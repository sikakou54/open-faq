## 2. システム概要

### 2.1 背景

FAQ AI ウィジェット SaaS は、顧客サイトに埋め込み可能なウィジェットを通じてエンドユーザーの質問に AI が自動回答する SaaS サービス。FAQ 限定ポリシーで AI が学習しない・新事実を生成しない方針を採用し、企業 FAQ コンテンツのみを参照する設計。

### 2.2 目的

- エンドユーザーからの質問を AI が自動回答(信頼度しきい値超過時のみ)
- 信頼度未達・FAQ 未登録の質問は管理者ユーザーが個別チャットで対応
- 対応結果を FAQ に登録することで自動回答率を継続的に改善
- 全運営費用を従量課金制で利用者に転嫁(MVP は単一プラン + 月次無料枠 + 14 日トライアル)

### 2.3 システムで解決する課題

- 24/7 のエンドユーザー対応の自動化(AI 回答による即時応答)
- 管理者ユーザーの対応コスト削減(高頻度質問の FAQ 化)
- 不正確な AI 回答による顧客誤誘導の防止(FAQ 限定 + 信頼度しきい値)

### 2.4 対象ユーザー

| ユーザー種別 | 概要 |
|---|---|
| オーナーアカウント | 契約あたり 1 アカウント固定、全権 + オーナー専有機能(課金 / 退会 / 規約再同意)|
| メンバーアカウント | 0..N、5 種権限フラグ(`faq:manage` / `chat:respond` / `users:manage` / `project:manage` / `logs:view`)+ プロジェクト割当で制御 |
| エンドユーザー | ウィジェット利用者(顧客サイト訪問者)|
| サービス運営者 | 本 SaaS 提供者(運営者画面 = [02_運営者システム/02_基本設計書.md](../02_運営者システム/02_基本設計書.md))|

### 2.5 対象業務

- FAQ 管理(登録 / 編集 / 公開 / 削除 / インポート / AI 下書き生成)
- 未解決質問対応(個別チャット / FAQ 候補化)
- ウィジェット設置 / 設定
- 利用量・課金管理 / 月次集計 / 退会
- メンバー / プロジェクト管理 / 権限管理
- お知らせ受信 / 規約再同意

### 2.6 SaaS としての提供形態

- **マルチオーナー境界モデル**: 全テーブルに `owner_account_id` カラム + API 層でオーナー境界判定(`actor.owner_account_id == target.owner_account_id`)+ 違反時は **404 偽装**
- 単一 D1 + owner_account_id 分離方式(150 契約到達時に D1 シャーディング設計を着手)
- 課金: **MVP 単一プラン** = 完全従量課金制 + 月次無料枠 + 14 日トライアル(階層プランは Future)

## 3. システム全体構成

### 3.1 システム構成概要

Cloudflare スタック上に構築:

| コンポーネント | 役割 |
|---|---|
| Cloudflare Workers | 全 API 実装(管理 API / ウィジェット API / 連携 IF 受信側) |
| Cloudflare D1 | プライマリ DB(SQLite 互換、`owner_account_id` で分離) |
| Cloudflare KV | キャッシュ(レート制限 / セッション / AI しきい値 / 許可ドメイン) |
| Cloudflare R2 | オブジェクトストレージ(請求書 PDF / DLQ 退避) |
| Cloudflare Queues | 非同期処理(メール配信 / 月次集計 / お知らせ fan-out) |
| Cloudflare Workers AI | AI 推論(`@cf/meta/llama-3.1-8b-instruct` 等) |
| Stripe | 課金決済(サブスクリプション / Smart Retries / Credit Note) |
| Resend | メール配信(共通ドメイン送信、サプレスリスト全契約横断管理) |

### 3.2 アーキテクチャ方針

- **マルチオーナー境界モデル**: §2.6 参照
- **抽象化 IF**: AnswerProvider(AI 推論)/ EmailProvider(メール配信)を抽象化し、プロバイダ差し替え可能(詳細は [個別設計書群/03_API設計書.md §5.11-5.12](個別設計書群/03_API設計書.md))

### 3.5 マルチオーナー境界方針

**重要**: 本サービスは旧来の分離概念を廃止し、**オーナー直接所有モデル** を採用する。

- データ分離単位 = `accounts.owner_account_id`(オーナー行は `is_owner=1` AND `owner_account_id = id`)
- API 層で `actor.owner_account_id == target.owner_account_id` を必須検証
- 違反時は **404 偽装**(403 を返すと相手契約の存在が漏れる)
- 全契約横断テーブル: `email_suppression_list` / `terms_versions` / `service_announcements`(配信先は契約別)

## 5. 設計方針

### 5.1 画面設計方針

- 共通 UI 部品(Header / Sidebar / PageHeader / Breadcrumb 等)+ UI/UX 共通要件追加部品(Timeline / AutosaveIndicator / UnsavedChangesGuard 等)を統一
- レスポンシブ方針: PC ≥ 1024 / Tablet 768-1023 / SP < 768
- 色 + アイコン + テキストの三重識別(FR-390)
- 詳細は [個別設計書群/02_画面設計書.md](個別設計書群/02_画面設計書.md) 参照

### 5.2 API 設計方針

- REST + JSON、エラーは RFC 7807(`application/problem+json`)
- 共通仕様: Base URL `/api/v1`(管理)/ `/widget/v1`(ウィジェット)/ `/internal/admin-integration/v1`(内部連携)
- 冪等性キー: `Idempotency-Key` ヘッダ
- 連携 IF #1〜#12: mTLS + JWT
- 詳細は [個別設計書群/03_API設計書.md](個別設計書群/03_API設計書.md) 参照

### 5.3 データ設計方針

- D1(SQLite)、`owner_account_id` で分離、ULID 主キー
- 命名規則: snake_case 複数形(テーブル)/ snake_case 単数形(カラム)
- 暗号化方針: `{column}_encrypted`(AES-256-GCM)+ `{column}_hmac`(検索用)の 2 列構成
- 保持期間: §11.4 / 詳細は [個別設計書群/04_テーブル定義書.md](個別設計書群/04_テーブル定義書.md) 参照

### 5.4 マルチオーナー境界設計方針

- 全テーブルに `owner_account_id` 必須(全契約横断テーブルを除く)
- API / 認可レイヤで境界検証 → 違反時 **404 偽装**
- 150 契約到達時に D1 シャーディング設計を着手(Future)

### 5.5 認証・認可設計方針

- パスワード認証(Argon2id)+ MVP は admin に MFA を必須化しない(リスク受容)
- 重要操作の再認証(FR-005、5 分以内 / 1 回限り)
- ログイン失敗ロックアウト(5 回 / 15 分、`IP × user_id`)
- Principal は KV キャッシュ TTL 60s
- 詳細は [個別設計書群/09_認証認可設計書.md](個別設計書群/09_認証認可設計書.md) 参照

### 5.6 権限設計方針

- ロール: `admin` / `service_operator` / `end_user`
- メンバー権限フラグ 5 種(`faq:manage` / `chat:respond` / `users:manage` / `project:manage` / `logs:view`)
- オーナー専有機能: 課金 / 退会 / 規約再同意
- オーナー保護(FR-333) / 自己操作不可(FR-334) / 単一オーナー所属(FR-335)
- 詳細は [個別設計書群/05_権限設計書.md](個別設計書群/05_権限設計書.md) 参照

### 5.7 エラー設計方針

- エラー分類 11 種 + プレフィックス(`E-INPUT-*` / `E-AUTH-*` / `E-AUTHZ-*` / `E-BIZ-*` / `E-QUOTA-*` / `E-BILL-*` / `E-SYS-*` / `E-EXT-*` / `E-IF-NNN-*` / `E-WH-*` / `E-LEGAL-*`)
- 主要設計原則 A〜D(透過再試行 / ユーザー再試行 / 即時 fail-fast / 業務分岐)
- 認可エラーは 403 統一、**オーナー境界違反は 404 偽装**
- **`case_status=closed` の自動遷移禁止**(FR-079、auto retention で `case_status` を **変更しない**、admin の明示操作のみで自動遷移させず)
- 詳細は [個別設計書群/06_エラー設計書.md](個別設計書群/06_エラー設計書.md) 参照

### 5.8 メッセージ設計方針

- メッセージ ID 体系: `MSG-<画面ID>-<種別>-<連番>`
- 通知重要度 4 値: `low` / `normal` / `high` / `critical` ── `critical` はメール強制送信(オーナー + `users:manage` 保持者へ配信)
- 文言の共通基準(ボタン文言ルール、エラーメッセージルール等)
- i18n キー化必須(MVP 日本語のみ、Future で多言語対応)
- 詳細は [個別設計書群/07_メッセージ一覧.md](個別設計書群/07_メッセージ一覧.md) 参照

### 5.9 セキュリティ設計方針

- PII 暗号化: AES-256-GCM + オーナー派生鍵
- 鍵管理: Master Key 年次ローテーション + 60 日 dual-decrypt
- Web 脆弱性対策: CSP / HSTS / X-Frame-Options / Double Submit Cookie / XSS 対策
- 監査ログ 3 区分保持(1y / 5y / 7y、運営者側が正本のハッシュチェーン構造)
- 不正利用検知(FR-195 / NFR-308): メイン側で実装、IF #9 で運営者へ通知
- メール配信信頼性: `email_suppression_list` 全契約横断 + DLQ 4 日 / R2 30 日
- プロンプト注入対策: AC-019 × AC-035 の 2 段構えテスト
- 詳細は [個別設計書群/10_セキュリティ設計書.md](個別設計書群/10_セキュリティ設計書.md) 参照

### 5.10 課金・契約設計方針

- **MVP 単一プラン**: 完全従量課金制 + 月次無料枠 + 14 日トライアル
- 契約状態: `accounts.contract_status` 4 値(active / suspended / deleted_pending / deleted)
- 課金単位: 質問数(1,000 件/月) / FAQ 件数(100 件) / 個別チャット部屋数(30 部屋/月)
- 80/100/125% 三段階アクション(FR-122、**質問数は拒否しない・事後課金**)
- Stripe Smart Retries 委譲(独自リトライ禁止)+ 7 日猶予
- 契約別利用上書き: `owner_quota_overrides.resource_kind` で統一実装(運営者操作 = SCR-093)
- 詳細は [個別設計書群/11_課金請求設計書.md](個別設計書群/11_課金請求設計書.md) 参照

### 5.11 通知 / AI 設計方針

#### 5.11.1 通知方針

- 通知重要度 4 値 + `critical` メール強制送信(§5.8)
- 通知契機 21 種(FR-140〜149b、メイン側)+ 通知テンプレート 11 種
- メール配信信頼性 + DLQ(§5.9)

#### 5.11.2 AI 方針

- AI スコアリング(信頼度・関連度しきい値)、3 階層適用(`global` / `owner` / `project`、`ai_threshold_persistent_cache`)
- 矛盾検知(複数 FAQ 間で内容食い違い検出 → 未解決登録)
- AI モデル切替時の回帰テスト(FR-063、20 ケース以上、四半期更新)
- FAQ 限定ポリシー固定(プロンプト注入対策、§5.9)
- AI 学習利用禁止(`learn=false` 強制)

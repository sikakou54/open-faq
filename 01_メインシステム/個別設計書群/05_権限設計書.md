## 2. ロール一覧

(§3.1 から移管)

| ロールID | ロール名 | 概要 | 利用者 | DB 値 |
|---|---|---|---|---|
| OWNER | オーナーアカウント | 契約あたり 1 アカウント固定、全権 + オーナー専有機能 | 契約代表者 | `role='admin'` AND `is_owner=1` AND `owner_account_id=id` |
| MEMBER | メンバーアカウント | 0..N、権限フラグ + プロジェクト割当で制御 | 契約参加者 | `role='admin'` AND `is_owner=0` AND `owner_account_id=<owner_id>` |
| END_USER | エンドユーザー | ウィジェット利用者(自分の問い合わせ ID のみ)| 一般訪問者 | `role='end_user'` |
| SERVICE_OPERATOR | 運営者 | 本 SaaS 提供者(運営者側)| 運営事業者 | `role='service_operator'` |

## 3. 権限フラグ定義(`account_permissions.permission_kind` 5 値)

(§3.2.0 から移管)

### 3.1 権限フラグ概要

| `permission_kind` | 表示名(UI)| できること |
|---|---|---|
| `faq:manage` | FAQ 管理 | FAQ の登録・編集・削除・公開、FAQ AI 下書き生成、FAQ インポート |
| `chat:respond` | 個別チャット対応 | 未解決質問の閲覧・対応状態更新・担当割当、個別チャット返信、対応不要終了 |
| `users:manage` | ユーザー管理 | メンバーの招待・権限フラグ変更・停止・削除、招待再送・取消(オーナー対象は除く)|
| `project:manage` | プロジェクト設定 | プロジェクトの作成・編集・削除、ウィジェット設定、許可ドメイン設定 |
| `logs:view` | ログ参照 | 質問ログ閲覧、自契約分の監査ログ閲覧、データエクスポート、未解決質問・個別チャット内容の閲覧 |

### 3.2 主管 SCR / FR 対応

| `permission_kind` | 主な操作対象 | オーナーとの関係 |
|---|---|---|
| `faq:manage` | SCR-012(FAQ 管理)、FR-040〜046, FR-104, FR-310 | オーナーは常時保持 |
| `chat:respond` | SCR-011(未解決質問)、SCR-013(個別チャット)、FR-070〜079, FR-086 | オーナーは常時保持 |
| `users:manage` | SCR-017(ユーザー管理)、FR-015〜021 系列 | オーナーは常時保持し、保持メンバーがいなくても運用継続可 |
| `project:manage` | SCR-010 / SCR-014、FR-030〜036, FR-150 | オーナーは常時保持 |
| `logs:view` | SCR-015、質問ログ・自契約監査ログ、FR-070, NFR-602 系列 | オーナーは常時保持 |

### 3.3 オーナー専有機能(メンバーには `account_permissions` で付与不可)

- **課金**(SCR-015 の月次予算変更 / 支払い方法変更)
- **退会**(SCR-024)
- **規約再同意**(SCR-025 の同意 / 不同意)

UI 上は非表示 + API は 403(`E-AUTHZ-OWNER-ONLY`)。

| 機能ID | 機能名 | 必要権限 | オーナー | 制限事項 |

| F-PRJ-* | プロジェクト | `project:manage` | ◯ | 削除は再認証必須 |

## 7. 契約別利用制限 / 運営者上書き

### 7.1 単一プランの利用上限([11_課金請求設計書.md §3](11_課金請求設計書.md) 参照)

MVP は単一プラン(完全従量課金 + 月次無料枠 + 14 日トライアル)。プラン階層による差別化はない。利用上限は **`owner_quota_overrides.resource_kind`** で統一実装される(テーブル定義書 04 §3.26 参照)。

| `resource_kind` | 用途 | 主管 SCR(運営者)|
|---|---|---|
| `widget_ask_per_min` | ウィジェット質問レート | SCR-093 |
| `widget_inquiry_per_min` | ウィジェット未解決質問送信レート | SCR-093 |
| `widget_chat_per_min` | ウィジェットチャット投稿レート | SCR-093 |
| `email_per_hour` | メール送信レート | SCR-093 |
| `admin_api_per_min` | 管理 API レート | SCR-093 |

### 7.2 80/100/125% 三段階アクション([11_課金請求設計書.md §8](11_課金請求設計書.md) の参照)

| データ種別 | 80% | 100% | 125% |
|---|---|---|---|
| 質問数 | inbox 通知(`normal`)| inbox + email(`high`)、**拒否しない**(事後課金)| 翌 1 時間レート制限 1/2 + 運営者エスカレーション |
| FAQ 件数 | inbox 通知 | 新規登録拒否(編集可)| + 運営者エスカレーション |
| 個別チャット部屋数 | inbox 通知 | 新規作成拒否(返信可)| + 運営者エスカレーション |

詳細は [11_課金請求設計書.md §8](11_課金請求設計書.md) を正本とする。

## 8. 権限不足・契約上書き時の動作

| ケース | システム動作 | 画面表示 | メッセージID | 関連エラーID |
|---|---|---|---|---|
| 権限不足(`faq:manage` 未保持で SCR-012 編集)| 403 統一 | 「権限がありません」+ オーナーへ連絡誘導 | MSG-COMMON-AUTHZ-403 | E-AUTHZ-FORBIDDEN |
| **オーナー境界違反**(他契約のリソース参照)| **404 偽装**(403 だと相手契約の存在が漏れる)| 「リソースが見つかりません」 | MSG-COMMON-NOT-FOUND | E-AUTHZ-OWNER-BOUNDARY |
| プロジェクト割当不足 | 404 偽装 | 「リソースが見つかりません」 | MSG-COMMON-NOT-FOUND | E-AUTHZ-PROJECT-DENIED |
| オーナー専有機能(`is_owner=0` でアクセス) | 403 + 専用コード、UI 非表示 | 「この機能はオーナーのみご利用いただけます」 | MSG-COMMON-OWNER-ONLY | E-AUTHZ-OWNER-ONLY |
| オーナー保護違反(オーナー停止 / 削除)| 403 | 「オーナーアカウントは停止・削除できません」 | MSG-COMMON-OWNER-PROTECTED | E-AUTHZ-OWNER-PROTECTED |
| 自己操作不可(自分の `users:manage` 剥奪)| 403、UI 非活性 | 「自分の権限は剥奪できません」 | MSG-COMMON-SELF-MUTATION | E-AUTHZ-SELF-MUTATION |
| 契約停止中(`contract_status=suspended`)| 403 + 専用コード、課金画面 / 退会画面 / データエクスポートのみ許可 | サスペンション理由 + 再決済誘導 | MSG-BILL-CONTRACT-SUSPENDED | E-BIZ-CONTRACT-SUSPENDED |
| 月次無料枠 100% 超過(質問数以外)| 429 + `Retry-After` | 「上限に到達しました」+ 課金ダッシュボード誘導 | MSG-QUOTA-* | E-BIZ-QUOTA-EXCEEDED |

詳細は [06_エラー設計書.md §5](06_エラー設計書.md) を正本とする。

## 9. 緊急時のみ可操作(§3.2.1-2 から移管)

### 9.1 「△ 緊急時のみ可」操作の発動条件

| 条件 | 実装方針(メイン側) |
|---|---|
| 1. サポートチケット ID 必須 | 連携 IF #4 / #5 / #6 / #12 のリクエスト JSON に `ticket_id` フィールドを必須化。メイン側で空文字検証 + 監査ログ記録 |
| 2. 運営者 2 名の承認(自己承認不可)| 顧客管理側で承認を取得し、結果を連携 IF 経由で伝搬。メイン側は **メタフィールド `approvals[]`(承認者 2 名分)** の検証(承認者 ID が異なる、承認時刻が 24 時間以内、kid 検証)を実施 |
| 3. 管理者ユーザーへの即時通知 | メイン側で `inbox_messages` レコードを生成(`category=system`、`importance=high`)+ メール送信を Queue 投入 |
| 4. 監査ログ必須記録 | `<resource>.<verb>.by_operator` 形式の `action` コードで `audit_logs` に記録。`retention_class='operator_high_priv'`(5 年保持)|

### 9.2 緊急区分 × 検知ハンドラ × 承認 SLA

| 緊急区分 | 検知ハンドラ | 承認 SLA | エスカレーション経路 |
|---|---|---|---|
| 重大障害 | `monitoring:main_outage` 監視 / 連携 IF #2 ヘルスチェック(5s × 3 連続失敗)| 15 分以内に 2 名承認 | RB-021(メイン障害時の運営者側縮退運転)と連動 |
| 全員ロックアウト | `audit_logs(auth.locked)` 集約検知(該当契約内全 admin が `auth.locked` 状態)| 30 分以内に CS マネージャーへ通報 | サポート窓口経由でオーナーと連絡、紙ベース回復コード(RB-014)も可 |
| セキュリティインシデント | NFR-602c 監査ログ完全性検証バッチ / SIEM / 不正アクセス検知(FR-178)| 即時 PagerDuty 起動 | セキュリティ責任者 → CTO → 運営本部 |
| 法令対応 | (運用側で個別判断)| 個別判断 | 法務 → CTO |

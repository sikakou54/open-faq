## 1. 運用方針

| 項目 | 内容 |
|---|---|
| 基本方針 | 本番運用は Cloudflare Workers / D1 / R2 / KV / Queues の SaaS マネージドサービスを前提とし、サービス運営者は SLO 遵守・障害一次切り分け・KPI 監視・データ整合性検証に責任を持つ |
| データリージョン | 日本リージョン(apac)固定。データ越境ゼロ(APPI / NFR-805 整合) |
| SLO 目標 | 公開ウィジェット可用性 99.9%(NFR-201)、管理画面可用性 99.5%(NFR-202)、API p95 各 NFR 値以下 |
| エラーバジェット | 短期(1h)と長期(24h)の二重 burn-rate 監視 |
| オンコール | MVP は 1 名(24/7、PagerDuty 受信)、平日昼帯は日中業務として一次受け |
| 計画停止 | 月次合計 2 時間まで、深夜 02:00-03:00 JST、7 日前事前告知 |
| 障害告知 | 公開ステータスページ + 管理画面お知らせ受信箱(system 種別)+ critical 時メール の 3 経路併用 |
| 環境分離 | dev / staging / prod の 3 環境。prod デプロイは 2 名承認(GitHub OIDC + Environment Protection Rules) |

## 2. 運用対象

| 区分 | 対象 | 補足 |
|---|---|---|
| アプリ | `@faq-saas/main-api`、`@faq-saas/widget-api`、`@faq-saas/internal-api`、`@faq-saas/webhook`、`@faq-saas/queue-consumer`、`@faq-saas/cron`、`@faq-saas/admin`、`@faq-saas/widget`、`@faq-saas/public` | Cloudflare Workers / Pages |
| DB | D1(`main-db-prod`) | D1 Time Travel(直近 30 日) |
| インフラ | Cloudflare Workers + KV + R2 + Queues + Cron Triggers | apac リージョン |
| 外部サービス | Stripe(課金 Webhook)、Resend(メール送信)、Workers AI(AI 推論) | 連携 IF #1〜#12 |
| バッチ | tombstone 日次 / 未解決質問保持期間処理 / 月次集計 / 月次確定 cron / 無料枠超過・支払方法未登録判定 / 監査ログ完全性検証 / RetentionConsistencyChecker 等 | §06_運用手順 §1 を参照 |
| ログ | アプリログ(R2 / 90 日)、エラーログ(180 日)、監査ログ(1y / 5y / 7y) | JSON Lines |
| 監視対象 | NFR-804 (a)〜(h) の 8 KPI + AI 品質メトリクス + D1 容量 + 連携 IF 状況 + サーキットブレーカ状態 | Cloudflare Workers Analytics Engine |

## 3. 運用体制

| 役割 | 担当範囲 | 連絡経路 | エスカレーション |
|---|---|---|---|
| 第 1 階層 on-call | 一次切り分け / Runbook 実行 / 5 分以内 Ack | PagerDuty(critical)/ Slack(normal-high) | 15 分未対応で第 2 階層 |
| 第 2 階層 on-call backup | 第 1 階層の代替 / 一次対応継続 | PagerDuty | さらに 15 分未対応で第 3 階層 |
| 第 3 階層 on-call manager | リソース調整 / サービス縮退 / 計画停止判断 | 個人携帯 | 30 分未対応で経営陣 |
| 経営陣(CTO / SRE Lead) | データ損失リスク評価 / 顧客告知トーン決定 | 全社オンコール(`#incident-all-hands`) | — |
| PO + 開発リード | 重大障害時の意思決定支援 / 顧客対応 | メール + Slack | — |
| サポート | 管理者からの問い合わせ一次対応(平日 9:00〜18:00 JST) | メール + チャット | オンコールへ引継 |

エラー設計書(メイン)

このページは、メインシステムで発生するエラーの ID 体系・発生条件・HTTP ステータス・記録方針を定義し、その正本とします。 あわせて、AI が回答できなかった「未解決登録分岐」はエラーではなく正常な業務分岐であることを明確にし、両者を別カテゴリとして切り分けます。

2. エラー分類

この節では、メインシステムのエラーをプレフィックスで分類します。E-AUTH-*(認証)/ E-AUTHZ-*(認可)/ E-BIZ-*(業務)は本ページが正本です。表のあとに、§3 で「処理エラー」と「未解決登録分岐(正常)」を別カテゴリとして切り分けます。

エラー分類 概要 プレフィックス 代表例
入力エラー バリデーション失敗 E-INPUT-* 必須項目欠落 / 形式不正 / 重複
認証エラー 認証失敗・期限切れ E-AUTH-* パスワード不一致 / ロックアウト / トークン期限切れ
認可エラー 権限不足 E-AUTHZ-* プロジェクトロール不足 / オーナー境界違反 / オーナー専有機能 / 最後の管理者保護
業務エラー ビジネスルール違反 E-BIZ-* 未解決質問状況の直接変更禁止 / 退会済み / 論理削除済みリソースへの操作

3. 処理エラーと未解決登録分岐の分離

処理エラー(通信障害・上流障害・入力不備・認可エラー)と、AI が回答できなかった未解決登録分岐(FAQ なし・信頼度不足・FAQ 矛盾)は、まったく別の事象です。前者はエラーであり未解決登録しません。後者はエラーではなく正常な業務分岐であり、未解決質問として登録します(FR-111 / FR-057)。この切り分けを誤ると、障害を未解決質問に取り込んでしまうため、両者を別カテゴリとして設計します。

3.1 処理エラー(未解決登録しない)

処理エラーは AI 回答の前提が崩れた・操作が成立しなかった事象です。回復可能性に応じて A〜C の 3 カテゴリへ分け、いずれも未解決質問として登録しません。AI 推論のタイムアウト(8 秒超過、FR-347)は処理エラーであり、信頼度不足による未解決分岐とは区別します。

カテゴリ 該当事象 自動リトライ ウィジェット利用者への提示 HTTP 未解決登録
A. 透過再試行可 一時ネットワーク断・一過性の上流障害 ◯ サーバ側指数 BO 3 回 「しばらくしてから再度お試しください」(成功時は通常応答) 200(成功)/ 503(失敗) しない
B. ウィジェット利用者再試行 レート超過・ロックアウト・上限到達・サスペンション × 「再試行してください」+ 待機時間表示(Retry-After) 429 / 423 しない
C. 即時 fail-fast 入力不備・認証 / 認可エラー・前提条件未充足・AI 推論タイムアウト(8 秒超過、FR-347)・サーバ内部エラー × エラー固有メッセージ + 是正案内(5xx はエラー識別子を提示、FR-113) 400 / 401 / 403 / 404 / 409 / 410 / 422 / 500 / 503 しない
AI 推論タイムアウトは処理エラーAI 推論が 8 秒(FR-347)を超過した場合は推論を打ち切り、処理エラー(カテゴリ C)として扱います。信頼度・関連度しきい値の未達による未解決分岐(§3.2)とは異なり、未解決質問としては登録しません。該当エラー ID は E-SYS-WORKERS-AI-TIMEOUT です。

3.2 未解決登録分岐(エラーではない正常分岐)

AI が登録済み FAQ で回答できなかった場合は、エラーではなく正常な業務分岐として扱います。HTTP 200 を返し、管理用の問い合わせ ID を付与して未解決質問を登録します(FR-056)。ウィジェットには問い合わせ ID を表示せず、回答できなかった旨と確認済みプロジェクト連絡先メールをシステム返信として案内します(FR-156d)。

分岐理由 発生条件 根拠 FR HTTP 登録 / 案内
FAQ なし 質問に該当する公開中 FAQ が関連度しきい値未満で見つからない FR-056 200 未解決登録 + 連絡先案内(問い合わせ ID 非表示)
信頼度不足 AI 回答の信頼度がしきい値未満で確定回答にできない FR-054 200 同上
FAQ 矛盾 参照 FAQ 同士に矛盾があり断定回答できない FR-054 200 同上
しきい値の正本関連度・信頼度のしきい値(MVP 初期値 関連度 0.50 / 信頼度 0.60)と階層適用・矛盾検知方式は要件 FR05 AI 回答 および FR20 AI 推論動作 を正本とします。本ページは分類のみを扱い、しきい値そのものは再掲しません。各分岐に対応するエラー ID は E-BIZ-AI-*(分類上は業務分岐)です。

4. エラー一覧

この節では、分類ごとのエラー ID を発生条件・メッセージ ID・HTTP ステータス・システム動作・ログレベルとともに定義します。HTTP ステータスは本ページが正本です。メッセージ文言の全文は 06_メッセージ一覧.html を正本とし、ここではメッセージ ID で対応づけます。

4.7 システムエラー(E-SYS-*)

サーバー内部起因のエラーは 500 で扱い、内部詳細を応答に含めず、利用者にはエラー識別子のみを提示します(FR-113)。AI 推論の 8 秒タイムアウト(FR-347)は E-SYS-WORKERS-AI-TIMEOUT として処理エラー扱いとし、未解決登録しません。

E-SYS-INTERNAL サーバー内部エラー / 予期しない例外 500 エラー ID を利用者提示(FR-113)、内部詳細は応答に含めない ERROR
E-SYS-DB DB エラー 500 同上 ERROR
E-SYS-WORKERS-AI-TIMEOUT AI 推論のタイムアウト超過(8 秒、FR-347)。超過時に推論を打ち切る 503 処理エラー(カテゴリ C)として扱い未解決登録しない。再試行案内を提示(FR-112)。信頼度不足の未解決分岐(§3.2)とは区別する WARN
E-SYS-NETWORK-TRANSIENT 一時ネットワーク断 503 → 200(成功) サーバ側指数 BO 3 回(分類 A 透過再試行可) INFO

E-EXT-EMAIL-FAILED Resend 配信失敗 notification_logs.delivery_state='failed' → 自動 BO 3 回(最大 24h) WARN
E-EXT-EMAIL-BOUNCED メール bounce(hard) email_suppression_list 即時追加(全契約横断) INFO
E-EXT-EMAIL-SOFT-BOUNCE メール bounce(soft)5 連続 サプレス登録(復帰可、SCR-093 経由) INFO
E-EXT-EMAIL-COMPLAINT メール苦情(email.complained) 永久サプレス(復帰不可) WARN

4.9 連携 IF エラー(E-IF-NNN-*)

顧客管理システムとの連携 IF #1〜#12 のエラーです。冪等性違反はリトライ安全性のため 200 + 既存処理結果を返し、認証失敗は 401 + 監査記録とします。IF 仕様の正本は 02_API設計.html の連携 IF 節です。

E-IF-COMMON-SCHEMA JSON Schema 不整合 400 WARN
E-IF-001-* IF #1(契約停止イベント)関連 - WARN
E-IF-002-* IF #2(強制ログアウト)関連 - WARN
E-IF-004-* IF #4(復元実行)関連 - WARN
E-IF-005-* IF #5(レート制限上書き)関連 - INFO
E-IF-006-* IF #6(AI しきい値上書き)関連 - INFO
E-IF-007-* IF #7(お知らせ生成)関連 - INFO
E-IF-008-* IF #8(監視メトリクス取得、送信側)関連 - INFO
E-IF-009-* IF #9(不正利用検知通知、送信側)関連 - WARN
E-IF-010-* IF #10(課金 Webhook 受信)関連 - WARN

4.10 Webhook エラー(E-WH-*)

外部 Webhook(Resend / Stripe)受信のエラーです。署名検証失敗は 401 + 監査記録、冪等性違反は 200 + 既存処理結果とし、ペイロード差分検出時は自動上書きを禁止して運営者レビュー待ちにします。

エラーID 発生条件 システム動作 ログレベル
E-WH-SIGNATURE-INVALID Resend / Stripe 署名 HMAC 不正 401 + 監査記録 + 運営者 high alert WARN
E-WH-IDEMPOTENCY-VIOLATION event_id 重複 200 + 既存処理結果 INFO
E-WH-PAYLOAD-DIFF 同 event_id + ハッシュ不一致 200 + webhook_payload_diffs INSERT + 運営者 high alert + 自動上書き禁止 WARN
E-WH-DLQ-FAILED DLQ 自動 BO 3 回失敗 永久失敗 → 運営者手動リプレイ待ち WARN

4.11 規約エラー(E-LEGAL-*)

規約・プライバシーポリシー未同意のエラーです。未同意は SCR-015 の再同意フローへ誘導し、同意期限超過時は機能制限ガードを適用します。再同意フローの根拠は 08_認証・認可設計.html §6 を正本とします。

エラーID 発生条件 システム動作
E-LEGAL-TERMS-NOT-AGREED 規約未同意 SCR-015 へリダイレクト
E-LEGAL-TERMS-EXPIRED 同意期限超過(発効 + 14 日) 機能制限ガード(課金 / 新規プロジェクト不可、FAQ / ウィジェット稼働は継続)

5. 異常系共通方針

この節では、分類ごとの共通的な扱い方針を定めます。再試行案内・連打防止のリトライ回数とクールダウンは §5.10 を本ページの正本とします。

5.1 入力エラー時の方針

入力バリデーション失敗は 400 とフィールド単位のエラー配列で返し、画面側で各項目の近くにインライン表示します。値域・範囲チェックは入力スキーマで検証します。

  • 必須項目欠落 / 形式不正は 400 + フィールド単位のエラー配列 を返却
  • メッセージは画面側で項目近くにインライン表示
  • 文字数 / 範囲チェックは入力スキーマで検証(02_API設計.html を正本)

5.2 認証エラー時の方針

認証失敗・期限切れは 401 に統一し、攻撃者へヒントを与えません。ログイン 5 回失敗で (IP × user_id) を 15 分ロックアウトし、解除は時間経過のみです。

  • パスワード不一致 / トークン期限切れは 401 統一
  • ログイン失敗 5 回で (IP × user_id) を 15 分ロックアウト(FR-007。フローは 08_認証・認可設計.html §4.1 が正本)
  • ロックアウト解除は時間経過のみ(管理者解除なし)
  • 攻撃者へのヒント禁止(メール / パスワードのどちらが誤りかを示さない)

5.3 認可エラー時の方針

権限不足は 403 に統一し、レスポンスに権限名を含めません。オーナー境界違反は他契約リソースの存在を漏らさないため 404 偽装とし、クロス契約アクセス検知(AC-017)は月次自動検証で確認します。

  • プロジェクトロール不足は 403 統一
  • オーナー境界違反は 404 偽装(403 だと相手契約のリソース存在が漏れる)
  • レスポンスボディには具体的な権限名を含めない(列挙攻撃対策)
  • AC-017(クロス契約アクセス検知)は 月次自動検証

5.4 業務エラー時の方針

FAQ の状態(draft / published / hidden)は相互に自由遷移でき、専用の公開ガード・公開エラーは設けません。ステータス値域の妥当性は API 側で必ず検証します。

  • FAQ の状態(draft / published / hidden)は相互に自由遷移でき、専用の公開ガード・公開エラーは設けない(SCR-006「保存」で状態をそのまま保存)。status の値域は CHECK 制約(draft/published/hidden/deleted)で検証する
  • ステータス値域の妥当性は API 側で必ず検証(画面側だけの判定に頼らない)

5.5 利用上限エラー時の方針(プロジェクト単位)

質問数の月次上限件数 100% 到達は 429 + Retry-After で当該プロジェクトのウィジェット受付を停止します。無料枠超過分の課金は別概念であり、同時接続上限は 503 + Retry-After で絞ります。

  • 質問数の月次上限件数 100% 到達は 429 + Retry-After(秒数 = 月末までの秒数)。支払方法の有無に関わらず当該プロジェクトのウィジェット新規質問受付を停止し、エンドユーザーへ連絡先メール誘導する(E-QUOTA-QUESTIONS-LIMIT、§4.5)。FAQには月次上限エラーを設けない
  • 無料枠超過分の課金(無料枠 の超過)は事後課金(支払方法登録済)。これはウィジェット停止の 上限件数 判定とは別概念(FR-122)
  • 同時接続上限は 503 + Retry-After(NFR-114)

5.6 課金エラー時の方針

契約停止中(contract_status=suspended)は 403 で課金 / 退会画面のみ許可します。支払い失敗は 7 日猶予期間のカウントを開始し、critical メールを通知します。

  • 契約停止中(contract_status=suspended)は 403 + 専用コード E-BILL-CONTRACT-SUSPENDED
  • 課金画面 / 退会画面のみ許可(FR-138)
  • 支払い失敗 → 7 日猶予期間カウント開始 + メール通知(critical)

5.7 外部連携エラー時の方針

AI 推論失敗は質問数を計測しつつ課金対象外とし、タイムアウトも処理エラー扱いで未解決登録しません。メール配信失敗は自動バックオフ再送し、bounce / 苦情はサプレスリストで再送を抑止します。

  • AI 推論失敗: 質問数は計測するが metering_billable=false(課金対象外)。タイムアウト(8 秒、FR-347)も処理エラー扱いで未解決登録しない
  • Resend 配信失敗: notification_logs.delivery_state='failed' → DLQ 投入 + 自動 BO 3 回(最大 24h)
  • メール bounce: email_suppression_list 即時追加(全契約横断、R-003 共通ドメイン対策)
  • メール苦情(complained): 永久サプレス、復帰不可

5.8 連携 IF エラー時の方針

連携 IF の冪等性違反はリトライ安全性のため 200 + 既存処理結果を返します。認証失敗(mTLS / JWT)は 401 統一とし、JSON Schema 不整合は 400 + 監査記録とします。

  • 冪等性違反: 200 + 既存処理結果 を返却(リトライ安全性)
  • 認証失敗(mTLS / JWT): 401 統一、署名検証失敗は別ログ出力
  • JSON Schema 不整合: 400 + 監査記録

5.9 Webhook エラー時の方針

Webhook の署名検証失敗は 401 + 監査記録 + 運営者アラートで即時破棄します。冪等性違反は 200 + 既存処理結果を返し、ペイロード差分検出時は自動上書きを禁止して運営者レビュー待ちにします。

  • 署名検証失敗: 401 + 監査記録 + 運営者 high alert + 即時破棄
  • 冪等性違反: 200 + 既存処理結果
  • ペイロード差分検出: 200 + webhook_payload_diffs INSERT + 自動上書き禁止(SCR-098 レビュー待ち)

5.10 システムエラー・再試行案内・記録(FR-112 / FR-113 / FR-114 正本)

サーバー内部起因のエラーは 500 で内部詳細を隠し、利用者にはエラー識別子のみを提示します。再試行のリトライ回数・クールダウン、提示する識別子、記録から除外する項目は本ページを正本とします(FR-113 / FR-112 / FR-114)。

  • システムエラー: 500 統一、内部詳細(スタックトレース・内部メッセージ)は応答に含めない
  • エラー識別子の利用者提示(FR-113): 5xx 系(カテゴリ C)はサポート問い合わせと紐付けられるようエラー識別子を画面に表示する。識別子は error_logs の主キーに対応づける
  • 監査が必要な操作の途中で発生した場合は audit_logs にも記録する
再試行案内・連打防止の正本値(FR-112)
  • 自動リトライ(カテゴリ A 透過再試行可): サーバ側で指数バックオフ 最大 3 回。回復しない場合のみウィジェット利用者へ手動再試行を案内する
  • 手動再試行: 再試行ボタンは押下後 3 秒クールダウンで連打を防止する。クールダウン中はボタンを非活性にする
  • 待機指示: カテゴリ B(429 / 423)は Retry-After の秒数を待機時間として表示する
記録から除外する項目(FR-114)エラー記録には個人情報(PII)・認証トークン・パスワードハッシュ・カード情報を含めない。ウィジェット利用者の入力は全文保存せず、先頭 100 文字 + SHA-256 ハッシュのみを最小限の手掛かりとして記録する。詳細は §7 を参照。

5.11 不正利用検知エラー時の方針(FR-195)

不正利用検知パターン(FR-195 / NFR-308)に該当した場合は IF #9 で運営者へ通知します。閾値超過時は自動サスペンションと管理者・メール通知を行います。

  • 不正利用検知パターン(FR-195 / NFR-308)に該当 → IF #9 送出 + 運営者通知
  • 閾値超過時は自動 suspension + 管理者 + email 通知

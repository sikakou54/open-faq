# 権限設計書(メイン)

## 1. 文書概要

### 1.1 目的

メインシステムの権限モデル(オーナー / プロジェクト管理者 / メンバー / エンドユーザー / 運営者)、プロジェクト別ロール 2 値、画面別・機能別・API 別の操作可否、**契約別利用上書き** を一元化する。

## 2. ロール一覧

利用者側は 3 つのロール(オーナー / プロジェクト管理者 / メンバー)で構成する。プロジェクト管理者とメンバーは「プロジェクト単位」のロールであり、同一メンバーがプロジェクト A では `admin`、プロジェクト B では `member`、プロジェクト C には未割当、といった組合せをとり得る。

| ロールID | 表示名 | 説明文(UI) | DB 値 |
|---|---|---|---|
| OWNER | 契約オーナー | 契約全体を管理し、プロジェクト作成・課金・退会などすべての設定を操作できます。 | `users` 行存在 AND `contract_owners.user_id = users.id` 行存在 |
| PROJECT_ADMIN | プロジェクト管理者 | 担当プロジェクトのFAQ、問い合わせ対応、メンバー招待・ロール変更を行えます。 | `users` 行存在 AND `contract_owners` 行を持たない AND `project_users.role='admin'`(プロジェクト単位)|
| MEMBER | プロジェクトメンバー | 担当プロジェクトのFAQや問い合わせを確認・対応できます。 | `users` 行存在 AND `contract_owners` 行を持たない AND `project_users.role='member'`(プロジェクト単位)|
| END_USER | チャット利用者 | ウィジェットから個別チャットを開始し、履歴確認・投稿ができます。 | `end_users` 行存在(プロジェクト単位識別) |
| SERVICE_OPERATOR | サービス運営者 | サービス提供者として、サポートや障害対応に必要な運用操作を行えます。 | **別 D1**(運営者システム `service_operators` テーブル)── メイン側 `users` には格納しない。メイン API への運営者アクセスは IF 経由の認可コンテキストで実現し、`audit_logs.actor_role='service_operator'` の値として記録する |

**用語の衝突回避**: 管理者の認証主体は `users` テーブル(全行が管理者)で、エンドユーザーは独立テーブル `end_users` で管理される。`project_users.role='admin'` はプロジェクト管理者ロールを示し、`users` テーブルのロール概念とは別軸。`audit_logs.actor_role` は操作者種別(`admin` / `end_user` / `system` / `service_operator`)を記録するもので、`users` のロール定義とは独立した概念(§ [03_テーブル設計.md §3.23](03_テーブル設計.md) 参照)。

## 3. ロール定義(`project_users.role` 2 値 + オーナー)

### 3.1 ロール概要

| ロール | 表示名(UI)| 範囲 | できること |
|---|---|---|---|
| オーナー | 契約オーナー | 契約全体 | すべてのプロジェクトと契約設定を管理できます。プロジェクト作成・課金・退会・規約同意・メンバー管理も行えます。 |
| `admin` | プロジェクト管理者 | 該当プロジェクト | 担当プロジェクトのFAQ公開、問い合わせ対応、ログ確認、データ出力、メンバー招待・ロール変更ができます。 |
| `member` | プロジェクトメンバー | 該当プロジェクト | 担当プロジェクトのFAQ編集、問い合わせ対応、ログ確認、データ出力ができます。 |

**オーナーの project_users 行**: オーナーはプロジェクト作成時に `project_users(role='admin', valid=1)` 行が自動 INSERT され、当該プロジェクトの明示的なプロジェクト管理者として行を保持する。認可上は Principal 構築時の `isOwner=true` 判定(`contract_owners` 行存在)で bypass が優先され、`project_users` 行は critical 通知の宛先解決と画面表示の事実情報として参照される(詳細は [08_認証・認可設計.md §6.1](08_認証・認可設計.md))。

### 3.2 ロール × 主管 SCR / FR 対応

| ロール | 主な操作対象 SCR / FR |
|---|---|
| オーナー | 全 SCR、FR-015a / FR-015c / FR-030 / FR-030a / FR-030b / FR-115〜127(課金)/ FR-189(退会)/ FR-193(API キー)/ 規約 |
| `admin`(プロジェクト管理者)| SCR-011 / 012 / 013 / 015(該当プロジェクト範囲)/ SCR-017 / SCR-017-M1(操作者が `admin` のプロジェクト範囲)/ 選択中プロジェクト詳細 / FR-015 / FR-016 / FR-018a〜c / FR-040〜046 / FR-070〜079 / FR-086 / NFR-602 系列 |
| `member` | SCR-001(ダッシュボード)/ SCR-011 / 012 / 013 / 015(該当プロジェクト範囲)/ FR-040〜046 / FR-070〜079 / FR-086(返信のみ)|
| **招待トークン保有者(未認証)** | **SCR-031 メンバーアカウント有効化のみ**(`access_tokens.purpose='activation'` 有効時のみ到達可能)/ FR-016 / FR-016a / FR-016d / FR-016e。本画面以外には到達不可、有効化完了後は SCR-001 ログインへ遷移して通常 Principal が構築される |
| **連絡先確認トークン保有者(未認証)** | **SCR-032 プロジェクト連絡先メール確認完了のみ**(`access_tokens.purpose='contact_verify'` 有効時のみ到達可能)/ FR-033a。アカウント作成は不要、第三者の共有メールアドレスでも本フローを進められる。確認完了後は静的ランディングに留まる |

### 3.3 オーナー専有機能(プロジェクト別ロールでは付与不可)

- **プロジェクトの作成・編集**(SCR-010 / SCR-010-M1、許可ドメイン等のプロジェクト設定全般を含む)
- **プロジェクトの削除**(SCR-015 プロジェクト情報パネルのオーナー専有 DangerSection のみに動線集約、`DELETE /projects/{id}`)── 削除時の挙動は論理削除(`projects.status='deleted' + valid=0`)+ 当該プロジェクトの `project_users` 全行 `valid=0` + 他 PJ 割当が無くなった(非オーナー)メンバーの `users.valid=0` 論理削除 + セッション失効
- **課金**(SCR-015 の月次予算変更 / 支払い方法変更)
- **退会**(SCR-024)
- **規約再同意**(SCR-025 の同意 / 不同意)
- **メンバーアカウント削除**(`DELETE /projects/{id}/members/{accountId}`、アカウント全体論理削除)── オーナー操作時は全メンバー削除可、プロジェクト管理者(オーナー以外)操作時は当該プロジェクトの member のみ削除可(他 admin は member 在籍中削除不可)。詳細は FR-019 / FR-019a / FR-019b を正本とする
- **契約ホーム閲覧 / OnboardingChecklist 操作**(SCR-026 全体、FR-326)── プロジェクト管理者 / メンバーは契約ワークスペース自体が WorkspaceSwitcher に出現しないため到達不可。URL 直叩き(`/owner/*`)時は 403 + 既定ワークスペースへリダイレクト

UI 上は非表示 + API は 403(`E-AUTHZ-OWNER-ONLY`)。

**SCR-031 メンバーアカウント有効化ページ**: 上記表外の特別画面。Cookie セッション認証を持たない未認証ユーザーが招待リンク経由でアクセスする。認可条件は `access_tokens.purpose='activation'` の有効トークン(期限内 + 未使用)を保持していることのみで、ロール列(オーナー / プロジェクト管理者 / メンバー)は適用されない。トークン認証 Principal(`user_id` + `access_tokens.meta.invitedProjectId` / `invitedRole`)を構築し、本画面以外でこの Principal は使用しない。詳細は 08_認証・認可設計.md §3.4 / §5.1 を参照。

**SCR-032 プロジェクト連絡先メール確認完了ページ**: 同様に未認証ユーザーがプロジェクト連絡先確認トークン経由でアクセスする画面。`access_tokens.purpose='contact_verify'` の有効トークン保持が唯一の認可条件で、連絡先メール所有者はアカウントを持つ必要がない(第三者の共有メールアドレスでも構わない)。トークン認証 Principal(`access_tokens.meta.projectId` のみ、`user_id` は持たない)を構築し、`projects.contact_email_verified_at` の単純な書き込みのみを行う。本画面以外でこの Principal は使用しない。詳細は 08_認証・認可設計.md §5.1 を参照。

| 機能ID | 機能名 | 必要ロール | オーナー | 制限事項 |

## 7. 契約別利用制限 / 運営者上書き

### 7.1 単一プランの利用上限([11_課金請求設計書.md §3](11_課金請求設計書.md) 参照)

## 9. 緊急時のみ可操作(§3.2.1-2 から移管)

### 9.1 「△ 緊急時のみ可」操作の発動条件

| 条件 | 実装方針(メイン側) |
|---|---|
| 1. サポートチケット ID 必須 | 連携 IF #4 / #5 / #6 / #12 のリクエスト JSON に `ticket_id` フィールドを必須化。メイン側で空文字検証 + 監査ログ記録 |
| 2. 運営者 2 名の承認(自己承認不可)| 顧客管理側で承認を取得し、結果を連携 IF 経由で伝搬。メイン側は **メタフィールド `approvals[]`(承認者 2 名分)** の検証(承認者 ID が異なる、承認時刻が 24 時間以内、kid 検証)を実施 |
| 3. 管理者ユーザーへの即時通知 | メイン側で `inbox_messages` レコードを生成(`category=system`、`importance=high`)+ メール送信を Queue 投入 |
| 4. 監査ログ必須記録 | `<resource>.<verb>.by_operator` 形式の `action` コードで `audit_logs` に記録。`retention_class='operator_high_priv'`(5 年保持)|

### 9.2 緊急区分 × 検知ハンドラ × 承認 SLA

| 緊急区分 | 検知ハンドラ | 承認 SLA | エスカレーション経路 |
|---|---|---|---|
| 重大障害 | `monitoring:main_outage` 監視 / 連携 IF #2 ヘルスチェック(5s × 3 連続失敗)| 15 分以内に 2 名承認 | RB-021(メイン障害時の運営者側縮退運転)と連動 |
| 全員ロックアウト | `audit_logs(auth.locked)` 集約検知(該当契約内全 admin が `auth.locked` 状態)| 30 分以内に CS マネージャーへ通報 | サポート窓口経由でオーナーと連絡、紙ベース回復コード(RB-014)も可 |
| セキュリティインシデント | NFR-602c 監査ログ完全性検証バッチ / SIEM / 不正アクセス検知(FR-178)| 即時 PagerDuty 起動 | セキュリティ責任者 → CTO → 運営本部 |
| 法令対応 | (運用側で個別判断)| 個別判断 | 法務 → CTO |

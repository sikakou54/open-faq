## 2. ロール一覧

利用者側は 3 つのロール(オーナー / プロジェクト管理者 / メンバー)で構成する。プロジェクト管理者とメンバーは「プロジェクト単位」のロールであり、同一メンバーがプロジェクト A では `admin`、プロジェクト B では `member`、プロジェクト C には未割当、といった組合せをとり得る。

| ロールID | 表示名 | 説明文(UI) | DB 値 |
|---|---|---|---|
| OWNER | 契約オーナー | 契約全体を管理し、プロジェクト作成・課金・退会などすべての設定を操作できます。 | `users` 行存在 AND `contract_owners.user_id = users.id` 行存在 |
| PROJECT_ADMIN | プロジェクト管理者 | 担当プロジェクトのFAQ、問い合わせ対応、メンバー招待・ロール変更を行えます。 | `users` 行存在 AND `contract_owners` 行を持たない AND `project_users.role='admin'`(プロジェクト単位)|
| MEMBER | プロジェクトメンバー | 担当プロジェクトのFAQや問い合わせを確認・対応できます。 | `users` 行存在 AND `contract_owners` 行を持たない AND `project_users.role='member'`(プロジェクト単位)|
| END_USER | ウィジェット利用者 | 公開ウィジェットからFAQ質問を送信し、AI回答・問い合わせID・連絡先案内を確認できます。 | 永続ロールなし。公開キー + ウィジェットセッション |
| SERVICE_OPERATOR | サービス運営者 | サービス提供者として、サポートや障害対応に必要な運用操作を行えます。 | **別 D1**(運営者システム `service_operators` テーブル)── メイン側 `users` には格納しない。メイン API への運営者アクセスは IF 経由の認可コンテキストで実現し、`audit_logs.actor_role='service_operator'` の値として記録する |

**用語の衝突回避**: 管理コンソールの認証主体は `users` テーブル(全行が管理者)で、ウィジェット利用者は永続アカウントを持たない。`project_users.role='admin'` はプロジェクト管理者ロールを示し、`users` テーブルのロール概念とは別軸。`audit_logs.actor_role` は操作者種別(`admin` / `system` / `service_operator`)を記録するもので、`users` のロール定義とは独立した概念。

## 3. ロール定義(`project_users.role` 2 値 + オーナー)

### 3.1 ロール概要

| ロール | 表示名(UI)| 範囲 | できること |
|---|---|---|---|
| オーナー | 契約オーナー | 契約全体 | すべてのプロジェクトと契約設定を管理できます。プロジェクト作成・課金・退会・規約同意・メンバー管理も行えます。 |
| `admin` | プロジェクト管理者 | 該当プロジェクト | 担当プロジェクトのFAQ公開、問い合わせ対応、ログ確認、データ出力、メンバー招待・ロール変更ができます。 |
| `member` | プロジェクトメンバー | 該当プロジェクト | 担当プロジェクトのFAQ編集、問い合わせ対応、ログ確認、データ出力ができます。 |

**オーナーの project_users 行**: オーナーはプロジェクト作成時に `project_users(role='admin', valid=1)` 行が自動 INSERT され、当該プロジェクトの明示的なプロジェクト管理者として行を保持する。認可上は Principal 構築時の `isOwner=true` 判定(`contract_owners` 行存在)で bypass が優先され、`project_users` 行は critical 通知の宛先解決と画面表示の事実情報として参照される(詳細は [08_認証・認可設計.md §6.1](08_認証・認可設計.md))。

### 3.2 ロール × 主管 SCR / FR 対応

| ロール | 主な操作対象 SCR / FR |
|---|---|
| オーナー | 全プロジェクト画面 + SCR-016 / SCR-004 / SCR-022〜027、FR-015a / FR-015c / FR-030 / FR-115〜127 / FR-189 / FR-193 / 規約 |
| `admin`(プロジェクト管理者)| SCR-005 / SCR-006 / SCR-008(該当プロジェクト範囲)/ SCR-009 / SCR-009-M1(操作者が `admin` のプロジェクト範囲)/ 選択中プロジェクト詳細 / FR-015 / FR-016 / FR-018a〜c / FR-040〜046 / FR-070〜077 / NFR-602 系列 |
| `member` | SCR-001(ダッシュボード)/ SCR-005 / SCR-006 / SCR-008(該当プロジェクト範囲)/ FR-040〜046 / FR-070〜077 |
| **招待トークン保有者(未認証)** | **SCR-018 メンバーアカウント有効化のみ**(`access_tokens.purpose='activation'` 有効時のみ到達可能)/ FR-016 / FR-016a / FR-016d / FR-016e。本画面以外には到達不可、有効化完了後は SCR-001 ログインへ遷移して通常 Principal が構築される |
| **連絡先確認トークン保有者(未認証)** | **SCR-019 プロジェクト連絡先メール確認完了のみ**(`access_tokens.purpose='contact_verify'` 有効時のみ到達可能)/ FR-033a。アカウント作成は不要、第三者の共有メールアドレスでも本フローを進められる。確認完了後は静的ランディングに留まる |

### 3.3 オーナー専有機能(プロジェクト別ロールでは付与不可)

- **プロジェクトの作成・編集・削除**(作成 = SCR-004-M1、編集・削除 = SCR-027)
- **料金・請求**(SCR-023)── プロジェクトの質問数上限・アラート変更はSCR-021-M1でプロジェクト管理者にも付与する
- **退会**(SCR-014)
- **規約再同意**(SCR-015 の同意 / 不同意)
- **契約メンバーのアカウント削除**(`DELETE /owner/members/{userId}`、SCR-024)── プロジェクト管理者はプロジェクト割当解除のみ可能
- **契約概要・契約利用状況・操作履歴・契約設定**(SCR-016 / SCR-022〜026)

UI 上は非表示 + API は 403(`E-AUTHZ-OWNER-ONLY`)。

**SCR-019 プロジェクト連絡先メール確認完了ページ**: 同様に未認証ユーザーがプロジェクト連絡先確認トークン経由でアクセスする画面。`access_tokens.purpose='contact_verify'` の有効トークン保持が唯一の認可条件で、連絡先メール所有者はアカウントを持つ必要がない(第三者の共有メールアドレスでも構わない)。トークン認証 Principal(`access_tokens.meta.projectId` のみ、`user_id` は持たない)を構築し、`projects.contact_email_verified_at` の単純な書き込みのみを行う。本画面以外でこの Principal は使用しない。詳細は 08_認証・認可設計.md §5.1 を参照。

## 5. 機能別権限

| 機能ID | 機能名 | 必要ロール | オーナー | 制限事項 |
|---|---|---|---|---|
| F-AUTH-* | 認証 | 不要 | ◯ | - |
| F-PRJ-* | プロジェクト管理(作成・編集・削除)| **オーナー専有** | ◯ | 削除は再認証必須 |
| F-FAQ-* | FAQ 管理 | `member`+(該当 PJ)| ◯ | 状態(`draft` / `published` / `hidden`)は SCR-006 で「状態」ラジオ選択 + 「保存」により自由遷移(状態遷移ガードなし)|
| F-INQ-* | 未解決質問 | `member`+(該当 PJ)| ◯ | 状況(`open` / `closed`)は SCR-005 詳細画面からの手動操作のみで切替。FAQ 下書き保存・FAQ 公開は連動しない(FR-077)|
| F-WIDGET-* | ウィジェット設定(プロジェクト単位)| `admin`+(該当 PJ)| ◯ | 公開キー再発行は L3(再認証必須)。`member` は閲覧 + 埋め込みコードのコピーのみ可。配置はプロジェクトワークスペース(SCR-007)|
| F-BILL-* | 課金(支払方法・請求・解約)| **オーナー専有** | ◯ | 支払方法変更 / 解約は再認証必須。質問数の月次上限件数・アラート変更は F-LIMIT-* へ分離 |
| F-LIMIT-* | プロジェクトの質問数上限・アラート | `admin`+(該当 PJ)/ オーナー(自動 admin)| ◯ | 変更は再認証必須(L3、SCR-021-M1)。`member` は閲覧のみ。`project_quota_limits.resource_kind='question_monthly_limit'` の上限とアラート閾値だけを扱う |
| F-USER-* | プロジェクトメンバー管理 | `admin`+(該当 PJ)/ オーナー | ◯ | 招待、ロール変更、当該PJ割当解除。アカウント削除は不可 |
| F-OWNER-MEMBER-* | 契約メンバー管理 | オーナー専有 | ◯ | 契約横断参照とアカウント論理削除 |
| F-ACC-* | 個人設定 | 全認証ユーザー | ◯ | 自分のプロフィール・セキュリティのみ |
| F-CONTRACT-* | 契約設定 / 退会 | オーナー専有 | ◯ | 退会は再認証必須 |
| F-NOTIF-* | お知らせ | 全 admin | ◯ | 既読化のみ |
| F-LEGAL-* | 利用規約 / プライバシーポリシー 閲覧・同意(文書ごとに個別)| **オーナー専有(同意)** / 全ユーザー(閲覧、認証不要)| ◯ | 閲覧は SCR-010(利用規約)/ SCR-020(プライバシーポリシー)|

「`member`+(該当 PJ)」は該当プロジェクトに対するロールが `member` または `admin` であることを示す(オーナーは bypass)。「`admin`+(該当 PJ)」は該当プロジェクトに対するロールが `admin` のみを示す。

## 7. 利用制限 / 運営者上書き(レート=契約単位 / 月次上限件数=プロジェクト単位)

### 7.1 利用上限の構成([10_課金・請求設計.md §3 / §3a](10_課金・請求設計.md) 参照)

MVP は単一プラン(完全従量課金 + 月次無料枠、無料トライアル期間なし)。プラン階層による差別化はない。利用上限は **2 系統** で実装される:

- **レート制限(契約単位)**: DDoS / Bot / 暴走対策のセキュリティ施策。`owner_quota_overrides.resource_kind`(レート種別)で実装。主管 SCR = SCR-093(運営者、4-eyes 承認)。
- **月次上限件数・無料枠(プロジェクト単位)**: 暴走コスト防止と課金境界の設定。`project_quota_limits`(§3.24)で実装。SCR-021 は質問数だけを主管し、FAQ無料枠は SCR-093 の運営者料金設定で管理する。

| `resource_kind`(`owner_quota_overrides`、契約単位)| 用途 | 主管 SCR |
|---|---|---|
| `widget_ask_per_min` | ウィジェット質問レート | SCR-093 |
| `widget_inquiry_per_min` | ウィジェット未解決質問送信レート | SCR-093 |
| `email_per_hour` | メール送信レート | SCR-093 |
| `admin_api_per_min` | 管理 API レート | SCR-093 |

| `resource_kind`(`project_quota_limits`、プロジェクト単位)| 用途 | 主管 SCR |
|---|---|---|
| `question_monthly_limit` | 質問数の月次上限件数・無料枠 | SCR-021(オーナー / PJ admin)/ SCR-093(運営者上書き)|
| `faq_monthly_limit` | FAQ 件数の無料枠(上限なし) | SCR-093 |

### 7.2 選択閾値アラート / 100%停止 / 125%運営者通知(プロジェクト単位に独立判定、[10_課金・請求設計.md §8](10_課金・請求設計.md) の参照)

## 9. 緊急時のみ可操作(§3.2.1-2 から移管)

### 9.1 「△ 緊急時のみ可」操作の発動条件

| 条件 | 実装方針(メイン側) |
|---|---|
| 1. サポートチケット ID 必須 | 連携 IF #4 / #5 / #6 / #12 のリクエスト JSON に `ticket_id` フィールドを必須化。メイン側で空文字検証 + 監査ログ記録 |
| 2. 運営者 2 名の承認(自己承認不可)| 顧客管理側で承認を取得し、結果を連携 IF 経由で伝搬。メイン側は **メタフィールド `approvals[]`(承認者 2 名分)** の検証(承認者 ID が異なる、承認時刻が 24 時間以内、kid 検証)を実施 |
| 3. 管理者ユーザーへの即時通知 | メイン側で `inbox_messages` レコードを生成(`category=system`、`importance=high`)+ メール送信を Queue 投入 |
| 4. 監査ログ必須記録 | `<resource>.<verb>.by_operator` 形式の `action` コードで `audit_logs` に記録。`retention_class='operator_high_priv'`(5 年保持)|

### 9.2 緊急区分 × 検知ハンドラ × 承認 SLA

テーブル設計

このページは、メインシステムのデータベース(Cloudflare D1 / SQLite)に存在する全テーブルのスキーマ(カラム定義 / 主キー / 外部キー / インデックス / 制約 / コード値)を定義します。

版数 v2.2更新 2026-06-17承認済

1. 文書概要

本書が対象とするテーブルの範囲と、各テーブル節(§3.x)で用いる統一フォーマットを示します。対象は利用者向け MVP データ(認証 / プロジェクト / FAQ / 未解決質問 / 通知 / 課金 / お知らせ / 監査 / 規約 等)で、運営者側専用テーブル(operator_* / ハッシュチェーン正本側 audit_logs / webhook_events / operator_approvals / accounts_retired 等)は 運営者システムのテーブル設計が正本です。contract_owners.contract_status やオーナーユーザーモデルなどの共有概念は 共有概念対応表を参照します。

1.5 フォーマット規約

各テーブル(§3.x)は概要 / カラム定義 / 主キー / 外部キー / インデックス / 制約 / コード値・区分値の 7 サブセクションで統一して記述します。DDL ブロックは持たず、本書の 7 サブセクション表が正本です。次の表は各サブセクションの記載内容を示します。

各テーブル(§3.x)は以下の 7 サブセクション構成。

サブセクション 内容
N.1 概要 テーブル名 / 論理名 / 説明 / 主管 FR / 保持期間 / contract_owner_user_id / retention_class
N.2 カラム定義 No / 論理名 / 物理名 / データ型 / 桁数 / NULL / PK / FK / UNIQUE / DEFAULT / 説明(11 列)
N.3 主キー 主キー名 / 対象カラム
N.4 外部キー 外部キー名 / 対象カラム / 参照先テーブル / 参照先カラム / 更新時 / 削除時
N.5 インデックス インデックス名 / 対象カラム / UNIQUE / 用途
N.6 制約 制約名 / 種別(CHECK / UNIQUE / 複合CHECK) / 対象カラム / 条件・内容
N.7 コード値・区分値 カラム名 / 値 / 意味
  • 桁数: TEXT / INTEGER は空欄(-)、長さ制約は §N.6 制約の CHECK に集約
  • PK / FK / UNIQUE 列: 単一は ○、複合は ① / ② の番号付き
  • NULL 列: YES / NO
  • 外部キー名: SQLite は名前なしのため (暗黙) 固定
  • 更新時 / 削除時: NO ACTION / CASCADE / SET NULL / RESTRICT
  • 部分インデックス: 対象カラム欄末尾に WHERE … 併記
  • コード値の値表記: バッククォート(SC-007 通過のキー)。状態遷移詳細は §4.5.x へリンク

2.4 ユーザー権限保持方式(3 表モデル)

採用方式は 3 表分離型(users + contract_owners + project_users):

  • users: 全ロール共通の認証主体(認証情報 + PII)。「オーナーかどうか」は users 単独では判定せず、contract_owners 行存在で判定する
  • contract_owners: オーナー専有属性(contract_status / data_deletion_mode)を分離。user_id PRIMARY KEY により 1 契約 = 1 オーナー固定が DB 制約で担保される
  • project_users: プロジェクト単位ロール(admin / member)。(user_id, project_id) 複合 PK + contract_owner_user_id 冗長保持(認可境界クエリ高速化と意図明示) + granted_at / granted_by 保持で行レベル監査

オーナーも project_users に行を持つ(プロジェクト作成時に project_users(user_id=オーナー, project_id=新PJ, contract_owner_user_id=オーナー, role='admin', valid=1) を自動 INSERT。FR-030a)。認可フロー先頭の isOwner=true bypass(contract_owners 行存在で判定)は維持し、project_users 行は critical 通知宛先解決と画面表示(プロジェクト管理者一覧)の事実情報として参照する。個別権限フラグ方式は採用しない。

最後の管理者保護判定(E-AUTHZ-LAST-ADMIN-PROTECTED): オーナー由来の project_users 行を判定対象から除外する。具体的には pu.user_id NOT IN (SELECT user_id FROM contract_owners WHERE valid=1) で「メンバー由来 admin 行」のみを抽出して件数判定する(オーナーは常に admin 行を持つため判定対象外)。

valid カラムによる論理削除を全削除対象テーブルに適用する(対象テーブルは §4.6 削除モード表を正本)。valid INTEGER NOT NULL DEFAULT 1 CHECK (valid IN (0,1)) を持つテーブルでは、論理削除は UPDATE … SET valid=0 で表現する。論理削除データは updated_at 基準で 90 日経過後に物理削除バッチで物理削除する(DD13_バッチ・非同期処理.html を正本)。deleted_at カラムは持たず、論理削除日時は既存 updated_at(valid が 0 に更新された時刻)で代用する。例外テーブル: audit_logs(API 操作ログ専用、created_at + retention 期間 で物理 DELETE)、sessions(revoked_at で代用)、法定保管 7 年テーブル(billing_invoices / usage_metering / withdrawal_requests / terms_versions / terms_agreements)、全契約横断テーブル(email_suppression_list)、ログ系(notification_logs / error_logs / inbox_messages)、トークン / 履歴系(access_tokens / faq_revisions / faq_search_fts / question_log_faq_refs)、M:N 関係系(announcement_audiences)。

4. 横断仕様

個々のテーブルを横断する共通仕様を定義します。ER 図・命名規則・型方針・SaaS データ分離・全契約横断テーブル・主要な状態遷移・データ削除モード・保持調整制約・PII 取扱・インデックス方針を順に示します。

4.1 ER 図

ER 図は本書とは別の HTML ファイル ER図.html で管理する(md に直接 Mermaid を埋め込むと PDF 出力時に縮小されすぎて可読性が落ちるため)。HTML 上でブラウザサイドレンダリング + ズーム / パン / 全体表示ボタンを利用可能。

  • 掲載カラム: 主要キー(PK / FK)+ 業務キー(inquiry_code / widget_public_key 等)+ 状態カラム(inquiries.status / contract_status 等)+ 特徴的なカラム(actor_user_id 等)
  • 省略カラム: created_at / updated_at / valid / *_encrypted / *_key_version / metadata 等の共通技術カラム(完全なカラム定義は §3 を参照)
  • FK 制約を持たないが論理的に紐づく参照(error_logs.contract_owner_user_id / error_logs.user_id / announcement_recipients.contract_owner_user_id / announcement_recipients.user_id)は ER 図上で「FK制約なし」注記付きカラムとして表現し、関係線は引かない
  • ER 図の編集ルール: 図を変更する場合は ER図.html の <pre id="mermaid-src"> ブロックを直接編集する(本書 md には Mermaid を二重保持しない)

4.3 SaaS データ分離

  • 全契約スコープのテーブルに contract_owner_user_id カラム必須(全契約横断テーブルを除く ── §4.4 参照)
  • 「契約境界(owner)」を表す FK 系列 → contract_owners(user_id) を参照(契約境界違反を物理的に弾く)。「操作者・主体(actor)」系 → users(id) を参照
  • API / 認可レイヤで境界検証: principal.contractOwnerUserId === target.contract_owner_user_id(08_認証・認可設計.html §5)
  • 違反時は 404 偽装(403 だと相手契約のリソース存在が漏れる)

4.3.1 オーナー / プロジェクト管理者 / メンバーの判定

  • オーナー判定: contract_owners.user_id = users.id 行が存在するユーザー(行存在 = オーナー)
  • プロジェクト管理者: 非オーナー かつ project_users.role='admin' AND valid=1 を 1 件以上保持するユーザー
  • メンバー: 非オーナー かつ project_users.role='member' AND valid=1 のみを保持するユーザー
  • 同一ユーザーがオーナー兼任 / 複数プロジェクト所属する場合の権限解決は、認可フロー先頭の isOwner bypass を最優先(08_認証・認可設計.html §5 を正本)
  • 1 契約 = 1 オーナー固定は contract_owners.user_id PRIMARY KEY で DB レベル担保(is_owner カラムや owner_account_id 自己参照は持たない)

4.4 全契約横断テーブル(contract_owner_user_id を持たない)

契約境界キー contract_owner_user_id を持たない、全契約横断で共有するテーブルを列挙します。次の表は各テーブルと横断保持とする理由を示します。

テーブル名 理由

service_announcements 運営者発信のお知らせ本体(配信先は announcement_audiences / announcement_recipients で契約別)

4.5 状態遷移

状態カラムを持つ主要テーブル(FAQ / 未解決質問 / 通知 / 契約)の状態遷移を定義します。inquiries.status と contract_owners.contract_status は共有概念の正本としてここで定義します。

4.5.1 FAQ 状態遷移(faqs.status)

From To トリガー 条件
- draft FAQ 作成 admin のみ
draft / hidden ↔︎ published (相互) SCR-006 で「状態」ラジオを選択して保存(PATCH /faqs/{id}) admin のみ。状態遷移ガードなし(draft/published/hidden を自由遷移)
draft ↔︎ hidden (相互) 同上 admin のみ
any deleted 論理削除 admin のみ
deleted draft 復元 運営者のみ(IF #4)

運用方針: draft / published / hidden の 3 状態は SCR-006 編集画面の「状態」ラジオで選択し「保存」ボタンで相互に自由遷移する。専用の公開 API・状態遷移ガードは持たない。published を選択して保存する操作が管理者(admin)による公開前の内容確認(FR-045)を兼ねる。published 設定時のみ保存前に確認ダイアログを表示する(UI 上の確認であり遷移制約ではない)。

4.5.5 契約状態遷移(contract_owners.contract_status、共有概念正本)

4.6 データ削除モード

設定: contract_owners.data_deletion_mode(physical / anonymize、初期値 physical)で保持する独立カラム。

論理削除 は UPDATE … SET valid=0 で行い、90 日経過後の物理削除バッチ(DD13)で物理削除モード(physical / anonymize)に従って実体を消去する。SCR-009-001 の「プロジェクトから外す」は対象 project_users 行だけを論理削除する。オーナーによるメンバーの契約からの削除は users と当該ユーザーの全 project_users を論理削除する。SCR-004-001(編集モード)のプロジェクト削除は projects と関連 project_users 行を論理削除する。

4.6.2 論理削除の物理削除バッチ(DD13 正本)

  • 日次バッチで valid=0 AND updated_at < now() - 90d の行を物理削除モード(contract_owners.data_deletion_mode 準拠)に従って処理
  • users 行が物理 DELETE された時点で ON DELETE CASCADE で関連 contract_owners / project_users / sessions / access_tokens / terms_agreements 等が連鎖物理削除
  • projects 行が物理 DELETE された時点で ON DELETE CASCADE で関連 faqs / inquiries 等が連鎖物理削除
  • billing_subscriptions は物理削除バッチの対象外(永久保持)

4.7 データ保持の調整制約(NFR-702)

  • データ保持期間はシステム固定値とし、契約・管理者ユーザーによる UI 上の変更は不可(短縮・長期化いずれも不可)
  • 固定値は本書 §2 のテーブル一覧「保持期間」列および要件定義 NFR-701 / NFR-705 を正本とする
  • 短縮が必要な場合(法令・契約義務範囲内)はサポート窓口経由の運営者対応とする
  • SCR-017 個人設定にデータ保持期間設定 UI を持たない(01_画面設計/index.html §5.24 参照)
  • usage_metering / billing_invoices / audit_logs(retention_class='billing') は電子帳簿保存法・国税法の要請により最低 7 年保持(NFR-602b)、これを下回る短縮は不可

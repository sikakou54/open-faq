テーブル構造設計

このページは、メインシステムの各テーブルを横断する設計意図と取り決め(テーブル間関係・3 表ユーザーモデル・契約境界キーによるデータ分離・状態遷移・削除モード・PII 取扱・インデックス方針)を、図解付きで集約します。 個々のテーブルのスキーマ定義は テーブル設計(一覧)と各テーブルファイル(users.html 等)が正本です。

  • 主要テーブルがどのような関係(カーディナリティ)で結ばれるか
  • ユーザー権限を保持する 3 表モデル(users / contract_owners / project_users)の役割分担
  • 契約境界キー contract_owner_user_id によるマルチテナント分離の仕組み
  • 命名規則・型方針・SaaS データ分離・全契約横断テーブルの扱い
  • 主要な状態遷移(FAQ / 未解決質問 / 通知 / 契約)とデータ削除モード・保持調整制約・PII 取扱・インデックス方針

2. ユーザー権限保持方式(3 表モデル)

権限(オーナー / プロジェクト管理者 / メンバー)を 3 テーブルに分離保持する方式です。役割分担を図解し、各表の責務を述べます。

2.1 3 表モデル図

users(認証主体)に対し、contract_owners 行の存在がオーナーを、project_users 行がプロジェクト割当とロール(admin / member)を表します。project_users.contract_owner_user_id はオーナー(契約境界)を冗長保持します。

2.2 各表の責務

採用方式は 3 表分離型(users + contract_owners + project_users)です。個別権限フラグ方式は採用しません。

  • users: 全ロール共通の認証主体(認証情報 + PII)。「オーナーかどうか」は users 単独では判定せず、contract_owners 行存在で判定する
  • contract_owners: オーナー専有属性(contract_status / data_deletion_mode)を分離。user_id PRIMARY KEY により 1 契約 = 1 オーナー固定が DB 制約で担保される
  • project_users: プロジェクト単位ロール(admin / member)。(user_id, project_id) 複合 PK + contract_owner_user_id 冗長保持(認可境界クエリ高速化と意図明示) + granted_at / granted_by 保持で行レベル監査

オーナーも project_users に行を持つ(プロジェクト作成時に project_users(user_id=オーナー, project_id=新PJ, contract_owner_user_id=オーナー, role='admin', valid=1) を自動 INSERT。FR-030a)。認可フロー先頭の isOwner=true bypass(contract_owners 行存在で判定)は維持し、project_users 行は critical 通知宛先解決と画面表示(プロジェクト管理者一覧)の事実情報として参照する。

最後の管理者保護判定(E-AUTHZ-LAST-ADMIN-PROTECTED): pu.user_id NOT IN (SELECT user_id FROM contract_owners WHERE valid=1) で「メンバー由来 admin 行」のみを抽出して件数判定する(オーナーは常に admin 行を持つため判定対象外)。

valid カラムによる論理削除を全削除対象テーブルに適用する(対象は §6 削除モード表が正本)。valid INTEGER NOT NULL DEFAULT 1 CHECK (valid IN (0,1)) を持つテーブルでは論理削除を UPDATE … SET valid=0 で表す。論理削除データは updated_at 基準で 90 日経過後に物理削除バッチで物理削除する(DD13_バッチ・非同期処理.html 正本)。deleted_at は持たず、論理削除日時は updated_at(valid が 0 に更新された時刻)で代用する。例外テーブル: audit_logs(API 操作ログ専用、created_at + retention 期間 で物理 DELETE)、sessions(revoked_at で代用)、法定保管 7 年テーブル(billing_invoices / usage_metering / withdrawal_requests / terms_versions / terms_agreements)、全契約横断テーブル(email_suppression_list)、ログ系(notification_logs / error_logs / inbox_messages)、トークン / 履歴系(access_tokens / faq_revisions / faq_search_fts / question_log_faq_refs)、M:N 関係系(announcement_audiences)。

3. 設計意図の補足

各テーブルの概要メタから省いたテーブル横断の設計意図(契約境界キーの持ち方・retention_class 分類・論理削除と物理削除バッチ)をここに集約します。

3.1 契約境界キー contract_owner_user_id

マルチテナント(SaaS)分離の境界キーが contract_owner_user_id です。多くのテーブルは本列を直接持ち、一部は親テーブル経由で契約所属を間接表現します。下表で直接保持 / 間接表現を整理します。

保持方式 対象テーブル(代表) 取り決め
直接保持(contract_owner_user_id カラムを持つ) projects / faqs / question_logs / inquiries / notification_logs / billing_subscriptions / billing_invoices / usage_metering / owner_quota_overrides / project_quota_limits / audit_logs / announcement_recipients / inbox_messages / withdrawal_requests 等 全契約スコープのテーブルに必須。境界 FK 系列は contract_owners(user_id) を参照し、契約境界違反を物理的に弾く
間接表現(親テーブル経由) users(contract_owners.user_id 行存在 / project_users.contract_owner_user_id 経由)、sessions / access_tokens / terms_agreements(users 経由)、project_legacy_keys / allowed_domains(projects 経由)、faq_revisions / faq_search_fts(faqs 連動)、question_log_faq_refs(question_logs 経由) users は contract_owner_user_id を持たず、contract_owners.user_id(行存在=オーナー判定)/ project_users.contract_owner_user_id で契約所属を表現する
契約境界そのもの contract_owners 本テーブル自身が契約境界(user_id = contract_owner_user_id)
全契約横断(境界キーを持たない) email_suppression_list / terms_versions / service_announcements 全契約共通で共有するため境界キーを持たない(§4.2 参照)

3.2 retention_class 分類の設計意図

監査ログの保持クラス(retention_class = general / billing)は対応する保持規制を表します。値の正本表は テーブル設計 §1.1 を参照してください。

クラス 保持規制(設計意図) 保持期間
general 一般業務監査(法定保管要件なし、運用上の証跡) 1 年(NFR-602a)
billing 電子帳簿保存法・国税法の課金/法定保管要請に対応 7 年(NFR-602b)

保持規制の詳細は 09_セキュリティ設計.html §7 を参照する。

3.3 論理削除(valid)と物理削除バッチ・匿名化モード

論理削除は valid=0 で表し、90 日経過後に物理削除バッチが data_deletion_mode(physical / anonymize)に従って実体を処理します。詳細表は §6 が正本です。

  • 論理削除: 全削除対象テーブル(§6 削除モード表が正本)で UPDATE … SET valid=0。論理削除日時は updated_at で代用(deleted_at は持たない)
  • 物理削除バッチ(90 日): 日次バッチが valid=0 AND updated_at < now() - 90d の行を contract_owners.data_deletion_mode に従って処理する(DD13 正本)
  • 匿名化モード: data_deletion_mode='anonymize' 選択時は、原文復元不可な置換(email_encrypted → anon_<sha256>、name → 「匿名」等)を行い集計値を維持する(§6 参照)

4. 命名規則・型方針 / SaaS データ分離

全テーブル共通の命名規則・型方針と、SaaS マルチテナントのデータ分離方式を示します。

4.1 命名規則・型方針

テーブル名・カラム名・データ型の全テーブル共通規則です。次の表に対象と規則を並べます。

対象 規則
テーブル snake_case 複数形
主キー 文字列 ID(ULID)/ *_versions 等の例外あり
外部キー {table_singular}_id
日時 各テーブルのデータ型欄は「日時」と表記し、実体は TEXT に ISO 8601 UTC(UTC 固定)で保持する(NFR-1103)
状態 status または {対象}_status
真偽値 INTEGER 0/1
JSON TEXT 列に JSON 文字列で保存
暗号化 {column}_encrypted(AES-256-GCM) + {column}_hmac(検索用)の 2 列構成

4.2 SaaS データ分離

契約境界キー contract_owner_user_id によるマルチテナント分離の概念と判定方式です。概念図で分離の仕組みを俯瞰します。

flowchart TD
  subgraph A["契約 A(オーナー A)"]
    AO["contract_owners(user_id=A)"]
    AP["projects / faqs / inquiries …
contract_owner_user_id = A"]
    AO --- AP
  end
  subgraph B["契約 B(オーナー B)"]
    BO["contract_owners(user_id=B)"]
    BP["projects / faqs / inquiries …
contract_owner_user_id = B"]
    BO --- BP
  end
  REQ["API リクエスト
principal.contractOwnerUserId"]
  REQ -->|"=== target.contract_owner_user_id"| AP
  REQ -.->|"境界違反は 404 偽装"| BP
  • 全契約スコープのテーブルに contract_owner_user_id カラム必須(全契約横断テーブルを除く ── §4.3 参照)
  • 「契約境界(owner)」を表す FK 系列 → contract_owners(user_id) を参照(契約境界違反を物理的に弾く)。「操作者・主体(actor)」系 → users(id) を参照
  • API / 認可レイヤで境界検証: principal.contractOwnerUserId === target.contract_owner_user_id(08_認証・認可設計.html §5)
  • 違反時は 404 偽装(403 だと相手契約のリソース存在が漏れる)

4.2.1 オーナー / プロジェクト管理者 / メンバーの判定

  • オーナー判定: contract_owners.user_id = users.id 行が存在するユーザー(行存在 = オーナー)
  • プロジェクト管理者: 非オーナー かつ project_users.role='admin' AND valid=1 を 1 件以上保持するユーザー
  • メンバー: 非オーナー かつ project_users.role='member' AND valid=1 のみを保持するユーザー
  • 同一ユーザーがオーナー兼任 / 複数プロジェクト所属する場合の権限解決は、認可フロー先頭の isOwner bypass を最優先(08_認証・認可設計.html §5 を正本)
  • 1 契約 = 1 オーナー固定は contract_owners.user_id PRIMARY KEY で DB レベル担保(is_owner カラムや owner_account_id 自己参照は持たない)

4.3 全契約横断テーブル(contract_owner_user_id を持たない)

契約境界キーを持たず全契約横断で共有するテーブルを、横断保持とする理由とともに列挙します。

テーブル名 理由
email_suppression_list メール bounce / 苦情の全契約横断管理(R-003 共通ドメイン対策)
terms_versions 規約 / プライバシーポリシーの版(全利用者共通)
service_announcements お知らせ本体(配信先は announcement_audiences / announcement_recipients で契約別)

6. データ削除モード

退会・削除時のデータ処理モード(物理削除 / 匿名化)と、対象テーブルごとの削除挙動を定義します。

設定: contract_owners.data_deletion_mode(physical / anonymize、初期値 physical)の独立カラムで保持する。

論理削除 は UPDATE … SET valid=0 で行い、90 日経過後の物理削除バッチ(DD13)が data_deletion_mode(physical / anonymize)に従って実体を消去する。論理削除対象は、メンバーのプロジェクト割当解除=対象 project_users 行、オーナーによるメンバー削除=users + 当該ユーザーの全 project_users、プロジェクト削除=projects + 関連 project_users 行。

notification_logs (valid なし、ログ系で論理削除概念外) DELETE recipient_email_hmac → SHA-256, vars → 空 JSON
announcement_audiences (valid なし、配信関係 M:N) service_announcements / contract_owners 物理削除時に CASCADE 連鎖 (匿名化対象外)
access_tokens (valid なし、used_at で代用) DELETE (常に物理削除、匿名化対象外)
sessions (valid なし、revoked_at で代用) DELETE (匿名化対象外)
audit_logs (valid なし、メイン側は API 操作ログ専用) created_at + retention 期間 経過後に物理 DELETE(general 1y / billing 7y) (匿名化対象外。ハッシュチェーンで完全性を検証)

6.2 論理削除の物理削除バッチ(DD13 正本)

  • 日次バッチで valid=0 AND updated_at < now() - 90d の行を物理削除モード(contract_owners.data_deletion_mode 準拠)に従って処理
  • users 行が物理 DELETE された時点で ON DELETE CASCADE で関連 contract_owners / project_users / sessions / access_tokens / terms_agreements 等が連鎖物理削除
  • projects 行が物理 DELETE された時点で ON DELETE CASCADE で関連 faqs / inquiries 等が連鎖物理削除
  • billing_subscriptions は物理削除バッチの対象外(永久保持)

7. データ保持の調整制約(NFR-702)

データ保持期間の変更可否に関する制約を示します。保持期間はシステム固定値であり、UI からの変更はできません。

  • データ保持期間はシステム固定値とし、契約・管理者ユーザーによる UI 上の変更は不可(短縮・長期化いずれも不可)
  • 固定値は要件定義 NFR-701 / NFR-705 を正本とする
  • 短縮が必要な場合(法令・契約義務範囲内)はサポート窓口経由の対応とする
  • データ保持期間は利用者が設定できない(固定値)
  • usage_metering / billing_invoices / audit_logs(retention_class='billing') は電子帳簿保存法・国税法の要請により最低 7 年保持(NFR-602b)、これを下回る短縮は不可

10. ER 図

完全な ER 図は別 HTML で管理します(Mermaid を直接埋め込むと PDF 出力時に縮小されすぎ可読性が落ちるため)。本ページ §1.1 の概観図は主要 FK の俯瞰用です。

  • ER 図は ER図.html で管理する。HTML 上でブラウザサイドレンダリング + ズーム / パン / 全体表示ボタンを利用可能
  • 掲載カラム: 主要キー(PK / FK)+ 業務キー(inquiry_code / widget_public_key 等)+ 状態カラム(inquiries.status / contract_status 等)+ 特徴的なカラム(actor_user_id 等)
  • 省略カラム: created_at / updated_at / valid / *_encrypted / *_key_version / metadata 等の共通技術カラム(完全なカラム定義は テーブル設計各テーブルファイルを参照)
  • FK 制約を持たないが論理的に紐づく参照(error_logs.contract_owner_user_id / error_logs.user_id / announcement_recipients.contract_owner_user_id / announcement_recipients.user_id)は ER 図上で「FK制約なし」注記付きカラムとして表現し、関係線は引かない
  • ER 図の編集ルール: 図を変更する場合は ER図.html の <pre id="mermaid-src"> ブロックを直接編集する(本書には Mermaid を二重保持しない)

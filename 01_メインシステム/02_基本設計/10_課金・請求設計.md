## 2. 契約モデル(MVP は単一プラン、共有概念正本)

### 2.1 単一プラン定義

MVP は **Free / Basic / Pro / Enterprise の階層プランを採用しない**。以下の単一モデルを全契約に適用する。

| 項目 | MVP 仕様 | 正本 |
|---|---|---|
| 課金モデル | **完全従量課金制 + 月次無料枠** | 要件 §8.11.1 / 基本設計書 §6.6 |
| トライアル | **14 日**(終了日翌日 00:00 JST に自動サスペンション)| §5.1 |
| 再課金リトライ | **Stripe Smart Retries 委譲、独自リトライ禁止** | §11.4 |
| 契約別カスタマイズ | `owner_quota_overrides`(`resource_kind` 統一実装)で契約単位に上書き(運営者操作)| §3 / [05_権限設計書.md §7](05_権限設計書.md) |
| 月次境界 | JST 暦月、無料枠は毎月リセット | §8.2 |
| 訂正請求 | Stripe Credit Note API、運営者の手動操作のみ(自動禁止)| §11.5 |

### 2.2 階層プランへの拡張余地

将来的にプラン階層を導入する場合(Future)、本書は `pricing_plans` テーブル + `accounts.plan_id` 追加で拡張可能な構造を残す。MVP では `pricing_plans` テーブルは作成しない。

## 4. 契約状態(`accounts.contract_status` 4 値 = 共有概念正本)

(基本設計書 §4.9 から移管)

| 状態 | 説明 | 利用可否 | システム動作 | 関連メッセージID |
|---|---|---|---|---|
| `active` | 有効 | 全機能 | 通常 | - |
| `suspended` | サスペンション中(支払い失敗 / トライアル終了)| オーナーのみ、課金画面 / 退会画面 / データエクスポート画面 | ウィジェット応答は機能停止旨を返す(**423**、FR-138)| MSG-BILL-CONTRACT-SUSPENDED |
| `deleted_pending` | 退会申請中(30 日猶予)| オーナーのみ、退会キャンセル / データエクスポート | データ参照可、新規操作不可 | MSG-BILL-DELETE-PENDING |
| `deleted` | 削除済み | × | アカウントスラグのみ `accounts_retired` に永久保持(運営者側)。`contract_status='deleted'` 設定時は必ず `accounts.valid=0` も同時セット(両者は同時遷移)| MSG-BILL-DELETED |

#### `contract_status` × `valid` の整合(03_テーブル設計.md §3.1 と同期)

| `is_owner` | `contract_status` | `valid` | 状態 |
|---|---|---|---|
| 1 | `active` / `suspended` | 1 | オーナー稼働中 / サスペンション中 |
| 1 | `deleted_pending` | 1 | オーナー退会申請中(30 日猶予) |
| 1 | `deleted` | 0 | オーナー退会完了(`valid=0` と必ず同時セット) |
| 0 | `active`(デフォルト) | 1 | メンバー有効 |
| 0 | `active`(デフォルト) | 0 | メンバー論理削除済み(`contract_status` は触らない、`valid=0` のみ) |

オーナー退会完了時の `valid=0` は退会バッチ([DD14_バッチ・非同期処理.md](../03_詳細設計/DD14_バッチ・非同期処理.md))で `contract_status='deleted'` 更新と同一トランザクション内でセットする。メンバー論理削除時は `contract_status` を変更しない(メンバー行では `contract_status` は意味を持たない)。

### 4.1 状態遷移詳細(付録 B.5 から移管)

| From | To | トリガー | reason | セッション処理 |
|---|---|---|---|---|
| - | `active` | 契約作成 | - | - |
| `active` | `suspended` | IF #1(支払い失敗 7 日猶予経過)| `payment_failure_grace_expired` | 継続(課金 / エクスポート / 退会のみ)|
| `active` | `suspended` | IF #1(運営者手動)| `operator_manual` / `tos_violation` | **5 秒以内全削除**(FR-022)|
| `active` | `suspended` | Cron(トライアル終了 + 支払方法未登録)| `trial_expired_no_payment_method` | 継続 |
| `suspended` | `active` | IF #1(支払再開 / 解除)| - | 再ログイン要求 |
| `active` | `deleted_pending` | admin 退会申請(SCR-024)| - | 継続(参照中心)|
| `deleted_pending` | `active` / `suspended`(元の状態)| admin 撤回(30 日以内)| - | 継続 |
| `deleted_pending` | `deleted` | 30 日経過 + 物理削除 Batch | - | 全セッション無効化 |
| `suspended` | `deleted_pending` | 運営者退会・削除移行 | - | - |

## 5. 契約ライフサイクル

### 5.1 トライアル・本契約フロー

| イベント | 変更前状態 | 変更後状態 | トリガー | 関連API |
|---|---|---|---|---|
| トライアル開始 | (新規)| `active`(`trial=true`)| SCR-002 新規登録完了 | `POST /accounts` |
| トライアル終了(支払方法未登録)| `active`(`trial=true`)| `suspended` | 終了日翌日 00:00 JST cron | (cron) |
| 本契約開始 | `active`(`trial=true`)| `active`(`trial=false`)| 支払方法登録 + 初回決済成功 | `POST /billing/payment-method` |

トライアル按分: 質問 1,000/月相当を 14 日按分 ≒ 466 件
通知: トライアル開始日、残り 3 日、終了日に inbox(billing / normal)+ email
起動: メイン Cron(JST 00:00 daily)

### 5.2 サスペンションフロー(§6.2.7 から移管)

起点イベント:
- 支払い失敗 連携 IF #1 受信 → `reason=payment_failure_grace_expired`(**7 日猶予タイマー開始**)
- 運営者手動停止 → `reason=operator_manual` / `tos_violation`(即時 suspended)
- トライアル終了 メイン Cron → `reason=trial_expired_no_payment_method`(即時 suspended、**猶予なし**)

認可ゲート(`accounts.contract_status=suspended` 中):
- **許可**: 課金画面 / データエクスポート画面 / 退会画面のみアクセス可
- **その他**: 403
- **ウィジェット応答**: 423(FR-138)

解除条件: 顧客管理側の連携 IF #1 受信(支払い成功 / 運営者解除)で `accounts.contract_status='active'`(オーナー行)へ

### 5.3 退会・解約フロー(§6.2.8 から移管)

```
admin が SCR-024 で退会申請
  → 再認証 FR-005
  → accounts.contract_status='deleted_pending'(オーナー行)
  → ウィジェット応答停止、新規 FAQ 登録不可
  → 当月末まで利用可
  → 月末締めで利用停止(月割りなし)
  → 30 日エクスポート猶予開始
  → 物理削除バッチ実行: accounts.contract_status='deleted'(オーナー行)
  → 全データ削除 + announcement_recipients 削除
```

| 観点 | 仕様 |
|---|---|
| 解約予告期間 | 当月末まで利用可、**月末締めで停止**(予告期間なし)|
| データエクスポート猶予 | 退会確定から **30 日**、SCR-024 退会申請画面 / SCR-015 課金部から JSON/CSV エクスポート可(旧 SCR-016 経由は廃止)|
| 物理削除 | 30 日経過後にバッチで物理削除(NFR-704)|
| クーリングオフ | B2B サービスのため特商法適用外。利用規約に明示 |
| 月途中の日割り | しない(FR-125)|
| 未請求残高返金 | 返金しない旨を利用規約に明記 |
| 解約 vs 退会 | 解約申請(サブスクリプション終了)と退会(契約削除)は別フロー |

## 6. 契約とオーナーの関係

| 項目 | 内容 |
|---|---|
| 契約単位 | アカウント(`accounts.is_owner=1` の行)= 1 契約 |
| オーナー / 組織との関係 | 契約 = 1 オーナーアカウント。組織概念は MVP 範囲外 |

## 8. 課金方式 / 課金単位

### 8.1 課金単位(要件 §8.11.1 / 基本設計書 §6.6.1 正本)

| 課金対象 | 計測単位 | 計測タイミング | 失敗時の扱い | MVP 無料枠初期値 | MVP 超過課金単価初期値 |
|---|---|---|---|---|---|
| 質問数 | 1 質問 = エンドユーザーが「送信」操作した 1 リクエスト | リクエスト到達時(同期)| AI 推論失敗時もカウントするが `metering_billable=false` | 1,000 件 / 月 | 0.5 円 / 件 |
| FAQ 件数 | 公開状態の FAQ 件数(下書き・非公開・削除除外)| 月末 23:59:59 JST スナップショット | n/a | 100 件 | 5 円 / 件 / 月 |
| 個別チャット部屋数 | 1 部屋 = 新規 `chat_room` レコード作成 1 件 | 部屋作成成功時 | 再オープン(自動クローズ後 30 日以内の同一 room_id 復活)はカウントしない | 30 部屋 / 月 | 30 円 / 部屋 |
| AI 利用コスト(原価)| 推論呼び出しトークン換算(社内可視化用)| 呼び出し成功時のみ | 失敗は計測のみ・課金対象外。再試行は最初の 1 回のみ計測 | サービス側吸収(MVP)| n/a(質問数単価に内包)|

### 8.2 80/100/125% 三段階アクション(FR-122)

| データ種別 | 80% 到達 | 100% 到達 | 125% 到達 |
|---|---|---|---|
| 質問数 | inbox 通知(billing / normal)| inbox + email(billing / high)。**拒否はしない**(エンドユーザー体験を優先、事後課金)| 同左 + 翌 1 時間レート制限を 1/2 + 運営者へエスカレーション |
| FAQ 件数 | inbox 通知 | 新規 FAQ 登録を **拒否**(既存編集は可)| 同左 + 運営者へエスカレーション |
| 個別チャット部屋数 | inbox 通知 | 新規部屋作成を **拒否**(既存返信は可)| 同左 + 運営者へエスカレーション |

### 11.4 Stripe Smart Retries 委譲(基本設計書 §6.6.4)

| 観点 | 仕様 |
|---|---|
| 再課金リトライ | Stripe Smart Retries に委譲、**独自リトライ実装は行わない** |
| 失敗確定 | 最大 3 回失敗後にメイン側へ「決済失敗確定」イベント送信(連携 IF #1)|
| 猶予期間 | 7 日間(顧客管理側からのイベント受信時刻起点)|
| 猶予中の操作 | 管理者ユーザーは管理画面の課金画面・データエクスポート画面・退会画面のみアクセス可、それ以外は 403 |
| 解除条件 | 管理者ユーザーが課金画面から手動で支払方法更新 → 即時再決済 → 成功なら顧客管理側が解除イベント発火 → 連携 IF #1 で `accounts.contract_status='active'`(オーナー行) |
| 訂正請求 | Stripe Credit Note API 経由、**運営者の手動操作のみ**(自動禁止)|

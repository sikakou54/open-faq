## 2. 契約モデル(MVP は単一プラン、共有概念正本)

### 2.1 単一プラン定義

MVP は **Free / Basic / Pro / Enterprise の階層プランを採用しない**。以下の単一モデルを全契約に適用する。

| 項目 | MVP 仕様 | 正本 |
|---|---|---|
| 課金モデル | **完全従量課金制 + 月次無料枠** | 要件 §8.11.1 / 基本設計書 §6.6 |
| 契約開始 | 新規登録完了時点で `active`(**無料トライアル期間なし**)。支払方法登録は任意、無料枠超過時にウィジェット質問受付を停止して必須化(支払方法ゲート、FR-136。契約は `active` 維持)| §5.1 |
| 再課金リトライ | **Stripe Smart Retries 委譲、独自リトライ禁止** | §11.4 |
| 利用設定のカスタマイズ | レート制限 = `owner_quota_overrides` / 質問数上限・課金対象別無料枠 = `project_quota_limits`。管理者ユーザーが変更できるのは質問数設定のみ | §3 / §3a |
| 月次境界 | JST 暦月、無料枠は毎月リセット | §8.2 |
| 訂正請求 | Stripe Credit Note API、運営者の手動操作のみ(自動禁止)| §11.5 |

### 2.2 階層プランへの拡張余地

## 3. レート制限上書き(契約単位)

運営者が契約単位で `owner_quota_overrides` テーブルを操作する(SCR-093、4-eyes 承認)。DDoS / Bot / 暴走対策のセキュリティ施策であり、レート種別(`*_per_*`)のみを `resource_kind` で扱う(月次上限件数は §3a の `project_quota_limits` へ分離)。

| `resource_kind` | 用途 | 主管 SCR(運営者)| KV ミラー |
|---|---|---|---|
| `widget_ask_per_min` | ウィジェット質問レート | SCR-093 | あり(60s TTL)|
| `widget_inquiry_per_min` | ウィジェット未解決質問送信レート | SCR-093 | あり |
| `email_per_hour` | メール送信レート | SCR-093 | あり |
| `admin_api_per_min` | 管理 API レート | SCR-093 | あり |

詳細は [03_テーブル設計.md §3.28](03_テーブル設計.md) 参照。

## 3a. 質問数上限・課金対象別無料枠(プロジェクト単位)

質問数の月次上限件数・上限アラートと、MVP課金対象の無料枠を **プロジェクト単位** で `project_quota_limits` に保持する。質問数は `threshold` に値がある場合を上限ON、NULLを上限OFFとして扱う。管理者ユーザー向け SCR-036 / SCR-036-M1 が表示・変更するのは質問数の上限とアラートだけとし、無料利用枠は表示・変更しない。無料枠は運営者向け SCR-093 で管理し、FAQ 件数の `threshold` は NULL 固定で月次上限なしとする。

| `resource_kind` | 用途 | 主管 SCR | KV ミラー |
|---|---|---|---|
| `question_monthly_limit` | 質問数の月次上限件数・無料枠 | SCR-036(オーナー / PJ admin)/ SCR-093(運営者プロジェクト上書き)| あり(30s TTL)|
| `faq_monthly_limit` | FAQ 件数の無料枠(上限なし) | SCR-093 | あり |

質問数の優先順位は「プロジェクト設定(`source='owner'`)> 運営者上書き(`source='operator'`)> デフォルト推奨値」。FAQ無料枠の優先順位は「運営者上書き(`source='operator'`)> デフォルト値」とし、`source='owner'` 行は作成しない。デフォルト値は KV `usage-limit:default:{kind}` / `usage-limit:free-default:{kind}` で解決する。詳細は [03_テーブル設計.md §3.36](03_テーブル設計.md) 参照。

## 4. 契約状態(`contract_owners.contract_status` 4 値 = 共有概念正本)

(基本設計書 §4.9 から移管)

| 状態 | 説明 | 利用可否 | システム動作 | 関連メッセージID |
|---|---|---|---|---|
| `active` | 有効 | 全機能 | 通常 | - |
| `suspended` | サスペンション中(支払い失敗 / 運営者手動停止)| オーナーのみ、課金画面 / 退会画面 / データエクスポート画面 | ウィジェット応答は機能停止旨を返す(**423**、FR-138)| MSG-BILL-CONTRACT-SUSPENDED |
| `deleted_pending` | 退会申請中(30 日猶予)| オーナーのみ、退会キャンセル / データエクスポート | データ参照可、新規操作不可 | MSG-BILL-DELETE-PENDING |
| `deleted` | 削除済み | × | アカウントスラグのみ `accounts_retired` に永久保持(運営者側)。`contract_status='deleted'` 設定時は必ず `contract_owners.valid=0` も同時セット(両者は同時遷移)| MSG-BILL-DELETED |

#### `contract_status` × `valid` の整合(03_テーブル設計.md §3.1 と同期)

| ユーザー種別 | `contract_owners.contract_status` | `valid`(行種別)| 状態 |
|---|---|---|---|
| オーナー(`contract_owners` 行存在)| `active` / `suspended` | `contract_owners.valid=1` + `users.valid=1` | オーナー稼働中 / サスペンション中 |
| オーナー(`contract_owners` 行存在)| `deleted_pending` | `contract_owners.valid=1` + `users.valid=1` | オーナー退会申請中(30 日猶予) |
| オーナー(`contract_owners` 行存在)| `deleted` | `contract_owners.valid=0` + `users.valid=0` | オーナー退会完了(`valid=0` と必ず同時セット) |
| メンバー(`contract_owners` 行なし)| (該当なし、`contract_owners` 行を持たない)| `users.valid=1` | メンバー有効 |
| メンバー(`contract_owners` 行なし)| (該当なし、`contract_owners` 行を持たない)| `users.valid=0` | メンバー論理削除済み(`users.valid=0` のみ) |

オーナー退会完了時の `valid=0` は退会バッチ([DD14_バッチ・非同期処理.md](../03_詳細設計/DD14_バッチ・非同期処理.md))で `contract_status='deleted'` 更新と同一トランザクション内で `contract_owners.valid=0` + `users.valid=0` を同時セットする。メンバー論理削除時は `contract_owners` 行を持たないため `contract_status` は無関係で、`users.valid=0` のみ更新する。

### 4.1 状態遷移詳細(付録 B.5 から移管)

| From | To | トリガー | reason | セッション処理 |
|---|---|---|---|---|
| - | `active` | 契約作成 | - | - |
| `active` | `suspended` | IF #1(支払い失敗 7 日猶予経過)| `payment_failure_grace_expired` | 継続(課金 / エクスポート / 退会のみ)|
| `active` | `suspended` | IF #1(運営者手動)| `operator_manual` / `tos_violation` | **5 秒以内全削除**(FR-022)|
| `suspended` | `active` | IF #1(支払再開 / 解除)| - | 再ログイン要求 |
| `active` | `deleted_pending` | admin 退会申請(SCR-024)| - | 継続(参照中心)|
| `deleted_pending` | `active` / `suspended`(元の状態)| admin 撤回(30 日以内)| - | 継続 |
| `deleted_pending` | `deleted` | 30 日経過 + 物理削除 Batch | - | 全セッション無効化 |
| `suspended` | `deleted_pending` | 運営者退会・削除移行 | - | - |

## 5. 契約ライフサイクル

### 5.1 契約開始フロー(無料トライアル期間なし)

| イベント | 変更前状態 | 変更後状態 | トリガー | 関連API |
|---|---|---|---|---|
| 契約開始 | (新規)| `active` | SCR-002 新規登録完了 | `POST /accounts` |
| 支払方法登録(任意)| `active` | `active`(支払方法登録済)| 支払方法登録 + Setup Intent 成功 | `POST /billing/payment-method` |

- **無料トライアル期間は設けない**。新規登録完了時点で契約は即 `active` となり、月次無料枠の範囲を無料で利用できる。
- **支払方法の登録は任意**。未登録でも無料枠内の利用およびダッシュボード閲覧は可能。
- **支払方法ゲート(FR-136)**: 無料利用枠80%到達時に支払方法登録が必要になる旨を SCR-026 / SCR-037 のバナー + 受信箱 / メールで事前告知する。
- **支払方法未登録で質問数無料枠を超過した場合、契約状態は `active` のまま、ウィジェットの新規質問受付のみを停止**する。FAQ 件数は無料枠超過や月次件数を理由に新規作成を拒否しない。
- **支払方法登録済の契約**は、全課金対象の無料枠超過分を事後課金とする。
- ウィジェット質問受付停止は支払方法登録(Setup Intent 成功)で即時再開する。以降の無料枠超過分は事後課金となる。

### 5.2 サスペンションフロー(§6.2.7 から移管)

起点イベント:
- 支払い失敗 連携 IF #1 受信 → `reason=payment_failure_grace_expired`(**7 日猶予タイマー開始**)
- 運営者手動停止 → `reason=operator_manual` / `tos_violation`(即時 suspended)

> **支払方法ゲート(FR-136)は契約サスペンションではない**。支払方法未登録の契約が無料枠を超過した場合は契約 `active` のままウィジェット質問受付のみを停止する(§8.2 / DD13 §3.7a)。本フローの対象外。

認可ゲート(`contract_owners.contract_status=suspended` 中):
- **許可**: 課金画面 / データエクスポート画面 / 退会画面のみアクセス可
- **その他**: 403
- **ウィジェット応答**: 423(FR-138)

解除条件:
- `payment_failure_grace_expired` / `operator_manual` / `tos_violation`: 顧客管理側の連携 IF #1 受信(支払い成功 / 運営者解除)で `contract_owners.contract_status='active'` へ

### 5.3 退会・解約フロー(§6.2.8 から移管)

```
admin が SCR-024 で退会申請
  → 再認証 FR-005
  → contract_owners.contract_status='deleted_pending'
  → ウィジェット応答停止、新規 FAQ 登録不可
  → 当月末まで利用可
  → 月末締めで利用停止(月割りなし)
  → 30 日エクスポート猶予開始
  → 物理削除バッチ実行: contract_owners.contract_status='deleted' + contract_owners.valid=0 + users.valid=0(同時セット)
  → 全データ削除 + announcement_recipients 削除
```

| 観点 | 仕様 |
|---|---|
| 解約予告期間 | 当月末まで利用可、**月末締めで停止**(予告期間なし)|
| データエクスポート猶予 | 退会確定から **30 日**、SCR-024 退会申請画面 / SCR-041 契約設定から CSV エクスポート可 |
| 物理削除 | 30 日経過後にバッチで物理削除(NFR-704)|
| クーリングオフ | B2B サービスのため特商法適用外。利用規約に明示 |
| 月途中の日割り | しない(FR-125)|
| 未請求残高返金 | 返金しない旨を利用規約に明記 |
| 解約 vs 退会 | 解約申請(サブスクリプション終了)と退会(契約削除)は別フロー |

## 6. 契約とオーナーの関係

| 項目 | 内容 |
|---|---|
| 契約単位 | `contract_owners` 行 = 1 契約(`user_id` PRIMARY KEY で 1 契約 = 1 オーナー固定)|
| オーナー / 組織との関係 | 契約 = 1 オーナーアカウント。組織概念は MVP 範囲外 |
| 複数契約可否 | 同一メールで複数契約は不可(`users.email_hmac` UNIQUE)|
| 契約管理者 | オーナーアカウント(MVP では譲渡不可、FR-333)|
| 契約停止時の利用制限 | 課金画面 / 退会画面 / データエクスポートのみ |

## 7. プラン変更時の仕様(MVP は上限上書きのみ)

MVP は単一プランのため、プラン階層変更は存在しない。**レート上書き(契約単位)/ 月次上限件数・無料枠(プロジェクト単位)の追加・変更・削除** のみが行われる。

| 変更種別 | 反映タイミング | 課金処理 | 利用制限の扱い | 備考 |
|---|---|---|---|---|
| レート上書き追加・削除(運営者操作)| 即時(KV TTL 60s 経過後)| - | 即時適用 / 削除で標準値復帰 | SCR-093 / 4-eyes 承認ログ(契約単位)|
| プロジェクトの質問数上限・アラート変更(オーナー / PJ admin)| 即時(KV TTL 30s 経過後)| — | 即時適用 | SCR-036-M1(プロジェクト単位)|
| プロジェクト上限の運営者上書き追加・削除 | 即時(KV TTL 30s)| 翌月から | 即時適用 / 削除でオーナー設定 or デフォルトへ復帰 | SCR-093 / 4-eyes 承認ログ(プロジェクト単位)|

## 8. 課金方式 / 課金単位

### 8.1 課金単位(要件 §8.11.1 / 基本設計書 §6.6.1 正本)

| 課金対象 | 計測単位 | 計測タイミング | 失敗時の扱い | MVP 無料枠初期値 | MVP 超過課金単価初期値 |
|---|---|---|---|---|---|
| 質問数 | 1 質問 = エンドユーザーが「送信」操作した 1 リクエスト | リクエスト到達時(同期)| AI 推論失敗時もカウントするが `metering_billable=false` | 1,000 件 / 月(プロジェクト単位デフォルト)| 0.5 円 / 件 |
| FAQ 件数 | 公開状態の FAQ 件数(下書き・非公開・削除除外)| 月末 23:59:59 JST スナップショット | n/a | 100 件(プロジェクト単位デフォルト)| 5 円 / 件 / 月 |
| AI 利用コスト(原価)| 推論呼び出しトークン換算(社内可視化用)| 呼び出し成功時のみ | 失敗は計測のみ・課金対象外。再試行は最初の 1 回のみ計測 | サービス側吸収(MVP、契約単位指標)| n/a(質問数単価に内包)|

> 無料枠初期値は**プロジェクトごとのデフォルト値**とする。全課金対象の無料枠は運営者が SCR-093 で管理し、SCR-036-M1 では表示・更新しない。上限判定は質問数のみ、無料枠判定はMVP課金対象でプロジェクト単位とする。

### 8.2 選択閾値アラート / 100%停止 / 125%運営者通知(FR-122, FR-127)

**判定基準はプロジェクトごとの質問数月次上限件数**。選択可能なアラート閾値25/50/80/90/100%、100%停止、125%運営者通知は質問数だけに適用し、FAQ 件数は対象外とする。

> **無料枠と上限件数の区別**: 無料枠はMVP課金対象の課金境界。月次上限件数は質問数だけの利用停止境界。FAQは無料枠を超えても利用を継続し、超過分を従量課金する。

下表は質問数のアクション。質問数100%停止はアラート選択と支払方法の有無に関わらず適用される。

| データ種別 | 選択閾値到達(25/50/80/90/100%)| 100% 到達(上限件数比)| 125% 到達(上限件数比)|
|---|---|---|---|
| 質問数 | 選択した各閾値への当月初回到達時に、オーナー + 当該 PJ 管理者へメール送信。複数閾値選択可、同一メールアドレスは重複排除 | **当該プロジェクトのウィジェット新規質問受付を停止**(429 + `Retry-After`=当月末。エンドユーザーへ連絡先メール誘導、MSG-WIDGET-QUESTION-LIMIT)。100%アラート選択時だけ管理者へ閾値メールを送信。契約は `active` のまま。復帰=翌月リセット or SCR-036-M1 で上限引き上げ | 同左(既に停止中)+ 運営者へエスカレーション |

### 8.3 月次境界

JST 暦月。無料枠は毎月 1 日 00:00 JST にリセット。

## 9. 請求対象

| 対象 | 課金条件 | 金額算出方法 | 備考 |
|---|---|---|---|
| 月次無料枠超過分(質問数)| 100% 超過 + **支払方法登録済**| (超過件数 × 単価)| **拒否しない**、事後課金 |
| 月次無料枠超過分(FAQ 件数)| 100% 超過 | (超過件数 × 単価)| **拒否しない**。上限なしで継続利用 |
| 月次無料枠内 | 課金しない | 0 円 | 支払方法未登録でも利用可 |
| 無料枠超過 + **支払方法未登録** | 課金しない | 0 円 | 契約 `active` 維持、ウィジェット質問受付停止(429、支払方法ゲート FR-136)。支払方法登録で再開 |
| サスペンション中 | 課金しない(月次集計対象外)| 0 円 | - |

### 11.4 Stripe Smart Retries 委譲(基本設計書 §6.6.4)

| 観点 | 仕様 |
|---|---|
| 再課金リトライ | Stripe Smart Retries に委譲、**独自リトライ実装は行わない** |
| 失敗確定 | 最大 3 回失敗後にメイン側へ「決済失敗確定」イベント送信(連携 IF #1)|
| 猶予期間 | 7 日間(顧客管理側からのイベント受信時刻起点)|
| 猶予中の操作 | 管理者ユーザーは管理画面の課金画面・データエクスポート画面・退会画面のみアクセス可、それ以外は 403 |
| 解除条件 | 管理者ユーザーが課金画面から手動で支払方法更新 → 即時再決済 → 成功なら顧客管理側が解除イベント発火 → 連携 IF #1 で `contract_owners.contract_status='active'` |
| 訂正請求 | Stripe Credit Note API 経由、**運営者の手動操作のみ**(自動禁止)|

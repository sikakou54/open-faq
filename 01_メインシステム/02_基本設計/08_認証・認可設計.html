1. 文書概要

1.1 目的

メインシステムの認証(ログイン / ログアウト / パスワード再設定 / 招待受諾 / 規約再同意 / メール確認)、セッション・トークン、認可モデル(オーナー境界判定 / Principal 構築)、認可判定ルール、アカウント状態、認証・認可エラーを一元化する。

1.2 対象範囲

  • 対象: 利用者(オーナー / プロジェクト管理者 / メンバー)の認証・認可全般、公開ウィジェットのセッション検証

3.1 ログイン(AUTH-FLOW-001)

  1. メール + パスワード入力 → Turnstile 検証(FR-177 発動時)
  2. Argon2id 検証 → sessions レコード作成 + Cookie 発行
  3. 規約再同意状態確認 → 未同意なら SCR-015 へリダイレクト
  4. 契約状態確認(contract_owners.contract_status)→ suspended で許可機能外なら 403、deleted なら 403
  5. 既定の管理範囲に応じて SCR-016 利用状況または SCR-008 概要へ

エラー: 401 INVALID_CREDENTIALS(E-AUTH-CREDENTIAL)、423 LOCKED_OUT(E-AUTH-LOCKED)、403 CONTRACT_SUSPENDED

3.2 ログアウト

セッション失効 + Cookie 削除。SCR-001 へ。

3.3 パスワード再設定(AUTH-FLOW-003)

  1. SCR-003 段階 1: メール入力 → POST /auth/password-reset-request(列挙攻撃対策で存在有無を返さない)
  2. メール送信(access_tokens.purpose='password_reset'、HMAC-SHA256、有効期限 1 時間、1 回限り、FR-006)
  3. SCR-003 段階 2: トークン検証 + 新パスワード入力 → POST /auth/password/reset
  4. 全セッション失効 → SCR-001 へ

3.4 招待受諾(AUTH-FLOW-004)

  1. プロジェクト WS の SCR-009-M1 から POST /projects/{id}/members(リクエスト: { email, role }、氏名 displayName は受け付けない)→ 招待メール送信(access_tokens.purpose='activation'、有効期限 7 日、meta={"invitedProjectId": "...", "invitedRole": "..."} を JSON 保持、FR-016 / FR-016d / FR-016e)。同時に users 行は name=NULL / status='pending_activation' で作成、project_users 行は valid=0 で予約作成
  2. メール内リンク → SCR-018 メンバーアカウント有効化ページ → POST /auth/invitations/{token}/preview(招待情報パネル表示用、トークン検証 + プレビュー情報取得)
  3. SCR-018 で氏名(表示名) + 初回パスワード + 利用規約同意 + プライバシーポリシー同意 + Turnstile を入力 → POST /auth/invitations/{token}/activate
  4. 同一トランザクションで以下を実行: UPDATE users SET name=?, password_hash=?, status='active' + INSERT INTO terms_agreements (...)(利用規約・プライバシーポリシーの各最新版に対し doc_type 別の 2 行)+ UPDATE project_users SET valid=1 WHERE user_id=? AND project_id=? + consumeToken('activation') + writeAudit('user.activation_completed', ...)
  5. SCR-001 ログイン画面へ遷移

他プロジェクトに参加させる場合は、対象プロジェクトの WS で個別に招待操作を行い、FR-021c 名前解決により既存ユーザー(users.name 設定済 / status='active')に当該プロジェクト分の project_users 行(valid=1)が直接追加される(SCR-018 はバイパス)。氏名は本人のみが入力可能(FR-016d、個人情報原則)。

3.5 規約再同意(AUTH-FLOW-005)

  1. ログイン直後に利用規約 / プライバシーポリシーのいずれかに未同意(最新版)があれば SCR-015 強制割込み(改定された文書のみを対象に判定)
  2. 利用規約同意 → POST /terms/agree、プライバシーポリシー同意 → POST /privacy/agree(双方の最新版同意が必須。改定された文書のみ再同意)
  3. terms_agreements レコード作成(doc_type 別に各 1 行)
  4. 同意期限超過後の機能制限ガード: 課金画面操作 + 新規プロジェクト作成は不可、FAQ 編集とウィジェット稼働は継続

3.6 メール確認(AUTH-FLOW-006)

  1. SCR-002 新規登録 → 確認メール送信(access_tokens.purpose='email_verify'、有効期限 24 時間)
  2. メール内リンク → POST /auth/email-verifications/{token}
  3. users.email_verified_at 設定 → SCR-001 へ

3.8 強制ログアウト(AUTH-FLOW-008)

連携 IF #2 受信時、operator_manual / tos_violation の場合は 5 秒以内 に全セッション無効化(FR-022)。payment_failure_grace_expired 等は既存セッション継続。

5. 認可モデル

5.1 認可方式

操作者種別(管理者 = users 行存在 / 公開ウィジェット = セッショントークン / 運営者 = 別 D1)+ プロジェクト別ロール project_users.role = admin / member + オーナー判定 = contract_owners 行存在

SCR-018 メンバーアカウント有効化ページの例外: 未認証ユーザーが招待トークン経由でアクセスする画面のため、Cookie セッションを持たない。access_tokens.purpose='activation' の有効トークン(期限内 + 未使用)を保持していることが唯一の認可条件で、トークン認証 Principal(user_id を主体とし、access_tokens.meta.invitedProjectId / invitedRole を併せて取得)を構築する。本画面以外でこの Principal は使用しない(有効化完了 → SCR-001 ログイン → 通常 Principal 構築の流れ)。

SCR-019 プロジェクト連絡先メール確認完了ページの例外: 同様に未認証ユーザーがプロジェクト連絡先確認トークン経由でアクセスする画面で、access_tokens.purpose='contact_verify' の有効トークン保持が唯一の認可条件。トークン認証 Principal(access_tokens.meta.projectId のみを取得、user_id は持たない ── 連絡先メール所有者がアカウントを持つとは限らない)を構築し、当該プロジェクトの projects.contact_email_verified_at を更新する単純な書き込みのみを行う。完了後は静的ランディングに留まり、認証コンソールへの遷移は行わない。

5.2 オーナー境界判定式(共有概念正本)

すべての操作で以下の判定式を必須:

principal.contractOwnerUserId === target.contract_owner_user_id
  • principal = 操作するユーザー(セッションから取得した Principal)
  • target = 操作対象リソース(テーブルの contract_owner_user_id カラム)
  • 一致しない場合は 404 偽装(403 を返すと相手契約の存在が漏れる、エラー設計書 06 §5.3)

5.3 ユーザー種別の DB 値(§3.1 から移管)

ユーザー種別 DB 値 概要
オーナー users 行存在 AND contract_owners.user_id = users.id 行が存在 契約あたり 1 オーナー固定(contract_owners.user_id PRIMARY KEY で DB レベル担保)。全権 + オーナー専有機能
プロジェクト管理者 / メンバー users 行存在 AND contract_owners 行を持たない AND project_users 行を保持 プロジェクト別ロール(project_users.role = admin/member)で操作範囲が決まる
ウィジェット利用者 DB主体行なし 公開キーとウィジェットセッションでFAQ質問送信のみ。管理画面Principalは構築しない
サービス運営者 別 D1(運営者システム service_operators テーブル) 本 SaaS 提供者(運営者側、メイン側 D1 には格納しない)

管理者は users 行を主体とする。MVPのウィジェット利用者は管理画面Principalや永続的な利用者アカウントを持たず、公開キーと短期ウィジェットセッションでAPIを制限する。 オーナー判定: contract_owners.user_id = users.id 行存在(is_owner カラムや owner_account_id 自己参照は持たない) contract_owners.contract_status 値域: active / suspended / deleted_pending / deleted の 4 値固定

5.4 オーナー専有機能

プロジェクト別ロール(admin / member)では付与不可な機能:

  • プロジェクトの作成・編集・削除(プロジェクト設定全般: 名称・許可ドメイン・ウィジェット鍵ローテーション 等)
  • 請求(SCR-022)と設定(SCR-023)── 質問数上限・アラート変更はプロジェクト単位(SCR-021-M1)で該当PJ管理者にも付与する
  • 退会(SCR-014)
  • 規約再同意(SCR-015 の同意 / 不同意)
  • メンバーアカウント論理削除(DELETE /projects/{id}/members/{userId})── オーナー操作時は全メンバー削除可、プロジェクト管理者(オーナー以外)操作時は当該プロジェクトの member のみ削除可(他 admin は member 在籍中削除不可)

UI 上は非表示 + API は 403(E-AUTHZ-OWNER-ONLY)。

5.5 プラン制限との関係

MVP は単一プラン。利用上限は認可判定の 次段 で適用される(まず認可 → 通過したら上限チェック → 通過したら処理実行)。上限は 2 系統 ── レート制限(契約単位、owner_quota_overrides)と月次上限件数・無料枠(プロジェクト単位、project_quota_limits、対象 project_id で解決)。なお 上限の編集操作 は本判定(上限チェック)とは別で、プロジェクト上限の編集は requireProjectRole(admin)(オーナーは isOwner bypass)で認可する(オーナー専有ではない)。詳細は 10_課金・請求設計.html §3 / §3a 参照。

6. 認可判定ルール

6.1 Principal 構築フロー(§3.3.1 から移管)

Principal オブジェクト構造

Principal {
  userId:                 string
  contractOwnerUserId:    string                          // 本人がオーナーなら自分の userId、メンバーなら所属契約の owner user_id
  actorRole:              'admin' | 'service_operator'
  isOwner:                boolean                         // contract_owners 行が存在するかの boolean
  projectRoles:           Map<string /*projectId*/, 'admin' | 'member'>   // オーナーも project_users を持つが認可は isOwner=true bypass を優先
}

Principal 構築クエリ(参考)

-- 管理者(users)向け Principal 構築
SELECT u.id, u.status, u.valid,
       (co.user_id IS NOT NULL)                          AS is_owner,
       COALESCE(co.user_id, pu.contract_owner_user_id)   AS contract_owner_user_id
FROM users u
LEFT JOIN contract_owners co ON co.user_id = u.id AND co.valid = 1
LEFT JOIN project_users   pu ON pu.user_id = u.id AND pu.valid = 1
WHERE u.id = ?1 AND u.valid = 1
LIMIT 1;

projectRoles は別途 SELECT project_id, role FROM project_users WHERE user_id=? AND valid=1 で取得する。

ウィジェット利用者は管理コンソール用Principalを構築しない。公開キー・許可ドメイン・ウィジェットセッションで質問受付範囲を検証する。

認可ミドルウェア実装

  • オーナーはプロジェクト作成時に自動で project_users(role='admin', valid=1) 行を保持するため、users と contract_owners / project_users を結合する処理はオーナー / メンバー共通。ただし認可上は isOwner=true 判定(= contract_owners 行存在)が成立した時点で projectRoles の中身を参照せず bypass で全プロジェクト許可する(認可優先順位)
  • projectRoles は通知宛先解決と画面表示の事実情報として参照される(空 Map ではなく、オーナー自身の admin 行も含む)
  • 取得クエリには WHERE project_users.valid=1 フィルタを必須付与(論理削除済み行は除外)
  • 認可判定ヘルパ:
    • requireOwner(): principal.isOwner でない場合 403(E-AUTHZ-OWNER-ONLY)
    • requireProjectRole(projectId, minRole: 'admin' | 'member'): principal.isOwner ならば常時許可。それ以外は principal.projectRoles.get(projectId) が未定義なら 404 偽装(E-AUTHZ-PROJECT-DENIED)、minRole='admin' のときロールが admin でなければ 403(E-AUTHZ-FORBIDDEN)
  • Principal: KV キャッシュ(TTL 60 秒)経由で読み取り
  • ロール変更時の即時反映よりも認可レイテンシ最適化を優先(FR-338)

6.2 認可判定の順序

  1. セッション検証(sessions テーブル参照、無操作 30 分 / 絶対 12 時間)── 失敗 → 401
  2. ユーザー論理削除確認(users.valid=1)── valid=0 なら 401(セッション失効扱い、ログイン画面へリダイレクト)
  3. 規約再同意状態確認(terms_agreements を doc_type 別に最新版照合。利用規約・プライバシーポリシーのいずれかが未同意)── 未同意 → SCR-015 へリダイレクト
  4. 契約状態確認(contract_owners.contract_status、所属契約)── suspended で許可機能外なら 403、deleted_pending / deleted なら 403
  5. アカウント状態確認(users.status='active')── pending_activation / pending_verification なら 403
  6. オーナー境界判定(principal.contractOwnerUserId === target.contract_owner_user_id)── 違反 → 404 偽装
  7. プロジェクトロール判定(requireProjectRole):
    • principal.isOwner=true のとき自契約配下の全プロジェクトに無条件アクセス可(bypass)
    • メンバーは対象 project_id が principal.projectRoles に含まれ、必要ロール以上を満たすこと
    • 違反 → 404 偽装(E-AUTHZ-PROJECT-DENIED)/ ロール不足 → 403(E-AUTHZ-FORBIDDEN)
  8. オーナー専有機能判定(requireOwner、contract_owners 行を持たないユーザーがプロジェクト作成 / 課金 / 退会 / 規約再同意 等にアクセス)── 違反 → 403(E-AUTHZ-OWNER-ONLY)
  9. 最後の管理者保護(ロール変更 / 離脱対象が、当該プロジェクトの最後の admin 行になる場合)── 違反 → 403(E-AUTHZ-LAST-ADMIN-PROTECTED)。判定はオーナー由来 admin 行を contract_owners JOIN または NOT IN サブクエリで除外し、メンバー由来の admin 行のみで件数判定する
  10. 再認証判定(重要操作の場合、FR-005)── 5 分以内 + 1 回限り、未保持なら 403
  11. 利用上限判定 ── レート制限 = owner_quota_overrides(契約単位)/ 月次上限件数 = project_quota_limits(プロジェクト単位、対象 project_id で解決、KV usage-limit:<owner>:<projectId>)── 上限到達 → 429 + Retry-After

6.3 critical 通知の宛先解決(共有概念正本)

critical 重要度の通知 = メール強制送信。送信先は 「project_users.role='admin' AND valid=1 を 1 件でも保持するユーザー」(オーナー自身もプロジェクト作成時に admin 行を自動取得するため、本クエリ単独でオーナー + プロジェクト管理者を網羅できる)。

SELECT DISTINCT u.id, u.email_encrypted, u.email_hmac, u.email_key_version
FROM users u
JOIN project_users pu ON pu.user_id = u.id
WHERE pu.contract_owner_user_id = :contractOwnerUserId
  AND pu.role = 'admin' AND pu.valid = 1
  AND u.status = 'active' AND u.valid = 1;

クエリ設計: オーナー自身が admin 行を持つため、admin 行の JOIN 条件単独でオーナー + プロジェクト管理者を網羅できる。DISTINCT でメンバーが複数プロジェクトで admin を持つ場合の二重カウントを防ぐ。オーナー側も project_users 行が 1 件以上あるため DISTINCT 対象に含まれる(二重カウント無し)。

部分インデックス idx_project_users_role_admin(WHERE role = 'admin' AND valid = 1)で高速化。詳細は 06_メッセージ一覧.html §2.4 も参照。

プロジェクト単位アラートの宛先解決: 質問数上限の選択アラート閾値(25/50/80/90/100%、FR-127)は 当該プロジェクトの管理者(オーナー + 当該 PJ の admin)のみにメール配信する。上記の契約横断クエリに AND pu.project_id = :targetProjectId を加えた プロジェクト限定版を用い、ユーザーIDと正規化メールアドレスで重複排除する。100%到達時の停止処理はアラート選択と独立して実行する。

SELECT DISTINCT u.id, u.email_encrypted, u.email_hmac, u.email_key_version
FROM users u
JOIN project_users pu ON pu.user_id = u.id
WHERE pu.contract_owner_user_id = :contractOwnerUserId
  AND pu.project_id = :targetProjectId
  AND pu.role = 'admin' AND pu.valid = 1
  AND u.status = 'active' AND u.valid = 1;

7.1 サスペンション中の認可ゲート(§6.2.7 から移管)

contract_owners.contract_status='suspended' 中:

その他 403

reason 別のセッション扱い:

  • operator_manual / tos_violation: 5 秒以内に全セッション無効化(FR-022)
  • payment_failure_grace_expired: 既存セッション継続(課金 / 退会のみ)

  • payment_failure_grace_expired / operator_manual / tos_violation: 顧客管理側の連携 IF #1 受信(支払い成功 / 運営者解除)で contract_owners.contract_status='active' へ。

7.2 退会・解約フロー(§6.2.8 から移管)

オーナーが SCR-014 で退会申請
  → 再認証 FR-005
  → contract_owners.contract_status='deleted_pending'
  → ウィジェット応答停止、新規 FAQ 登録不可
  → 当月末まで利用可
  → 月末締めで利用停止(月割りなし)
  → 30 日削除猶予開始(物理削除まで)
  → 物理削除バッチ実行: contract_owners.contract_status='deleted' + valid=0(CHECK で同時セット強制)
  → 全データ削除 + announcement_recipients 削除

詳細は 11_課金請求設計書.html §5 参照。

7.3 規約改定フロー(§6.2.9 から移管)

運営者が新規約を準備
  → 発効 30 日前: inbox(announcement/critical)+ メール + ステータスページ
  → 契約単位で段階的に発効(全契約同日強制ではない)
  → 新規約発効
  → ログイン時に同意状況チェック
    ├─ Yes(同意済み): 通常画面へ
    └─ No: SCR-015 規約再同意割込み
        ├─ 同意期限 14 日以内
        │  ├─ Yes(同意): 通常画面へ
        │  └─ No(不同意): 退会フロー(SCR-014)へ誘導
        └─ 同意期限超過時: 機能制限ガード(課金画面操作と新規プロジェクト作成は不可、FAQ 編集とウィジェット稼働は継続)
観点 仕様
通知タイミング 改定発効日の 30 日前
通知経路 inbox(announcement/critical) + メール(強制送信)+ ステータスページ
同意期限 発効日 + 14 日(計 44 日の余裕)
同意期限超過時 SCR-015 強制割込み。ログイン自体は可能、機能制限
段階的実施 契約単位で発効日を分散
非同意時 退会フロー(SCR-014)へ誘導

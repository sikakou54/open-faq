認証・認可設計(メイン)

このページは、利用者側(オーナー / プロジェクト管理者 / メンバー)の認証(本人確認)と認可(操作可否)の方針・判定フローを定義します。

1. 認証方式

本システムが採用する本人確認の手段を一覧で示します。次の表は、各方式の採否と用途、有効期限などの制約・正本リンクを並べたもので、利用者の登録から日常ログイン、パスワード再設定、メンバー有効化までを 1 つの方式表で見渡せます。

認証方式採否説明制約・参照
メールアドレス + パスワード 採用 主たる認証方式。ログイン時にパスワードを検証する パスワード要件は §2、ハッシュ方式は 09_セキュリティ設計.html が正本
パスワード再設定リンク 採用 メールに送る短期トークンで本人到達を確認し再設定させる 有効期限 1 時間 / 1 回限り(FR-006)。トークン定義は access_tokens
招待トークン(メンバー有効化) 採用 招待リンクから本人が初回パスワード・氏名・規約同意を入力して有効化する 有効期限 7 日(FR-016 / FR-020)。フローは §3

Bot 対策(CAPTCHA 等) 採用 必要に応じてログイン等で発動する(FR-177) 判定方針は 09_セキュリティ設計.html が正本

3. 認証フロー

ログインからログアウト、パスワード再設定、メンバー有効化、メール確認、運営者起動の強制ログアウトまで、認証にかかわる各フローの処理手順をまとめます。以降の各小節は、それぞれのフローを順序付きの手順表で示します。

3.1 ログイン

通常ログイン時の処理を入口から初期画面遷移まで順に追います。次のシーケンス図は、資格情報の検証・セッション確立に加え、規約再同意ゲート(§6)と契約状態ゲート(§5.7)をログイン直後に通す順序と、不一致・停止状態などの分岐を示します。

sequenceDiagram
  participant U as ブラウザ(利用者)
  participant W as worker-main-api
  participant D as D1
  participant K as KV (session)

  U->>W: メール + パスワード送信
  W->>D: 資格情報を照会
  D-->>W: 照合結果(一致 / 不一致)
  alt 不一致
    W-->>U: 認証失敗(失敗カウント加算・ヒントなし)
  else 一致
    W->>K: セッション確立(sessions 行作成)
    K-->>W:
    W-->>U: セッション Cookie 発行
    W->>D: 規約再同意の要否を確認
    D-->>W: 未同意文書(あり / なし)
    opt 未同意あり
      W-->>U: 規約再同意へ割込み(SCR-015)
    end
    W->>D: 契約状態を確認
    D-->>W: 契約状態(active / suspended / deleted_pending / deleted)
    alt active
      W-->>U: 管理範囲に応じた初期画面へ遷移
    else suspended / deleted_pending / deleted
      W-->>U: 停止時アクセス制限(§5)を適用
    end
  end
補足不一致時は失敗カウントを加算し、ヒントを与えない応答を返します(必要に応じ Bot 対策チャレンジを挟む、FR-177)。契約状態が suspended / deleted_pending / deleted の場合は §5 のアクセス制限に従います。

3.6 強制ログアウト(運営者操作)

運営者システム側の操作を契機に、配下利用者のセッションをどう失効させるかを定めます。次のシーケンス図は、手動停止・規約違反停止での全セッション無効化、長時間処理中の扱い、停止後の次回操作での再認証・停止時アクセス制限(§5)を示します(FR-022 / BR-034)。

sequenceDiagram
  participant OP as 運営者システム
  participant W as worker-main-api
  participant K as KV (session)
  participant U as 配下利用者

  OP->>W: 停止 / 規約違反停止を契機として連携
  W->>K: 配下管理者ユーザーの全セッションを速やかに無効化
  K-->>W:
  W-->>OP:
  Note over W: 進行中の長時間処理は完了まで継続(中断しない)
  U->>W: 次回操作

運営者起動の主体強制ログアウトは運営者システム側の操作を契機とします。連携の方式・宛先解決は 02_API設計/index.html を正本とし、本書は「契機を受けてセッションをどう失効させるか」の方針に限定します。

複数の失効要因が同時に当てはまるとき、どれを優先して判定するかを定めます。次の表は運営者の強制ログアウトを最優先とし、絶対タイムアウト・無操作タイムアウト・通常セッションの順で評価する優先順位です。重要操作の再認証(§2)はこの寿命判定とは別軸で働きます。

5. 認可モデルと判定フロー

誰が何を操作できるかを判定する認可モデルを定めます。ユーザー種別とオーナー判定、契約・プロジェクトの境界、判定段の順序、専有・保護制約、通知宛先解決、アカウント状態・契約状態によるアクセス制限を以降の小節で順に示します。

5.1 ユーザー種別とオーナー判定

認可の起点となるユーザー種別と、その見分け方を定めます。次の表は contract_owners 行の有無でオーナーかどうかを判定し、非オーナーは project_users.role で操作範囲が決まることを示します(FR-015a / FR-015b)。

ユーザー種別判定権限の表し方
オーナー contract_owners 行が存在する(contract_owners.user_id PRIMARY KEY で 1 契約 1 オーナーを DB レベル担保) 全プロジェクトへの全権 + オーナー専有機能。役割変更・降格・譲渡なし(FR-015a)
プロジェクト管理者 / メンバー contract_owners 行を持たず、project_users 行を保持する プロジェクトごとの役割(project_users.role = admin / member)で操作範囲が決まる(FR-015b / FR-336)
ウィジェット利用者 管理画面の主体行を持たない 公開キーとウィジェットセッションで質問送信のみ。管理画面の Principal は構築しない
オーナー判定 = isOwner bypass を先頭にプロジェクト作成時にオーナーは当該プロジェクトの project_users(role='admin', valid=1) 行を自動取得しますが(FR-015e)、認可は contract_owners 行存在による isOwner=true bypass を最優先とし、オーナーは自契約配下の全プロジェクトに無条件でアクセスできます(FR-015d / FR-021b)。自動付与された admin 行は認可の根拠には使わず、通知宛先解決と画面表示の事実情報として参照します(§5.5)。

権限外は原則非表示・直アクセス拒否権限外の主ナビゲーションと操作は原則として表示しない。共同作業上、存在を知らせる価値がある項目のみ読み取り専用で表示し、必要権限と依頼先を説明する。権限のない画面への直アクセスは、同等の説明と戻り先を添えて拒否する(FR-341)。表示制御の画面仕様は 01_画面設計/index.html が正本。
役割・割当変更の反映タイミング(D-03)役割やプロジェクト割当の追加・剥奪では既存セッションを即時失効させず、次回の認可チェック時点で反映します(FR-338)。認可判定で参照する役割・割当情報のキャッシュ反映上限(TTL)は 60 秒以内 とし、これを本書で正本化します(FR-338 が反映上限時間とキャッシュ方式を本書へ委任)。変更から最大 60 秒で次回認可チェックに反映されます。

5.4 オーナー専有機能・保護制約

特定のロールに付与してはならない専有機能と、運用が止まらないための保護制約を定めます。次の表はオーナー専有機能・オーナー保護・自己操作の禁止・最後の管理者保護をまとめたもので、各制約に該当する操作は §5.3 の対応する判定段で拒否されます(FR-333 / FR-334)。

制約内容根拠
オーナー専有機能 課金情報・支払方法・請求履歴・退会・規約再同意・契約設定、プロジェクトの作成・編集・削除はオーナー専有。いずれの役割にも付与しない。プロジェクト単位の利用上限・無料枠は、オーナーまたは当該プロジェクト管理者が変更できる FR-015c / BR-044
オーナー保護 オーナーに対する退会・停止・削除・降格・譲渡を一切受け付けない。1 契約 = 1 オーナーのため孤立状態は構造的に発生しない FR-333
自己操作の禁止 自分自身の利用状態変更・削除はできない。当該プロジェクトの管理者が自分 1 人のとき、自身を管理者からメンバーへ降格できない。自身のアカウント削除は常に不可で、他人を経由して自身を削除する経路も設けない FR-334
最後の管理者保護 役割変更・離脱の対象が当該プロジェクトの最後の管理者になる場合は拒否する。オーナー由来の自動 admin 行は件数判定から除外し、メンバー由来の admin 行のみで判定する FR-018a / FR-334

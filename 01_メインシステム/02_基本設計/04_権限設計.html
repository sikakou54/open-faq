権限設計書(メイン)

1. 文書概要

1.1 目的

メインシステムの権限モデル(オーナー / プロジェクト管理者 / メンバー / エンドユーザー / 運営者)、プロジェクト別ロール 2 値、画面別・機能別・API 別の操作可否、契約別利用上書き を一元化する。

1.2 対象範囲

  • 対象: 利用者側の操作者種別(管理者 = users テーブル、公開ウィジェット = セッショントークン。運営者は別 D1 の service_operators テーブルで管理し、メイン側 users には格納しない)、プロジェクト別ロール(project_users.role = admin / member)、オーナー専有機能(contract_owners 行存在で判定)、契約別利用上書き
  • 対象外: 運営者ロール(02_運営者システム/個別設計書群/05_権限設計書.html 参照)、認証フロー(09_認証認可設計書.html 参照)

2. ロール一覧

利用者側は 3 つのロール(オーナー / プロジェクト管理者 / メンバー)で構成する。プロジェクト管理者とメンバーは「プロジェクト単位」のロールであり、同一メンバーがプロジェクト A では admin、プロジェクト B では member、プロジェクト C には未割当、といった組合せをとり得る。

ロールID 表示名 説明文(UI) DB 値
OWNER 契約オーナー 契約全体を管理し、プロジェクト作成・課金・退会などすべての設定を操作できます。 users 行存在 AND contract_owners.user_id = users.id 行存在
PROJECT_ADMIN プロジェクト管理者 担当プロジェクトのFAQ、問い合わせ対応、メンバー招待・ロール変更を行えます。 users 行存在 AND contract_owners 行を持たない AND project_users.role='admin'(プロジェクト単位)
MEMBER プロジェクトメンバー 担当プロジェクトのFAQや問い合わせを確認・対応できます。 users 行存在 AND contract_owners 行を持たない AND project_users.role='member'(プロジェクト単位)
END_USER ウィジェット利用者 公開ウィジェットからFAQ質問を送信し、AI回答・連絡先案内を確認できます。管理用の問い合わせIDは表示されません。 永続ロールなし。公開キー + ウィジェットセッション
SERVICE_OPERATOR サービス運営者 サービス提供者として、サポートや障害対応に必要な運用操作を行えます。 別 D1(運営者システム service_operators テーブル)── メイン側 users には格納しない。メイン API への運営者アクセスは IF 経由の認可コンテキストで実現し、audit_logs.actor_role='service_operator' の値として記録する

用語の衝突回避: 管理コンソールの認証主体は users テーブル(全行が管理者)で、ウィジェット利用者は永続アカウントを持たない。project_users.role='admin' はプロジェクト管理者ロールを示し、users テーブルのロール概念とは別軸。audit_logs.actor_role は操作者種別(admin / system / service_operator)を記録するもので、users のロール定義とは独立した概念。

3. ロール定義(project_users.role 2 値 + オーナー)

3.1 ロール概要

ロール 表示名(UI) 範囲 できること
オーナー 契約オーナー 契約全体 契約全体の利用状況・請求・設定を確認し、各プロジェクトの設定とメンバー管理を行えます。
admin プロジェクト管理者 該当プロジェクト 担当プロジェクトのFAQ公開、問い合わせ対応、ログ確認、データ出力、メンバー招待・ロール変更ができます。

オーナーの project_users 行: オーナーはプロジェクト作成時に project_users(role='admin', valid=1) 行が自動 INSERT され、当該プロジェクトの明示的なプロジェクト管理者として行を保持する。認可上は Principal 構築時の isOwner=true 判定(contract_owners 行存在)で bypass が優先され、project_users 行は critical 通知の宛先解決と画面表示の事実情報として参照される(詳細は 08_認証・認可設計.html §6.1)。

3.3 オーナー専有機能(プロジェクト別ロールでは付与不可)

  • プロジェクトの作成・編集・削除(作成・編集・削除 = SCR-004-M1 プロジェクト作成・編集モーダル)
  • 利用状況・請求・設定(SCR-016 / SCR-022 / SCR-023)── プロジェクトの質問数上限・アラート変更はSCR-021-M1でプロジェクト管理者にも付与する
  • 退会(SCR-014)
  • 規約再同意(SCR-015 の同意 / 不同意)

UI 上は非表示 + API は 403(E-AUTHZ-OWNER-ONLY)。

SCR-018 メンバーアカウント有効化ページ: 上記表外の特別画面。Cookie セッション認証を持たない未認証ユーザーが招待リンク経由でアクセスする。認可条件は access_tokens.purpose='activation' の有効トークン(期限内 + 未使用)を保持していることのみで、ロール列(オーナー / プロジェクト管理者 / メンバー)は適用されない。トークン認証 Principal(user_id + access_tokens.meta.invitedProjectId / invitedRole)を構築し、本画面以外でこの Principal は使用しない。詳細は 08_認証・認可設計.html §3.4 / §5.1 を参照。

SCR-019 プロジェクト連絡先メール確認完了ページ: 同様に未認証ユーザーがプロジェクト連絡先確認トークン経由でアクセスする画面。access_tokens.purpose='contact_verify' の有効トークン保持が唯一の認可条件で、連絡先メール所有者はアカウントを持つ必要がない(第三者の共有メールアドレスでも構わない)。トークン認証 Principal(access_tokens.meta.projectId のみ、user_id は持たない)を構築し、projects.contact_email_verified_at の単純な書き込みのみを行う。本画面以外でこの Principal は使用しない。詳細は 08_認証・認可設計.html §5.1 を参照。

7. 利用制限 / 運営者上書き(レート=契約単位 / 月次上限件数=プロジェクト単位)

7.1 利用上限の構成(10_課金・請求設計.html §3 / §3a 参照)

MVP は単一プラン(完全従量課金 + 月次無料枠、無料トライアル期間なし)。プラン階層による差別化はない。利用上限は 2 系統 で実装される:

  • レート制限(契約単位): DDoS / Bot / 暴走対策のセキュリティ施策。owner_quota_overrides.resource_kind(レート種別)で実装。主管 SCR = SCR-093(運営者、4-eyes 承認)。
  • 月次上限件数・無料枠(プロジェクト単位): 暴走コスト防止と課金境界の設定。project_quota_limits(§3.24)で実装。SCR-021 は質問数だけを主管し、FAQ無料枠は SCR-093 の運営者料金設定で管理する。

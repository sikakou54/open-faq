権限設計(メイン)

このページは、メインシステムで「誰が・どの画面で・何をできるか」を操作 × ロールの網羅マトリクスとして定義します。

版数 v1.7更新 2026-06-17承認済

1. ロールモデル

利用者側のロールは オーナー / プロジェクト管理者 / メンバー の 3 種です。オーナーは契約全体の全権を恒久的に持ち、プロジェクト管理者とメンバーは「プロジェクト単位」のロールである点が要です。同一ユーザーがプロジェクト A では管理者、B ではメンバー、C には未割当という組合せを取り得ます。判定キーとテーブルの正本は 03_テーブル設計/index.html と 共有概念 §6 です。

2. オーナー専有操作

契約の単一障害点化と事故を避けるため、契約レベルの操作は オーナー専有とし、いずれのプロジェクト別ロールにも付与しません。非オーナーが要求した場合は UI 上で非表示にし、API は 403(E-AUTHZ-OWNER-ONLY)を返します。専有機能の認可上の扱いの正本は 08_認証・認可設計.html §5.4、エラー定義の正本は 05_エラー設計.html §4.3 です。

3. SCR × ロール アクセス制御マトリクス

本節が「画面ごとに各ロールが何を参照・操作できるか」の網羅マトリクスの正本です。SCR の定義そのものは 01_画面設計/index.html が正本で、ここでは可否のみを示します。表の凡例は次のとおりです。

  • ○ = 参照・操作とも可
  • 閲覧 = 参照のみ可(操作不可)
  • × = 不可(権限外。原則 UI 非表示 + 直アクセス拒否、FR-341)
  • — = ロール概念が適用されない(未認証トークン専用画面)

「プロジェクト管理者」「メンバー」列は当該プロジェクトに割当があり、そのロールを保持する場合の挙動を示します。割当のないプロジェクトへの直アクセスは 404 偽装で拒否します(08_認証・認可設計.html §5.2)。

3.1 未認証トークン専用画面(SCR-018 / SCR-019)

SCR-018 と SCR-019 は Cookie セッション認証を持たない未認証ユーザーがアクセスする画面で、ロール(オーナー / 管理者 / メンバー)の概念が適用されません。認可条件はトークンの有効性のみで、トークン認証 Principal の構築仕様は 08_認証・認可設計.html §5.1 が正本です。

4. 利用上限の変更権限

月次の質問数上限・無料枠はプロジェクト単位でオーナーまたは当該プロジェクト管理者が変更できます。値・判定ロジックの正本は 10_課金・請求設計.html、テーブルの正本は 03_テーブル設計/index.html §3.24 です。本節は「誰が変更できるか」のみを定めます。

5. ロールを満たしても拒否される制約

必要ロールを満たしていても、事故や契約の単一障害点化を防ぐために拒否される操作があります。本節はその不可制約の網羅です。判定順序とエラーへの落とし込みの正本は 08_認証・認可設計.html §5.4、エラー ID・メッセージの正本は 05_エラー設計.html §4.3 です。

6. 契約状態による許可操作の制限

契約が停止・退会猶予中のときは、ロールを満たしていても管理画面の許可操作を必要最小限に絞ります。契約状態値(contract_status)の定義とウィジェット側挙動の正本は 10_課金・請求設計.html、管理画面側の認可ゲートの正本は 08_認証・認可設計.html §5.7 です。本節は権限の観点で許可範囲を整理します。

契約状態管理画面の許可操作根拠 FR
停止中(suspended) 課金(請求情報更新)と退会のみ許可。その他の機能と新規ウィジェット応答は機能停止応答に切替 FR-138
退会申請中(deleted_pending) 管理画面ログインと既存データ参照は可。新規 FAQ 登録・新規ウィジェット応答・課金変更などの新規書込みは不可。退会キャンセル動線を提供 FR-167
退会済み(deleted) ログイン不可・操作不可 —
支払方法ゲートは契約停止ではない支払方法未登録 + 無料枠超過によるウィジェット質問受付の停止は契約サスペンションではなく、contract_status は active のままです。本表の制限対象外で、管理画面は通常どおり利用できます(正本は 10_課金・請求設計.html)。

セキュリティ設計(メイン)

このページは、メインシステム(利用者向け FAQ ウィジェット SaaS)のセキュリティ方式 ── データ暗号化・鍵管理・Web 脆弱性対策・監査ログ・不正利用検知・メール配信信頼性・ウィジェット保護・AI 安全性 ── を定義します。

版数 v1.2更新 2026-06-17承認済

2. セキュリティ基本方針

メインシステムは「多層防御・最小権限・監査の網羅・改ざん検知」を基本とする。本節で守るべき対象と想定脅威を先に示し、以降の各節で具体策を定める。

2.1 保護対象

本システムが守るべき資産を列挙する。以降の各節の対策は、ここに挙げた対象のいずれかを保護するために設ける。

  • 利用者の PII(メール / 名前 / 個人識別情報)
  • エンドユーザーの質問入力
  • 認証情報(パスワード / セッション / トークン)
  • 暗号鍵(マスター / バックアップ / オーナー派生 / グローバル派生)
  • 課金・決済情報(Stripe 連携、カード情報は Stripe に委譲)
  • 監査ログ(改ざん検知必須)

2.2 想定脅威

保護対象に対して想定する代表的な脅威を列挙する。各脅威にはリスク ID(R-xxx)を対応させ、§5 以降の個別対策で打ち消す。

  • 不正アクセス(クレデンシャルスタッフィング / ブルートフォース / R-005)
  • データ漏洩(他契約データ参照 / R-004)
  • インジェクション(SQLi / XSS / プロンプト注入)
  • メールレピュテーション悪化(R-003 / 共通ドメイン使用)
  • DDoS / Bot 暴走
  • AI 出力経由の機密漏洩 / PII 漏洩

2.3 基本対策方針

脅威に共通して適用する基本姿勢を 4 本柱で定める。個別の節はこの方針を具体化したものとして読む。

  • 多層防御(認証 → 認可 → 入力検証 → 出力検査 → 監査)
  • 最小権限原則(オーナー専有 / プロジェクト別ロール admin / member(M_project_user_assignments.role)の二段階)
  • 監査の網羅性(変更系操作は全て監査、参照系は高権限操作のみ)
  • 改ざん検知(ハッシュチェーン + 完全性検証バッチによる完全性検証を行う)

3. データ保護

機密データは「暗号化(復号が必要なもの)」と「ハッシュ保存(照合だけ必要なもの)」を使い分け、PII はオーナー派生鍵で暗号化する(FR-171)。PII 暗号化方式とオーナー派生鍵はメインを共有概念正本とする(共有概念.html #9)。

3.1 PII 暗号化(AES-256-GCM)+ オーナー派生鍵

利用者の PII は AES-256-GCM とオーナー派生鍵で暗号化し、トークンは原文を保存せず HMAC ハッシュのみを保持する。いずれもログ出力は不可とする。暗号化対象カラムの定義は 03_テーブル設計/テーブル構造設計.html §8 を正本とする。

対象データ 暗号化方式 鍵 保存先 ログ出力
利用者メール AES-256-GCM オーナー派生鍵 M_owners / M_project_users の email_encrypted + email_hmac 不可
招待トークン / 各種トークン HMAC-SHA256 オーナー派生鍵 / グローバル派生鍵 直接保存しない(ハッシュのみ) 不可

3.2 保護層別方針

通信・パスワード・トークン・監査・境界・マスキングといった保護層ごとに採用方式と関連 NFR / FR を一覧する。各方式の詳細は対応する正本へリンクし、ここでは層と方式の対応を示す。

層 方式 関連
通信 HTTPS 必須 NFR-301
HSTS Strict-Transport-Security: max-age=31536000; includeSubDomains; preload NFR-315
パスワード Argon2id m=64MB, t=3, p=4, salt=16B NFR-304
トークン HMAC-SHA256 ハッシュ保存、オーナー / グローバル派生鍵使い分け NFR-305
監査ログ 最小限情報、PII 除外 FR-114 / §7.2
オーナー境界 全 API で境界検証、自動テスト月次 50 ケース + 年次手動レビュー NFR-302
個人情報マスキング AI 出力・管理画面表示で §13 PII Scrubber 層適用 FR-060 / NFR-319
AI 学習利用禁止 AnswerProvider learn=false 強制、Workers AI ログ無効化 NFR-405

3.3 データ削除モード(物理削除 / 匿名化)

退会時の契約データ削除(FR-163)は、物理削除と匿名化のいずれかをオーナー設定で選ぶ。匿名化は原文復元不可な置換のみとし、集計値は維持する。削除ポリシーと data_deletion_mode カラムは 03_テーブル設計/テーブル構造設計.html §6 を正本とし、本書は秘匿の担保事項を定める(共有概念 #17)。

オーナー設定 data_deletion_mode(physical / anonymize、初期値 physical)

対象テーブル 物理削除 匿名化
H_question_logs DELETE user_question 空字、ip_address / user_agent NULL、集計値維持
T_inquiries DELETE user_question 空字、inquiry_code 維持、closed_reason 空字
H_notification_logs DELETE recipient_email_hmac → SHA-256、vars → 空 JSON
T_access_tokens DELETE (常に物理削除、匿名化対象外)
H_audit_logs 削除しない (tombstone 方式、匿名化対象外)

担保事項:

  • 不可逆性: メールアドレスハッシュ化など、原文復元不可な置換のみ
  • 設定管理: M_owners.data_deletion_mode は契約締結時に既定 physical。MVP では UI 上の変更操作を提供せず、変更が必要な場合はサポート窓口経由の対応とする(SCR-017 個人設定にデータ保持・削除モード設定 UI は持たない)
  • 監査: H_audit_logs.action='owner.data_deletion_mode.change'(retention_class=general)

3.4 お知らせ二層構成

運営お知らせ等の通知は、配信元・配信宛先・受信箱の三層に分けて記録する。重複生成を抑止しつつ配信集計と監査を両立させるための分離方針を示す。各テーブルの定義は 03_テーブル設計/index.html §1 を正本とする。

三層構造:

  • 配信元層(Control Plane): M_service_announcements テーブル
  • 配信宛先層: T_announcement_recipients(送信日時・配信失敗フラグ等、監査用)
  • 受信箱層(Tenant): T_inbox_messages(管理者のアプリ内閲覧用)
種別 トリガー 二層構成経由 特記
announcement お知らせを公開 M_service_announcements → T_announcement_recipients → T_inbox_messages 配信集計が必要

system 上限接近・超過等 直接 T_inbox_messages ヘッダベル + メール(high/critical)

重複生成対策(R-012): 同一 (contract_owner_user_id, event_kind, dedup_key) で 60 分以内の複数発火は 1 件に集約

4. 入力値検証

5. Web 脆弱性対策

XSS・CSRF・SQLi・クリックジャッキング等の代表的な Web 脆弱性は、応答ヘッダ(CSP / HSTS 等)とアプリ層の対策(エスケープ・トークン突合・プリペアドステートメント)を組み合わせて防ぐ。管理画面とウィジェットでは frame-ancestors の方針を分ける。

5.1 HTTP 応答ヘッダ

全レスポンスに以下のセキュリティヘッダを付与する。管理画面は frame-ancestors 'none' で iframe 化を禁止し、ウィジェットは顧客サイト埋め込みのため許可ドメインに限定する。

Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-<random>';
  style-src 'self' 'nonce-<random>';
  img-src 'self' data: https:;
  object-src 'none';
  frame-ancestors 'none';
  base-uri 'self';
  form-action 'self'

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()

ウィジェット側は frame-ancestors *(顧客サイト埋め込みのため)。

5.2 主要脅威別対策

XSS・CSRF・SQLi・SSRF・クリックジャッキング・ブルートフォース等の脅威ごとに、採用する対策と適用対象箇所を一覧する。管理画面とウィジェットで frame-ancestors の方針が分かれる点に注意する。

脅威 対策 対象箇所
XSS CSP script-src 'self' 'nonce-<random>' + 自動エスケープ + お知らせ本文 2 重サニタイズ 全画面
CSRF Double Submit Cookie + X-CSRF-Token ヘッダ突合 全更新系 API
SQL インジェクション プリペアドステートメント / ORM 全 DB アクセス
SSRF 外部 URL 取得時のホワイトリスト (該当機能なし、MVP)
クリックジャッキング(管理画面) X-Frame-Options: DENY / frame-ancestors 'none' 管理画面
クリックジャッキング(ウィジェット) frame-ancestors を M_allowed_domains で制限 ウィジェット iframe
ブルートフォース ログイン失敗 5 回ロックアウト SCR-001 / SCR-003
HSTS Strict-Transport-Security 全レスポンス

5.3 お知らせ本文サニタイズ(2 重防御)

運営お知らせの HTML 本文は、入稿時(受信時)と表示時の二重でサニタイズし、許可タグ・属性のホワイトリストで XSS を防ぐ。エンドユーザー入力はお知らせ本文に一切含めない(NFR-309)。

  • 入稿時: お知らせ入稿時にサーバ側 sanitize(許可タグ・属性ホワイトリスト)
  • 表示時: GET /me/announcements/{id} 時に再度 sanitize
  • 許可タグ: <p><div><h2-4><ul><ol><li><blockquote><pre><a><strong><em><code><span><br><img>
  • 禁止: <script><iframe><object><embed><form><input><style> + on* イベント属性
  • エンドユーザー入力: お知らせ本文に一切含めない(NFR-309)

7.3 改ざん検知方式(NFR-306)

監査ログの改ざん検知はハッシュチェーンで行う(NFR-306)。チェーン構造・ダイジェスト保管・完全性検証バッチにより H_audit_logs の完全性検証を行う。

7.4 エラーログ(H_error_logs)

サーバー内部起因のエラーは運用確認用に記録するが、PII・トークン・カード情報は一切残さない(FR-113 / FR-114)。テーブル定義は 03_テーブル設計/index.html §3.19 を正本とする。

  • 180 日保持(JST 05:00 Cron で自動物理削除、UC-140)
  • スタックトレース / リクエスト ID を記録
  • PII / トークン / カード情報を記録禁止(FR-114)
  • 入力値ログ: 先頭 100 文字 + SHA-256 ハッシュのみ
  • カテゴリ C の 5xx 系エラーは利用者にエラー ID を提示(FR-113)

8. 秘密情報管理(暗号鍵管理 = メイン側が共有概念正本)

マスター鍵・派生鍵・ウィジェット公開キーの保管とローテーションを定める(FR-171 / FR-193 / FR-194、NFR-320〜324)。暗号鍵管理はメインを共有概念正本とする(共有概念.html #10)。鍵を用いた実装手順は 詳細設計(DD10) をリンク先とする。

8.1 暗号鍵管理

マスター鍵はクラウド機密保管領域に置き年次ローテーションする(NFR-320 / NFR-321)。バックアップ鍵はマスター鍵と分離する(NFR-324)。

鍵種別 保管場所 用途 ローテーション周期
Master Key Cloudflare Secrets Store オーナー派生鍵・グローバル派生鍵のベース 年次
オーナー派生鍵 動的生成(HKDF-SHA256, info=owner:<id>) 旧エンドユーザーメール暗号化・トークン HMAC Master Key 更新時に再 KDF
グローバル派生鍵 動的生成(HKDF-SHA256, info=global) パスワードリセット・API キー生成 Master Key 更新時に再 KDF
API キー 動的生成・管理者操作で再発行 ウィジェット公開キー(pk_live_<32-char base62>) 無期限。失効は再発行(ローテーション)でのみ。再発行時に旧キーを 24 時間猶予で失効(FR-193 / FR-194)
Backup Key 別管理(Master Key と分離) 復旧用(AES-256-GCM 暗号化) 年次ローテーション(NFR-324)

8.2 ローテーション運用

マスター鍵交換中は新旧両鍵で復号を試みる dual-decrypt 期間を設け、ウィジェット公開キーの再発行は 24 時間の猶予を置く(NFR-322 / FR-193)。マスター鍵ローテーションは承認を要する。

操作 方式
Master Key ローテーション中 60 日 dual-decrypt(旧鍵と新鍵の両方で復号を試行)(NFR-321)、ローテーションは承認を要する
API キーローテーション猶予 24 時間(FR-193)、SCR-007 に「旧キー使用中」バッジ表示
外部プロバイダ API キー(Resend / Stripe) Workers Secrets Store 保管、90 日ローテーション を目標(NFR-323)
漏洩検知時 (a)Secrets Store 即時ローテーション、(b)該当プロバイダ管理画面から旧キー revoke、(c)影響契約へ inbox/high 通知、(d)全管理者セッション無効化

8.2.1 鍵バージョン参照

dual-decrypt 期間中はレコードごとにどの世代の鍵で暗号化したかを email_key_version で明示し、ローテーション完了後に旧鍵で暗号化された行を再暗号化する。鍵バージョン列の定義は 03_テーブル設計/index.html §3.1 を正本とし、本書は運用フローを定める。

対象テーブル カラム 説明
M_owners / M_project_users email_key_version 管理者(オーナー / プロジェクト管理者 / メンバー)のメールアドレス暗号化バージョン

運用フロー:dual-decrypt 期間中の暗号化・復号・再暗号化の流れを次のシーケンス図に示す。INSERT / UPDATE は最新鍵で暗号化し、SELECT は行の email_key_version に応じて N / N-1 を試行し、完了後にバッチで旧版行を再暗号化する。

      A->>S: 旧鍵を削除
      S-->>A:
    end
  end

ローテーション完了の判定:SELECT COUNT(*) FROM M_owners WHERE email_key_version < N + SELECT COUNT(*) FROM M_project_users WHERE email_key_version < N がいずれも 0 件であること。

例外:M_owners.contract_status='deleted'・M_owners.valid=0 または M_project_users.valid=0 の論理削除済み行は、90 日経過後の物理削除バッチに任せ、再暗号化対象外とする(復元時の dual-decrypt 整合性を維持するため、ローテーション期間は 60 日 + 物理削除遅延 90 日を考慮した運用窓設計とする)。

8.3 ウィジェット API キー(M_projects / T_project_legacy_keys)

ウィジェット公開キーは無期限で、失効は再発行(ローテーション)でのみ行う(FR-194)。再発行時は旧キーを 24 時間併存させ猶予後に完全失効する(FR-193)。キーと旧キーのテーブル定義は 03_テーブル設計/index.html §3.7 を正本とする。

  • 形式: pk_live_<32-char base62>
  • 有効期限: なし(無期限)。失効は再発行(ローテーション)でのみ行う
  • ローテーション: 旧キー 24 時間併存
  • 旧キー使用検知: SCR-007 に警告バッジ + 件数 + 最終アクセス時刻

10. 不正利用検知(FR-195 / NFR-308 = 共有概念正本)

大量送信・Bot 疑い・規約違反を自動検知し、自動ブロック(レート制限緊急引き下げ・契約 suspended 遷移)を行ったうえで、管理者ユーザーへ即時通知し申し立て窓口を案内する(FR-195、NFR-308 / NFR-317)。不正利用検知はメインを共有概念正本とする(共有概念.html #13)。判定しきい値は契約別に上書きできる。

10.1 検知・対応体系

検知ロジックと自動ブロックはメインシステム側で実行する。しきい値は契約別に上書きでき、発火時は管理者へ inbox + メールで通知する。次のシーケンス図は、しきい値超過の判定から自動ブロック(レート緊急引き下げ / 契約 suspended 遷移)・監査記録・管理者通知までの流れと、しきい値内かどうかの分岐を示す。続く表は図に表れない検知ロジックの所在・上書き契機・監査 action を補う。

sequenceDiagram
  participant E as ウィジェット利用者
  participant W as worker-main-api
  participant D as D1 (H_audit_logs)
  participant M as 管理者ユーザー

  E->>W: 大量送信 / Bot 疑い / 規約違反となる挙動

  alt しきい値超過

  end

検知ロジック メインシステム側で実装(FR-195(a))
しきい値 契約別に上書き可能(FR-195(b))。既定しきい値はメイン側検知ロジックが保持する

11. メール配信信頼性・レピュテーション保護

この §11 が配信信頼性方式の正本メール再送・バウンス / 苦情検知・全契約横断サプレスといった配信信頼性の方式は本節を正本とする(11_メール設計.html §5 から参照)。11_メール設計はテンプレート単位の挙動差のみを扱う。

通知メールは Queue 経由で送信し、一時失敗は当方制御で最大 3 回・最大遅延 24 時間まで自動再送する(FR-141 / FR-147、NFR-502 / NFR-506)。バウンス・苦情は検知して送信停止し、停止リストは全契約横断で共有する(FR-142 / FR-149b、NFR-503)。受信 Webhook は冪等処理し、滞留は 1 時間以内にアラートする(NFR-505 / NFR-507)。次のシーケンス図は、Queue 投入から送信成功 / 一時失敗時の再送 / バウンス・苦情 Webhook によるサプレス / 上限超過時の DLQ までの全体フローと分岐を示す。各節の表は図に無いしきい値・保持期間・イベント別処理を補う。

sequenceDiagram

  participant Q as Cloudflare Queues
  participant R as Resend
  participant N as D1 (H_notification_logs)
  participant DLQ as DLQ
  participant AL as アラート通知

  A->>Q: 送信ジョブを投入(同期処理外)
  Q-->>A:
  Q->>R: Consumer が送信

    N-->>A:
  else 一時失敗

    opt 上限到達

11.1 メール配信信頼性(NFR-502〜507)

送信から再送・冪等・DLQ までの配信信頼性方針を一覧する。当方の再送カウンタは Resend 側の自動再送と区別する。

項目 方針
送信 API 同期処理に含めず Queue 経由(NFR-507)
一時失敗時の自動再送 Queue Consumer による自動再送(指数バックオフ)、最大 3 回、最大遅延 24 時間(NFR-502 / NFR-506)
Resend 自動再送 email.delivery_delayed で Resend が裏で再送中。当方の再送カウンタに含めない(NFR-506)
バウンス Resend Webhook 検知、対象アドレスを通知停止(M_email_suppression_list 追加)(NFR-503)
苦情 Resend Webhook 検知、即時永久サプレス(復帰不可)(NFR-503)
監視 契約別レート・バウンス率・苦情率をしきい値監視(NFR-804)
サンドボックス 開発・検証環境は Resend サンドボックス使用
冪等性 event_id で重複処理を防止(NFR-505)
DLQ Cloudflare Queues DLQ、1 時間以内にアラート(NFR-507)

11.2 共通ドメインレピュテーション保護(FR-149b / NFR-504)

サービス共通ドメインの送信レピュテーションを守るため、契約別送信レート・バウンス率・苦情率を監視し、超過時に送信を抑制する(FR-144、NFR-504)。送信停止リスト M_email_suppression_list(03_テーブル設計/index.html §3.17)は全契約横断で共有する(FR-149b)。

項目 方針
送信レート制限 契約別時間あたり通数の上限設定(調整可)
バウンス率監視 しきい値超過時にアラート / 送信抑制
苦情率監視 同上
件名・本文 エンドユーザー入力をそのまま使わない(FR-145、なりすまし対策)
通知停止 バウンス・苦情・オプトアウトの即時反映
異常検知 短時間急増・未配信率急上昇を検知して通知
全契約横断サプレス M_email_suppression_list を全契約横断で共有(FR-149b)
復帰承認 サプレスリスト復帰は承認を要する

11.3 Resend Webhook 処理

送信プロバイダ(Resend)からの配信状態 Webhook を受信し、H_notification_logs(03_テーブル設計/index.html §3.15)の配信状態を更新する。署名検証に失敗した受信は即時破棄し通知する(NFR-310)。

イベント 処理
email.sent H_notification_logs.delivery_state='sent'
email.delivered delivery_state='delivered'
email.bounced(hard) サプレス + 管理者ユーザー inbox 通知
email.bounced(soft) カウンタ +1、5 回でサプレス
email.complained 即時永久サプレス + high 通知
email.delivery_delayed Resend 側自動再送待機、24h 後も未配信なら当方カウンタ +1
email.opened/clicked 集計のみ(個人特定なし)
署名検証失敗 即時破棄 + high 通知(NFR-310)

11.4 サプレスリスト復帰フロー

サプレス事由ごとの復帰可否と復帰方法を示す。苦情(complaint)は永久サプレスで復帰不可、ハード / ソフトバウンスは承認を経て復帰できる。

イベント アクション 復帰方法
ハードバウンス 即時サプレス、全契約横断共有 SCR-023 設定からサポート申請 → 承認 → 復帰
ソフトバウンス 5 連続 サプレス、全契約横断共有 同上(承認 → 復帰)
苦情(complaint) 即時永久サプレス 復帰不可
レピュテーション低下警告 契約送信レート -50% 24h 経過 + バウンス率 <1% で自動復帰

11.5 DLQ 運用

配信に失敗した通知ジョブは DLQ で保持し、一定時間は自動再試行、それ以降は明示的リプレイで処理する(NFR-507)。署名検証失敗は再処理せず即時破棄する。

項目 仕様
DLQ 保持期間 4 日間(Cloudflare Queues 現行制限値)。超える保持が必要なイベント → R2 にコピー保存
自動再処理 DLQ 投入から 1 時間以内 は自動指数 BO 再試行、それ以降は明示的リプレイ操作のみ
リプレイ範囲 直近 30 日(R2 退避分を含む)。30 日超は不可
DLQ 監視 DLQ 滞留が上限値を超えた瞬間に high 通知 + メール
署名検証失敗 即時破棄 + high 通知(NFR-310)

12. ウィジェット保護

ウィジェットは顧客サイトに埋め込まれるため、埋め込み元ドメイン検証(Origin / Referer)で許可ドメイン以外からの読み込みを拒否し(FR-173、NFR-312)、公開キー署名・CORS・iframe sandbox・レート制限・Bot 対策で多層に保護する(FR-172 / FR-177)。許可ドメインの定義は 03_テーブル設計/index.html §3.8(M_allowed_domains)を正本とする。

項目 方式
許可ドメイン検証 Origin / Referer で判定、完全一致 + *.example.com ワイルドカード
CORS 許可ドメインのみ、Access-Control-Max-Age=600
API 署名 公開キー(pk_live_<32-char base62>)+ セッショントークン組み合わせ認証
iframe sandbox sandbox="allow-scripts allow-forms allow-same-origin allow-popups"(allow-top-navigation なし)
CSP default-src 'self' https://api.<service>; script-src 'self'; frame-ancestors *
レート制限 プロジェクト / IP / セッション単位(調整可)
Bot 対策 Turnstile を必要に応じて適用(FR-177)
入力制限 文字数 2,000、連続送信(EU 10 件/分)、危険文字列無害化

13. AI 安全性(PII マスキング・プロンプト注入対策・出力検査)

AI 回答の安全性は、入力(プロンプト注入耐性)・処理(FAQ 限定方針の固定)・出力(参照 FAQ 外要素の検出と PII マスキング)の三段で担保する(FR-058 / FR-060)。PII マスキングはメインを共有概念正本とする(共有概念.html #9)。AI モデル / プロンプトの変更時は攻撃パターンを含む回帰テストを実行する(FR-058 / FR-349)。

13.1 PII マスキング

個人情報は Cloudflare ネットワーク内で検出し、所定形式でマスキングして返却する。検出種別は記録し、マスク前データは残さない(FR-060)。検出ルールは保守可能とする。

観点 方針
検出器配置 Cloudflare ネットワーク内(Workers AI + 正規表現ルール)、外部 API 送信不可
検出ルール保守 KV に保持、更新可能
監査記録 マスキング発動: H_audit_logs.action='ai.pii_masked'、検査 NG: ai.output_inspection_failed(マスク前データ非記録)
誤検出改善ループ 管理者ユーザーが「誤検出」報告 → 集約 → 検出ルール改善(AC-036、3 営業日以内判定)

13.2 プロンプト注入対策・出力検査

ウィジェット利用者の入力で FAQ 限定回答方針が変わらないよう、システム指示はサーバ側で固定しクライアント上書きを禁じる(FR-058)。出力前に参照 FAQ 外の固有名詞・数値・手順を検出した回答は返却せず未解決へ倒す(FR-060)。AI 回答ロジックの正本は FR05 を要件正本、方式は AI 回答の基本設計を正本とする。

対策 内容
FAQ 限定ポリシー固定 サーバ側でシステム指示を固定し、クライアント上書き不可(FR-058)
候補なし時 AI 生成へ進まない
出力検査 参照 FAQ 外の固有名詞・数値・手順を検出し未解決へ倒す(FR-060)
不審入力検知 命令上書き等を検知・記録(H_audit_logs.action='ai.prompt_injection_detected')
失敗時 安全側に倒し未解決登録または安全エラー
2 段構えテスト 社内 100+ ケース 95% × 必須 20 ケース 100%(AC-019 × AC-035)

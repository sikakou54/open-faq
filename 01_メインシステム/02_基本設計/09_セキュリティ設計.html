2. セキュリティ基本方針

2.1 保護対象

  • 利用者の PII(メール / 名前 / 個人識別情報)
  • エンドユーザーの質問入力
  • 認証情報(パスワード / セッション / トークン)
  • 暗号鍵(マスター / バックアップ / オーナー派生 / グローバル派生)
  • 課金・決済情報(Stripe 連携、カード情報は Stripe に委譲)
  • 監査ログ(改ざん検知必須)

2.2 想定脅威

  • 不正アクセス(クレデンシャルスタッフィング / ブルートフォース / R-005)
  • データ漏洩(他契約データ参照 / R-004)
  • インジェクション(SQLi / XSS / プロンプト注入)
  • メールレピュテーション悪化(R-003 / 共通ドメイン使用)
  • DDoS / Bot 暴走
  • AI 出力経由の機密漏洩 / PII 漏洩

2.3 基本対策方針

  • 多層防御(認証 → 認可 → 入力検証 → 出力検査 → 監査)
  • 最小権限原則(オーナー専有 / プロジェクト別ロール admin / member(project_users.role)の二段階)
  • 監査の網羅性(変更系操作は全て監査、参照系は高権限操作のみ)
  • 改ざん検知(ハッシュチェーン + 完全性検証バッチ、運営者側 §7 正本)

7.3 改ざん検知方式(NFR-306)

詳細は運営者側 10_セキュリティ設計書.html §7 を正本とする。本書では概要のみ:

  • ハッシュチェーン実装: 各行に prev_hash(直前行の current_hash) と current_hash(SHA-256(id, contract_owner_user_id, actor_user_id, action, target_type, target_id, payload, created_at, prev_hash))
  • 日次ダイジェスト保管: JST 03:00 に当日分最終 current_hash 抽出 + HMAC-SHA256 署名 + R2 へ書き出し

  • 180 日保持(JST 05:00 Cron で自動物理削除、UC-140)

  • PII / トークン / カード情報を記録禁止(FR-114)
  • 入力値ログ: 先頭 100 文字 + SHA-256 ハッシュのみ
  • カテゴリ C の 5xx 系エラーは利用者にエラー ID を提示(FR-113)

8. 秘密情報管理(共有概念正本 = メイン側)

8.1 暗号鍵管理(§10.9 から移管)

鍵種別 保管場所 用途 ローテーション周期
Master Key Cloudflare Secrets Store オーナー派生鍵・グローバル派生鍵のベース 年次
オーナー派生鍵 動的生成(HKDF-SHA256, info=owner:<id>) 旧エンドユーザーメール暗号化・トークン HMAC Master Key 更新時に再 KDF

8.2 ローテーション運用

操作 方式
Master Key ローテーション中 60 日 dual-decrypt(旧鍵と新鍵の両方で復号を試行)(NFR-321)、MVP ハードゲート 4-eyes 対象(要件 §6.2.3)

8.2.1 鍵バージョン参照

dual-decrypt 期間中に「どのレコードがどの世代の鍵で暗号化されているか」を明示するため、PII 暗号化列を持つテーブルに鍵バージョンを保持する。

対象テーブル カラム 説明
users email_key_version 管理者(オーナー / プロジェクト管理者 / メンバー)のメールアドレス暗号化バージョン

運用フロー:

  1. INSERT / UPDATE 時:アプリは最新鍵バージョン(Secrets Store の email_key_active_version から取得)で暗号化し、その値を email_key_version に記録する
  2. SELECT 時:復号処理は email_key_version を読み取り、対応する鍵を Secrets Store から取得して復号する。dual-decrypt 期間中は version N と N-1 の両方を保持し、ヒットした方を採用する
  3. ローテーション完了後:バッチで email_key_version < N の行を新鍵で再暗号化(UPDATE … SET email_encrypted=新値, email_key_version=N)
  4. 全行再暗号化完了後:旧鍵を Secrets Store から削除

ローテーション完了の判定:SELECT COUNT(*) FROM users WHERE email_key_version < N が 0 件であること。

例外:contract_owners.contract_status='deleted' または users.valid=0 の論理削除済み行は、90 日経過後の物理削除バッチに任せ、再暗号化対象外とする(復元時の dual-decrypt 整合性を維持するため、ローテーション期間は 60 日 + 物理削除遅延 90 日を考慮した運用窓設計とする)。

8.3 ウィジェット API キー(projects / project_legacy_keys、§8.3.4 から移管)

  • 形式: pk_live_<32-char base62>
  • 有効期限: なし(無期限)。失効は再発行(ローテーション)でのみ行う
  • ローテーション: 旧キー 24 時間併存
  • 旧キー使用検知: SCR-007 に警告バッジ + 件数 + 最終アクセス時刻

11.5 DLQ 運用(§11.9 から移管)

項目 仕様

自動再処理 DLQ 投入から 1 時間以内 は自動指数 BO 再試行、それ以降は運営者の明示的リプレイ操作のみ(SCR-096)

DLQ 監視 DLQ 滞留が §8.7.2 の IF 別上限値を超えた瞬間に運営者 high 通知 + メール
署名検証失敗 即時破棄 + 運営者 high 通知(NFR-310)

13. PII マスキング・プロンプト注入対策

13.1 PII マスキング(§10.7 から移管)

観点 方針
検出器配置 Cloudflare ネットワーク内(Workers AI + 正規表現ルール)、外部 API 送信不可
検出ルール保守 KV に保持、運営者システム(SCR-097)で更新可能

誤検出改善ループ 管理者ユーザーが「誤検出」報告 → SCR-097 に集約 → 検出ルール改善(AC-036、3 営業日以内判定)

失敗時 安全側に倒し未解決登録または安全エラー

### 2.1 retention_class 値の正本定義(`audit_logs.retention_class`)

| 値 | 保持期間 | 用途 | 例 |
|---|---|---|---|
| `general` | **1 年**(NFR-602a)| 業務監査 | 通常の操作 / チャット / FAQ 編集 / ログイン |
| `billing` | **7 年**(NFR-602b、電子帳簿保存法)| 課金・法定保管 | 請求 / 支払 / サブスク / 予算 / `owner.suspend` prefix |
| `operator_high_priv` | **5 年**(NFR-602d、SOX 類似)| 運営者高権限操作 | 復元 / 契約停止 / 物理削除 / AI 変更 / レート上書き / 鍵ローテーション |

詳細は [10_セキュリティ設計書.md §7](10_セキュリティ設計書.md)(運営者側 §7 が正本)を参照。

### 2.3 メンバー権限保持方式(設計判断履歴、§7.1.1 から移管)

採用方式は **プロジェクト単位ロール方式**(`account_project_grants.role`)。`(account_id, project_id)` 複合 PK + `role` 列(`admin` / `member`) + `granted_at` / `granted_by` 保持で行レベル監査を維持。

**v1.10 でオーナー(`accounts.is_owner=1`)も `account_project_grants` に行を持つ**(プロジェクト作成時に `account_project_grants(account_id=オーナー, project_id=新PJ, role='admin', valid=1)` を自動 INSERT)。従来の「オーナーは行を持たない暗黙の admin」設計は廃止した。認可フロー先頭の `isOwner=true` bypass は維持し、`account_project_grants` 行は通知宛先解決(critical 通知)と画面表示(プロジェクト管理者一覧)の事実情報として参照する。MVP 前期に検討した「個別権限フラグ方式(`account_permissions` 別テーブル + `permission_kind` 5 値)」は、運用パターンが固定的なため UI / 認可ロジック双方の複雑さに見合わず廃止した。

**v1.10 で `valid` カラムによる論理削除を全削除対象テーブルに導入**(対象テーブルは §9 削除モード表を正本)。`valid INTEGER NOT NULL DEFAULT 1 CHECK (valid IN (0,1))` を追加し、論理削除は `UPDATE … SET valid=0` で表現する。論理削除データは `updated_at` 基準で 90 日経過後に物理削除バッチで物理削除する([DD14_バッチ・非同期処理.md](../03_詳細設計/DD14_バッチ・非同期処理.md) を正本)。`deleted_at` カラムは追加せず、論理削除日時は既存 `updated_at`(`valid` が 0 に更新された時刻)で代用する。**v1.11 で `accounts` / `projects` / `faqs` / `inquiries` に残置していた `deleted_at` 列を削除し、本ルールに整合させた**(退会日時の証跡は `withdrawal_requests.applied_at`、業務クローズ日時の `inquiries.closed_at` は別概念として維持)。例外テーブル: `audit_logs`(メイン側は v1.11 で API 操作ログ専用化、`created_at + retention 期間` で物理 DELETE。ハッシュチェーン正本は運営者側)、`sessions`(`revoked_at` で代用)、法定保管 7 年テーブル(`billing_invoices` / `usage_metering` / `withdrawal_requests` / `terms_versions` / `terms_agreements`)、全契約横断テーブル(`email_suppression_list`)、ログ系(`notification_logs` / `error_logs` / `inbox_messages`)、トークン / 履歴系(`access_tokens` / `faq_revisions` / `faq_search_fts` / `chat_messages` / `inquiry_status_history` / `question_log_faq_refs` / `chat_room_reminder_history`(v1.11 新設))、M:N 関係系(`announcement_audiences`(v1.11 新設))。

## 3. テーブル詳細

### 3.1 `accounts`(アカウント)

| 項目 | 内容 |
|---|---|
| テーブル名 | `accounts` |
| 論理名 | アカウント |
| 概要 | オーナー(`is_owner=1`)とメンバー(`is_owner=0`)の両方を 1 表に保持 |
| 主キー | `id`(TEXT, ULID) |
| 設計判断 | オーナー行は `owner_account_id = id` で自己参照。**v1.10 でオーナー(`is_owner=1`)も `account_project_grants` に `role='admin'` 行を保持する**(プロジェクト作成時に自動 INSERT。認可上は `isOwner=true` bypass を優先する)。論理削除は `valid` カラムで表現(`valid=0` = 論理削除済、`valid=1` = 有効)。**v1.11 で v1.10 ルール「`deleted_at` は追加しない / 論理削除日時は `updated_at` で代用」に整合させ、本テーブルからも `deleted_at` 列を削除した**。退会日時の証跡は `withdrawal_requests.applied_at` で代替する。`email_key_version` は鍵ローテーション dual-decrypt 期間中の鍵バージョン参照用(v1.11 で新設)|

#### DDL(詳細設計書 行 5827-5859 由来)

```sql
CREATE TABLE accounts (
  id                       TEXT PRIMARY KEY,
  owner_account_id         TEXT NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
  email_encrypted          TEXT NOT NULL,
  email_hmac               TEXT NOT NULL,
  email_key_version        INTEGER NOT NULL DEFAULT 1,   -- v1.11: 鍵ローテーション dual-decrypt 期間中の鍵バージョン
  password_hash            TEXT,
  name                     TEXT,
  role                     TEXT NOT NULL CHECK (role IN ('admin','end_user')),  -- v1.11: 'service_operator' 削除(運営者は別 D1)
  is_owner                 INTEGER NOT NULL DEFAULT 0 CHECK (is_owner IN (0,1)),
  status                   TEXT NOT NULL DEFAULT 'pending_verification'
                           CHECK (status IN ('pending_verification','pending_activation','active')),
  contract_status          TEXT NOT NULL DEFAULT 'active'
                           CHECK (contract_status IN ('active','suspended','deleted_pending','deleted')),
  data_deletion_mode       TEXT NOT NULL DEFAULT 'physical'
                           CHECK (data_deletion_mode IN ('physical','anonymize')),
  valid                    INTEGER NOT NULL DEFAULT 1 CHECK (valid IN (0,1)),
  last_login_at            TEXT,
  email_verified_at        TEXT,
  created_at               TEXT NOT NULL,
  updated_at               TEXT NOT NULL,
  -- v1.11: deleted_at 列を削除(v1.10 ルール「論理削除日時は updated_at で代用」に整合)
  CHECK ((is_owner = 1 AND owner_account_id = id) OR (is_owner = 0 AND owner_account_id <> id))
);
-- v1.10: 同一オーナー配下で valid=1 のメール重複のみ禁止(論理削除済みの valid=0 行は重複許容、同メールでの再招待で新規 accounts 行を作成可能にする)
CREATE UNIQUE INDEX uq_accounts_owner_email ON accounts(owner_account_id, email_hmac) WHERE valid = 1;
CREATE INDEX idx_accounts_email_hmac     ON accounts(email_hmac);
CREATE INDEX idx_accounts_owner_role_status ON accounts(owner_account_id, role, status);
CREATE INDEX idx_accounts_is_owner       ON accounts(is_owner) WHERE is_owner = 1;
CREATE INDEX idx_accounts_valid          ON accounts(valid) WHERE valid = 0;
```

### 3.2 `account_project_grants`(オーナー / メンバープロジェクト割当 + プロジェクト別ロール)

オーナー(`accounts.is_owner=1`)とメンバー(`is_owner=0`)の両方に対し、プロジェクトごとに `admin`(プロジェクト管理者) / `member`(メンバー)のロールを付与する。**v1.10 でオーナーも本テーブルに行を持つ**(プロジェクト作成時に `(account_id=オーナー, project_id=新PJ, role='admin', valid=1)` 行を自動 INSERT)。認可フロー先頭の `isOwner=true` bypass は維持し、本テーブルは通知宛先解決と画面表示に使う。

**割当ゼロ状態の発生条件(v1.7 で FR-015d 不変条件を撤廃 / v1.10 で論理削除化)**: メンバー削除を **アカウント論理削除**(`UPDATE accounts SET valid=0`)に統一し(v1.7 物理削除統一を v1.10 で論理削除へ再統一)、`account_project_grants` 行のみの単独削除(離脱)操作は提供しない。論理削除に伴い `account_project_grants` 行も `valid=0` に同時 UPDATE する。`valid=1` 行が 0 行残った状態は構造的に発生しない(アカウント論理削除と同時に発生するため)。プロジェクト削除起因で `valid=1` 割当が 0 件になるメンバーは同一トランザクション内で `accounts.valid=0` に論理削除する(下記「プロジェクト削除時の孤立メンバー処理」)。最後の admin の有無に関しては、v1.10 でオーナーが常に admin 行を保持するため「プロジェクトに admin 行が 0 件になる」状態は構造的に発生しない(`E-AUTHZ-LAST-ADMIN-PROTECTED` の判定はオーナー行除外、メンバー由来の admin のみで判定)。

#### DDL

```sql
CREATE TABLE account_project_grants (
  account_id  TEXT NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
  project_id  TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  role        TEXT NOT NULL CHECK (role IN ('admin','member')),
  valid       INTEGER NOT NULL DEFAULT 1 CHECK (valid IN (0,1)),
  granted_at  TEXT NOT NULL,
  granted_by  TEXT REFERENCES accounts(id) ON DELETE SET NULL,  -- v1.11: 招待者削除でも整合維持(監査履歴は audit_logs で補完)
  updated_at  TEXT NOT NULL,
  PRIMARY KEY (account_id, project_id)
);
-- v1.10: 同一 (account_id, project_id) ペアで valid=1 行は 1 件のみ、再招待時は valid=0 履歴行を残したまま valid=1 行を上書き(UPSERT)で復活
-- 物理 PK は (account_id, project_id) のままだが、論理削除済の同ペアを保持する目的では新規行を作成せず既存行の valid を 0 ⇔ 1 で切り替える
CREATE INDEX idx_account_project_grants_project ON account_project_grants(project_id);
-- プロジェクト管理者の高速検索用(critical 通知宛先解決でも使用、valid=1 の admin のみ)

  ON account_project_grants(account_id) WHERE role = 'admin' AND valid = 1;
CREATE INDEX idx_account_project_grants_valid ON account_project_grants(valid) WHERE valid = 0;
```

> **PK と再招待**: `(account_id, project_id)` の物理 PK は維持。同じメンバー × 同じプロジェクトの再招待は、既存行の `valid` を `0 → 1` に切り替える UPSERT(`INSERT … ON CONFLICT(account_id, project_id) DO UPDATE SET valid=1, role=?, granted_at=now(), granted_by=?, updated_at=now()`)で表現する。これにより、同一ペアで `valid=1` 行は最大 1 件となり、部分 UNIQUE インデックスは不要。論理削除済み(`valid=0`)行の `granted_at` は最初の招待時刻を維持し、再招待時に上書きしない仕様としても良いが、MVP では復活時に最新値で上書きする(監査ログ側で履歴を担保)。

|---|---|---|
| `admin` | プロジェクト管理者 | FAQ 管理 / 個別チャット対応 / ログ参照 / ユーザー管理(該当プロジェクトへの招待・離脱・ロール変更)|

#### プロジェクト削除時の孤立メンバー処理(アプリ層、v1.10 で論理削除化)

v1.10 でプロジェクト削除と孤立メンバー cleanup を論理削除に統一。次の処理はアプリ層のトランザクション内で明示する:

1. `UPDATE account_project_grants SET valid=0, updated_at=now() WHERE project_id=? AND valid=1`(オーナー自身の admin 行も含む全行)
2. UPDATE 後に「他プロジェクト割当(`valid=1` 行)が 0 件かつ `is_owner=0`」となったメンバー `accounts` 行を抽出して `UPDATE accounts SET valid=0, updated_at=now() WHERE id IN (...)` および全セッション失効(`UPDATE sessions SET revoked_at=now()`)
3. `UPDATE projects SET status='deleted', valid=0, updated_at=now() WHERE id=?`(v1.11: `deleted_at` 列廃止、`updated_at` で削除日時代用)
4. 関連テーブル(`allowed_domains` / `project_ip_allowlist` / `faqs` / `inquiries` / `inquiry_contacts` / `chat_rooms` / `question_logs`)の対象行を `valid=0` に伝播

オーナーは `accounts` 本体の論理削除対象外(`is_owner=1` は常に `valid=1` 維持)。当該プロジェクトに対する `account_project_grants` 行は `valid=0` に論理削除する。論理削除データは `updated_at < now() - 90d AND valid=0` の物理削除バッチ(DD14)で 90 日後に物理削除される。物理削除時に `accounts` の物理 DELETE が走ると `ON DELETE CASCADE` で関連行も連鎖物理削除されることに留意。詳細は [DD02_プロジェクト・FAQ管理.md](../03_詳細設計/DD02_プロジェクト・FAQ管理.md) を参照。

## 7. SaaS データ分離観点

### 7.1 オーナー境界によるデータ分離方式(§2.6 + §7.1 から移管)

- 全契約スコープのテーブルに `owner_account_id` カラム必須(全契約横断テーブルを除く ── §7.3 参照)
- API / 認可レイヤで境界検証: `actor.owner_account_id == target.owner_account_id`(認証・認可設計書 09 §5)
- 違反時は **404 偽装**(403 だと相手契約のリソース存在が漏れる)
- 150 契約到達時に D1 シャーディング設計を着手予定(Future 参照)

### 7.2 オーナー行とメンバー行の判定

- オーナー行: `accounts.is_owner=1` かつ `owner_account_id = id`(自己参照)
- メンバー行: `accounts.is_owner=0` かつ `owner_account_id <> id`
- `accounts` テーブルの CHECK 制約で強制: `((is_owner = 1 AND owner_account_id = id) OR (is_owner = 0 AND owner_account_id <> id))`

### 7.3 全契約横断テーブル(`owner_account_id` を持たない)

| テーブル名 | 理由 |
|---|---|
| `email_suppression_list` | メール bounce / 苦情の全契約横断管理(R-003 共通ドメイン対策)|
| `terms_versions` | 規約 / プライバシーポリシーの版(全利用者共通)|
| `service_announcements` | 運営者発信のお知らせ本体(配信先は `announcement_recipients` で契約別)|

## 9. データ削除モード(§7.8 から移管、v1.10 で論理削除モード追加)

**設定**: `accounts.data_deletion_mode`(`physical` / `anonymize`、初期値 `physical`)で保持(v1.6 で `notification_preferences` JSON 内から独立カラムに昇格)。

v1.10 で **第 3 のモード「論理削除」** を導入。論理削除は `UPDATE … SET valid=0` で行い、90 日経過後の物理削除バッチ(DD14)で物理削除モード(`physical` / `anonymize`)に従って実体を消去する。SCR-017-M1 のメンバー削除と SCR-026 のプロジェクト削除は常に論理削除を経由する(v1.10)。

### 9.1 削除モード対応表(v1.10 改訂)

| 対象テーブル | 論理削除(v1.10 新設)| 物理削除(90 日後バッチ or 即時、`data_deletion_mode='physical'`)| 匿名化(`data_deletion_mode='anonymize'`)|
|---|---|---|---|
| `accounts` | `valid=0` | DELETE(ON DELETE CASCADE で関連行連鎖)| `email_encrypted` → `anon_<sha256>`, `name` → 「匿名」 |

| `projects` | `status='deleted', valid=0` | DELETE(関連 `faqs` / `inquiries` 等は CASCADE 経由)| `name` → 「削除済みプロジェクト」 |
| `allowed_domains` | `valid=0` | DELETE | (匿名化対象外)|
| `project_ip_allowlist` | `valid=0` | DELETE | (匿名化対象外)|
| `faqs` | `status='deleted', valid=0` | DELETE | `title` / `body` 空字 |
| `question_logs` | `valid=0` | DELETE | `user_question` 空字, `ip_address` / `user_agent` NULL, 集計値維持 |

### 9.2 論理削除の物理削除バッチ(v1.10、DD14 正本)

- 日次バッチで `valid=0 AND updated_at < now() - 90d` の行を物理削除モード(`accounts.data_deletion_mode` 準拠)に従って処理
- `accounts` 行が物理 DELETE された時点で `ON DELETE CASCADE` で関連 `account_project_grants` / `sessions` 等が連鎖物理削除
- `projects` 行が物理 DELETE された時点で `ON DELETE CASCADE` で関連 `faqs` / `inquiries` 等が連鎖物理削除
- `billing_subscriptions` は物理削除バッチの対象外(永久保持)

## 10. データ保持の調整制約(NFR-702)

- **データ保持期間はシステム固定値とし、契約・管理者ユーザーによる UI 上の変更は不可**(短縮・長期化いずれも不可)
- 固定値は本書 §2 のテーブル一覧「保持期間」列および要件定義 NFR-701 / NFR-705 を正本とする
- 短縮が必要な場合(法令・契約義務範囲内)はサポート窓口経由の運営者対応とし、長期化が必要な場合は将来要件(FUT01_認証・セキュリティ強化 / FUT 配下)で再検討する
- SCR-028 アカウント設定にデータ保持期間設定 UI を持たない(基本設計 01_画面設計.md §5.20b 参照。SCR-016 は v1.5 で SCR-028 に統合・廃止)
- `usage_metering` / `billing_invoices` / `audit_logs(retention_class='billing')` は電子帳簿保存法・国税法の要請により最低 7 年保持(NFR-602b)、これを下回る短縮は不可

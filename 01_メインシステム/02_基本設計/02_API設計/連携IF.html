連携 IF

このページは、顧客管理システムとの連携 IF #1〜#12 のうちメインが受信・送信するエンドポイントの契約を定義します。

このページの API

  • API-LNK-001 IF #1 契約停止イベント受信(顧管 → メ)
  • API-LNK-002 IF #2 強制ログアウト受信
  • API-LNK-003 IF #4 復元実行受信
  • API-LNK-004 IF #5 レート制限上書き受信
  • API-LNK-005 IF #5(月次上限件数上書き受信 / プロジェクト単位)
  • API-LNK-006 IF #6 AI しきい値上書き受信
  • API-LNK-007 IF #7 お知らせ生成受信
  • API-LNK-008 IF #8 監視メトリクス取得(送信側、メ → 顧管)
  • API-LNK-009 IF #9 不正利用検知通知(送信側、メ → 顧管)
  • API-LNK-010 IF #10 課金 Webhook 受信(顧管 → メ)
  • API-LNK-011 IF #12 運営者操作通知受信

API-LNK-001 IF #1 契約停止イベント受信(顧管 → メ)

顧客管理システムからの契約停止イベントを受信し、契約状態とセッションを反映します。

POST /internal/admin-integration/v1/owner/suspend

認証: mTLS + JWT(aud=main、TTL 5min)。

リクエストパラメータ

Query Parameters

{}

Path Parameters

{}

Request Body

{
  "operationId": "01HZ...",
  "contractOwnerUserId": "01HZ...",
  "targetStatus": "suspended",
  "reason": "payment_failed|terms_violation|manual_admin",
  "occurredAt": "2026-05-13T10:00:00Z",
  "operatorAccountId": "..."
}

ヘッダ: Idempotency-Key: (operation_id, contract_owner_user_id, target_status)。

レスポンス(200)

{ "ok": true, "newStatus": "suspended", "appliedAt": "..." }

動作:

  1. JWT + Idempotency-Key 検証
  2. contract_owners.contract_status 更新
  3. KV キャッシュ無効化(セッション/KV/レート制限)
  4. reason に応じてセッション処理(operator_manual / tos_violation は 5 秒以内 に全セッション無効化)

エラー

HTTP ステータスエラーコード内容
401JWT_INVALIDJWT 検証失敗
409CONFLICT冪等性競合

API-LNK-004 IF #5 レート制限上書き受信

契約単位のレート制限上書きを受信し、即時反映します。

POST /internal/admin-integration/v1/rate-limit/override

リクエストパラメータ

Query Parameters

{}

Path Parameters

{}

Request Body

{
  "contractOwnerUserId": "...",
  "overrideId": "...",
  "limits": [{ "kind": "ask", "limit": 30, "windowSec": 60, "validUntil": "..." }],
  "reason": "...",
  "operatorAccountId": "..."
}

動作:

  1. JWT + Idempotency-Key 検証
  2. owner_quota_overrides テーブル更新(レート種別のみ。契約単位)
  3. KV キー ratelimit:{contractOwnerUserId}:{kind} 更新(即時反映)
補足本 IF はレート制限(契約単位)専用です。月次上限件数の運営者上書きはプロジェクト単位のため API-055 を用います。

API-LNK-005 IF #5(月次上限件数上書き受信 / プロジェクト単位)

プロジェクト単位の月次上限件数上書き(4-eyes 承認済み)を受信し、即時反映します。

POST /internal/admin-integration/v1/usage-limit/override

リクエストパラメータ

Query Parameters

{}

Path Parameters

{}

Request Body

{
  "contractOwnerUserId": "...",
  "projectId": "prj_...",
  "overrideId": "...",
  "limits": [{ "kind": "question_monthly_limit", "limit": 8000, "freeQuota": 2000, "validUntil": "..." }],
  "reason": "...",
  "operatorAccountId": "..."
}

動作:

  1. JWT + Idempotency-Key 検証(4-eyes 承認済みであること)
  2. project_quota_limits テーブル更新(source='operator'、project_id で対象プロジェクトを特定、reason / valid_until / created_by を記録)
  3. KV キー usage-limit:{contractOwnerUserId}:{projectId} 更新(30s TTL、即時反映)

kind は question_monthly_limit / faq_monthly_limit。質問数は limit と freeQuota、FAQ は運営者が管理する freeQuota だけを受け付ける。FAQ の limit は常に NULL とし、指定時は 422。質問数の優先順位は「プロジェクト設定 > 運営者上書き > デフォルト推奨値」、FAQ 無料枠は「運営者上書き > デフォルト値」。

API-LNK-007 IF #7 お知らせ生成受信

運営者が生成したお知らせを受信し、配信先を解決して inbox へ fan-out します。

POST /internal/admin-integration/v1/announcement/inbound

リクエストパラメータ

Query Parameters

{}

Path Parameters

{}

Request Body

{
  "announcementId": "...",
  "contractOwnerUserId": null,
  "title": "...",
  "bodyHtml": "<p>...</p>",
  "importance": "low|normal|high|critical",
  "audienceContractOwnerUserIds": null,
  "publishedAt": "...",
  "operatorAccountId": "..."
}
  • audienceContractOwnerUserIds:
    • null または空配列 → 全契約配信(contract_owners.contract_status='active' AND valid=1 を動的解決)
    • 配列 → 列挙された contract_owner_user_id への限定配信(各要素は ULID 形式)

動作:

  1. JWT + Idempotency-Key 検証
  2. service_announcements レコード作成
  3. audienceContractOwnerUserIds が配列の場合、各要素に対し announcement_audiences(announcement_id, contract_owner_user_id) 行を INSERT(配信先は中間テーブル化されており、service_announcements 側に JSON 列は持たない)
  4. announcement_recipients を宛先範囲に応じて生成(announcement_audiences 行が 0 件なら全契約、1 件以上なら列挙された owner のみ)
  5. announcement-fanout-queue に投入、inbox_messages へ fan-out

API-LNK-010 IF #10 課金 Webhook 受信(顧管 → メ)

運営者が一次受信した Stripe 課金イベントを受信し、請求 / 契約状態へ反映します(FR-139)。

POST /internal/admin-integration/v1/billing-webhook/forward

リクエストパラメータ

Query Parameters

{}

Path Parameters

{}

Request Body

{
  "stripeEventId": "evt_...",
  "eventType": "invoice.paid|invoice.payment_failed|customer.subscription.created|...",
  "contractOwnerUserId": "...",
  "payload": { /* Stripe イベント */ },
  "receivedAt": "...",
  "signatureVerifiedAt": "..."
}

動作:

  1. JWT 検証 + event_id 冪等性
  2. Stripe イベント種別に応じてハンドラ分岐:
    • invoice.paid: billing_invoices.status='paid'、suspended → active 復帰
    • invoice.payment_failed: 通知、7 日猶予タイマー開始
    • customer.subscription.created: billing_subscriptions 作成
    • charge.refunded: 訂正請求行記録
  3. IF #1 経由で contract_owners.contract_status 反映

Idempotency-Key: event_id(Stripe evt_*)。DLQ 滞留上限: 1000/24h。

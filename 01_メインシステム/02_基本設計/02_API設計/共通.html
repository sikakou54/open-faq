2. 共通仕様

全 API に共通する形式・認証ヘッダ・冪等性・論理削除フィルタ・Webhook 規約をここで一括定義します。各エンドポイントの個別仕様(§4)は本節を前提とし、差分のみを記載します。

2.1 API 共通仕様

Base URL・データ形式・ID 形式・ページング方式など、全 API が従う基本ルールです。

項目値
Base URL/api/v1(管理)/ /widget/v1(ウィジェット)
データ形式JSON
エラー形式RFC 7807(application/problem+json)
日時ISO 8601 + 末尾 Z(UTC)
ID 形式ULID(26 文字)、Stripe ID 例外
ページングカーソル方式(cursor, limit 50〜200)
CSRFDouble Submit Cookie 検証(状態変更 API)
CookieSecure; HttpOnly; SameSite=Lax; Path=/

2.2 オーナー境界判定

全操作で契約境界を強制し、違反時は存在を秘匿します。

すべての操作で actor.owner_id == target.owner_id を必須とする(正本: 08_認証・認可設計.html §5.2)。違反時は 404 偽装で応答する(正本: 05_エラー設計.html §5.3)。

2.3 論理削除フィルタ

valid カラムを持つテーブルへの GET 系 API は論理削除済みの行を返しません。

valid カラムを持つテーブル(M_OWNERS / M_PRJ_USERS / M_PRJ_USER_ASGN / M_PROJECTS / M_ALLOWED_DOMAINS / M_FAQS / H_QUESTION_LOGS / T_INQUIRIES / T_BILL_SUBS)に対する全 GET 系 API(一覧 / 詳細)のクエリは、原則として WHERE <table>.valid=1 フィルタを追加する。論理削除済み(valid=0)の行は通常 API 経由では返却しない。

例外:

  • 監査ログ参照系 API は valid 概念外(H_AUDIT_LOGS には valid カラムを追加しない)
  • 認証ミドルウェアは、セッションの主体(actor_type='owner' / 'project_user')に応じて M_OWNERS.valid=0 / M_PRJ_USERS.valid=0 の行へのログインを 401 で拒否する(セッション内 actor_id の valid を確認、valid=0 ならセッションも失効扱い)

WHERE valid=1 フィルタの漏れは IDOR 類似の脆弱性として扱う(09_セキュリティ設計.html)。

2.4 冪等性キー

書き込み系 API はリトライ安全性のため冪等性キーを受け付けます。

書き込み系 API は Idempotency-Key ヘッダ(ULID 推奨、24h 保管)を受け付ける。重複時は既存処理結果を 200 で返却(リトライ安全性)。

2.5 Webhook 共通仕様

受信 Webhook の署名検証・冪等性・リトライ・DLQ の共通方針です。

項目仕様
署名検証HMAC-SHA256(送信元との共有 Secret)
冪等性event_id で重複処理防止
リトライ3 回(指数 BO: 1m → 4m → 16m)、超過で DLQ
DLQ 保持4 日(Cloudflare Queues)、その後 R2 退避 30 日
タイムアウト30 秒(超過で 5xx 返却)

2.6 認証ヘッダ

API カテゴリごとに用いる認証ヘッダの一覧です。

ヘッダ用途例

Authorization: Bearer <wst_*>ウィジェットセッションウィジェット API

5. 外部 Webhook

メール配信プロバイダからの配信状態 Webhook を受信し、配信状態の更新と抑制リスト登録を行います。署名検証と冪等性は §2.5 の共通仕様に従います。

受信 Webhook(Resend)の詳細仕様は API-WHK-001 外部 Webhook(Resend) を参照。

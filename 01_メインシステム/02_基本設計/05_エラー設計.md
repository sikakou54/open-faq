## 2. エラー分類

| エラー分類 | 概要 | プレフィックス | 代表例 |
|---|---|---|---|
| 入力エラー | バリデーション失敗 | `E-INPUT-*` | 必須項目欠落 / 形式不正 / 重複 |
| 認証エラー | 認証失敗・期限切れ | `E-AUTH-*` | パスワード不一致 / ロックアウト / トークン期限切れ |
| 認可エラー | 権限不足 | `E-AUTHZ-*` | プロジェクトロール不足 / オーナー境界違反 / オーナー専有機能 / 最後の管理者保護 |
| 業務エラー | ビジネスルール違反 | `E-BIZ-*` | FAQ 自動公開禁止 / `case_status=closed` 自動遷移禁止 / 退会済み |
| 契約上限エラー | プラン / 契約上限到達 | `E-QUOTA-*` | 月次無料枠 100% / 125% 超過 |
| 課金エラー | 決済 / 課金処理エラー | `E-BILL-*` | Stripe 連携失敗 / 支払い方法未登録 / サスペンション中 |
| システムエラー | サーバー内部エラー | `E-SYS-*` | DB エラー / 内部例外 / OOM |
| 外部連携エラー | 外部 API エラー | `E-EXT-*` | Workers AI 推論失敗 / Resend 配信失敗 |
| 連携 IF エラー | 連携 IF #1〜#12 エラー | `E-IF-NNN-*` | IF 受信時の冪等性違反 / 認証失敗 |
| Webhook エラー | 外部 Webhook(Resend など)受信エラー | `E-WH-*` | 署名検証失敗 / 冪等性違反 |
| 規約エラー | 規約・プライバシー関連 | `E-LEGAL-*` | 規約未同意状態でのアクセス |

## 3. 主要設計原則(エラー分類 A〜D、§6.2.13 から移管)

| カテゴリ | 自動リトライ | 利用者提示 | HTTP | 未解決登録 |
|---|---|---|---|---|
| **A. 透過再試行可** | ◯ 指数 BO 3 回 | 「しばらくしてから再度お試しください」 | 200(成功)/ 503(失敗)| しない |
| **B. ユーザー再試行** | × | 「再試行してください」+ 待機時間表示 | 429 / 423 | しない |
| **C. 即時 fail-fast** | × | エラー固有メッセージ + 是正案内 | 400 / 401 / 403 / 404 / 409 / 422 | しない |
| **D. 業務分岐(正常)** | (該当なし) | 「個別チャットへ誘導」 | 200 | **する** |

## 5. 異常系共通方針

### 5.1 入力エラー時の方針

- 必須項目欠落 / 形式不正は **400 + フィールド単位のエラー配列** を返却
- メッセージは画面側で項目近くにインライン表示
- 文字数 / 範囲チェックは Zod スキーマ(API 設計書 §3 参照)

### 5.2 認証エラー時の方針

- パスワード不一致 / トークン期限切れは **401 統一**
- ログイン失敗 5 回で `(IP × user_id)` を **15 分ロックアウト**(基本設計書 §10.1.1 / FR-007 / FR-177)
- ロックアウト解除は時間経過のみ(管理者解除なし)
- 攻撃者へのヒント禁止(メール / パスワードのどちらが誤りかを示さない)

### 5.3 認可エラー時の方針

- プロジェクトロール不足は **403 統一**
- **オーナー境界違反は 404 偽装**(403 だと相手契約のリソース存在が漏れる)
- レスポンスボディには具体的な権限名を含めない(列挙攻撃対策)
- AC-017(クロス契約アクセス検知)は **月次自動検証**

### 5.4 業務エラー時の方針

- FAQ 自動公開禁止(基本設計書 §4.2.1)/ `case_status=closed` の自動遷移禁止(FR-079)は **422 + 専用エラーコード**
- ステータス遷移可否は API 側で必ず検証(画面側だけの判定に頼らない)

### 5.5 契約上限エラー時の方針

- 月次無料枠 100% / 125% 超過は **429 + `Retry-After`**(秒数 = 月末までの秒数)
- **質問数のみ拒否しない**(基本設計書 §6.6.2 / FR-122 = 事後課金、エンドユーザー体験を優先)
- 同時接続上限は **503 + `Retry-After`**(NFR-114)

### 5.6 課金エラー時の方針

- 契約停止中(`contract_status=suspended`)は **403 + 専用コード** `E-BILL-CONTRACT-SUSPENDED`
- 課金画面 / 退会画面 / データエクスポート画面のみ許可(基本設計書 §6.2.7 / FR-124)
- 支払い失敗 → 7 日猶予期間カウント開始 + メール通知(`critical`)

### 5.7 外部連携エラー時の方針

- **Workers AI 推論失敗**: 質問数は計測するが `metering_billable=false`(基本設計書 §6.6.1)
- **Resend 配信失敗**: `notification_logs.delivery_state='failed'` → DLQ 投入 + 自動 BO 3 回(最大 24h)
- **メール bounce**: `email_suppression_list` 即時追加(全契約横断、R-003 共通ドメイン対策)
- **メール苦情(complained)**: 永久サプレス、**復帰不可**

### 5.8 連携 IF エラー時の方針

- 冪等性違反: **200 + 既存処理結果** を返却(リトライ安全性)
- 認証失敗(mTLS / JWT): 401 統一、署名検証失敗は別ログ出力
- JSON Schema 不整合: 400 + 監査記録

### 5.9 Webhook エラー時の方針

- 署名検証失敗: **401 + 監査記録 + 運営者 `high` alert** + 即時破棄
- 冪等性違反: 200 + 既存処理結果
- ペイロード差分検出: 200 + `webhook_payload_diffs` INSERT + **自動上書き禁止**(SCR-099 レビュー待ち)

### 5.10 システムエラー時の方針

- 500 統一、内部詳細は応答に含めない
- **エラー ID を利用者提示**(FR-113、カテゴリ C の 5xx 系)
- `error_logs` テーブルに記録(180 日保持、JST 05:00 Cron で自動物理削除)
- 監査が必要な操作の途中で発生した場合は `audit_logs` にも記録
- 再試行ボタン 3 秒クールダウン(FR-112)

### 5.11 不正利用検知エラー時の方針(FR-195)

- 不正利用検知パターン(FR-195 / NFR-308)に該当 → IF #9 送出 + 運営者通知
- 閾値超過時は自動 suspension + 管理者 + email 通知

## 7. `error_logs` テーブル設計・運用

| 項目 | 仕様 |
|---|---|
| 保持期間 | **180 日**(NFR-704)|
| 削除スケジュール | JST 05:00 Cron で自動物理削除(UC-140) |
| 記録内容 | エラーコード / エラーメッセージ / スタックトレース |
| **除外項目** | **個人情報(PII)/ 認証トークン / パスワードハッシュ**(FR-114)|
| 入力値ログ | 先頭 100 文字 + SHA-256 ハッシュのみ記録 |
| エラー ID 提示 | カテゴリ C の 5xx 系エラーは利用者に提示(FR-113)|
| 再試行防止 | 再試行ボタン 3 秒クールダウン(FR-112)|

ER 図の見方

  • 掲載カラム: MVPの主要キー(PK / FK)+ 業務キー(inquiry_code / widget_public_key 等)+ 状態カラム(T_INQUIRIES.status / contract_status 等)+ 特徴的なカラム(actor_user_id 等)。
  • 省略カラム: created_at / updated_at / valid / *_encrypted / *_key_version / metadata 等の共通技術カラム。完全なカラム定義は 03_テーブル設計/index.html §3 を参照。
  • 関係性: 実線 = FK 制約あり / 注記 FK制約なし = 論理参照のみ。本図は FK 制約に基づく関係のみを線で描く。アクター横断参照(作成者 / 更新者 / 監査アクター / セッション主体 / 受信者)は M_OWNERS と M_PRJ_USERS の 2 マスタに分かれるため物理 FK を張らず、actor_type(owner / project_user)+ actor_id の論理参照で表現する(線を引かない)。T_ANNOUNCE_RCPT → T_INBOX_MSG の受信箱 fan-out と T_BILL_SUBS → T_BILL_INVOICES の請求書生成も FK 制約を持たないアプリ層の論理 fan-out のため線を引かない(T_INBOX_MSG / T_BILL_INVOICES はいずれも M_OWNERS に FK で帰属)。
  • 主体モデル(2 マスタ完全分離): M_OWNERS(オーナーの認証属性 + 契約属性〔contract_status / data_deletion_mode〕を統合した契約マスタ。M_OWNERS.id 自体が契約境界キー本体)+ M_PRJ_USERS(プロジェクトユーザーの認証マスタ。オーナーを含まず、メール一意空間もオーナーと完全分離)。プロジェクトへの割当・ロールは M_PRJ_USER_ASGN(role = admin / member)で表現する。ウィジェット利用者(end_users)は永続アカウントを持つ別マスタ。
  • SaaS 境界キー: owner_id(全契約スコープのテーブルに冗長保持し M_OWNERS.id を参照、認可境界クエリ高速化)。

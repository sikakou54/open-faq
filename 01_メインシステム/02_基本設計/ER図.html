ER 図の見方

  • 掲載カラム: MVPの主要キー(PK / FK)+ 業務キー(inquiry_code / widget_public_key 等)+ 状態カラム(inquiries.status / contract_status 等)+ 特徴的なカラム(actor_user_id 等)。
  • 省略カラム: created_at / updated_at / valid / *_encrypted / *_key_version / metadata 等の共通技術カラム。完全なカラム定義は 03_テーブル設計/index.html §3 を参照。
  • 関係性: 実線 = FK 制約あり / 注記 FK制約なし = 論理参照のみ。本図は FK 制約に基づく関係のみを線で描く。アクター横断参照(作成者 / 更新者 / 監査アクター / セッション主体 / 受信者)は owners と project_users の 2 マスタに分かれるため物理 FK を張らず、actor_type(owner / project_user)+ actor_id の論理参照で表現する(線を引かない)。announcement_recipients → inbox_messages の受信箱 fan-out と billing_subscriptions → billing_invoices の請求書生成も FK 制約を持たないアプリ層の論理 fan-out のため線を引かない(inbox_messages / billing_invoices はいずれも owners に FK で帰属)。
  • 主体モデル(2 マスタ完全分離): owners(オーナーの認証属性 + 契約属性〔contract_status / data_deletion_mode〕を統合した契約マスタ。owners.id 自体が契約境界キー本体)+ project_users(プロジェクトユーザーの認証マスタ。オーナーを含まず、メール一意空間もオーナーと完全分離)。プロジェクトへの割当・ロールは project_user_assignments(role = admin / member)で表現する。ウィジェット利用者(end_users)は永続アカウントを持つ別マスタ。
  • SaaS 境界キー: contract_owner_user_id(全契約スコープのテーブルに冗長保持し owners.id を参照、認可境界クエリ高速化)。

課金・請求設計書(メイン)

1. 文書概要

1.1 目的

メインシステムの課金・契約・請求・決済を一元化する。MVP の課金モデル(完全従量課金 + 月次無料枠、無料トライアル期間なし)、契約状態、質問数の月次上限、課金対象別無料枠、決済フローを定義する。質問数だけ月次上限を持ち、FAQ 件数は上限なしで計測・課金を継続する。

1.2 対象範囲

  • 対象: 利用者向け課金 UI(SCR-016 利用状況 / SCR-022 請求 / SCR-023 設定 / SCR-014)、契約状態管理、月次利用量計測、Stripe 連携、請求書管理
  • 対象外: Stripe Webhook の一次受信・DLQ・リプレイ(02_運営者システム/個別設計書群/11_課金請求設計書.html §13 正本)

2. 契約モデル(MVP は単一プラン、共有概念正本)

2.1 単一プラン定義

MVP は Free / Basic / Pro / Enterprise の階層プランを採用しない。以下の単一モデルを全契約に適用する。

項目 MVP 仕様 正本
課金モデル 完全従量課金制 + 月次無料枠 要件 §8.11.1 / 基本設計書 §6.6
契約開始 新規登録完了時点で active(無料トライアル期間なし)。支払方法登録は任意、無料枠超過時にウィジェット質問受付を停止して必須化(支払方法ゲート、FR-136。契約は active 維持) §5.1

2.2 階層プランへの拡張余地

3. レート制限上書き(契約単位)

運営者が契約単位で owner_quota_overrides テーブルを操作する(SCR-093、4-eyes 承認)。DDoS / Bot / 暴走対策のセキュリティ施策であり、レート種別(*_per_*)のみを resource_kind で扱う(月次上限件数は §3a の project_quota_limits へ分離)。

resource_kind 用途 主管 SCR(運営者) KV ミラー
widget_ask_per_min ウィジェット質問レート SCR-093 あり(60s TTL)
widget_inquiry_per_min ウィジェット未解決質問送信レート SCR-093 あり
email_per_hour メール送信レート SCR-093 あり
admin_api_per_min 管理 API レート SCR-093 あり

詳細は 03_テーブル設計.html §3.23 参照。

3a. 質問数上限・課金対象別無料枠(プロジェクト単位)

質問数の月次上限件数・上限アラートと、MVP課金対象の無料枠を プロジェクト単位 で project_quota_limits に保持する。質問数は threshold に値がある場合を上限ON、NULLを上限OFFとして扱う。管理者ユーザー向け SCR-021 / SCR-021-M1 が表示・変更するのは質問数の上限とアラートだけとし、無料利用枠は表示・変更しない。無料枠は運営者向け SCR-093 で管理し、FAQ 件数の threshold は NULL 固定で月次上限なしとする。

resource_kind 用途 主管 SCR KV ミラー
question_monthly_limit 質問数の月次上限件数・無料枠 SCR-021(オーナー / PJ admin)/ SCR-093(運営者プロジェクト上書き) あり(30s TTL)
faq_monthly_limit FAQ 件数の無料枠(上限なし) SCR-093 あり

質問数の優先順位は「プロジェクト設定(source='owner')> 運営者上書き(source='operator')> デフォルト推奨値」。FAQ無料枠の優先順位は「運営者上書き(source='operator')> デフォルト値」とし、source='owner' 行は作成しない。デフォルト値は KV usage-limit:default:{kind} / usage-limit:free-default:{kind} で解決する。詳細は 03_テーブル設計.html §3.24 参照。

5. 契約ライフサイクル

5.1 契約開始フロー(無料トライアル期間なし)

イベント 変更前状態 変更後状態 トリガー 関連API
契約開始 (新規) active SCR-002 新規登録完了 POST /accounts
支払方法登録(任意) active active(支払方法登録済) 支払方法登録 + Setup Intent 成功 POST /billing/payment-method
  • 無料トライアル期間は設けない。新規登録完了時点で契約は即 active となり、月次無料枠の範囲を無料で利用できる。
  • 支払方法の登録は任意。未登録でも無料枠内の利用およびダッシュボード閲覧は可能。
  • 支払方法ゲート(FR-136): 無料利用枠80%到達時に支払方法登録が必要になる旨を SCR-016 / SCR-022 のバナー + 受信箱 / メールで事前告知する。
  • 支払方法未登録で質問数無料枠を超過した場合、契約状態は active のまま、ウィジェットの新規質問受付のみを停止する。FAQ 件数は無料枠超過や月次件数を理由に新規作成を拒否しない。
  • 支払方法登録済の契約は、全課金対象の無料枠超過分を事後課金とする。
  • ウィジェット質問受付停止は支払方法登録(Setup Intent 成功)で即時再開する。以降の無料枠超過分は事後課金となる。

5.2 サスペンションフロー(§6.2.7 から移管)

起点イベント:

  • 支払い失敗 連携 IF #1 受信 → reason=payment_failure_grace_expired(7 日猶予タイマー開始)
  • 運営者手動停止 → reason=operator_manual / tos_violation(即時 suspended)

支払方法ゲート(FR-136)は契約サスペンションではない。支払方法未登録の契約が無料枠を超過した場合は契約 active のままウィジェット質問受付のみを停止する(§8.2 / DD12 §3.7a)。本フローの対象外。

認可ゲート(contract_owners.contract_status=suspended 中):

  • 許可: 課金画面 / 退会画面のみアクセス可
  • その他: 403
  • ウィジェット応答: 423(FR-138)

解除条件:

  • payment_failure_grace_expired / operator_manual / tos_violation: 顧客管理側の連携 IF #1 受信(支払い成功 / 運営者解除)で contract_owners.contract_status='active' へ

5.3 退会・解約フロー(§6.2.8 から移管)

admin が SCR-014 で退会申請
  → 再認証 FR-005
  → contract_owners.contract_status='deleted_pending'
  → ウィジェット応答停止、新規 FAQ 登録不可
  → 当月末まで利用可
  → 月末締めで利用停止(月割りなし)
  → 30 日削除猶予開始(物理削除まで)
  → 物理削除バッチ実行: contract_owners.contract_status='deleted' + contract_owners.valid=0 + users.valid=0(同時セット)
  → 全データ削除 + announcement_recipients 削除

解約 vs 退会 解約申請(サブスクリプション終了)と退会(契約削除)は別フロー

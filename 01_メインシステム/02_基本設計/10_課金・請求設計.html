課金・請求設計(メイン)

このページは、メインシステムの課金ロジック・契約状態遷移・利用量集計方針・課金プロバイダ Webhook 受信方針を定義します。

版数 v2.1更新 2026-06-17承認済

1. 課金モデルと判定単位

メインシステムは完全従量課金 + 月次無料枠の単一モデルを全契約に適用する。無料枠・月次上限はプロジェクト単位、レート制限は契約単位で判定し、計測はプロジェクト単位・請求集計は契約単位とする。階層プランは MVP では持たない。

観点 方針 判定 / 集計単位
課金モデル 完全従量課金 + 月次無料枠(無料トライアルなし) —
月次境界 JST 暦月。無料枠は毎月 1 日 00:00 JST にリセット(BR-086) —
無料枠 / 月次上限件数 project_quota_limits で保持(03_テーブル設計.html §3.24 正本) プロジェクト
レート制限(DDoS / Bot / 暴走対策) owner_quota_overrides で保持(03_テーブル設計.html §3.23 正本)。上限件数・無料枠とは別軸 契約
利用量計測 / 請求集計 usage_metering をプロジェクト単位で計測し、契約単位で集計して請求(03_テーブル設計.html §3.22 正本) 計測=プロジェクト / 請求=契約
テーブル定義は 03 が正本billing_subscriptions / billing_invoices / usage_metering / owner_quota_overrides / project_quota_limits のカラム・制約・コード値は 03_テーブル設計.html §3 を正本とする。本ページは判定ロジックと状態遷移のみを扱う。

1.1 課金対象と無料枠初期値

課金対象は質問数・FAQ 件数・AI 利用コストの 3 種。質問数のみが月次上限件数による停止対象で、FAQ 件数は上限なしで従量課金を継続する。AI 利用コストは MVP ではサービス側が吸収する社内可視化指標とする。無料枠・単価の初期値は要件(BR-086)を正本とし、運営者が料金設定として管理する。

課金対象 計測単位 計測タイミング 失敗時の扱い 上限件数判定
質問数 ウィジェット利用者が「送信」した 1 リクエスト リクエスト到達時(同期) AI 推論失敗時もカウントするが計測フラグで課金対象外として区別 対象(月次上限件数・100% 停止)
FAQ 件数 公開状態の FAQ 件数(下書き・非公開・削除除外) 月末 23:59:59 JST スナップショット — 対象外(上限なし・従量課金のみ継続)
AI 利用コスト(原価) 推論呼び出しのトークン換算(社内可視化用) 呼び出し成功時のみ 失敗は計測のみ・課金対象外。再試行は最初の 1 回のみ計測 対象外(契約単位の社内指標)
無料枠と上限件数は別の境界無料枠は課金が始まる課金境界、月次上限件数は質問数の受付停止境界である。FAQ 件数は無料枠を超えても受付を止めず、超過分を従量課金する。無料枠・単価の初期値(質問数・FAQ 件数・AI コスト)は FR09 BR-086 を正本とし、本ページでは再掲しない。

1.2 上限・無料枠設定の優先順位

プロジェクトの質問数上限・アラートはオーナー / 当該プロジェクト管理者が設定し、運営者がプロジェクト単位で上書きできる。優先順位はプロジェクト設定 > 運営者上書き > デフォルト推奨値とする。管理者ユーザー(SCR-021 / SCR-021-001)が触れるのは質問数の上限とアラートだけで、無料枠は運営者(SCR-093)が管理する。

設定対象 設定主体 優先順位 主管画面
質問数の月次上限件数・アラート閾値 オーナー / 当該プロジェクト管理者 プロジェクト設定 > 運営者上書き > デフォルト推奨値 SCR-021 / SCR-021-001 / 運営者 SCR-093
課金対象別の無料枠 運営者(料金設定) 運営者上書き > デフォルト値(プロジェクト設定行は作らない) 運営者 SCR-093

質問数は月次上限件数を設定した場合を上限オン、設定しない場合を上限オフとして扱う。FAQ 件数は上限・アラートの対象とせず、管理者ユーザー向けの設定も提供しない。上限オフ時は利用率の算出・アラート・上限到達停止を行わない。

2. 質問数上限とウィジェット受付停止

プロジェクトの質問数上限がオンのとき、利用が上限件数の 100% に達した時点で、支払方法の有無に関わらず当該プロジェクトのウィジェット新規質問受付を停止する。契約は active のまま、ウィジェット利用者には安全な定型文を表示し送信を無効化する。125% は集計遅延・誤差対策の最終ガードとして運営者へ追加通知する。

到達点(上限件数比) システム動作 支払方法の有無
選択アラート閾値(25 / 50 / 80 / 90 / 100%) 当月初回到達時に管理者向けアラートメールを送信(§3) 無関係
100% 当該プロジェクトのウィジェット新規質問受付を停止。ウィジェット利用者へプロジェクト連絡先メール誘導の定型文をチャット内システム返信として表示し、入力・送信を無効化。契約は active を維持 有無を問わず停止
125% 既に停止中。集計遅延・誤差の最終ガードとして運営者へエスカレーション通知 無関係

復帰は次のいずれかで即時とする。

  1. 翌月 1 日の月次リセット
  2. 質問数上限の引き上げ(SCR-021-001)
  3. 質問数上限のオフ
支払方法ゲートとは別経路支払方法未登録の契約でプロジェクトが無料枠を超過した場合も同様にウィジェット受付を停止するが、これは上限件数 100% 停止ではなく支払方法ゲート(§4)による。両者ともウィジェット受付のみを止め、契約はサスペンションしない。ウィジェット応答 HTTP ステータス・エラー ID(E-QUOTA-* / E-BILL-*)は 05_エラー設計.html §4.5 / §4.6 を正本とする。

3. 上限アラート通知方針

プロジェクトごとに選択した質問数上限アラート閾値へ当月初回到達した時点で、宛先へメールを送信する。宛先はオーナー + 当該プロジェクトの有効な管理者(project_users.role='admin' AND valid=1)とし、同一メールアドレスは重複排除する。全閾値未選択時はアラート通知を行わない。

項目 方針
選択可能な閾値 25% / 50% / 80% / 90% / 100%(複数選択可)
送信契機 選択した各閾値へ当月初回到達した時のみ(同一月の同一閾値で再送しない)
宛先 オーナー + 当該プロジェクトの有効な管理者。同一メールアドレスは重複排除
全閾値未選択時 アラート通知なし。ただし 100% 到達によるウィジェット受付停止処理自体は行う
FAQ 件数 アラート対象外
通知契機・件名本文は別正本通知契機表(配信先・重要度)は 06_メッセージ一覧.html §6、メール件名・本文テンプレート全文は 11_メール設計.html §4.8〜§4.11 を正本とする。本ページは送信条件と宛先解決ルールのみを扱う。

4. 支払方法ゲート

新規契約は支払方法未登録でも無料枠内を利用できる。いずれかのプロジェクトが無料枠 80% に到達したら事前告知し、支払方法未登録のまま無料枠を超過したら当該プロジェクトのウィジェット受付のみを停止する。契約はサスペンションせず管理画面は継続利用でき、支払方法登録で即時再開する。

段階 契機 システム動作 契約状態
事前告知 いずれかのプロジェクトが無料枠 80% 到達 利用状況(SCR-016)・お知らせ・メールで支払方法登録が必要になる旨を告知 active
受付停止 支払方法未登録のまま無料枠を超過 当該プロジェクトのウィジェット新規質問受付のみを停止し、支払方法登録へ誘導。管理画面は全機能継続。オーナー + 当該プロジェクト管理者へ通知 active(サスペンションしない)
再開 支払方法登録(Setup Intent 成功) 当該プロジェクトのウィジェット受付を即時再開。以降の無料枠超過分は事後課金 active
支払方法ゲート ≠ サスペンション支払方法ゲートはウィジェット受付のみを止める措置で、契約 contract_status は active のままである。決済失敗起因のサスペンション(§5)とは別経路であり、混同しないこと。支払方法登録済の契約は全課金対象の無料枠超過分を事後課金とし、受付は止めない(質問数上限件数 100% 停止は別途適用)。

5. 契約状態ライフサイクル

contract_owners.contract_status は active / suspended / deleted_pending / deleted の 4 値。状態定義そのものは共有概念(メイン正本)に従い、本ページは課金起因のサスペンション遷移と退会起因の削除遷移を記述する。決済失敗の猶予は 7 日、退会の猶予は 90 日で、両者は別の期間である。

状態 意味 アカウント利用者の可否 ウィジェット応答
active 有効 全機能 通常応答
suspended サスペンション中(決済失敗猶予経過 / 運営者手動停止) 課金・退会のみ 機能停止の旨を示す応答(FR-138)
deleted_pending 退会申請中(90 日猶予) 退会撤回・データ参照中心 停止
deleted 削除済み × 停止。contract_status='deleted' 設定時は contract_owners.valid=0 を同時セット
contract_status は共有概念正本に従う4 値の定義・valid との整合・テーブル CHECK 制約は 共有概念.html #1 および 03_テーブル設計.html §4.5.5 を正本とする。本ページは課金 / 退会起因の遷移条件のみを扱う。

5.1 決済失敗からサスペンションへ(7 日猶予)

課金プロバイダから決済失敗確定の通知を受信した時刻を起点に 7 日間の猶予を計測する。次のシーケンス図は、猶予開始から猶予経過による suspended 移行までの流れと、猶予中 / サスペンション中いずれでも再決済成功・運営者解除の通知受信で即時 active へ復帰する分岐を示す。猶予中も管理画面ログインと支払方法更新は可能である。

sequenceDiagram
  participant P as 課金プロバイダ
  participant W as worker-main-api
  participant D as D1 (contract_owners)
  participant O as オーナー

  P->>W: 決済失敗確定の通知
  W->>D: 受信時刻を起点に 7 日タイマー開始(active 維持)
  D-->>W:
  W->>O: 重要度 critical 通知
  Note over W: 猶予中は管理画面ログイン・支払方法更新を許可
  alt 7 日以内に再決済成功の通知受信
    P->>W: 再決済成功の通知
    W->>D: 猶予解除(active 維持)
    D-->>W:
  else 7 日経過
    W->>D: contract_status を suspended へ移行
    D-->>W:
    Note over W: ウィジェット応答を機能停止へ切替・管理画面を課金/退会のみに限定
    P->>W: 再決済成功 / 運営者解除の通知
    W->>D: contract_status を active へ復帰
    D-->>W:
  end
7 日は 7 日のまま決済失敗の猶予は 7 日で固定し、退会猶予 90 日とは混同しない。サスペンション中は課金・退会のみ可で、それ以外の管理画面操作とウィジェット応答は停止する。サスペンション中の操作制限は 08_認証・認可設計.html、対応エラー ID は 05_エラー設計.html §4.6 を正本とする。

5.2 退会から削除へ(90 日猶予)

オーナーが退会申請すると deleted_pending へ移行し、90 日の猶予の後に物理削除バッチが deleted へ確定させる。猶予中は退会撤回が可能で、撤回時は元の状態へ戻す。退会・解約はオーナー専有操作とする。

遷移 契機 挙動
active / suspended → deleted_pending オーナーが退会申請(再認証を伴う) ウィジェット応答を停止。当月末まで参照中心で利用可、月末締めで停止(日割りなし)。90 日の削除猶予を開始
deleted_pending → 元の状態 90 日以内の退会撤回 申請前の active / suspended へ戻す
deleted_pending → deleted 90 日経過後の物理削除バッチ contract_status='deleted' と contract_owners.valid=0 + users.valid=0 を同一トランザクションで同時セットし、全データを物理削除
退会猶予は 90 日退会申請(deleted_pending)から物理削除(deleted)までの猶予は 90 日とする。論理削除データは updated_at 基準で 90 日経過後に物理削除バッチが処理する。退会バッチの実装詳細は DD13 バッチ・非同期処理 を参照。

6. 利用量集計方針

利用量は発生時にリアルタイム集計し、管理画面向けサマリは 5 分以内の遅延で反映する。月次集計はプロジェクト単位で行い、契約単位へ集約して請求する。課金対象外(推論失敗時)の計測は区別して保持する。

項目 方針
リアルタイム集計 質問発生時に usage_metering へ計測。選択アラート閾値到達・100% 停止・125% 通知の判定に用いる
サマリ反映遅延 管理画面向け利用量サマリは 5 分以内に反映(FR-126)
課金対象の区別 推論失敗の質問はカウントするが計測フラグで課金対象外として区別。AI 利用コストは成功時のみ計測
月次集計 → 請求 プロジェクト単位で月次集計し、契約単位で全プロジェクトの超過分を集約して請求書を作成(契約単位)。同一月の二重集計は冪等にスキップ
サスペンション中 月次集計対象外(課金しない)
月次請求確定 cron は運営者側が正本月初の月次請求確定バッチ・課金プロバイダ請求作成・請求書発行は運営者側 02_運営者システム 10_課金・請求設計.html を正本とする。メイン側は計測と上限判定、請求見込みの表示を担う。

7. 課金プロバイダ Webhook 受信方針

課金プロバイダからの通知(イベント)を受信する際は、送信元の正当性を検証し、重複処理を冪等に防止する。次のシーケンス図は、署名検証・冪等チェック・契約状態更新までの受信フローと、署名不正 / 重複イベント / 処理失敗の分岐を示す。続く表は、図に表れない受信履歴の保持期間やメイン側で扱う受信種別などのパラメータを補う。

sequenceDiagram
  participant P as 課金プロバイダ
  participant W as worker-main-api
  participant D as D1
  participant O as 運営者

  P->>W: イベント通知(署名付き)
  W->>W: 署名検証
  alt 署名不正
    W-->>P: 拒否(処理しない)
  else 署名正当
    W->>D: イベント識別子で重複受信を照会
    D-->>W: 受信判定(初回 / 重複)
    alt 重複(処理済み)
      W-->>P: 冪等にスキップ
    else 初回
      W->>D: contract_status 更新 + キャッシュ無効化 + セッション処理
      D-->>W: 更新結果(成功 / 失敗)
      alt 失敗
        W->>O: 処理失敗を通知(再処理を受付)
      else 成功
        W-->>P: 受領
      end
    end
  end
項目 内容
受信履歴 直近 30 日の受信履歴を保持する(BR-095)
再処理 処理失敗時は運営者の指示による再処理を受け付ける
メイン側で扱う受信 契約停止 / 解除イベント受信による contract_status 更新 + キャッシュ無効化 + セッション処理
Webhook 一次受信・DLQ・リプレイは運営者が正本Webhook 一次受信・冪等基盤・DLQ・リプレイ・ペイロード差分検出(webhook_events)は 共有概念.html #12 のとおり運営者 10_課金・請求設計.html §13 を正本とする。連携 IF #1〜#12 の通信仕様・主管責任は 02_API設計.html §6 連携 IF 通信仕様一覧 と運営者 02 API 設計を両方正本とする。

8. 請求画面(SCR-022)で確認できる内容

オーナーは請求画面で、支払方法・請求状態・請求履歴・契約全体の請求見込み・プロジェクト別の課金内訳・上限詳細を確認できる。画面項目・レイアウトは画面設計が正本で、本ページは確認できる情報の範囲を定義する。

  • 支払方法(登録有無・更新導線)
  • 請求状態(未請求 / 請求済 / 支払成功 / 支払失敗確定 等)と請求履歴
  • 契約全体の当月請求見込み(全プロジェクトの超過分を契約単位で集約)
  • プロジェクト別の課金内訳(課金対象別の利用量・超過分)
  • プロジェクト単位の上限詳細(当月利用・上限件数・利用率)
画面項目は 01 が正本SCR-022 請求・SCR-021 利用量と上限・SCR-021-001 質問数上限設定モーダルの項目定義・バリデーションは 01_画面設計/index.html §5.27 / §5.12 / §5.13 を正本とする。課金・契約メッセージ文言は 06_メッセージ一覧.html §7 を正本とする。

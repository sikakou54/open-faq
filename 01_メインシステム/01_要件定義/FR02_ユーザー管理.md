# FR02: ユーザー管理(オーナー + メンバー)

## 1. 機能概要

| 項目 | 内容 |
|---|---|
| 概要 | オーナーアカウント(1 契約 = 1 アカウント固定)とメンバーアカウント(0..N、プロジェクト単位のロール `admin` / `member` で制御)のユーザーモデルを管理する。招待・アクティベーション・ロール変更・削除の各操作を扱う |
| 利用場面 | オーナーによる契約全体のメンバー管理、およびプロジェクト管理者ロール保持者による該当プロジェクト範囲のメンバー招待・離脱 |
| 優先度 | Must |

## 2. 収録 FR-NNN 一覧

| ID | 要件 | 優先度 |
|---|---|:---:|
| FR-015 | オーナーおよびプロジェクト管理者(`project_users.role='admin'` 保持者)は、自分が現在開いているプロジェクトワークスペースにおいて、メールアドレス・当該プロジェクトに対するロール(`admin` / `member`)を指定してメンバーを招待できること。**招待モーダル(SCR-017-M1)では氏名(表示名)を入力しない**(個人情報原則により、他人の氏名を管理者が任意に登録することを禁止)。氏名は招待された本人が SCR-031 メンバーアカウント有効化ページで入力する(FR-016d)。**メンバー管理は常にプロジェクト単位**で行い、SCR-017-M1 では当該プロジェクトの 1 件の割当のみを扱う。同一メンバーを複数プロジェクトに参加させる場合は各プロジェクト WS で個別に招待操作を行い、同一オーナースコープでメールアドレスが一致するアカウントは名前解決により単一ユーザーに紐付き、新規プロジェクト分の `project_users` 行のみが追加される(FR-021c と整合) | P0 |
| FR-015a | オーナーはアカウント新規登録時に自動的に決定され、全プロジェクトに対する全権(プロジェクト管理者ロール相当)とオーナー専有機能を恒久的に保持すること。**オーナーは「プロジェクト作成時に `project_users(role='admin', valid=1)` 行が自動 INSERT される明示的なプロジェクト管理者」となる**。認可フロー先頭の `isOwner=true` bypass(`contract_owners` 行存在で判定)は維持し、`project_users` 行は通知宛先解決と画面表示の事実情報として参照する。オーナーはロールの編集対象外とし、降格・譲渡操作は提供しない(MVP)。1 契約 = 1 オーナー(`contract_owners.user_id` の PRIMARY KEY で担保)とする | P0 |
| FR-015b | メンバーの権限は「プロジェクトごとに付与するロール」で表現し、ロールは `admin`(プロジェクト管理者)と `member`(メンバー)の 2 種とすること。`admin` は該当プロジェクトに対する FAQ 管理 + 未解決質問管理 + ログ参照 + ユーザー管理(該当プロジェクト範囲の招待・離脱)を、`member` は該当プロジェクトに対する FAQ 管理 + 未解決質問管理 + ログ参照のみを許可する。個別の権限フラグ、プリセットロール、権限テンプレートは提供しない | P0 |
| FR-015c | 契約単位の課金情報・支払方法・請求履歴・退会申請・規約再同意・契約設定、およびプロジェクトの作成・編集・削除はオーナー専有機能とし、いずれのロールでも付与できないこと。プロジェクト単位の利用上限・無料利用枠は、オーナーまたは当該プロジェクト管理者が変更できること(FR-121 / FR-125 と整合) | P0 |
| FR-015d | メンバーは割当されたプロジェクトに帰属するデータ(FAQ・要対応の質問・質問ログ等)のみアクセス可能とすること。プロジェクトから外す操作は対象の `project_users` 行のみを論理削除し、他プロジェクトの割当と `users` 行には影響させない。非オーナーの有効な割当が 0 件になる操作は拒否し、契約から完全に削除する場合はオーナーが SCR-039 契約メンバーからアカウント削除を行う。オーナーは割当の有無に関わらず全プロジェクトを閲覧・操作でき、各プロジェクトの `admin` 行も保持する | P0 |
| FR-015e | プロジェクト作成時、オーナーは自動で当該プロジェクトのプロジェクト管理者となる(`project_users(role='admin', valid=1)` を自動 INSERT。FR-030a と整合)。SCR-010-M1 には「このプロジェクトの管理者を指定」セクション(自己管理者チェックボックス + 他者管理者メール指定)を設けない。他者をプロジェクト管理者として招待する操作は、当該プロジェクトの SCR-017 / SCR-017-M1(プロジェクト WS)で行う | P0 |
| FR-016 | 招待されたメンバーは招待リンクから **SCR-031 メンバーアカウント有効化ページ** に遷移し、本人による **氏名(表示名)入力 + 初回パスワード設定 + 利用規約・プライバシーポリシー同意** を行ってアカウントを有効化できること。招待リンクの有効期限は **7 日**(FR-020 と整合)。期限切れの場合はオーナーまたは該当プロジェクトの管理者ロール保持者が再送信する | P0 |
| FR-016a | 招待時に指定されたロール割当(プロジェクト × `admin` / `member`)は、招待対象者が SCR-031 でアクティベーション(氏名入力 + パスワード設定 + 規約同意)を完了した時点で有効化されること(`project_users.valid` を `0` → `1` に更新) | P0 |
| FR-016b | 招待リンクの再送信は、オーナーまたは該当プロジェクト管理者のみが実行できること。再送信時は旧リンクを失効させ、新リンクを発行する。再送 API リクエストには氏名(`displayName`)を含めない。招待中メンバーをキャンセルしたい場合は招待取消ではなくメンバー削除(FR-019)で対応する | P0 |
| FR-016c | アクティベーション完了前のメンバーは、招待時に指定されたロール割当を予約状態(`project_users.valid=0`)として保持するが、ログインおよび認可チェックは通らないこと(SCR-031 完了時点で初めて有効化される) | P0 |
| FR-016d | 招待されたメンバーは、招待リンク先の **SCR-031 メンバーアカウント有効化ページ** で氏名(表示名)・初回パスワード・**利用規約同意・プライバシーポリシー同意(それぞれ独立した必須同意)** を入力してアカウント有効化を完了する。利用規約とプライバシーポリシーは個別の文書として版管理され、双方の最新版への同意が有効化の必須条件となる。氏名は本人のみが入力でき、招待元の管理者は事前に氏名を入力できない(個人情報原則)。氏名は本人が SCR-028 個人設定で後から変更できる | P0 |
| FR-016e | 招待時点で `users` レコードは `name=NULL` / `status='pending_activation'` で作成され、SCR-031 有効化完了時に `users.name` 更新 + `users.status='active'` + `project_users.valid=1` + `terms_agreements` INSERT(**利用規約・プライバシーポリシーで各 1 行、計 2 行を `doc_type` 別に記録**)+ `access_tokens.used_at` セットを同一トランザクションで実行する。招待トークン(`access_tokens.purpose='activation'`)の `meta` フィールド(JSON)には `invitedProjectId` / `invitedRole` を保持し、有効化完了時に当該プロジェクトの予約行の特定に用いる | P0 |
| FR-017 | 利用者は SCR-017 プロジェクトメンバーで、現在開いているプロジェクトに割当のあるメンバー一覧と詳細を参照できること。詳細には種別(契約オーナー / プロジェクト管理者 / プロジェクトメンバー)、ステータス(`利用中` / `招待中`)、当該プロジェクトのロールを含める。最終ログイン日時は表示せず、招待期限と再送導線は SCR-017-M1 に集約する。加えてオーナーは SCR-039 契約メンバーで、契約内の全メンバー、参加プロジェクト、プロジェクト別ロールを横断して参照できること | P0 |
| FR-018 | オーナーおよびプロジェクト管理者は、自分(オーナー)に紐づくメンバーの **メールアドレスのみ** を更新できること(氏名の更新は不可)。**氏名(表示名)は本人のみが SCR-031 メンバーアカウント有効化ページおよび SCR-028 個人設定で変更可能**(個人情報原則により、管理者は他人の氏名を編集できない)。SCR-017-M1 編集モードでは氏名は read-only 表示(未入力時は「未設定」を灰色テキストで表示)。オーナーの氏名・メールの更新はオーナー本人のみが行えること。プロジェクト管理者が更新できる対象は、当該プロジェクトに割当のあるメンバーに限る。**個人ごとの通知受信オプトアウト(重要/通常/低の受信切替)は MVP では提供しない**。プロジェクト関連通知は v2.0 で常時 ON 固定とし、当該プロジェクトの全メンバー(オーナー + admin + member)に即時配信する(画面設定なし) | P0 |
| FR-018a | プロジェクト管理者(オーナー含む)は、SCR-017-M1 で当該プロジェクトの 1 件の割当のみを編集できること(`admin` / `member` のラジオ選択 + 「プロジェクトから外す」操作)。他プロジェクトの割当とアカウント状態には影響しない。自分自身の降格・割当解除、および非オーナーの有効な割当が 0 件になる解除は不可とする。他メンバーを `admin` から `member` へ降格する操作は誤操作防止レベル L2を必須とする | P0 |
| FR-018b | ロール変更・削除時には、変更前後のプロジェクト × ロール差分を監査ログに記録し、当該メンバーへお知らせ受信箱とメールで通知できること(FR-145a と整合) | P0 |
| FR-018c | オーナーは、メンバーを別プロジェクトへ追加で割当する場合、当該別プロジェクトの WS に切り替えて SCR-017-M1 から招待する。プロジェクト管理者は他プロジェクトへの割当はできず、自分が管理者であるプロジェクト範囲でのみ招待・ロール変更・削除ができる(範囲内のロール制約は FR-019 を正本)| P0 |
| FR-019 | アカウントレベルの削除はオーナー専有とし、SCR-039 契約メンバーの「契約から削除」からのみ実行できること。対象ユーザーの `users.valid=0` と全 `project_users.valid=0` を同一トランザクションで更新し、全セッションと未使用招待トークンを失効する。オーナー自身は削除対象外とし、削除操作は誤操作防止レベル L3(対象表示名のタイプ確認 + 再認証)とする。プロジェクト管理者にはアカウント削除権限を付与しない | P0 |
| FR-019a | SCR-017-M1 の「プロジェクトから外す」は当該 `project_users` 行だけを `valid=0` にする。対象が他プロジェクトに有効な割当を持つ場合は解除を許可し、対象ユーザーのセッションは即時失効させず次回認可チェックから反映する。対象が非オーナーで最後の有効な割当である場合は `E-BIZ-MEMBER-NO-GRANT` で拒否し、オーナーに SCR-039 からの契約削除を案内する | P0 |
| FR-019b | SCR-039 の契約削除確認では、参加中の全プロジェクト名と件数、全参加解除、セッション失効、90日後の物理削除予定を明示する。削除後に同じメールアドレスで再招待された場合は、論理削除済み行を維持して新規 `users` 行を作成する | P0 |
| FR-020 | 招待リンクには有効期限を設けること。**有効期限: 7 日**(顧客管理システム FR-020 と整合)。期限切れ後は招待の再送信で新リンクが発行される | P0 |
| FR-021 | メンバー数に固定の上限を設けず、利用量しきい値に基づいて上限接近や急増を検知し、利用者(オーナー / メンバー)へアラート通知できること | P0 |
| FR-021a | メンバー数しきい値接近・超過時のアラート通知先は、オーナーおよび同一契約内で 1 件以上の `admin` ロールを保持する全プロジェクト管理者とすること(critical 通知の宛先と一致) | P1 |
| FR-021b | 契約にメンバーが 1 人も登録されていない状態でも、オーナーが単独で契約運営を継続できること(オーナーは全機能保持) | P0 |
| FR-021c | 同一オーナーに紐づく既存の有効・招待中アカウントと同一メールアドレスでメンバーを重複招待できないこと。別オーナー配下では別アカウントとして扱う(FR-335 と整合) | P0 |

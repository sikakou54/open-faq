1. 機能概要

項目 内容
概要 オーナー配下で FAQ・ウィジェット・ログを分けて管理する単位「プロジェクト」を作成・編集・削除する。許可ドメイン設定、ウィジェット設定、プロジェクト連絡先メールも含む
利用場面 製品別 / サイト別にプロジェクトを分けて運用する場合、ウィジェット設置先サイトの追加、プロジェクト終了時の整理
優先度 Must

2. 収録 FR-NNN 一覧

ID 要件 優先度
FR-030 オーナーはプロジェクトを作成、編集、削除できること。新規作成・編集・削除はいずれも SCR-004-M1(作成・編集兼用モーダル)で行う。プロジェクト一覧(SCR-004)ではプロジェクト名を主リンクとし、クリックで SCR-004-M1 を編集モードで開く。一覧に ID 列・更新日時列は設けず、プロジェクト ID は編集モーダル内に読み取り専用 + コピー可能な補助情報として表示する。削除動線は SCR-004-M1 のモーダル内 DangerSection のみに置き、プロジェクト名タイプ確認 + 再認証(レベル L3)を必須とする P0
FR-030a プロジェクト作成時には、当該プロジェクトの作成者であるオーナー自身が常にプロジェクト管理者となる。SCR-004-M1 は作成・編集兼用とし、管理者指定は扱わない(作成時にオーナーが自動で当該プロジェクトの管理者となる)。一般メンバー追加・他者をプロジェクト管理者として招待する操作は SCR-009 / SCR-009-M1 で行う P0
FR-030b プロジェクト削除時は当該プロジェクトの project_users を全行論理削除(UPDATE … SET valid=0)する。削除の結果、他プロジェクト割当(valid=1 の project_users 行)が 0 件かつ contract_owners 行を持たない(=非オーナー)ユーザーは users.valid=0 で論理削除する(孤立メンバー cleanup)。他プロジェクト割当があるメンバーは users.valid=1 を維持し、当該プロジェクトに対する project_users 行のみ valid=0 で論理削除する。オーナー(contract_owners 行を保持するユーザー)の users 行は本処理の対象外(オーナー自身が当該プロジェクトに対し保持していた project_users(role='admin') 行は valid=0 に更新するが、オーナー users 行と contract_owners 行は維持)。論理削除データは updated_at から 90 日経過後に物理削除バッチで物理削除する P0
FR-031 プロジェクトごとにウィジェット設定(公開キー・埋め込みコード・見た目・プレビュー)を管理できること。公開キーはプロジェクト単位で発行・ローテーションし、契約共通ではない。公開キーの再発行は当該プロジェクトの管理者(project_users.role='admin')以上に許可し、誤操作防止レベル L3(再認証 + 「既存埋め込みコードが即座に無効化される」旨の確認)を必須とする。メンバー(role='member')は閲覧と埋め込みコードのコピーのみ可。SCR-007 はプロジェクトワークスペース内に配置する(契約ワークスペースには配置しない) P0
FR-032 プロジェクトごとにFAQ、質問ログ、未解決質問を分けて管理できること P0
FR-033 許可するWebサイトのドメインを複数設定できること P0
FR-033a プロジェクトごとに連絡先メールアドレスを 1 件設定でき、確認メール(TPL-PROJECT_CONTACT_VERIFY、24h 有効、access_tokens.purpose='contact_verify'、meta.projectId を JSON 保持)を経て有効化できること。確認メール内リンクは SCR-019 プロジェクト連絡先メール確認完了ページ(未認証可、共通領域)に着地し、トークン検証成功時に projects.contact_email_verified_at=now() をセットする。確認完了前のアドレスはウィジェット表示に用いない。メール送信時の Reply-To としては利用しない(エンドユーザー向けメールは常に共通ドメインの no-reply 送信で固定) P0
FR-033c 公開ウィジェットにおいて、確認完了済みのプロジェクト連絡先メールを未解決時・質問受付制限時のお問い合わせ先として表示できること。未設定または未確認の場合はメールアドレスを表示しないこと P0
FR-034 プロジェクト削除時の関連データ(FAQ、質問ログ、未解決質問)の取扱いを利用者が確認・選択できること P0
FR-035 プロジェクト数に固定の上限を設けず、利用状況を監視し急激な増加を検知してオーナーへ通知できること P1

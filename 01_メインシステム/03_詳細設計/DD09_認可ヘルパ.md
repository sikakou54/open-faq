## 3. 詳細設計本文

### 3.1 認可ヘルパ実装

```ts
// app/workers/main-api/src/lib/authz.ts
export type ProjectRole = 'admin' | 'member';

export function requireTenant(c: Context, expectedTenantId: string) {
  const principal = c.get('principal');
  if (principal.contractOwnerUserId !== expectedTenantId) throw new HTTPException(404); // 404 偽装
}

export function requireOwner(c: Context) {
  const principal = c.get('principal');
  if (!principal.isOwner) throw new HTTPException(403); // E-AUTHZ-OWNER-ONLY
}

export async function requireProjectRole(c: Context, projectId: string, minRole: ProjectRole) {
  const principal = c.get('principal');
  const project = await c.env.DB.prepare(
    `SELECT contract_owner_user_id FROM projects WHERE id = ?1`
  ).bind(projectId).first<{ contract_owner_user_id: string }>();
  if (!project) throw new HTTPException(404);
  if (project.contract_owner_user_id !== principal.contractOwnerUserId) throw new HTTPException(404); // 404 偽装

  if (principal.isOwner) return; // オーナーは全プロジェクト bypass(暗黙の admin)

  const granted = principal.projectRoles.get(projectId);
  if (!granted) throw new HTTPException(404); // E-AUTHZ-PROJECT-DENIED(404 偽装)
  if (minRole === 'admin' && granted !== 'admin') throw new HTTPException(403); // E-AUTHZ-FORBIDDEN
  // minRole='member' は admin/member いずれでも通過
}

export async function requireInquiry(c: Context, inquiryId: string) {
  const principal = c.get('principal');
  const row = await c.env.DB.prepare(
    `SELECT contract_owner_user_id, project_id FROM inquiries WHERE id = ?1`
  ).bind(inquiryId).first<{ contract_owner_user_id: string; project_id: string }>();
  if (!row) throw new HTTPException(404);
  if (row.contract_owner_user_id !== principal.contractOwnerUserId) throw new HTTPException(404);
  await requireProjectRole(c, row.project_id, 'member');
}

export function requireRole(...roles: Role[]): MiddlewareHandler {
  return async (c, next) => {
    const principal = c.get('principal');
    if (!roles.includes(principal.role)) throw new HTTPException(403);
    await next();
  };
}
```

### 3.2 Principal 構築（参照のみ）

`c.get('principal')` で取得する Principal の構築フローは [../02_基本設計/08_認証・認可設計.md §6.1](../02_基本設計/08_認証・認可設計.md) を正本とする。Principal には以下を含む。

- `userId`: ユーザー ID(オーナー / プロジェクト管理者 / メンバー全員)
- `contractOwnerUserId`: 所属契約の owner user_id(本人がオーナーなら自身の userId、メンバーなら所属契約の owner user_id)
- `role`: 契約ロール(`users.role` = `admin` / `service_operator` / `end_user`)
- `isOwner`: オーナー判定(`contract_owners` 行存在で判定 = `co.user_id IS NOT NULL`)
- `projectRoles`: `Map<projectId, 'admin' | 'member'>`。オーナー / メンバー共通で `project_users WHERE valid=1` を 1 クエリで取得して構築する。認可ヘルパは `isOwner=true` のときに `projectRoles` の中身を参照せず bypass する優先順位を保つ。`projectRoles` は通知宛先解決と画面表示の事実情報として参照可能

#### Principal 構築クエリ(参考、基本設計と同期)

```sql
SELECT u.id, u.role, u.status, u.valid,
       (co.user_id IS NOT NULL)                          AS is_owner,
       COALESCE(co.user_id, pu.contract_owner_user_id)   AS contract_owner_user_id
FROM users u
LEFT JOIN contract_owners co ON co.user_id = u.id AND co.valid = 1
LEFT JOIN project_users   pu ON pu.user_id = u.id AND pu.valid = 1
WHERE u.id = ?1 AND u.valid = 1
LIMIT 1;
```

`projectRoles` は別途 `SELECT project_id, role FROM project_users WHERE user_id=? AND valid=1` で取得する。

### 3.3 オーナー境界（NFR-301）

すべての CRUD API は **`contract_owner_user_id` フィルタを必須** とする。`requireTenant` / `requireProject` / `requireInquiry` で境界チェックを実施し、他オーナーのデータには 404 を返す（漏洩防止のため 403 ではなく 404）。

### 3.4 不正アクセスの監査記録

境界違反検知時は `authz.owner_boundary_violation` / `authz.project_boundary_violation` を `retention_class=general` で記録する。詳細は [DD10_監査ログ書込・完全性検証.md](DD10_監査ログ書込・完全性検証.md) を参照。

// tests/isolation/cross-owner.test.ts
describe('クロス契約アクセス試行', () => {
  it('他契約の inquiry にアクセスできない', async () => {
    const ownerA = await createOwnerAccount();
    const ownerB = await createOwnerAccount();
    const inquiryB = await createInquiry(ownerB);
    const sessionA = await loginAs(ownerA.adminEmail);
    const res = await fetch(`/api/v1/inquiries/${inquiryB.id}`, { headers: { Cookie: sessionA } });
    expect(res.status).toBe(404);  // 漏洩しないために 403 ではなく 404
  });

### 5.2 その他観点

| 観点 | 内容 |
|---|---|
| 単体 | `requireTenant` / `requireProject` / `requireInquiry` / `requireRole` の各分岐 |
| 結合 | プロジェクト割当がない(`project_users` 行なし)メンバーによる FAQ 編集試行で 404 偽装(`E-AUTHZ-PROJECT-DENIED`)/ 該当 PJ の `member`+ なら 200 |
| 異常系 | 削除済オーナー所属メンバーのアクセス時 401 + セッション失効 |
| 境界値 | アカウント削除はアカウントレベル論理削除のため、現存メンバーは常に 1 件以上の割当を保持。割当 0 件はテスト境界として「データ不整合時のフェイルセーフ動作」(`/auth/login` への強制ログアウト)を検証する |
| 月次 | クロス契約侵入試行 50 ケース月次 + 5 ケース年次手動 |

## 3. 詳細設計本文

### 3.1 利用者・権限詳細設計（参照のみ）

ユーザー種別、3 ロール(オーナー / プロジェクト管理者 / メンバー)、画面別権限マトリクス、認証フロー、Principal 構築は [../02_基本設計/04_権限設計.md](../02_基本設計/04_権限設計.md) および [../02_基本設計/08_認証・認可設計.md](../02_基本設計/08_認証・認可設計.md) を正本とする。

利用者側アカウントはオーナーとメンバーに分かれ、メンバーはプロジェクトごとに `admin` / `member` を持つ。SCR-009-M1の「プロジェクトから外す」は対象 `project_users` 行を論理削除し、最後の有効割当だった場合は `users.valid=0` と全セッション・未使用招待を同一トランザクションで失効する。

### 3.2 状態詳細設計（参照のみ）

契約状態（`contract_owners.contract_status`: `active` / `suspended` / `deleted_pending` / `deleted`）の全状態遷移は [../02_基本設計/03_テーブル設計.md](../02_基本設計/03_テーブル設計.md) §4.5.5 を正本とする。

### 3.3 画面詳細設計（参照のみ）

SCR-001〜003 / SCR-009 / SCR-016 / SCR-017 / SCR-023 の表示・入力・操作・メッセージIDは基本設計を正本とする。

### 3.4 機能詳細設計（参照のみ）

### 3.6 関連する横断設計

#### 3.6.1 トークン発行・検証

メール確認・パスワードリセット・管理者ユーザー有効化（activation）の各トークンは [DD07_トークン発行・検証.md](DD07_トークン発行・検証.md) の HMAC-SHA256 保存方式を利用する。一回限りトークンは `consumeToken()` を検証直後に必ず呼ぶ。

#### 3.6.2 認可ヘルパ

`requireTenant` / `requireProject` / `requireRole` の各ヘルパは [DD08_認可ヘルパ.md](DD08_認可ヘルパ.md) を利用する。

#### 3.6.3 監査ログ書込

`auth.signup` / `auth.login_success` / `auth.login_failure` / `auth.logout` / `auth.password_reset` / `auth.re_auth.success` / `auth.ip_blocked` / `auth.terms_agree` などの action コードはすべて [DD09_監査ログ書込・完全性検証.md](DD09_監査ログ書込・完全性検証.md) の `writeAudit()` で書込み、`retention_class=general` とする。

#### 3.6.4 暗号化・鍵管理

`email` / `phone` などの PII はオーナー派生鍵で AES-256-GCM 暗号化し、検索用に HMAC-SHA256 列（`email_hmac` 等）を併設する。詳細は [DD10_暗号化・鍵管理.md](DD10_暗号化・鍵管理.md) を参照。

### 3.7 オーナー / メンバー / プロジェクト割当

- **オーナー判定** = `contract_owners` 行(`valid=1`)を持つユーザー。`contract_owners.user_id` が当該オーナーの `users.id` を直接参照する(`is_owner` カラムも `owner_account_id` 自己参照も持たない)
- メンバーは `contract_owners` 行を持たず、`project_users.contract_owner_user_id` 経由で所属契約に紐付く
- メンバーの権限は `project_users.role`(`admin` / `member`)とプロジェクト割当の組合せで管理
- オーナーもプロジェクト作成時に `project_users(role='admin', valid=1)` 行を自動取得し、明示的なプロジェクト管理者として行を保持する。認可フロー先頭の `isOwner=true` bypass(`contract_owners` 行存在で判定)は維持し、`project_users` 行は通知宛先解決と画面表示の事実情報として参照する
- 新規オーナー登録時のトランザクション: `INSERT INTO users(...)` → `INSERT INTO contract_owners(user_id, contract_status='active', valid=1, ...)` を同一 BEGIN IMMEDIATE で実行(`users.id` = `contract_owners.user_id` = `contractOwnerUserId`、物理値は同一 ULID)
- 共有概念は [../../共有/共有概念.md](../../共有/共有概念.md) を正本とする

### 3.7a メンバーアカウント有効化フロー(SCR-018 / FR-016d / FR-016e)

招待されたメンバー本人が招待リンク先で氏名 + 初回パスワード + 規約同意を入力してアカウントを有効化するフローの実装ロジック。基本設計の正本は [../02_基本設計/08_認証・認可設計.md §3.4](../02_基本設計/08_認証・認可設計.md) と [../02_基本設計/02_API設計.md §5.1.7 / §5.1.8](../02_基本設計/02_API設計.md) を参照。

#### 3.7a.1 招待時の DB 状態作成(`POST /projects/{id}/members`)

```ts
await db.transaction(async (tx) => {
  // (1) users 行を name=NULL で作成(個人情報原則により、招待者は氏名を入力しない)
  const userId = ulid()
  await tx.users.insert({
    id: userId,
    email_encrypted: encrypt(email, ownerKey),
    email_hmac: hmac(email, ownerKey),
    name: null,                              // SCR-018 で本人が入力
    password_hash: null,                     // SCR-018 で本人が入力
    status: 'pending_activation',
    valid: 1,
  })
  // (2) project_users 予約行(valid=0)
  await tx.projectUsers.insert({
    user_id: userId,
    contract_owner_user_id: contractOwnerUserId,
    project_id: projectId,
    role: invitedRole,                       // 'admin' | 'member'
    valid: 0,                                // SCR-018 完了時に 1 に更新
  })
  // (3) access_tokens(activation, 7d, meta に invitedProjectId / invitedRole を保持)
  const rawToken = generateRawToken()
  await tx.accessTokens.insert({
    id: ulid(),
    user_id: userId,
    token_hash: hmac(rawToken, hashKey),
    purpose: 'activation',
    meta: JSON.stringify({ invitedProjectId: projectId, invitedRole }),
    expires_at: addDays(now(), 7),
  })
  // (4) 招待メール送信(TPL-ADMIN_USER_REGISTER または TPL-PROJECT_ADMIN_INVITE)
  await enqueueEmail({ template, activationUrl: buildActivationUrl(rawToken) })
})
```

#### 3.7a.2 プレビュー API(`POST /auth/invitations/{token}/preview`)

未認証可。トークン検証 + 招待情報パネル用のメタ情報を返却。

```ts
const { tokenRecord, userId } = await verifyToken(rawToken, 'activation', env)
const meta = JSON.parse(tokenRecord.meta)
const project = await db.projects.findById(meta.invitedProjectId)
const owner = await db.users.findById(project.contract_owner_user_id)
const user = await db.users.findById(userId)
return {
  userId,
  email: decrypt(user.email_encrypted, ownerKey),
  projectId: project.id,
  projectName: project.name,
  invitedRole: meta.invitedRole,
  inviterOwnerName: owner.name,
  expiresAt: tokenRecord.expires_at,
}
```

#### 3.7a.3 有効化 API(`POST /auth/invitations/{token}/activate`)

同一トランザクションで `users.name` / `users.password_hash` / `users.status` を更新 + `terms_agreements` INSERT + `project_users.valid=1` + `consumeToken()` + 監査ログ。

```ts
await verifyTurnstile(req.turnstileToken)
const { tokenRecord, userId } = await verifyToken(rawToken, 'activation', env)
const meta = JSON.parse(tokenRecord.meta)
validateDisplayName(req.displayName)        // 1〜100 文字
validatePassword(req.password)              // FR-006 強度要件
if (req.password !== req.passwordConfirm) throw E_INPUT_VALIDATION
if (!req.termsAgreed || !req.privacyAgreed) throw E_INPUT_REQUIRED

await db.transaction(async (tx) => {
  await tx.users.update(userId, {
    name: req.displayName.trim(),
    password_hash: await hashPassword(req.password),
    status: 'active',
    updated_at: now(),
  })
  // 利用規約・プライバシーポリシーを doc_type 別の 2 行で記録
  await tx.termsAgreements.insertMany([
    { id: ulid(), user_id: userId, doc_type: 'terms_of_service', terms_version: latestTermsVersion, agreed_at: now() },
    { id: ulid(), user_id: userId, doc_type: 'privacy_policy', terms_version: latestPrivacyVersion, agreed_at: now() },
  ])
  await tx.projectUsers.update({
    user_id: userId,
    project_id: meta.invitedProjectId,
  }, { valid: 1, updated_at: now() })
  await tx.accessTokens.update(tokenRecord.id, { used_at: now() })
  await tx.auditLogs.insert({
    action: 'user.activation_completed',
    actor_user_id: userId,
    target_user_id: userId,
    meta: JSON.stringify({ project_id: meta.invitedProjectId, role: meta.invitedRole }),
    retention_class: 'general',
  })
})
return { userId, redirectUrl: '/auth/login' }
```

#### 3.7a.4 招待再送(`POST /members/{id}/resend-invitation`)

旧 `access_tokens` を `used_at=now()` で失効させ、`meta` を同一(`invitedProjectId` / `invitedRole`)で新規トークンを発行。リクエストボディは空(`displayName` を受け付けない)。

#### 3.7a.5 名前解決による既存アカウント招待(FR-021c)

既存アカウントが同一オーナースコープに存在する場合は、SCR-018 をバイパスして `project_users` 行のみを `valid=1` で追加 + 既存アカウントのログイン URL を含む簡易メールを送信(氏名 / パスワードは既存値を保持)。

### 3.7b プロジェクト割当解除とアカウント停止

`DELETE /projects/{id}/members/{userId}`:

1. `requireProjectRole(..., 'admin')` と自己・オーナー保護を実行する。
2. 対象ユーザーの他プロジェクト有効割当数を取得する。
3. 対象 `project_users` 行を `valid=0` にする。
4. 他プロジェクト有効割当数が0件なら `users.valid=0`、全セッション失効、未使用招待失効を同一トランザクションで実行する。
5. `project.member_removed` を監査記録し、アカウント停止を伴う場合は `account_deactivated=true` をmetadataへ記録する。

### 3.8 退会フロー（参照のみ）

- オーナーが退会要求 → `UPDATE contract_owners SET contract_status='deleted_pending'` + `withdrawal_requests` レコード作成 + `scheduled_deletion_at`（30 日後）
- 30 日経過後、§14.1.9 定期削除 cron が `UPDATE contract_owners SET contract_status='deleted', valid=0` に更新し、後続バッチで `users.valid=0` を揃える
- 詳細は [DD13_バッチ・非同期処理.md](DD13_バッチ・非同期処理.md) を参照

### 3.9 フロント URL ルーティング規約(FR-325)

利用者向け管理コンソールの SPA は **スコープ別 4 系列** のルートツリーで構成する。スコープ判定はフロント router の `beforeEach` ガードと API ミドルウェア `authenticate.ts` + `requireTenant` の二重化で行う。

```
/auth/*                        ── 認証フロー(未認証可)
/owner/*                       ── 契約 WS(オーナー専有 = `contract_owners` 行存在)
/projects/:projectId/*         ── プロジェクト WS(`project_users` 保持者)
/account/*                     ── 共通領域(認証済全員)
/eu/*                          ── エンドユーザー(認証コンソール外、トークン認証)
```

#### 3.9.1 フロント router のガード(疑似コード)

```ts
router.beforeEach(async (to) => {
  const scope = parseScope(to.path)  // 'auth' | 'owner' | 'projects' | 'account' | 'eu'
  const me = await ensureSession()    // 未認証なら null

  if (scope === 'auth') {
    if (me) return resolveDefaultWorkspace(me)  // 認証済は既定 WS へリダイレクト(§4.4.7)
    return true
  }

  if (!me) return '/auth/login'

  if (scope === 'owner' && !me.isOwner) {
    return resolveDefaultWorkspace(me)  // 403 同等の扱い、UI 上は権限不足トーストを表示
  }

  if (scope === 'projects') {
    const projectId = to.params.projectId
    const projectUser = me.projectUsers.find(pu => pu.project_id === projectId)
    if (!me.isOwner && !projectUser) {
      // 最後の割当解除時は users.valid=0 となるため、通常は projectUsers.length >= 1
      return resolveDefaultWorkspace(me)
    }
  }

  return true
})
```

#### 3.9.2 既定ワークスペース決定(§4.4.7 と同期)

`resolveDefaultWorkspace(me)` は次の順で評価する:

1. localStorage の `lastWorkspace` が現在の `project_users` で有効ならその WS のホーム
2. オーナーかつ未保存なら `/owner` (SCR-016)
3. メンバーで `project_users` 行が 1 件なら `/projects/{onlyRow.project_id}` (SCR-008)
4. メンバーで `project_users` 行が複数なら最新 `last_accessed_at` のプロジェクト

ユーザー削除はユーザーレベル論理削除(`users.valid=0`)のため、現存するメンバー(`users.valid=1`)は `project_users.valid=1` を 1 件以上満たし、「`project_users` 行が 0 件のケース」は構造的に発生しない。データ不整合等で `WHERE valid=1` 条件下で `projectUsers.length === 0` を検出した場合のフェイルセーフは Sentry へ重大度 high を通知し `/auth/login` へ強制ログアウトする(運営者 IF #2 強制ログアウト発火を期待)。

**認証ミドルウェアの追加チェック**: 認証成功直後に `users.valid=1` を確認し、`valid=0` の場合は 401 でログイン拒否(セッション失効扱い)。これにより論理削除済みユーザーのセッションが残存しても再アクセスを防止する。オーナーの場合は加えて `contract_owners.valid=1 AND contract_status NOT IN ('deleted')` も検証(`deleted_pending` は 30 日間の猶予期間中は許容)。

### 3.10 SCR-016 プロジェクト0件の空状態(FR-326)

`GET /owner/projects/usage` の `summary.projectCount=0` の場合、SCR-016は全体サマリーと一覧を表示せず、「プロジェクトがまだありません」と「最初のプロジェクトを作成」の単一CTAだけを表示する。専用のオンボーディングAPIや進捗状態は保持しない。

### 3.11 契約共通項目の「利用中のプロジェクト」集計(FR-327)

契約共通項目の「利用中のプロジェクト: N」表示用に参照プロジェクト件数を返す。今月の利用上限・無料利用枠はプロジェクト単位のため対象外とする。

- AI 利用コスト消化率等の契約共通指標: `SELECT COUNT(*) FROM projects WHERE contract_owner_user_id = :contractOwnerUserId AND deleted_at IS NULL`(契約配下の全プロジェクト数。契約全体に適用される指標のため、全プロジェクトが利用先となる)
- 集約レスポンスは `GET /me/onboarding-status` と同一 API にまとめて返却(`shared_assets_usage` フィールド)し、SCR-016 の 1 ロードで全データを揃える

N=0 の場合のラベル切替は基本設計 [01_画面設計.md §4.4.4](../02_基本設計/01_画面設計.md) のメッセージ ID 体系(`MSG-SCR-016-LBL-005` / `LBL-006`)に従う。

### 5.1 結合テスト例（ロックアウト）

```ts
// tests/integration/auth/login.test.ts
describe('POST /api/v1/auth/login', () => {
  it('正しい資格情報でログイン成功', async () => {
    await seedAccount({ email: 'admin@example.com', password: 'P@ss12345678', role: 'admin' });
    const res = await (await worker).fetch('http://localhost/api/v1/auth/login', {
      method: 'POST', body: JSON.stringify({ email: '...', password: '...' }),
    });
    expect(res.status).toBe(200);
    expect(res.headers.get('Set-Cookie')).toContain('session=');
  });
  it('5 回失敗でロックアウト', async () => {
    for (let i = 0; i < 5; i++) {
      await (await worker).fetch('...', { method: 'POST', body: JSON.stringify({ ..., password: 'wrong' }) });
    }
    const res = await (await worker).fetch('...', { method: 'POST', body: JSON.stringify({ ..., password: 'correct' }) });
    expect(res.status).toBe(423);  // LOCKED_OUT
  });
});
```

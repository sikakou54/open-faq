3.5 FAQ 状態の保存

faqs.status は CHECK 制約（draft / published / hidden / deleted）で値域固定。状態は SCR-006 編集画面の「状態」ラジオで選択した値を PATCH /api/v1/faqs/{id} の status フィールドで保存する。draft / published / hidden は相互に自由遷移でき、専用の公開 API・状態遷移ガードは持たない。domain/faq-status.ts の責務は status 値域の妥当性検証のみ（deleted への直接遷移は DELETE /faqs/{id} 経由に限定、deleted → draft の復元は運営者の IF #4 経由に限定）。status='published' を選択して保存する操作が管理者(admin)・メンバー(member)による公開前の内容確認（FR-045）を兼ね、UI 上は保存前に確認ダイアログを表示する。published 設定時は published_at をサーバ側で記録する。

3.6 FTS5 検索

faq_search_fts は SQLite の FTS5 virtual table で構築し、faqs.title || faqs.body をインデックス対象とする。FAQ 公開・更新時に同期更新。検索クエリは BM25 ランキングを利用：

SELECT f.*, bm25(faq_search_fts) AS rank
FROM faq_search_fts JOIN faqs f ON f.rowid = faq_search_fts.rowid
WHERE faq_search_fts MATCH ?1 AND f.project_id = ?2 AND f.status = 'published'
ORDER BY rank LIMIT 20

詳細は DD03_AI回答パイプライン.html §3.2 関連度計算を参照。

3.7 埋め込みベクトル事前計算

FAQ 公開時に @cf/baai/bge-base-en-v1.5 で埋め込みベクトルを計算し、faq_embeddings テーブル（または KV embedding:{faqId}）に保存。AI 推論時のリランキング用。

3.8 一括インポート（faq.bulk_import）

CSV ファイル（UTF-8、BOM 許容）→ R2 ステージング → Queue 投入 → consumer が分割 INSERT。1 リクエスト最大 1000 件。JSON 形式は受理しない（不正形式は E-INPUT-CSV-INVALID）。UI は SCR-006-M1 FAQ CSV インポートモーダル（POST /faqs/import）。詳細は DD13_バッチ・非同期処理.html を参照。

文字コード事前検証（クライアント側）: SCR-006-M1 のドラッグ&ドロップ / ファイル選択の時点で、ファイル先頭バイト（BOM）および本文のバイト列から文字コードを判定し、UTF-8（BOM 許容）以外はアップロードせず即エラー（E-INPUT-CSV-INVALID / MSG-SCR-006-M1-ERR-005、検出した文字コード名を併記）。これによりサーバ側の文字化け INSERT を未然に防ぐ。

CSV 列定義（ヘッダ行必須）: FAQ ID, 質問, 回答, カテゴリ。status（状態）列は受理しない。

テンプレート生成（GET /faqs/import/template）: ヘッダ行のみ（FAQ ID,質問,回答,カテゴリ）を text/csv; charset=utf-8（BOM 付与）でストリーム返却。動的データなし（静的文字列）、Queue 不経由の同期レスポンス。

FAQ ID 行判定ロジック（consumer 側、行単位）:

  1. FAQ ID 空欄 → 新規 INSERT（status='draft'、version=1）。
  2. FAQ ID が当該契約（contract_owner_user_id 境界）内の既存 faqs.id と一致 → UPDATE（status は維持、faq_revisions に改訂行を追記、version をインクリメント）。
  3. FAQ ID が当該契約内に存在しない → 当該行を失敗（E-INPUT-CSV-FAQID-NOTFOUND）。他行は継続（部分成功）。
  4. 質問・回答の欠落 / 文字数超過 → 当該行を失敗（E-INPUT-CSV-INVALID 包含）。カテゴリは任意。

失敗行は「行番号 + エラー理由」の構造化リストとしてジョブ結果（GET /faqs/import/jobs/{jobId} 等のポーリング応答）に含め、SCR-006-M1 進捗パネル内のスクロール領域に画面表示する（エラー CSV のダウンロードは行わない）。

3.10 プロジェクト作成時のオーナー自動 admin 付与(FR-015e / FR-030a)

POST /projects 実行時は SCR-004-M1(新規作成モード)から name / allowedDomains 等の基本情報のみを受け取る(initialAdmins フィールドは受け取らない)。オーナーは作成時に自動で当該プロジェクトの管理者となる(project_users(role='admin', valid=1) 自動 INSERT)。他者をプロジェクト管理者として招待する操作は、プロジェクト作成後に SCR-009-M1 経由(POST /projects/{id}/members + role='admin')で行う。

処理ステップ:

  1. リクエスト検証: name(1〜100 文字必須)、allowedDomains(1 件以上必須、形式チェック)。違反は 400 VALIDATION_ERROR
  2. projects INSERT(ウィジェット公開鍵を発行、valid=1)
  3. オーナー自動 admin 行 INSERT: INSERT INTO project_users(user_id=actor.contractOwnerUserId, project_id=newPid, contract_owner_user_id=actor.contractOwnerUserId, role='admin', valid=1, granted_at=now(), granted_by=actor.userId)
  4. 監査ログ project.created_with_owner_admin(metadata に { projectId, contractOwnerUserId }、retention_class=general)を記録

擬似コード:

async function createProject(actor: Principal, req: CreateProjectRequest) {
  requireOwner(actor); // E-AUTHZ-OWNER-ONLY
  validateCreateProjectRequest(req); // 400
  return await db.transaction(async (tx) => {
    const project = await tx.insert('projects', {
      ...req,
      contract_owner_user_id: actor.contractOwnerUserId,
      valid: 1,
    });
    // オーナー admin 行を自動 INSERT(project_users)
    await tx.insert('project_users', {
      user_id: actor.contractOwnerUserId,  // オーナー本人 = contract_owners.user_id = users.id
      project_id: project.id,
      contract_owner_user_id: actor.contractOwnerUserId,  // 認可境界クエリ高速化の冗長保持
      role: 'admin',
      valid: 1,
      granted_by: actor.userId,
      granted_at: now(),
    });
    await tx.insertAudit({
      action: 'project.created_with_owner_admin',
      target_type: 'project',
      target_id: project.id,
      metadata: { contractOwnerUserId: actor.contractOwnerUserId },
      retention_class: 'general',
    });
    return project;
  });
}

3.11 プロジェクト削除時の論理削除カスケード + 孤立メンバー cleanup(FR-030b)

DELETE /projects/{id} 実行時は SCR-004-M1 プロジェクト作成・編集モーダル(編集モード)のオーナー専有 DangerSectionのみ から起動可能。以下のトランザクション境界で論理削除する:

  1. 削除前のスナップショット: 当該プロジェクトに valid=1 で割当のあるメンバー userId 一覧を取得(オーナー含む)
  2. UPDATE project_users SET valid=0, updated_at=now() WHERE project_id=? AND valid=1(オーナー自身の admin 行も含む全行を論理削除)
  3. 孤立メンバーの抽出: 「ステップ 1 のメンバーのうち、他プロジェクト割当(valid=1 の project_users)が 0 件かつ contract_owners 行を持たない(非オーナー)もの」
  4. 孤立メンバーの全セッションを失効(UPDATE sessions SET revoked_at=now())+ 未使用招待トークンを失効(UPDATE access_tokens SET used_at=now() WHERE used_at IS NULL)+ UPDATE users SET valid=0, updated_at=now() で論理削除
  5. UPDATE projects SET status='deleted', valid=0, deleted_at=now(), updated_at=now() WHERE id=? + 関連テーブル(allowed_domains / faqs / inquiries / question_logs / project_quota_limits)を valid=0, updated_at=now() に伝播(usage_metering は法定保管 7 年のため論理削除せず保持)
  6. 監査ログ project.logical_delete(metadata に { projectId, logicallyDeletedUserIds: [...] }、retention_class=general)を記録

オーナーの users 行は本処理の論理削除対象外(contract_owners 行を持つユーザーは常に valid=1 維持)。当該プロジェクトに対するオーナー自身の project_users admin 行は他のメンバー行と同様に valid=0 に論理削除される。論理削除データは updated_at < now() - 90d AND valid=0 の物理削除バッチ(DD13_バッチ・非同期処理.html §3.12)で 90 日後に物理削除される。

3.11a プロジェクト連絡先メール確認フロー(FR-033a / SCR-019)

SCR-004-M1(新規モードで初期入力、または編集モードで projects.contact_email を変更)した時に、TPL-PROJECT_CONTACT_VERIFY確認メールを送信する。受信者はSCR-019へ着地し、POST /auth/contact-verifications/{token} で確認完了する。

// (1) 連絡先メール変更時(POST/PATCH /projects 内)
await env.DB.prepare(
  `UPDATE projects SET contact_email = ?1, contact_email_verified_at = NULL, updated_at = ?2 WHERE id = ?3`
).bind(newContactEmail, now, projectId).run();
const rawToken = await generateToken('contact_verify', { projectId }, env, 24 * 3600);
await enqueueEmail({
  template: 'TPL-PROJECT_CONTACT_VERIFY',
  to: newContactEmail,
  verifyUrl: `${env.WEB_BASE}/auth/contact-verify?token=${encodeURIComponent(rawToken)}`,
});

// (2) SCR-019 着地時(POST /auth/contact-verifications/{token})
const { payload } = await verifyToken(rawToken, 'contact_verify', env);  // 410 if expired/used
const { projectId } = payload;
await env.DB.batch([
  env.DB.prepare(`UPDATE projects SET contact_email_verified_at = ?1, updated_at = ?1 WHERE id = ?2`).bind(now, projectId),
]);
await consumeToken(rawToken, 'contact_verify', env);
await writeAudit({ action: 'project.contact_email_verified', target_project_id: projectId });

注意:

  • 連絡先メール再変更時は旧トークンを consumeToken() で失効させ、新トークンを発行する
  • contact_email_verified_at が NULL の間はウィジェット表示に用いない(FR-033c)
  • 受信者(連絡先メール所有者)はアカウントを持つ必要がない ── 第三者の共有メールアドレスでも本フローを進められる
  • Turnstile は不要(状態変更は単純な verified_at セットのみ、トークン HMAC + レート制限で十分)
  • 詳細なトークン発行ロジックは DD07_トークン発行・検証.html §3.3b を参照

3.12 90 日後物理削除バッチ(概要)

詳細は DD13_バッチ・非同期処理.html §3.12 を正本とする。日次バッチで valid=0 AND updated_at < now() - 90d の行を物理削除する。users 行が物理 DELETE されると ON DELETE CASCADE で関連 contract_owners / project_users / sessions / access_tokens / terms_agreements 等が連鎖物理削除される。projects 行が物理 DELETE されると ON DELETE CASCADE で関連 faqs / inquiries / chat_* / project_quota_limits 等が連鎖物理削除される。billing_subscriptions / usage_metering は物理削除バッチの対象外(電子帳簿保存法 7 年保持、永久維持。usage_metering はプロジェクト単位行も含め 7 年保持)。匿名化モード(contract_owners.data_deletion_mode='anonymize')は物理削除前に匿名化処理を実施する。

3.5 FAQ 状態の保存

FAQ の状態を専用 API ではなく PATCH の status フィールドで保存する方針と、その値域・遷移ルールを定めます。

faqs.status は CHECK 制約（draft / published / hidden / deleted）で値域固定。状態は SCR-006 編集画面の「状態」ラジオで選択した値を PATCH /api/v1/faqs/{id} の status フィールドで保存する。draft / published / hidden は相互に自由遷移でき、専用の公開 API・状態遷移ガードは持たない。domain/faq-status.ts の責務は status 値域の妥当性検証のみ（deleted への直接遷移は DELETE /faqs/{id} 経由に限定）。status='published' を選択して保存する操作が管理者(admin)・メンバー(member)による公開前の内容確認（FR-045）を兼ね、UI 上は保存前に確認ダイアログを表示する。published 設定時は published_at をサーバ側で記録する。

3.6 FTS5 検索

公開 FAQ の全文検索を FTS5 virtual table + BM25 ランキングで実現する構成と検索クエリを示します。

faq_search_fts は SQLite の FTS5 virtual table で構築し、faqs.title || faqs.body をインデックス対象とする。FAQ 公開・更新時に同期更新。検索クエリは BM25 ランキングを利用：

SELECT f.*, bm25(faq_search_fts) AS rank
FROM faq_search_fts JOIN faqs f ON f.rowid = faq_search_fts.rowid
WHERE faq_search_fts MATCH ?1 AND f.project_id = ?2 AND f.status = 'published'
ORDER BY rank LIMIT 20

詳細は DD03_AI回答パイプライン.html §3.2 関連度計算を参照。

AI 推論時のリランキングに用いる埋め込みベクトルを、FAQ 公開時に事前計算して保存する処理を定めます。

3.8 一括インポート（faq.bulk_import）

CSV を R2 へステージングし Queue 経由で consumer が分割取込する一括インポートの判定ロジックと、失敗行の扱いを定めます。文字コード事前検証・列定義・テンプレート生成・行単位の INSERT/UPDATE 判定が要点です。

CSV ファイル（UTF-8、BOM 許容）→ R2 ステージング → Queue 投入 → consumer が分割 INSERT。1 リクエスト最大 1000 件。JSON 形式は受理しない（不正形式は E-INPUT-CSV-INVALID）。UI は SCR-006-002 FAQ CSV インポートモーダル（POST /faqs/import）。詳細は DD13_バッチ・非同期処理.html を参照。

文字コード事前検証（クライアント側）: SCR-006-002 のドラッグ&ドロップ / ファイル選択の時点で、ファイル先頭バイト（BOM）および本文のバイト列から文字コードを判定し、UTF-8（BOM 許容）以外はアップロードせず即エラー（E-INPUT-CSV-INVALID / MSG-SCR-006-002-ERR-005、検出した文字コード名を併記）。これによりサーバ側の文字化け INSERT を未然に防ぐ。

CSV 列定義（ヘッダ行必須）: FAQ ID, 質問, 回答, カテゴリ。status（状態）列は受理しない。

テンプレート生成（GET /faqs/import/template）: ヘッダ行のみ（FAQ ID,質問,回答,カテゴリ）を text/csv; charset=utf-8（BOM 付与）でストリーム返却。動的データなし（静的文字列）、Queue 不経由の同期レスポンス。

FAQ ID 行判定ロジック（consumer 側、行単位）。consumer は CSV 各行について FAQ ID の有無と当該契約境界（contract_owner_user_id）での突合結果に応じて INSERT / UPDATE / 失敗を分岐する。質問・回答の欠落 / 文字数超過はいずれの分岐でも当該行を失敗させる（カテゴリは任意）。

sequenceDiagram
  participant Q as Queue consumer
  participant D as D1 (faqs)

  Q->>Q: 行をパース(FAQ ID / 質問 / 回答 / カテゴリ)
  alt 質問・回答が欠落 / 文字数超過
    Q->>Q: 当該行を失敗(E-INPUT-CSV-INVALID)
  else FAQ ID 空欄
    Q->>D: 新規 INSERT(status='draft', version=1)
    D-->>Q:
  else FAQ ID あり
    Q->>D: 当該契約境界内の faqs.id を突合
    D-->>Q: 突合結果(一致 / 不存在)
    alt 一致
      Q->>D: UPDATE(status 維持・faq_revisions 追記・version++)
      D-->>Q:
    else 不存在
      Q->>Q: 当該行を失敗(E-INPUT-CSV-FAQID-NOTFOUND)
    end
  end
部分成功失敗行があっても他行の取込は継続する（部分成功）。UPDATE 分岐では status を維持し、faq_revisions に改訂行を追記して version をインクリメントする。

失敗行は「行番号 + エラー理由」の構造化リストとしてジョブ結果（GET /faqs/import/jobs/{jobId} 等のポーリング応答）に含め、SCR-006-002 進捗パネル内のスクロール領域に画面表示する（エラー CSV のダウンロードは行わない）。

3.10 プロジェクト作成(FR-015e / FR-030a)

プロジェクト作成処理を定めます。オーナーは owners 行存在(isOwner)による全権でプロジェクトを管理するため、作成時に自身の割当行(project_user_assignments)を作る必要はありません。1 トランザクション内で projects INSERT と監査ログ記録を行う流れを示します。

POST /projects 実行時は SCR-004-001(新規作成モード)から name / allowedDomains 等の基本情報のみを受け取る(initialAdmins フィールドは受け取らない)。オーナーは isOwner 全権で当該プロジェクトを管理でき、プロジェクトへの自動 admin 割当行(旧 v1.10)は作成しない。他者をプロジェクト管理者として割り当てる操作は、プロジェクト作成後に SCR-009-001 経由(POST /projects/{id}/members + role='admin')で project_user_assignments に行う。

sequenceDiagram
  participant U as オーナー
  participant W as worker-main-api
  participant D as D1

  U->>W: POST /projects(name / allowedDomains)
  W->>W: requireOwner + リクエスト検証
  alt 検証 NG
    W-->>U: 400 VALIDATION_ERROR
  else 検証 OK
    W->>D: projects INSERT(公開鍵発行・valid=1)
    D-->>W: projectId
    W->>D: 監査ログ project.create(retention_class=general)
    D-->>W:
    W-->>U: 作成したプロジェクト
  end

擬似コード(上記トランザクションの実体、INSERT 各列値を含む):

async function createProject(actor: Principal, req: CreateProjectRequest) {
  requireOwner(actor); // E-AUTHZ-OWNER-ONLY(owners 行存在 = isOwner)
  validateCreateProjectRequest(req); // 400
  return await db.transaction(async (tx) => {
    const project = await tx.insert('projects', {
      ...req,
      contract_owner_user_id: actor.contractOwnerUserId,  // → owners.id
      valid: 1,
    });
    // オーナーは isOwner 全権で管理するため、自身の割当行(project_user_assignments)は作成しない
    await tx.insertAudit({
      action: 'project.create',
      actor_type: 'owner', actor_id: actor.actorId,
      target_type: 'project',
      target_id: project.id,
      metadata: { contractOwnerUserId: actor.contractOwnerUserId },
      retention_class: 'general',
    });
    return project;
  });
}

3.11 プロジェクト削除時の論理削除カスケード + 孤立メンバー cleanup(FR-030b)

プロジェクト削除時に、関連行を論理削除へ伝播させ、他に割当の無くなった孤立メンバーを併せて無効化する 1 トランザクションの流れを定めます。DELETE /projects/{id} は SCR-004-001 プロジェクト作成・編集モーダル(編集モード)のオーナー専有 DangerSection のみ から起動可能です。

sequenceDiagram
  participant U as オーナー
  participant W as worker-main-api
  participant D as D1

  U->>W: DELETE /projects/{id}
  W->>D: 割当プロジェクトユーザー一覧を取得(valid=1)
  D-->>W: 割当一覧
  W->>D: project_user_assignments を valid=0 へ(当該プロジェクトの全割当行)
  D-->>W:
  W->>D: 孤立プロジェクトユーザーを抽出(他 valid=1 割当 0 件)
  D-->>W: 孤立ユーザー一覧
  opt 孤立プロジェクトユーザーあり
    W->>D: 孤立ユーザーの sessions 失効・未使用 access_tokens 失効・project_users を valid=0
    D-->>W:
  end
  W->>D: projects を status='deleted', valid=0, deleted_at へ + 関連テーブルへ valid=0 伝播
  D-->>W:
  W->>D: 監査ログ project.logical_delete(retention_class=general)
  D-->>W:
  W-->>U: 削除完了
論理削除の伝播先と除外関連テーブル(allowed_domains / faqs / inquiries / question_logs / project_quota_limits)へ valid=0, updated_at=now() を伝播する。usage_metering は法定保管 7 年のため論理削除せず保持する。孤立プロジェクトユーザーの抽出条件は「当該プロジェクトに割り当たっていたプロジェクトユーザーのうち、他プロジェクト割当(valid=1 の project_user_assignments)が 0 件のもの」。project_users はオーナーを含まない完全分離マスタのため、抽出対象は常にプロジェクトユーザーのみとなる。監査ログ metadata には { projectId, logicallyDeletedProjectUserIds: [...] } を含める。

オーナー(owners 行)は project_users マスタの外にあり、本処理の論理削除対象に含まれない(owners は常に valid=1 維持)。オーナーは isOwner 全権でプロジェクトを管理し自身の割当行を持たないため、削除されるのは当該プロジェクトの project_user_assignments 行のみである。論理削除データは updated_at < now() - 90d AND valid=0 の物理削除バッチ(DD13_バッチ・非同期処理.html §3.12)で 90 日後に物理削除される。

3.11a プロジェクト連絡先メール確認フロー(FR-033a / SCR-019)

プロジェクト連絡先メールを変更したときに、確認メールのトークンで到達確認を行い contact_email_verified_at をセットするまでの流れを定めます。

SCR-004-001(新規モードで初期入力、または編集モードで projects.contact_email を変更)した時に、TPL-PROJECT_CONTACT_VERIFY確認メールを送信する。受信者はSCR-019へ着地し、POST /auth/contact-verifications/{token} で確認完了する。

sequenceDiagram
  participant U as オーナー / 管理者
  participant W as worker-main-api
  participant D as D1
  participant M as メール送信
  participant R as 連絡先メール受信者

  U->>W: POST/PATCH /projects(contact_email 変更)
  W->>D: contact_email 更新 + contact_email_verified_at=NULL
  D-->>W:
  W->>W: contact_verify トークン発行(有効 24h)
  W->>M: TPL-PROJECT_CONTACT_VERIFY を enqueue
  M-->>R: 確認メール(verifyUrl)
  R->>W: POST /auth/contact-verifications/{token}(SCR-019)
  W->>W: verifyToken
  alt 期限切れ / 使用済み
    W-->>R: 410
  else 有効
    W->>D: contact_email_verified_at をセット
    D-->>W:
    W->>W: consumeToken + 監査ログ project.contact_email_verified
    W-->>R: 確認完了
  end

下記の擬似コードは上記フローの実体(SQL バインドとトークン操作の詳細):

// (1) 連絡先メール変更時(POST/PATCH /projects 内)
await env.DB.prepare(
  `UPDATE projects SET contact_email = ?1, contact_email_verified_at = NULL, updated_at = ?2 WHERE id = ?3`
).bind(newContactEmail, now, projectId).run();
const rawToken = await generateToken('contact_verify', { projectId }, env, 24 * 3600);
await enqueueEmail({
  template: 'TPL-PROJECT_CONTACT_VERIFY',
  to: newContactEmail,
  verifyUrl: `${env.WEB_BASE}/auth/contact-verify?token=${encodeURIComponent(rawToken)}`,
});

// (2) SCR-019 着地時(POST /auth/contact-verifications/{token})
const { payload } = await verifyToken(rawToken, 'contact_verify', env);  // 410 if expired/used
const { projectId } = payload;
await env.DB.batch([
  env.DB.prepare(`UPDATE projects SET contact_email_verified_at = ?1, updated_at = ?1 WHERE id = ?2`).bind(now, projectId),
]);
await consumeToken(rawToken, 'contact_verify', env);
await writeAudit({ action: 'project.contact_email_verified', target_project_id: projectId });

注意:

  • 連絡先メール再変更時は旧トークンを consumeToken() で失効させ、新トークンを発行する
  • contact_email_verified_at が NULL の間はウィジェット表示に用いない(FR-033c)
  • 受信者(連絡先メール所有者)はアカウントを持つ必要がない ── 第三者の共有メールアドレスでも本フローを進められる
  • Turnstile は不要(状態変更は単純な verified_at セットのみ、トークン HMAC + レート制限で十分)
  • 詳細なトークン発行ロジックは DD07_トークン発行・検証.html §3.3b を参照

3.12 90 日後物理削除バッチ(概要)

論理削除済みデータを 90 日後に物理削除する日次バッチの概要を示します。詳細ロジックは DD13 が正本のため、本節は対象と CASCADE の範囲のみを述べます。

詳細は DD13_バッチ・非同期処理.html §3.12 を正本とする。日次バッチで valid=0 AND updated_at < now() - 90d の行を物理削除する。owners 行が物理 DELETE されると ON DELETE CASCADE で関連 project_users / project_user_assignments / sessions / access_tokens / terms_agreements(actor_type='owner' 分)等が連鎖物理削除される。project_users 行が物理 DELETE されると project_user_assignments / 当該ユーザーの sessions / access_tokens 等が連鎖物理削除される。projects 行が物理 DELETE されると ON DELETE CASCADE で関連 faqs / inquiries / chat_* / project_quota_limits 等が連鎖物理削除される。billing_subscriptions / usage_metering は物理削除バッチの対象外(電子帳簿保存法 7 年保持、永久維持。usage_metering はプロジェクト単位行も含め 7 年保持)。匿名化モード(owners.data_deletion_mode='anonymize')は物理削除前に匿名化処理を実施する。

3.6 関連する横断設計

本機能が依存する横断的な実装(トークン・認可ヘルパ・監査ログ・暗号化)の正本 DD を対応付けます。各小節は委譲先と本書側で守るべき呼び出し規約だけを示します。

3.6.1 トークン発行・検証

本機能のメール確認・パスワードリセット・有効化トークンは DD07 の保存方式を利用します。

メール確認・パスワードリセット・管理者ユーザー有効化（activation）の各トークンは DD07_トークン発行・検証.html の HMAC-SHA256 保存方式を利用する。一回限りトークンは consumeToken() を検証直後に必ず呼ぶ。

3.6.2 認可ヘルパ

テナント・プロジェクト・ロールの認可判定は DD08 のヘルパに委譲します。

requireTenant / requireProject / requireRole の各ヘルパは DD08_認可ヘルパ.html を利用する。

3.6.3 監査ログ書込

本機能が発行する auth.* 系 action コードは DD09 の writeAudit() で記録し、保持クラスを general に固定します。

auth.signup / auth.login_success / auth.login_failure / auth.logout / auth.password_reset / auth.re_auth.success / auth.ip_blocked / auth.terms_agree などの action コードはすべて DD09_監査ログ書込・完全性検証.html の writeAudit() で書込み、retention_class=general とする。

3.6.4 暗号化・鍵管理

PII の暗号化と検索用ハッシュ列の方式は DD10 が正本です。

email / phone などの PII はオーナー派生鍵で AES-256-GCM 暗号化し、検索用に HMAC-SHA256 列（email_hmac 等）を併設する。詳細は DD10_暗号化・鍵管理.html を参照。

3.7 オーナー / メンバー / プロジェクト割当

オーナーとプロジェクトユーザーを 2 マスタでどう完全分離し、契約・プロジェクトへどう紐付けるかの実装規約を定めます。次の箇条書きは、オーナー判定が M_OWNERS 行存在であること、新規オーナー登録時の同一トランザクション処理を示します。

  • オーナー判定(isOwner) = M_OWNERS 行(valid=1)が存在すること。M_OWNERS.id が契約境界キー本体であり、オーナーは M_OWNERS マスタで認証・契約属性を一元管理する(プロジェクトユーザーとは別マスタ・別認証系統)
  • プロジェクトユーザーは M_PRJ_USERS マスタで認証情報を持ち(オーナーは含まない完全分離)、M_PRJ_USERS.contract_owner_user_id(→ M_OWNERS.id)経由で所属契約に紐付く
  • プロジェクトユーザーの権限は M_PRJ_USER_ASGN.role(admin / member)とプロジェクト割当の組合せで管理する。割当行は M_PRJ_USER_ASGN(project_user_id → M_PRJ_USERS, project_id, contract_owner_user_id → M_OWNERS.id, role)
  • オーナーは M_OWNERS 行存在(isOwner=true)による全権 bypass で認可され、プロジェクトへの自動 admin 割当行(旧 v1.10)は持たない。プロジェクト別の admin は M_PRJ_USER_ASGN の割当でのみ付与される
  • 新規オーナー登録時のトランザクション: INSERT INTO M_OWNERS(id, contract_status='active', valid=1, ...) を BEGIN IMMEDIATE で実行(M_OWNERS.id = contractOwnerUserId = 契約境界キー、ULID)。認証情報(メール / パスワード)も M_OWNERS 行に持つ
  • 共有概念は ../../共有/共有概念.html を正本とする

3.7a メンバーアカウント有効化フロー(SCR-018 / FR-016d / FR-016e)

招待されたメンバー本人が招待リンク先で氏名 + 初回パスワード + 規約同意を入力してアカウントを有効化するフローの実装ロジックを定めます。次のシーケンス図は、招待時の予約行・トークン作成から、本人によるプレビュー・有効化、招待再送、既存アカウント名前解決による有効化バイパスまでの全体像を示します。各 API の DB 操作の実体は続く小節のコードで定めます。

3.7a.1 招待時の DB 状態作成(POST /projects/{id}/members)

招待時に作成する DB 状態を定めます。次のコードは、氏名・パスワード未設定の M_PRJ_USERS 行(pending_activation)、予約状態の M_PRJ_USER_ASGN 行(valid=0)、有効期限 7 日の T_ACCESS_TOKENS(activation)を同一トランザクションで作成する実体です。

await db.transaction(async (tx) => {
  // (1) M_PRJ_USERS 行を name=NULL で作成(個人情報原則により、招待者は氏名を入力しない)
  const projectUserId = ulid()
  await tx.projectUsers.insert({
    id: projectUserId,
    email_encrypted: encrypt(email, ownerKey),
    email_hmac: hmac(email, ownerKey),
    name: null,                              // SCR-018 で本人が入力
    password_hash: null,                     // SCR-018 で本人が入力
    status: 'pending_activation',
    valid: 1,
  })
  // (2) M_PRJ_USER_ASGN 予約行(valid=0)
  await tx.projectUserAssignments.insert({
    project_user_id: projectUserId,
    contract_owner_user_id: contractOwnerUserId,
    project_id: projectId,
    role: invitedRole,                       // 'admin' | 'member'
    valid: 0,                                // SCR-018 完了時に 1 に更新
  })
  // (3) T_ACCESS_TOKENS(activation, 7d, meta に invitedProjectId / invitedRole を保持)
  const rawToken = generateRawToken()
  await tx.accessTokens.insert({
    id: ulid(),
    actor_type: 'project_user', actor_id: projectUserId,
    token_hash: hmac(rawToken, hashKey),
    purpose: 'activation',
    meta: JSON.stringify({ invitedProjectId: projectId, invitedRole }),
    expires_at: addDays(now(), 7),
  })
  // (4) 招待メール送信(TPL-ADMIN_USER_REGISTER または TPL-PROJECT_ADMIN_INVITE)
  await enqueueEmail({ template, activationUrl: buildActivationUrl(rawToken) })
})

3.7a.2 プレビュー API(POST /auth/invitations/{token}/preview)

有効化ページ表示前に招待情報を返すプレビュー API の実装です。未認証可とし、トークン検証後に招待情報パネル用のメタ情報を返却します。次のコードがその返却ロジックの実体です。

const { tokenRecord, projectUserId } = await verifyToken(rawToken, 'activation', env)
const meta = JSON.parse(tokenRecord.meta)
const project = await db.M_PROJECTS.findById(meta.invitedProjectId)
const owner = await db.M_OWNERS.findById(project.contract_owner_user_id)
const user = await db.projectUsers.findById(projectUserId)
return {
  projectUserId,
  email: decrypt(user.email_encrypted, ownerKey),
  projectId: project.id,
  projectName: project.name,
  invitedRole: meta.invitedRole,
  inviterOwnerName: owner.name,
  expiresAt: tokenRecord.expires_at,
}

3.7a.3 有効化 API(POST /auth/invitations/{token}/activate)

本人入力を受けてアカウントを有効化する API の実装です。次のコードは、Turnstile・入力検証の後に、M_PRJ_USERS 更新・T_TERMS_AGREE INSERT・M_PRJ_USER_ASGN.valid=1・consumeToken()・監査ログを 1 つのトランザクションで確定する実体を示します。

await verifyTurnstile(req.turnstileToken)
const { tokenRecord, projectUserId } = await verifyToken(rawToken, 'activation', env)
const meta = JSON.parse(tokenRecord.meta)
validateDisplayName(req.displayName)        // 1〜100 文字
validatePassword(req.password)              // FR-006 強度要件
if (req.password !== req.passwordConfirm) throw E_INPUT_VALIDATION
if (!req.termsAgreed || !req.privacyAgreed) throw E_INPUT_REQUIRED

await db.transaction(async (tx) => {
  await tx.projectUsers.update(projectUserId, {
    name: req.displayName.trim(),
    password_hash: await hashPassword(req.password),
    status: 'active',
    updated_at: now(),
  })
  // 利用規約・プライバシーポリシーを doc_type 別の 2 行で記録
  await tx.termsAgreements.insertMany([
    { id: ulid(), actor_type: 'project_user', actor_id: projectUserId, doc_type: 'terms_of_service', terms_version: latestTermsVersion, agreed_at: now() },
    { id: ulid(), actor_type: 'project_user', actor_id: projectUserId, doc_type: 'privacy_policy', terms_version: latestPrivacyVersion, agreed_at: now() },
  ])
  await tx.projectUserAssignments.update({
    project_user_id: projectUserId,
    project_id: meta.invitedProjectId,
  }, { valid: 1, updated_at: now() })
  await tx.accessTokens.update(tokenRecord.id, { used_at: now() })
  await tx.auditLogs.insert({
    action: 'user.activation_completed',
    actor_type: 'project_user', actor_id: projectUserId,
    target_type: 'project_user', target_id: projectUserId,
    meta: JSON.stringify({ project_id: meta.invitedProjectId, role: meta.invitedRole }),
    retention_class: 'general',
  })
})
return { projectUserId, redirectUrl: '/auth/login' }

3.7a.4 招待再送(POST /members/{id}/resend-invitation)

期限切れ等で再招待するときのトークン再発行ロジックを定めます。旧トークンを失効させ、招待時の meta を引き継いで新規トークンを発行します。

旧 T_ACCESS_TOKENS を used_at=now() で失効させ、meta を同一(invitedProjectId / invitedRole)で新規トークンを発行。リクエストボディは空(displayName を受け付けない)。

3.7a.5 名前解決による既存アカウント招待(FR-021c)

同一オーナー配下に既存アカウントがある場合の有効化バイパスを定めます。氏名入力を伴う SCR-018 を経ず、割当のみを追加します。

既存アカウントが同一オーナースコープに存在する場合は、SCR-018 をバイパスして M_PRJ_USER_ASGN 行のみを valid=1 で追加 + 既存アカウントのログイン URL を含む簡易メールを送信(氏名 / パスワードは既存 M_PRJ_USERS 値を保持)。

3.7b プロジェクト割当解除とアカウント停止

メンバーをプロジェクトから外す処理と、最後の割当だった場合のアカウント自動停止の実装手順を定めます。次の手順は、保護チェックから論理削除・全セッション失効・監査記録までの順序を示します。

DELETE /projects/{id}/members/{userId}:

  1. requireProjectRole(..., 'admin') と自己・オーナー保護を実行する。
  2. 対象ユーザーの他プロジェクト有効割当数を取得する。
  3. 対象 M_PRJ_USER_ASGN 行を valid=0 にする。
  4. 他プロジェクト有効割当数が0件なら M_PRJ_USERS.valid=0、全セッション失効、未使用招待失効を同一トランザクションで実行する。
  5. project.member_removed を監査記録し、アカウント停止を伴う場合は account_deactivated=true をmetadataへ記録する。

3.8 退会フロー（参照のみ）

オーナーの退会から物理削除までの流れを概観します。詳細バッチは DD13 が正本で、本節は契約状態の遷移と削除予約のみを示します。

  • オーナーが退会要求 → UPDATE M_OWNERS SET contract_status='deleted_pending' + T_WITHDRAW_REQ レコード作成 + scheduled_deletion_at（30 日後）
  • 30 日経過後、§14.1.9 定期削除 cron が UPDATE M_OWNERS SET contract_status='deleted', valid=0 に更新し、後続バッチで配下 M_PRJ_USERS.valid=0 を揃える
  • 詳細は DD13_バッチ・非同期処理.html を参照

3.9 フロント URL ルーティング規約(FR-325)

利用者向け管理コンソール SPA の URL スコープ構成を定めます。次のルートツリーは、認証 / オーナー / プロジェクト / 共通 / エンドユーザーの 5 系列と、各系列のアクセス主体を示します。

利用者向け管理コンソールの SPA は スコープ別 4 系列 のルートツリーで構成する。スコープ判定はフロント router の beforeEach ガードと API ミドルウェア authenticate.ts + requireTenant の二重化で行う。

/auth/*                        ── 認証フロー(未認証可)
/owner/*                       ── 契約 WS(オーナー専有 = `M_OWNERS` 行存在 = isOwner)
/projects/:projectId/*         ── プロジェクト WS(`M_PRJ_USER_ASGN` 保持者)
/account/*                     ── 共通領域(認証済全員)
/eu/*                          ── エンドユーザー(認証コンソール外、トークン認証)

3.9.1 フロント router のガード(疑似コード)

スコープ別アクセス判定を行う router ガードの実装を示します。次の疑似コードは、未認証リダイレクト・オーナー専有スコープ・プロジェクト割当チェックの分岐の実体です。

router.beforeEach(async (to) => {
  const scope = parseScope(to.path)  // 'auth' | 'owner' | 'M_PROJECTS' | 'account' | 'eu'
  const me = await ensureSession()    // 未認証なら null

  if (scope === 'auth') {
    if (me) return resolveDefaultWorkspace(me)  // 認証済は既定 WS へリダイレクト(§4.4.7)
    return true
  }

  if (!me) return '/auth/login'

  if (scope === 'owner' && !me.isOwner) {
    return resolveDefaultWorkspace(me)  // 403 同等の扱い、UI 上は権限不足トーストを表示
  }

  if (scope === 'M_PROJECTS') {
    const projectId = to.params.projectId
    const assignment = me.assignments.find(a => a.project_id === projectId)
    if (!me.isOwner && !assignment) {
      // 最後の割当解除時は M_PRJ_USERS.valid=0 となるため、通常は assignments.length >= 1
      return resolveDefaultWorkspace(me)
    }
  }

  return true
})

3.9.2 既定ワークスペース決定(§4.4.7 と同期)

ログイン後や権限不足時に遷移する既定ワークスペースの決定ロジックを定めます。次の評価順は、前回ワークスペース・オーナー・単一/複数プロジェクトの優先順位を示します。

resolveDefaultWorkspace(me) は次の順で評価する:

  1. localStorage の lastWorkspace が現在の M_PRJ_USER_ASGN で有効ならその WS のホーム
  2. オーナーかつ未保存なら /owner (SCR-016)
  3. プロジェクトユーザーで M_PRJ_USER_ASGN 行が 1 件なら /projects/{onlyRow.project_id} (SCR-008)
  4. プロジェクトユーザーで M_PRJ_USER_ASGN 行が複数なら最新 last_accessed_at のプロジェクト

プロジェクトユーザー削除はユーザーレベル論理削除(M_PRJ_USERS.valid=0)のため、現存するプロジェクトユーザー(M_PRJ_USERS.valid=1)は M_PRJ_USER_ASGN.valid=1 を 1 件以上満たし、「割当行が 0 件のケース」は構造的に発生しない。データ不整合等で WHERE valid=1 条件下で assignments.length === 0 を検出した場合のフェイルセーフは Sentry へ重大度 high を通知し /auth/login へ強制ログアウトする。

認証ミドルウェアの追加チェック: 認証成功直後に、プロジェクトユーザーは M_PRJ_USERS.valid=1 を確認し、valid=0 の場合は 401 でログイン拒否(セッション失効扱い)。これにより論理削除済みユーザーのセッションが残存しても再アクセスを防止する。オーナーの場合は M_OWNERS.valid=1 AND contract_status NOT IN ('deleted') を検証(deleted_pending は 30 日間の猶予期間中は許容)。オーナーとプロジェクトユーザーはセッションの actor_type('owner' / 'project_user')で別系統に判別する。

3.10 SCR-016 プロジェクト0件の空状態(FR-326)

プロジェクトが 1 件もないときの利用状況画面の表示ロジックを定めます。全体サマリーを出さず、作成 CTA のみを表示します。

GET /owner/projects/usage の summary.projectCount=0 の場合、SCR-016は全体サマリーと一覧を表示せず、「プロジェクトがまだありません」と「最初のプロジェクトを作成」の単一CTAだけを表示する。専用のオンボーディングAPIや進捗状態は保持しない。

3.11 契約共通項目の「利用中のプロジェクト」集計(FR-327)

契約共通項目に表示するプロジェクト件数の集計ロジックを定めます。月次上限・無料枠はプロジェクト単位のため本集計の対象外です。

契約共通項目の「利用中のプロジェクト: N」表示用に参照プロジェクト件数を返す。今月の利用上限・無料利用枠はプロジェクト単位のため対象外とする。

  • AI 利用コスト消化率等の契約共通指標: SELECT COUNT(*) FROM M_PROJECTS WHERE contract_owner_user_id = :contractOwnerUserId AND deleted_at IS NULL(契約配下の全プロジェクト数。契約全体に適用される指標のため、全プロジェクトが利用先となる)
  • 集約レスポンスは GET /me/onboarding-status と同一 API にまとめて返却(shared_assets_usage フィールド)し、SCR-016 の 1 ロードで全データを揃える

N=0 の場合のラベル切替は基本設計 01_画面設計/index.html §4.4.4 のメッセージ ID 体系(MSG-SCR-016-LBL-005 / LBL-006)に従う。

5.1 結合テスト例（ロックアウト）

ログイン成功とロックアウトの結合テストの書き方を例示します。次のコードは、正常ログインで Cookie が発行されること・5 回失敗で 423 になることを検証する実体です。

// tests/integration/auth/login.test.ts
describe('POST /api/v1/auth/login', () => {
  it('正しい資格情報でログイン成功', async () => {
    await seedAccount({ email: 'admin@example.com', password: 'P@ss12345678', role: 'admin' });
    const res = await (await worker).fetch('http://localhost/api/v1/auth/login', {
      method: 'POST', body: JSON.stringify({ email: '...', password: '...' }),
    });
    expect(res.status).toBe(200);
    expect(res.headers.get('Set-Cookie')).toContain('session=');
  });
  it('5 回失敗でロックアウト', async () => {
    for (let i = 0; i < 5; i++) {
      await (await worker).fetch('...', { method: 'POST', body: JSON.stringify({ ..., password: 'wrong' }) });
    }
    const res = await (await worker).fetch('...', { method: 'POST', body: JSON.stringify({ ..., password: 'correct' }) });
    expect(res.status).toBe(423);  // LOCKED_OUT
  });
});

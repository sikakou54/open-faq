DD12: ウィジェット配信

このページは、エンドユーザー向けウィジェット(widget.js / iframe)と worker-widget-api による配信・チャット UI の実装方針を定義します。 bootstrap での公開鍵・許可ドメイン検証、質問送信(/ask)から AI 応答・未解決質問登録までのリクエストフロー、性能目標・レート制限・質問受付停止・ウィジェット UI 状態遷移を扱います(FR-150〜156 / FR-156a〜d / FR-070〜077 / FR-194 / AC-015 / AC-025 / AC-033)。

版数 v1.6更新 2026-06-17承認済

3. 詳細設計本文

ウィジェット API のリクエストフロー、配信構成、実装モジュール、性能目標、応答経路最適化、レート制限・質問受付停止、ウィジェット UI 状態遷移を順に定義します。以降の各小節は、bootstrap から質問送信・AI 応答までの処理とその最適化・制御を扱います。

3.1 ウィジェット API リクエストフロー

ページロード時の bootstrap(公開鍵 pk_live_xxx と Origin 検証 → セッション発行)から、質問送信(/ask)を経て AI 応答または未解決質問登録に至るまでの流れを定義します。次のシーケンス図で、許可ドメイン判定の分岐と回答可能 / 未解決の分岐を示します。

sequenceDiagram
  participant E as エンドユーザー
  participant W as widget.js (iframe 内)
  participant API as worker-widget-api
  participant K as KV (M_allowed_domains / threshold / session)
  participant D as D1
  participant AI as Workers AI

  E->>W: ページ表示時にロード
  W->>API: POST /widget/v1/bootstrap (pk_live_xxx, Origin)
  API->>K: GET M_allowed_domains:{project_id}
  K-->>API: 許可ドメイン一覧(ヒット / ミス)
  opt キャッシュミス
    API->>D: SELECT M_projects
    D-->>API: プロジェクト設定
  end
  alt Origin が許可ドメイン
    API->>K: SET widget-session:{sid} (TTL 1h)
    API-->>W: { session_token, project_config }
    W-->>E: 右下に丸型ランチャーバッジを表示
  else Origin が不許可
    API-->>W: 403 (ウィジェット非表示)
  end
  E->>W: バッジをクリック / Enter / Space
  W-->>E: チャット UI を展開
  E->>W: 質問入力 → 送信
  W->>API: POST /widget/v1/ask { question, session_token }
  API->>D: FTS5 検索 (TP_faq_search_fts)
  D-->>API: FAQ 候補(最大 5 件)
  API->>K: GET ai_threshold:{contract_owner_user_id}:{project_id}
  K-->>API: しきい値
  API->>AI: 推論（質問 + FAQ 抜粋）
  AI-->>API: 推論結果(スコア / 回答)
  API->>D: INSERT H_question_logs (metering_billable=...)
  alt 回答可能
    API-->>W: { type: "answered", answer, ... }
    W-->>E: AI 回答を会話履歴へ追加
  else 未解決
    API->>D: INSERT T_inquiries
    API-->>W: { type: "unanswered", inquiryId, inquiryCode, contactEmail }
    W-->>E: 回答不能メッセージ + 連絡先表示（問い合わせ ID は描画しない）
  end
公開鍵と Origin の検証bootstrap は公開鍵 pk_live_xxx の検証と Origin が許可ドメインに一致するかを確認し、一致時のみ session_token を発行する。以降の /ask は session_token で認証する。検証ロジックの正本は基本設計(認証・認可設計)。

3.4 性能目標（NFR-101 / NFR-102）

各ウィジェット API の p95 応答目標と達成方策を定義します。次の表は、bootstrap と ask(FAQ ヒット / ミス)の目標値と KV キャッシュ・FTS5 等の達成手段の対応です。

API p95 目標 達成方策
POST /widget/v1/bootstrap < 200ms KV キャッシュ（60s）優先、D1 ヒット時のみフォールバック
POST /widget/v1/ask (FAQ ヒット) < 1000ms FTS5 検索 + Workers AI バインディング内呼出、最大 5 FAQ
POST /widget/v1/ask (FAQ miss) < 500ms AI 推論スキップ

3.5 ウィジェット応答経路の最適化

応答性能を満たすための KV キャッシュ・埋め込み事前計算・AI モデル選択の方針を示します。これらにより bootstrap / ask の p95 目標を達成します。

  • KV 60s キャッシュ: project_key, M_allowed_domains, ai_threshold
  • 埋め込みベクトル: FAQ 公開時に事前計算、KV または faq_embeddings テーブル保存
  • AI モデル: @cf/meta/llama-3.1-8b-instruct を第一選択（早い）、品質要件で llama-3.1-70b に切替可能

3.6 AI 推論タイムアウト + フォールバック

AI 推論が 5000ms を超えた場合に unanswered(reason='ai_timeout')へフォールバックする実装を定義します。次の擬似コードは Promise.race によるタイムアウト処理です。

const inferencePromise = provider.answer(input);
const timeoutPromise = new Promise<never>((_, reject) =>
  setTimeout(() => reject(new Error('AI_TIMEOUT')), 5000),
);
try {
  const result = await Promise.race([inferencePromise, timeoutPromise]);
  // ...
} catch (e) {
  if ((e as Error).message === 'AI_TIMEOUT') {
    // unanswered + 管理者ユーザー誘導
    return { type: 'unanswered', reason: 'ai_timeout' };
  }
  throw e;
}

3.7 レート制限（FR-194 / AC-033）

ウィジェット質問の契約単位レート制限と同時並列上限を定義します。レート制限ロジックの正本は DD11 とし、本節は適用値のみ示します。

ウィジェット質問レートは 50/min（警告）/ 60/min（拒否、429 RATE_LIMITED + Retry-After 付）。同時並列はオーナー 10 / グローバル 200。詳細は DD11_利用量計測・課金.html §3.1 参照。

3.7a 質問受付停止時のウィジェット応答（FR-122 / FR-136）

上限件数到達(FR-122)または支払方法ゲート(FR-136)に該当するプロジェクトのウィジェット質問を、AI 推論・FTS5 検索を実行せず即時停止応答する仕様を定義します。次の表で 2 つの停止契機の条件と reason、続けて停止応答の共通仕様を示します。

POST /widget/v1/ask の処理冒頭で、公開キーから解決した project_id について以下 2 つの契機のいずれか を満たす場合、AI 推論・FTS5 検索を実行せず 即時に停止応答する(KV usage-limit:<owner>:<projectId> 30s TTL を参照。判定はプロジェクト単位):

契機 条件 reason
上限件数到達(FR-122) 当月の 当該プロジェクト 質問数 ≥ M_project_quota_limits.question_monthly_limit(当該プロジェクト設定) question_monthly_limit_reached
支払方法ゲート(FR-136) 支払方法未登録(契約単位)かつ 当月の 当該プロジェクト 質問数 ≥ 当該プロジェクトの無料枠(M_project_quota_limits.free_quota or デフォルト 1,000) payment_method_required

停止応答の共通仕様:

項目 内容
応答コード 429(E-QUOTA-QUESTIONS-LIMIT)、Retry-After = 当月末 23:59:59 JST までの秒数
レスポンスボディ { type: "limited", reason: <上表>, contactEmail: <確認済みプロジェクト連絡先メール or null> }
ウィジェット表示 contactEmail あり → MSG-WIDGET-QUESTION-LIMIT(メール誘導)。null → MSG-WIDGET-QUESTION-LIMIT-NOEMAIL(フォールバック)
計上 H_question_logs に計上しない(metering_billable=false、停止質問はカウント外)
レート制限との区別 一時的なレート超過(60/min)は RATE_LIMITED「混み合っています」、本停止は別メッセージ
契約状態 contract_status は変更しない(active のまま、当該プロジェクトのウィジェット質問受付のみ停止。他プロジェクトは継続。契約サスペンション 423 とは別)
復帰 上限件数到達: 翌月リセット or オーナー / 当該プロジェクト管理者の上限引き上げ(SCR-021-001)。支払方法ゲート: 支払方法登録(Setup Intent 成功)で即時再開(契約単位の支払方法登録で全プロジェクトのゲートが解除)

連絡先メール表示は確認済み(M_projects.contact_email_verified_at IS NOT NULL)のもののみ。停止判定・通知ロジックの正本は上限件数到達 = DD11_利用量計測・課金.html §3.8、支払方法ゲート = DD13_バッチ・非同期処理.html §3.1.3。

3.7b ウィジェット UI 状態遷移（FR-156a〜d）

丸型ランチャーバッジによる開閉状態と、回答結果に応じた会話状態の遷移を定義します。以降の 2 つの表で開閉状態(collapsed ↔ expanded)と会話状態(normal / limited)の遷移を示します。

開閉状態

丸型ランチャーバッジの操作による collapsed と expanded の遷移と、各遷移時の UI 処理(会話履歴・入力内容のメモリ保持を含む)を示します。

現在状態 契機 次状態 UI 処理
collapsed(初期状態) 丸型ランチャーバッジのクリック / Enter / Space expanded 右下固定の直径56pxバッジを非表示にし、チャット UI を表示
expanded ヘッダーの閉じるボタン押下 collapsed チャット UI を非表示にし、丸型ランチャーバッジを再表示。会話履歴、入力内容、normal / limited の受付状態をメモリ上で保持

会話状態

/ask の応答(answered / unanswered / 制限 429)に応じた会話状態の遷移と UI 処理を示します。制限 429 では入力・送信を無効化します。

現在状態 契機 次状態 UI 処理
normal /ask が answered normal AI 回答を会話履歴へ追加し、入力を継続
normal /ask が unanswered normal T_inquiriesを管理用問い合わせID付きで登録し、MSG-WIDGET-UNANSWERED と確認済み連絡先メールを会話履歴へ追加。inquiryCode は描画せず、FAQ質問入力は継続
normal /ask が制限 429 limited MSG-WIDGET-QUESTION-LIMIT または MSG-WIDGET-QUESTION-LIMIT-NOEMAIL をシステム返信として追加し、入力・送信を無効化

ウィジェットヘッダーは全状態でウィジェットタイトル、受付状態、閉じるボタンのみを表示する。ハンバーガーメニューその他の操作メニュー、および SCR-010 利用規約・SCR-020 プライバシーポリシーへの導線は表示しない。閉じるボタンには aria-label="FAQチャットを閉じる"、丸型ランチャーバッジには aria-label="FAQチャットを開く" を設定する。

5.2 負荷試験

ウィジェット配信基盤の性能を検証する負荷シナリオを定義します。次の表は、均等分布・偏在・急増・AI 高並列・ピークの 5 シナリオと各々の検証目的です。

シナリオ 構成 目的
(S1) 均等分布 200 契約 × 50 FAQ 平均、ask 100 RPS 全体 ベースライン性能
(S2) 偏在 1 契約 10,000 FAQ + 199 契約 × 25 FAQ 大規模契約時の FTS5 / D1 限界
(S3) 急増 50 → 200 契約 / 5 分で線形増加 スケール時のオートスケール反応
(S4) AI 高並列 100 契約 × ask 5 RPS、AI 推論 200 並列上限到達 AI 並列上限・タイムアウト挙動
(S5) ピーク (オプション) 月次キャンペーン想定、200 → 400 RPS スパイク エラーバジェット消費量検証

合格基準: 全シナリオで p95 ≤ §3.4 目標値 + http_req_failed < 1%。

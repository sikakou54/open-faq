## 3. 詳細設計本文

### 3.1 画面詳細設計（参照のみ）

### 3.2 機能詳細設計（参照のみ）

チャット部屋の状態遷移（`open` / `waiting_user` / `waiting_admin` / `closed` 等）、メッセージ送受信、自動クローズ 6 段階、再入室フローは [../02_基本設計/02_API設計.md](../02_基本設計/02_API設計.md) および [../02_基本設計/03_テーブル設計.md](../02_基本設計/03_テーブル設計.md) を正本とする。

### 3.3 実装モジュール構成

```
worker-main-api/src/
├── routes/
│   └── chat-rooms.ts         # /chat-rooms/* CRUD + メッセージ送受信（管理者側）
├── handlers/
├── domain/
│   └── chat-room-status.ts   # 部屋状態遷移ガード（純関数）
├── repository/
│   ├── chat-rooms.ts
│   └── chat-messages.ts
└── middleware/
    └── authorize.ts          # requireInquiry / requireProject 連動

worker-widget-api/src/
├── routes/
│   ├── chat-rooms.ts         # /widget/v1/chat-rooms/* CRUD（エンドユーザー側）
│   └── reentry.ts            # SCR-027 連携、再入室トークン検証
└── middleware/
    └── widget-session.ts
```

### 3.4 再入室トークン

```ts
// 再入室トークンは長寿命のため、access_tokens テーブルに meta.inquiryId を内包
// 失効・ローテーション運用:
// - チャット部屋作成時に発行
// - 30 日経過で expires_at に達し失効
// - 部屋を closed → reopen した時に新トークン発行 (旧トークンは expires_at まで有効)
```

詳細は [DD08_トークン発行・検証.md](DD08_トークン発行・検証.md) §3.4 を参照。

### 3.5 自動クローズ評価（cron 5 分間隔）

5 分間隔で `chat_rooms` を走査し、最終アクティビティからの経過時間に応じて自動クローズ 6 段階を進める。詳細は [DD14_バッチ・非同期処理.md](DD14_バッチ・非同期処理.md) §14.1.6 を参照。

### 3.6 関連する横断設計

- 認可: [DD09_認可ヘルパ.md](DD09_認可ヘルパ.md) の `requireInquiry` でオーナー境界を検証
- 通知: メッセージ受信時に管理者へメール通知（[DD07_通知ロジック.md](DD07_通知ロジック.md)）
- 監査ログ: `chat.create` / `chat.message.send` / `chat.close` / `chat.reopen` を `retention_class=general` で記録

### 5.1 その他観点

| 観点 | 内容 |
|---|---|
| 単体 | `chat-room-status.ts` 全 6 段階自動クローズ判定 / 再入室時の新トークン発行 |
| 結合 | 管理者送信 → エンドユーザー再入室 → 返信 → 自動クローズ閾値到達 |
| 異常系 | 期限切れ再入室トークン使用時の 400 `TOKEN_EXPIRED` |
| 境界値 | 再入室トークン TTL 30 日ぴったり / 自動クローズ各段階の閾値 |
| 性能 | `POST /api/v1/chats/.../messages` p95 < 500ms（NFR-104） |
| ショートカット | SCR-013 で `Ctrl+Enter`（macOS では `Cmd+Enter`）でメッセージ送信 |

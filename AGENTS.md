# リポジトリ作業ガイド

## 概要

このリポジトリは `open-faq` の仕様書一式を管理するドキュメントリポジトリです。アプリケーションのソースコードを管理するリポジトリではありません。

仕様書は次の 2 システムを並列で管理します。

- `01_メインシステム/`: 利用者向け管理画面および FAQ ウィジェット
- `02_運営者システム/`: サービス運営者向け管理コンソール

完全な保守ルールは `CLAUDE.md` に定義されています。複数ドキュメントにまたがる変更を行う前に必ず確認してください。本ファイルと `CLAUDE.md` の内容が異なる場合は、`CLAUDE.md` を正本とします。

## リポジトリ構成

各システムは同じ 5 層のドキュメント構成を持ちます。

1. `01_要件定義/`: 要件および受入条件
2. `02_基本設計/`: 画面、API、テーブル、権限、エラー、メッセージ、トレーサビリティ、認証認可、セキュリティ、課金、メール
3. `03_詳細設計/`: 実装レベルの詳細設計
4. `04_運用設計/`: 監視、バックアップ、ログ、障害対応、リリース、運用手順
5. `05_future/`: MVP 後の将来仕様

その他の重要なファイル・ディレクトリ:

- `共有/共有概念.md`: システム間で共有する概念と正本の定義
- `番号整理対応表.md`: 番号変更および削除項目の履歴
- `99_script/`: 検証スクリプトおよび PDF 生成スクリプト
- `PDF出力/`: 生成された PDF 成果物

## 変更手順

仕様変更は次の順番で調査・反映します。

1. 要件定義
2. 基本設計
3. 詳細設計
4. 画面変更がある場合は画面遷移図
5. 運用設計および将来対応

依頼で一方のシステムや単一ドキュメントだけが指定されていても、両システムへの影響を確認してください。関連する基本設計ファイルは `CLAUDE.md` の「変更種別 → 影響ドキュメント・マトリクス」に従って特定します。

共有概念を複数箇所へ重複記載しないでください。`共有/共有概念.md` で指定された正本を更新し、他方のシステムでは参照のみを更新します。

## 番号体系

- 各システム内の FR および DD ファイル名は連番を維持します。
- 数値形式の SCR ID は連番を維持します。モーダルは親画面の ID に `-M1`、`-M2` のようなサフィックスを付けます。
- カテゴリ別番号帯や `SCR-AUTH` など、意味を持つ記号形式の画面 ID は維持します。
- ID を変更した場合は、リンク、参照、HTML 画面遷移図、索引、トレーサビリティ、メッセージ、エラー、検査スクリプトも更新します。
- 廃番項目は現行仕様から削除します。変更前後の履歴だけを `番号整理対応表.md` に残します。
- 外部契約で予約されている連携 IF 番号は、契約変更が明示されない限り再採番しません。

## 編集方針

- 有効な要件内容は、意味変更が必要な依頼でない限り変更しません。
- 変更範囲は番号、並び順、参照、廃番項目の削除を中心に限定します。
- 要件定義には WHAT を記載します。DDL、JSON Schema、cron 式、Worker 名、実装パラメータは記載しません。
- 詳細設計では、基本設計が正本とする画面、API、テーブル、権限、エラー、メッセージの定義を再掲せず参照します。
- PDF は生成物であり正本ではありません。先に Markdown または HTML を更新してください。

## 検証

仕様書を編集した後は、同期スモークチェックを実行します。

```sh
bash 99_script/check-spec-sync.sh
```

コミット前には差分の形式も確認します。

```sh
git diff --check
```

同期スモークチェックは SC-001〜SC-012 を実行し、ファイル連番、SCR 集合、共有概念、リンク、章節番号、削除済み ID の再混入などを検査します。

PDF の確認を依頼された場合は、変更されたドキュメントを PDF 化します。

```sh
bash 99_script/build_all_pdfs.sh --changed-only
```

Markdown の PDF 化には Pandoc と XeLaTeX、HTML の PDF 化には Chrome 系ブラウザのヘッドレス実行環境または WeasyPrint が必要です。

## コミット方針

- 大規模な再採番は、ステージ前には削除と追加の組み合わせに見える場合があります。`git status` を注意して確認してください。
- ファイル名変更、参照更新、検査スクリプト、`番号整理対応表.md` は関連する一式としてステージします。
- 無関係な生成 PDF やローカル成果物をコミットに含めません。
- コミットメッセージは仕様変更の内容が分かる簡潔な表現にします。

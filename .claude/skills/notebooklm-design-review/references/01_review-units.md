# レビュー実施単位

NotebookLMレビューは一度に全体をまとめて行わず、以下の単位に分けて実施する。

---

## 4.1 全体構成レビュー

**対象:**
- 要件定義全体 / 基本設計全体 / 詳細設計全体
- `index.md` / ディレクトリ構成 / ID体系

**目的:**
- ドキュメント構成が適切か
- 粒度が混在していないか
- 読み順が明確か
- 全体の責務分離ができているか
- 上位設計から下位設計への流れが自然か
- 課題や判断履歴がObsidianに保存されているか

---

## 4.2 成果物別レビュー

**対象:**
- 画面詳細設計 / API詳細設計 / バッチ処理詳細設計
- モジュール設計 / SQL設計 / DB詳細設計
- クラス図 / 状態遷移図 / 共通定義設計

**目的:**
- 各成果物が担当すべき内容だけを記載しているか
- 他成果物の責務を侵食していないか
- 基本設計から詳細設計へ正しく落とし込まれているか
- 実装に入れる粒度になっているか

---

## 4.3 処理フローレビュー（UI → API → モジュール → SQL → DB）

**対象:**
- 画面詳細設計 / API詳細設計 / モジュール設計 / SQL設計 / DB詳細設計

**目的:**
- 呼び出し関係が追跡できるか
- APIがSQLを直接呼んでいないか
- モジュールがSQL呼び出し責務を持っているか
- SQLがDB設計と整合しているか

---

## 4.4 バッチフローレビュー（バッチ → モジュール → SQL → DB）

**対象:**
- バッチ処理詳細設計 / モジュール設計 / SQL設計 / DB詳細設計

**目的:**
- バッチからSQLへの直接呼び出しがないか
- 多重起動制御・リトライ・異常終了時・再実行方針が定義されているか
- 実行周期・閾値が共通定義IDで参照されているか

---

## 4.5 共通定義レビュー

**対象:**
- `10_共通定義設計` と各詳細設計書全体

**目的:**
- 状態値・区分値・閾値・設定値が共通定義設計に集約されているか
- 各設計書から `CONST ID` を参照しているか
- 表示ラベルと内部値が混同されていないか
- DB保存値と画面表示名が混同されていないか

---

## 4.6 状態遷移レビュー

**対象:**
- 状態遷移図 / 共通定義設計 / DB詳細設計の状態カラム
- 状態更新API / 状態更新モジュール / 状態更新SQL

**目的:**
- 状態値の意味が共通定義設計に一元管理されているか
- 状態遷移条件が状態遷移図に定義されているか
- API / バッチ / モジュール / SQL と状態遷移が追跡できるか
- 未確定の状態遷移や課題がObsidianで管理されているか

---

## 4.7 Obsidian課題管理レビュー

**対象:**
- `99_obsidian_context/04_issues_and_solutions`
- `INDEX_課題一覧.md` / レビュー記録 / 設計判断メモ / Agent間引継ぎメモ

**目的:**
- レビュー指摘から発生した課題がObsidianに保存されているか
- 課題に発見元・根拠・影響範囲・解決方針が定義されているか
- 課題がレビュー記録・設計判断・関連成果物とリンクされているか
- 解決済み課題に対応内容・対応結果・再レビュー結果が記録されているか
- 未解決課題が一覧で管理されているか

---

## 4.8 再レビュー

**対象:**
- 前回指摘一覧 / 修正後の設計書 / Obsidian課題メモ

**目的:**
- 前回指摘された問題が修正されているか
- 修正により新たな不整合が発生していないか
- 修正内容が要件定義・基本設計に存在しない仕様追加になっていないか
- 未解消指摘がObsidian課題として保存されているか

# NotebookLMプロンプト集

各レビュー種別に対応するNotebookLMへの投入プロンプト。
`nlm_ask.py` で投入するか、NotebookLMのチャット欄に直接貼り付ける。

---

## 6.1 全体構成レビュー用プロンプト

```
あなたは、要件定義・基本設計・詳細設計の整合性を確認する外部レビュー担当です。

アップロードされているドキュメント全体を対象に、以下の観点でレビューしてください。

# レビュー観点

1. 要件定義・基本設計・詳細設計の粒度が適切に分離されているか
2. 要件定義に基本設計や詳細設計の粒度が混入していないか
3. 基本設計に詳細設計の実装粒度が混入していないか
4. 詳細設計が基本設計の概要粒度に留まっていないか
5. ドキュメント構成・読み順・ディレクトリ構成が自然か
6. ID体系に不整合や参照切れがないか
7. 要件定義 → 基本設計 → 詳細設計の流れが追跡できるか
8. 逆参照、つまり要件定義が基本設計や詳細設計を参照していないか
9. 基本設計と詳細設計に重複記載がないか
10. 詳細設計に、要件定義・基本設計にない仕様が混入していないか
11. 重要な設計判断・課題・解決方法がObsidianに保存されているか

# 出力形式

| 指摘ID | 重要度 | 指摘分類 | 対象ドキュメント | 該当箇所 | 指摘内容 | 根拠 | 推奨対応 | Obsidian課題化要否 | ステータス |
|---|---|---|---|---|---|---|---|---|---|

重要度は「高 / 中 / 低」で分類してください。
指摘分類は「粒度不適切 / 落とし込み漏れ / 上位設計にない仕様 / 重複 / 逆参照 / 責務混在 / トレーサビリティ不備 / Obsidian保存漏れ / 要確認」から選択してください。
根拠が不明な場合は断定せず、指摘分類を「要確認」にしてください。
提供されたソースのみに基づいて回答し、`===== <path> =====` セパレータからの相対パスを引用してください。
```

## 6.2 基本設計から詳細設計への落とし込みレビュー用プロンプト

```
あなたは、基本設計から詳細設計への落とし込みを確認する外部レビュー担当です。

アップロードされている基本設計書と詳細設計書を比較し、以下の観点でレビューしてください。

# レビュー観点

1. 基本設計で定義された画面が、画面詳細設計に漏れなく落とし込まれているか
2. 基本設計で定義されたAPIが、API詳細設計に漏れなく落とし込まれているか
3. 基本設計で定義されたバッチが、バッチ処理詳細設計に漏れなく落とし込まれているか
4. 基本設計で定義されたテーブルが、DB詳細設計に漏れなく落とし込まれているか
5. 基本設計で定義された処理が、API / バッチ / モジュール / SQL に適切に分解されているか
6. 詳細設計に、基本設計に存在しない仕様が追加されていないか
7. 詳細設計が基本設計の説明をコピーしただけになっていないか
8. 詳細設計が実装に必要な入出力・条件・分岐・例外を定義しているか
9. 落とし込み漏れや上位設計にない仕様がObsidian課題として保存されているか

# 出力形式

| 指摘ID | 重要度 | 基本設計ID | 詳細設計ID | 対象ドキュメント | 指摘内容 | 根拠 | 推奨対応 | Obsidian課題化要否 | ステータス |
|---|---|---|---|---|---|---|---|---|---|

基本設計にあるが詳細設計にないものは「落とし込み漏れ」としてください。
詳細設計にあるが基本設計に根拠がないものは「上位設計にない仕様」としてください。
提供されたソースのみに基づいて回答し、推測しないでください。
```

9. 参照切れのIDがないか
10. API詳細設計・モジュール設計・SQL設計・DB詳細設計の責務が混在していないか

## 6.7 Obsidian課題管理レビュー用プロンプト

# レビュー観点

1. NotebookLMレビューで発見された重要課題がObsidianに保存されているか
2. 課題メモに発見元・関連レビュー・指摘ID・重要度・指摘分類が記載されているか

7. 課題メモが関連するレビュー記録・設計判断・設計方針とリンクされているか
8. INDEX_課題一覧.md が作成・更新されているか
9. 未解決課題と解決済み課題が区別されているか
10. 解決済み課題が削除されず、履歴として残っているか

## 6.8 再レビュー用プロンプト

```
あなたは、前回レビュー指摘が解消されたかを確認する外部レビュー担当です。

前回のレビュー指摘一覧と、修正後の設計書、Obsidian課題メモを比較し、各指摘が解消されているか確認してください。

# 確認観点

1. 前回指摘された問題が修正されているか
2. 修正により新たな不整合が発生していないか
3. 修正内容が要件定義・基本設計に存在しない仕様追加になっていないか
4. 修正後も責務分離が保たれているか
5. 修正後もUI → API → モジュール → SQL → DBが追跡できるか
6. 修正後も共通定義IDの参照が正しいか
7. 未解決または要確認の指摘がObsidian課題として保存されているか
8. 解決済み指摘について、Obsidian課題メモに対応結果と再レビュー結果が記録されているか

# 出力形式

| 指摘ID | 前回指摘内容 | 確認結果 | 解消状況 | 新規指摘有無 | 追加指摘内容 | Obsidian課題 | 根拠 | ステータス |
|---|---|---|---|---|---|---|---|---|

解消状況は「解消済み / 一部解消 / 未解消 / 別問題発生 / Obsidian課題管理中 / 要確認」から選択してください。
提供されたソースのみに基づいて回答し、推測しないでください。
```

## 7次元レビュープロンプト（全層フルレビュー）

`prompts/` ディレクトリ内のテンプレートを使用する。  
`nlm_ask.py` で投入する場合:

```bash
SK=.claude/skills/notebooklm-design-review/scripts
PROMPTS=.claude/skills/notebooklm-design-review/prompts

python3 $SK/nlm_ask.py --notebook "$NB" \
  --prompt-file $PROMPTS/xref_mismatch.md \
  --sources requirements,usecases,traceability,crosscut \
  --ids-map _nlm/source_ids.json \
  --out _nlm/results/xref
```

| 次元 | プロンプトファイル | `--sources` |
|---|---|---|
| 相互参照取り違え | `prompts/xref_mismatch.md` | requirements,usecases,traceability,crosscut |
| 整合性 | `prompts/consistency.md` | （全ソース） |
| 逆参照・委譲 | `prompts/reverse_reference.md` | rules,requirements,usecases |
| 構成・記載粒度 | `prompts/granularity.md` | rules,system,sequences,apis,database |
| 画面フォーマット適合 | `prompts/format_conformance.md` | rules,screens |
| 抜け漏れ | `prompts/gaps.md` | （全ソース） |
| 総合/再レビュー | `prompts/overall.md` | （全ソース） |

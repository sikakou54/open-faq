# 指摘分類・レビュー観点一覧

---

## 指摘分類

| 指摘分類 | 内容 |
|---|---|
| 粒度不適切 | 要件定義・基本設計・詳細設計の粒度が混在している |
| 落とし込み漏れ | 上位設計の内容が下位設計へ反映されていない |
| 上位設計にない仕様 | 下位設計に上位設計の根拠がない仕様がある |
| 重複 | 複数ドキュメントに同じ内容が重複している |
| 逆参照 | 上位設計が下位設計を参照している |
| 責務混在 | 設計書の責務外の内容が記載されている |
| トレーサビリティ不備 | IDや対応関係を追跡できない |
| 参照切れ | 参照しているIDの定義が存在しない |
| 共通定義不備 | 状態値・区分値・閾値などが共通定義にない |
| SQL/DB不整合 | SQLで使用するテーブル・カラムがDB設計と不整合 |
| 状態遷移不整合 | 状態値・状態遷移・DB保存値が不整合 |
| Obsidian保存漏れ | 重要な計画・方針・課題・解決方法がObsidianに保存されていない |
| 課題メモ不足 | 課題の発見元・根拠・影響範囲・解決方針が不足している |
| 要確認 | 根拠不足により断定できない |

---

## 5.1 構成・粒度レビュー

- ドキュメント構成が自然な読み順になっているか
- 要件定義・基本設計・詳細設計の粒度が混在していないか
- 詳細設計が基本設計の概要粒度に留まっていないか
- 詳細設計に、実装に必要な入出力・条件・分岐・例外が定義されているか
- 詳細設計に、基本設計へ戻すべき概要説明が混入していないか
- 要件定義に、画面ID・API ID・SQL IDなどの基本設計以降の粒度が混入していないか

---

## 5.2 落とし込みレビュー

- 業務要件が機能要件へ落とし込まれているか
- 機能要件が業務ユースケースへ落とし込まれているか
- 業務ユースケースが基本設計へ落とし込まれているか
- 基本設計の画面が画面詳細設計へ落とし込まれているか
- 基本設計のAPIがAPI詳細設計へ落とし込まれているか
- 基本設計のバッチがバッチ処理詳細設計へ落とし込まれているか
- 基本設計のテーブルがDB詳細設計へ落とし込まれているか
- 基本設計の処理がAPI / バッチ / モジュール / SQLへ分解されているか

---

## 5.3 上位設計にない内容の混入レビュー

- 詳細設計に、要件定義・基本設計に存在しない仕様が追加されていないか
- 基本設計に、要件定義に存在しない業務仕様が追加されていないか
- 詳細設計で追加されたように見える仕様が、上位設計のどこに根拠を持つか
- 根拠が見つからない仕様は、仕様追加ではなく「要確認」として扱われているか

---

## 5.4 重複レビュー

- 要件定義と基本設計で同じ内容を重複記載していないか
- 基本設計と詳細設計で同じ内容を重複記載していないか
- 画面詳細設計とAPI詳細設計で同じ処理説明を重複記載していないか
- API詳細設計とモジュール設計で責務が重複していないか
- モジュール設計とSQL設計でSQL仕様を重複記載していないか
- SQL設計とDB詳細設計でテーブル定義を重複記載していないか
- 状態値・区分値・閾値・設定値が共通定義設計以外に重複定義されていないか

---

## 5.5 逆参照レビュー

- 要件定義が基本設計・詳細設計を参照していないか
- 基本設計が詳細設計の実装仕様に依存していないか
- 要件定義に、画面ID・API ID・SQL ID・テーブルIDなどの下位設計IDが混入していないか

---

## 5.6 責務分離レビュー

- 画面詳細設計にAPI内部処理・モジュール処理・SQL文・DB制約を書いていないか
- API詳細設計にSQL文・SQL詳細条件・モジュール内部ロジックを書いていないか
- バッチ処理詳細設計にSQL文・モジュール内部ロジックを書いていないか
- モジュール設計にAPI受付仕様・SQL文・テーブル定義を書いていないか
- SQL設計にAPI処理全体・モジュール処理全体・DB定義全文を書いていないか
- DB詳細設計にSQL文・API処理・モジュール処理を書いていないか
- 共通定義設計に状態遷移条件・処理フローを書いていないか

---

## 5.7 UI/API/モジュール/SQL/DBレビュー

- 画面操作から呼び出すAPIが定義されているか
- APIから呼び出すモジュールが定義されているか
- モジュールから呼び出すSQLが定義されているか
- SQLが参照・更新するDBテーブルが定義されているか
- APIがSQLを直接呼び出す構造になっていないか
- バッチがSQLを直接呼び出す構造になっていないか
- 各IDの参照先が存在するか
- 参照関係が `index.md` の対応表で追跡できるか

---

## 5.8 SQL・DB整合性レビュー

- SQL設計に入力引数・出力項目・SQL文が定義されているか
- SQL設計の対象テーブルがDB詳細設計に存在するか
- SQL設計で使用しているカラムがDB詳細設計に存在するか
- WHERE条件・JOIN条件がDB制約やインデックスと整合しているか
- SQL設計で使用する状態値・区分値が共通定義設計を参照しているか
- DB詳細設計にER図・制約・インデックスが定義されているか

---

## 5.9 状態遷移レビュー

- 状態を持つ主要エンティティが抽出されているか
- 状態値の意味が共通定義設計に定義されているか
- 状態遷移図に初期状態・終了状態が定義されているか
- 状態遷移図にイベント・遷移元・遷移先・遷移条件が定義されているか
- 状態遷移に関連するAPI / バッチ / モジュール / SQL / DBが追跡できるか
- DB上の状態カラムと共通定義設計のステータス定義が整合しているか
- 不正な状態遷移や未定義の遷移がないか

---

## 5.10 共通定義レビュー

- 状態値・区分値・閾値・上限値・固定値・設定値が共通定義設計に一元化されているか
- 各設計書で値や意味を重複定義していないか
- 各設計書から `CONST ID` を参照しているか
- 表示ラベルと内部値が区別されているか
- DB保存値と画面表示名が混同されていないか
- 環境ごとに変わる設定値と固定値が区別されているか
- 共通定義の変更時の影響範囲が定義されているか

---

## 5.11 Obsidian課題管理レビュー

- レビュー指摘から発生した課題がObsidianに保存されているか
- 課題メモに発見元・関連レビュー・指摘ID・重要度・指摘分類が記載されているか
- 課題メモに根拠・原因・影響範囲が記載されているか
- 課題メモに解決方針・対応内容・対応結果が記載されているか
- 再レビュー結果が記録されているか
- `INDEX_課題一覧.md` が作成・更新されているか
- 未解決課題と解決済み課題が区別されているか

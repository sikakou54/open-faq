# 観点(ドキュメント間のつながり / 設計根拠の明確さ)
1. 業務要件→機能要件→業務ユースケース→基本設計の関係が読み取れるか。上位要件から基本設計への落とし込みが自然か。
2. 要件定義と基本設計の間で矛盾・重複・未反映(上位を変えたのに下位へ反映されていない)がないか。
3. ある画面・API・システム処理が「なぜ必要か」(どの業務目的・UC に由来するか)が読み取れない箇所はないか。
4. 要件定義が基本設計の内容(画面ID・API名・テーブル物理名・エンドポイント・HTTPコード等)を**逆参照**しているように見える箇所はないか。あれば必ず挙げる。

注意: このリポジトリは層横断トレースを各文書に逆引きで書かず[トレーサビリティ一覧]に一元管理する方針。したがって「各設計書に対応UC表が無い」こと自体は仕様であり指摘にしない。問題にするのは、業務的文脈すら読み取れず**設計の存在理由が追えない**ケースに限る。

厳守:
- 提供ソースのみ。推測で補完しない。各指摘は相対パス + 実在 ID を根拠に。
- HTML/パイプ表が空に見えても指摘にしない。
- 指摘が無ければ「指摘なし」と明記。

# レビュー観点(画面設計フォーマット適合・記載粒度)
1. **§1 画面概要**が、長い散文ではなく簡潔な箇条書き(3〜6点)になっているか。1点1文で目的・対象ロール差・主な導線・主要な表示状態を述べているか。
2. **フォーマット外記述が無いか**。8 セクション固定の表・セクションの外側に、箇条書き・補足・コメント・`> [!NOTE]`/`> [!IMPORTANT]` などが置かれていないか。各セクションのリード文が定型(1文)を超えて仕様・例外・補足を盛り込み、勝手に増補・改変されていないか。
3. **§7 画面イベント詳細が基本設計の粒度か**。各イベントが「業務的な結果(どの画面へ遷移する・どの API を呼ぶか・何を表示/保存する・成功/失敗で何が起きる)」に留まっているか。**画面が内部で管理する変数・状態(選択状態・絞り込み/並び替え状態の保持や引継ぎ、画面内フラグ)、クエリ/パラメータの構築・受け渡し、API 呼び出しの手順・処理ステップの列挙**といった詳細設計レベルの内部挙動が混入していないか。例)「現在選択中の期間・絞り込みを引き継ぎ、未解決で絞り込んだ状態で…へ遷移する」=NG(遷移先のみ書く)。
4. **呼び出す API は記載されているか**(§7 で API を呼ぶイベントが、どの API を呼ぶかをリンクで示しているか)。逆に、API リンクごと落として業務結果が不明瞭になっていないか。
5. **データパターン**(選択肢・状態値)が、§4 画面項目表の直下の「データパターン」表(画面項目/表示名/補足)で定義され、表示名は業務語で、内部コード値が混入していないか。表外のインライン箇条書きで列挙されていないか。
6. **詳細設計粒度の本文転記が無いか**。変数名・物理カラム名・JSON キー名・エンドポイントパス・HTTP ステータスコード・アルゴリズム等が本文に書かれていないか(API/テーブルはリンクで辿る)。
7. **要件ID(BR/FR/NFR/RULE/UC)・mermaid 実装詳細**が画面設計に混入していないか。
8. **別セクションを見れば分かる内容の重複記載が無いか**。特に §2 画面遷移図が示す遷移フロー(「成功時は◯◯へ遷移し、条件Aなら△△へ割込み遷移する」等)を §1 概要・§7 で散文として再記述していないか(各 §7 イベントの「◯◯へ遷移する」結果は可)。§4 表示条件・§5 検証・§8 メッセージ文面を他セクションで繰り返していないか。
9. **§7 で呼び出す API が API-ID 付きで表示されているか**(例「ログイン(API-002)」)。リンク文に API-ID が見えない、または API を呼ぶイベントなのに API 参照が無い場合は指摘する。

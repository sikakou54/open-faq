# レビュー観点(構成・記載粒度)
## 要件定義層を見る場合
1. 業務要件/機能要件/非機能要件/業務ユースケースの粒度が混在していないか(BR に数値しきい値・手順番号・状態名・画面文言・FR とほぼ同文が混入していないか)。
2. 要件としては細かすぎる内容(画面項目・UI 文言・実装手順・アルゴリズム)が混入していないか。
3. BR と FR で同一内容の重複記載がないか(BR=業務意図1〜2文、FR=システム機能の細粒度)。
4. 要件が基本設計前提の後付け説明(How 寄り)になっていないか。

## 基本設計層を見る場合
5. 基本設計に詳細設計・実装レベル(SQL・クラス/関数/メソッド名・ORM・テーブルCRUD表記・物理カラム名・算出式・ピクセル値)が混入していないか。特にシーケンス図に SQL/メソッド名/CRUD表記が無いか。
6. 詳細設計へ移管すべき内容が基本設計に残っていないか。
7. 逆に、基本設計として必要な項目(対応UC・利用テーブル・エラー結線・I/O 定義等)が欠落していないか。

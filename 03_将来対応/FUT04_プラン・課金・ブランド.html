FUT04: プラン・課金・ブランド

0. 文書情報

項目 内容

版数 v1.0
ステータス 検討中(MVP 外)

1. 概要

項目 内容
概要 MVP では単一プラン(従量課金 + 月次無料枠) / 提供事業者ロゴ必須表示 / 1 アカウント = 1 オーナーアカウント固定 / クレジットカード払い必須 / オーナー譲渡不可で提供している領域について、Post-MVP でプラン別機能差別化 / ホワイトラベル化 / 担当者ハブ / 請求書払い / オーナー譲渡 / ロールプリセットを段階導入する。
背景 エンタープライズ契約獲得には、プラン選択肢・ブランド表示自由度・複数契約所属・請求書払い・譲渡フローの整備が必要となる。
期待効果 エンタープライズ契約獲得余地、顧客解約率低下、内部統制要件への適合、運用負荷(オーナー譲渡サポート等)の軽減。
優先度(将来対応内) 中
想定時期 T3(GA 判定まで)〜 Post-MVP

2. 関連要件・候補

2.1 要件候補(将来 FR)

プラン・課金・ブランド

ID 要件候補 MVP の前提 判定時期 判定基準 / 完了条件
FUT-REQ-PLAN-001 プラン別機能差別化 単一プラン(従量課金 + 月次無料枠) ベータ以降 契約要望件数、競合プラン構成、原価試算をもとに導入可否を判定
FUT-REQ-PLAN-002 ホワイトラベル化 本サービス提供事業者ロゴを必須表示 GA エンタープライズ層の要望、プラン別機能差別化との整合、ブランドリスクをもとに判定
FUT-REQ-PLAN-003 担当者ハブ(複数契約所属) 1 アカウント = 1 オーナーアカウント固定 GA 契約要望件数、認可モデルの複雑度、監査ログの全契約横断検索要件をもとに判定
FUT-REQ-PLAN-004 請求書払い / 未請求残高返金 / KYC / 反社チェック クレジットカード払い必須、B2B 低リスク前提、返金なし Post-MVP 法務・財務レビュー、法人審査、エンタープライズ契約の必要性をもとに判定
FUT-REQ-PLAN-005 カスタム SLA / 専用 NER モデル 単一プラン内の標準提供 Post-MVP 契約規模、運用原価、SLA 違反時の補償コストをもとに判定
FUT-REQ-PLAN-006 オーナー譲渡 契約作成者がオーナー固定、譲渡不可。MVP では緊急時のみサポート対応 GA 譲渡 UI 仕様、譲渡時の規約再同意・課金責任者切替、監査ログ・誤譲渡防止策(2 名承認 / 再認証等)が確定すること
FUT-REQ-PLAN-007 カスタムロール / プロジェクト管理者サブロール分割 MVP では 3 ロール固定(オーナー / プロジェクト管理者 / メンバー)。M_PRJ_USER_ASGN.role は admin / member の 2 値のみ。契約横断カスタムロール定義やプロジェクト管理者の権限細分化は提供しない GA 利用者側からのカスタムロール要望件数、プロジェクト管理者の権限細分化ニーズ、ロール定義の運用責務が確定すること

4. 実施条件

項目 内容
実施判断条件 プラン別機能差別化は契約要望件数 + 競合プラン構成 + 原価試算、ホワイトラベルは GA でエンタープライズ要望ベース、担当者ハブは認可モデルの複雑度評価、請求書払いは法務・財務レビュー、オーナー譲渡は誤譲渡防止策(2 名承認 / 再認証)確定
期限 T3(プラン別機能、ホワイトラベル、担当者ハブ、オーナー譲渡、ロールプリセット)、Post-MVP(請求書払い / KYC、カスタム SLA / 専用 NER)
依存関係 課金フロー(Stripe Webhook 受信 / DLQ 30 日リプレイ / 月次請求確定)との整合、共有概念の 3 ロールユーザーモデル(2 マスタ完全分離: M_OWNERS マスタ + M_PRJ_USERS マスタ + 割当 M_PRJ_USER_ASGN.role)拡張

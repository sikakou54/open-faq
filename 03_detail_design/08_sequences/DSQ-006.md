# <span id="DSQ-006"></span>DSQ-006: アカウント退会→カスケード削除 詳細シーケンス

> **この詳細シーケンスは「アカウント利用者本人による即時退会の同期確定から、退会済みアカウントが所有するプロジェクトの運用データの日次カスケード物理削除、保持期間経過後の課金・請求・認証データの物理削除までの内部コンポーネント連携とトランザクション境界」を定義します。**

*種別 詳細シーケンス図 ・ ステータス ドラフト*

## 1. 目的

本フローは、退会確定(同期 API 処理)・退会済みアカウントが所有するプロジェクトの運用データの日次カスケード物理削除(依存順の多段階削除・長時間ジョブ保護待機を伴う)・保持期間経過後の課金/請求/認証データの物理削除という 3 段階の高リスク破壊的操作を含み、各段のトランザクション境界・ロールバック戦略・待機/再評価分岐を実装粒度で確定する必要があるため詳細化する。詳細化元は退会済み・論理削除データの物理削除バッチ([SEQ-089](../../02_basic_design/03_sequences/SEQ-089.md#SEQ-089))であり、その「サーバー・DB」抽象を Route Handler / Service / Repository / D1 / Cron Triggers / Queue の連携へ写像する。同期退会確定は [API-056](../../02_basic_design/02_backend/03_apis/API-056.md#API-056)、削除対象か否かの判定ロジックは [IPO-010](../04_ipo/IPO-010.md#IPO-010)、日次バッチの実行機構は [BAT-009](../05_batch/BAT-009.md#BAT-009)(運用データ)・[BAT-013](../05_batch/BAT-013.md#BAT-013)(課金・請求・認証データ)を参照する。

## 3. 正常系シーケンス

退会確定(同期)から、日次の運用データカスケード物理削除(長時間ジョブ保護待機を含む)、保持期間経過後の課金・請求・認証データ物理削除までを、各段のトランザクション境界とともに示す。運用データの物理削除は[SYS-027](../../02_basic_design/02_backend/01_system/SYS-027.md#SYS-027)の網羅順序(子→親)に従いプロジェクト単位で直列に進め、プロジェクトごとに個別コミットする。

```mermaid
sequenceDiagram
    autonumber
    participant Client as "Client(SCR-019)"
    participant RH as "Route Handler(API-056)"
    participant Svc as WithdrawalService
    participant Repo as Repository
    participant D1 as D1
    Note over Client,D1: 段階1 同期退会確定
    Client->>RH: 退会実行(再認証トークン・確認メール・退会理由)
    RH->>Svc: 退会確定を依頼
    Svc->>Repo: 課金アカウント状態を確認
    Repo->>D1: 課金アカウント状態を照会
    D1-->>Repo: 状態(active/suspended/withdrawn/deleted)
    Repo-->>Svc: 現在状態
    alt 既に withdrawn または deleted
        Svc-->>RH: 重複退会エラー
        RH-->>Client: 409 を返す
    else 退会可能
        Note over Svc,D1: トランザクション開始
        Svc->>Repo: 本人がメンバーの割当を解除(離脱)
        Repo->>D1: 割当解除を更新
        D1-->>Repo: 更新完了
        Svc->>Repo: 本人がオーナーのプロジェクトを削除 deleted へ遷移
        Repo->>D1: プロジェクト状態を更新
        D1-->>Repo: 更新完了
        Svc->>Repo: 課金アカウントを withdrawn へ遷移し退会日時を記録
        Repo->>D1: 課金アカウント状態・退会日時を更新
        D1-->>Repo: 更新完了
        Svc->>Repo: 退会記録を登録し請求データの削除予定日を記録
        Repo->>D1: 退会記録を登録
        D1-->>Repo: 登録完了
        Note over Svc,D1: コミット
        Svc-->>RH: 退会結果(withdrawn 状態・退会日時)
        RH-->>Client: 退会完了を返す
    end
```

```mermaid
sequenceDiagram
    autonumber
    participant Cron as "Cron Triggers(BAT-009)"
    participant Svc as PurgeService
    participant Repo as Repository
    participant D1 as D1
    Note over Cron,D1: 段階2 運用データ日次カスケード物理削除 SYS-027
    Cron->>Svc: 日次起動(多重起動チェック)
    alt 実行中インスタンスあり
        Svc-->>Cron: 本起動をスキップして終了
    else 実行中インスタンスなし
        Svc->>Repo: 削除対象走査を依頼 IPO-010
        Repo->>D1: 退会済みアカウント所有プロジェクトの運用データ、<br/>猶予期間経過済みの一般論理削除行を照会
        D1-->>Repo: 削除対象一覧
        Repo-->>Svc: 削除対象一覧
        alt 削除対象なし
            Svc-->>Cron: 削除なしで正常終了
        else 削除対象あり
            Svc->>Svc: 課金・請求・利用者・認証データを対象から除外
            loop 削除対象プロジェクトを1件ずつ処理
                Svc->>Repo: 対象プロジェクトの実行中長時間処理の有無を確認
                Repo->>D1: FAQ取込ジョブ等の状態を照会
                D1-->>Repo: 終端状態か否か
                Repo-->>Svc: 判定結果
                alt 実行中の長時間処理あり
                    Svc->>Svc: 終端状態 完了/失敗確定 まで待機し保護
                    alt 想定処理時間内に終端到達
                        Svc->>Svc: 削除実行へ進む
                    else 想定処理時間を超過
                        Svc->>Svc: 当該プロジェクトのみ持ち越しとして記録し次のプロジェクトへ
                    end
                else 実行中の長時間処理なし
                    Svc->>Svc: 削除実行へ進む
                end
                opt 削除実行へ進んだ場合
                    Note over Svc,D1: トランザクション開始 プロジェクト単位
                    Svc->>Repo: 従属データを依存順に物理削除 許可ドメイン/旧鍵/未解決質問/質問ログ/利用量/FAQ/FAQ化履歴/AIしきい値キャッシュ/お知らせ受信状況/受信箱/通知ログ/エラーログ/レート上書き/クォータ上限
                    Repo->>D1: 従属データを削除
                    D1-->>Repo: 削除完了
                    Svc->>Repo: プロジェクトメンバー割当を物理削除
                    Repo->>D1: 割当を削除
                    D1-->>Repo: 削除完了
                    Svc->>Repo: プロジェクト本体を物理削除
                    Repo->>D1: プロジェクトを削除
                    D1-->>Repo: 削除完了
                    Svc->>Repo: 削除内容を監査記録として登録
                    Repo->>D1: 監査記録を登録
                    D1-->>Repo: 登録完了
                    Note over Svc,D1: コミット プロジェクト単位
                end
            end
            loop 猶予期間経過済みの一般論理削除行
                Note over Svc,D1: トランザクション開始 行単位
                Svc->>Repo: 依存順に物理削除し監査記録を登録
                Repo->>D1: 削除・監査記録を登録
                D1-->>Repo: 完了
                Note over Svc,D1: コミット 行単位
            end
            Svc-->>Cron: 処理結果サマリ 削除件数/持ち越し件数/失敗件数 を記録し正常終了
        end
    end
```

## 6. 後続工程への引き継ぎ事項

テスト設計・詳細ロジック設計・DB 物理設計へ渡す観点を示す。

- 同期退会確定トランザクション(No.2)の境界値(割当解除件数 0 件・オーナープロジェクト 0 件のケースでも課金アカウント遷移・退会記録登録は必ず行われること)をテスト設計でケース化する。
- 段階2(運用データ日次削除)のプロジェクト単位トランザクションが個別コミットであり、一部プロジェクトの削除失敗が他プロジェクトの削除に波及しないことを検証する([BAT-009](../05_batch/BAT-009.md#BAT-009) §6)。
- 長時間ジョブ保護待機([SYS-027](../../02_basic_design/02_backend/01_system/SYS-027.md#SYS-027) PR-06)の終端状態到達・想定処理時間超過による持ち越しの双方の分岐をテスト設計でケース化する。
- 段階3(保持期間経過後削除)のアカウント単位処理で、課金・請求従属 → 認証従属 → 課金アカウント・アカウント確定の依存順が厳守され、中途状態(一部のみ削除済み)が放置されないことを検証する([BAT-013](../05_batch/BAT-013.md#BAT-013) §9)。
- 段階1〜3を通じた冪等性(同一日の再起動・多重起動抑止境界を含む)による二重削除・二重監査記録の防止を、[BAT-009](../05_batch/BAT-009.md#BAT-009) §6・[BAT-013](../05_batch/BAT-013.md#BAT-013) §6 の冪等キー方針に沿って検証する。
- 3 段階すべてで削除内容が監査記録([TBL-027](../../02_basic_design/02_backend/04_database/TBL-027.md#TBL-027))として漏れなく残り、監査記録書き込み失敗時に削除のみが先行して残る期間が滞留監視でアラートされることを確認する。
- 削除順序(FK 依存順・子→親)の実装が [DB 設計の ER 図](../../02_basic_design/02_backend/04_database/index.md) と齟齬なく、途中失敗時に中途半端な参照切れが残らないことをテストで検証する。

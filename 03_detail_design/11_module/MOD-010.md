## 1. 目的

本機能領域は、運営お知らせの配信予定日時到来または即時配信指示を契機とした宛先解決・受信箱 fan-out・お知らせメール送信([BAT-007](../05_batch/BAT-007.md#BAT-007))、アカウント利用者向けの受信箱閲覧・未読件数取得・既読管理([CLS-011](../10_class/CLS-011.md#CLS-011) `InboxService`)、メール配信事業者(Resend)との送信/Webhook 連携([EIF-003](../06_external_if/EIF-003.md#EIF-003))、送信品質監視による通知送信抑制([BAT-002](../05_batch/BAT-002.md#BAT-002))・配信失敗通知の再送([BAT-008](../05_batch/BAT-008.md#BAT-008))、プロジェクト単位の通知配信状態サマリ参照([CLS-011](../10_class/CLS-011.md#CLS-011) `NotificationDispatchService`)までを一つの実装単位として定義する。モジュール分割は Next.js on Cloudflare の物理配置(`app/`・`lib/service`・`lib/repository`・`lib/gateway`・`workers/cron`・`workers/queues`)へ写像し、依存は内向き(frontend → api → service → repository、メール配信連携は service → external-gateway)に統一して逆依存・循環依存を作らない。CLS-011 で確定したクラス構成([`AnnouncementRouteHandler`](../10_class/CLS-011.md#CLS-011)・[`InboxService`](../10_class/CLS-011.md#CLS-011)・[`NotificationDispatchService`](../10_class/CLS-011.md#CLS-011)・[`EmailProvider`](../10_class/CLS-011.md#CLS-011)/`ResendEmailProvider`・各 Repository)をそのまま物理モジュールへ写像し、命名を一致させる。

## 6. 後続工程への引き継ぎ事項

実装・テスト設計へ引き継ぐ観点(依存方向の逸脱検出・非同期境界・外部連携の切り離しテスト)を箇条書きで示す。

- 内向き依存の逸脱検証: D1 クライアント(M-18)を利用するのは Repository 群(M-12〜M-17)のみで、Service/Cron/API から直接 D1 を触らないこと。逆依存(Repository → Service)・循環依存が生じていないこと。
- 同期経路の境界検証: お知らせ API(M-03)・通知配信状態 API(M-04)から Service(M-06/M-07)への委譲が [CLS-011](../10_class/CLS-011.md#CLS-011) のクラス構成と一致すること。Webhook 受信 API(M-05)から通知送信抑制判定(M-09)への委譲は [API-059](../../02_basic_design/02_backend/03_apis/API-059.md#API-059) ・ [IPO-016](../04_ipo/IPO-016.md#IPO-016) の入出力と一致すること(CLS-011 の対象外)。
- 非同期境界の検証: お知らせ配信 Cron(M-19)から Queues 経由での fan-out コンシューマ(M-20)への投入・消費・冪等・DLQ 滞留([BAT-007](../05_batch/BAT-007.md#BAT-007))、送信品質監視 Cron(M-21)・配信失敗再送 Cron(M-22)の Cron Triggers 起動制御・排他・冪等([BAT-002](../05_batch/BAT-002.md#BAT-002) ・ [BAT-008](../05_batch/BAT-008.md#BAT-008))。
- external-gateway スタブ化: EmailProvider(M-10/M-11)をスタブ化した通知送信抑制判定(M-09)・fan-out コンシューマ(M-20)・配信失敗再送 Cron(M-22)の単体テストで、送信成功/失敗・署名検証成功/失敗の各分岐を分離検証すること。
- 正本未確定の引き継ぎ: 送信品質監視 Cron(M-21)の起動周期・許容水準の具体値、プロジェクト単位の送信抑制状態の物理カラムは基本設計側で正本未確定であり、確定後に M-09/M-21 の実装へ反映すること([IPO-016](../04_ipo/IPO-016.md#IPO-016) §5 ・ [BAT-002](../05_batch/BAT-002.md#BAT-002) §9)。
- モジュール境界の契約整合: お知らせ API・通知配信状態 API と Service 間、Service と各 Repository 間の入出力契約が [CLS-011](../10_class/CLS-011.md#CLS-011) の DTO/Entity 定義と一致すること。
- 集約・冪等境界の検証: 未読お知らせ集約判定(M-08)の集約時間窓境界での同時到達、fan-out コンシューマ(M-20)・配信失敗再送 Cron(M-22)の冪等キーによる重複実行防止をテスト設計でケース化すること。

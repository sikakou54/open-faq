# <span id="IO-014"></span>IO-014: 退会 入出力

> **この入出力設計書は「退会画面と再認証 API・アカウント退会 API の入力/出力項目・入力チェック・編集加工」を論理項目単位で定義します。**

*種別 入出力設計 ・ ステータス ドラフト*

| 項目 | 値 |
|----|----|
| 入出力設計ID | IO-014 |
| 業務ユースケースID | [UC-022](../../01_requirements/04_business_usecases/UC-022.md#UC-022) |
| 関連画面 | [SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) |
| 関連 API | [API-005](../../02_basic_design/02_backend/03_apis/API-005.md#API-005) ・ [API-056](../../02_basic_design/02_backend/03_apis/API-056.md#API-056) |

## 1. 目的

本入出力設計は、ユーザーが自身のアカウントを即時退会する際の確認入力(登録メールアドレス一致・再認証)と実行結果の入力/出力項目を、画面と API を突き合わせて論理項目単位で確定する。橋渡しする画面は [SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) の確認ダイアログ確定操作(SCR-019 EVT-03)、API は再認証を担う [API-005](../../02_basic_design/02_backend/03_apis/API-005.md#API-005) と退会実行を担う [API-056](../../02_basic_design/02_backend/03_apis/API-056.md#API-056) である。

## 2. 対象範囲

本設計が対象とする画面・API・データと対象外を明示する。退会に伴うプロジェクト離脱・削除処理そのものの内部手順や請求データの保持・削除予定管理は別文書へ委ねる。

| 区分 | 対象 |
|----|----|
| 対象画面 | [SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) 退会(本体パネルおよび確認ダイアログ) |
| 対象API | [API-005](../../02_basic_design/02_backend/03_apis/API-005.md#API-005) 再認証 ・ [API-056](../../02_basic_design/02_backend/03_apis/API-056.md#API-056) アカウント退会(即時) |
| 対象データ | 退会理由・登録メールアドレス確認入力・再認証パスワード・退会影響の注意事項表示・退会実行結果 |
| 対象外 | プロジェクト割当解除・作成プロジェクト削除の内部処理手順、請求データの保持・削除予定の記録処理([SYS-027](../../02_basic_design/02_backend/01_system/SYS-027.md#SYS-027))、退会後の請求情報閲覧(別画面) |

## 3. 入力項目一覧

退会画面から受け取り API へ渡す入力項目を論理項目単位で定義する。文字数上限などの具体値は基本設計へ ID 参照する。

| 項目ID | 項目名 | データ型 | 必須 | 初期値 | 入力制約 | 入力例 | 備考 |
|----|----|----|----|----|----|----|----|
| I-01 | 退会理由 | 文字列 | — | — | 0〜500 文字([SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) §4 #2) | サービスを利用しなくなったため | SCR-019 §4 #2 に対応。API-056 リクエスト `reason` |
| I-02 | 登録メールアドレス確認入力 | 文字列 | ◯ | — | 最大 255 文字([SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) §4 #5)。アカウントの登録メールアドレスと完全一致 | user@example.com | SCR-019 §4 #5 に対応。API-056 リクエスト `confirmEmail` |
| I-03 | パスワード(再認証用) | 文字列 | ◯ | — | 最大 128 文字([SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) §4 #6) | — | SCR-019 §4 #6 に対応。画面には表示しない。API-005 リクエスト `password` |
| I-04 | 再認証トークン | 文字列 | ◯ | — | API-005 応答で発行された有効な再認証トークン | — | 画面には表示しない。API-005 応答 `token` を API-056 リクエスト `reauthToken` へ引き継ぐ |

## 4. 出力項目一覧

API レスポンス・画面へ返す出力項目を論理項目単位で定義する。

| 項目ID | 項目名 | データ型 | 表示形式 | 表示条件 | 備考 |
|----|----|----|----|----|----|
| O-01 | 退会影響の注意事項 | 文字列(固定文) | 本体パネルに常時表示 | 初期表示時 | アカウント単位・即時・不可逆・作成プロジェクト削除・参加プロジェクト離脱・請求情報は閲覧のみ可能である旨。SCR-019 §4 #1 に対応 |
| O-02 | 再認証結果 | 真偽値 | 画面非表示 | 再認証実行後 | API-005 応答 `ok`。真のとき再認証トークン(I-04)を取得し退会確定へ進む |
| O-03 | 退会受付結果 | 真偽値 | — | 退会実行後 | API-056 応答に含まれる退会受付の成否 |
| O-04 | 退会後の課金アカウント状態 | 文字列 | 画面非表示 | 退会成功時 | 遷移後の課金アカウント状態。意味は[状態モデル §2](../../02_basic_design/08_state-model.md#2-課金アカウント状態)。API-056 応答に含まれる |
| O-05 | 退会日時 | 日時 | 画面非表示 | 退会成功時 | 退会が確定した日時(データ保持期間の起算点。[API-056](../../02_basic_design/02_backend/03_apis/API-056.md#API-056) P-06)。API-056 応答に含まれる |

## 5. 入力チェック仕様

各入力項目のチェック内容・実施主体(クライアント / サーバー)・エラー時参照を定義する。クライアント検証違反時は確定操作を中止し、サーバー検証違反時はエラー応答に応じて確認ダイアログの表示状態を制御する。文面は本ページに再掲せず ID 参照する。

| 項目名 | チェック内容 | タイミング | エラー時メッセージ | 備考 |
|----|----|----|----|----|
| 退会理由(I-01) | 最大文字数チェック(500 文字以内) | クライアント | [SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) EM-01 | 入力時に判定([SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) §5 #2) |
| 退会理由(I-01) | 最大文字数チェック(500 文字以内) | サーバー | [ERR-001](../../02_basic_design/05_errors/ERR-001.md#ERR-001) | 超過時は `400 VALIDATION_ERROR`([API-056](../../02_basic_design/02_backend/03_apis/API-056.md#API-056) `## バリデーション`) |
| 登録メールアドレス確認入力(I-02) | 登録メールアドレス一致チェック | クライアント | [SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) EM-02 | 入力時・確定操作前に判定([SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) §5 #5)。不一致時は確定するボタン(#8)を非活性化 |
| 登録メールアドレス確認入力(I-02) | アカウントのメールアドレスとの完全一致チェック | サーバー | [ERR-001](../../02_basic_design/05_errors/ERR-001.md#ERR-001) | 不一致・未入力時は `400 VALIDATION_ERROR`([API-056](../../02_basic_design/02_backend/03_apis/API-056.md#API-056) P-02) |
| パスワード(I-03) | 未入力チェック | クライアント | [SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) EM-03 | 確定操作時に判定([SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) §5 #6) |
| パスワード(I-03) | 再照合チェック | サーバー | [SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) EM-04 / [ERR-005](../../02_basic_design/05_errors/ERR-005.md#ERR-005) | 不一致時は `401 INVALID_PASSWORD`([API-005](../../02_basic_design/02_backend/03_apis/API-005.md#API-005) P-02) |
| 再認証トークン(I-04) | トークン有効性チェック | サーバー | [SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) EM-04 / [ERR-013](../../02_basic_design/05_errors/ERR-013.md#ERR-013) | 無効・未提示時は `401 REAUTH_REQUIRED`([API-056](../../02_basic_design/02_backend/03_apis/API-056.md#API-056) P-00) |
| 課金アカウント状態(重複退会防止) | 既に `withdrawn` / `deleted` でないことのチェック | サーバー | [SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) EM-06 / [ERR-023](../../02_basic_design/05_errors/ERR-023.md#ERR-023) | 該当時は `409 CONFLICT`([API-056](../../02_basic_design/02_backend/03_apis/API-056.md#API-056) P-03)。状態の意味は[状態モデル §2](../../02_basic_design/08_state-model.md#2-課金アカウント状態) |

## 6. 編集・加工仕様

入力値の正規化・出力値の整形など編集加工を定義する。

| 項目名 | 加工内容 | 加工タイミング | 備考 |
|----|----|----|----|
| 登録メールアドレス確認入力(I-02) | 前後空白の除去・小文字化 | 一致判定前(サーバー) | アカウントのメールアドレスとの一致判定は加工後の値で行う |
| 退会理由(I-01) | 前後空白の除去 | 保存前(サーバー) | 未入力時は退会理由なしとして扱う |
| 再認証トークン(I-04) | 画面には表示せず内部でのみ保持し API-056 呼び出し時に引き継ぐ | 再認証成功後(クライアント) | 確定操作(SCR-019 EVT-03)実行時に I-04 を伴って API-056 を呼び出す |

## 7. 後続工程への引き継ぎ事項

テスト設計・DB 物理設計・詳細ロジック設計へ引き継ぐ観点を挙げる。プロジェクト離脱・削除処理の詳細と請求データの保持・削除予定管理は別文書へ委ねる。

- 退会理由(I-01)の境界値(0 / 500 / 501 文字)、登録メールアドレス確認入力(I-02)の一致 / 不一致パターンはテスト設計でケース化する。
- 再認証失敗(EM-04・ERR-013・ERR-005)・重複退会(EM-06・ERR-023)・退会失敗(EM-05)の各エラー応答時に確認ダイアログを維持するか閉じるかの画面挙動([SCR-019](../../02_basic_design/01_frontend/01_screens/SCR-019.md#SCR-019) §7 EVT-03)はテスト設計でケース化する。
- 退会に伴うプロジェクト離脱・削除・課金アカウント状態遷移・請求データの保持予定記録は [API-056](../../02_basic_design/02_backend/03_apis/API-056.md#API-056) の処理内容および [SYS-027](../../02_basic_design/02_backend/01_system/SYS-027.md#SYS-027) を参照する。
- 論理項目 ↔ 物理カラムの対応は DB 物理設計へ引き継ぐ。

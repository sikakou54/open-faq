# <span id="IO-002"></span>IO-002: ログイン 入出力

> **この入出力設計書は「ログイン画面とログイン API の入力/出力項目・入力チェック・編集加工」を論理項目単位で定義します。**

*種別 入出力設計 ・ ステータス ドラフト*

| 項目 | 値 |
|----|----|
| 入出力設計ID | IO-002 |
| 業務ユースケースID | [UC-001](../../01_requirements/04_business_usecases/UC-001.md#UC-001) |
| 関連画面 | [SCR-001](../../02_basic_design/01_frontend/01_screens/SCR-001.md#SCR-001) |
| 関連 API | [API-002](../../02_basic_design/02_backend/03_apis/API-002.md#API-002) |

## 1. 目的

本入出力設計は、未認証ユーザーがメールアドレス・パスワードで本人確認を受けセッションを確立する一連の入力/出力項目を、画面と API を突き合わせて論理項目単位で確定する。橋渡しする画面は [SCR-001](../../02_basic_design/01_frontend/01_screens/SCR-001.md#SCR-001) の「ログイン」押下(SCR-001 EVT-02)、API は [API-002](../../02_basic_design/02_backend/03_apis/API-002.md#API-002) である。

## 2. 対象範囲

本設計が対象とする画面・API・データと対象外を明示する。ロックアウトの判定ロジック自体と初期表示時の遷移分岐は別文書へ委ねる。

| 区分 | 対象 |
|----|----|
| 対象画面 | [SCR-001](../../02_basic_design/01_frontend/01_screens/SCR-001.md#SCR-001) ログイン(「ログイン」押下 SCR-001 EVT-02) |
| 対象API | [API-002](../../02_basic_design/02_backend/03_apis/API-002.md#API-002) ログイン |
| 対象データ | メールアドレス・パスワード、認証成功時の利用者識別子・再規約同意要否・有効セッション一覧 |
| 対象外 | ロックアウトの判定ロジック([IPO-014](../04_ipo/IPO-014.md#IPO-014))・初期表示時のログイン済み判定分岐(SCR-001 EVT-01)・パスワード再設定(別 IO で扱う)・アカウント登録(別 IO で扱う) |

## 3. 入力項目一覧

ログイン画面から受け取り API へ渡す入力項目を論理項目単位で定義する。文字数上限などの具体値は基本設計へ ID 参照する。

| 項目ID | 項目名 | データ型 | 必須 | 初期値 | 入力制約 | 入力例 | 備考 |
|----|----|----|----|----|----|----|----|
| I-01 | メールアドレス | 文字列 | ◯ | — | 最大 254 文字・メールアドレス形式([SCR-001](../../02_basic_design/01_frontend/01_screens/SCR-001.md#SCR-001) §4 #1 / §5) | user@example.com | SCR-001 §4 #1 に対応。API リクエスト `email` |
| I-02 | パスワード | 文字列 | ◯ | — | 最大 128 文字([SCR-001](../../02_basic_design/01_frontend/01_screens/SCR-001.md#SCR-001) §4 #2 / §5) | — | SCR-001 §4 #2 に対応。画面には平文表示しない(input password)。API リクエスト `password` |
| I-03 | 冪等キー | 文字列 | — | — | 再送重複防止の一意識別子(ULID) | — | 任意。認証ヘッダ `Idempotency-Key` で送信([API-002](../../02_basic_design/02_backend/03_apis/API-002.md#API-002) `## リクエスト`)。画面には表示しない |

## 4. 出力項目一覧

API レスポンス・画面へ返す出力項目を論理項目単位で定義する。認証成功時と認証失敗時(ロックアウト・停止含む)で出力項目が分岐する。

| 項目ID | 項目名 | データ型 | 表示形式 | 表示条件 | 備考 |
|----|----|----|----|----|----|
| O-01 | 利用者識別子 | 文字列 | 画面非表示 | 認証成功時 | 認証された利用者の識別子。以降の画面遷移・API 呼び出しの前提となる認証状態確立に用いる。API `userId` |
| O-02 | 再規約同意要否 | 真偽値 | — | 認証成功時 | `true` のとき規約再同意(SCR-020)へ遷移、`false` のとき概要(SCR-012)へ遷移([SCR-001](../../02_basic_design/01_frontend/01_screens/SCR-001.md#SCR-001) §2 / §7 EVT-02)。API `requireTermsAgreement` |
| O-03 | 有効セッション一覧 | 配列 | 画面非表示 | 認証成功時 | 現在有効なセッションの一覧(セッション識別子・IP アドレス・User-Agent・作成日時)。本画面では描画しない。API `activeSessions` |
| O-04 | メッセージ表示エリア内容 | 文字列 | メッセージ表示エリアに表示(SCR-001 §4 #6) | 認証失敗時(エラーまたはロックアウト警告) | 認証エラー時は EM-04、ロックアウト条件到達時は EM-05 を表示。文面は本ページに再掲せず ID 参照する(§5) |

## 7. 後続工程への引き継ぎ事項

テスト設計・DB 物理設計・詳細ロジック設計へ引き継ぐ観点を挙げる。ロックアウト判定ロジックの詳細は別文書へ委ねる。

- メールアドレス(I-01)・パスワード(I-02)の境界値(最大長ちょうど / 超過)はテスト設計でパターン化する。
- 認証失敗(EM-04)とロックアウト警告(EM-05)の表示分岐、および連続失敗回数がしきい値到達直前・到達時の境界はテスト設計でケース化する([IPO-014](../04_ipo/IPO-014.md#IPO-014) 引き継ぎ事項も参照)。
- 再規約同意要否(O-02)による遷移先分岐(SCR-020 / SCR-012)の組み合わせはテスト設計でケース化する。
- ロックアウトの判定ロジック(失敗計上・ロック発動・自動解除・手動解除)は [IPO-014](../04_ipo/IPO-014.md#IPO-014) を参照する。
- 論理項目 ↔ 物理カラムの対応は [DBP-002](../07_db_physical/DBP-002.md#DBP-002) で確定する。

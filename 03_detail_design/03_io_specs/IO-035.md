## 1. 目的

本入出力設計は、画面を持たない外部連携 4 API(AI 推論 IF・メール配信 IF・メール配信状態 Webhook・課金プロバイダ Webhook)について、入力/出力項目・入力チェック・異常時の応答を論理項目単位で確定する。橋渡しする API は [API-057](../../02_basic_design/02_backend/03_apis/API-057.md#API-057)・[API-058](../../02_basic_design/02_backend/03_apis/API-058.md#API-058)・[API-059](../../02_basic_design/02_backend/03_apis/API-059.md#API-059)・[API-060](../../02_basic_design/02_backend/03_apis/API-060.md#API-060) である。

## 2. 対象範囲

本設計が対象とする API・データと対象外を明示する。AI 推論・メール配信の外部連携の性質・連携図・例外処理の詳細は EIF へ委ねる。

| 区分 | 対象 |
|----|----|
| 対象API | [API-057](../../02_basic_design/02_backend/03_apis/API-057.md#API-057) AI 推論 IF(`AnswerProvider`)・[API-058](../../02_basic_design/02_backend/03_apis/API-058.md#API-058) メール配信 IF(`EmailProvider`)・[API-059](../../02_basic_design/02_backend/03_apis/API-059.md#API-059) 外部 Webhook(Resend)・[API-060](../../02_basic_design/02_backend/03_apis/API-060.md#API-060) 課金プロバイダ Webhook 受信 |
| 対象データ | AI 推論の要求/応答項目、メール送信の要求/応答項目、メール配信状態 Webhook の受信項目、課金プロバイダ Webhook の受信項目 |
| 対象外 | AI 推論の連携方式・接続確認・矛盾検出アルゴリズム([EIF-001](../06_external_if/EIF-001.md#EIF-001))、回答可否判定・信頼度しきい値判定([IPO-001](../04_ipo/IPO-001.md#IPO-001))、通知配信状態の遷移・保持期間([STS-001](../01_state_transitions/STS-001.md#STS-001) 以外の通知状態遷移は状態モデル)、課金アカウント・サブスクリプション・請求書状態への反映ロジック([SYS-004](../../02_basic_design/02_backend/01_system/SYS-004.md#SYS-004))、抑制リスト運用 |

| 項目名 | チェック内容 | タイミング | エラー時メッセージ | 備考 |
|----|----|----|----|----|
| 質問文(I-01) | 未入力・最大長チェック | サーバー(上位 API-038 で実施済み) | [ERR-001](../../02_basic_design/05_errors/ERR-001.md#ERR-001) | 本 IF は上位からの整形済み値を受け取る前提([IO-001](IO-001.md#IO-001) §5) |
| 署名ヘッダ(I-08) | Resend(Svix)署名(HMAC-SHA256)の検証 | サーバー | [ERR-031](../../02_basic_design/05_errors/ERR-031.md#ERR-031) | 検証失敗時は 401 とし受信を取り込まない([API-059](../../02_basic_design/02_backend/03_apis/API-059.md#API-059) P-01) |
| 配信イベント種別(I-09) | 重複受信チェック(同一イベントの再送) | サーバー | [ERR-032](../../02_basic_design/05_errors/ERR-032.md#ERR-032) | 重複時は前回結果を 200 で返し冪等に処理(状態変更なし) |

各 Webhook 受信項目の取りうる値の全集合と意味を示す。値の全集合の正本は各 API の `## 列挙値` / インターフェース定義。

## 7. 後続工程への引き継ぎ事項

テスト設計・DB 物理設計・詳細ロジック設計へ引き継ぐ観点を挙げる。AI 連携の詳細な例外処理・接続確認は EIF、課金プロバイダ通知の取込ロジックは SYS を参照する。

- AI 推論応答(O-01〜O-06)の結果種別ごとの分岐(`answered` / `unanswerable` 理由 3 値 / `error` 理由 3 値)はテスト設計でケース化する。タイムアウト・プロバイダエラー・レート制限の例外処理詳細は [EIF-001](../06_external_if/EIF-001.md#EIF-001) を参照する。
- メール配信状態 Webhook(I-08〜I-10)・課金プロバイダ Webhook(I-11〜I-13)は署名検証失敗・重複受信・未知イベント種別の 3 パターンをテスト設計でケース化する。
- 課金プロバイダ通知の課金アカウント・サブスクリプション・請求書状態への反映ロジックは [SYS-004](../../02_basic_design/02_backend/01_system/SYS-004.md#SYS-004)、決済失敗からサスペンションへの遷移は [課金・請求設計 §5.1](../../02_basic_design/05_billing-design.md#51-決済失敗からサスペンションへ) を参照する。
- 論理項目 ↔ 物理カラムの対応は [DBP-001](../07_db_physical/DBP-001.md#DBP-001)(質問ログ)を参照し、通知配信ログ・課金 Webhook 受信ログの物理対応は該当テーブル([TBL-026](../../02_basic_design/02_backend/04_database/TBL-026.md#TBL-026)・[TBL-032](../../02_basic_design/02_backend/04_database/TBL-032.md#TBL-032))で確定する。

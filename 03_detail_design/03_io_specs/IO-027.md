# <span id="IO-027"></span>IO-027: 通知配信状態 入出力

> **この入出力設計書は「通知配信状態画面と通知配信状態サマリ API の入力/出力項目・入力チェック・編集加工」を論理項目単位で定義します。**

*種別 入出力設計 ・ ステータス ドラフト*

| 項目 | 値 |
|----|----|
| 入出力設計ID | IO-027 |
| 業務ユースケースID | [UC-077](../../01_requirements/04_business_usecases/UC-077.md#UC-077) |
| 関連画面 | [SCR-031](../../02_basic_design/01_frontend/01_screens/SCR-031.md#SCR-031) |
| 関連 API | [API-061](../../02_basic_design/02_backend/03_apis/API-061.md#API-061) |

## 1. 目的

本入出力設計は、オーナー / メンバーが当該プロジェクトの通知配信状態(送信待ち・送信済み・配信済み・失敗・バウンス・苦情・送信停止)と失敗件数・バウンス件数を確認する一連の入力/出力項目を、画面と API を突き合わせて論理項目単位で確定する。橋渡しする画面は [SCR-031](../../02_basic_design/01_frontend/01_screens/SCR-031.md#SCR-031) の初期表示(EVT-01)・「最新の状態に更新」(EVT-02)、API は [API-061](../../02_basic_design/02_backend/03_apis/API-061.md#API-061) である。

## 2. 対象範囲

本設計が対象とする画面・API・データと対象外を明示する。配信状態そのものの遷移契機・更新処理は別文書へ委ねる。

| 区分 | 対象 |
|----|----|
| 対象画面 | [SCR-031](../../02_basic_design/01_frontend/01_screens/SCR-031.md#SCR-031) 通知配信状態(初期表示 EVT-01・更新 EVT-02) |
| 対象API | [API-061](../../02_basic_design/02_backend/03_apis/API-061.md#API-061) 通知配信状態サマリ |
| 対象データ | 対象プロジェクト・集計対象期間、件数サマリー(送信済み・配信済み・失敗・バウンス)、通知種別ごとの配信状態内訳 |
| 対象外 | 配信状態の遷移契機・更新処理([STS-009](../01_state_transitions/STS-009.md#STS-009))・サプレスリスト登録処理・再送処理([SYS-025](../../02_basic_design/02_backend/01_system/SYS-025.md#SYS-025)) |

## 3. 入力項目一覧

画面から受け取り API へ渡す入力項目を論理項目単位で定義する。本画面に利用者による入力項目は無く、初期表示・更新操作の契機で自動的に確定する値のみを扱う。

| 項目ID | 項目名 | データ型 | 必須 | 初期値 | 入力制約 | 入力例 | 備考 |
|----|----|----|----|----|----|----|----|
| I-01 | 対象プロジェクト | 文字列 | ◯ | — | 有効なプロジェクト識別子 | — | 画面表示中のプロジェクトコンテキストから確定。画面には表示しない。API リクエスト `projectId` |
| I-02 | 集計対象期間 | 列挙値 | — | `current_month` | [API-061](../../02_basic_design/02_backend/03_apis/API-061.md#API-061) `## リクエスト` に定義された値(`current_month`) | `current_month` | 画面に選択操作は無く既定値のみ送信。画面には表示しない。API リクエスト `period` |

## 4. 出力項目一覧

API レスポンス・画面へ返す出力項目を論理項目単位で定義する。取得成功時と集計前 / 取得失敗時で画面の表示分岐が生じる。

| 項目ID | 項目名 | データ型 | 表示形式 | 表示条件 | 備考 |
|----|----|----|----|----|----|
| O-01 | 集計対象期間 | 文字列 | 画面見出し下に表示(SCR-031 §4 #2) | 取得成功時 | 集計対象とした期間。API レスポンスの集計期間情報に対応 |
| O-02 | 最終更新日時 | 日時 | 画面見出し下に表示(SCR-031 §4 #2) | 取得成功時 | サマリー集計の最終更新日時。API レスポンスの集計時刻情報に対応 |
| O-03 | 送信済み件数 | 数値 | カード表示(SCR-031 §4 #4) | 取得成功時 | 当該プロジェクトの送信済み件数サマリー |
| O-04 | 配信済み件数 | 数値 | カード表示(SCR-031 §4 #5) | 取得成功時 | 当該プロジェクトの配信済み件数サマリー |
| O-05 | 失敗件数 | 数値 | カード表示(SCR-031 §4 #6) | 取得成功時 | 当該プロジェクトの失敗件数サマリー |
| O-06 | バウンス件数 | 数値 | カード表示(SCR-031 §4 #7) | 取得成功時 | 当該プロジェクトのバウンス件数サマリー |
| O-07 | 配信状態の内訳 | 配列 | テーブル表示(SCR-031 §4 #8) | 取得成功時 | 通知種別ごとの状態別件数の一覧。行内訳は O-08〜O-15 |
| O-08 | 通知種別 | 文字列 | テーブル行の通知種別列(SCR-031 §4 #8) | 取得成功時 | 配信状態の内訳テーブルの行を識別する通知種別。値の全集合は [TBL-026](../../02_basic_design/02_backend/04_database/TBL-026.md#コード値区分値) `notification_type` |
| O-09 | 送信待ち件数 | 数値 | テーブル行の送信待ち列(SCR-031 §4 #8) | 取得成功時 | 通知種別ごとの `queued`/`sending` 件数。状態の意味は [状態モデル §8.2](../../02_basic_design/08_state-model.md#82-通知配信状態) |
| O-10 | 送信済み件数(内訳) | 数値 | テーブル行の送信済み列(SCR-031 §4 #8) | 取得成功時 | 通知種別ごとの `sent` 件数 |
| O-11 | 配信済み件数(内訳) | 数値 | テーブル行の配信済み列(SCR-031 §4 #8) | 取得成功時 | 通知種別ごとの `delivered` 件数 |
| O-12 | 失敗件数(内訳) | 数値 | テーブル行の失敗列(SCR-031 §4 #8) | 取得成功時 | 通知種別ごとの `failed` 件数 |
| O-13 | バウンス件数(内訳) | 数値 | テーブル行のバウンス列(SCR-031 §4 #8) | 取得成功時 | 通知種別ごとの `bounced` 件数 |
| O-14 | 苦情件数 | 数値 | テーブル行の苦情列(SCR-031 §4 #8) | 取得成功時 | 通知種別ごとの `complained` 件数 |
| O-15 | 送信停止件数 | 数値 | テーブル行の送信停止列(SCR-031 §4 #8) | 取得成功時 | 通知種別ごとの `suppressed` 件数 |
| O-16 | 空状態表示 | 文字列 | 空状態案内(SCR-031 §4 #9) | 集計前 / 取得失敗時 | O-01〜O-15 は表示しない |

## 5. 入力チェック仕様

各入力項目のチェック内容・実施主体(クライアント / サーバー)・エラー時参照を定義する。本画面に利用者入力は無いため、チェックは対象プロジェクトの権限判定・存在確認に限られる。

| 項目名 | チェック内容 | タイミング | エラー時メッセージ | 備考 |
|----|----|----|----|----|
| 対象プロジェクト(I-01) | 当該プロジェクトに割当のあるオーナー / メンバーかの判定 | サーバー | [ERR-030](../../02_basic_design/05_errors/ERR-030.md#ERR-030) | サーバーのみ判定([API-061](../../02_basic_design/02_backend/03_apis/API-061.md#API-061) P-01)。403 応答時は EVT-01/02 いずれも空状態表示(SCR-031 §4 #9)へ遷移 |
| 対象プロジェクト(I-01) | 対象プロジェクトの存在確認 | サーバー | [ERR-011](../../02_basic_design/05_errors/ERR-011.md#ERR-011) | 404 応答時は EVT-01/02 いずれも空状態表示(SCR-031 §4 #9)へ遷移 |
| 集計対象期間(I-02) | 列挙値チェック | サーバー | [ERR-001](../../02_basic_design/05_errors/ERR-001.md#ERR-001) | 画面からは既定値のみ送信するため通常経路では発生しない |

## 6. 編集・加工仕様

入力値の正規化・出力値の整形など編集加工を定義する。

| 項目名 | 加工内容 | 加工タイミング | 備考 |
|----|----|----|----|
| 配信状態の内訳(O-07) | 通知種別ごとに状態別件数を集約して行配列へ整形する | 集計時(サーバー) | 送信停止(抑制)対象を加味して区分集計する([API-061](../../02_basic_design/02_backend/03_apis/API-061.md#API-061) P-03) |
| 件数サマリー(O-03〜O-06) | 通知種別横断で件数を合算する | 集計時(サーバー) | 内訳(O-09〜O-15)の合算値と整合させる |

## 7. 後続工程への引き継ぎ事項

テスト設計・DB 物理設計・詳細ロジック設計へ引き継ぐ観点を挙げる。配信状態の遷移契機・サプレスリスト登録処理の詳細は別文書へ委ねる。

- 集計前(実績なし)と取得失敗(403/404)を区別せず空状態表示(O-16)へ統一する画面挙動はテスト設計でケース化する。
- 配信状態の内訳(O-07)の合算値と件数サマリー(O-03〜O-06)の整合はテスト設計で境界ケース(0 件・大量件数)を含めて検証する。
- 配信状態(`delivery_state`)の遷移契機・更新処理は [STS-009](../01_state_transitions/STS-009.md#STS-009)、サプレスリスト([TBL-007](../../02_basic_design/02_backend/04_database/TBL-007.md#TBL-007))登録契機は同書を参照する。
- 論理項目 ↔ 物理カラムの対応は `H_NOTIF_LOGS`([TBL-026](../../02_basic_design/02_backend/04_database/TBL-026.md#TBL-026))の DB 物理設計 [DBP-012](../07_db_physical/DBP-012.md#DBP-012) で確定する。

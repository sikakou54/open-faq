# <span id="IO-003"></span>IO-003: アカウント登録 入出力

> **この入出力設計書は「アカウント登録画面と新規登録 API の入力/出力項目・入力チェック・編集加工」を論理項目単位で定義します。**

*種別 入出力設計 ・ ステータス ドラフト*

| 項目 | 値 |
|----|----|
| 入出力設計ID | IO-003 |
| 業務ユースケースID | [UC-002](../../01_requirements/04_business_usecases/UC-002.md#UC-002) ・ [UC-003](../../01_requirements/04_business_usecases/UC-003.md#UC-003) |
| 関連画面 | [SCR-002](../../02_basic_design/01_frontend/01_screens/SCR-002.md#SCR-002) |
| 関連 API | [API-001](../../02_basic_design/02_backend/03_apis/API-001.md#API-001) |

## 1. 目的

本入出力設計は、未認証ユーザーが表示名・メールアドレス・パスワードを入力し規約へ同意してアカウントを新規作成する一連の入力/出力項目を、画面と API を突き合わせて論理項目単位で確定する。橋渡しする画面は [SCR-002](../../02_basic_design/01_frontend/01_screens/SCR-002.md#SCR-002) の登録入力・送信(EVT-05)、API は [API-001](../../02_basic_design/02_backend/03_apis/API-001.md#API-001) 新規登録である。

## 2. 対象範囲

本設計が対象とする画面・API・データと対象外を明示する。確認メール再送・メール確認自体の入出力は別 API 処理のため対象外とする。

| 区分 | 対象 |
|----|----|
| 対象画面 | [SCR-002](../../02_basic_design/01_frontend/01_screens/SCR-002.md#SCR-002) アカウント登録(登録入力・送信 EVT-05) |
| 対象API | [API-001](../../02_basic_design/02_backend/03_apis/API-001.md#API-001) 新規登録(`POST /auth/signup`) |
| 対象データ | 表示名・メールアドレス・パスワード・利用規約 / プライバシーポリシー同意、受付結果・送信先メールアドレスの各出力項目 |
| 対象外 | 確認メール再送(`POST /auth/signup/email-verification`)・メール確認([SCR-018](../../02_basic_design/01_frontend/01_screens/SCR-018.md#SCR-018) / [API-006](../../02_basic_design/02_backend/03_apis/API-006.md#API-006))・課金アカウント作成・プロジェクト作成 |

## 3. 入力項目一覧

登録画面から受け取り API へ渡す入力項目を論理項目単位で定義する。文字数上限などの具体値は基本設計へ ID 参照する。

| 項目ID | 項目名 | データ型 | 必須 | 初期値 | 入力制約 | 入力例 | 備考 |
|----|----|----|----|----|----|----|----|
| I-01 | 表示名 | 文字列 | ◯ | — | 1〜100 文字([SCR-002](../../02_basic_design/01_frontend/01_screens/SCR-002.md#SCR-002) §4 #1) | 山田 太郎 | SCR-002 §4 #1 に対応。API リクエスト `name` |
| I-02 | メールアドレス | 文字列 | ◯ | — | RFC 5322 準拠のメール形式・最大 254 文字([SCR-002](../../02_basic_design/01_frontend/01_screens/SCR-002.md#SCR-002) §4 #2 / [API-001](../../02_basic_design/02_backend/03_apis/API-001.md#API-001) `## リクエスト`) | user@example.com | SCR-002 §4 #2 に対応。API リクエスト `email` |
| I-03 | パスワード | 文字列 | ◯ | — | 12 文字以上、英大文字 / 小文字 / 数字 / 記号のうち 3 種類以上([FR-006](../../01_requirements/02_functional_requirement/01_account-fr.md#FR-006) / [RULE-003](../../01_requirements/01_business_requirement/08_rule.md#RULE-003))・最大 128 文字([SCR-002](../../02_basic_design/01_frontend/01_screens/SCR-002.md#SCR-002) §4 #3) | — | SCR-002 §4 #3 に対応。画面には表示しない。API リクエスト `password` |
| I-04 | 利用規約・プライバシーポリシー同意 | 真偽値 | ◯ | 未チェック | `true` であること(未チェックは送信不可) | true | SCR-002 §4 #4 に対応。API リクエスト `agreedTerms` / `agreedPrivacy`(画面上は単一チェックボックスで両方に同意) |
| I-05 | 冪等キー | 文字列 | — | — | 再送重複防止の一意識別子(ULID) | — | 任意。[API-001](../../02_basic_design/02_backend/03_apis/API-001.md#API-001) 認証ヘッダ `Idempotency-Key` |

## 4. 出力項目一覧

API レスポンス・画面へ返す出力項目を論理項目単位で定義する。

| 項目ID | 項目名 | データ型 | 表示形式 | 表示条件 | 備考 |
|----|----|----|----|----|----|
| O-01 | 受付結果 | 真偽値 | 画面非表示 | 常時 | 受理時 `true`。API `ok`。成功時は [SCR-018](../../02_basic_design/01_frontend/01_screens/SCR-018.md#SCR-018) へ遷移する契機となる |
| O-02 | 送信先メールアドレス | 文字列 | メール確認待ち画面に表示 | 受付成功時 | 登録に用いたメールアドレス(I-02)をそのまま画面遷移先へ引き継ぐ。SCR-018 §4 #2 に対応 |

## 6. 編集・加工仕様

入力値の正規化・出力値の整形など編集加工を定義する。最大長・重複判定は加工後の値に対して行う。

| 項目名 | 加工内容 | 加工タイミング | 備考 |
|----|----|----|----|
| メールアドレス(I-02) | 前後空白の除去・小文字化 | 保存前(サーバー) | 重複判定(§5)は加工後の値で行う |
| 表示名(I-01) | 前後空白の除去 | 保存前(サーバー) | 文字数チェック(§5)は加工後の値で判定する |
| 送信先メールアドレス(O-02) | 登録時のメールアドレス(I-02、加工後)をそのまま整形せず引き継ぐ | 応答生成時(サーバー) | SCR-018 の送信先メールアドレス表示に用いる |

## 7. 後続工程への引き継ぎ事項

テスト設計・DB 物理設計・詳細ロジック設計へ引き継ぐ観点を挙げる。

- 表示名(I-01)・パスワード(I-03)の境界値(文字数上限・強度種類数)、メールアドレス(I-02)の重複判定はテスト設計でパターン化する。
- 利用規約・プライバシーポリシー同意(I-04)は画面上単一チェックボックスで `agreedTerms` / `agreedPrivacy` の 2 値へ分岐する対応関係をテスト設計でケース化する。
- メール確認トークンの発行・有効期限は [SCR-018](../../02_basic_design/01_frontend/01_screens/SCR-018.md#SCR-018) 側の入出力設計(別 IO)および[システム仕様書](../../02_basic_design/07_system-spec.md)へ引き継ぐ。
- 論理項目 ↔ 物理カラムの対応は [DBP-002](../07_db_physical/DBP-002.md#DBP-002) で確定する。

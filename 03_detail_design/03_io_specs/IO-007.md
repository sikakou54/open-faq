# <span id="IO-007"></span>IO-007: 連絡先メール確認完了 入出力

> **この入出力設計書は「連絡先メール確認完了画面とプロジェクト連絡先メール確認 API・連絡先確認メール再送 API の入力/出力項目・入力チェック・編集加工」を論理項目単位で定義します。**

*種別 入出力設計 ・ ステータス ドラフト*

| 項目 | 値 |
|----|----|
| 入出力設計ID | IO-007 |

## 1. 目的

本入出力設計は、連絡先メールアドレスの所有者が確認案内をたどって到達し、確認情報の検証結果(完了 / 期限切れ・無効 / 既使用)を受け取る一連の入力/出力項目を、画面と API を突き合わせて論理項目単位で確定する。橋渡しする画面は [SCR-024](../../02_basic_design/01_frontend/01_screens/SCR-024.md#SCR-024) の初期表示(EVT-01)、API は [API-009](../../02_basic_design/02_backend/03_apis/API-009.md#API-009) である。確認が行えない状態からの再送は [API-011](../../02_basic_design/02_backend/03_apis/API-011.md#API-011) が担う。

## 2. 対象範囲

対象とする画面・API・データと、対象外を明示する。

| 区分 | 対象 |
|----|----|
| 対象画面 | [SCR-024](../../02_basic_design/01_frontend/01_screens/SCR-024.md#SCR-024) 連絡先メール確認完了(初期表示 EVT-01) |
| 対象API | [API-009](../../02_basic_design/02_backend/03_apis/API-009.md#API-009) プロジェクト連絡先メール確認 ・ [API-011](../../02_basic_design/02_backend/03_apis/API-011.md#API-011) 連絡先確認メール再送 |
| 対象データ | 連絡先確認トークン(パスパラメータ)・確認結果(対象プロジェクト名 / 確認日時)・確認できない場合の状態区分、再送対象プロジェクト識別子・再送結果 |
| 対象外 | 連絡先メールの登録・変更([SCR-005](../../02_basic_design/01_frontend/01_screens/SCR-005.md#SCR-005) EVT-05)・確認メールの初回送信契機・[SCR-005](../../02_basic_design/01_frontend/01_screens/SCR-005.md#SCR-005) の画面項目一覧全体(本設計は EVT-06 の再送呼び出しのみを対象とする) |

## 3. 入力項目一覧

画面から受け取り API へ渡す入力項目を論理項目単位で定義する。文字数上限などの具体値は基本設計へ ID 参照する。

| 項目ID | 項目名 | データ型 | 必須 | 初期値 | 入力制約 | 入力例 | 備考 |
|----|----|----|----|----|----|----|----|
| I-01 | 連絡先確認トークン | 文字列 | ◯ | — | 確認案内メール内リンクの URL パスから取得する有効なトークン([API-009](../../02_basic_design/02_backend/03_apis/API-009.md#API-009) `## リクエスト` Path Parameters) | — | 画面には表示しない。SCR-024 初期表示(EVT-01)で自動送信。API パスパラメータ `token` |
| I-02 | 冪等キー | 文字列 | — | — | 再送重複防止の一意識別子 | — | 任意。API-009 呼び出し時の重複防止([API-009](../../02_basic_design/02_backend/03_apis/API-009.md#API-009) `Idempotency-Key`) |
| I-03 | 再送対象プロジェクト識別子 | 文字列 | ◯ | — | 有効なプロジェクト識別子(確認メール再送時) | proj_01HMK4Z8Q... | SCR-005 の「確認メールを再送」(EVT-06)押下時に画面状態から確定して送信。画面には入力欄を持たない。API-011 リクエスト `projectId` |

## 4. 出力項目一覧

API レスポンス・画面へ返す出力項目を論理項目単位で定義する。API-009 の確認結果は成功 / 期限切れ・無効 / 使用済みでレスポンスが分岐する。

| 項目ID | 項目名 | データ型 | 表示形式 | 表示条件 | 備考 |
|----|----|----|----|----|----|
| O-01 | 確認対象プロジェクト名 | 文字列 | 確認完了メッセージ内に表示(SCR-024 §4 #1) | トークン検証成功時 | API-009 レスポンス `projectName` |
| O-02 | 確認日時 | 文字列(ISO 8601) | 確認完了メッセージ内に表示(SCR-024 §4 #1) | トークン検証成功時 | API-009 レスポンス `verifiedAt` |
| O-03 | 確認結果区分 | 列挙値 | — | 常時 | `verified`(確認完了)/ `expired_or_invalid`(期限切れ・無効)/ `already_used`(既使用)。取りうる値と意味は §列挙値。SCR-024 §4 #1〜#4 の表示分岐に対応 |
| O-04 | 閉じるボタン活性状態 | 真偽値 | ボタン活性(SCR-024 §4 #2) | 確認結果区分が `verified` | トークン検証成功時のみ活性表示 |
| O-05 | 再送受付結果 | 真偽値 | — | 再送要求時 | 受理時 `true`。API-011 レスポンス `ok` |
| O-06 | 再送可能までの残り秒数 | 数値 | トースト文言へ反映(SCR-005 §7 EVT-06) | 再送要求時 | レート制限内は `0`。API-011 レスポンス `retryAfter` |

## 6. 編集・加工仕様

入力値の正規化・出力値の整形など編集加工を定義する。

| 項目名 | 加工内容 | 加工タイミング | 備考 |
|----|----|----|----|
| 連絡先確認トークン(I-01) | URL パスからの抽出 | 受信時(クライアント) | メール内リンクの URL パスをそのまま API パスパラメータへ引き渡す。画面上での加工は行わない |
| 確認日時(O-02) | 表示用日時形式への整形 | 表示時(クライアント) | ISO 8601 形式のレスポンス値を確認完了メッセージ(SCR-024 §4 #1)の文言へ組み込む |
| 再送可能までの残り秒数(O-06) | トースト文言への埋め込み | 応答受信時(クライアント) | 「再送は X 分後に再試行できます」(SCR-005 §7 EVT-06)へ分単位に換算して反映 |

## 列挙値

出力の確認結果区分(O-03)の取りうる値と意味を示す。値は API-009 のレスポンス分岐(成功時 200 / エラー時 410・404)から画面が導出する。

| 項目 | 取りうる値 | 意味 |
|----|----|----|
| 確認結果区分(O-03) | `verified` | 確認完了(トークン検証成功・連絡先メールを確認済みとして記録) |
| 確認結果区分(O-03) | `expired_or_invalid` | 確認案内の有効期限切れ、またはトークンが存在しない([ERR-006](../../02_basic_design/05_errors/ERR-006.md#ERR-006) / [ERR-008](../../02_basic_design/05_errors/ERR-008.md#ERR-008)) |
| 確認結果区分(O-03) | `already_used` | トークンが既に使用済み(既に確認完了済み)([ERR-007](../../02_basic_design/05_errors/ERR-007.md#ERR-007)) |

> [!NOTE]
> **確認結果区分は API-009 のレスポンスから画面側で 1 つに確定する。** 期限切れ(`TOKEN_EXPIRED`)と存在しない(`TOKEN_NOT_FOUND`)は SCR-024 上では同一の期限切れ・無効メッセージ(§4 #3)へ集約し、区分としては `expired_or_invalid` にまとめる。既使用(`TOKEN_USED`)のみ既使用メッセージ(§4 #4)として区別する。

## 7. 後続工程への引き継ぎ事項

テスト設計・DB 物理設計・詳細ロジック設計へ引き継ぐ観点を挙げる。

- 確認結果区分(O-03)の 3 状態(`verified` / `expired_or_invalid` / `already_used`)の表示分岐と、再送要求(I-03)のレート制限成功 / 失敗の組み合わせはテスト設計でケース化する。
- 連絡先確認トークンの有効期限([システム仕様書 §4](../../02_basic_design/07_system-spec.md#4-データ保持期間削除猶予))境界(期限直前 / 直後)の検証はテスト設計でパターン化する。
- 論理項目 ↔ 物理カラムの対応は [DBP-002](../07_db_physical/DBP-002.md#DBP-002)(`T_ACCESS_TOKENS`)・[DBP-006](../07_db_physical/DBP-006.md#DBP-006)(`M_PROJECTS`)で確定する。

# <span id="BAT-008"></span>BAT-008: 配信失敗通知の再送

> **本バッチは、定期起動を契機に配信に失敗した通知メールを検出し、抑制リスト照合・再送上限判定・指数バックオフに基づいて再送上限の範囲内で自動的に再送する実行機構(起動・排他・冪等・監視・異常終了時の扱い)の詳細設計を定義します。配信状態の遷移そのものは [STS-009](../01_state_transitions/STS-009.md#STS-009) を正本とし、本バッチでは重複定義しません。**

*種別 バッチ処理設計 ・ ステータス ドラフト*

## 1. 目的

本バッチは、定期起動により配信に失敗した通知メール([TBL-026](../../02_basic_design/02_backend/04_database/TBL-026.md#TBL-026) `delivery_state = failed`)を検出し、宛先がメールサプレスリスト([TBL-007](../../02_basic_design/02_backend/04_database/TBL-007.md#TBL-007))非該当かつ再送回数が上限未満の対象を、指数バックオフの次回間隔到達分から上限の範囲内で再送する。詳細化元のシステム設計は [SYS-025](../../02_basic_design/02_backend/01_system/SYS-025.md#SYS-025)(配信失敗通知の再送)であり、その `batch` 種別・トリガー(定期起動の再送処理)を Cloudflare Cron Triggers の定時実行へ写像する。配信状態(`queued`/`sent`/`failed`/`bounced`等)の遷移契機・ガード条件そのものは [STS-009](../01_state_transitions/STS-009.md#STS-009) を正本とし、本バッチは起動制御・抑制リスト照合・再送上限判定・指数バックオフ判定・排他・監視・異常終了時の扱いを定義する。

## 2. バッチ概要

本バッチは定期スケジュールにより自律的に起動する再送処理である。起動方式・多重起動可否・リトライ有無・想定規模を示す。

| 項目 | 値 |
|----|----|
| バッチ ID | BAT-008 |
| バッチ名 | 配信失敗通知の再送 |
| 起動方式 | Cron Triggers(定時。[SYS-025](../../02_basic_design/02_backend/01_system/SYS-025.md#SYS-025) の `batch` 種別・「定期起動の再送処理(失敗分の検出)」を写像) |
| 起動タイミング | 定期起動(周期は [システム仕様書 §7](../../02_basic_design/07_system-spec.md#7-バッチ運用設計値)(Cron スケジュール [SYS-025](../../02_basic_design/02_backend/01_system/SYS-025.md#SYS-025)・15 分ごと)を正本とする。指数バックオフの最短間隔([システム仕様書 §3](../../02_basic_design/07_system-spec.md#3-タイムアウトセッション認証) の通知再送バックオフ)を検出できる粒度) |
| 多重起動可否 | 不可(同時実行させない) |
| リトライ有無 | あり(個々の再送呼び出し失敗時は当該対象のみ次回バッチで再評価。バッチ全体の再送方針は [NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040) を参照し本文へ回数を再掲しない) |
| 想定処理件数 | 1 起動あたりの再送対象件数は通知ログの想定規模([NFR-069](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-069))に準拠 |
| 想定処理時間 | 全対象の再送可否判定・再送呼び出しが次回起動までの周期内に完了する範囲(具体的な許容時間は運用設計に委ねる) |

## 3. 入力

本バッチは定期起動時に再送対象候補を取得し、抑制リストと照合する。

| 入力元 | 内容 | 取得条件 | 備考 |
|----|----|----|----|
| Cron Triggers 起動イベント | 起動時刻 | 定期スケジュール到達時 | 起動制御は §6 |
| 通知ログ | 再送対象候補(`delivery_state = failed` の通知・試行回数・失敗日時) | `delivery_state = 'failed'` かつ試行回数が[システム仕様書 §3](../../02_basic_design/07_system-spec.md#3-タイムアウトセッション認証)の通知再送回数上限未満の行を取得する | [TBL-026](../../02_basic_design/02_backend/04_database/TBL-026.md#TBL-026) |
| メールサプレスリスト | 再送対象候補の宛先が送信停止対象か否か | 通知ログの受信者メール HMAC でサプレスリストを照合する | [TBL-007](../../02_basic_design/02_backend/04_database/TBL-007.md#TBL-007) |

## 4. 処理内容

本バッチの処理を実行順に示す。[SYS-025](../../02_basic_design/02_backend/01_system/SYS-025.md#SYS-025) の処理項目([PR-01](../../02_basic_design/02_backend/01_system/SYS-025.md#PR-01)〜[PR-06](../../02_basic_design/02_backend/01_system/SYS-025.md#PR-06))に対応する。配信状態遷移のガード条件・更新操作の詳細は [STS-009](../01_state_transitions/STS-009.md#STS-009) §5 を正本とし、本表では重複定義しない。

| No | 処理 | 条件 | 正常時 | 異常時 | 備考 |
|----|----|----|----|----|----|
| 1 | 多重起動チェック | Cron Triggers 起動時 | 実行中インスタンスが無ければ後続へ進む | 実行中インスタンスがあれば本起動をスキップして終了(§6) | 排他制御は §6 |
| 2 | 再送対象抽出(SYS-025 [PR-02](../../02_basic_design/02_backend/01_system/SYS-025.md#PR-02)) | 多重起動チェック通過時 | `delivery_state = failed` かつ試行回数が通知再送回数上限([システム仕様書 §3](../../02_basic_design/07_system-spec.md#3-タイムアウトセッション認証))未満の通知を再送対象候補として抽出する | 抽出失敗はバッチ全体を異常終了とする | [TBL-026](../../02_basic_design/02_backend/04_database/TBL-026.md#TBL-026) |
| 3 | 抑制リスト照合(SYS-025 [PR-03](../../02_basic_design/02_backend/01_system/SYS-025.md#PR-03)) | 再送対象候補ごとに実行 | 宛先の受信者メール HMAC が[メールサプレスリスト](../../02_basic_design/02_backend/04_database/TBL-007.md#TBL-007)に非該当であれば次判定へ進む | 該当する場合は再送せず No.6(確定失敗記録)へ | 認証関連の最優先通知を除く扱いは UC-062 例外フロー参照。本バッチの対象(配信失敗通知の再送)には当該除外を適用しない |
| 4 | 指数バックオフ判定(SYS-025 [PR-03](../../02_basic_design/02_backend/01_system/SYS-025.md#PR-03)) | 抑制リスト非該当の対象 | 直近の失敗日時からの経過時間が試行回数に応じた通知再送バックオフ([システム仕様書 §3](../../02_basic_design/07_system-spec.md#3-タイムアウトセッション認証))の次回間隔に達していれば再送可と判定する | 次回間隔未達の対象は本回はスキップし次回起動で再評価する | バックオフ段数は試行回数に対応させる |
| 5 | 通知再送(SYS-025 [PR-04](../../02_basic_design/02_backend/01_system/SYS-025.md#PR-04)) | 抑制外 かつ 上限未満 かつ バックオフ次回間隔到達 | [API-058](../../02_basic_design/02_backend/03_apis/API-058.md#API-058) `send` を冪等キー付きで呼び出し、結果に応じて配信状態と試行回数を更新する(遷移詳細は [STS-009](../01_state_transitions/STS-009.md#STS-009)) | 再送呼び出し自体が失敗した場合は当該対象をスキップし次回バッチで再評価する(バッチ全体は継続) | 冪等キーの構成方針は §6 |
| 6 | 確定失敗記録(SYS-025 [PR-06](../../02_basic_design/02_backend/01_system/SYS-025.md#PR-06)) | 再送回数が通知再送回数上限([システム仕様書 §3](../../02_basic_design/07_system-spec.md#3-タイムアウトセッション認証))に到達 または 抑制リスト該当 | 再送を行わず確定失敗として記録する(状態値自体は `failed` のまま。[STS-009](../01_state_transitions/STS-009.md#STS-009) §5 参照) | 記録失敗は当該対象のみ失敗として次回バッチで再評価する | — |
| 7 | 完了判定 | 全再送対象候補の再送 / スキップ / 確定失敗のいずれかが完了したとき | 処理結果サマリ(再送件数・スキップ件数・確定失敗件数)を記録し正常終了する | 一部対象が失敗しても正常終了とし、失敗分は §8 の異常終了時の扱いに従い個別に扱う | バッチ全体としては部分失敗があっても継続完了とする |

## 6. 排他・並列処理

同一起動を単一インスタンスに限定し、再送対象単位の冪等キーで重複送信を防ぐ。

| 項目 | 内容 |
|----|----|
| 排他制御 | 同一起動時刻のバッチ実行を同時に 1 インスタンスへ限定する(多重起動不可)。ロックの具体実装は詳細実装に委ねる |
| 並列処理 | バッチ実行インスタンスは単一とし、インスタンス内では再送対象候補を順次または分割して処理する。並列度・分割方針は詳細実装に委ねる |
| 多重起動時の扱い | 実行中インスタンスが既にある場合、後着の起動はスキップして終了する |
| 冪等キー | [API-058](../../02_basic_design/02_backend/03_apis/API-058.md#API-058) `send` の呼び出しに通知ログ識別子 × 試行回数を用いた冪等キーを付与し、再送の重複送信を防止する([API-058](../../02_basic_design/02_backend/03_apis/API-058.md#API-058) P-01)。冪等性の保証は [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) を参照し、キーの物理構成は詳細実装に委ねる |

## 8. 異常終了時の扱い

異常検知時の対応・通知先・リトライ方針を示す。リトライ上限は本文に再掲せず NFR を参照する。

| 異常内容 | 対応 | 通知先 | リトライ | 備考 |
|----|----|----|----|----|
| 再送対象抽出の失敗(No.2) | バッチ全体を異常終了とし当該起動は未処理のまま終える | 運用 | あり([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040) を参照し次回起動または手動再起動で再試行) | 抽出前の異常のため部分再送は生じない |
| 対象単位の再送呼び出し失敗(No.5) | 当該対象のみ失敗として記録し、他対象の処理を継続 | 運用 | あり(次回起動時にバックオフ判定・再送回数上限判定に従い再評価) | 冪等キーにより再送の二重送信を防止 |
| 確定失敗記録の更新失敗(No.6) | 当該対象のみ失敗として記録し、他対象の処理を継続 | 運用 | あり(次回起動で再評価) | 記録未了のまま `failed` として残留し得るため次回起動での再評価で解消する |
| 想定処理時間超過・滞留 | 未処理対象の滞留を監視しアラートする | 運用 | 手動再処理 | 滞留アラートは [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) に従う |

## 9. 後続工程への引き継ぎ事項

テスト設計・運用設計へ渡す確認観点を箇条書きで示す。

- 境界値: 再送対象 0 件 / 再送上限到達直前 / 再送上限到達 / バックオフ次回間隔未達での抽出・判定・記録の動作。
- 冪等性: 同一対象への重複配信・自動リトライ時に二重送信が発生しないこと(冪等キーによる担保)。
- 排他: 同一起動時刻での多重起動が発生しないこと(Cron Triggers の遅延・重複発火時を含む)。
- 部分失敗: 一部対象の再送呼び出し・確定失敗記録が失敗しても他対象の処理が継続され、失敗分が次回起動で再評価されること。
- 状態遷移との整合: 配信状態(`failed`→`sent`/`failed`→`failed`)の遷移契機・ガード条件は [STS-009](../01_state_transitions/STS-009.md#STS-009) を正本とし、本バッチの再送呼び出し・確定失敗記録がその遷移契機と齟齬なく対応することをテストで確認する。
- 起動周期の正本: 定期起動の周期は [システム仕様書 §7](../../02_basic_design/07_system-spec.md#7-バッチ運用設計値)(Cron スケジュール [SYS-025](../../02_basic_design/02_backend/01_system/SYS-025.md#SYS-025)・`*/15 * * * *`・15 分ごと)を正本とする。この周期は指数バックオフの最短間隔を検出できる粒度である。Cron Triggers のスケジュール式はこの正本値に一致させる。

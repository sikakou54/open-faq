# <span id="BAT-007"></span>BAT-007: 運営お知らせ配信

> **本バッチは、配信予定日時の到来または運営による即時配信の指示を契機に、運営お知らせの宛先範囲を解決して対象のアカウント利用者へ受信箱お知らせを一括生成(fan-out)し、重要度に応じてお知らせメールを送信する処理の詳細設計を定義します。**

*種別 バッチ処理設計 ・ ステータス ドラフト*

## 1. 目的

本バッチは、運営が登録した運営お知らせの配信予定日時到来または即時配信の指示を契機に、宛先範囲(全アカウント / 特定オーナー / 特定プロジェクト)から対象のアカウント利用者を解決し、各利用者の受信箱へお知らせを生成(fan-out)したうえで、重要度に応じてお知らせメールを送信する。詳細化元のシステム設計は [SYS-022](../../02_basic_design/02_backend/01_system/SYS-022.md#SYS-022) であり、その `batch` トリガー・処理項目([PR-01](../../02_basic_design/02_backend/01_system/SYS-022.md#PR-01)〜[PR-06](../../02_basic_design/02_backend/01_system/SYS-022.md#PR-06))を Cloudflare Cron Triggers(配信予定日時到来の検知)と Cloudflare Queues(宛先解決後の受信箱生成・メール送信の fan-out 処理)へ写像する。

## 2. バッチ概要

本バッチは配信予定日時の到来を Cron Triggers が検知して起動する経路と、運営操作による即時配信指示を契機に Queues へ投入して起動する経路の 2 系統を持つ。起動方式・多重起動可否・リトライ有無・想定規模を示す。

| 項目 | 値 |
|----|----|
| バッチ ID | BAT-007 |
| バッチ名 | 運営お知らせ配信 |
| 起動方式 | Cron Triggers(配信予定日時到来の定期検知)+ Queues(即時配信指示の投入契機、および Cron 検知後の宛先ごと fan-out 処理) |
| 起動タイミング | 配信予定日時に到達したお知らせの定期検知(Cron)、または運営が即時配信を指示した時(Queues 投入) |
| 多重起動可否 | 不可(同一お知らせの配信処理は同時実行させない。宛先ごとの fan-out メッセージは並列処理可) |
| リトライ有無 | あり([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040) / [NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036) を参照。本文に回数を再掲しない) |
| 想定処理件数 | 宛先範囲が全アカウントの場合は全アカウント利用者数規模([NFR-069](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-069) を参照) |
| 想定処理時間 | メール通知の許容遅延([NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036))内 |

| 2 | 宛先解決(SYS-022 [PR-02](../../02_basic_design/02_backend/01_system/SYS-022.md#PR-02)) | 配信ジョブ読み込み完了時 | 宛先範囲(全アカウント / 特定オーナー / 特定プロジェクト)に応じて対象のアカウント利用者を解決する。特定オーナー / 特定プロジェクトは[TBL-011](../../02_basic_design/02_backend/04_database/TBL-011.md#TBL-011)の限定指定からユーザー ID 一覧を取得し、全アカウントは全利用者を対象とする | 宛先解決失敗時は当該お知らせの配信を失敗としてリトライへ回す | 解決したユーザー ID 一覧を宛先ごとの fan-out メッセージへ分割する |
| 3 | 宛先ありなし判定(SYS-022 [PR-02](../../02_basic_design/02_backend/01_system/SYS-022.md#PR-02)) | 宛先解決完了時 | 宛先が 1 件以上あれば No.4 へ進む | 宛先が 0 件の場合は No.7(宛先なし正常終了)へ | — |
| 4 | 冪等チェック + 受信箱お知らせ生成(SYS-022 [PR-03](../../02_basic_design/02_backend/01_system/SYS-022.md#PR-03)) | 宛先ありのとき、宛先ユーザーごとに fan-out | 「お知らせ ID × ユーザー ID」で[TBL-021](../../02_basic_design/02_backend/04_database/TBL-021.md#TBL-021)に既存行が無いことを確認したうえで、当該ユーザーの受信箱にお知らせを生成し、お知らせ受信者行を配信状態「配信待ち」で作成する | 既存行があれば当該ユーザーは生成済みとしてスキップし正常継続。受信箱生成の書き込み失敗時は当該ユーザーのみ失敗として再処理へ回し他ユーザーの処理は継続 | 受信箱生成は[TBL-022](../../02_basic_design/02_backend/04_database/TBL-022.md#TBL-022)への作成。配信状態の意味は[状態モデル §8.1](../../02_basic_design/08_state-model.md#81-お知らせ配信状態)を参照 |

## 6. 排他・並列処理

同一お知らせの配信処理は多重起動させず、宛先ユーザーごとの fan-out 処理は並列に消費して受信箱生成・メール送信の重複を冪等キーで防止する。

| 項目 | 内容 | 
|----|----|
| 排他制御 | 同一お知らせ ID に対する配信ジョブの同時実行を抑止する。既に処理中の同一お知らせ ID への配信契機は後着を待機またはスキップする |
| 並列処理 | 宛先解決後の受信箱生成・メール送信は宛先ユーザー単位で Queues による fan-out・並列消費を行う。並列度の具体値は負荷想定([NFR-069](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-069) / [NFR-074](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-074))を参照する |
| 多重起動時の扱い | 同一お知らせ ID の配信ジョブが重複して検知された場合は後着をスキップし、宛先ごとの fan-out メッセージが自動リトライ等で重複配信されても冪等キーの既処理判定でスキップする |
| 冪等キー | 「お知らせ ID × 宛先ユーザー ID」を冪等キーとし、受信箱お知らせ生成・お知らせ受信者行作成・メール送信の重複実行を抑止する。冪等性の保証は[NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041)を参照し、キーの物理構成は詳細実装に委ねる |

## 8. 異常終了時の扱い

異常検知時の対応・通知先・リトライ方針を示す。リトライ上限は本文に再掲せず NFR を参照する。

| 異常内容 | 対応 | 通知先 | リトライ | 備考 |
|----|----|----|----|----|
| 配信契機検知・お知らせ読み込み失敗 | 当該お知らせの配信ジョブを失敗として再処理へ回す | 運用 | あり([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040)) | 他のお知らせの配信ジョブは継続 |
| 宛先解決失敗 | 当該お知らせの配信ジョブを失敗として再処理へ回す | 運用 | あり(同上) | 宛先が一部でも解決できた fan-out 済みメッセージは重複させず冪等キーで整合を保つ |
| 受信箱生成・お知らせ受信者記録失敗(宛先ユーザー単位) | 当該ユーザーのみ失敗として再処理へ回す。他ユーザーの処理は継続 | 運用 | あり(同上。Queues 経由は自動リトライ、上限超過分は DLQ へ退避) | 部分失敗時も他宛先への配信は継続する |
| メール送信 API 呼び出し失敗(宛先ユーザー単位) | 受信箱お知らせは生成済みのまま、当該ユーザーの配信状態を配信失敗として記録し再処理へ回す | 運用 | あり(同上) | 受信箱お知らせの生成は取り消さない。メール再送は通知再送の業務([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040))が扱う |
| リトライ上限超過(DLQ 退避) | DLQ に退避し滞留を監視して手動再処理する | 運用 | 手動再処理 | 再処理待ち滞留は[NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041)に従いアラート |

## 9. 後続工程への引き継ぎ事項

テスト設計・運用設計へ渡す確認観点、および基本設計で未確定の点を箇条書きで示す。

- 境界値: 宛先 0 件(正常終了) / 宛先 1 件 / 全アカウント規模の大量宛先での fan-out 処理時間・完了サマリの正しさ。
- 冪等性: 同一お知らせ ID × 同一宛先ユーザー ID の fan-out メッセージが重複配信・自動リトライされても、受信箱お知らせ・お知らせ受信者行・メール送信が二重にならないこと。
- 排他: 同一お知らせ ID への配信契機(Cron 検知と運営の即時配信指示)が同時に発生しても配信ジョブが二重起動しないこと。
- 部分失敗: 一部宛先ユーザーの受信箱生成・メール送信が失敗しても他宛先の処理が継続し、失敗分がリトライ / DLQ へ正しく回ること。
- DLQ 運用: DLQ 滞留の検知・アラート・手動再処理手順([NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041))。
- 強制送信・オプトアウト: 重要度 `critical` は送信停止対象宛先を除きオプトアウト設定によらず送信されること、`normal` / `high` はオプトアウト設定・サプレスリスト([TBL-007](../../02_basic_design/02_backend/04_database/TBL-007.md#TBL-007))に従いスキップされること([MSG-012](../../02_basic_design/06_messages/MSG-012.md#MSG-012))。
- 特定プロジェクト宛先の展開: 特定プロジェクト宛先は、**お知らせ作成時(配信対象スコープ確定時)**に通知サービス(Service 層)がプロジェクト関係者(オーナー = `M_PROJECTS.owner_user_id` + 有効メンバー = `M_PRJ_USERS.valid=1`)をユーザー単位へ展開し、確定した宛先ユーザー ID 一覧を [TBL-011](../../02_basic_design/02_backend/04_database/TBL-011.md#TBL-011) へ保持する。本配信バッチ([SYS-022](../../02_basic_design/02_backend/01_system/SYS-022.md#SYS-022) の配信・No.2 宛先解決)は [TBL-011](../../02_basic_design/02_backend/04_database/TBL-011.md#TBL-011) の確定済み一覧を読み出すのみで、配信時にプロジェクト→ユーザーの再展開は行わない。

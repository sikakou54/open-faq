# <span id="BAT-004"></span>BAT-004: 質問数上限アラート通知

> **本バッチは、質問数上限アラートの通知要否判定([IPO-005](../04_ipo/IPO-005.md#IPO-005))で確定した宛先リストを受け取り、各宛先の受信箱へお知らせを生成し、上限アラートメールを送信して送信結果を記録する処理の詳細設計を定義します。閾値到達の判定・宛先解決は本バッチでは行いません(判定の正本は IPO-005)。**

*種別 バッチ処理設計 ・ ステータス ドラフト*

## 1. 目的

本バッチは、[IPO-005](../04_ipo/IPO-005.md#IPO-005)(SYS-017 `PR-01`〜`PR-03` の判定ロジック)が確定した「通知要・確定宛先リスト」を契機に、各宛先の受信箱へ上限アラートのお知らせを生成し、上限アラートメールを送信し、送信結果を記録する。詳細化元のシステム設計は [SYS-017](../../02_basic_design/02_backend/01_system/SYS-017.md#SYS-017)(質問数上限アラート通知)であり、その `async` 契機・処理項目([PR-04](../../02_basic_design/02_backend/01_system/SYS-017.md#PR-04)〜[PR-07](../../02_basic_design/02_backend/01_system/SYS-017.md#PR-07))を Cloudflare Queues の消費処理へ写像する。閾値到達検知・受信者ごとの初回到達判定・宛先解決は本バッチの対象外であり [IPO-005](../04_ipo/IPO-005.md#IPO-005) が正本である。

## 2. バッチ概要

本バッチは [IPO-005](../04_ipo/IPO-005.md#IPO-005) が通知要と判定したことを契機に、確定宛先リストを Cloudflare Queues へ投入したメッセージを消費して起動する非同期処理である。起動方式・多重起動可否・リトライ有無・想定規模を示す。

| 項目 | 値 |
|----|----|
| バッチ ID | BAT-004 |
| バッチ名 | 質問数上限アラート通知 |
| 起動方式 | Queues(イベント契機。[IPO-005](../04_ipo/IPO-005.md#IPO-005) の通知要否判定が「通知要」を確定した時にメッセージを投入し、その消費で起動) |
| 起動タイミング | アラート閾値到達契機([BAT-001](BAT-001.md#BAT-001))を受けて [IPO-005](../04_ipo/IPO-005.md#IPO-005) が通知要と判定するたび |
| 多重起動可否 | 可(宛先単位で並列消費。同一宛先への二重送信は §6 の冪等キーで抑止) |
| リトライ有無 | あり([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040) / [NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036) を参照。本文に回数を再掲しない) |
| 想定処理件数 | 1 通知契機あたり確定宛先リストの件数(オーナー + 当該プロジェクトの有効メンバー数の想定上限に準拠) |
| 想定処理時間 | メール通知の許容最大遅延([NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036)を参照)の許容内 |

## 3. 入力

本バッチは通知要否判定が確定した宛先リストのメッセージを入力とし、お知らせ生成・メール送信に必要な情報を参照する。

| 入力元 | 内容 | 取得条件 | 備考 |
|----|----|----|----|
| Queues メッセージ(通知要確定イベント) | 対象プロジェクト・請求年月・到達閾値・確定宛先リスト(ユーザー ID) | [IPO-005](../04_ipo/IPO-005.md#IPO-005) が通知要と判定した時に投入したメッセージを消費する | 判定・宛先解決は行わない。宛先リストは重複排除済み(IPO-005 で確定) |
| プロジェクト上限・アラート取得 | 到達閾値・当月質問数・上限件数(お知らせ / メール本文の表示値) | メッセージの `project_id` で当該プロジェクトの上限・利用状況を取得する | 参照は [API-046](../../02_basic_design/02_backend/03_apis/API-046.md#API-046)。表示値の正本は [TBL-009](../../02_basic_design/02_backend/04_database/TBL-009.md#TBL-009) / [TBL-020](../../02_basic_design/02_backend/04_database/TBL-020.md#TBL-020) |
| テンプレート定義 | 上限アラートの件名・本文テンプレート(サブ契機 `question_limit_threshold_reached`) | 通知種別に応じてテンプレートを解決する | テンプレート正本は [MSG-013](../../02_basic_design/06_messages/MSG-013.md#MSG-013) |

## 4. 処理内容

本バッチの処理を実行順に示す。正常時・異常時の分岐を各行で表す。SYS-017 の処理項目([PR-04](../../02_basic_design/02_backend/01_system/SYS-017.md#PR-04)〜[PR-07](../../02_basic_design/02_backend/01_system/SYS-017.md#PR-07))に対応する。

| No | 処理 | 条件 | 正常時 | 異常時 | 備考 |
|----|----|----|----|----|----|
| 1 | 冪等チェック | 常時 | メッセージの冪等キー(§6)が当該宛先で未処理なら後続へ進む | 処理済みなら以降をスキップし正常完了(当該宛先の二重通知防止) | 冪等キーは §6 参照 |
| 2 | 表示値の参照 | 冪等チェック通過時 | [API-046](../../02_basic_design/02_backend/03_apis/API-046.md#API-046) 経由で到達閾値・当月質問数・上限件数を取得する | 取得失敗時は当該メッセージのみ失敗として Queues の自動リトライへ委ね、バッチ全体は継続 | お知らせ・メール本文へ渡す表示値のみを参照し、判定は行わない |
| 3 | お知らせ生成(SYS-017 [PR-04](../../02_basic_design/02_backend/01_system/SYS-017.md#PR-04)) | 常時(冪等チェック通過時) | 当該宛先の受信箱へ上限アラートのお知らせを生成する | 生成失敗時は当該メッセージを失敗としてリトライ | テンプレートは [MSG-013](../../02_basic_design/06_messages/MSG-013.md#MSG-013)。書き込み先は [TBL-022](../../02_basic_design/02_backend/04_database/TBL-022.md#TBL-022) |
| 4 | メール送信(SYS-017 [PR-05](../../02_basic_design/02_backend/01_system/SYS-017.md#PR-05)) | お知らせ生成後 | [API-058](../../02_basic_design/02_backend/03_apis/API-058.md#API-058)(`EmailProvider.send`)で当該宛先へ上限アラートメールを送信する | 送信失敗時は当該メッセージを失敗としてリトライ。一時失敗は自動再送([NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036)) | 件名・本文は [MSG-013](../../02_basic_design/06_messages/MSG-013.md#MSG-013)。送信時は §6 の冪等キーを `idempotency_key` として渡す |
| 5 | 送信結果記録(SYS-017 [PR-06](../../02_basic_design/02_backend/01_system/SYS-017.md#PR-06)) | メール送信後 | 送信結果(メッセージ ID・配信状態)を記録する | 記録失敗時は当該メッセージを失敗としてリトライ | 記録先は [TBL-026](../../02_basic_design/02_backend/04_database/TBL-026.md#TBL-026)(`notification_type=NOTIF-SYSTEM_NOTICE`) |
| 6 | 冪等スキップ(SYS-017 [PR-07](../../02_basic_design/02_backend/01_system/SYS-017.md#PR-07)) | 当該宛先が当月内で既に送信成功済みのとき | 通知せず処理を終える | — | 受信者ごとの初回到達判定自体は [IPO-005](../04_ipo/IPO-005.md#IPO-005) の責務。本行は本バッチ側での二重送信の最終防止線 |

## 6. 排他・並列処理

宛先単位で並列消費しつつ、同一プロジェクト・同一請求月・同一閾値・同一宛先への二重通知を冪等キーで抑止する。

| 項目 | 内容 |
|----|----|
| 排他制御 | 同一宛先への同時二重送信を防ぐため、冪等キー単位で送信前に送信済みチェックを条件付き更新で直列化する。ロックの具体実装は詳細実装に委ねる |
| 並列処理 | 確定宛先リスト内の異なる宛先は Queues のバッチ消費で並列に送信する。並列度・バッチサイズの具体値は負荷想定([NFR-069](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-069) / [NFR-074](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-074))を参照する |
| 多重起動時の扱い | 同一メッセージが自動リトライ等で重複配信されても、冪等キーの既送信判定で当該宛先への再送を行わない |
| 冪等キー | `question-limit-alert:{projectId}:{yyyyMM}:{thresholdPercent}` + 受信者 ID([メッセージ設計 index](../../02_basic_design/06_messages/index.md) の重複配信抑止方針)。冪等性の保証は [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) を参照し、キーの物理構成は詳細実装に委ねる |

## 8. 異常終了時の扱い

異常検知時の対応・通知先・リトライ方針を示す。リトライ上限は本文に再掲せず NFR を参照する。

| 異常内容 | 対応 | 通知先 | リトライ | 備考 |
|----|----|----|----|----|
| 表示値参照失敗(API-046 一時障害) | 当該メッセージのみ失敗として再処理へ回す。バッチ全体は継続 | 運用 | あり([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040)。Queues 経由は自動リトライ、上限超過分は DLQ へ退避) | 表示値の参照のみで判定は行わないため再処理で二重判定は生じない |
| お知らせ生成失敗 | 当該メッセージを失敗として再処理へ回す | 運用 | あり(同上) | 冪等キーにより再処理時の重複お知らせを防止 |
| メール送信失敗(一時失敗) | 当該メッセージを失敗として再処理へ回す | 運用 | あり(同上。一時失敗は自動再送 [NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036)) | 冪等キーにより再送時の二重配信を防止 |
| 送信結果記録失敗 | 当該メッセージを失敗として再処理へ回す | 運用 | あり(同上) | メール自体は送信済みのため記録の再実行のみで二重送信は生じない(冪等キーで判定) |
| リトライ上限超過(DLQ 退避) | DLQ に退避し滞留を監視して手動再処理する | 運用 | 手動再処理 | 再処理待ち滞留は [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) に従いアラート |

## 9. 後続工程への引き継ぎ事項

テスト設計・運用設計へ渡す確認観点を箇条書きで示す。

- 境界値: 確定宛先リストが 1 件 / 複数件(オーナー + 複数有効メンバー)の場合の並列送信動作。
- 冪等性: 同一メッセージ重複配信・自動リトライ時に同一宛先へ二重送信・二重お知らせが生成されないこと(冪等キー `question-limit-alert:{projectId}:{yyyyMM}:{thresholdPercent}` + 受信者 ID の一意性確認)。
- 部分失敗: 確定宛先リストの一部宛先で送信失敗しても他宛先の処理が継続し、失敗分がリトライ / DLQ へ正しく回ること。
- DLQ 運用: DLQ 滞留の検知・アラート・手動再処理手順([NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041))。
- 責務境界の確認: 閾値到達検知([BAT-001](BAT-001.md#BAT-001))・受信者ごとの初回到達判定と宛先解決([IPO-005](../04_ipo/IPO-005.md#IPO-005))・お知らせ生成とメール配信(本バッチ)の三者間で、判定結果と配信実行が二重に行われないことを結合テストで確認する。
- 配信遅延: メール通知の許容最大遅延([NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036))内に配信されること。超過時の再送挙動([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040))を確認する。

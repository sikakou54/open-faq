# <span id="BAT-011"></span>BAT-011: 保持期間超過データの自動削除

> **本バッチは、データ保持期間([システム仕様書 §4](../../02_basic_design/07_system-spec.md#4-データ保持期間削除猶予))を超過した質問ログ・未解決質問・通知ログ・お知らせ受信箱を日次で抽出し、対象テーブルの削除方式(論理削除 / 物理削除)に従って削除する処理の実行機構の詳細設計を定義します。削除対象か否かの判定ロジックそのものは本バッチでは行いません([IPO-010](../04_ipo/IPO-010.md#IPO-010) が正本)。**

*種別 バッチ処理設計 ・ ステータス ドラフト*

## 1. 目的

本バッチは、日次の定期起動を契機に、データ保持期間([システム仕様書 §4](../../02_basic_design/07_system-spec.md#4-データ保持期間削除猶予))を超過した質問ログ・未解決質問・通知ログ・お知らせ受信箱を抽出し、対象テーブルが論理削除列を持つかに応じて論理削除(`valid=0`)または物理削除を実行する。詳細化元のシステム設計は [SYS-032](../../02_basic_design/02_backend/01_system/SYS-032.md#SYS-032)(保持期間超過データの自動削除)であり、その `batch` 種別・「日次の定期処理」トリガーを Cloudflare Cron Triggers の定時実行へ写像する。削除対象か否かの判定(起算点の決定・保持期間との比較)は [IPO-010](../04_ipo/IPO-010.md#IPO-010) §4 No.3・No.4(c) に委ね、本バッチでは重複定義しない。論理削除(`valid=0`)後の物理削除は [SYS-027](../../02_basic_design/02_backend/01_system/SYS-027.md#SYS-027) が猶予期間経過後に別途担う。

## 2. バッチ概要

本バッチは日次スケジュールにより自律的に起動する定期処理である。起動方式・多重起動可否・リトライ有無・想定規模を示す。

| 項目 | 値 |
|----|----|
| バッチ ID | BAT-011 |
| バッチ名 | 保持期間超過データの自動削除 |
| 起動方式 | Cron Triggers(定時。[SYS-032](../../02_basic_design/02_backend/01_system/SYS-032.md#SYS-032) の `batch` 種別・「日次の定期処理」を写像) |
| 起動タイミング | 日次(周期内の具体時刻は運用設計で確定。§9 課題) |
| 多重起動可否 | 不可(同時実行させない) |
| リトライ有無 | あり(当該対象・当該行のみ失敗として継続し、バッチ全体は次回日次起動で再評価する。Cron Triggers 起動のためメール再送 [NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040) は適用対象外) |
| 想定処理件数 | 質問ログ・未解決質問の想定規模([NFR-069](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-069))に準拠 |
| 想定処理時間 | 日次起動の周期内に完了する範囲(具体的な許容時間は運用設計で確定。§9 課題) |

## 4. 処理内容

本バッチの処理を実行順に示す。正常時・異常時の分岐を各行で表す。[SYS-032](../../02_basic_design/02_backend/01_system/SYS-032.md#SYS-032) の処理項目([PR-01](../../02_basic_design/02_backend/01_system/SYS-032.md#PR-01)〜[PR-04](../../02_basic_design/02_backend/01_system/SYS-032.md#PR-04))に対応する。

| No | 処理 | 条件 | 正常時 | 異常時 | 備考 |
|----|----|----|----|----|----|
| 1 | 超過データ抽出(SYS-032 [PR-01](../../02_basic_design/02_backend/01_system/SYS-032.md#PR-01)) | 日次起動時 | 質問ログ・未解決質問・通知ログ・お知らせ受信箱の各対象について、[IPO-010](../04_ipo/IPO-010.md#IPO-010) の判定ロジックを呼び出して削除対象行を確定する | 判定呼び出し失敗時はバッチ全体を異常終了し次回日次起動で再評価する | 対象が存在しない場合は削除を行わず正常完了する |
| 2 | 論理削除(SYS-032 [PR-02](../../02_basic_design/02_backend/01_system/SYS-032.md#PR-02)) | 削除対象に論理削除列を持つ対象(質問ログ・未解決質問)が含まれるとき | 対象行を `valid=0` へ更新する | 当該行のみ失敗として記録し処理を継続する。物理削除は [SYS-027](../../02_basic_design/02_backend/01_system/SYS-027.md#SYS-027) が猶予期間経過後に別途担う | 論理削除確定時刻の記録方法は [IPO-010](../04_ipo/IPO-010.md#IPO-010) §5 の課題候補を参照 |
| 3 | 物理削除(SYS-032 [PR-03](../../02_basic_design/02_backend/01_system/SYS-032.md#PR-03)) | 削除対象に論理削除列を持たない追記専用の対象(通知ログ・お知らせ受信箱)が含まれるとき | 対象行を物理削除する | 当該行のみ失敗として記録し処理を継続する | 不可逆(復元不可) |
| 4 | 削除結果記録(SYS-032 [PR-04](../../02_basic_design/02_backend/01_system/SYS-032.md#PR-04)) | 論理削除・物理削除を実行したとき | 実行結果(対象・件数・成功/失敗)を監査ログへ記録する | 記録自体の失敗は運用へ通知し次回日次起動で削除結果の整合を確認する | 出力先は §5 |

## 6. 排他・並列処理

日次起動 1 回の中で対象種別ごとに走査・削除を行い、多重起動と対象種別間の相互干渉を防ぐ。

| 項目 | 内容 |
|----|----|
| 排他制御 | 同一実行日の重複起動を抑止する(前回実行が完了する前に次回起動しない)。ロックの具体的実装は詳細実装に委ねる |
| 並列処理 | 対象種別(質問ログ / 未解決質問 / 通知ログ / お知らせ受信箱)ごとに独立して抽出・削除する。種別間の依存関係はないため並列実行を妨げない。並列度の具体値は負荷想定([NFR-069](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-069))を参照する |
| 多重起動時の扱い | 前回実行が完了していない場合は当該日の起動をスキップし、次回日次起動時に未処理分も含めて再評価する |
| 冪等キー | 対象行の識別子([TBL-025](../../02_basic_design/02_backend/04_database/TBL-025.md#TBL-025) / [TBL-017](../../02_basic_design/02_backend/04_database/TBL-017.md#TBL-017) / [TBL-026](../../02_basic_design/02_backend/04_database/TBL-026.md#TBL-026) / [TBL-022](../../02_basic_design/02_backend/04_database/TBL-022.md#TBL-022) の各 ID)を冪等キーとする。論理削除はすでに `valid=0` の行を再度対象化しない(判定条件で対象外化)ため冪等。物理削除は削除済み行が抽出条件に再度現れないため冪等。キーの物理構成は詳細実装に委ねる |

## 8. 異常終了時の扱い

異常検知時の対応・通知先・リトライ方針を示す。

| 異常内容 | 対応 | 通知先 | リトライ | 備考 |
|----|----|----|----|----|
| 削除対象判定([IPO-010](../04_ipo/IPO-010.md#IPO-010))の呼び出し失敗 | バッチ全体を異常終了し、削除を行わない | 運用 | あり(次回日次起動で再評価) | 判定未了の中途状態を残さない |
| 対象行単位の論理削除・物理削除失敗(競合・一時障害) | 当該行のみ失敗として記録し、他の対象行の処理を継続する | 運用 | あり(次回日次起動で再評価) | バッチ全体は継続。失敗行は次回起動時に再度削除対象として抽出される |
| 削除結果記録([TBL-027](../../02_basic_design/02_backend/04_database/TBL-027.md#TBL-027))への記録失敗 | 削除自体は成立させたまま記録失敗を運用へ通知する | 運用 | あり(次回日次起動で整合確認) | 削除実行と記録の不整合が生じないよう次回起動時に照合する |

## 9. 後続工程への引き継ぎ事項

テスト設計・運用設計へ渡す確認観点、および基本設計で未確定の点を箇条書きで示す。

- 境界値: 0 件 / 保持期間到達直前 / 保持期間到達当日での抽出・削除の動作。
- 冪等性: 同一日の重複起動抑止時に対象行の重複削除・重複記録が発生しないこと。
- 部分失敗: 一部対象行の削除失敗時に他対象種別・他対象行の処理が継続し、失敗分が次回日次起動で再評価されること。
- 削除方式の境界: 論理削除(質問ログ・未解決質問)と物理削除(通知ログ・お知らせ受信箱)を同一バッチ内で対象種別ごとに正しく振り分けること。
- 監査記録: 削除実行結果が [TBL-027](../../02_basic_design/02_backend/04_database/TBL-027.md#TBL-027) へ確実に記録され、削除対象が存在しない日は記録が生成されないこと。
- **課題候補**: 日次起動の具体的な時刻・周期内の許容処理時間が基本設計・システム仕様書に明示されておらず、運用設計での確定が必要(§2 起動タイミング・想定処理時間)。
- **課題候補**: [IPO-010](../04_ipo/IPO-010.md#IPO-010) §5 で指摘済みの、論理削除確定時刻(`valid=0` へ更新した時刻)を記録する専用カラムの要否(DBP 側での物理カラム追加要否)は本バッチの論理削除実行にも影響するため、あわせて確認が必要。

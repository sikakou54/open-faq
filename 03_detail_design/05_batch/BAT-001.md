# <span id="BAT-001"></span>BAT-001: 利用量リアルタイム集計

> **本バッチは、ウィジェットへの質問到達を契機に、同期加算済みの当月利用量(質問数)を参照して上限・アラート閾値の到達を判定し、各契機を担当処理へ引き渡し、管理画面の利用量サマリへ反映する処理の詳細設計を定義します。質問数の加算は本バッチでは行いません(質問送信 API の同期トランザクションが正本)。**

*種別 バッチ処理設計 ・ ステータス ドラフト*

## 1. 目的

本バッチは、ウィジェット利用者の質問がプロジェクトへ到達するたびに、質問送信 API が同一トランザクションで加算済みの当月質問数([TBL-020](../../02_basic_design/02_backend/04_database/TBL-020.md#TBL-020) `question_count` / `unbillable_question_count`。加算の正本は [API-038](../../02_basic_design/02_backend/03_apis/API-038.md#API-038))を参照し、月次上限がオンのプロジェクトでは利用率を算出してアラート閾値到達・上限到達・最終ガード到達の各契機を担当処理へ引き渡し、集計結果を管理画面向けの利用量サマリへ反映する。本バッチは質問数の加算を行わない。詳細化元のシステム設計は [SYS-016](../../02_basic_design/02_backend/01_system/SYS-016.md#SYS-016)(利用量集計・サマリ反映)であり、その `async` 契機・処理項目([PR-01](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-01)〜[PR-06](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-06))を Cloudflare Queues の消費処理へ写像する(利用率算定の分母は課金対象件数=`question_count - unbillable_question_count`・[RULE-013](../../01_requirements/01_business_requirement/08_rule.md#RULE-013))。

## 2. バッチ概要

本バッチはウィジェット質問送信 API が質問到達時にイベントを Cloudflare Queues へ投入したことを契機に起動する非同期消費処理である。起動方式・多重起動可否・リトライ有無・想定規模を示す。

| 項目 | 値 |
|----|----|
| バッチ ID | BAT-001 |
| バッチ名 | 利用量リアルタイム集計 |
| 起動方式 | Queues(イベント契機。[API-038](../../02_basic_design/02_backend/03_apis/API-038.md#API-038) が質問到達時にメッセージを投入し、その消費で起動) |
| 起動タイミング | ウィジェットへの質問がプロジェクトに到達するたび(1 質問 = 1 メッセージ) |
| 多重起動可否 | 可(メッセージ単位で並列消費。質問数カウンタは同期側で更新済みのため本バッチは加算せず、閾値契機引き渡しの当月一度性を後述の冪等で担保する) |
| リトライ有無 | あり(Queues の自動リトライ。上限超過分は DLQ へ退避。[NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036) / [NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040) を参照し本文へ回数を再掲しない) |
| 想定処理件数 | 1 プロジェクトあたり質問ログ件数の想定上限([NFR-069](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-069))に準拠 |
| 想定処理時間 | サマリ反映は利用量サマリ反映遅延([システム仕様書 §2](../../02_basic_design/07_system-spec.md#2-課金利用量上限))の許容内 |

本バッチは質問到達イベントのメッセージを入力とし、集計・判定に必要な設定を参照する。物理カラム名の羅列は行わず、テーブルはリンクで辿る。

| 入力元 | 内容 | 取得条件 | 備考 |
|----|----|----|----|
| Queues メッセージ(質問到達イベント) | 対象プロジェクト・請求年月・課金対象区分・冪等キーとなる質問ログ識別子 | [API-038](../../02_basic_design/02_backend/03_apis/API-038.md#API-038) が質問ログを記録・質問数を同期加算した時に 1 質問 = 1 件投入したメッセージを消費する | 課金対象区分は課金対象 / 対象外の別。推論失敗(`result_type=error` / `result_reason_code=ai_unavailable`)は課金対象外として区別され、同期側で `unbillable_question_count` に計上済み([TBL-025](../../02_basic_design/02_backend/04_database/TBL-025.md#TBL-025)) |
| プロジェクト別利用設定 | 月次上限件数・無料枠・アラート閾値の設定 | メッセージの `project_id` で当該プロジェクトの有効設定を取得する | 月次上限オフ時は判定を行わずサマリ反映のみ([TBL-009](../../02_basic_design/02_backend/04_database/TBL-009.md#TBL-009)) |
| 利用量計測 | 当該プロジェクト × 当該請求月の同期加算済み現在値(`question_count` / `unbillable_question_count`) | メッセージの `project_id` と請求年月で一意行を取得する(同期側で加算済みのため本バッチは参照のみ・行を作らない) | 一意キーは `(project_id, billing_ym)`([TBL-020](../../02_basic_design/02_backend/04_database/TBL-020.md#TBL-020)) |

## 4. 処理内容

本バッチの処理を実行順に示す。正常時・異常時の分岐を各行で表す。SYS-016 の処理項目([PR-01](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-01)〜[PR-06](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-06))に対応する。

| No | 処理 | 条件 | 正常時 | 異常時 | 備考 |
|----|----|----|----|----|----|
| 1 | 冪等チェック | 常時 | メッセージの冪等キー(質問ログ識別子)が本バッチで未処理なら後続へ進む | 処理済みなら以降をスキップし正常完了(閾値契機の二重引き渡し防止) | 冪等キーは §6 参照。質問数の加算は同期側で冪等済み |
| 2 | 同期加算済み利用量の参照(SYS-016 [PR-01](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-01)) | 冪等チェック通過時 | 当該プロジェクト × 請求月の同期加算済み値(`question_count` / `unbillable_question_count`)を参照する。本バッチは質問数を加算しない | 値の取得失敗時は当該メッセージのみ失敗として Queues の自動リトライへ委ね、バッチ全体は継続 | 加算の正本は同期側([API-038](../../02_basic_design/02_backend/03_apis/API-038.md#API-038))。課金対象件数=`question_count - unbillable_question_count`([TBL-020](../../02_basic_design/02_backend/04_database/TBL-020.md#TBL-020) ・ [RULE-013](../../01_requirements/01_business_requirement/08_rule.md#RULE-013)) |
| 3 | 利用率算出(SYS-016 [PR-02](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-02)) | 月次上限がオンのとき | 課金対象件数(`question_count - unbillable_question_count`)を分子、月次上限件数を分母に当月利用率を算出する | 上限設定の取得失敗時は当該メッセージを失敗としてリトライ | 月次上限オフ時は本処理をスキップし No.7 へ。分母解釈は [RULE-013](../../01_requirements/01_business_requirement/08_rule.md#RULE-013) |
| 4 | アラート契機引き渡し(SYS-016 [PR-03](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-03)) | 月次上限オンでアラート閾値([RULE-014](../../01_requirements/01_business_requirement/08_rule.md#RULE-014))に当月はじめて到達したとき | 上限アラート通知の契機を [SYS-017](../../02_basic_design/02_backend/01_system/SYS-017.md#SYS-017) へ引き渡す | 引き渡し投入失敗時は当該メッセージを失敗としてリトライ | 初回到達の判定は §6 の冪等方針で当月内一度だけに保つ |
| 5 | 受付停止契機引き渡し(SYS-016 [PR-04](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-04)) | 月次上限オンで受付停止しきい値(100%・[システム仕様書 §2](../../02_basic_design/07_system-spec.md#2-課金利用量上限) / [RULE-013](../../01_requirements/01_business_requirement/08_rule.md#RULE-013))に到達したとき | ウィジェット新規質問受付停止の契機を [SYS-018](../../02_basic_design/02_backend/01_system/SYS-018.md#SYS-018) へ引き渡す | 引き渡し失敗時は当該メッセージを失敗としてリトライ | 実際の受付可否判定は SYS-018 が質問送信時に同期で行う |
| 6 | 最終ガード契機引き渡し(SYS-016 [PR-05](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-05)) | 月次上限オンで最終ガード追加通知しきい値(125%・[システム仕様書 §2](../../02_basic_design/07_system-spec.md#2-課金利用量上限) / [RULE-013](../../01_requirements/01_business_requirement/08_rule.md#RULE-013))に到達したとき | 追加アラート通知の契機を [SYS-017](../../02_basic_design/02_backend/01_system/SYS-017.md#SYS-017) へ引き渡す | 引き渡し失敗時は当該メッセージを失敗としてリトライ | 集計遅延・誤差対策の最終ガード |

## 6. 排他・並列処理

メッセージ単位で並列消費しつつ、質問数の加算は同期側([API-038](../../02_basic_design/02_backend/03_apis/API-038.md#API-038))で確定済みのため本バッチは加算しない。閾値到達契機の引き渡しは当月一度だけに保ち、二重通知・二重受付停止契機を防ぐ。

| 項目 | 内容 |
|----|----|
| 排他制御 | 質問数カウンタ加算は本バッチの対象外(同期側の Tx が正本)。本バッチは閾値契機の当月一度性を担保するため、当該プロジェクト × 請求月 × 閾値の引き渡し済みマーカーを条件付き更新で直列化する。ロックの具体実装は詳細実装に委ねる |
| 並列処理 | 異なるプロジェクト / 請求月のメッセージは Queues のバッチ消費で並列に処理する。並列度・バッチサイズの具体値は負荷想定([NFR-069](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-069) / [NFR-074](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-074))を参照する |
| 多重起動時の扱い | 同一メッセージが自動リトライ等で重複配信されても、冪等キーの既処理判定で判定・契機引き渡しを重複させない。処理中の同一キーは後着をスキップまたは再試行に回す |
| 冪等キー | 質問到達イベントに紐づく質問ログ識別子([TBL-025](../../02_basic_design/02_backend/04_database/TBL-025.md#TBL-025) の ID)を冪等キーとし、判定・サマリ反映の再処理を抑止する。閾値到達の契機引き渡しは「当該プロジェクト × 請求月 × 閾値」で当月一度だけに保つ。冪等性の保証は [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) を参照し、キーの物理構成は詳細実装に委ねる |

## 9. 後続工程への引き継ぎ事項

テスト設計・運用設計へ渡す確認観点、および基本設計で未確定の点を箇条書きで示す。

- 境界値: 0 件 / 上限直前 / 100% 到達 / 125% 到達での加算・判定・契機引き渡しの動作。
- 冪等性: 同一メッセージ重複配信・自動リトライ時に二重計上せず、閾値到達契機が当月一度だけ引き渡されること。
- 排他: 同一プロジェクト × 同一請求月への高頻度同時到達時にカウンタ更新の取りこぼしがないこと。
- 部分失敗: 一部メッセージ失敗時に他メッセージの処理が継続し、失敗分がリトライ / DLQ へ正しく回ること。
- DLQ 運用: DLQ 滞留の検知・アラート・手動再処理手順([NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041))。
- 反映遅延: 利用量サマリの反映が反映遅延許容([システム仕様書 §2](../../02_basic_design/07_system-spec.md#2-課金利用量上限))内に収まること。
- 役割分担(確定): 質問数計上は質問送信 API([API-038](../../02_basic_design/02_backend/03_apis/API-038.md#API-038))が同期トランザクション内で [TBL-020](../../02_basic_design/02_backend/04_database/TBL-020.md#TBL-020) を加算する経路を正本とする。本バッチ(SYS-016)は加算せず、同期加算済み値を参照して閾値到達判定・契機引き渡し・サマリ反映のみを行う。テストでは同期加算と本バッチ判定の整合(参照タイミングによる判定ずれ)を確認する。
- 課金対象外の保持(確定): 推論失敗(`ai_unavailable` / 課金対象外)は同期側が [TBL-020](../../02_basic_design/02_backend/04_database/TBL-020.md#TBL-020) の `unbillable_question_count` に計上する。課金対象件数(無料枠消費・受付停止・請求の分母)は `question_count - unbillable_question_count`([RULE-013](../../01_requirements/01_business_requirement/08_rule.md#RULE-013))で算定する。テストでは推論失敗が分母に含まれず無料枠を消費しないことを確認する。

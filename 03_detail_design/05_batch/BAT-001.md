# <span id="BAT-001"></span>BAT-001: 利用量リアルタイム集計

> **本バッチは、ウィジェットへの質問到達を契機に当月の利用量を非同期で集計し、上限・アラート閾値の到達を判定して各契機を担当処理へ引き渡し、管理画面の利用量サマリへ反映する処理の詳細設計を定義します。**

*種別 バッチ処理設計 ・ ステータス ドラフト*

## 1. 目的

本バッチは、ウィジェット利用者の質問がプロジェクトへ到達するたびに当月質問数をリアルタイムに積み上げ、月次上限がオンのプロジェクトでは利用率を算出してアラート閾値到達・上限到達・最終ガード到達の各契機を担当処理へ引き渡し、集計結果を管理画面向けの利用量サマリへ反映する。詳細化元のシステム設計は [SYS-016](../../02_basic_design/02_backend/01_system/SYS-016.md#SYS-016)(利用量リアルタイム集計・サマリ反映)であり、その `async` 契機・処理項目([PR-01](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-01)〜[PR-06](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-06))を Cloudflare Queues の消費処理へ写像する。

## 2. バッチ概要

本バッチはウィジェット質問送信 API が質問到達時にイベントを Cloudflare Queues へ投入したことを契機に起動する非同期消費処理である。起動方式・多重起動可否・リトライ有無・想定規模を示す。

| 項目 | 値 |
|----|----|
| バッチ ID | BAT-001 |
| バッチ名 | 利用量リアルタイム集計 |
| 起動方式 | Queues(イベント契機。[API-038](../../02_basic_design/02_backend/03_apis/API-038.md#API-038) が質問到達時にメッセージを投入し、その消費で起動) |
| 起動タイミング | ウィジェットへの質問がプロジェクトに到達するたび(1 質問 = 1 メッセージ) |
| 多重起動可否 | 可(メッセージ単位で並列消費。同一 `project_id` × 同一請求月のカウンタ更新のみ後述の排他で直列化) |
| リトライ有無 | あり(Queues の自動リトライ。上限超過分は DLQ へ退避。[NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036) / [NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040) を参照し本文へ回数を再掲しない) |
| 想定処理件数 | 1 プロジェクトあたり質問ログ件数の想定上限([NFR-069](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-069))に準拠 |
| 想定処理時間 | サマリ反映は利用量サマリ反映遅延([システム仕様書 §2](../../02_basic_design/07_system-spec.md#2-課金利用量上限))の許容内 |

## 3. 入力

本バッチは質問到達イベントのメッセージを入力とし、集計・判定に必要な設定を参照する。物理カラム名の羅列は行わず、テーブルはリンクで辿る。

| 入力元 | 内容 | 取得条件 | 備考 |
|----|----|----|----|
| Queues メッセージ(質問到達イベント) | 対象プロジェクト・請求年月・課金対象区分・冪等キーとなる質問ログ識別子 | [API-038](../../02_basic_design/02_backend/03_apis/API-038.md#API-038) が質問ログを記録した時に 1 質問 = 1 件投入したメッセージを消費する | 課金対象区分は課金対象 / 対象外の別。推論失敗(`result_type=error` / `result_reason_code=ai_unavailable`)は課金対象外として区別する([TBL-025](../../02_basic_design/02_backend/04_database/TBL-025.md#TBL-025)) |
| プロジェクト別利用設定 | 月次上限件数・無料枠・アラート閾値の設定 | メッセージの `project_id` で当該プロジェクトの有効設定を取得する | 月次上限オフ時は判定を行わず計測のみ([TBL-009](../../02_basic_design/02_backend/04_database/TBL-009.md#TBL-009)) |
| 利用量計測 | 当該プロジェクト × 当該請求月の現在値 | メッセージの `project_id` と請求年月で一意行を取得する(存在しなければ初期化して新規行を作る) | 一意キーは `(project_id, billing_ym)`([TBL-020](../../02_basic_design/02_backend/04_database/TBL-020.md#TBL-020)) |

## 4. 処理内容

本バッチの処理を実行順に示す。正常時・異常時の分岐を各行で表す。SYS-016 の処理項目([PR-01](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-01)〜[PR-06](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-06))に対応する。

| No | 処理 | 条件 | 正常時 | 異常時 | 備考 |
|----|----|----|----|----|----|
| 1 | 冪等チェック | 常時 | メッセージの冪等キー(質問ログ識別子)が未処理なら後続へ進む | 処理済みなら以降をスキップし正常完了(二重計上防止) | 冪等キーは §6 参照 |
| 2 | 利用量加算(SYS-016 [PR-01](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-01)) | 冪等チェック通過時 | 当該プロジェクト × 請求月の質問数を加算する。課金対象外(推論失敗)は件数に数えつつ課金対象としては区別して保持する | 当該メッセージのみ失敗として Queues の自動リトライへ委ね、バッチ全体は継続 | 課金対象 / 対象外の区別は [TBL-020](../../02_basic_design/02_backend/04_database/TBL-020.md#TBL-020)。区別の格納方式は §9 で後続へ引き渡す |
| 3 | 利用率算出(SYS-016 [PR-02](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-02)) | 月次上限がオンのとき | 月次上限件数を分母に当月利用率を算出する | 上限設定の取得失敗時は当該メッセージを失敗としてリトライ | 月次上限オフ時は本処理をスキップし No.7 へ |
| 4 | アラート契機引き渡し(SYS-016 [PR-03](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-03)) | 月次上限オンでアラート閾値([RULE-014](../../01_requirements/01_business_requirement/08_rule.md#RULE-014))に当月はじめて到達したとき | 上限アラート通知の契機を [SYS-017](../../02_basic_design/02_backend/01_system/SYS-017.md#SYS-017) へ引き渡す | 引き渡し投入失敗時は当該メッセージを失敗としてリトライ | 初回到達の判定は §6 の冪等方針で当月内一度だけに保つ |
| 5 | 受付停止契機引き渡し(SYS-016 [PR-04](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-04)) | 月次上限オンで受付停止しきい値(100%・[システム仕様書 §2](../../02_basic_design/07_system-spec.md#2-課金利用量上限) / [RULE-013](../../01_requirements/01_business_requirement/08_rule.md#RULE-013))に到達したとき | ウィジェット新規質問受付停止の契機を [SYS-018](../../02_basic_design/02_backend/01_system/SYS-018.md#SYS-018) へ引き渡す | 引き渡し失敗時は当該メッセージを失敗としてリトライ | 実際の受付可否判定は SYS-018 が質問送信時に同期で行う |
| 6 | 最終ガード契機引き渡し(SYS-016 [PR-05](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-05)) | 月次上限オンで最終ガード追加通知しきい値(125%・[システム仕様書 §2](../../02_basic_design/07_system-spec.md#2-課金利用量上限) / [RULE-013](../../01_requirements/01_business_requirement/08_rule.md#RULE-013))に到達したとき | 追加アラート通知の契機を [SYS-017](../../02_basic_design/02_backend/01_system/SYS-017.md#SYS-017) へ引き渡す | 引き渡し失敗時は当該メッセージを失敗としてリトライ | 集計遅延・誤差対策の最終ガード |
| 7 | サマリ反映(SYS-016 [PR-06](../../02_basic_design/02_backend/01_system/SYS-016.md#PR-06)) | 常時(上限オン / オフ共通) | 集計結果を管理画面向け利用量サマリへ反映遅延([システム仕様書 §2](../../02_basic_design/07_system-spec.md#2-課金利用量上限))内に反映する | 反映失敗時は当該メッセージを失敗としてリトライ | 参照は [API-040](../../02_basic_design/02_backend/03_apis/API-040.md#API-040) |

## 6. 排他・並列処理

メッセージ単位で並列消費しつつ、同一プロジェクト × 同一請求月のカウンタ更新は競合を避けて直列化し、二重計上を防ぐ。

| 項目 | 内容 |
|----|----|
| 排他制御 | 同一 `project_id` × 同一 `billing_ym` の利用量計測行への加算は D1 のトランザクション内で条件付き更新(現在値を条件に含む更新)により直列化し、同時更新の取りこぼしを防ぐ。ロックの具体実装は詳細実装に委ねる |
| 並列処理 | 異なるプロジェクト / 請求月のメッセージは Queues のバッチ消費で並列に処理する。並列度・バッチサイズの具体値は負荷想定([NFR-069](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-069) / [NFR-074](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-074))を参照する |
| 多重起動時の扱い | 同一メッセージが自動リトライ等で重複配信されても、冪等キーの既処理判定で二重計上しない。処理中の同一キーは後着をスキップまたは再試行に回す |
| 冪等キー | 質問到達イベントに紐づく質問ログ識別子([TBL-025](../../02_basic_design/02_backend/04_database/TBL-025.md#TBL-025) の ID)を冪等キーとし、計上済みキーの再処理を抑止する。閾値到達の契機引き渡しは「当該プロジェクト × 請求月 × 閾値」で当月一度だけに保つ。冪等性の保証は [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) を参照し、キーの物理構成は詳細実装に委ねる |

## 8. 異常終了時の扱い

異常検知時の対応・通知先・リトライ方針を示す。リトライ上限は本文に再掲せず NFR を参照する。

| 異常内容 | 対応 | 通知先 | リトライ | 備考 |
|----|----|----|----|----|
| 利用量計測の更新失敗(競合・一時障害) | 当該メッセージのみ失敗として再処理へ回す。バッチ全体は継続 | 運用 | あり([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040)。Queues 経由は自動リトライ、上限超過分は DLQ へ退避) | 冪等キーにより再処理での二重計上を防止 |
| 上限・閾値設定の取得失敗 | 当該メッセージを失敗として再処理へ回す | 運用 | あり(同上) | 計測のみ完了・判定未了の中途状態を残さない |
| 契機引き渡し(SYS-017 / SYS-018)の投入失敗 | 当該メッセージを失敗として再処理へ回す | 運用 | あり(同上) | 通知本体の再送は SYS-017 が担う |
| リトライ上限超過(DLQ 退避) | DLQ に退避し滞留を監視して手動再処理する | 運用 | 手動再処理 | 再処理待ち滞留は [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) に従いアラート |

## 9. 後続工程への引き継ぎ事項

テスト設計・運用設計へ渡す確認観点、および基本設計で未確定の点を箇条書きで示す。

- 境界値: 0 件 / 上限直前 / 100% 到達 / 125% 到達での加算・判定・契機引き渡しの動作。
- 冪等性: 同一メッセージ重複配信・自動リトライ時に二重計上せず、閾値到達契機が当月一度だけ引き渡されること。
- 排他: 同一プロジェクト × 同一請求月への高頻度同時到達時にカウンタ更新の取りこぼしがないこと。
- 部分失敗: 一部メッセージ失敗時に他メッセージの処理が継続し、失敗分がリトライ / DLQ へ正しく回ること。
- DLQ 運用: DLQ 滞留の検知・アラート・手動再処理手順([NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041))。
- 反映遅延: 利用量サマリの反映が反映遅延許容([システム仕様書 §2](../../02_basic_design/07_system-spec.md#2-課金利用量上限))内に収まること。
- 未確定: 質問到達時の質問数計上を同期(API-038 の利用テーブル更新)で行うか本バッチの非同期集計で行うか、両者の役割分担・整合方針(集計の正本の所在)は基本設計で確定していないため、テスト設計前に確定を要する。
- 未確定: 課金対象外(推論失敗)を利用量計測でどの粒度で区別保持するか(件数区分の格納方式)は基本設計に定義がないため、DB 物理設計([TBL-020](../../02_basic_design/02_backend/04_database/TBL-020.md#TBL-020))と併せて確定を要する。

# <span id="BAT-005"></span>BAT-005: 月次請求確定バッチ

> **本バッチは、月初の定期起動を契機に対象月(前月分)の利用量をオーナー単位(課金アカウント単位)で確定し、請求確定通知を生成するまでの実行機構(起動・冪等スキップ・排他・監視・異常終了時の扱い)の詳細設計を定義します。無料枠控除・超過単価計算・請求額算定そのものは本バッチでは行いません(算定ロジックの正本は [IPO-002](../04_ipo/IPO-002.md#IPO-002))。**

*種別 バッチ処理設計 ・ ステータス ドラフト*

## 1. 目的

本バッチは、月初の定期起動により前月分を対象として、プロジェクト単位で計測済みの利用量をオーナー単位(課金アカウント単位)へ集約して請求を確定し、確定通知を生成する。詳細化元のシステム設計は [SYS-019](../../02_basic_design/02_backend/01_system/SYS-019.md#SYS-019)(月次請求確定)であり、その `batch` 種別・トリガー(月初定期起動)を Cloudflare Cron Triggers の定時実行へ写像する。請求額算定ロジック(月次集計・無料枠控除・超過単価計算・オーナー集約)は [IPO-002](../04_ipo/IPO-002.md#IPO-002) に委ね、本バッチは起動制御・オーナー単位の処理順序・冪等スキップ・排他・監視・異常終了時の扱いを定義する。

## 2. バッチ概要

本バッチは月初に前月分を対象として定期起動する Cron Triggers 契機のバッチ処理である。起動方式・多重起動可否・リトライ有無・想定規模を示す。

| 項目 | 値 |
|----|----|
| バッチ ID | BAT-005 |
| バッチ名 | 月次請求確定バッチ |
| 起動方式 | Cron Triggers(定時。[SYS-019](../../02_basic_design/02_backend/01_system/SYS-019.md#SYS-019) の `batch` 種別・「月初に前月分を対象として定期起動」を写像) |
| 起動タイミング | 月初(JST 暦月境界。[システム仕様書 §2](../../02_basic_design/07_system-spec.md#2-課金利用量上限) の質問数無料枠リセット境界と同一の月次境界に基づく)。前月分を対象月として起動する |
| 多重起動可否 | 不可(同時実行させない) |
| リトライ有無 | あり(バッチ全体としては単発実行だが、オーナー単位の処理は失敗時にオーナー単位で再試行対象とする。再送・遅延の方針は [NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040) を参照し本文へ回数を再掲しない) |
| 想定処理件数 | 請求対象オーナー(課金アカウント)数分。1 オーナーあたりの集計対象プロジェクト数はプロジェクト数・メンバー数の目安([システム仕様書 §5](../../02_basic_design/07_system-spec.md#5-レート制限キャパシティ目安))に準拠 |
| 想定処理時間 | 全対象オーナーの確定・通知が当日の月初起動ウィンドウ内に完了する範囲(具体的な許容時間は運用設計に委ねる) |

## 3. 入力

本バッチは月初起動時に対象月(前月)を確定し、請求確定ロジックへ渡す請求対象オーナーの一覧を取得する。

| 入力元 | 内容 | 取得条件 | 備考 |
|----|----|----|----|
| Cron Triggers 起動イベント | 起動時刻(対象月=前月の算定根拠) | 月初の定時起動 | 対象月の判定は起動時刻から前月を導出する(JST 暦月境界) |
| 課金アカウント | 請求対象のオーナー一覧 | 課金アカウント状態が課金対象([課金・請求設計書 §6](../../02_basic_design/05_billing-design.md#6-利用量集計方針)。サスペンション中は集計対象外) | [TBL-002](../../02_basic_design/02_backend/04_database/TBL-002.md#TBL-002) |
| 利用量計測 / プロジェクト別利用設定 | オーナーが作成した各プロジェクトの対象月計測値・無料枠・超過単価 | 対象月・当該オーナーが作成したプロジェクトで取得。算定ロジックへの受け渡しは [IPO-002](../04_ipo/IPO-002.md#IPO-002) が担う | [TBL-020](../../02_basic_design/02_backend/04_database/TBL-020.md#TBL-020) ・ [TBL-009](../../02_basic_design/02_backend/04_database/TBL-009.md#TBL-009) |

## 4. 処理内容

本バッチの処理を実行順に示す。請求額算定そのものの手順([SYS-019](../../02_basic_design/02_backend/01_system/SYS-019.md#SYS-019) `PR-01`〜`PR-03`)は [IPO-002](../04_ipo/IPO-002.md#IPO-002) に委ね、本表では起動制御・呼び出し・オーナー単位の継続処理のみを示す。

| No | 処理 | 条件 | 正常時 | 異常時 | 備考 |
|----|----|----|----|----|----|
| 1 | 起動・対象月確定 | 月初の Cron Triggers 起動時 | 起動時刻から対象月(前月)を確定し、多重起動チェックへ進む | 起動処理自体の異常はバッチ全体を異常終了とする | 対象月の算定は JST 暦月境界に基づく |
| 2 | 多重起動チェック | 対象月確定後 | 同一対象月の実行中インスタンスが無ければ後続へ進む | 実行中インスタンスがあれば本起動をスキップして終了(§6) | 排他制御は §6 |
| 3 | 請求対象オーナー一覧の取得 | 多重起動チェック通過時 | 課金対象の課金アカウント一覧を取得しオーナー単位のループへ渡す | 一覧取得失敗はバッチ全体を異常終了とする | [TBL-002](../../02_basic_design/02_backend/04_database/TBL-002.md#TBL-002) |
| 4 | オーナー単位の冪等判定 | オーナーごとに実行 | 当該オーナー・当該対象月が未確定であれば請求確定ロジック呼び出しへ進む | 確定済み / 対象なしのオーナーはスキップし次のオーナーへ継続([SYS-019](../../02_basic_design/02_backend/01_system/SYS-019.md#SYS-019) `PR-07`) | 冪等キーは §6 |
| 5 | 請求確定ロジック呼び出し | 冪等判定で未確定のオーナー | [IPO-002](../04_ipo/IPO-002.md#IPO-002) を呼び出し、月次集計・無料枠控除・超過単価計算・オーナー集約・請求確定・記録・通知送信までを一括で委ねる | 当該オーナーの呼び出し失敗は当該オーナーのみ失敗として記録し、後続オーナーの処理は継続 | 算定ロジック内部の分岐・入出力は [IPO-002](../04_ipo/IPO-002.md#IPO-002) を参照 |
| 6 | 全オーナー処理完了判定 | 全オーナーの確定 / スキップ / 失敗のいずれかが完了したとき | 処理結果サマリ(確定件数・スキップ件数・失敗件数)を記録し正常終了する | 失敗オーナーが存在する場合も正常終了とし、失敗分は §8 の異常終了時の扱いに従い個別に扱う | バッチ全体としては部分失敗があっても継続完了とする |

|----|----|----|----|

## 6. 排他・並列処理

同一対象月の実行を単一インスタンスに限定し、オーナー単位の冪等判定で確定済みオーナーの再確定を防ぐ。

| 項目 | 内容 |
|----|----|
| 排他制御 | 同一対象月のバッチ実行を同時に 1 インスタンスへ限定する(多重起動不可)。ロックの具体実装は詳細実装に委ねる |
| 並列処理 | バッチ実行インスタンスは単一とし、インスタンス内ではオーナー単位に順次処理する。オーナー間の並列度・分割方針は詳細実装に委ねる |
| 多重起動時の扱い | 同一対象月の実行中インスタンスが既にある場合、後着の起動はスキップして終了する |
| 冪等キー | 「オーナー(課金アカウント)× 対象月」を冪等キーとし、確定済みオーナーは請求確定ロジックを呼び出さずスキップする([SYS-019](../../02_basic_design/02_backend/01_system/SYS-019.md#SYS-019) `PR-07`)。冪等性の保証は [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) を参照し、キーの物理構成は詳細実装に委ねる |

| ログ種別 | 出力内容 | 出力タイミング | 備考 |
|----|----|----|----|
| 起動ログ | 対象月・起動時刻・多重起動チェック結果 | バッチ起動時 | — |
| オーナー処理ログ | オーナー識別子・確定 / スキップ / 失敗の別 | オーナー単位の処理完了時 | 請求金額の内訳は含めない |
| 完了ログ | 対象月・確定件数・スキップ件数・失敗件数・処理時間 | バッチ完了時 | 想定処理時間超過の監視に用いる |
| 異常ログ | オーナー単位の失敗理由・バッチ全体異常終了の理由 | 異常検知時 | 滞留監視は §8 |

## 8. 異常終了時の扱い

異常検知時の対応・通知先・リトライ方針を示す。リトライ上限は本文に再掲せず NFR を参照する。

| 異常内容 | 対応 | 通知先 | リトライ | 備考 |
|----|----|----|----|----|
| 請求対象オーナー一覧の取得失敗(No.3) | バッチ全体を異常終了とし当該対象月は未確定のまま終える | 運用 | あり([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040) を参照し次回起動または手動再起動で再試行) | 一覧取得前の異常のため部分確定は生じない |
| オーナー単位の請求確定ロジック呼び出し失敗(No.5) | 当該オーナーのみ失敗として記録し、他オーナーの処理を継続([IPO-002](../04_ipo/IPO-002.md#IPO-002) を参照) | 運用 | あり(次回起動時に冪等判定で当該オーナーのみ再試行対象となる) | 冪等キー(オーナー×対象月)により再確定・二重請求を防止 |
| 確定通知送信の失敗 | 請求確定自体はロールバックせず、通知失敗のみを記録([IPO-002](../04_ipo/IPO-002.md#IPO-002) の通知失敗時挙動に従う) | 運用 | あり([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040)。通知の再送方針は [API-058](../../02_basic_design/02_backend/03_apis/API-058.md#API-058) 側に従う) | 請求確定と通知送信は別トランザクション境界(詳細は [IPO-002](../04_ipo/IPO-002.md#IPO-002)) |
| 想定処理時間超過・滞留 | 未処理オーナーの滞留を監視しアラートする | 運用 | 手動再処理 | 滞留アラートは [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) に従う |

## 9. 後続工程への引き継ぎ事項

テスト設計・運用設計へ渡す確認観点、および基本設計で未確定の点を箇条書きで示す。

- 境界値: 請求対象オーナー 0 件 / 全オーナー確定済み(全件スキップ)/ 想定上限件数での起動・完了時間。
- 冪等性: 同一対象月に対する再起動(手動再実行含む)で、確定済みオーナーが再確定・二重請求されないこと。
- 排他: 同一対象月の多重起動が発生しないこと(Cron Triggers の遅延・重複発火時を含む)。
- 部分失敗: 一部オーナーの請求確定ロジック呼び出しが失敗しても他オーナーの確定・通知が継続され、失敗分が次回起動で冪等に再試行されること。
- 算定ロジックとの境界: 月次集計・無料枠控除・超過単価計算・オーナー集約・トランザクション境界は [IPO-002](../04_ipo/IPO-002.md#IPO-002) を正本とし、本バッチは呼び出し制御・オーナー単位の継続処理のみを担うことをテストで区別して確認する。

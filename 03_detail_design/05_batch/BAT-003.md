# <span id="BAT-003"></span>BAT-003: FAQ一括取り込みジョブ非同期実行

> **本バッチは、FAQ CSV インポート API が受け付けた取込ジョブを Cloudflare Queues 経由で非同期に起動し、取込対象を行単位に読み込んで新規登録・既存上書き・行失敗のいずれかへ反映し、全行処理後に成功・失敗件数と失敗明細を集計して依頼者への完了通知まで実行する処理の詳細設計を定義します。**

*種別 バッチ処理設計 ・ ステータス ドラフト*

## 1. 目的

本バッチは、FAQ CSV インポート API([API-028](../../02_basic_design/02_backend/03_apis/API-028.md#API-028))が受け付けた取込ジョブ([TBL-033](../../02_basic_design/02_backend/04_database/TBL-033.md#TBL-033) `TP_IMPORT_JOBS`)を画面応答を待たせずに Cloudflare Queues 経由で非同期に処理し、取込対象を行ごとに読み込んで新規登録・既存上書き・行失敗のいずれかへ反映したうえで、全行処理後に成功件数・失敗件数と失敗明細を集計して依頼したアカウント利用者へ完了を知らせる。詳細化元のシステム設計は [SYS-014](../../02_basic_design/02_backend/01_system/SYS-014.md#SYS-014)(FAQ一括取り込みジョブ非同期実行)であり、その `async` 契機・処理項目([PR-01](../../02_basic_design/02_backend/01_system/SYS-014.md#PR-01)〜[PR-07](../../02_basic_design/02_backend/01_system/SYS-014.md#PR-07))を Cloudflare Queues の消費処理へ写像する。行単位の判定・バリデーション・反映ロジックは [IPO-015](../04_ipo/IPO-015.md#IPO-015) に委ね、ジョブの状態遷移(`queued`/`processing`/`completed`/`failed`)は [STS-006](../01_state_transitions/STS-006.md#STS-006) に委ねる。

## 2. バッチ概要

本バッチは FAQ CSV インポート API がジョブを受け付けた時点で Cloudflare Queues へメッセージを投入したことを契機に起動する非同期消費処理である。起動方式・多重起動可否・リトライ有無・想定規模を示す。

| 項目 | 値 |
|----|----|
| バッチ ID | BAT-003 |
| バッチ名 | FAQ一括取り込みジョブ非同期実行 |
| 起動方式 | Queues(イベント契機。[API-028](../../02_basic_design/02_backend/03_apis/API-028.md#API-028) がジョブ受付時([STS-006](../01_state_transitions/STS-006.md#STS-006) `(なし)→queued`)にメッセージを投入し、その消費で起動)。加えて `processing` 滞留の異常終了確定(§8)は Cron 監視経路([SYS-014](../../02_basic_design/02_backend/01_system/SYS-014.md#SYS-014) SEV-070)で起動する |
| 起動タイミング | FAQ CSV インポート依頼が受け付けられジョブ(`status=queued`)が作成されるたび(1 ジョブ = 1 メッセージ)。滞留監視は[システム仕様書 §7](../../02_basic_design/07_system-spec.md#7-バッチ運用設計値)の FAQ 取込ジョブ 滞留監視間隔で定期実行する |
| 多重起動可否 | 不可(同一ジョブは 1 ワーカーのみが処理する。多重配信時は §6 の競合検知でスキップ) |
| リトライ有無 | あり(Queues の自動リトライ。上限超過分は DLQ へ退避。[NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036) を参照し本文へ回数を再掲しない) |
| 想定処理件数 | 1 ファイルあたり最大件数(設計値・[API-028](../../02_basic_design/02_backend/03_apis/API-028.md#API-028))に準拠。1 プロジェクトあたり FAQ 件数の運用想定上限は [NFR-068](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-068) を参照 |
| 想定処理時間 | 1,000 件 / 10 分以内(目安値・非 SLA。[システム仕様書 §7](../../02_basic_design/07_system-spec.md#7-バッチ運用設計値)を参照) |

| 入力元 | 内容 | 取得条件 | 備考 |
|----|----|----|----|
| Queues メッセージ(取込ジョブ受付イベント) | 対象取込ジョブ ID・対象プロジェクト ID | [API-028](../../02_basic_design/02_backend/03_apis/API-028.md#API-028) がジョブを `queued` で作成した時に 1 ジョブ = 1 件投入したメッセージを消費する | ジョブ本体・受付済み CSV の格納方式(一時保管先)は基本設計に未定義(§9 引き継ぎ) |
| 取込ジョブ | 処理対象ジョブの現行状態 | メッセージの取込ジョブ ID で当該ジョブを取得する | `status=queued` の場合のみ処理開始対象とする([TBL-033](../../02_basic_design/02_backend/04_database/TBL-033.md#TBL-033)・[STS-006](../01_state_transitions/STS-006.md#STS-006)) |
| 取込対象 CSV の各行 | 行ごとの `FAQ ID` / 質問 / 回答 / カテゴリ | ジョブに紐づく取込対象を行番号順に読み込む | 受付時点の形式検証(CSV 形式・文字コード・ヘッダ行・件数上限・サイズ上限)は [API-028](../../02_basic_design/02_backend/03_apis/API-028.md#API-028) が既に通過済み。本バッチは行単位バリデーションのみ行う([IPO-015](../04_ipo/IPO-015.md#IPO-015)) |
| 既存 FAQ | 対象プロジェクトの既存 FAQ 一覧(`FAQ ID` 照合用) | メッセージの対象プロジェクト ID で当該プロジェクトの [`M_FAQS`](../../02_basic_design/02_backend/04_database/TBL-006.md#TBL-006) を取得する | 新規 / 上書き / 行失敗の判定は [IPO-015](../04_ipo/IPO-015.md#IPO-015) No.2 に従う |

## 4. 処理内容

本バッチの処理を実行順に示す。正常時・異常時の分岐を各行で表す。SYS-014 の処理項目([PR-01](../../02_basic_design/02_backend/01_system/SYS-014.md#PR-01)〜[PR-07](../../02_basic_design/02_backend/01_system/SYS-014.md#PR-07))に対応する。行単位の判定・バリデーション・反映の疑似コードは [IPO-015 §4](../04_ipo/IPO-015.md#IPO-015) を正本としここでは再定義しない。

| No | 処理 | 条件 | 正常時 | 異常時 | 備考 |
|----|----|----|----|----|----|
| 1 | 冪等チェック・処理開始判定 | メッセージ受信時 | 対象ジョブが `status=queued` であれば処理開始対象とし、先着ワーカーとして `status` を `processing` へ更新する([STS-006](../01_state_transitions/STS-006.md#STS-006) `queued→processing`) | `status` が既に `processing`/`completed`/`failed` であれば後着として当該メッセージをスキップし正常完了とする(多重配信対策) | 冪等キーは §6 参照。更新は条件付き更新で直列化する |
| 2 | 総行数確定 | 処理開始判定通過時 | 取込対象の総行数を確定してジョブへ記録する([SYS-014](../../02_basic_design/02_backend/01_system/SYS-014.md#PR-01)) | 記録失敗時は当該ジョブを失敗として継続し No.6 の異常終了処理へ進む | [TBL-033](../../02_basic_design/02_backend/04_database/TBL-033.md#TBL-033) `total_count` |
| 3 | 行単位ループ処理 | 未処理行が残っている間、行番号順に 1 行ずつ | 各行について行バリデーション → 新規 / 上書き / 行失敗判定 → 反映(新規登録は `status=draft` で登録、既存上書きは `status` 維持で上書き)を行う。処理済み行数・成功行数・失敗行数を逐次カウントする(判定条件・疑似コードは [IPO-015 No.1〜5](../04_ipo/IPO-015.md#IPO-015) を参照) | 当該行の反映失敗(制約違反等)は当該行のみ行失敗へ振り替え、他行の処理を継続する(1 行の失敗は他行に影響しない) | ループの各回はジョブ全体を異常終了させない。行失敗の記録先は [TBL-033](../../02_basic_design/02_backend/04_database/TBL-033.md#TBL-033) `error_summary` |
| 4 | 進捗記録 | 行単位ループ処理中、一定行数ごと | 処理済み行数・成功行数・失敗行数をジョブへ中間反映し、画面ポーリング([API-067](../../02_basic_design/02_backend/03_apis/API-067.md#API-067))から参照可能にする | 中間反映の失敗は次回の進捗記録機会で再試行し、ループ処理自体は継続する | 反映間隔は 50 行ごと([システム仕様書 §7](../../02_basic_design/07_system-spec.md#7-バッチ運用設計値) FAQ 取込ジョブ 進捗記録間隔) |
| 5 | 件数上限ガード・件数集計・完了確定 | 全行の処理完了後([SYS-014](../../02_basic_design/02_backend/01_system/SYS-014.md#PR-06)) | 件数上限ガード([IPO-015 No.6](../04_ipo/IPO-015.md#IPO-015))が範囲内のとき、成功件数・失敗件数・処理済み件数・失敗明細を確定してジョブへ記録し `status` を `completed` へ更新する([STS-006](../01_state_transitions/STS-006.md#STS-006) `processing→completed`。全件成功・部分失敗を区別しない共通状態) | 件数上限ガード超過時は当該ジョブで反映した全行をロールバックしジョブを `failed` へ更新する(理由 `CSV_FAQ_LIMIT_EXCEEDED`・[ERR-037](../../02_basic_design/05_errors/ERR-037.md#ERR-037)・部分取込しない)。集計・更新処理自体の失敗もジョブを `failed` へ更新する([STS-006](../01_state_transitions/STS-006.md#STS-006) `processing→failed`) | 取込は単一トランザクション。件数上限ガードは [IPO-015 No.6](../04_ipo/IPO-015.md#IPO-015)、集計式は [IPO-015 No.7](../04_ipo/IPO-015.md#IPO-015) を参照 |
| 6 | 完了通知([SYS-014](../../02_basic_design/02_backend/01_system/SYS-014.md#PR-07)) | 件数集計・完了確定後 | 依頼したアカウント利用者へ完了を知らせ、全件成功時と一部失敗時で種別を分けて成功・失敗の件数と失敗明細を提示する | — | 通知は画面ポーリング([API-067](../../02_basic_design/02_backend/03_apis/API-067.md#API-067))による結果参照のみで行い、受信箱・メール等の別チャネル通知は行わない([SYS-014](../../02_basic_design/02_backend/01_system/SYS-014.md#PR-07)) |
| 7 | 異常終了時のジョブ確定 | ワーカー処理中に回復不能な異常(DB 書込失敗の継続・予期しない例外等)が発生したとき | `status` を `failed` へ更新する。既に確定済みの行単位結果(成功/行失敗)は巻き戻さない([STS-006](../01_state_transitions/STS-006.md#STS-006) `processing→failed`) | 異常内容をログへ記録し運用へ通知する(§8) | ジョブ全体の失敗は行単位の失敗(No.3)とは区別する |

## 6. 排他・並列処理

ジョブ単位で 1 ワーカーのみが処理を進行させ、行単位の反映は当該ジョブ内で順に処理する。

| 項目 | 内容 |
|----|----|
| 排他制御 | 同一取込ジョブへの処理開始は `status` の条件付き更新(`queued→processing` を先着のみ成立させる)で直列化する。後着ワーカーは更新不成立を検知して当該メッセージをスキップする([STS-006](../01_state_transitions/STS-006.md#STS-006)) |
| 並列処理 | 異なる取込ジョブは Queues の消費で並列に処理する。同一ジョブ内の行単位ループは順次処理とする(トランザクション境界・並列化方式の具体実装は詳細実装に委ねる) |
| 多重起動時の扱い | 同一ジョブに対する重複メッセージ配信時は、処理開始判定(No.1)で後着をスキップする。処理中の同一ジョブへの追加メッセージも同様にスキップする |
| 冪等キー | 取込ジョブ ID([TBL-033](../../02_basic_design/02_backend/04_database/TBL-033.md#TBL-033) の `id`)を冪等キーとし、処理開始・完了確定の重複実行を抑止する。行単位の反映結果は行番号を単位に一意に確定し、再処理時に同一行を二重反映しない(冪等性の保証は [NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036) を参照し、キーの物理構成は詳細実装に委ねる) |

## 8. 異常終了時の扱い

異常検知時の対応・通知先・リトライ方針を示す。リトライ上限は本文に再掲せず NFR を参照する。

| 異常内容 | 対応 | 通知先 | リトライ | 備考 |
|----|----|----|----|----|
| 行単位の反映失敗(制約違反等) | 当該行のみ行失敗として記録し、ジョブ全体は継続する([IPO-015](../04_ipo/IPO-015.md#IPO-015) No.5) | — | なし(行単位は即時失敗記録・ジョブとしては正常進行) | 1 行の失敗は他行に影響しない |
| 総行数確定・進捗記録・集計確定処理自体の失敗(競合・一時障害) | 当該メッセージを失敗として Queues の自動リトライへ回す。上限超過時はジョブを `failed` へ確定する | 運用 | あり([NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036)。Queues 経由は自動リトライ、上限超過分は DLQ へ退避) | 既に確定済みの行単位結果は巻き戻さない |
| ワーカー処理中の回復不能な例外 | `status` を `failed` へ更新し処理を打ち切る([STS-006](../01_state_transitions/STS-006.md#STS-006) `processing→failed`) | 運用 | あり(同上) | 部分結果は保証しない([STS-006 §6](../01_state_transitions/STS-006.md#STS-006)) |
| `processing` 滞留(タイムアウト監視) | 監視処理(Cron Triggers)が進捗の更新されない滞留ジョブを検知し `status` を `failed` へ更新する。ワーカー側が同時に `completed` へ更新済みの場合は先着(ワーカー側)を優先する | 運用 | 監視契機による確定(自動) | 監視間隔・滞留判定しきい値は[システム仕様書 §7](../../02_basic_design/07_system-spec.md#7-バッチ運用設計値)を参照([SYS-014](../../02_basic_design/02_backend/01_system/SYS-014.md#SYS-014) SEV-070) |
| リトライ上限超過(DLQ 退避) | DLQ に退避し滞留を監視して手動再処理する | 運用 | 手動再処理 | 再処理待ち滞留は [NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036) に従いアラート |

## 9. 後続工程への引き継ぎ事項

テスト設計・運用設計へ渡す確認観点、および基本設計で未確定の点を箇条書きで示す。

- 境界値: 0 行 / 1 ファイルあたり最大件数(設計値)での行単位ループ・件数集計・完了通知の動作。
- 部分失敗: 一部行の反映失敗時に他行の処理が継続し、成功・失敗件数と失敗明細が正しく確定すること([IPO-015](../04_ipo/IPO-015.md#IPO-015))。
- 冪等性: 同一取込ジョブへの重複メッセージ配信・自動リトライ時に処理開始・完了確定が二重に行われないこと、行単位の反映が再処理時に二重登録・二重更新を起こさないこと。
- 排他: 同一ジョブへの多重ワーカー処理開始時に後着が正しくスキップされること(先着優先)。
- FAQ 件数上限との関係: 新規登録行が [NFR-068](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-068) の運用想定上限(段階通知 80% / 100%・強制拒否 120%)に到達・超過する場合の取込時の扱いが基本設計に未定義のため、[IPO-015 §5](../04_ipo/IPO-015.md#IPO-015) と同様にテスト観点として引き継ぐ(課題化要否の確認)。

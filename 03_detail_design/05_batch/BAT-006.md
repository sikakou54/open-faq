# <span id="BAT-006"></span>BAT-006: 決済失敗猶予・サスペンション移行

> **本バッチは、課金プロバイダからの決済失敗確定 / 再決済成功 / 解除の通知受信と、猶予経過を判定する定期起動という複合トリガーを Cloudflare の実行基盤へ写像し、サスペンション移行・通常状態復帰の確定を実行機構の観点(起動契機・スケジュール・冪等・排他・異常終了時の扱い)から定義する処理の詳細設計を定義します。判定ロジックそのものは [IPO-003](../04_ipo/IPO-003.md#IPO-003)、状態遷移そのものは [STS-003](../01_state_transitions/STS-003.md#STS-003) を正本とします。**

*種別 バッチ処理設計 ・ ステータス ドラフト*

## 1. 目的

本バッチは、詳細化元のシステム設計 [SYS-020](../../02_basic_design/02_backend/01_system/SYS-020.md#SYS-020)(決済失敗猶予・サスペンション移行、種別 `monitor`)が定義する「決済失敗確定 / 再決済成功 / 解除の通知受信」と「猶予経過を判定する定期起動」という複合トリガーを、Cloudflare Queues(通知受信契機)と Cron Triggers(猶予経過の定期判定契機)へ写像し、起動契機・冪等・排他・異常終了時の扱いを実行機構として定義する。判定条件・状態確定ロジックは [IPO-003](../04_ipo/IPO-003.md#IPO-003) に委ね、本書は重複しない。

## 2. バッチ概要

本バッチは 2 系統のトリガー(通知受信によるイベント契機・猶予経過判定の定時契機)を持つ複合トリガー型の監視処理である。起動方式・多重起動可否・リトライ有無・想定規模を示す。

| 項目 | 値 |
|----|----|
| バッチ ID | BAT-006 |
| バッチ名 | 決済失敗猶予・サスペンション移行 |
| 起動方式 | Queues(イベント契機。決済失敗確定 / 再決済成功 / 解除の通知受信)+ Cron Triggers(定時。猶予経過判定) |
| 起動タイミング | 通知受信契機は課金プロバイダ通知の検証・取込([SYS-004](../../02_basic_design/02_backend/01_system/SYS-004.md#SYS-004))が完了しメッセージを投入するたび。定期判定契機の起動周期は[システム仕様書 §7](../../02_basic_design/07_system-spec.md#7-バッチ運用設計値)(Cron スケジュール SYS-020・毎時 0 分)を正本とする(設計判断・運用確認対象は §9) |
| 多重起動可否 | 通知受信契機はメッセージ単位で並列消費可(可)。定期判定契機は同一時刻の重複起動を不可とする(不可。多重起動時の扱いは §6) |
| リトライ有無 | あり([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040) / [NFR-036](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036) を参照。本文に回数を再掲しない) |
| 想定処理件数 | 定期判定契機は猶予監視中の課金アカウント一覧が対象(想定規模は運用開始後の実績に基づき確定・§9 課題) |
| 想定処理時間 | 通知受信契機は 1 メッセージ単位で完結。定期判定契機は次回起動までに完了する範囲 |

## 3. 入力

本バッチは通知受信契機のメッセージと定期判定契機の起動信号を入力とし、判定に必要な現在状態を参照する。物理カラム名の羅列は行わず、テーブルはリンクで辿る。

| 入力元 | 内容 | 取得条件 | 備考 |
|----|----|----|----|
| Queues メッセージ(決済失敗確定通知契機) | 対象課金アカウント、通知の真正性検証結果、冪等性キー `(provider, event_id)` | [SYS-004](../../02_basic_design/02_backend/01_system/SYS-004.md#SYS-004) が通知の受信・検証・取込を完了した時に投入したメッセージを消費する | 検証・冪等性キーの物理照合そのものは SYS-004 が担う。本バッチは検証済み結果を受け取る |
| Queues メッセージ(再決済成功 / 解除通知契機) | 対象課金アカウント、通知の真正性検証結果、冪等性キー `(provider, event_id)` | 同上(SYS-004 経由) | 猶予中 / サスペンション中いずれの状態でも受理する([IPO-003](../04_ipo/IPO-003.md#IPO-003) No.6) |
| Cron Triggers 起動信号(猶予経過判定契機) | 起動時刻のみ | 定時起動 | 対象抽出は次行 |
| 課金アカウント状態 | 猶予監視中かつ `active` の課金アカウント一覧・各アカウントの猶予起点 | 定期判定契機の起動時に、[TBL-018](../../02_basic_design/02_backend/04_database/TBL-018.md#TBL-018) `grace_started_at IS NOT NULL` かつ `status == active` を条件に対象を抽出する | 猶予起点の保持先は [TBL-018](../../02_basic_design/02_backend/04_database/TBL-018.md#TBL-018) `grace_started_at` で確定([IPO-003 §5](../04_ipo/IPO-003.md#IPO-003))。状態の意味は[状態モデル §2](../../02_basic_design/08_state-model.md#2-課金アカウント状態)([TBL-002](../../02_basic_design/02_backend/04_database/TBL-002.md#TBL-002)) |

## 4. 処理内容

本バッチの処理を実行順に示す。判定条件・状態確定ロジックの詳細は [IPO-003](../04_ipo/IPO-003.md#IPO-003) を参照し、本書は実行機構(トリガ振り分け・冪等制御・呼び出し順)に限定する。正常時・異常時の分岐を各行で表す。

| No | 処理 | 条件 | 正常時 | 異常時 | 備考 |
|----|----|----|----|----|----|
| 1 | トリガー振り分け | 常時 | メッセージ種別(決済失敗確定 / 再決済成功・解除)または Cron 起動を判別し、対応する処理へ振り分ける | 種別判別不能なメッセージは処理せず失敗として記録 | 振り分け先は No.2〜4 |
| 2 | 決済失敗確定通知の処理呼び出し | 決済失敗確定通知メッセージを消費したとき | 冪等チェック通過後、[IPO-003](../04_ipo/IPO-003.md#IPO-003) No.1・No.2・No.3(通知真正性・新規性判定、猶予監視開始、オーナーへの重要通知)を呼び出す | 判定結果が「処理せず冪等スキップ」の場合は当該メッセージを正常完了として終了 | 通知の真正性検証・冪等性キー物理照合は [SYS-004](../../02_basic_design/02_backend/01_system/SYS-004.md#SYS-004) 済み結果を用いる |
| 3 | 猶予経過判定の呼び出し(定期起動) | Cron Triggers 起動時 | 猶予監視中かつ `active` の課金アカウント一覧を抽出し、各アカウントに対し [IPO-003](../04_ipo/IPO-003.md#IPO-003) No.3・No.4(猶予経過判定、サスペンション移行確定)を呼び出す | 一覧抽出・個別アカウントの判定失敗は当該アカウントのみ失敗として記録し、他アカウントの処理は継続 | 抽出対象・境界値判定は IPO-003 を正本とする |
| 4 | 再決済成功 / 解除通知の処理呼び出し | 再決済成功 / 解除通知メッセージを消費したとき | 冪等チェック通過後、[IPO-003](../04_ipo/IPO-003.md#IPO-003) No.5・No.6(通知真正性・新規性判定、即時復帰確定)を呼び出す | 判定結果が「処理せず冪等スキップ」の場合は当該メッセージを正常完了として終了 | 猶予経過判定(No.3)と同時到達した場合は復帰確定を優先する([IPO-003](../04_ipo/IPO-003.md#IPO-003) No.5 備考・§6) |
| 5 | 状態確定の記録 | No.2〜4 いずれかが状態変化を確定したとき | 課金アカウント状態([TBL-002](../../02_basic_design/02_backend/04_database/TBL-002.md#TBL-002))を更新する | 更新失敗時は当該処理単位(メッセージ / アカウント)のみ失敗として記録し継続 | 状態遷移の許可・禁止は [STS-003](../01_state_transitions/STS-003.md#STS-003) を正本とする |

## 6. 排他・並列処理

通知受信契機はメッセージ単位で並列消費し、定期判定契機は同一時刻の重複起動を避ける。猶予経過判定によるサスペンション移行確定と、通知受信による即時復帰確定がほぼ同時に到達した場合は復帰を優先する([IPO-003](../04_ipo/IPO-003.md#IPO-003) No.5 備考)。

| 項目 | 内容 |
|----|----|
| 排他制御 | 猶予経過判定(定期起動)によるサスペンション移行確定と、通知受信による即時復帰確定が同一課金アカウントに対しほぼ同時に到達した場合は、復帰確定を優先する順序で状態を確定する。実装上の排他制御(楽観ロック / 更新順序)は詳細実装に委ねる([IPO-003 §5](../04_ipo/IPO-003.md#IPO-003)の引き継ぎに対応) |
| 並列処理 | 通知受信契機は課金アカウント単位で並列消費してよい。定期判定契機は抽出した課金アカウント一覧を並列に判定してよいが、同一課金アカウントへの並列書込は排他制御(前行)で直列化する |
| 多重起動時の扱い | 定期判定契機(Cron Triggers)は前回起動が完了していない場合、新規の重複起動は行わない(スキップ)。通知受信契機(Queues)は同一メッセージの重複配信を冪等キーで判定・スキップする |
| 冪等キー | 通知受信契機は冪等性キー `(provider, event_id)`(物理照合は [SYS-004](../../02_basic_design/02_backend/01_system/SYS-004.md#SYS-004) が担う)を用い、猶予監視開始・即時復帰確定の重複実行を防ぐ。冪等性の保証は [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) を参照する。定期判定契機は課金アカウント単位の状態確定が自然に冪等(既に `suspended` / 猶予未経過なら移行しない)であり、追加の冪等キーを要しない |

## 8. 異常終了時の扱い

異常検知時の対応・通知先・リトライ方針を示す。リトライ上限は本文に再掲せず NFR を参照する。

| 異常内容 | 対応 | 通知先 | リトライ | 備考 |
|----|----|----|----|----|
| 通知受信契機の処理失敗(検証結果照合・状態更新の一時障害) | 当該メッセージのみ失敗として再処理へ回す。他メッセージの処理は継続 | 運用 | あり([NFR-040](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040)。Queues 経由は自動リトライ、上限超過分は DLQ へ退避) | 冪等キーにより再処理時の二重確定は防止される |
| 定期判定契機の一覧抽出失敗 | 当該起動全体を失敗として記録し、次回起動での再判定に委ねる | 運用 | 次回定期起動での再判定 | 猶予経過の見逃しを避けるため周期の余裕を運用設計で確認する(§9) |
| 定期判定契機の個別課金アカウント判定・状態更新失敗 | 当該課金アカウントのみ失敗として記録し、他アカウントの判定は継続 | 運用 | 次回定期起動での再判定 | 部分失敗時も他アカウントのサスペンション移行判定を妨げない |
| リトライ上限超過(DLQ 退避、通知受信契機) | DLQ に退避し滞留を監視して手動再処理する | 運用 | 手動再処理 | 再処理待ち滞留は [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) に従いアラート |

## 9. 後続工程への引き継ぎ事項

テスト設計・運用設計へ渡す確認観点、および設計判断・運用確認の対象を箇条書きで示す。

- 境界値: 猶予期間ちょうど経過([IPO-003](../04_ipo/IPO-003.md#IPO-003) `>=` 判定)での移行有無、猶予監視中に決済失敗確定通知を重複受信した場合の猶予起点非更新([IPO-003](../04_ipo/IPO-003.md#IPO-003) No.2)。
- 冪等性: 通知受信契機の重複配信・自動リトライ時に猶予監視開始・即時復帰確定が二重に発生しないこと。
- 排他: 猶予経過判定によるサスペンション移行確定と即時復帰確定がほぼ同時到達した場合に復帰が優先されること(§6)。
- 部分失敗: 定期判定契機で一部課金アカウントの判定・更新が失敗しても他アカウントの判定が継続すること。
- DLQ 運用: 通知受信契機の DLQ 滞留検知・アラート・手動再処理手順([NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041))。
- 定期判定の起動周期: 猶予期間([システム仕様書 §4](../../02_basic_design/07_system-spec.md#4-データ保持期間削除猶予)、7 日)に対する定期判定の起動周期は[システム仕様書 §7](../../02_basic_design/07_system-spec.md#7-バッチ運用設計値)(Cron スケジュール SYS-020・毎時 0 分)を正本とする(設計判断・運用確認対象)。猶予経過の見逃し・確定遅延が生じない周期であることをテスト設計・運用設計で確認する。
- 猶予起点の保持先: [TBL-018](../../02_basic_design/02_backend/04_database/TBL-018.md#TBL-018) `grace_started_at` で確定(決済失敗確定通知の受信時刻を `past_due` 遷移時に記録し、`active` 復帰時に NULL へリセット)。本バッチの §3 入力(猶予監視中の一覧抽出条件)は `grace_started_at IS NOT NULL` を条件とする。物理制約・抽出用部分インデックスは [DBP-011](../07_db_physical/DBP-011.md#DBP-011) を参照。

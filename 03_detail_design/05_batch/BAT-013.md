# <span id="BAT-013"></span>BAT-013: 保持期間経過アカウントの物理削除

> **本バッチは、日次の定期起動を契機に、退会から保持期間([システム仕様書 §4](../../02_basic_design/07_system-spec.md#4-データ保持期間削除猶予))を経過した課金アカウントの課金・請求従属データおよび認証従属データを依存順に物理削除し、アカウントを削除済み(`deleted`)のトムストーンへ移行する実行機構(起動・排他・冪等・監視・異常終了時の扱い)の詳細設計を定義します。削除対象か否かの判定ロジックそのものは本バッチでは行いません([IPO-010](../04_ipo/IPO-010.md#IPO-010) が正本)。**

*種別 バッチ処理設計 ・ ステータス ドラフト*

## 1. 目的

本バッチは、日次の定期起動により、退会済みの課金アカウントのうち退会日時 `withdrawn_at` から保持期間([システム仕様書 §4](../../02_basic_design/07_system-spec.md#4-データ保持期間削除猶予))を経過したものを走査し、課金・請求従属データおよび認証従属データを依存順に物理削除したうえで、課金アカウントを物理削除しアカウント(利用者)を削除済み(`deleted`)のトムストーンへ移行する。詳細化元のシステム設計は [SYS-034](../../02_basic_design/02_backend/01_system/SYS-034.md#SYS-034)(保持期間経過アカウントの物理削除)であり、その `batch` 種別・「日次の実行スケジュールによる自動起動」を Cloudflare Cron Triggers の定時実行へ写像する。退会日時起点の保持期間経過判定そのものは [IPO-010](../04_ipo/IPO-010.md#IPO-010) に委ね、本バッチは起動制御・依存順の削除実行・排他・冪等・監視・異常終了時の扱いを定義する。

## 6. 排他・並列処理

課金アカウント単位で削除処理を進行し、アカウントをまたぐ処理は互いに独立させる一方、同一アカウント内は依存順を厳守する。

| 項目 | 内容 |
|----|----|
| 排他制御 | バッチ全体の同時実行を抑止し(多重起動不可)、単一インスタンスが日次の削除対象アカウントを順に処理する。ロックの具体的実装は詳細実装に委ねる |
| 並列処理 | アカウント単位では独立して処理してよいが、同一アカウント内の依存順削除(課金・請求従属 → 認証従属 → 課金アカウント → アカウント)は直列に実行する。並列度の具体値は負荷想定([システム仕様書 §5](../../02_basic_design/07_system-spec.md#5-レート制限キャパシティ目安))を参照する |
| 多重起動時の扱い | 前回起動が完了していない場合、新規の重複起動は行わない(スキップ) |
| 冪等キー | 課金アカウント識別子を冪等キーとし、既に物理削除済み(または `deleted` へ移行済み)のアカウントは再処理をスキップする。冪等性の保証は [NFR-041](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041) を参照し、キーの物理構成は詳細実装に委ねる |

## 9. 後続工程への引き継ぎ事項

テスト設計・運用設計へ渡す確認観点、および基本設計・関連設計で未確定の点を箇条書きで示す。

- 境界値: 0 件 / 保持期間経過当日 / 保持期間未経過での抽出・非抽出の動作。
- 依存順削除: 課金・請求従属 → 認証従属 → 課金アカウント → アカウントの順で削除され、FK 制約違反が発生しないこと。
- 冪等性: 同一アカウントへの重複実行(再起動・多重起動抑止の境界含む)で二重削除・二重トムストーン化が発生しないこと。
- 部分失敗: アカウント単位の削除失敗時に他アカウントの処理が継続し、失敗分が次回起動で正しく再評価されること。中途状態(依存順の一部のみ削除済み)が放置されないこと。
- トムストーン化: アカウントが `status='deleted'` へ移行後、識別子が再利用されないこと([NFR-051](../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-051))。
- 監査記録: 削除内容が保持クラス `billing` として監査ログに記録され、保持義務の範囲で扱われること([システム仕様書 §4](../../02_basic_design/07_system-spec.md#4-データ保持期間削除猶予))。
- 運用データとの役割分担(確定): 退会済みアカウントの運用データ(FAQ・プロジェクト・質問ログ・利用量・通知・お知らせ受信箱など)は退会時に速やかに削除済み([SYS-027](../../02_basic_design/02_backend/01_system/SYS-027.md#SYS-027) が担当)であり、本バッチは課金・請求・認証の保持対象データを保持期間経過後に確定削除する役割に限定される。両バッチとも走査起点は課金アカウント([TBL-002](../../02_basic_design/02_backend/04_database/TBL-002.md#TBL-002))の `status='withdrawn'` で統一されている(`M_USER.status` の値集合とは別系統。[状態モデル §1](../../02_basic_design/08_state-model.md#1-アカウント状態) / [§2](../../02_basic_design/08_state-model.md#2-課金アカウント状態))。テストでは両バッチの対象データが重複・欠落なく分担されていることを確認する。
- 対応する詳細シーケンス(DSQ)は本書執筆時点で未作成(退会カスケードの詳細シーケンスとして起票が想定される)。作成後は本書 §1 からの参照要否を確認する。

## 1. 目的

本物理設計は、運営お知らせの本体と配信先限定指定、実配信先ごとの受信者・配信状態、利用者個人の受信箱、メール通知の送信履歴という一連のお知らせ配信系 5 テーブルの物理写像を Cloudflare D1(SQLite)上で確定する。詳細化元は基本設計 [TBL-010](../../02_basic_design/02_backend/04_database/TBL-010.md#TBL-010)・[TBL-011](../../02_basic_design/02_backend/04_database/TBL-011.md#TBL-011)・[TBL-021](../../02_basic_design/02_backend/04_database/TBL-021.md#TBL-021)・[TBL-022](../../02_basic_design/02_backend/04_database/TBL-022.md#TBL-022)・[TBL-026](../../02_basic_design/02_backend/04_database/TBL-026.md#TBL-026)であり、そのカラム物理名・コード値・インデックスと齟齬を生じさせず、fan-out(1 件のお知らせから多数の受信者・受信箱行を生成する)構造・既読状態の索引化・配信履歴保持のインデックス設計を確定する。

## 4. インデックス定義

## 7. 後続工程への引き継ぎ事項

テスト設計・マイグレーション設計・詳細ロジック設計へ渡す観点を箇条書きで示す。

- CHECK 制約違反(`importance` / `delivery_status` / `delivery_state` / `category` / `priority` / 桁数上限)・NOT NULL 違反・FK 違反時のふるまいはテスト設計でケース化する。
- `M_SERVICE_ANNOUNCE.created_by_type` / `created_by_id` の論理参照(物理 FK を張らない参照先切替)は、参照整合性をアプリ層でどう担保するかを詳細ロジック設計で確定し、テスト設計へ反映する。
- `M_ANNOUNCE_AUD` を `WITHOUT ROWID`(複合主キー)テーブルとするか通常テーブル + 一意インデックスとするかは、`idx_announcement_audiences_user` の書込みコストと合わせてマイグレーション設計で確定する。
- fan-out(`T_ANNOUNCE_RCPT` / `T_INBOX_MSG` への一括挿入)のトランザクション境界・部分失敗時の再試行方式([SYS-022](../../02_basic_design/02_backend/01_system/SYS-022.md#SYS-022))は詳細ロジック設計で確定する。
- `idx_inbox_dedup` による 60 分集約判定([SYS-026](../../02_basic_design/02_backend/01_system/SYS-026.md#SYS-026))の同時到達時の整合(同一 `dedup_key` への競合挿入の排他制御)は詳細ロジック設計・テスト設計の双方で確認する。
- `T_INBOX_MSG` / `H_NOTIF_LOGS` は論理削除列を持たない追記専用テーブルであるため、保持期間超過分は [SYS-032](../../02_basic_design/02_backend/01_system/SYS-032.md#SYS-032) により `created_at` 起点で直接物理削除される。退会時の即時削除([SYS-012](../../02_basic_design/02_backend/01_system/SYS-012.md#SYS-012))との削除順序・再試行方式はマイグレーション設計・詳細ロジック設計で確定する。
- 論理項目 ↔ 物理カラムの対応は詳細設計の入出力設計書(お知らせ一覧・未読件数・既読化の各 API 対応 IO)と突き合わせて整合を確認する。

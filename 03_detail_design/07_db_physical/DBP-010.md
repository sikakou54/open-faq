## 1. 目的

本物理設計は、プロジェクト単位の月次利用量を計測する `T_USAGE_METER`、プロジェクト単位の月次上限件数・無料枠・アラート閾値を保持する `M_PRJ_QUOTA_LIMITS`、オーナー単位のレート制限上書きを保持する `M_OWNER_QUOTA_OVR` という利用量・上限ドメインの 3 テーブルの物理写像を Cloudflare D1(SQLite)上で確定する。詳細化元は基本設計 [TBL-020](../../02_basic_design/02_backend/04_database/TBL-020.md#TBL-020)・[TBL-009](../../02_basic_design/02_backend/04_database/TBL-009.md#TBL-009)・[TBL-008](../../02_basic_design/02_backend/04_database/TBL-008.md#TBL-008) であり、そのカラム物理名・コード値・インデックスと齟齬を生じさせず、SQLite の型親和性・主キー採番・一意制約・JSON 検証・容量性能見積を確定する。しきい値・単価・無料枠そのものの具体値は本書で再定義せず、[システム仕様書 §2](../../02_basic_design/07_system-spec.md#2-課金利用量上限)・[システム仕様書 §5](../../02_basic_design/07_system-spec.md#5-レート制限キャパシティ目安) を正本とする。

## 7. 後続工程への引き継ぎ事項

テスト設計・マイグレーション設計・詳細ロジック設計へ渡す観点を箇条書きで示す。

- 主キーを `WITHOUT ROWID`(ULID 主キー)とするか通常テーブルとするかは、`T_USAGE_METER` の高頻度な同期加算更新([SYS-016](../../02_basic_design/02_backend/01_system/SYS-016.md#SYS-016))の書込みコストと合わせてマイグレーション設計で確定する。
- CHECK 制約違反(`resource_kind` / `source` / 真偽フラグ / `unbillable_question_count <= question_count` / `M_PRJ_QUOTA_LIMITS` の複合制約 `ck_pql_kind_threshold_shape`)・NOT NULL 違反・FK 違反・一意制約違反(`(project_id, billing_ym)` / `(project_id, resource_kind, source)`)時のふるまいはテスト設計でケース化する。特に月初の `T_USAGE_METER` 新規行作成と同時加算の競合(初回加算時に行が存在しない場合の作成順序)を確認する。
- `M_PRJ_QUOTA_LIMITS.alert_thresholds` の配列要素(許可値 `25`/`50`/`80`/`90`/`100`・重複排除・昇順化)は SQLite の CHECK では表現が困難なため、API 層検証(正規化)に委ねることをテスト設計へ引き継ぐ。値の全集合・意味は [TBL-009 `alert_thresholds` JSON 構造](../../02_basic_design/02_backend/04_database/TBL-009.md#alert_thresholds)を正本とする。
- `M_PRJ_QUOTA_LIMITS.created_by_id` / `M_OWNER_QUOTA_OVR.created_by_id` は `created_by_type` に応じて参照先テーブルが変わる論理参照であり、DB レベルの FK 制約を張らない(アプリ層で整合を保証する)方針の妥当性を詳細ロジック設計で確定する。
- `M_PRJ_QUOTA_LIMITS.billing_ym` 形式(`YYYY-MM`)の厳密検証(数字・ハイフン位置)を DB CHECK で強制するか、アプリ層検証のみに委ねるかは詳細ロジック設計で確定する。
- `M_OWNER_QUOTA_OVR.valid_until` 超過後の行を能動的に物理削除するバッチの要否・削除タイミングは詳細ロジック設計・バッチ設計([BAT](../05_batch/index.md))で確定する。
- 課金対象件数(= `question_count - unbillable_question_count`)を集計クエリの都度算出するか、参照頻度に応じて別途キャッシュ列を持つかは性能検証の結果を踏まえ詳細ロジック設計で確定する。
- 論理項目 ↔ 物理カラムの対応は各画面・API の入出力設計書(IO)と突き合わせて整合を確認する。

## 1. 目的

本物理設計は、プロジェクト本体・ウィジェット埋め込みを許可するドメイン・鍵ローテーション時の旧公開キーという密接に関連する 3 テーブルの物理写像を Cloudflare D1(SQLite)上で確定する。詳細化元は基本設計 [TBL-004](../../02_basic_design/02_backend/04_database/TBL-004.md#TBL-004)(`M_PROJECTS`)・[TBL-005](../../02_basic_design/02_backend/04_database/TBL-005.md#TBL-005)(`M_ALLOWED_DOMAINS`)・[TBL-015](../../02_basic_design/02_backend/04_database/TBL-015.md#TBL-015)(`T_PRJ_LEGACY_KEYS`)であり、そのカラム物理名・コード値・インデックスと齟齬を生じさせず、SQLite の型親和性・主キー採番・オーナー単位検索の索引化・鍵ローテーション保持・容量性能見積を確定する。

## 7. 後続工程への引き継ぎ事項

テスト設計・マイグレーション設計・詳細ロジック設計へ渡す観点を箇条書きで示す。

- 主キーを `WITHOUT ROWID`(ULID 主キー)とするか通常テーブルとするかは、3 テーブルとも書込み頻度・セカンダリインデックス数と合わせてマイグレーション設計で確定する。
- CHECK 制約違反(`status` / 文字数上限)・NOT NULL 違反・FK 違反・一意制約違反(`widget_public_key` / `(project_id, domain)`)時のふるまいはテスト設計でケース化する。特に `M_ALLOWED_DOMAINS` の部分 UNIQUE(`WHERE valid = 1`)が論理削除後の再登録を正しく許可するかを境界値として確認する。
- `M_ALLOWED_DOMAINS.domain` の形式検証(完全一致 / `*.example.com` ワイルドカード・IP アドレスやプロトコル・ポート・パスの拒否)を DB CHECK で強制するか、[API-017](../../02_basic_design/02_backend/03_apis/API-017.md#API-017) のアプリ層検証のみに委ねるかは詳細ロジック設計で確定する(SQLite の `CHECK` では正規表現相当の検証が困難なため)。
- `T_PRJ_LEGACY_KEYS` は遅延失効方式([API-037](../../02_basic_design/02_backend/03_apis/API-037.md#API-037) P-02)を採るが、`grace_until` 超過後の行の物理削除は [SYS-032](../../02_basic_design/02_backend/01_system/SYS-032.md#SYS-032) が `grace_until` 起点で担う(確定)。
- `M_PROJECTS.settings` の JSON 構造妥当性検証(キー・型・範囲)は DB 制約では担保せず、[API-017](../../02_basic_design/02_backend/03_apis/API-017.md#API-017) / [API-018](../../02_basic_design/02_backend/03_apis/API-018.md#API-018) のアプリ層検証に委ねることをテスト設計へ引き継ぐ。
- 論理項目 ↔ 物理カラムの対応は各画面・API の入出力設計書(IO)と突き合わせて整合を確認する。

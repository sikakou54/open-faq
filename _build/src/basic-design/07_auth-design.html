認証・認可設計書

このページは、利用者側(オーナー / プロジェクト管理者 / メンバー)の認証(本人確認)と認可(操作可否)の方針・判定フローを定義します。オーナーは M_CONTRACT マスタ、プロジェクト管理者 / メンバーは M_PRJ_USERS マスタで管理し、ログイン経路・メール一意空間・セッションを完全に分離します。

版数 v2.0更新 2026-06-17承認済

1. 認証方式

本システムが採用する本人確認の手段を一覧で示します。次の表は、各方式の採否と用途、有効期限などの制約・正本リンクを並べたもので、利用者の登録から日常ログイン、パスワード再設定、メンバー有効化までを 1 つの方式表で見渡せます。

認証方式採否説明制約・参照
メールアドレス + パスワード 採用 主たる認証方式。ログイン時にパスワードを検証する パスワード要件は §2、ハッシュ方式は 09_セキュリティ設計.html が正本
パスワード再設定リンク 採用 メールに送る短期トークンで本人到達を確認し再設定させる 有効期限 1 時間 / 1 回限り(FR-006)。トークン定義は T_ACCESS_TOKENS
招待トークン(メンバー有効化) 採用 招待リンクから本人が初回パスワード・氏名・規約同意を入力して有効化する 有効期限 7 日(FR-016 / FR-020)。フローは §3
メール確認トークン 採用 新規登録後の本人確認(FR-003) 画面は 01_画面設計/index.html(SCR-013)
規約再同意の割込み 採用 利用規約・プライバシーポリシー改定時にログイン後フローへ割込む(FR-011) 認可割込みルールは §6
Bot 対策(CAPTCHA 等) 採用 必要に応じてログイン等で発動する(FR-177) 判定方針は 09_セキュリティ設計.html が正本

3. 認証フロー

ログインからログアウト、パスワード再設定、メンバー有効化、メール確認、強制ログアウトまで、認証にかかわる各フローの処理手順をまとめます。以降の各小節は、それぞれのフローを順序付きの手順表で示します。

3.1 ログイン

通常ログイン時の処理を入口から初期画面遷移まで順に追います。ログイン経路はオーナーとプロジェクトユーザーで完全に分離し、入口ごとに照合先マスタ(オーナー = M_CONTRACT / プロジェクトユーザー = M_PRJ_USERS)を切り替えます。メール一意性も両マスタで別空間のため、同一メールでも別系統のアカウントとして扱います。次のシーケンス図は、資格情報の検証・セッション確立に加え、規約再同意ゲート(§6)と契約状態ゲート(§5.7)をログイン直後に通す順序と、不一致・停止状態などの分岐を示します。

sequenceDiagram
  participant U as ブラウザ(利用者)
  participant W as worker-main-api
  participant D as D1 (M_USER)
  participant K as KV (session)

  U->>W: メール + パスワード送信(ログイン経路で主体種別を識別)
  W->>D: ログイン経路に対応するマスタ(M_USER)で資格情報を照会
  D-->>W: 照合結果(一致 / 不一致)
  alt 不一致
    W-->>U: 認証失敗(失敗カウント加算・ヒントなし)
  else 一致
    W->>K: セッション確立(主体を actor_type='owner'/'project_user' + actor_id で保持)
    K-->>W: 
    W-->>U: セッション Cookie 発行
    W->>D: 規約再同意の要否を確認
    D-->>W: 未同意文書(あり / なし)
    opt 未同意あり
      W-->>U: 規約再同意へ割込み(SCR-015)
    end
    W->>D: 契約状態を確認
    D-->>W: 契約状態(active / suspended / deleted_pending / deleted)
    alt active
      W-->>U: 管理範囲に応じた初期画面へ遷移
    else suspended / deleted_pending / deleted
      W-->>U: 停止時アクセス制限(§5)を適用
    end
  end
補足不一致時は失敗カウントを加算し、ヒントを与えない応答を返します(必要に応じ Bot 対策チャレンジを挟む、FR-177)。契約状態が suspended / deleted_pending / deleted の場合は §5 のアクセス制限に従います。

4.2 失効の優先順位

複数の失効要因が同時に当てはまるとき、どれを優先して判定するかを定めます。次の表は強制ログアウトを最優先とし、絶対タイムアウト・無操作タイムアウト・通常セッションの順で評価する優先順位です。重要操作の再認証(§2)はこの寿命判定とは別軸で働きます。

5. 認可モデルと判定フロー

誰が何を操作できるかを判定する認可モデルを定めます。ユーザー種別とオーナー判定、契約・プロジェクトの境界、判定段の順序、専有・保護制約、通知宛先解決、アカウント状態・契約状態によるアクセス制限を以降の小節で順に示します。

5.1 ユーザー種別とオーナー判定

認可の起点となるユーザー種別と、その見分け方を定めます。オーナーとプロジェクトユーザーは別マスタで完全に分離し、次の表は主体が M_CONTRACT マスタ由来ならオーナー、M_PRJ_USERS マスタ由来なら M_PRJ_USERS.role で操作範囲が決まることを示します(FR-015a / FR-015b)。

ユーザー種別判定権限の表し方
オーナー 主体が M_CONTRACT マスタ由来(actor_type='owner')。M_CONTRACT.id が契約境界キー本体で 1 契約 1 オーナーを担保する 全プロジェクトへの全権 + オーナー専有機能。役割変更・降格・譲渡なし(FR-015a)
プロジェクト管理者 / メンバー 主体が M_PRJ_USERS マスタ由来(actor_type='project_user')。オーナーは M_PRJ_USERS に含まれない(完全分離) プロジェクトごとの役割(M_PRJ_USERS.role = admin / member)で操作範囲が決まる(FR-015b / FR-336)
ウィジェット利用者 管理画面の主体行を持たない 公開キーとウィジェットセッションで質問送信のみ。管理画面の Principal は構築しない
オーナー判定 = isOwner bypass を先頭にオーナーは M_CONTRACT マスタの独立した主体であり、M_PRJ_USERS / M_PRJ_USERS には行を持ちません。認可は主体が M_CONTRACT 由来(actor_type='owner')であることによる isOwner=true bypass を最優先とし、オーナーは自契約配下の全プロジェクトに無条件でアクセスできます(FR-015d / FR-021b)。プロジェクト管理者権限は M_PRJ_USERS.role='admin' の割当でのみ付与され、オーナーへの自動付与は行いません。通知宛先解決と画面表示では、オーナーを M_CONTRACT から、当該プロジェクトの管理者を M_PRJ_USERS.role='admin' から個別に解決します(§5.5)。

5.2 オーナー境界判定

5.4 オーナー専有機能・保護制約

特定のロールに付与してはならない専有機能と、運用が止まらないための保護制約を定めます。次の表はオーナー専有機能・オーナー保護・自己操作の禁止・最後の管理者保護をまとめたもので、各制約に該当する操作は §5.3 の対応する判定段で拒否されます(FR-333 / FR-334)。

制約内容根拠
オーナー専有機能 課金情報・支払方法・請求履歴・退会・規約再同意・契約設定、プロジェクトの作成・編集・削除はオーナー専有。いずれの役割にも付与しない。プロジェクト単位の利用上限・無料枠は、オーナーまたは当該プロジェクト管理者が変更できる FR-015c / BR-044
オーナー保護 オーナーに対する退会・停止・削除・降格・譲渡を一切受け付けない。1 契約 = 1 オーナーのため孤立状態は構造的に発生しない FR-333
自己操作の禁止 自分自身の利用状態変更・削除はできない。当該プロジェクトの管理者が自分 1 人のとき、自身を管理者からメンバーへ降格できない。自身のアカウント削除は常に不可で、他人を経由して自身を削除する経路も設けない FR-334
最後の管理者保護 役割変更・離脱の対象が当該プロジェクトの最後の管理者になる場合は拒否する。判定は M_PRJ_USERS.role='admin' の割当件数で行う(オーナーは M_PRJ_USERS に含まれず、isOwner bypass で常に全権を持つため件数判定の対象外) FR-018a / FR-334

5.5 critical 通知の宛先解決(共有概念正本)

重複排除同一のプロジェクトユーザーが複数プロジェクトで admin を持つ場合の二重配信を避けるため、主体(actor_type + actor_id)と正規化メールアドレスで重複排除する。両マスタはメール一意空間が別のため、合算後の重複排除も正規化メールアドレスで行う—
プロジェクト限定通知質問数上限の選択アラート等は対象プロジェクトに限定し、当該プロジェクトの管理者(オーナー + 当該 PJ の role='admin' 割当)のみに配信するFR-021a
配信契機・文面の正本通知契機・配信先一覧は 06_メッセージ一覧.html、本文テンプレートは 11_メール設計.html が正本です。本書は「誰が宛先か」を決める解決ロジックのみを定めます。

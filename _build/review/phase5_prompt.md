# 総合レビュー観点
## 1. 設計書の構成・記載粒度
- 要件定義に書くべき内容と基本設計に書くべき内容が分離されているか。BR/FR/UC/基本設計の粒度混在、要件への細かすぎる内容(画面ID・API ID・DB 項目)の混入、基本設計への詳細設計・実装レベル(SQL/メソッド名/算出式/物理名)の混入がないか。
## 2. 要件定義が基本設計を逆参照していないか
- 要件が下流(基本設計/画面/API/テーブル)を参照・委譲(「…を正本とする」「基本設計に明記」等)していないか。要件が下流識別子に依存していないか。トレース方向が要件→基本設計か。
## 3. 抜け漏れ
- 要件→基本設計の落とし込み漏れ(画面/API/テーブル/権限/エラー/メッセージ/バッチ/外部連携)。異常系・例外系・権限不足・入力不正・データ不整合・外部連携失敗の考慮漏れ。非機能(運用/監査/ログ/セキュリティ/性能)の考慮漏れ。
## 4. ドキュメント間整合性
- 用語/ID/ステータス/ロール/権限/メッセージ/エラーコードの統一。旧体系 ID の残存。廃止/重複/矛盾の残留。同一対象の説明ブレ。

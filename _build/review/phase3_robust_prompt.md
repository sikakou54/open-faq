# レビュー対象
全ソース(要件定義 + 基本設計)。判定基準は 00_rules。

# レビュー観点(異常系・外部連携・非機能の考慮漏れ)
1. 異常系・例外系の考慮漏れ: 権限不足・入力不正・データ不整合・競合(同時更新)・外部連携失敗(メール送信失敗・決済失敗・Webhook 重複/順序逆転)・タイムアウト・リトライ等が、API/SEQ/ERR で扱われているか。扱われていない重要ケースはないか。
2. エラー設計(ERR)とメッセージ(MSG)が、実際に各 API・画面イベントで発生し得るエラーを網羅しているか。API のエラー表にあるが ERR 未定義、または ERR にあるが画面/メッセージ未提示、の不整合はないか。
3. 権限設計(PERM)が、ロール(オーナー/メンバー等)× 操作の可否を必要分カバーしているか。権限判定が必要なのに PERM 欠落の操作はないか。
4. 非機能要件(NFR)で要求される 運用・監査・ログ・セキュリティ・パフォーマンス・可用性・バックアップ等が、基本設計(SYS/SEV/API/TBL/NFR)で具体化されているか。要求はあるが設計が無いものはないか。
5. データ保持・削除・退会・論理削除・保持期間などのライフサイクルが、要件と基本設計(TBL の論理削除/監査項目、削除バッチ等)で整合して設計されているか。抜けはないか。

# 出力形式(厳守)

| No | 重要度 | 起点要件 | 落とし込み先 | 現状 | 抜け漏れ内容 | 理由 | 修正方針 |

重要度=高/中/低。**指摘が無ければ「抜け漏れに関する指摘なし」と明記。** 根拠の無い推測指摘は挙げないこと(実在 ID を伴わない指摘は不可)。

# レビュー観点(要件定義の逆参照)
要件仕様(BR/FR/NFR/RULE)について、次を確認してください。
1. 基本設計層・下流ドキュメントの識別子が本文に混入していないか(画面ID `SCR-NNN` / 画面イベント `EVT-NNN` / API `API-NNN` / テーブル `TBL-NNN` / シーケンス `SEQ-NNN` / 権限 `PERM-NNN` / エラー `ERR-NNN` / メッセージ `MSG-NNN`)。
2. 「基本設計を参照」「画面設計を参照」「API設計を参照」「テーブル設計を参照」「…を正本とする」等、下流へ委譲・参照する記述がないか。
3. テーブル物理名(`M_`/`T_`/`H_`/`TP_` 接頭辞)・カラム名・SQL・エラーコード・エンドポイント・HTTPメソッドなど、基本設計/詳細設計レベルの識別子が混入していないか。
4. 「何を実現したいか(What/Why)」ではなく「どう実装するか(How)」に寄りすぎた、あるいは基本設計に合わせた後付けの要件記述になっていないか。
5. 基本設計から要件へ仕様を逆流させたような記述がないか。トレース方向が「要件定義 → 基本設計」になっているか。
6. CLAUDE.md の規約上 BR/FR が `### 関連` セクション(対応業務UC・関連NFR 等の下流リンク)を持っていないか。

# 出力形式(厳守)
最初に総括を1〜2文。続けて、指摘を次の Markdown 表で列挙してください。対象箇所は `===== <相対パス> =====` 区切りに基づく相対パス + 該当 ID または引用で必ず特定してください。

| No | 重要度 | 対象ドキュメント | 対象箇所 | 逆参照の内容 | 問題点 | 修正方針 |

重要度は 高(要件の独立性を損なう重大な逆参照)/ 中(記載粒度・参照方向の問題)/ 低(表現上の軽微な参照・表記ゆれ)。
**指摘が一件も無い場合は、表の代わりに「逆参照なし」とだけ明記してください。** 根拠の無い指摘は挙げないでください。

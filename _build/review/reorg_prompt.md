ルール:
- 指摘は必ず該当する相対パス(例: `CLAUDE.md` / `templates/05_business-usecase.md`)を添える。
- ソースに書かれている事実のみに基づくこと。ソースに無いことは推測で補完せず「ソースに記載なし」と述べる。
- 各観点について、問題があれば「指摘: …(該当パス)」、なければ「問題なし」と明示する。

レビュー観点:

1. テンプレート本文の分離: CLAUDE.md 本体に、各層テンプレートの章立て・骨格・記載例の本文(例「6 セクション固定」「概要 + 13 項目」「## 項目」「アクターは固定3者」などの雛形そのもの)が残っていないか。残っていれば該当箇所を指摘。テンプレートは templates/ 配下に分離されているか。

2. 運用ルール中心の構成: CLAUDE.md が運用ルール(構成・採番・粒度分離・影響調査・トレーサビリティ・ポータル/ナビ・レビュー方針・Agent 実行・PDCA・検証・課題管理・ID/用語)中心の内容になっているか。

3. 粒度分離の明確さ: 「業務要件 → 機能要件・非機能要件 → 業務ユースケース → 基本設計」の役割分担が明確で、層の混在を禁じているか。業務要件・業務ユースケースに 画面ID/API ID/テーブル名/実装詳細 を書かない旨が明記されているか。

4. 逆参照の禁止と整合: 要件が業務UC・基本設計を参照する逆参照を禁じているか。業務ルール RULE の `### 適用UC` に関する例外の扱いが、CLAUDE.md と templates/04_business-rule.md の間で矛盾していないか(整合が取れているか)。

5. ドキュメント間整合(CLAUDE.md ↔ templates): CLAUDE.md の記述と各 templates/ の骨格との間に、矛盾・不要な重複・抜けがないか(例: 相互参照アンカーの正本が一本化されているか、画面モック手順、検証手順、共通記載スタイルの参照関係)。

6. 6 つの運用ルールの網羅: 次が漏れなく定義されているか。(A) ドキュメント粒度分離 (B) 上位設計変更時の影響調査 (C) NotebookLM レビュー活用(スキルを必ず使う旨の明記を含む)(D) 統括Agent + 作業Agent の実行体制 (E) PDCA (F) GitHub Issue 運用(登録ケース・Close 前確認)+ Issue 本文の記載ルール(中学生にも分かる表現・提案必須・概要/背景/影響/提案/確認したいこと/関連ドキュメントの 6 見出し)。

7. 抜け漏れ・改善点: 上記以外で、運用ルールとして抜けている/曖昧な点・改善余地があれば挙げる。

## <span id="2-共通仕様"></span>2. 共通仕様

全 API に共通する形式・認証ヘッダ・冪等性・論理削除フィルタ・Webhook 規約をここで一括定義します。各エンドポイントの個別仕様(§4)は本節を前提とし、差分のみを記載します。

### <span id="31-api-共通仕様82"></span>2.1 API 共通仕様

Base URL・データ形式・ID 形式・ページング方式など、全 API が従う基本ルールです。

| 項目       | 値                                          |
|------------|---------------------------------------------|
| Base URL   | `/api/v1`(管理)/ `/widget/v1`(ウィジェット) |
| データ形式 | JSON                                        |
| エラー形式 | RFC 7807(`application/problem+json`)        |
| 日時       | ISO 8601 + 末尾 Z(UTC)                      |
| ID 形式    | ULID(26 文字)、Stripe ID 例外               |
| ページング | カーソル方式(`cursor`, `limit` 50〜200)     |
| CSRF       | Double Submit Cookie 検証(状態変更 API)     |
| Cookie     | `Secure; HttpOnly; SameSite=Lax; Path=/`    |

### <span id="32-オーナー境界判定"></span>2.2 オーナー境界判定

全操作で契約境界を強制し、違反時は存在を秘匿します。

すべての操作で `actor.contract_id == target.contract_id` を必須とする(正本: 08_認証・認可設計.html §5.2)。違反時は **404 偽装**で応答する(正本: 05_エラー設計.html §5.3)。

### <span id="32a-論理削除フィルタ"></span>2.3 論理削除フィルタ

`valid` カラムを持つテーブルへの GET 系 API は論理削除済みの行を返しません。

`valid` カラムを持つテーブル(`M_USER` / `M_CONTRACT` / `M_PRJ_USERS` / `M_PROJECTS` / `M_ALLOWED_DOMAINS` / `M_FAQS` / `H_QUESTION_LOGS` / `T_INQUIRIES` / `T_BILL_SUBS`)に対する全 GET 系 API(一覧 / 詳細)のクエリは、原則として `WHERE <table>.valid=1` フィルタを追加する。論理削除済み(`valid=0`)の行は通常 API 経由では返却しない。

例外:

- 監査ログ参照系 API は `valid` 概念外(`H_AUDIT_LOGS` には `valid` カラムを追加しない)
- 認証ミドルウェアは、セッションの認証主体(`user_id` → `M_USER`)が無効化済み(`M_USER.valid=0`)ならログイン / 継続を 401 で拒否し、セッションも失効扱いとする。オーナー / メンバーの種別は `M_CONTRACT.user_id` 一致 / `M_PRJ_USERS` の有効割当で導出する

`WHERE valid=1` フィルタの漏れは IDOR 類似の脆弱性として扱う(09_セキュリティ設計.html)。

### <span id="33-冪等性キー"></span>2.4 冪等性キー

書き込み系 API はリトライ安全性のため冪等性キーを受け付けます。

書き込み系 API は `Idempotency-Key` ヘッダ(ULID 推奨、24h 保管)を受け付ける。重複時は既存処理結果を 200 で返却(リトライ安全性)。

### <span id="34-webhook-共通仕様873"></span>2.5 Webhook 共通仕様

受信 Webhook の署名検証・冪等性・リトライ・DLQ の共通方針です。

| 項目         | 仕様                                          |
|--------------|-----------------------------------------------|
| 署名検証     | HMAC-SHA256(送信元との共有 Secret)            |
| 冪等性       | event_id で重複処理防止                       |
| リトライ     | 3 回(指数 BO: 1m → 4m → 16m)、超過で DLQ      |
| DLQ 保持     | 4 日(Cloudflare Queues)、その後 R2 退避 30 日 |
| タイムアウト | 30 秒(超過で 5xx 返却)                        |

### <span id="36-認証ヘッダ"></span>2.6 認証ヘッダ

API カテゴリごとに用いる認証ヘッダの一覧です。

| ヘッダ                          | 用途                   | 例               |
|---------------------------------|------------------------|------------------|
| `Cookie: session=<token>`       | 利用者セッション       | 管理 API         |
| `Authorization: Bearer <wst_*>` | ウィジェットセッション | ウィジェット API |
| `X-CSRF-Token: <token>`         | CSRF 対策              | 状態変更系       |
| `Idempotency-Key: <ULID>`       | 冪等性                 | 書き込み系       |

## <span id="3-レート制限キャパシティ"></span>3. レート制限・キャパシティ

API 層で適用する警告 / 拒否レベルと超過時の挙動の正本です。プロジェクト単位の月次上限・無料枠は 10_課金・請求設計.html が業務正本で、本表は API での enforcement 仕様を示します。

| 項目 | 警告レベル | 拒否レベル | 契約上書き | 超過時の挙動 | 正本 |
|----|----|----|----|----|----|
| 同時アクティブ契約 | 150 でシャーディング着手 | 200 | 不可 | 新規契約受付一時停止 + 通知 | NFR-110 |
| FAQ 件数(契約共通) | 8,000(80%) | 12,000(120%) | 不可 | 拒否レベルで 429 | FR-046 |
| FAQ 質問文字数 | — | 500 文字 | 不可 | 超過は 400 | FR-046 |
| FAQ 回答文字数 | — | 5,000 文字 | 不可 | 超過は 400 | FR-046 |
| 月間質問数(プロジェクト単位) | 80%(通知) | 100%(設定上限件数到達でウィジェット質問受付停止。無料枠超過分は事後課金、**無料枠超過自体は拒否しない**) | プロジェクト別上書き可(`M_PRJ_QUOTA_LIMITS`、SCR-021) | 125% で追加制限 | FR-122 |
| Workers AI 月間コスト | 80% | 100%(停止) | (社内原価指標、契約単位) | 同上 | FR-122 |
| 公開 API `/widget/ask` | — | 60 req/min/IP | — | 429 | FR-128 |
| 公開 API `/widget/bootstrap` | — | 30 req/min/IP | — | 429 | FR-128 |
| ログイン失敗 | — | 5 回/15 分(`IP × user_id` ロック) | 不可 | アカウントロック | FR-007 |
| 管理者ユーザー数 | 80% | 100% | 可(プラン依存) | 429、通知 | FR-021 |
| プロジェクト数 | 平均×3 超過時通知 | — | — | 急増検知通知のみ | FR-035 |
| 同時オンライン管理者 | — | 1,000 名 | — | 503 + Retry-After | NFR-114 |
| ウィジェット同時接続(EU) | — | 10,000 並列 | — | 503 + Retry-After | NFR-114 |
| D1 容量 | 8GB(80%) | 10GB | — | 高優先度アラート + シャーディング検討 | NFR-115 |
| Workers CPU 時間 | 50ms 目標 / 100ms 警告 | — | — | 100ms 超過が 1h で 1% 超で通知 | NFR-116 |

詳細は 10_課金・請求設計.html §8 の選択閾値アラート / 100%停止 / 125%通知を参照。

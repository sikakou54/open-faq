# アクティビティ図(ACT)テンプレート

> **本テンプレートはアクティビティ図(ACT)の記載骨格(業務×システム処理を俯瞰する E2E フロー 1 件 = 1 ファイル)を定義します。**

運用ルールの正本は [../CLAUDE.md](../CLAUDE.md)。共通記載スタイルは [共通記載スタイル](00_common-style.md) を参照する。

- **配置先**: `03_detail_design/09_activities/`(`index.md` + `ACT-001.md`〜)
- **採番**: `ACT-001`〜(E2E フロー 1 件 = 1 ファイル・ゼロ詰め 3 桁・欠番なし・`<span id>` 出現順 001)。ID は各設計書の H1 に `# <span id="ACT-001"></span>ACT-001: 名称` で保持する。

## 骨格(6 セクション固定)

**E2E フロー 1 件 = 1 ファイル(`ACT-NNN`)**。基本設計の業務ユースケース(UC の基本フロー)とシーケンス設計(SEQ)を入力とし、シーケンスより上位の抽象度で「業務処理とシステム処理がどの主体で・どの順で・どの分岐で流れるか」を 1 枚のアクティビティ図として俯瞰する詳細設計層の文書。フロント(Server / Client Components)・バックエンド(Route Handlers / Service 層)・外部サービス(Stripe / Resend 等)の分担をスイムレーンで示す。個々の API 呼び出し・DB 操作の要求応答は SEQ が担うため列挙せず、判定・分岐・状態遷移を伴う主要フローの俯瞰に用いる(**少数の E2E フローに限定**)。6 セクション固定:

1. **目的**(本フローが俯瞰する業務・システム処理と、詳細化元の UC / SEQ との対応を 1〜2 文で示す)
2. **対象範囲**(開始条件 / 終了条件 / 対象ロール / 対象機能 を項目表で示す。加えて業務ユースケースID `UC` / 関連 SEQ / 関連画面 `SCR` / 関連 API・SYS の関連表を置く)
3. **アクティビティ図**(mermaid `flowchart`。フロント / バックエンド / 外部サービスの分担が要るときは `subgraph` でスイムレーンを構成する)
4. **処理フロー一覧**(No / 実行主体 / 処理 / 条件 / 次処理 / 備考)
5. **分岐条件**(分岐 / 条件 / 遷移先 / 備考)
6. **後続工程への引き継ぎ事項**(詳細シーケンス `DSQ`・テスト設計へ渡す観点を箇条書き)

各 `##`/`###` 直後に 1〜2 文のリード文を置いてから表・図へ入る(表・図をいきなり置かない)。

## 記載ルール

- **具体的な設計値(しきい値・単価・無料枠・タイムアウト・保持期間)・状態名の意味は本書で再定義しない。** [システム仕様書](../02_basic_design/07_system-spec.md) / [状態モデル](../02_basic_design/08_state-model.md) / [用語集](../01_requirements/00_glossary.md) へリンクまたは ID 参照で送る(例: 「信頼度しきい値([システム仕様書 §1](../02_basic_design/07_system-spec.md#1-aiしきい値))未満のとき未解決へ回す」)。「等」で打ち切らない。
- **判定条件・分岐条件・取得/更新条件は業務語+条件で書いてよい。** ただし実装コード本文・物理カラム名の羅列・SQL 本文・アルゴリズム詳細・変数名 / JSON キー名は書かない。テーブル参照はリンクで辿り、物理名を本文へ転記しない。個々の API 要求応答・DB 読み書きの往復は SEQ の領分なので図・表に列挙しない(主体と処理の流れを俯瞰する粒度に留める)。
- **層横断トレーサビリティ(`TR-NNN`)は本文に書かない。** §2 の関連表には業務ユースケースID(`UC-NNN`)と設計上必要な関連 ID(関連 `SEQ` / `SCR` / `API` / `SYS`)のみを置く。全層の厳密な紐付けはトレーサビリティ一覧表 [`00_traceability/index.md`](../02_basic_design/00_traceability/index.md) に一元管理する。要件 ID(`FR` / `BR` / `NFR` / `RULE`)は設計根拠の括弧引用としてのみ許容し、逆引きの対応表・専用欄は作らない。
- **mermaid 図は描画可能であること。** コードフェンスの開き(` ```mermaid `)・閉じ(` ``` `)は必ず独立行に置き、後続の文・要素を同一行へ連結しない。ノードラベルに丸括弧等の特殊文字を含めるときは `"..."` で囲む。**絵文字は使わない。** スイムレーンは `subgraph フロント` / `subgraph バックエンド` / `subgraph 外部サービス` 等で構成し、レーン名は実装スタック(Server / Client Components・Route Handlers / Service 層・Stripe / Resend)ではなく業務主体で示す(実装写像は §4 の「実行主体」欄で補う)。
- 相互参照アンカーの記法・相対パスは [共通記載スタイル](00_common-style.md) の相互参照アンカー例に従う(`03_detail_design/09_activities/` からの深さは `../02_basic_design/...` / `../01_requirements/...` / `../CLAUDE.md`)。

## 記載例(セクション骨格)

```markdown
# <span id="ACT-001"></span>ACT-001: 利用者が質問して AI 回答を得る

> **本アクティビティ図は「利用者の質問受付から AI 回答提示・未解決振り分けまで」の業務×システム処理を俯瞰します。**

*種別 アクティビティ図 ・ ステータス ドラフト*

## 1. 目的

本フローが俯瞰する業務・システム処理と、詳細化元の業務ユースケース([UC-030](../../01_requirements/04_business_usecases/UC-030.md#UC-030))・シーケンス([SEQ-040](../../02_basic_design/03_sequences/SEQ-040.md#SEQ-040))との対応を 1〜2 文で示す。

## 2. 対象範囲

本フローの開始・終了条件と対象ロール・対象機能を示す。

| 項目 | 値 |
|----|----|
| 開始条件 | 利用者がウィジェットで質問を送信したとき |
| 終了条件 | AI 回答を提示、または未解決として振り分けたとき |
| 対象ロール | エンドユーザー(サイト訪問者) |
| 対象機能 | AI 回答生成・提示可否判定・未解決振り分け |

関連:

| 区分 | 参照 |
|----|----|
| 業務ユースケースID | [UC-030](../../01_requirements/04_business_usecases/UC-030.md#UC-030) |
| 関連 SEQ | [SEQ-040](../../02_basic_design/03_sequences/SEQ-040.md#SEQ-040) |
| 関連画面 | [SCR-018](../../02_basic_design/01_frontend/01_screens/SCR-018.md#SCR-018) |
| 関連 API / SYS | [API-030](../../02_basic_design/02_backend/03_apis/API-030.md#API-030) |

## 3. アクティビティ図

業務主体ごとのスイムレーンで、開始から終了までの処理と分岐を俯瞰する。

\`\`\`mermaid
flowchart TB
    subgraph フロント
        A([質問を送信]) --> B[受付結果を表示]
    end
    subgraph バックエンド
        C[入力を検証] --> D{"提示可否を判定"}
        D -->|提示可| E[AI 回答を確定]
        D -->|提示不可 / 判定不能| F[未解決として振り分け]
    end
    subgraph 外部サービス
        G[回答候補を生成]
    end
    A --> C
    C --> G
    G --> D
    E --> B
    F --> B
\`\`\`

## 4. 処理フロー一覧

図の各処理を実行順に、実行主体と次処理・条件とともに一覧化する。

| No | 実行主体 | 処理 | 条件 | 次処理 | 備考 |
|----|----|----|----|----|----|
| 1 | 利用者(Client Component) | 質問を送信する | — | 2 | — |
| 2 | サーバー(Route Handler / Service 層) | 入力を検証し回答候補生成を要求する | 検証 OK | 3 | 検証 NG は受付を中止し案内 |
| 3 | 外部サービス(AI) | 回答候補を生成する | — | 4 | 個々の要求応答は [SEQ-040](../../02_basic_design/03_sequences/SEQ-040.md#SEQ-040) を参照 |
| 4 | サーバー(Service 層) | 提示可否を判定する | しきい値([システム仕様書 §1](../../02_basic_design/07_system-spec.md#1-aiしきい値))と比較 | 5 / 6 | 分岐は §5 |
| 5 | サーバー(Service 層) | AI 回答を確定して返す | 提示可 | 終了 | — |
| 6 | サーバー(Service 層) | 未解決として振り分ける | 提示不可 / 判定不能 | 終了 | — |

## 5. 分岐条件

図中の分岐ノードごとに、遷移先を決める条件を示す。

| 分岐 | 条件 | 遷移先 | 備考 |
|----|----|----|----|
| 提示可否判定 | 信頼度・関連度がしきい値以上 | AI 回答を確定(No.5) | しきい値の正本は [システム仕様書 §1](../../02_basic_design/07_system-spec.md#1-aiしきい値) |
| 提示可否判定 | いずれか未満 / スコア欠損 | 未解決として振り分け(No.6) | 判定不能も未解決扱い |

## 6. 後続工程への引き継ぎ事項

詳細シーケンス(DSQ)・テスト設計へ渡す確認観点を箇条書きで示す。

- 分岐の境界値(しきい値の等号を含むか)と判定不能(スコア欠損)の扱いの確認。
- 外部サービス応答遅延・失敗時のフロー(タイムアウトは正本を参照)と未解決振り分けの整合。
```

## テンプレ運用の注記

本節はテンプレの使い方ガイドであり、**実ドキュメントには転記しない。**

- **記載しない項目**: 実装コード本文・SQL 本文・物理カラム名の羅列・アルゴリズム詳細・変数名 / JSON キー名。個々の API 要求応答・DB 読み書きの往復(SEQ の領分)。層横断トレーサビリティ(`TR-NNN`)と要件 ID の逆引き対応表。正本で定義済みの具体値(しきい値・単価・保持期間・タイムアウト・状態名の意味)の再定義。「等」での打ち切り。
- **基本設計との責務分担**: シーケンス設計(SEQ)が UC 単位で「どの主体が・どの API / DB とどんな要求応答を交わすか」を時系列で詳細に示すのに対し、ACT はそれより上位の抽象度で「業務処理とシステム処理が主体をまたいでどう流れ、どこで分岐・状態遷移するか」を 1 枚のアクティビティ図で俯瞰する。SEQ・UC で定義済みの業務意図・分岐と齟齬を生じさせず、§2 の関連表で対応 `UC` / `SEQ` を明示する。基本設計に無い仕様は推測で足さず、不明点は GitHub Issue で課題化する。適用対象は主要な少数の E2E フロー(申込〜開始・質問受付〜回答・請求確定〜サスペンション等)に限定し、全 UC を機械的に 1 対 1 で起こさない。
- **後続テスト設計への引き継ぎ観点**: §6 に、分岐の境界値・例外系・外部サービス失敗時のフロー・状態遷移の前後条件など、後続の詳細シーケンス(DSQ)・テスト設計が処理条件と成否条件を一意に読み取れる観点を残す。
```

# 詳細シーケンス図(DSQ)テンプレート

> **本テンプレートは詳細シーケンス図(DSQ)の記載骨格(内部コンポーネント連携・トランザクション境界・異常分岐を伴う複雑フロー 1 件 = 1 ファイル)を定義します。**

運用ルールの正本は [../CLAUDE.md](../CLAUDE.md)。共通記載スタイルは [共通記載スタイル](00_common-style.md) を参照する。

- **配置先**: `03_detail_design/08_sequences/`(`index.md` + `DSQ-001.md`〜)
- **採番**: `DSQ-001`〜(1 フロー = 1 ファイル・ゼロ詰め 3 桁・欠番なし・`<span id>` 出現順に `001` から採番)。ID は各設計書の H1 に `# <span id="DSQ-001"></span>DSQ-001: 名称` で保持する。

## 骨格(6 セクション固定)

**1 フロー = 1 ファイル(`DSQ-NNN`)**。基本設計のシーケンス設計([SEQ](12_sequence.md)。ユーザー / 画面 / サーバー / DB の抽象度)を、実装スタック(TypeScript + Next.js App Router + Service 層 → Repository 層)と実行基盤(Cloudflare Workers/Pages + D1 + Queues + Cron Triggers)の内部コンポーネント連携へ写像して詳細化する。基本設計 SEQ が「利用者・システムの流れ」を示すのに対し、DSQ は Route Handler / Service / Repository / D1 / External(Stripe / Resend 等)の連携順・トランザクション境界・リトライ / ロールバック・異常分岐を具体化する。6 セクション固定:

1. **目的**(本フローを詳細化する狙いと、詳細化元のシーケンス設計 `SEQ` との対応を 1〜2 文で示す)
2. **前提条件**(利用者 / 開始条件 / 前提状態 / 対象画面 / 対象 API / 対象 DB / 外部 IF を表で示す)
3. **正常系シーケンス**(mermaid `sequenceDiagram`。participant は内部コンポーネント。トランザクション境界を `Note` で注記)
4. **処理詳細**(No / 実行主体 / 処理内容 / 入力 / 出力 / 分岐・条件 / エラー時)
5. **異常系・例外系**(No / 発生箇所 / 発生条件 / エラー内容(`ERR` ID)/ 表示メッセージ(`MSG` ID)/ 後続処理)
6. **後続工程への引き継ぎ事項**(テスト設計・詳細ロジック設計へ渡す観点を箇条書き)

各 `##`/`###` 直後に 1〜2 文のリード文を置いてから表・図へ入る(表・図をいきなり置かない)。導出できない欄は `要確認`、該当なしは `—`(全角ダッシュ)。

## シーケンス図ルール

- **participant は内部コンポーネント**を用いる。基本設計 SEQ の `画面 / サーバー / DB` を詳細化し、`Client`(Server/Client Component) / `Route Handler`(`app/api/**/route.ts`) / `Service`(Service 層) / `Repository`(Repository 層) / `D1`(D1) / `External`(Stripe / Resend 等の連携先。具体名は EIF を参照)を分けて描く。非同期・バッチ起点は `Queue`(Cloudflare Queues) / `Cron`(Cron Triggers) / `Worker`(Webhook 受信 Worker)を参加者に加える。
- **トランザクション境界を `Note` で注記する。** `Note over Service,D1: トランザクション開始` / `Note over Service,D1: コミット` のように、どの範囲が 1 トランザクションかを図中に明示する。ロールバック契機は §5 と対応づける。
- **リトライ・ロールバック・異常分岐は `alt/else/opt/loop` で描く。** 基本設計 SEQ で `## 詳細設計への移管候補` へ逃がした行単位処理・冪等性・楽観ロック・部分失敗の分岐を、DSQ では図と §4 / §5 に具体化する。
- `autonumber` を付け、要求に応答線を対で描く。**図中に SQL・ORM クエリ・物理カラム名・変数名 / JSON キー名・関数 / メソッド名・アルゴリズム内部実装・Markdown リンク・`id=` は書かない**(検証スクリプトのため)。判定条件は業務語+条件式(自然言語)で示す。
- コードフェンスの開き(` ```mermaid `)・閉じ(` ``` `)は必ず独立行に置き、ノードラベルに丸括弧等の特殊文字を含めるときは `"..."` で囲む。**絵文字は使わない。**

## 記載ルール

- **本テンプレートは基本設計 SEQ(ユーザー / 画面 / サーバー / DB の抽象度)と重複させない。** 内部コンポーネント連携・トランザクション境界・リトライ / ロールバック・異常分岐を追加する複雑 / 高リスクフロー(月次請求確定・Stripe / Resend 連携・非同期投入と消費・部分失敗を伴う一括処理など)に限定して起こす。単純な 1 要求 1 応答フローは DSQ を起こさず基本設計 SEQ に留める。
- **具体的な設計値・状態名の意味を本書で再定義しない。** しきい値・単価・無料枠・タイムアウト・リトライ上限・再送遅延・保持期間は [システム仕様書](../02_basic_design/07_system-spec.md)、状態名の意味は [状態モデル](../02_basic_design/08_state-model.md)、用語は [用語集](../01_requirements/00_glossary.md) へ ID 参照 / リンクする(例:「再送遅延([NFR-040](../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040))」)。「等」で打ち切らない。
- **エラー内容・表示メッセージの文面を本書に書かない。** §5 の「エラー内容」は [エラー定義(ERR)](../02_basic_design/05_errors/ERR-001.md#ERR-001) の ID、「表示メッセージ」は [メッセージ(MSG)](../02_basic_design/06_messages/) の ID または画面表示なら SCR §8 の `EM-NN` を参照する(文面を重複させない)。
- **層横断トレーサビリティ(`TR-NNN`)は本文に書かない。** §2 前提条件には業務ユースケースID(`UC-NNN`)と設計上必要な関連 ID(対象 `SCR` / `API` / `TBL`、詳細化元 `SEQ`、参照 `IPO` / `EIF`)のみを置く。全層の厳密な紐付けはトレーサビリティ一覧表 [`00_traceability/index.md`](../02_basic_design/00_traceability/index.md) に一元管理する。要件 ID(`FR` / `BR` / `NFR` / `RULE`)は設計根拠の括弧引用としてのみ許容し、逆引きの対応表・専用欄は作らない。
- **実装コード本文・SQL 本文・物理カラム名の羅列・アルゴリズム内部実装は書かない。** 判定条件・取得 / 更新条件・疑似コード(自然言語の条件式)・冪等キーの構成方針は書いてよい。テーブルとの対応はリンクで辿る(物理名を本文へ転記しない。物理カラム定義は [DB物理設計(DBP)](11_database.md) の本務)。
- 相互参照アンカーの記法・相対パスは [共通記載スタイル](00_common-style.md) の相互参照アンカー例に従う(`03_detail_design/08_sequences/` からの深さは `../02_basic_design/...` / `../01_requirements/...` / `../CLAUDE.md`)。

## 3. 正常系シーケンス

正常完了までの内部コンポーネント連携をトランザクション境界とともに示す。

\`\`\`mermaid
sequenceDiagram
    autonumber
    participant Cron as Cron Triggers
    participant Service as 請求確定サービス
    participant Repo as Repository
    participant D1 as D1
    participant Stripe as External(Stripe)
    Cron->>Service: 月次締めを起動
    Service->>Repo: 対象課金アカウントの当月利用量を取得
    Repo->>D1: 利用量を照会
    D1-->>Repo: 利用量
    Repo-->>Service: 利用量
    Note over Service,D1: トランザクション開始
    Service->>Repo: 請求確定レコードを作成(冪等キーで重複防止)
    Repo->>D1: 請求確定を登録
    D1-->>Repo: 登録完了
    Note over Service,D1: コミット
    Service->>Stripe: 請求を作成・確定
    Stripe-->>Service: 請求結果
    alt 請求成功
        Service->>Repo: 請求状態を確定へ更新
        Repo->>D1: 状態を更新
        D1-->>Repo: 更新完了
    else 請求失敗
        Service->>Service: 失敗として記録し再試行対象へ回す(上限は NFR を参照)
    end
\`\`\`

## テンプレ運用の注記(実ドキュメントには転記しない)

本節はテンプレの使い方ガイドであり、以下の内容を実際の `DSQ-NNN.md` へ転記しない。

- **記載しない項目**: 実装コード本文・SQL 本文・ORM クエリ・物理カラム名の羅列・変数名 / JSON キー名・関数 / メソッド名・アルゴリズム内部実装。層横断トレーサビリティ(`TR-NNN`)と要件 ID の逆引き専用欄。正本で定義済みの具体値(しきい値・単価・無料枠・タイムアウト・リトライ上限・再送遅延・保持期間)と状態名の意味の再定義。エラー・メッセージ文面の再掲(ERR / MSG / SCR §8 の `EM-NN` へ参照)。「等」での打ち切り。
- **基本設計との責務分担**: 基本設計のシーケンス設計(SEQ)は「利用者・システムの流れ」をユーザー / 画面 / サーバー / DB の抽象度で示す。DSQ はその詳細設計として、同一フローを Route Handler / Service / Repository / D1 / External(Stripe / Resend 等) / Queue / Cron の内部コンポーネント連携へ写像し、トランザクション境界・リトライ / ロールバック・異常分岐を具体化する。DSQ を起こすのは、内部連携・トランザクション・部分失敗の分岐を追加する必要がある複雑 / 高リスクフローに限定し、単純フローは SEQ に留める(重複させない)。処理ロジックそのものは IPO、外部連携仕様は EIF、物理カラム定義は DBP が担い、DSQ はそれらを連携順の中で参照する。基本設計に無い仕様は推測で足さず GitHub Issue で課題化する。
- **後続テスト設計への引き継ぎ観点**: §6 に、トランザクション境界の部分コミット・ロールバック、冪等な再実行、多重起動・並行実行の競合、外部 IF 失敗 / タイムアウト時のリトライと滞留アラートの契機、異常分岐(§5)の網羅など、後続の詳細ロジック設計(IPO)・テスト設計が成否条件を一意に読み取れる観点を残す。

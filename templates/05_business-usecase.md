# 業務ユースケース(UC)テンプレート

> **本テンプレートは業務ユースケース(UC)の記載骨格(概要 + 13 項目)を定義します。**

運用ルールの正本は [../CLAUDE.md](../CLAUDE.md)。共通記載スタイルは [共通記載スタイル](00_common-style.md) を参照する。

- **配置先**: `01_requirements/04_business_usecases/`
- **採番**: `UC-001`〜(フラット連番・ゼロ詰め 3 桁・欠番なし。画面起点 → システム起点 → 要件起点の順)

## 骨格

**1 UC = 1 ファイル**。**要件定義ベースの業務処理単位**(「誰が・何の目的で・どの業務を・どのような流れで実行するか」)で記述する。前半が画面起点(利用者操作)、続いてシステム起点(バッチ・Webhook・非同期・通知)、末尾が要件起点(横断方針)。UC 名は **「誰が〜する」**(主アクター先頭)の業務動作で表す。骨格は **概要 + 11 項目**:

1. 業務ユースケースID(`# <span id="UC-NNN"></span>UC-NNN: 名称`)
2. 業務ユースケース名
3. 主アクター
4. 目的
5. 事前条件(システム起点はトリガー)
6. 基本フロー
7. 代替フロー
8. 例外フロー
9. 事後条件
10. トレーサビリティ(`TR`)
11. 備考

冒頭にタイトル直後の要約(`> **…**`)とメタ行、`## 概要` を置く。

> [!IMPORTANT]
> **UC 本文には基本設計レベルの情報を書かない。** 画面ID(`SCR`)/ 画面イベントID(`EVT`)/ API ID(`API`)/ テーブルID(`TBL`)/ シーケンスID(`SEQ`)/ エラーID(`ERR`)/ メッセージID(`MSG`)、画面名・ボタン名・フォーム名・入力項目名、API名 / エンドポイント / メソッド / ステータスコード、テーブル名・物理名・カラム名・SQL、クラス / 関数 / メソッド名、mermaid 図は **記載しない**。これらは業務語へ言い換えて除去する。

## 記載ルール

- トレースは **末尾 `## トレーサビリティ` 節に `TR-NNN` 1 件のみ本文保持**(`UC-NNN ↔ TR-NNN` の 1 対 1)。対応する要件(BR / FR / RULE)も実現する設計(画面・システム・API・テーブル・シーケンス。画面イベント `EVT` / システムイベント `SEV` を含む)も UC 本文には列挙せず、**トレーサビリティ一覧表 [`02_basic_design/00_traceability/index.md`](../../02_basic_design/00_traceability/index.md) の当該 TR 行で一元管理**する(UC 本文に `対応する業務要件ID` / `関連画面ID` 等は持たない)。
- 基本フローは業務粒度の「誰が / システムが 〜する」で 4〜8 ステップ。UI 操作の列挙にしない。詳細シーケンスは持たず SEQ 層へトレースする。
- 粒度: 1 画面イベント = 1 UC のような操作粒度に**しない**(統合する)。「FAQを管理する」のような umbrella にも**しない**(登録/編集/削除/公開… へ分割する)。
- トレーサビリティ規約・粒度の詳細ルールは [../CLAUDE.md](../CLAUDE.md) を参照。

# 外部インターフェース設計図(EIF)テンプレート

> **本テンプレートは外部インターフェース設計図(EIF)の記載骨格(連携先サービス単位 1 ファイル)を定義します。**

運用ルールの正本は [../CLAUDE.md](../CLAUDE.md)。共通記載スタイルは [共通記載スタイル](00_common-style.md) を参照する。

- **配置先**: `03_detail_design/06_external_if/`
- **採番**: `EIF-001`〜(連携先サービス単位 1 ファイル・ゼロ詰め 3 桁・欠番なし・`<span id>` 出現順に採番)。ID は各設計書の H1 に `# <span id="EIF-001"></span>EIF-001: 名称` で保持する。

## 骨格(7 セクション固定)

**連携先サービス単位 1 ファイル(`EIF-NNN`)**。基本設計の外部連携 API(課金プロバイダ Stripe / メール配信 Resend / AI 推論)を、実装者が連携方向・送受信項目・署名検証・タイムアウト・冪等・リトライまで迷わず実装できる粒度へ具体化する。骨格:

- H1 アンカー `# <span id="EIF-NNN"></span>EIF-NNN: <連携先サービス名>`
- ページ要約(引用ブロック)＋ 文書メタ(`*種別 外部インターフェース設計 ・ ステータス ドラフト*`)
- `## 項目`(EIF ID / 業務ユースケースID `UC-NNN` / 関連 API `API` / 関連 SYS / 関連テーブル `TBL` / 関連エラー `ERR` / 関連メッセージ `MSG`)。全層の厳密な紐付けはトレーサビリティ一覧表で一元管理し、EIF 本文に `TR-NNN` は記載しない。
- `## 1. 目的`
- `## 2. 連携概要`
- `## 3. 連携図`(mermaid `flowchart LR`)
- `## 4. 送信項目`(自システム → 外部)
- `## 5. 受信項目`(外部 → 自システム)
- `## 6. 例外処理`
- `## 7. 後続工程への引き継ぎ事項`

導出できない欄は `要確認`、該当なしは `—`。

## 記載ルール

- 各 `##`/`###` 直後に 1〜2 文のリード文を置いてから表へ入る(表をいきなり置かない)。
- **設計値(AI 推論タイムアウト・Webhook 受信履歴保持・冪等性キー保持・決済失敗の猶予・レート制限など)は本文で再定義せず、正本へリンク**する。タイムアウト・保持期間は [システム仕様書](../02_basic_design/07_system-spec.md)(例 [`AI 推論タイムアウト`](../02_basic_design/07_system-spec.md#3-タイムアウトセッション認証))、状態名は [状態モデル](../02_basic_design/08_state-model.md)、用語は [用語集](../01_requirements/00_glossary.md) を参照する。**「等」で打ち切らない**(取りうる値・イベント種別は全集合を列挙する)。
- **秘密情報の実値は書かない。** API キー・署名シークレット・エンドポイント URL の実値・Bearer トークン等は記載せず、「署名シークレット(環境変数で注入)」「送信元 API キー(秘匿)」のように参照名・注入方法のみを示す。
- 署名検証は方式(HMAC-SHA256 等)・検証対象ヘッダ(`Stripe-Signature` / `Svix-Signature` 等)・失敗時の HTTP ステータスと対応 `ERR` を明記する。実装コード本文・具体的な鍵導出アルゴリズムは書かない。
- 冪等性は重複排除キー(プロバイダ側イベント ID 等)と冪等応答の扱いを判定条件で示す。リトライはバックオフ回数・間隔の正本([システム仕様書](../02_basic_design/07_system-spec.md) の通知再送設定等)を参照する。
- **要件ID(`FR` / `BR` / `NFR` / `RULE`)は本文へ列挙しない**(業務的文脈は `UC-NNN` で示し、設計根拠の括弧引用に留める)。相互参照は [共通記載スタイル](00_common-style.md) の相互参照アンカー例に従う。
- mermaid はコードフェンスの開き(` ```mermaid `)・閉じ(` ``` `)を独立行に置き、ノードラベルに括弧等を含める場合は `"..."` で囲む。**絵文字は使わない。**

## 記載例(セクション骨格)

```markdown
# <span id="EIF-001"></span>EIF-001: 課金プロバイダ(Stripe)連携

> **本設計は「課金プロバイダ(Stripe)との決済・課金状態連携」の外部インターフェースを定義します。**

*種別 外部インターフェース設計 ・ ステータス ドラフト*

## 項目

| 項目 | 値 |
|----|----|
| EIF ID | EIF-001 |
| 業務ユースケースID | [UC-056](../../01_requirements/04_business_usecases/UC-056.md#UC-056) |
| 関連 API | [API-060](../../02_basic_design/02_backend/03_apis/API-060.md#API-060) |
| 関連 SYS | [SYS-004](../../02_basic_design/02_backend/01_system/SYS-004.md#SYS-004) |
| 関連テーブル | [TBL-032](../../02_basic_design/02_backend/04_database/TBL-032.md#TBL-032) |
| 関連エラー | [ERR-031](../../02_basic_design/05_errors/ERR-031.md#ERR-031) ・ [ERR-032](../../02_basic_design/05_errors/ERR-032.md#ERR-032) |
| 関連メッセージ | — |

## 1. 目的

本連携が何のために・どの業務(`UC-NNN`)を支えるために外部サービスとつながるかを 1〜2 文で示す。

## 2. 連携概要

連携先と連携の性質を一覧で示す。値の正本は各リンク先を参照する。

| 連携先 | 連携方向 | プロトコル | 連携タイミング | 認証方式 | セキュリティ要件 |
|----|----|----|----|----|----|
| Stripe(課金プロバイダ) | 双方向 | HTTPS / Webhook(POST) | 決済発生時・課金状態変更時 | 送信=API キー(秘匿) / 受信=署名検証(HMAC-SHA256) | TLS 必須・署名検証・重複排除 |

## 3. 連携図

自システムと外部サービス間のデータの流れを示す。

\`\`\`mermaid
flowchart LR
    S["自システム(Workers)"]
    EXT["Stripe(課金プロバイダ)"]
    S -->|"決済・購読操作を送信"| EXT
    EXT -->|"Webhook で状態通知を受信"| S
\`\`\`

## 4. 送信項目

自システムから外部サービスへ送信する項目を定義する(具体値・秘密情報は書かない)。

| 項目名 | データ型 | 必須 | 説明 | 備考 |
|----|----|----|----|----|
| 冪等キー | string | ◯ | 重複送信を排除するキー | 保持期間は [システム仕様書](../02_basic_design/07_system-spec.md#4-データ保持期間削除猶予) を参照 |

## 5. 受信項目

外部サービスから受信する項目を定義する。取りうるイベント種別は全集合を列挙する。

| 項目名 | データ型 | 必須 | 説明 | 備考 |
|----|----|----|----|----|
| イベント種別 | enum | ◯ | 通知イベント種別。全集合は [API-060 §列挙値](../../02_basic_design/02_backend/03_apis/API-060.md#API-060) を参照 | — |
| プロバイダ側イベント ID | string | ◯ | 重複排除キー | 冪等応答の判定に使用 |

## 6. 例外処理

タイムアウト・署名検証失敗・重複再送などの例外条件と自システムの処理を定義する。

| 発生条件 | 自システムの処理 | リトライ | 通知 | 備考 |
|----|----|----|----|----|
| 署名検証失敗 | 取り込まず 401 を返す | 不可 | 不正受信として記録 | 失敗時 [ERR-031](../../02_basic_design/05_errors/ERR-031.md#ERR-031) |
| 重複再送(同一イベント ID) | 冪等に受領応答(状態変更なし) | 不要 | — | [ERR-032](../../02_basic_design/05_errors/ERR-032.md#ERR-032) |
| 送信タイムアウト | 打ち切り・失敗として記録 | バックオフ再送 | 取込失敗を記録 | タイムアウト値は [システム仕様書](../02_basic_design/07_system-spec.md#3-タイムアウトセッション認証) を参照 |

## 7. 後続工程への引き継ぎ事項

実装・テスト設計へ引き継ぐ観点(署名検証の検証データ、冪等キーの生成規則、リトライ上限到達時の扱い等)を示す。詳細設計で確定できない事項は課題として分離する。
```

## テンプレ運用の注記

本注記はテンプレの使い方ガイドであり、実ドキュメントには転記しない。

- **記載しない項目**: API キー・署名シークレット・エンドポイント URL の実値、実装コード本文、鍵導出アルゴリズム、`TR-NNN`、要件ID(`FR` / `BR` / `NFR` / `RULE`)の列挙。設計値(タイムアウト・保持期間・単価・状態名の意味)は正本へリンクし本文で再定義しない。
- **基本設計との責務分担**: 基本設計(外部連携 API = Stripe `API-060` / Resend `API-058`・`API-059` / AI `API-057`・`API-038`)は「どの外部サービスと・どのイベントで連携するか」を定義する。EIF は同じ連携を「送受信項目・署名検証手順・タイムアウト・冪等・リトライ・例外時の自システムの処理」まで実装可能粒度へ具体化する。連携先 3 種(Stripe / Resend / AI)は署名検証(Webhook 受信の HMAC-SHA256)・タイムアウト([システム仕様書](../02_basic_design/07_system-spec.md))・冪等・リトライを必ず明記する。
- **後続テスト設計への引き継ぎ観点**: 署名検証(正当・改ざん・期限切れ)、冪等再送(同一イベント ID の重複)、タイムアウト時の打ち切りとリトライ上限到達時の扱い、順序逆転・遅延到達時の整合性を §7 に引き継ぐ。
```

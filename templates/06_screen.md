# 画面設計(SCR)テンプレート

> **本テンプレートは画面設計(SCR)の記載骨格(8 セクション固定)を定義します。**

運用ルールの正本は [../CLAUDE.md](../CLAUDE.md)。共通記載スタイルは [共通記載スタイル](00_common-style.md) を参照する。

- **配置先**: `02_basic_design/01_frontend/01_screens/`(画面モックは同階層配下の `mocks/`)
- **採番**: `SCR-001`〜(1 画面 = 1 ファイル(フラット。`-NNN` 従属は廃止)・ゼロ詰め 3 桁・欠番なし)

## 骨格(8 セクション固定)

**1 画面 = 1 ファイル(フラット `SCR-NNN`)**。8 セクション固定:

1. **画面概要**(`## 1.` 見出しの前(ファイル冒頭)に 画面 ID 表(画面ID / 画面名 / トレーサビリティID(`TR-NNN`。複数は ` ・ ` 区切り))と ステークホルダ表 を置く。続けて `## 1. 画面概要` 見出し＋本画面の目的を **箇条書き(3〜6 点)** で示す。**H1 タイトル・要約ブロックは付けない**)
2. **画面遷移図**(mermaid flowchart)
3. **画面レイアウト**(モック = **PNG 画像のみ**。HTML は `mocks/` に外出し)
4. **画面項目**(列: `#`(ページ内連番) / 項目 / 種類(HTML タグ。入力は `input(email)` / `input(password)` と型を併記、表示専用テキストは `label`、ほか `button` / `link` / `alert` / `widget`) / 必須(`◯` / `—`) / 最大長 / 初期値 / 表示条件。該当なしは `—`。各項目は `#N` で参照する)。**選択肢・状態値など値のパターンを持つ項目があるときは、画面項目表の直下に「データパターン」表を置く**(列: 画面項目(`#N`) / 表示名 / 補足。**利用者に見える選択肢・状態を業務語の表示名で列挙**し、`active` 等の内部コード値・実装値は書かない。本表が値パターンの正本。インライン箇条書きでは書かない)
5. **バリデーション**(入力検証の正本。列: 画面項目(§4 の `#N`) / タイミング(入力時・送信時 等) / ルール(`XXXXチェック` 粒度) / エラーコード(§8 の `EM-NN` を参照。メッセージ文面は §8 が正本で重複させない))
6. **イベント**(列に **EVT 列**(`<span id="EVT-NN"></span>EVT-NN` の非リンクアンカー。**EVT は画面ごとローカル採番**(各画面で `EVT-01` から・ゼロ詰め 2 桁・欠番なし・§6 出現順)。EVT は独立ページ・フォルダを持たず、この §6 にアンカーとして保持する)/ 画面項目(§4 の `#N`、無ければ `—`)/ イベント。処理は §7 で定義)
7. **画面イベント詳細**(列: EVT-ID(§6 と対応・プレーン表記)/ 処理。**基本設計の粒度で、イベントの業務的な結果**(どの画面へ遷移する・何を表示/保存する・成功/失敗で何が起きる)を記す。処理が分岐する場合は罫線ツリー `┣ ┃ ┗` を `<pre>` で記載。エラー表示は §8 の `EM-NN` で参照。**内部の挙動は書かない**(下記「記載ルール」参照))
8. **エラーメッセージ**(本画面が表示するエラー・警告メッセージの正本。列: エラーコード(`EM-NN`、ページ内ローカル連番) / エラーメッセージ。§5・§7 は `EM-NN` で参照し、文面を重複させない)

## 記載ルール

- **8 セクションの骨格・各セクションの説明(リード文)・表の列構成はこのテンプレートが正本。** 骨格・列・説明文を勝手に増減・改変しない。**定義された表・セクションの外側に、箇条書き・補足・コメント・`> [!NOTE]` などを一切書かない**(フォーマット外記述の禁止)。書きたい内容の置き場所が無い・判断に迷う場合は、勝手に追記せず **GitHub Issue で課題化** する。
- 各 `##` 直後のリード文は 1 文程度の定型に留める(表をいきなり置かない)。リード文に仕様・補足・例外を盛り込まない(仕様は各表の行で表現する)。
- **§1 画面概要は箇条書き(3〜6 点)。** 長い散文にしない。1 点 = 1 文を目安に、目的・対象ロール差・主要な表示状態を簡潔に挙げる。**画面間の遷移フローは §2 画面遷移図が示すので §1 で散文化しない**(他画面と分担する固有の関心事のみ書く)。
- **別セクションを見れば分かる内容を、別のセクションで重複記載しない。** 特に §2 画面遷移図が示す遷移フロー(「成功時は◯◯へ遷移し、条件Aなら△△へ割込み遷移する」等)を §1 概要・§7 等で散文として再記述しない(各 §7 イベントの「◯◯へ遷移する」という結果は書いてよいが、フロー全体の言い換えはしない)。§4 の項目・表示条件、§5 の検証ルール、§8 のメッセージ文面も、定義済みセクション以外で繰り返さない。
- **詳細設計粒度を書かない。** 変数名・物理カラム名・JSON キー名・アルゴリズム・コードポイント等の実装詳細は画面設計に書かない。値・状態は具体値と業務語で表す。API・テーブルとの対応はリンクで辿る(物理名を本文へ転記しない)。
- **§7 画面イベント詳細は基本設計の粒度に留める。** 各イベントは「業務的な結果」(どの画面へ遷移する・**どの API を呼ぶか**・何を表示/保存する・成功/失敗で何が起きる)を書く。**呼び出す API は記載する**(API 設計へのリンクで示す。どの API を呼ぶかは業務結果の一部)。一方、**画面が内部で管理する変数・状態は書かない**(=詳細設計の領分): 選択状態・絞り込み/並び替え状態の保持や引継ぎ、画面内フラグ、クエリ/パラメータの構築・受け渡し、API 呼び出しの順序・手続き、処理ステップの列挙。例) 「現在の期間・絞り込みを引き継ぎ、未解決で絞り込んだ状態で要対応の質問一覧へ遷移する」→ **「要対応の質問一覧(SCR-006)へ遷移する」**(遷移先という業務結果のみ)。「成功分のみ反映し失敗分の行は選択状態を保持する」→ **選択状態の保持は書かない**(画面内部状態)。
- 画面設計書は **H1 タイトル・要約ブロック・メタ行を付けない**。ファイル冒頭に 画面 ID 表(画面ID / 画面名 / トレーサビリティID)+ ステークホルダ表 を置き、続けて `## 1. 画面概要` から始める(共通記載スタイルのタイトル / 要約 / メタは SCR には適用しない)。
- §3 画面レイアウトは **PNG 画像で表示**する(MD は画像 1 行のみ)。HTML ソースは `mocks/<画面>-<n>.html` に外出しし、各 SCR からは `mocks/<画面>-<n>.png`(子階層相対・`../` なし)で参照する。画面モックの画像化手順は [../CLAUDE.md](../CLAUDE.md) の「画面モックの画像化」を参照。
- §1 の `トレーサビリティID` 行が本画面の TR 結線。UC ↔ 設計の対応はトレーサビリティ一覧表 [`00_traceability/index.md`](../00_traceability/index.md) の当該 TR 行で一元管理する(SCR 本文に `対応業務UC` / `関連画面` 等は持たない)。API 等へのリンクは [共通記載スタイル](00_common-style.md) の相互参照アンカー例に従う。
- 入力検証の正本は §5 バリデーション。**入力のみが目的のイベント(検証だけを行うイベント)は §6 イベント / §7 画面イベント詳細に重複記載しない**(検証は §5 に集約する)。
- **EVT-ID は画面ごとにローカル採番**(各画面で `EVT-01` から・ゼロ詰め 2 桁・欠番なし)。§6 で `<span id="EVT-NN">` を定義し、§7 はプレーン `EVT-NN` で参照する。**他層(API / ERR / PERM / SEQ / MSG)から参照するときは画面 ID を冠して `SCR-NNN EVT-MM` と書く**(画面ローカル ID 単独では参照しない)。
- §7 で処理が複数の結果・条件に分岐する場合は、罫線ツリー(`┣` 中間 / `┃` 縦線 / `┗` 末尾)を `<pre>` ブロックで記載し、分岐構造を可視化する。
- 表示するエラー・警告メッセージは §8 エラーメッセージに `EM-NN`(ページ内ローカル連番)で一元定義し、§5 バリデーション・§7 画面イベント詳細はコード(`EM-NN`)で参照する(文面を各所へ重複させない)。
- 要件ID(`FR` / `BR` / `NFR` / `RULE` / `UC`)は画面設計書に記載しない。しきい値・状態などは具体値・業務語で書き、要件との対応はトレーサビリティ一覧の `TR-NNN`(§1)で辿る。
- §4 画面項目は画面が **いずれかの状態で実際に表示する項目を網羅** し、各項目に `表示条件`(状態)を記す。§3 画面レイアウト(mock = 代表状態)に写る項目はすべて §4 に含め(モックにある項目を落とさない)、逆にどの状態でも表示しない幽霊項目は §4 に書かない。差異が出たら表示実態に合わせて §4 を直し、§5 バリデーション・§6/§7 イベント・§8 エラーメッセージまで連動修正する。状態ごとのモック画像作成は必須としない(不足状態の作図は別バックログ)。
- **mermaid 図は描画可能であること。** コードフェンスの開き(` ```mermaid `)・閉じ(` ``` `)は必ず独立行に置き、後続の文・要素を同一行へ連結しない。ノードラベルに丸括弧等の特殊文字を含めるときは `"..."` で囲む。

# IPO処理機能記述書(IPO)テンプレート

> **本テンプレートは IPO処理機能記述書(IPO)の記載骨格(主要ビジネスロジック 1 件 = 1 ファイル)を定義します。**

運用ルールの正本は [../CLAUDE.md](../CLAUDE.md)。共通記載スタイルは [共通記載スタイル](00_common-style.md) を参照する。

- **配置先**: `03_detail_design/04_ipo/`(`index.md` + `IPO-001.md`〜)
- **採番**: `IPO-001`〜(主要ビジネスロジック 1 件 = 1 ファイル・ゼロ詰め 3 桁・欠番なし・`<span id>` 出現順 001)。ID は各設計書の H1 に `# <span id="IPO-001"></span>IPO-001: 名称` で保持する。

## 骨格

**主要ビジネスロジック 1 件 = 1 ファイル(`IPO-NNN`)**。AI 回答可否判定・月次請求確定・サスペンション判定・しきい値伝播・アラート判定など、入力(Input)を受けて判定・分岐・状態変化を伴う処理(Process)を経て出力(Output)を確定する複雑ロジックを、実装者が迷わず組み立てられる粒度へ具体化する。基本設計(SEQ / API / SYS / FR / RULE)を入力とし、Next.js Service 層の責務単位で 1 ファイルに切る。骨格:

- H1 アンカー `# <span id="IPO-NNN"></span>IPO-NNN: 名称`
- ページ要約(引用ブロック)+ 文書メタ(斜体 1 行。共通記載スタイル参照)
- ヘッダー表(IPO ID / 業務ユースケースID / 関連 API・SYS / 参照 SEQ / 利用テーブル)。全層の厳密な紐付けはトレーサビリティ一覧表に一元管理し、IPO 本文に `TR-NNN` は記載しない。
- `## 1. 目的` / `## 2. 処理概要` / `## 3. IPO 一覧` / `## 4. 処理詳細` / `## 5. 後続工程への引き継ぎ事項`

各 `##`/`###` 直後に 1〜2 文のリード文を置いてから表へ入る(表をいきなり置かない)。

## 記載ルール

- **設計値(しきい値・単価・無料枠・タイムアウト・保持期間)・状態名の意味は本書で再定義しない。** [システム仕様書](../02_basic_design/07_system-spec.md) / [状態モデル](../02_basic_design/08_state-model.md) / [用語集](../01_requirements/00_glossary.md) へリンクまたは ID 参照で送る(例: 「信頼度しきい値([システム仕様書 §1](../02_basic_design/07_system-spec.md#1-aiしきい値))未満のとき」)。「等」で打ち切らない。
- **判定条件・取得/更新条件・疑似コードは可**。ただし実装コード本文・物理カラム名の羅列・SQL 本文・アルゴリズム内部実装(具体的なデータ構造・ライブラリ呼び出し)は書かない。テーブル参照はリンクで辿り、物理名を本文へ転記しない(物理カラム名の定義は DBP の本務)。
- 業務ユースケースID(`UC`)と設計上必要な関連 ID(`API` / `SYS` / `SEQ` / `TBL`)はヘッダー表に記載する。要件 ID(`FR` / `BR` / `NFR` / `RULE`)は設計根拠の括弧引用(例: 「(RULE-012)」)としてのみ許容し、逆引きの対応表・専用欄は作らない。
- mermaid を使う場合はフェンスの開き(` ```mermaid `)・閉じ(` ``` `)を独立行に置き、ノードラベルに丸括弧等を含めるときは `"..."` で囲む。**絵文字は使わない。**

## 記載例(セクション骨格)

```markdown
# <span id="IPO-001"></span>IPO-001: AI 回答可否判定

> **本記述書は「AI が生成した回答を利用者へ提示してよいか」を信頼度・関連度から判定する処理を定義します。**

*種別 IPO処理機能記述書 ・ 優先度 P0 ・ ステータス ドラフト*

| 項目 | 値 |
|----|----|
| IPO ID | IPO-001 |
| 業務ユースケースID | [UC-030](../../01_requirements/04_business_usecases/UC-030.md#UC-030) |
| 関連 API / SYS | [API-030](../../02_basic_design/02_backend/03_apis/API-030.md#API-030) ・ [SYS-015](../../02_basic_design/02_backend/01_system/SYS-015.md#SYS-015) |
| 参照 SEQ | [SEQ-040](../../02_basic_design/03_sequences/SEQ-040.md#SEQ-040) |
| 利用テーブル | [TBL-031](../../02_basic_design/02_backend/04_database/TBL-031.md#TBL-031) |

## 1. 目的

本処理が担う業務目的と、実装者が押さえるべき前提(適用されるしきい値の正本・依存する状態)を 1〜2 文で示す。

## 2. 処理概要

処理単位の起動から終了までの枠組みを 1 行で俯瞰する。

| 機能名 | 処理概要 | 起動条件 | 終了条件 |
|----|----|----|----|
| AI 回答可否判定 | 信頼度・関連度としきい値を比較し提示可否を確定する | 回答生成の完了を受けて呼び出されたとき | 提示可 / 不可 / 判定不能を確定して呼び出し元へ返したとき |

## 3. IPO 一覧

入力・処理・出力の対応と例外・分岐を 1 行 1 処理で一覧化する。

| No | Input | Process | Output | 例外・分岐 | 備考 |
|----|----|----|----|----|----|
| 1 | 信頼度・関連度スコア、対象プロジェクトの設定値有無 | 適用しきい値を決定(プロジェクト設定値優先・未登録はグローバル既定) | 適用しきい値 | 設定値取得不能時はグローバル既定へフォールバックしアラート([システム仕様書 §1](../02_basic_design/07_system-spec.md#1-aiしきい値)) | しきい値の正本はシステム仕様書 |
| 2 | 各スコア、適用しきい値 | しきい値以上か判定 | 提示可否(可 / 不可) | いずれか未満なら不可(未解決へ回す) | — |

## 4. 処理詳細

各処理の判定条件・入出力・エラー時挙動を実装可能な粒度で定義する(疑似コード・判定条件は可、実装コード本文・SQL 本文は書かない)。

| No | 処理名 | 処理内容(疑似コード / 判定条件) | 入力 | 出力 | 条件 | エラー時 |
|----|----|----|----|----|----|----|
| 1 | 適用しきい値決定 | `if プロジェクト設定値あり → 設定値 else グローバル既定値` | プロジェクト設定値有無 | 信頼度・関連度しきい値 | 対象プロジェクトの推論時 | 取得不能時はグローバル既定で継続しアラート |
| 2 | 提示可否判定 | `if 信頼度 ≥ しきい値 and 関連度 ≥ しきい値 → 提示可 else 提示不可` | 各スコア・適用しきい値 | 提示可 / 不可 | しきい値決定後 | スコア欠損時は判定不能として未解決扱い |

## 5. 後続工程への引き継ぎ事項

詳細シーケンス(DSQ)・テスト設計へ引き継ぐ観点(境界値・冪等性・並行実行・フォールバック契機)を挙げる。

- しきい値の境界値(等号を含むか)と、判定不能(スコア欠損)を未解決へ回す扱いの確認。
- 設定値取得不能時のフォールバックとアラート発火のテスト観点。
```

## テンプレ運用の注記

このセクションはテンプレの使い方ガイドであり、実ドキュメントには転記しない。

- **記載しない項目**: 実装コード本文・物理カラム名の羅列・SQL 本文・アルゴリズム内部実装・変数名/JSON キー名・`TR-NNN`(層横断トレーサビリティ)・要件 ID の逆引き対応表・具体的な設計値の再定義(しきい値/単価/保持期間は正本へリンク)。
- **基本設計との責務分担**: 基本設計(SEQ / API / SYS)が「どの契機で何を呼ぶか」の枠組みを示すのに対し、IPO は「入力を受けてどう判定・分岐し何を確定するか」の処理ロジックを実装可能な粒度へ具体化する。基本設計に無い仕様は推測で足さず、不明点は GitHub Issue で課題化する。適用範囲は AI 回答可否判定・月次請求確定・サスペンション判定・しきい値伝播・アラート判定など、判定・分岐・状態変化を伴う複雑ビジネスロジック(Next.js Service 層の責務単位)。バッチ/非同期/監視の起動制御そのものは BAT、外部連携仕様は EIF、物理カラム定義は DBP が担う。
- **後続テスト設計への引き継ぎ観点**: `## 5.` に境界値・例外系・冪等性・並行実行・フォールバック契機を残し、後続の詳細シーケンス(DSQ)・テスト設計が処理条件と成否条件を一意に読み取れる状態を保つ。

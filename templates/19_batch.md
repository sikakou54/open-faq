# バッチ処理設計(BAT)テンプレート

> **本テンプレートはバッチ処理設計(BAT)の記載骨格(1 バッチ = 1 ファイル)を定義します。**

運用ルールの正本は [../CLAUDE.md](../CLAUDE.md)。共通記載スタイルは [共通記載スタイル](00_common-style.md) を参照する。

- **配置先**: `03_detail_design/05_batch/`
- **採番**: `BAT-001`〜(1 バッチ = 1 ファイル・ゼロ詰め 3 桁・欠番なし・出現順 001)。ID は各設計書の H1 に `# <span id="BAT-001"></span>BAT-001: 名称` で保持する。

## 骨格(9 セクション固定)

**1 バッチ = 1 ファイル(`BAT-NNN`)**。基本設計のシステム設計(SYS。無人処理の §処理 / §イベント SEV)を詳細化した詳細設計層の文書で、実装スタック(TypeScript + Next.js + Repository 層)と実行基盤(Cloudflare)への写像を定義する。**定時起動は Cloudflare Cron Triggers、非同期・自動リトライは Cloudflare Queues(失敗継続時は DLQ)へ写像する。** モックは持たない。9 セクション固定:

1. **目的**(本バッチが実現する業務・上位 SYS との対応を 1〜2 文で示す)
2. **バッチ概要**(バッチ ID 表 = バッチ ID / バッチ名 / 起動方式 / 起動タイミング / 多重起動可否 / リトライ有無 / 想定処理件数 / 想定処理時間 + 関連表(業務ユースケースID / 対応システム `SYS` / 関連 API / 関連テーブル))
3. **入力**(入力元 / 内容 / 取得条件 / 備考)
4. **処理内容**(No / 処理 / 条件 / 正常時 / 異常時 / 備考)
5. **出力**(出力先 / 内容 / 出力条件 / 備考)
6. **排他・並列処理**(排他制御 / 並列処理 / 多重起動時の扱い / 冪等キー)
7. **ログ・監視**(ログ種別 / 出力内容 / 出力タイミング / 備考)
8. **異常終了時の扱い**(異常内容 / 対応 / 通知先 / リトライ / 備考)
9. **後続工程への引き継ぎ事項**(テスト設計・運用設計へ渡す観点を箇条書き)

## 記載ルール

- 各 `##`/`###` 直後に 1〜2 文のリード文を置いてから表へ入る(表をいきなり置かない)。
- **起動方式**は Cloudflare の実体へ写像して記す(定時 = `Cron Triggers` / 非同期 = `Queues` / イベント契機 = `Queues`(上流イベントの投入)/ 手動 = 運用者による手動実行)。基本設計 SYS の `種別`(`batch` / `cron` / `async` 等)・`トリガー・スケジュール`と齟齬を生じさせない。
- **具体的な設計値をテンプレ内・本文で再定義しない。** リトライ上限・再送遅延・保持期間・タイムアウト・しきい値・状態名の意味は正本へ ID 参照 / リンクする。「等」で打ち切らない。
  - リトライ上限・再送遅延: [NFR-040](../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-040)(通知メール再送 最大 3 回・最大遅延 24 時間)/ [NFR-036](../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-036) を ID 参照する。
  - 冪等性・滞留アラート: [NFR-041](../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-041)(通知ジョブは冪等性保証・再処理待ち滞留は 1 時間以内にアラート)を ID 参照する。
  - 保持期間・タイムアウト・単価・しきい値: [システム仕様書](../02_basic_design/07_system-spec.md) を参照する(本文へ数値を転記しない)。
  - 状態名の意味: [状態モデル](../02_basic_design/08_state-model.md) を参照する。用語は [用語集](../01_requirements/00_glossary.md) を正本とする。
- **層横断トレーサビリティ(`TR-NNN`)は本文に書かない。** §2 の関連表には業務ユースケースID(`UC-NNN`)と設計上必要な関連 ID(対応 `SYS` / 関連 `API` / 関連 `TBL`)のみを置く。全層の厳密な紐付けはトレーサビリティ一覧表 [`00_traceability/index.md`](../02_basic_design/00_traceability/index.md) に一元管理する。
- **実装コード本文・SQL 本文・物理カラム名の羅列・アルゴリズム詳細は書かない。** 判定条件・取得条件 / 更新条件・疑似コード・冪等キーの構成方針は書いてよい。テーブルとの対応はリンクで辿る(物理名を本文へ転記しない)。
- 相互参照アンカーの記法・相対パスは [共通記載スタイル](00_common-style.md) の相互参照アンカー例に従う(`03_detail_design/05_batch/` からの深さは `../02_basic_design/...` / `../01_requirements/...` / `../CLAUDE.md`)。

## テンプレ運用の注記

本節はテンプレの使い方ガイドであり、**実ドキュメントには転記しない。**

- **記載しない項目**: 実装コード本文・SQL 本文・ORM クエリ・物理カラム名の羅列・アルゴリズム詳細。層横断トレーサビリティ(`TR-NNN`)。正本で定義済みの具体値(リトライ上限・再送遅延・保持期間・タイムアウト・単価・しきい値・状態名の意味)の再定義。「等」での打ち切り。
- **基本設計との責務分担**: 基本設計のシステム設計(SYS)は「何を無人処理として実現するか」を業務語で定義する。BAT はその詳細設計として「Cloudflare のどの実行基盤(Cron Triggers / Queues / DLQ)にどう写像し、入力取得条件・処理順・出力・排他・冪等・監視・異常終了時の扱いをどう組むか」を定義する。SYS で定義済みの業務意図・トリガー種別と齟齬を生じさせず、§2 の関連表で対応 `SYS` を明示する。
- **後続テスト設計への引き継ぎ観点**: §9 に、境界値(0 件 / 想定上限件数)・部分失敗時の継続と件数集計・多重起動抑止・冪等な再実行・Queues 自動リトライと DLQ 滞留の再処理手順・想定処理時間超過時の扱いなど、テスト設計で検証すべき観点を残す。

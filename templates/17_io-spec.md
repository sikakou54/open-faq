# 入出力設計書(IO)テンプレート

> **本テンプレートは入出力設計書(IO)の記載骨格(画面・API の入力/出力項目とチェック・加工仕様を論理項目単位で確定する詳細設計)を定義します。**

運用ルールの正本は [../CLAUDE.md](../CLAUDE.md)。共通記載スタイルは [共通記載スタイル](00_common-style.md) を参照する。

- **配置先**: `03_detail_design/03_io_specs/`
- **採番**: `IO-001`〜(1 単位 = 1 ファイル・ゼロ詰め 3 桁・欠番なし・`<span id>` 出現順に `001` から採番)。ID は各設計書の H1 に `# <span id="IO-001"></span>IO-001: 名称` で保持する。

## 骨格(7 セクション固定)

**1 入出力単位 = 1 ファイル(`IO-NNN`)**。基本設計の [画面設計(SCR)](06_screen.md) の §4 画面項目・§5 バリデーションと [API設計(API)](10_api.md) の `## リクエスト` / `## レスポンス` / `## バリデーション` / `## 列挙値` を突き合わせ、実装に必要な論理項目・チェック・加工を詳細設計として確定する。骨格:

- H1 アンカー `# <span id="IO-NNN"></span>IO-NNN: 名称`
- ページ要約(引用ブロック)+ 文書メタ(斜体 1 行)
- ヘッダー表(入出力設計ID / 業務ユースケースID / 関連画面 `SCR` / 関連 API `API`)。全層の厳密な紐付けはトレーサビリティ一覧表に一元管理し、本文に `TR-NNN` は記載しない。
- `## 1. 目的`
- `## 2. 対象範囲`
- `## 3. 入力項目一覧`
- `## 4. 出力項目一覧`
- `## 5. 入力チェック仕様`
- `## 6. 編集・加工仕様`
- `## 7. 後続工程への引き継ぎ事項`

導出できない欄は `要確認`、該当なしは `—`(全角ダッシュ)。

## 記載ルール

- 各 `##`/`###` 直後に 1〜2 文のリード文を置いてから表へ入る(表をいきなり置かない)。
- **論理項目名で記述する(物理カラム名を書かない)。** 項目 ID は本ページ内ローカル連番(`I-NN` 入力 / `O-NN` 出力)で採番し、§5・§6 から参照する。テーブルの物理項目対応は [DB物理設計(DBP)](11_database.md) 側で定義する。
- **バリデーション閾値・上限・強度規則を本ページで再定義しない。** 文字数上限・パスワード強度・AI しきい値・保持期間などの具体値は [機能要件(FR)](../01_requirements/02_functional_requirement/02_faq-ai-fr.md) / [業務ルール(RULE)](../01_requirements/01_business_requirement/08_rule.md) / [システム仕様書](../02_basic_design/07_system-spec.md) を ID 参照で示す(例:「12 文字以上([FR-006](../01_requirements/02_functional_requirement/01_account-fr.md#FR-006))」)。状態値の意味は [状態モデル](../02_basic_design/08_state-model.md)、用語は [用語集](../01_requirements/00_glossary.md) を参照する。「等」で打ち切らない。
- **入力チェックのエラー文面は本ページに書かない。** エラー時メッセージは [エラー定義(ERR)](../02_basic_design/05_errors/ERR-001.md#ERR-001) / [メール文面(MSG)](13_permission-error-message.md)、または画面表示の場合は SCR §8 の `EM-NN` を ID 参照する(文面を重複させない)。
- **本ページは SCR と API の項目対応の橋渡しを担う。** SCR の画面項目(§4 の `#N`)と API のリクエスト/レスポンス項目の対応を論理項目で結ぶ。SCR の値パターン(§4 データパターン)・API の列挙値(§列挙値)を再掲せず参照で示す。
- **詳細設計粒度に留め、実装コード本文を書かない。** 変数名・JSON キー名・SQL 本文・関数名・アルゴリズム実装は書かない。判定条件・取得/更新条件・疑似コード(自然言語の条件式)は可。
- 要件ID(`FR` / `BR` / `NFR` / `RULE`)は設計根拠の括弧引用としてのみ用いる。層横断トレーサビリティ(`TR-NNN`)・逆引き(`関連FR/BR` 等の専用欄)は作らない。
- mermaid を用いる場合はコードフェンスの開き(` ```mermaid `)・閉じ(` ``` `)を独立行に置き、ノードラベルに括弧等があれば `"..."` で囲む。**絵文字は使わない。**

## 記載例(セクション骨格)

```markdown
# <span id="IO-001"></span>IO-001: 新規登録 入出力

> **この入出力設計書は「新規登録画面と新規登録 API の入力/出力項目・入力チェック・編集加工」を論理項目単位で定義します。**

*種別 入出力設計 ・ ステータス ドラフト*

| 項目 | 値 |
|----|----|
| 入出力設計ID | IO-001 |
| 業務ユースケースID | [UC-002](../../01_requirements/04_business_usecases/UC-002.md#UC-002) |
| 関連画面 | [SCR-002](../../02_basic_design/01_frontend/01_screens/SCR-002.md#SCR-002) |
| 関連 API | [API-001](../../02_basic_design/02_backend/03_apis/API-001.md#API-001) |

## 1. 目的

本入出力設計が確定する入力/出力の範囲と、橋渡しする画面・API を 1〜2 文で示す。

## 2. 対象範囲

対象とする画面・API・データと、対象外を明示する。

| 区分 | 対象 |
|----|----|
| 対象画面 | [SCR-002](../../02_basic_design/01_frontend/01_screens/SCR-002.md#SCR-002) アカウント登録 |
| 対象API | [API-001](../../02_basic_design/02_backend/03_apis/API-001.md#API-001) 新規登録 |
| 対象データ | 表示名・メールアドレス・パスワード・規約/プライバシー同意 |
| 対象外 | 確認メール再送(別 IO で扱う)・課金アカウント作成 |

## 3. 入力項目一覧

画面から受け取り API へ渡す入力項目を論理項目単位で定義する。

| 項目ID | 項目名 | データ型 | 必須 | 初期値 | 入力制約 | 入力例 | 備考 |
|----|----|----|----|----|----|----|----|
| I-01 | 表示名 | 文字列 | ◯ | — | 1〜100 文字([FR-005](../01_requirements/02_functional_requirement/01_account-fr.md#FR-005)) | 山田 太郎 | SCR-002 #1 に対応 |
| I-02 | メールアドレス | 文字列 | ◯ | — | RFC 5322 準拠・最大 254 文字 | user@example.com | — |
| I-03 | パスワード | 文字列 | ◯ | — | 12 文字以上・3 種類以上([FR-006](../01_requirements/02_functional_requirement/01_account-fr.md#FR-006)) | — | 画面には表示しない |

## 4. 出力項目一覧

API レスポンス・画面へ返す出力項目を論理項目単位で定義する。

| 項目ID | 項目名 | データ型 | 表示形式 | 表示条件 | 備考 |
|----|----|----|----|----|----|
| O-01 | 受付結果 | 真偽値 | — | 常時 | 受理時 `true` |
| O-02 | 送信先メールアドレス | 文字列 | メール確認画面に表示 | 受付成功時 | SCR-018 #2 に対応 |

## 5. 入力チェック仕様

各入力項目のチェック内容・実施タイミング・エラー時参照を定義する。

| 項目名 | チェック内容 | タイミング | エラー時メッセージ | 備考 |
|----|----|----|----|----|
| 表示名(I-01) | 未入力チェック・文字数上限チェック | クライアント・サーバー | [ERR-001](../02_basic_design/05_errors/ERR-001.md#ERR-001) / SCR-002 EM-01 | — |
| メールアドレス(I-02) | 形式チェック・重複チェック | サーバー | [ERR-014](../02_basic_design/05_errors/ERR-014.md#ERR-014) | 重複はサーバーのみ判定 |
| パスワード(I-03) | 強度チェック([FR-006](../01_requirements/02_functional_requirement/01_account-fr.md#FR-006)) | クライアント・サーバー | [ERR-001](../02_basic_design/05_errors/ERR-001.md#ERR-001) | — |

## 6. 編集・加工仕様

入力値の正規化・出力値の整形など編集加工を定義する。

| 項目名 | 加工内容 | 加工タイミング | 備考 |
|----|----|----|----|
| メールアドレス(I-02) | 前後空白の除去・小文字化 | 保存前(サーバー) | 重複判定は加工後の値で行う |
| 表示名(I-01) | 前後空白の除去 | 保存前(サーバー) | — |

## 7. 後続工程への引き継ぎ事項

テスト設計・DB 物理設計・詳細ロジック設計へ引き継ぐ観点を挙げる。

- 入力チェックの境界値(最小/最大文字数)はテスト設計でパターン化する。
- 論理項目 ↔ 物理カラムの対応は [DB物理設計(DBP)](11_database.md) で確定する。
```

## テンプレ運用の注記(実ドキュメントには転記しない)

本節はテンプレートの使い方ガイドである。以下の内容を実際の `IO-NNN.md` へ転記しない。

- **記載しない項目**: バリデーション閾値・上限・強度・しきい値・保持期間などの具体値(FR/RULE/システム仕様書へ ID 参照)、状態値の意味(状態モデルへ参照)、エラー/メッセージ文面(ERR/MSG または SCR §8 の `EM-NN` へ参照)、物理カラム名・SQL 本文・変数名・JSON キー名・関数名・アルゴリズム実装、層横断トレーサビリティ(`TR-NNN`)と逆引き専用欄。
- **基本設計との責務分担**: SCR は「画面が表示する項目・状態・画面内検証」、API は「エンドポイントのリクエスト/レスポンス構造・列挙値・API 検証」を定義する。IO は両者を論理項目で突き合わせ、実装に必要な入力/出力項目の確定・入力チェックの実施主体(クライアント/サーバー)・編集加工を詳細設計として確定する(基本設計の値・文面は再定義せず参照する)。
- **後続テスト設計への引き継ぎ観点**: 入力チェックの境界値・組み合わせ、必須/任意の網羅、加工前後の値の期待結果、出力項目の表示条件分岐は §7 に整理し、テスト設計でケース化する。論理項目 ↔ 物理カラムの対応は DB 物理設計へ引き継ぐ。

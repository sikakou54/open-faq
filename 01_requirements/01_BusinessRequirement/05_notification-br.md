# 通知・お知らせ(業務要件)

> **このページは、通知・お知らせ に関する業務要件を定義します。**

*複数の業務要件を一覧化したカテゴリ別ページ。掲載要件は全件 種別 業務要件 ・ 優先度 P0 ・ ステータス ドラフト。トレースは下流(UC 等)からの逆引きで辿る。*

<td>通知</td>
<td>請求確定・運営お知らせなどの契機が発生した時に、対象のアカウント利用者へメールまたはお知らせ受信箱で通知を届ける。</td>

<tr>
<td><span id="BR-077"></span>BR-077</td>
<td>請求確定通知</td>
<td>通知</td>
<td>月次の請求が確定した時に、支払い責任を負うオーナーが金額と根拠を確実に把握できるよう請求内容を通知する。同一の請求を重ねて課さない方針とする。</td>
</tr>
<tr>
<td><span id="BR-078"></span>BR-078</td>
<td>お知らせ配信</td>
<td>通知</td>
<td><ul>
<li><strong>システム：</strong>運営からのお知らせ(メンテナンス予告・機能追加・規約改定・価格改定など)を、対象のアカウント利用者へ確実に届ける</li>
<li><strong>アカウント利用者：</strong>受け取りたくないお知らせは種別ごとにオプトアウトできる。ただし規約改定・重要セキュリティ通知は周知が義務的なため強制送信としオプトアウトできない</li>
</ul></td>
</tr>
<tr>
<td><span id="BR-079"></span>BR-079</td>
<td>役割変更通知</td>
<td>通知</td>
<td>メンバーのプロジェクト別の役割割当が追加・変更・剥奪された時に、当該メンバーへお知らせ受信箱とメールで通知する。</td>
</tr>
<tr>
<td><span id="BR-080"></span>BR-080</td>
<td>システム通知の自動生成</td>
<td>通知</td>
<td>利用者が見落とすと不利益を被る運用イベント(利用上限への接近・到達、通知失敗の急増、サスペンション、復元など)を、システム側が自動で検知して「システム通知」として知らせる。</td>
</tr>
<tr>
<td><span id="BR-081"></span>BR-081</td>
<td>送信失敗時の再送</td>
<td>通知</td>
<td>一時的な障害で通知が届かなかった時も利用者に確実に伝わるよう再送する。ただし無制限の再送による負荷・迷惑を避けるため再送には上限を設ける。</td>
</tr>
<tr>
<td><span id="BR-082"></span>BR-082</td>
<td>バウンス・苦情対応</td>
<td>通知</td>
<td>バウンスや無効アドレス・苦情を検知した時に、送信元ドメインの評判悪化を防ぐため当該宛先への送信を停止し、管理者へ通知する。共通の送信ドメインを使うため、送信停止リストは全契約横断で共有する方針とする。</td>
</tr>
<tr>
<td><span id="BR-083"></span>BR-083</td>
<td>送信品質の監視</td>
<td>通知</td>
<td>送信元ドメインの信頼性を維持するため、契約 / プロジェクト単位で送信品質(送信レート・バウンス率・苦情率)を監視し、悪化した契約の送信を抑制する。</td>
</tr>
<tr>
<td><span id="BR-084"></span>BR-084</td>
<td>配信状態の可視化</td>
<td>通知</td>
<td>通知が相手に届いたか、失敗・停止していないかをアカウント利用者が把握し、トラブル時に対処できるよう、通知の配信状態を確認できる。</td>
</tr>
<tr>
<td><span id="BR-085"></span>BR-085</td>
<td>認証通知の強制送信</td>
<td>通知</td>
<td>アカウント認証関連の通知(パスワード再設定等)は本人のアカウント保護に不可欠なため、オプトアウトできない。</td>
</tr>
<tr>
<td><span id="BR-086"></span>BR-086</td>
<td>なりすまし・スパム対策</td>
<td>通知</td>
<td>ウィジェット利用者の入力を悪用したなりすまし・スパム埋め込みを防ぐため、入力文字列を外部に露出する通知箇所へそのまま使わない。</td>
</tr>
<tr>
<td><span id="BR-107"></span>BR-107</td>
<td>お知らせの確認</td>
<td>お知らせ</td>
<td>アカウント利用者が自契約宛のお知らせを見落とさず内容を把握できるよう、一覧と詳細で確認できる。</td>
</tr>
<tr>
<td><span id="BR-108"></span>BR-108</td>
<td>重要度の区分</td>
<td>お知らせ</td>
<td>利用者が緊急度に応じて優先して対応できるよう、お知らせを重要度で区分する。最上位の重要度は乱用すると埋もれてしまうため、規約改定・重大セキュリティ通知・サスペンション確定といった対応必須の場面に限定して付与する方針とする。</td>
</tr>
<tr>
<td><span id="BR-109"></span>BR-109</td>
<td>既読管理</td>
<td>お知らせ</td>
<td>アカウント利用者が未対応のお知らせを管理しやすいよう、確認済みのお知らせを個別または一括で既読にできる。お知らせが多くても目的のものへ素早く到達できるよう絞り込めること。</td>
</tr>
<tr>
<td><span id="BR-110"></span>BR-110</td>
<td>未読の即時把握</td>
<td>お知らせ</td>
<td>アカウント利用者が日常の操作を中断せずに未読のお知らせへ気づけるよう、未読件数と直近のお知らせを画面遷移なしに把握できる。</td>
</tr>
<tr>
<td><span id="BR-111"></span>BR-111</td>
<td>受信箱の自動生成</td>
<td>お知らせ</td>
<td>メールを見落としても管理画面上で重要な通知を確認できるよう、各契機(請求確定・運営お知らせ・運用イベント等)で対象アカウント利用者の受信箱へお知らせを生成する。メール通知と受信箱の内容が食い違わないよう、両者を同期して生成する方針とする。</td>
</tr>
<tr>
<td><span id="BR-112"></span>BR-112</td>
<td>重複生成の集約</td>
<td>お知らせ</td>
<td>同種のイベントが短時間に連続発火しても利用者の受信箱が大量のお知らせで埋もれないよう、同一契約・同一イベント種別のお知らせを 1 件に集約する。</td>
</tr>
<tr>
<td><span id="BR-113"></span>BR-113</td>
<td>閲覧範囲の限定</td>
<td>お知らせ</td>
<td>お知らせには契約の機微情報が含まれうるため、受信箱はアカウント利用者ごとに保持し本人のみが閲覧できる。エンドユーザーであるウィジェット利用者には公開しない。</td>
</tr>
<tr>
<td><span id="BR-114"></span>BR-114</td>
<td>退会・無効化時の削除</td>
<td>お知らせ</td>
<td>退会・無効化後に不要な個人データを残さないため、アカウントの無効化・退会が確定した時に当該アカウント利用者の受信箱を削除する。</td>
</tr>
<tr>
<td><span id="BR-115"></span>BR-115</td>
<td>操作の監査</td>
<td>お知らせ</td>
<td>広範囲に影響する一括既読操作について、後から「誰がいつ行ったか」を追跡できるよう監査記録する。</td>
</tr>

</table>

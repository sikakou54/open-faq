# 要件定義書 再構成結果

> **このページは、要件定義書を「1 要件 = 1 ファイル」のフラット個別ページ構成へ再構成した結果を、要件定義書の視点でまとめます。** 目的 / 新構成 / ID リナンバ / フェーズ実績 / トレーサビリティ状況 / 残課題を要約します。

*種別 再構成結果サマリ ・ 更新 2026-06-21 ・ ステータス ドラフト*

## 1. 目的と読み順

旧来のグループ別大型ファイル(`FR01.md`〜`FR21.md` に業務要件・機能要件を混載)を解体し、識別子 1 件 = 1 ファイルの**フラット個別ページ**へ再構成した。これにより、各要件を一意 ID で参照・トレースでき、下流(業務UC → 画面 → API → テーブル)との双方向結線が機械検証可能になった。

読み順(全体)は **要件定義 ＞ 業務ユースケース ＞ 画面設計 ＞ 画面イベント ＞ API設計 ＞ DB設計 ＞ シーケンス ＞ 権限 / エラー / メッセージ**。要件定義書はその起点である。

## 2. 新構成

| パス | 内容 |
|----|----|
| [`index.md`](index.md) | システム概要 / 背景・目的 / スコープ / 利用者・ロール / 制約・前提 / 読み順 / 要件仕様・業務UCへのポインタ |
| [`01_BusinessRequirement/`](01_BusinessRequirement/index.md) | 業務要件 `BR-*`(カテゴリ別 HTML テーブル)と業務ルール `RULE-*`(`08_rule.md`) |
| [`02_FunctionalRequirement/`](02_FunctionalRequirement/index.md) | 機能要件 `FR-*`(カテゴリ別ファイルの節) |

| [`04_business_usecases/`](04_business_usecases/index.md) | 業務ユースケース。`UC-*` を 1 UC = 1 ファイルで定義(画面起点 + システム起点) |

## 3. ID リナンバ要約

各系列をフラット連番・ゼロ詰め 3 桁・欠番なしへ再採番した。旧 ID(グループ別 `FRxx` 内の連番や旧 UC-SCR-*-EV* / UC-SYSTEM-*)からの対応は `99_management/01_crosswalk.md` / `crosswalk.json` / `uc_crosswalk.json` に保持する。

| 系列 | 件数 | 旧 → 新 方式 |
|----|----|----|
| 業務要件 `BR` | 146 | グループ別ファイル混載 → `BR-001`〜`BR-146`(文書出現順) |
| 機能要件 `FR` | 194 | グループ別ファイル混載 → `FR-001`〜`FR-194` |
| 非機能要件 `NFR` | 79 | カテゴリ別番号帯(`1xx`/`2xx`…)・範囲行 → 通し連番 `NFR-001`〜`NFR-079`(範囲行を展開) |
| 業務ルール `RULE` | 20 | FR / BR 文中の定量しきい値・ポリシーを新規抽出 → `RULE-001`〜`RULE-020` |
| 業務ユースケース `UC` | 247 | 旧 UC-SCR-*-EV* / UC-SYSTEM-* → `UC-001`〜`UC-247`(`UC-001`〜`UC-229` 画面起点 / `UC-230`〜`UC-247` システム起点) |

## 5. トレーサビリティ状況

要件 → 業務UC → 画面 → 画面イベント → API → テーブル の一気通貫トレースを確立し、UC 単位 **247 行**のマトリクスを `99_management/02_traceability_matrix.md` に生成した。各 FR / BR の `対応業務UC` 欄は逆引きで結線済み。

> [!WARNING]
> **要件被覆ギャップ(後続課題)** 現状、**132 FR + 91 BR が業務UC 未対応**(`(該当UCなし=ギャップ)`)、逆に **128 UC が要件(`対応要件ID`)未対応**である。UC は主に画面イベント由来の操作系で、対応する FR が未割当のものが多い。**要件被覆の精査(各 FR を実現する UC の結線、操作系 UC の FR 起票)は後続課題**として残る。詳細件数・対象 ID は `99_management/p7_gaps.json` を参照。

## 6. 既知の残課題

要件定義由来の残課題は `99_management/06_remaining_issues.md` に集約している(GitHub 再接続後に `[設計再構成][区分]` で起票予定)。主なもの:

- **traceability** — 上記の要件↔UC 被覆ギャップ(132 FR / 91 BR / 128 UC)の精査・結線。
- **migration** — 要件文に内在する実装寄り記述(具体値・技法)の基本設計移管要否の棚卸し。
- **needs-review** — `RULE` 抽出(20 件)の網羅性、`NFR` 採番(79 件)の最終確定。
- **cleanup** — 各 UC 備考の「旧 UC-… から導出」provenance 注記の整理。

GitHub Issue #3〜#12(エピック #3 + 各フェーズ)および P7 検出ギャップが対象。`99_management/` は再構成期間の作業ディレクトリで、最終的に内容を正式設計書 / GitHub Issues へ移管して削除する。

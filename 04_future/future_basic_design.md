## 0. 位置づけ

本書は、Post-MVP 要件を基本設計へ落とす際の方式・画面・状態・データ・連携の論点を管理する。要件の採否や判定基準は [future_requirements.md](future_requirements.md)、DDL / API / Worker / 移行 / テストの実装候補は [future_detailed_design.md](future_detailed_design.md) を正本とする。

## 1. 基本設計 Future 論点

### 1.1 認証・セキュリティ

| 項目 | 基本設計で具体化すること | 関連 Future 要件 |
|---|---|---|
| テナント側 admin / operator MFA | SCR-016 設定画面、MFA 登録・解除フロー、テナント単位強制状態、未設定ユーザーのログイン制御、リカバリ導線 | FUT-REQ-SEC-002 |
| 4-eyes 完全強制 | 10 操作の申請・承認・実行状態、SCR-096 のバイパス検出 KPI、緊急例外 RB-014 との分岐 | FUT-REQ-SEC-003 |

| SSO / パスキー | 認証プロバイダ選定、既存自前認証との併存、招待・退会・ロックアウトとの状態整合 | FUT-REQ-SEC-005 |
| 個別チャット列単位暗号化 | 鍵階層、検索・復元・監査への影響、障害時の復旧設計、運用者が本文を扱う導線の制限 | FUT-REQ-SEC-001 |

## 3. 設計レビュー観点

| 観点 | 確認事項 |
|---|---|
| MVP 本文との分離 | MVP ドキュメント本文に Future 文書への参照を戻していないこと |
| 要件との対応 | すべての基本設計 Future 論点が [future_requirements.md](future_requirements.md) の ID を持つこと |
| 実装粒度の分離 | DDL / API / Worker / Queue / テスト詳細は [future_detailed_design.md](future_detailed_design.md) へ送ること |
| 運用責務 | 運営者、テナント管理者、対応窓口担当者、エンドユーザーの責務境界が変わる場合は明示すること |
| リリース影響 | 課金、認証、監査、法務、SLA に影響するものはリリース判定条件を持つこと |

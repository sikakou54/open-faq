## 2. システム概要

### 2.1 背景

本サービス(open-faq)の SaaS 運営事業者向けに、契約管理 / 削除データ復元 / AI パラメータ調整 / 課金 Webhook 一次受信 / 監査 / 不正利用対応 / お知らせ配信を提供する内部運用コンソール。利用者システム(メインシステム)とは URL も DB も分離され、独立した認証・認可基盤を持つ。

### 2.2 目的

- 利用者システムの監視・運用・トラブル対応
- 課金関連の Webhook 一次受信(Stripe → 顧客管理 → メイン転送)
- 内部統制(4-eyes 原則 + 監査ログ全件 + ハッシュチェーン改ざん検知)
- 法令対応(APPI / GDPR / 電子帳簿保存法 / 国税通則法)

### 2.3 対象ユーザー

| ユーザー種別 | 概要 |
|---|---|
| サービス運営者 | 本 SaaS 提供者の自然人 1 名につき 1 アカウント(`service_operator` 単一ロール、FR-220)|

### 2.4 対象業務(運営者スコープ)

- 削除データ参照 / 復元(SCR-090 / SCR-091)
- AI 推論パラメータ管理(SCR-092、ハードゲート)
- 契約別レート・予算上書き(SCR-093)
- お知らせ作成・配信(SCR-094)
- 運営者活動監査 / 監査ログエクスポート(SCR-096)
- Stripe Webhook リプレイ / DLQ(SCR-097)
- PII 誤検出報告管理(SCR-098)
- Webhook ペイロード差分検出(SCR-099)
- 4-eyes 申請・承認(SCR-APPROVALS / M1 / M2)
- 月次請求確定 cron 監視 / 訂正請求

### 2.5 SaaS としての提供形態

運営者システムは独立したサブドメイン(`https://admin.open-faq.example.com`)で提供。利用者システムからは到達不可。

- **URL 分離**: 利用者側 `app.open-faq.example.com` ≠ 運営者側 `admin.open-faq.example.com`
- **DB 分離**: 運営者専用 D1(`operator_*` テーブル群)+ 共有 D1(`audit_logs` / `webhook_events` / `accounts_retired`)
- **認証分離**: `service_operators` テーブル(利用者の `accounts` とは別)+ MFA 必須 + IP allowlist

## 3. システム全体構成

### 3.1 システム構成概要

メインシステムと同じ Cloudflare スタック:

| コンポーネント | 役割 |
|---|---|
| Cloudflare Workers | 運営者管理 API + Stripe Webhook 一次受信 + cron |
| Cloudflare D1 | 運営者 DB + 共有 `audit_logs` 等 |
| Cloudflare KV | IP allowlist キャッシュ(60s)+ ハードゲート KV |
| Cloudflare R2 | 監査ログアーカイブ + DLQ 退避(30 日)|
| Cloudflare Queues | Webhook 内部転送 + 月次請求確定 |
| Cloudflare Secrets Store | Master Key / Webhook Secret / 内部 API JWT 鍵 |
| Stripe | 課金決済(Webhook 一次受信)|
| Resend | 運営者通知メール |

### 3.2 アーキテクチャ方針

- **6 段認可判定**(IP allowlist → セッション → MFA → role → 再認証 → 4-eyes)
- **4-eyes 承認フロー**(ハードゲート 3 + 承認ログ 7、自己承認禁止、payload_hash 改ざん検知)
- **ハッシュチェーン監査**(`record_hash = SHA-256(prev_hash || canonical_json)`、JCS RFC 8785 準拠、日次完全性検証)
- **Stripe Webhook 一次受信 + ペイロード差分検出**(冪等性 + 自動 BO + DLQ 30 日リプレイ)

### 3.3 利用技術(運営者固有)

| カテゴリ | 採用技術 |
|---|---|
| MFA | TOTP RFC 6238、HMAC-SHA1、6 桁、30 秒 |
| 認証ハッシュ | Argon2id 運営者強化プロファイル(`m=128MB, t=4, p=4, salt=16B`)|
| Stripe 署名検証 | HMAC-SHA256(Webhook Secret、90 日ローテーション)|
| 監査ハッシュチェーン | SHA-256(`record_hash = SHA-256(prev_hash || canonical_json)`)|
| 正規化 | JCS(JSON Canonicalization Scheme、RFC 8785)|

### 3.4 オーナー境界によるデータ分離方式の運営者側影響

運営者は **オーナー境界を適用しない**(全契約横断アクセス)。代わりに以下のガード:

1. IP allowlist
2. MFA
3. 再認証(5 分以内、操作 1 回限り)
4. ロール検証(`service_operator`)
5. 4-eyes 承認(10 操作のみ)
6. 監査ログ全件(`audit_logs.retention_class='5y'` または `7y`)

## 4. 法令対応スコープ

| 法令 | 影響箇所 | 物理化方針 |
|---|---|---|
| APPI 30 条(不適正利用禁止)| 監査ログ閲覧自体の記録 | `audit.search` action コード(retention_class='1y')|
| APPI 33 条(開示請求)| 運営者手動対応 | `inbox_messages` で通知 |
| GDPR 越境移転規制(46 条)| データリージョン管理 | MVP は APAC リージョン固定。EU 顧客は §4.1 |
| 国税通則法・法人税法施行規則 | 課金エビデンス 7 年保持 | `audit_logs.retention_class='7y'`、`billing_invoices` 7 年保持 |
| 電子帳簿保存法(真実性)| 取引データ改ざん防止 | 監査ログハッシュチェーン |
| 電子帳簿保存法(可視性)| 3 検索要件 | SCR-096 で `billing_invoices` 検索 UI |
| 改正電気通信事業法 | 外部送信規律 | SCR-094 で外部送信先一覧 |
| 特定電子メール法 | 運営者発通知 | メール本文末尾に法人名・住所・問合せ先・配信停止 URL |

### 4.1 GDPR 越境移転規制への MVP 運用ガード

EU/EEA/UK 顧客の検知時のワークフロー:

- **検知点**: 契約登録フォームで「事業所所在地」「主要顧客所在地」必須入力
- **判定基準**: 国コード ISO 3166-1 alpha-2 ホワイトリスト(EU 27 + EEA 3 + UK)
- **動作**: `owner_registration_reviews.status='pending_legal_review'` に申込情報を保存 + 運営者 inbox(legal/high)通知 + SCR-002 完了画面に「審査中」表示
- **4-eyes 承認**: 法務確認書面の事後添付と監査ログ記録を必須
- **同意取得書面**: 「日本データセンター処理同意書」を別途締結
- **監査ログ**: `owner.legal_review.{detect, approve, reject}`(retention_class='5y')
- **Runbook**: RB-023

詳細は [個別設計書群/10_セキュリティ設計書.md §10](個別設計書群/10_セキュリティ設計書.md) を正本とする。

## 6. 設計方針

### 6.1 6 段認可判定方針(本書の中核)

```
リクエスト到着 → 1. IP allowlist → 2. セッション → 3. MFA → 4. service_operator role → 5. 再認証(5 分以内) → 6. 4-eyes 承認
```

詳細フロー(Mermaid 図含む)は [個別設計書群/09_認証認可設計書.md §7.3](個別設計書群/09_認証認可設計書.md) を正本とする。

### 6.2 4-eyes 承認フロー方針

- ハードゲート 3 操作: 契約物理削除 / AI パラメータ変更 / マスター鍵ローテーション
- 承認ログ 7 操作: 契約無効化 / 復元 / 単価改定 / レート上書き / 強制停止 / 削除データ復元 / ハードゲート KV 切替
- 自己承認禁止 / 自己却下禁止(DB CHECK + アプリ層チェック)
- payload_hash 改ざん検知(SHA-256)
- 承認待ち TTL 72h + 承認後実行 TTL 72h
- ハードゲート KV `feature:hard-gate:<action_code>`(緊急一時無効化のみ)

詳細は [個別設計書群/09_認証認可設計書.md §8](個別設計書群/09_認証認可設計書.md) と [個別設計書群/05_権限設計書.md §7](個別設計書群/05_権限設計書.md) を正本とする。

### 6.3 監査ハッシュチェーン方針

- `record_hash = SHA-256(prev_hash || canonical_json(本レコード))`
- `canonical_json` = JCS(JSON Canonicalization Scheme、RFC 8785)準拠
- 日次バッチ `AuditChainVerifierWorker`(02:00 JST、全件再計算)
- 不一致検出時: 運営者 inbox(`critical`)+ メール + 検証ログ
- tombstone 方式(保持期間経過後の削除でもチェーン連続性維持)
- 3 区分保持(`1y` / `5y` / `7y`、retention_class)

詳細は [個別設計書群/10_セキュリティ設計書.md §7](個別設計書群/10_セキュリティ設計書.md) を正本とする。

### 6.4 Stripe Webhook 一次受信方針

- 署名検証: HMAC-SHA256 + トレランス 300s(Replay 攻撃防止)
- 冪等性: Stripe `event_id` を `webhook_events.event_id` PK
- ペイロード差分検出: JCS 正規化 + 除外フィールド削除 + SHA-256(不一致時は自動上書き禁止)
- DLQ: Cloudflare Queues 4 日 + R2 退避 30 日
- 自動 BO: 1m → 4m → 16m(3 回、超過で永久失敗)
- 手動リプレイ: SCR-097 から運営者の明示操作のみ(4-eyes 承認ログ)

詳細は [個別設計書群/11_課金請求設計書.md §13](個別設計書群/11_課金請求設計書.md) を正本とする。

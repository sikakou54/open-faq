# エラー設計書(運営者)

## 1. 文書概要

### 1.1 目的

運営者システムで発生するすべてのエラーについて、分類 / 発生条件 / システム動作 / ユーザー表示 / 復旧方法 / ログレベルを一元化する。プレフィックスは `E-OP-*` で、メイン側 `E-*` と識別する。

### 1.2 対象範囲

- 対象: 運営者向けエラー(認証 / MFA / 認可 / 4-eyes / IP allowlist / 再認証 / Webhook / 業務 / 連携 IF)

## 2. エラー分類

| エラー分類 | 概要 | プレフィックス | 代表例 |
|---|---|---|---|
| 認証エラー | 認証失敗・期限切れ | `E-OP-AUTH-*` | パスワード不一致 / セッション期限切れ |
| MFA エラー | MFA 関連 | `E-OP-MFA-*` | TOTP 失敗 / セットアップ期限切れ / 回復コード使用済み |
| IP allowlist エラー | IP 許可リスト拒否 | `E-OP-IP-*` | 未許可 IP からのアクセス |
| 再認証エラー | 5 分以内再認証 | `E-OP-REAUTH-*` | 再認証期限切れ |
| 4-eyes エラー | 4-eyes 関連 | `E-OP-4EYES-*` | 未承認 / 自己承認 / ハードゲート KV false / TTL 切れ / payload_hash 改ざん |
| 認可エラー | 権限不足 | `E-OP-AUTHZ-*` | (運営者では稀、主に 4-eyes 未承認)|
| 入力エラー | バリデーション失敗 | `E-OP-INPUT-*` | ticket_id 形式不正 / 必須項目欠落 |
| 業務エラー | ビジネスルール違反 | `E-OP-BIZ-*` | 削除済み契約の操作 / 既復元データの再復元 |
| Webhook エラー | Webhook 関連 | `E-OP-WEBHOOK-*` | 署名検証失敗 / 冪等性違反 / 差分検出 / リプレイ枠超過 |
| PII 報告エラー | PII 誤検出 | `E-OP-PII-*` | 報告ステータス不正 |
| 復元エラー | 削除データ復元 | `E-OP-RESTORE-*` | 復元期限切れ / 423 Locked / 既復元 |
| 連携 IF エラー | 連携 IF 関連 | `E-OP-IF-NNN-*` | 送信失敗 / 認証失敗 |
| システムエラー | サーバー内部エラー | `E-OP-SYS-*` | DB エラー / 内部例外 |

## 4. 異常系共通方針

### 4.1 IP allowlist 拒否

- **401 即時応答**(画面遷移なし、HTTP のみ)
- レスポンスボディには詳細を含めない(IP 範囲列挙攻撃対策)
- ログには発信元 IP を記録(セキュリティ設計書 10 §7、`audit_logs.action='auth.ip_blocked'`)

### 4.2 MFA エラー

- TOTP 失敗 5 回で `(IP × operator_id)` を **15 分ロックアウト**
- 回復コードは 1 回限り使用、使用済みは無効化
- **MFA リセットは自己リセット禁止**(他運営者承認必須)

### 4.3 再認証期限切れ

- **403 + 再認証画面誘導**(再認証モーダル表示)
- 再認証成功後、元の操作を再実行可能
- 残時間 5 分カウントダウンを ReauthCountdown で常時表示

### 4.4 4-eyes 未承認(本書の中核)

- 4-eyes 対象 10 操作の API 直接呼び出しは **403 統一**
- 「この操作には承認が必要です。承認待ち一覧から申請してください」を返却
- 4-eyes ハードゲート KV false の場合は別エラーコード(`E-OP-4EYES-HARDGATE`)
- 監査: `audit_logs.action='4eyes.approval.required'`

### 4.5 自己承認 / 自己却下

- **422 + 専用コード**
- 申請者と承認者が同一 operator_id の場合、`operator_approvals.CHECK` 制約 + アプリ層の二重チェック
- 監査: `audit_logs.action='4eyes.self_approval_rejected'`

### 4.6 承認 TTL 切れ

- **410 + 専用コード**(承認可能時間 72h 経過後)
- 再申請を促す
- 自動状態遷移 `expired`

### 4.7 payload_hash 改ざん検知

- **409 + 専用コード**(承認時の payload と実行時の payload が異なる)
- 監査: `audit_logs.action='4eyes.payload_mismatch'`
- 「申請内容が変更されている可能性があります」を表示し、最新で再申請

### 4.8 Webhook 関連

- **署名検証失敗**: 401 + 監査ログ記録(全件)
- **冪等性違反**: 200 + 既存処理結果(リトライ安全性)
- **ペイロード差分検出**: 200 受信 + SCR-099 で表示 + 運営者通知 `high` + **自動上書き禁止**

### 4.9 復元エラー

- `accounts_retired` 未存在: 404 + 「復元可能なデータが見つかりません」
- **deletion-queue 競合**: **423 Locked** + 「他の運営者が処理中のため復元できません」
- 既復元データ: 422 + 「既に復元済みです」
- プリチェックで確認が必要: 422 + 詳細表示

### 4.10 連携 IF エラー(送信側)

- 送信側(運営者 → メイン)失敗: DLQ 投入、自動リトライ(指数バックオフ 1m → 4m → 16m)
- 3 回失敗で運営者通知(`high`)
- 受信側(メイン → 運営者)エラー: 200 + エラーログ記録(リトライさせない)

メッセージ一覧(運営者)

1. 文書概要

1.1 目的

運営者システムで運営者および IF #12 経由で利用者に表示するすべての文言(画面ラベル / プレースホルダー / ボタン / 説明文 / エラー / 成功 / 警告 / 確認 / 空状態 / トースト / お知らせ・運営者操作通知テンプレート)を一元管理する。

1.2 対象範囲

  • 対象: SCR-090〜098 / SCR-AUTH / SCR-AUTH-M1 / SCR-HOME / SCR-APPROVALS / M1 / M2 の全画面文言、運営者向け通知(4-eyes / Webhook / 監査 / PII / DLQ)、IF #12 経由で利用者へ送られる運営者操作通知テンプレート、4-eyes 関連メッセージ
  • 対象外: 利用者向けメッセージ(01_メインシステム/個別設計書群/07_メッセージ一覧.html 参照)

2. 共通基準

メイン側 メッセージ一覧 §2 の共通基準を踏襲する。運営者固有の補足:

2.1 運営者向け文言の特徴

  • 専門用語(4-eyes, retention_class, payload_hash, ticket_id, DLQ, hardgate 等)は必要な場面に限って使い、画面表示では補足または業務語を併記する
  • 4-eyes 関連は「申請」「承認」「却下」「実行」「取下げ」を統一語彙とし、ボタンは「復元する」「配信する」など対象操作を明示する
  • 操作対象を明示({ContractName} / {operator_id} / {action_code} 等プレースホルダー多用)
  • セキュリティ理由から具体情報を伏せる場合あり(IP allowlist 拒否時の許可範囲非表示等)

2.2 4-eyes 用語

用語 説明
申請(request) 4-eyes 対象操作の最初の起票
承認(approve) 別運営者による承認
却下(reject) 別運営者による却下
取下げ(withdraw) 申請者本人による取下げ
実行(execute) 承認後の運用実行
ハードゲート(hardgate) KV フラグによる動的拒否
操作チケット ID(ticket_id) 4-eyes 申請時に必須入力する内部チケット
ペイロードハッシュ(payload_hash) 申請内容の改ざん検知用 SHA-256

2.3 通知重要度ルール(共有概念、メイン側 §2.4 参照)

announcement_drafts.severity の 3 値(low / normal / high)は内部運用用。利用者向けには通知重要度 4 値(low / normal / high / critical)と対応付ける。critical 通知は別建ての緊急通知パス(基本設計書 §9.2 / メイン側 10_セキュリティ設計書.html §11)で管理。

3. メッセージID 体系

メイン側 §3 を踏襲。SCR ID 部分は運営者画面 ID(SCR-090 / SCR-AUTH / SCR-APPROVALS-M1 等)を使用。運営者向け接頭辞は MSG- をそのまま使う(プレフィックス分離はなし)。エラー側は E-OP-* プレフィックスでメインと識別。

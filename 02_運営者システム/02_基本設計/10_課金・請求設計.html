課金・請求設計書(運営者)

1. 文書概要

1.1 目的

運営者側の課金・請求機能(Stripe Webhook 一次受信 / DLQ 30 日リプレイ / 月次請求確定 cron / 請求書 7 年保持 / Webhook ペイロード差分検出)を一元化する。

1.2 対象範囲

  • 対象: Stripe Webhook 一次受信処理、DLQ 管理、月次請求確定、請求書 7 年保持、Webhook ペイロード差分検出、契約別上書き(運営者操作)
  • 対象外: 利用者向け課金 UI(プロジェクト概要 / 料金・請求 / 退会申請)── メイン 10_課金・請求設計.html 参照、契約モデル(単一プラン定義はメイン側正本)

2. 契約モデル(メイン側参照)

MVP は単一プラン(完全従量課金 + 月次無料枠、無料トライアル期間なし)。詳細は メイン側 課金・請求設計書 §2 を正本とする。

3. 利用設定の運営者上書き(レート=契約単位 / 質問数上限・無料枠=プロジェクト単位)

運営者が SCR-093 から利用設定を上書きする。レート制限は契約単位、質問数上限と課金対象別無料枠はプロジェクト単位。FAQ 件数・個別チャット部屋数の上限は提供しない。

3.1 レート制限上書き(契約単位、owner_quota_overrides)

resource_kind 用途 主管 SCR 4-eyes 種別 KV ミラー
widget_ask_per_min ウィジェット質問レート SCR-093 承認ログ あり(60s TTL)
widget_inquiry_per_min ウィジェット未解決質問送信レート SCR-093 承認ログ あり
widget_chat_per_min ウィジェットチャット投稿レート SCR-093 承認ログ あり
email_per_hour メール送信レート SCR-093 承認ログ あり
admin_api_per_min 管理 API レート SCR-093 承認ログ あり

3.2 質問数上限・課金対象別無料枠上書き(プロジェクト単位、project_quota_limits)

PUT /admin/v1/overrides/usage-limit/{owner_id}/{project_id} で対象プロジェクトに上書きする。source='operator'、reason / validUntil / 運営者 ID を監査記録。KV ミラーは usage-limit:<owner>:<projectId>(30s TTL)。

resource_kind 用途 主管 SCR 4-eyes 種別
question_monthly_limit 質問数の月次上限件数・無料枠 SCR-093(プロジェクト選択) 承認ログ
faq_monthly_limit FAQ 件数の無料枠(上限なし) SCR-093 承認ログ
chat_room_monthly_limit 個別チャット部屋数の無料枠(上限なし) SCR-093 承認ログ

詳細は メイン 03_テーブル設計.html §3.23 / §3.24 参照。

4. 契約状態(メイン側参照)

contract_owners.contract_status 4 値はメイン側を正本とする(01_メインシステム/個別設計書群/11_課金請求設計書.html §4)。運営者側は連携 IF #1 経由で遷移をトリガーする。

遷移 トリガー 連携 IF 運営者画面
active → suspended(支払い失敗確定) Stripe Smart Retries 3 回失敗 IF #1 送信 -
suspended → active(再決済成功) Stripe Webhook 受信 IF #1 送信 -
suspended → active(運営者復帰) 4-eyes 承認 IF #1 送信 SCR-091 派生
active / suspended → deleted_pending(退会申請) 利用者操作 - -
deleted_pending → deleted(30 日経過) 月次 cron IF #4 送信 -

支払方法ゲート(FR-136、支払方法未登録 + 無料枠超過)は 契約状態を変更しない(active のままウィジェット質問受付のみ停止)。運営者側の関与・通知はなく、メイン側で完結する。

13. Webhook 処理(本書の中核、共有概念正本)

13.1 Stripe Webhook 一次受信(運営者側 → メイン転送)

(§6.3.3 / §8.5 から移管)

項目 内容
エンドポイント POST https://admin.open-faq.example.com/webhooks/stripe(運営者側 API)
ヘッダ Stripe-Signature: t=<unix_ts>,v1=<hmac_sha256>,v1=<hmac_sha256_alternative> / Content-Type: application/json
署名検証 Stripe-Signature ヘッダで HMAC-SHA256 検証(トレランス 300s)
冪等性 Stripe イベント ID(evt_*)で重複排除、webhook_events.event_id PK
保存 webhook_events テーブルに原文 + state(received)
状態遷移 received → verifying_signature → checking_idempotency → processing → succeeded / failed → dlq_*(リトライ失敗時)
メイン転送 IF #10 経由でメイン側へ転送
保持期間 7 年(電子帳簿保存法対応)
タイムアウト 30 秒(超過で 5xx 返却 → Stripe Smart Retries)

署名検証アルゴリズム

verifyStripeSignature(payload, header, secret, tolerance=300s):
  parts = parseHeader(header)
  timestamp = parts.t
  if |now - timestamp| > tolerance:
    return false  // Replay 攻撃防止
  signedPayload = timestamp + "." + payload
  expectedSig = HMAC-SHA256(secret, signedPayload)
  for sig in parts.v1[]:
    if constantTimeEqual(sig, expectedSig):
      return true
  return false

失敗時: HTTP 401 即返却 + 運営者 inbox(system/high)+ audit.webhook.signature_invalid(5y)

冪等性・差分検出

項目 仕様
同 event_id + ハッシュ一致 200 OK「duplicate, skipped, hash_match」+ ログ
同 event_id + ハッシュ不一致 200 OK + webhook_payload_diffs INSERT + SCR-098 記録 + 運営者 high alert + 自動上書き禁止

内部転送対象イベント(メイン側へ mTLS + JWT 同期)

event_type 処理 冪等性キー
invoice.paid 支払成功確定 → サスペンション解除 Stripe event_id
invoice.payment_failed 支払失敗 → grace 突入 Stripe event_id
invoice.finalized 請求書確定 Stripe event_id
customer.subscription.updated プラン変更反映 Stripe event_id
customer.subscription.deleted 解約 → suspended 遷移 Stripe event_id
charge.refunded 返金 → 請求書更新 Stripe event_id

13.2 Webhook ペイロード差分検出(§6.7 から移管)

項目 内容
目的 Stripe API 仕様変更 / 改ざんの早期検出
検出方式 過去ペイロードと正規化後 SHA-256 比較、差分があれば webhook_payload_diffs に記録
正規化方式 JCS(JSON Canonicalization Scheme、RFC 8785)準拠 + 除外フィールド削除 + UTF-8 SHA-256(セキュリティ設計書 10 §15 参照)
除外フィールド(付録 H、8 項目) created / request.id / request.idempotency_key / livemode / data.object.created / data.object.metadata.stripe_request_id / data.object.latest_charge.id / api_version

監査 audit_logs.action='webhook.payload_diff_detected'

13.3 DLQ 30 日リプレイ

項目 仕様
Cloudflare Queues 保持 4 日
R2 退避パス dlq-stripe-events/<event_id>.json
R2 保持期間 30 日(dlq_archived で以降リプレイ不可)
自動 BO 1m → 4m → 16m(3 回、超過で永久失敗)
手動リプレイ SCR-096 から運営者の明示操作のみ(4-eyes 承認ログ、action: webhook.replay)
リプレイ履歴 dlq_replay_log に保存
エラー処理 リプレイ失敗時は再度 DLQ、最大 3 回でアラート(high)

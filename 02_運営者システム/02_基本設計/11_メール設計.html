メール設計書(運営者システム)

1. 文書概要

1.1 目的

本書は 運営者システム(SaaS 提供事業者向け管理コンソール)が送信する全電子メールの件名・本文・テンプレート変数・送信契機を集中定義する正本 である。実装者・QA・運営オペレーターが本書の文言と差分なくメール内容を確認できることを保証する。

1.2 対象範囲

  • 運営者システムから自動送信される全メール(本書 §4 のテンプレート ID TPL-OP-* 全件)
  • 各テンプレートの件名 / プレーンテキスト本文 / HTML 本文(プレースホルダ含む)
  • 送信契機(通知 ID NOTIF-OP-*)/ 配信先 / 重要度 / 集約窓 / 失効・リンク有効期限
  • IF #12 経由で 利用者側(メイン契約管理者)に届くメール も対象に含める(運営者操作の通知主体は運営者システムのため)

対象外:

  • 運営者 inbox 専用通知の文言詳細(運営者画面内表示は 06_メッセージ一覧.html §8.4 を正本とする)
  • メインシステムが自発的に送信するメール(../../01_メインシステム/02_基本設計/11_メール設計.html)

2. 共通基準

2.1 送信元 / Reply-To / 差出人表記

項目 値
Envelope From bounces+<deliveryId>@ops.<service-domain> (VERP)
Header From open-faq Operations <ops-noreply@<service-domain>> 固定
Reply-To ops-team@<service-domain>(運営者宛通知のみ。IF #12 経由の利用者宛通知では設定しない)
DMARC / DKIM / SPF 全テンプレートで pass を必須

2.2 件名規則

  • 件名長: 半角換算 70 文字以内
  • 重要度プレフィックス:
    • critical: [重大] または [緊急](TPL-OP-FRAUD-DETECTED のみ [緊急])
    • high: [警告]
    • normal: [通知]
  • 4-eyes 申請関連は [承認待ち] / [承認完了] プレフィックスを用いる
  • 利用者宛通知(IF #12 経由)は重要度プレフィックスを付けず、メイン側メールと混在しても違和感のないフラットな文面とする

2.3 本文構造(運営者宛通知)

  1. 重要度マーカーと事象タイトル
  2. 一段落の状況説明
  3. メタデータ表(検知時刻 / 対象範囲 / 関連 ID 等)
  4. 推奨対応(Runbook URL 等)
  5. 関連画面 deeplink(SCR-* URL)
  6. 共通フッタ(返信不可注意、配信元署名)

2.4 本文構造(IF #12 経由 利用者宛通知)

  1. 宛名({recipient_display_name} 様)
  2. 運営者が当該操作を実行した旨の告知
  3. 対象と影響範囲(時刻 / 対象データ / 復元・解除手順)
  4. 利用者側で取れるアクション(該当画面 deeplink)
  5. 共通フッタ(運営者問い合わせ窓口 URL)

2.5 文字コード / 言語 / 時刻

  • 文字コード: UTF-8
  • HTML 版とテキスト版を両方生成し、multipart/alternative で送信
  • 全文 i18n キー化必須。MVP 対応言語: 日本語(ja)
  • 本文内の時刻表記は JST 固定 で YYYY-MM-DD HH:mm:ss(運営者宛は秒まで記載、利用者宛は分まで)

2.6 サニタイズ

  • 監査ログ・Webhook ペイロード・払い戻し payload 等の 生データ埋め込みは禁止。本文に含めるのは要約のみとし、詳細は SCR-* deeplink から運営者管理画面で確認する
  • 利用者宛通知では契約 ID(contract_owners.user_id)等の内部識別子も埋め込まない

2.7 配信信頼性

  • 一時失敗時の指数バックオフ再送、恒久失敗時の notification_logs.delivery_status='bounced' 記録は メイン側 §2.6 と同じ
  • critical 通知は配信失敗時に PagerDuty / Slack #ops-incidents への自動エスカレーションを併用(NFR-808、§9.1 障害対応)

4. テンプレート詳細

各テンプレートは以下のセクション構造で記載する:

  • 発火条件: どの操作・イベントで送信されるか
  • テンプレート変数: 本文中の {...} 変数
  • 件名: 件名(JA)
  • 本文(テキスト版): プレーンテキスト本文
  • 本文(HTML 版): HTML 版の構造概要
  • 備考: 注意事項

4.1 TPL-OP-4EYES-REQUEST(4-eyes 申請発生)

発火条件: 運営者が 4-eyes 対象操作(契約復元 / 契約停止 / レート上書き / AI パラメータ変更 / ウィジェット強制停止 等)を申請した時。承認待ちの全運営者へ送信。

{payload_preview} payload プレビュー(運営者 inbox 表示と同等の差分要約、200 字)
{approval_url} SCR-APPROVALS 詳細 URL
{request_at_jst} 申請日時(JST、秒まで)
{deadline_at_jst} 承認期限(JST、申請 + 48h)

件名: [承認待ち] {action_code} の承認依頼

本文(テキスト版):

運営者各位

4-eyes 承認待ちの申請が発生しました。

  - 操作: {action_label}({action_code})

  - 対象: {target_summary}
  - 申請日時: {request_at_jst}(JST)

{payload_preview}

5. 配信運用

5.1 配信先解決ロジック

通知 ID 配信先解決
運営者宛全テンプレート(TPL-OP-* の §3.1 全件) operator_accounts テーブルから disabled_at IS NULL の全運営者(TPL-OP-4EYES-APPROVED は申請者本人のみ、TPL-OP-ANN-TEST はテスト送信実行者のみ)
TPL-OP-IF12-* 対象 contract_owners.user_id 配下のオーナー + 同一契約スコープ(project_users.contract_owner_user_id)で project_users.role='admin' を 1 件でも保持するユーザー

5.2 集約窓(IF #12 経由 利用者宛通知)

  • FR-211 に従い、10 分以内の連続イベントは 1 通にまとめる(運営者が短時間に複数操作した場合、利用者への通知が乱発するのを防ぐ)
  • 集約時の件名は [まとめ] N 件の運営者操作通知、本文は時系列で操作ログを列挙

5.3 重要度別の強制送信ルール

  • critical: 受信オプトアウト不可、必ずメール送信。TPL-OP-HASH-CHAIN-NG / TPL-OP-SIG-FAIL / TPL-OP-FRAUD-DETECTED / TPL-OP-IF12-SUSPEND / TPL-OP-IF12-WIDGET-FORCE-STOP
  • high: 強制送信(運営者宛は inbox 通知も併用)
  • normal: 強制送信(運営者は MVP では個別オプトアウト不可)

5.4 配信ログ

  • 全送信は operator_notification_logs / notification_logs(IF #12 経由分)に記録
  • 列: メイン側 §5.4 と同一構造 + tenant_contract_owner_user_id(IF #12 経由分のみ)
  • 保持期間: 運営者宛 = 1 年(operator_high_priv retention class)、IF #12 経由 = 90 日

5.5 テスト送信

  • 開発・ステージング環境では MAIL_SAFE_SINK_OPS 環境変数で全宛先を ops-mail-sink@<internal-domain> に強制リダイレクト
  • 件名にプレフィックス [STAGING-OPS] / [DEV-OPS] を自動付与

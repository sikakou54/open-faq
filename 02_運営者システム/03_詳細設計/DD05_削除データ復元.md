## 3. 詳細設計本文

### 3.1 削除データのライフサイクル(D-07)

| `contract_owners.contract_status` | 説明 | 物理データ |
|---|---|---|
| `active` | 通常稼働 | `contract_owners` テーブル本体(`users` + `contract_owners` + `project_users` で構成)|
| `suspended` | 一時停止(運営者 `owner.suspend` 経由) | 本体保持、機能フラグで停止 |
| `deleted_pending` | 退会受付済、猶予期間中(30 日) | 本体保持、復元可能 |
| `deleted` | 物理削除済 | 運営者側 `accounts_retired` テーブルへ証跡 INSERT、`contract_owners` / `users` / `project_users` 本体は DELETE |

復元可能範囲:

- `suspended` / `deleted_pending` ── 通常復元(`POST /restorations`、承認ログ対象)
- `deleted` ── 物理削除後の復元は **不可**(`accounts_retired` は法令対応で永久保持されるが、本体への戻しは MVP では不可)

### 3.2 復元時の副作用ロールバック(FR-204、D-07)

退会・無効化処理で行った 7 種類の副作用を、復元時に逆順でロールバックする:

| 副作用 ID | 削除時の処理 | 復元時のロールバック |
|---|---|---|
| (a) | 全プロジェクト `archived` 遷移 | `archived` → `active` 戻し(削除前の状態に基づく) |
| (b) | 全 FAQ `unpublished` 遷移 | `unpublished` → 削除前の状態に基づく |
| (c) | 全ユーザー(管理者・メンバー)の `users.valid=0`(論理削除)| `valid=1` 戻し |
| (d) | API 鍵全失効 | 旧 API 鍵は再発行せず、復元後に新規発行を促す通知 |
| (e) | Webhook 配信先全停止 | 旧 Webhook URL を `disabled` 解除 |
| (f) | Stripe Subscription `cancel_at_period_end=true` | `POST /v1/subscriptions/{id}` で `cancel_at_period_end=false` |
| (g) | 通知配信停止 | inbox 通知再開 + 復元完了メール送信 |

ロールバックは **全 7 ステップを単一トランザクション** で実行し、途中失敗時は全件ロールバック + `restorations.state='failed'` 遷移 + 運営者 high alert。

### 3.3 復元実行 API

```yaml
/restorations:
  post:
    summary: 削除データ復元
    parameters:
      - $ref: "#/components/parameters/XOpTicketId"
      - $ref: "#/components/parameters/XApprovalId"
      - $ref: "#/components/parameters/IdempotencyKey"
      - $ref: "#/components/parameters/XCsrfToken"
    requestBody:
      required: true
      content:
        application/json:
          schema:
            type: object
            required: [resourceType, resourceId, reason]
            properties:
              resourceType:
                type: string
                enum: [owner, project, faq, account, announcement]
              resourceId: { type: string }
              reason: { type: string, maxLength: 1000 }
    responses:
      "200":
        content:
          application/json:
            schema:
              type: object
              properties:
                restorationId: { type: string }
                restoredAt: { type: string, format: date-time }
                rollback: { type: object }
      "423":
        content:
          application/problem+json:
            schema: { $ref: "#/components/schemas/Problem" }
```

`423 Locked` は復元対象が既に物理削除済(`accounts_retired` 移行済)で復元不可。

### 3.5 4-eyes 承認ログ対象(MVP)

`owner.restore_data` は **MVP Log Only / Beta Hard Gate**:

- MVP: 単独実行可能、承認ログ(`operator_approval.*`)を必ず残す + 事後監査
- Beta 以降: ハードゲート化、`X-Approval-Id` 必須

詳細は [DD02_4-eyes承認フロー.md](DD02_4-eyes承認フロー.md) を参照。

### 3.6 検索・一覧(SCR-090)

| 検索条件 | バリデーション |
|---|---|
| `resourceType` | `owner` / `project` / `faq` / `account` / `announcement` |
| `slug` 部分一致 | 1〜64 文字 |
| 削除日時範囲 | `from` ≤ `to`、最大 5 年遡及 |
| `contract_status` | `suspended` / `deleted_pending` / `deleted`(`deleted` は閲覧のみ、復元不可) |

検索結果は `(deleted_at, id)` カーソルページング(`limit` 既定 100、最大 200)。検索自体も監査(`audit.search`、1y)。

### 3.7 副作用 (a)〜(g) ロールバック詳細

#### (a) プロジェクト復元

```sql
-- 削除前の状態を deleted_projects から取得し、projects に戻す
INSERT INTO projects (...)
SELECT ... FROM deleted_projects WHERE contract_owner_user_id = ?;
UPDATE projects SET valid = 1 WHERE contract_owner_user_id = ? AND updated_at > ?;
```

#### (b) FAQ 復元

削除前の `is_published` 状態を復元(削除時に `is_published=0` 強制したものは元の値に戻す)。

#### (c) 管理者ユーザー復元

```sql
-- メンバーは users.valid=0 で論理削除されているため valid=1 へ戻す
UPDATE users SET valid = 1
WHERE id IN (
  SELECT user_id FROM project_users WHERE contract_owner_user_id = ?
)
AND updated_at > ?;
```

#### (d) API 鍵

旧 API 鍵は再発行しない(セキュリティ上、退会時点で失効済の鍵を復活させない)。復元完了後に inbox 通知で「新規 API 鍵を発行してください」と促す。

#### (e) Webhook 配信先

```sql
UPDATE webhook_endpoints SET disabled = 0
WHERE contract_owner_user_id = ? AND disabled_at > ?;
```

#### (f) Stripe Subscription

```text
const subscription = await stripe.subscriptions.retrieve(owner.stripe_subscription_id);
if (subscription.cancel_at_period_end === true) {
  await stripe.subscriptions.update(
    owner.stripe_subscription_id,
    { cancel_at_period_end: false },
    { idempotencyKey: `restore-${restorationId}-sub-resume` }
  );
}
```

#### (g) 通知配信再開

`contract_owners.notifications_disabled=0` に戻し、復元完了メール(`MSG-SCR-091-001`)を強制送信。

### 3.8 復元結果の記録

```sql
CREATE TABLE restorations (
  id TEXT PRIMARY KEY,
  resource_type TEXT NOT NULL CHECK (resource_type IN ('owner','project','faq','account','announcement')),
  resource_id TEXT NOT NULL,
  state TEXT NOT NULL CHECK (state IN ('executing','succeeded','failed')),
  rollback_summary TEXT,  -- JSON: (a)〜(g) 各ステップの結果
  requested_by TEXT NOT NULL,
  approval_id TEXT,
  reason TEXT NOT NULL,
  ticket_id TEXT NOT NULL,
  started_at TIMESTAMP NOT NULL,
  completed_at TIMESTAMP,
  FOREIGN KEY (requested_by) REFERENCES operator_accounts(id)
);
CREATE INDEX idx_restorations_resource ON restorations(resource_type, resource_id);
CREATE INDEX idx_restorations_started_at ON restorations(started_at);
```

完全な DDL は [基本設計 / テーブル設計](../02_基本設計/03_テーブル設計.md) を正本とする。

### 3.9 通知集約(FR-211、D-19)

復元完了通知は 10 分集約窓で運営者操作通知として送信:

| キー | 値 | TTL |
|---|---|---|
| `notify-batch:<contract_owner_user_id>:owner.restore_data` | `{ entries: [...], firstAt }` | 10 分 |

集約後、`OperatorNotifyAggregatorWorker`(1 分 cron で 10 分窓を判定)が連携 IF #12 経由でメインへ送信し、メインがオーナー + 全プロジェクト管理者(`project_users.role='admin'` を 1 件でも保持するユーザー)へ inbox 配信。

### 3.10 主要エラーコード

| エラー ID | HTTP | 説明 |
|---|---|---|
| `E-OP-RESTORE-001` | 423 | RESOURCE_PHYSICALLY_DELETED(`accounts_retired` 移行済) |
| `E-OP-RESTORE-002` | 409 | RESTORATION_IN_PROGRESS(同一リソースで `executing` 状態あり) |
| `E-OP-RESTORE-003` | 502 | STRIPE_RESUME_FAILED(副作用 (f) 失敗) |
| `E-OP-RESTORE-004` | 500 | ROLLBACK_TRANSACTION_FAILED(副作用 (a)〜(e) 失敗) |

完全な E-OP-RESTORE-* 一覧は [基本設計 / エラー設計](../02_基本設計/05_エラー設計.md) を正本とする。

## 5. テスト観点

### 5.1 ユニットテスト

- (a)〜(g) 各ステップのロールバック関数(個別単体)
- 422 / 423 / 409 各エラーケース
- 単一トランザクション内での全件成功 / 全件失敗 の保証
- `contract_status` 遷移ガード(`deleted` からは復元不可)

### 5.2 結合テスト(Miniflare)

- `suspended` → `active` 復元(全副作用ロールバック含む)
- `deleted_pending` → `active` 復元
- `deleted` 状態で 423 返却
- 復元中(`executing`)で同一リソースに再要求 → 409
- Stripe Subscription resume 失敗時の全件ロールバック

## 3. 詳細設計本文

### 3.1 状態遷移(`announcement_drafts.state`)

| From | To | トリガー |
|---|---|---|
| - | draft | POST /announcements |
| draft | preview | preview |
| preview | scheduled | schedule(再認証 + チケット必須) |
| scheduled | sending | AnnouncementSchedulerWorker(scheduled_at ≤ now+5min) |
| scheduled | cancelled | cancel(now < scheduled_at - 5min) |
| sending | sent | 連携 IF #7 200 OK |
| sending | failed | 連携 IF #7 一時失敗(5xx/Timeout)、`attempt_count++` |
| failed | sending | 自動指数 BO(1m → 4m → 16m、最大 3 回) |
| failed | dlq | 自動 BO 3 回失敗、運営者 inbox high |

詳細は [DD11_状態遷移詳細.md](DD11_状態遷移詳細.md) §B.4 を参照。

### 3.3 二段階 HTML サニタイズ(R-011)

| ステージ | タイミング | 対象 | 拒否条件 |
|---|---|---|---|
| 永続化前 | `POST /announcements` 受信時 | `bodyHtml` | 許可タグ外、`<script>` 含む、`href="javascript:"` 等 |
| 表示時 | メイン側 inbox 表示時 | 保管済 HTML | 同上(防御深層) |

#### 3.3.1 許可タグホワイトリスト

```text
a, b, br, code, em, i, li, ol, p, pre, span, strong, ul
```

#### 3.3.2 許可属性ホワイトリスト

```text
a: href, target="_blank", rel="noopener noreferrer"
span: class (限定: "highlight" のみ)
```

#### 3.3.3 拒否されるパターン例

- `<script>...</script>` ── タグ拒否
- `<img src=...>` ── タグ拒否(画像は MVP では非対応)
- `<a href="javascript:...">` ── 属性値スキーム拒否
- `<a href="data:...">` ── 同上
- HTML コメント `<!-- -->` ── 全削除

#### 3.3.4 拒否時のレスポンス

```json

  "type": "https://admin.open-faq.example.com/errors/E-OP-ANN-001",
  "title": "HTML サニタイザ拒否",
  "status": 422,
  "code": "E-OP-ANN-001",
  "detail": "Disallowed tag: <script>",
  "trace_id": "01J..."
}
```

### 3.4 予約配信スケジューラ(D-09)

```text
function announcementSchedule():
    candidates = D1.query("SELECT id FROM announcement_drafts WHERE state='scheduled' AND scheduled_at <= ? LIMIT ?", now + 5min, KV.get("feature:announcement-batch-size") || 100)
    for c in candidates:
        D1.exec("UPDATE announcement_drafts SET state='sending' WHERE id=? AND state='scheduled'", c.id)
        try:
            result = callMainIF(7, {announcementId: c.id, ...})
            if result.ok:
                D1.exec("UPDATE announcement_drafts SET state='sent', sent_at=? WHERE id=?", now, c.id)
                audit_logs(announcement.send, 5y, payload={announcement_id: c.id, recipients: result.recipients})
            else:
                handleRetry(c.id)
        catch e:
            handleRetry(c.id)
```

| 項目 | 仕様 |
|---|---|
| cron 周期 | `* * * * *`(毎分) |
| 早期判定窓 | `scheduled_at ≤ now + 5min`(早送りで境界跨ぎを吸収) |
| 最大件数 | KV `feature:announcement-batch-size`(既定 100、運営者設定可) |
| 取消可能期限 | `now < scheduled_at - 5min` |
| 自動 BO | 1m / 4m / 16m、最大 3 回(`failed` → `sending` 再試行) |
| DLQ 遷移 | 3 回連続失敗で `dlq`、運営者 inbox `high` 通知 |

### 3.5 連携 IF #7 JSON Schema(本書 → メイン)

```json
{
  "type": "object",
  "required": ["announcementId", "kind", "severity", "scope", "subject", "bodyHtml"],
  "properties": {
    "announcementId": { "type": "string" },
    "kind": { "type": "string", "enum": ["announcement", "system"] },
    "severity": { "type": "string", "enum": ["low", "normal", "high"] },
    "scope": {
      "type": "object",
      "required": ["type"],
      "properties": {
        "type": { "type": "string", "enum": ["all", "owners", "user_types"] },
        "contractOwnerUserIds": { "type": "array", "items": { "type": "string" } },
        "userTypes": { "type": "array", "items": { "type": "string", "enum": ["admin"] } }
      }
    },
    "subject": { "type": "string", "maxLength": 200 },
    "bodyHtml": { "type": "string", "maxLength": 10000 },
    "optOut": { "type": "string", "enum": ["optional", "mandatory"] },
    "scheduledAt": { "type": "string", "format": "date-time" }
  }
}
```

メインは受信後、対象オーナー + 管理者ユーザーの inbox に配信し、`MSG-SCR-094-*` テンプレートに従ってメール送信(opt-out 設定に従う)。

### 3.6 訂正告知(FR-189)

既に `sent` 状態のお知らせに対し、訂正告知を作成できる:

| 項目 | 仕様 |
|---|---|
| `kind` | `incident`(訂正告知も同 kind を使う、`correctionOf` フィールドで紐付け) |
| `correctionOf` | 訂正元 `announcementId` |
| 監査 | `announcement.correction.issue`(5y、`{newAnnouncementId, correctionOf}`) |

訂正告知の本文 HTML には自動的に「訂正前: [元告知タイトル]」リンクが挿入される(`SanitizedHtml` レンダラ側で `correctionOf` を参照)。

### 3.7 テスト送信(`POST /announcements/{id}/test-send`)

| 項目 | 仕様 |
|---|---|
| 送信先 | 申請者運営者本人のメール |
| 監査 | `announcement.test_send`(1y、`{announcementId, sentTo}`) |
| Resend 直叩き | 本書側 BillingWebhookWorker 同様、運営者通知 + テスト送信のみ Resend 直叩き |

### 3.8 通知配信(D-20)

- **オーナー + 全プロジェクト管理者(`project_users.role='admin'` を 1 件でも保持するユーザー)** に配信(`severity=high` の場合は強制送信)
- メインの `inbox_messages` テーブルに `kind=announcement` で 1 年保持(`retention_class='general'` 相当)、配信メタ情報を `audit_logs` に `retention_class='operator_high_priv'`(5y)で記録
- ユーザー個別の opt-out 設定が `mandatory` の場合はメールも強制送信

通知契機の正本は [基本設計 / メッセージ一覧](../02_基本設計/06_メッセージ一覧.md) §8 を参照。

### 3.9 10 分集約窓(D-19、運営者操作通知)

`announcement.send` 自体は集約対象外(個別に通知)。一方、SCR-094 経由の運営者操作(`announcement.schedule` / `announcement.cancel`)は同一オーナー宛通知が 10 分以内に複数発生する場合に集約。

| KV キー | TTL | 集約処理 |
|---|---|---|
| `notify-batch:<contract_owner_user_id>:announcement.schedule` | 10 分 | `OperatorNotifyAggregatorWorker` が 10 分窓終了で連携 IF #12 送信 |

### 3.10 主要エラーコード

| エラー ID | HTTP | 説明 |
|---|---|---|
| `E-OP-ANN-001` | 422 | HTML_SANITIZER_REJECT(タグ / 属性 / スキーム拒否) |
| `E-OP-ANN-002` | 409 | INVALID_STATE_TRANSITION(`draft → sent` 等、許可されない直接遷移) |
| `E-OP-ANN-003` | 410 | TOO_LATE_TO_CANCEL(`now ≥ scheduled_at - 5min`) |
| `E-OP-ANN-004` | 502 | UPSTREAM_MAIN_FAILED(連携 IF #7 5xx) |
| `E-OP-ANN-005` | 503 | DLQ_AFTER_3_ATTEMPTS |

完全な E-OP-ANN-* 一覧は [基本設計 / エラー設計](../02_基本設計/05_エラー設計.md) を正本とする。

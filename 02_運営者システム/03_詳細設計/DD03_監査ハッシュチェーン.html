3. 詳細設計本文

3.1 監査ハッシュチェーン構造

各 audit_logs レコードは前レコードの record_hash を prev_hash 列に保持し、暗号学的に連結する。

record_hash = HMAC-SHA256(
  key = HKDF(masterKey, info="audit-chain"),
  message = prev_hash || canonical_json({
    actor_id, action, target_type, target_id,
    payload_hash, retention_class, occurred_at
  })
)
項目 仕様
初期値 prev_hash "0" * 64(全 0、ジェネシス)
HMAC アルゴリズム HMAC-SHA256
HKDF info audit-chain
record_hash 長 64 文字(16 進エンコード)
順序保証 (occurred_at, id) の昇順、ULID 採番で時系列単調保証
改ざん検知 1 レコードを書換えると以降全レコードの record_hash が不一致

3.2 書込トランザクション(D-04)

監査ログ INSERT は次のトランザクションで実行する:

BEGIN;
  -- 直前レコードを排他取得
  SELECT id, record_hash FROM audit_logs
    ORDER BY occurred_at DESC, id DESC LIMIT 1
    FOR UPDATE;
  -- 新レコード INSERT
  INSERT INTO audit_logs (
    id, actor_id, action, target_type, target_id,
    payload, payload_hash, retention_class,
    prev_hash, record_hash, occurred_at, trace_id
  ) VALUES (?, ..., ?, computed_record_hash, ?, ?);
COMMIT;

D1 のトランザクション境界制約(現状 batch API ベース)を踏まえ、実装では db.batch([...]) で SELECT + INSERT を 1 つの論理単位として送出する。並行 INSERT で同一 prev_hash を使う競合は影響行数 = 0 で検出し、リトライ(最大 3 回)+ それでも失敗時は audit.write.conflict 監査 + 運営者 high alert。

3.3 日次ハッシュチェーン検証(AuditChainVerifierWorker、D-04)

function auditChainVerify():
    verify_date = today
    runId = ULID()
    audit_logs(audit.chain.verify.start, 5y, payload={verify_date, run_id: runId})
    
    prev_hash = "0" * 64
    mismatch_count = 0
    cursor = null
    do:
        rows = D1.query("SELECT id, prev_hash, record_hash, ... FROM audit_logs WHERE id > ? ORDER BY occurred_at, id LIMIT 1000", cursor)
        for r in rows:
            expected = sha256(prev_hash + canonical_json(r.{actor_id, action, ..., retention_class}))
            if expected != r.record_hash:
                audit_logs(audit.chain.verify.fail, 5y, payload={row_id: r.id, expected, actual: r.record_hash})
                notifyOperator(high, "Audit chain mismatch at row=${r.id}")
                mismatch_count++
            prev_hash = r.record_hash
            cursor = r.id
    while rows.length > 0
    
    audit_logs(audit.chain.verify, 5y, payload={verify_date, run_id: runId, mismatch_count, total_rows: ...})

cron 設定: 0 17 * * *(UTC、毎日 17:00 UTC = 翌日 02:00 JST)、Cron 詳細は DD08_Cron実装.html 参照。

所要時間目標: 1000 万行で 3600s 以内(KV: monitoring:thresholds:chain-verify-duration で監視、超過時に分割検証へ切替)。

3.6 監査エクスポート(HMAC 署名、D-17)

エクスポートは非同期ジョブ(audit-export:<job_id> KV TTL 24h)で実行する:

ステップ 処理
1 POST /audit-logs/exports(`format: csv
2 Worker が D1 クエリ + R2 マルチパート PUT(10 万行/ファイル、5 並列上限)
3 全ファイル完了後 HMAC 署名(HKDF info=audit-export、SHA-256)を .sig で生成
4 GET /audit-logs/exports/{job_id} で署名付き URL を取得(R2 Pre-signed、TTL 24h)
5 エクスポート完了で audit.export(5y、{exportId, format, rows})記録

外部検証ツール: admin/scripts/audit-export-verify.ts で .sig + マスター鍵公開 OK の HKDF 派生鍵から HMAC 再計算 → 完全性確認。運営者は別保管した鍵情報で第三者検証可能。

HMAC 計算:

hmacKey = HKDF(masterKey, info="audit-export")
signature = HMAC-SHA256(
  key = hmacKey,
  message = sha256(file_content) || jobId || createdAt
)

性能目標: 10 万行/ファイルで ≤ 60s(KV: monitoring:thresholds:audit-export-duration-p95)。

3.8 accounts_retired 永久保持

5. テスト観点

5.1 ユニットテスト

  • HMAC-SHA256 + HKDF info=audit-chain 派生鍵の確定的計算
  • canonical_json 正規化(キー順序ソート、Unicode NFC、数値表記)
  • 単一 INSERT で prev_hash + record_hash チェーン整合
  • 改ざんシミュレーション(1 レコード書換 → 検証で不一致検出)
  • HMAC 監査エクスポート署名計算 + 外部ツール検証

5.2 結合テスト(Miniflare)

  • 1000 件 INSERT 後にチェーン全件検証 = 不一致 0
  • 並行 INSERT で prev_hash 競合発生 → リトライ後成功
  • 監査検索カーソルページング動作
  • エクスポート(1000 行 × 3 ファイル)の HMAC 署名生成 + 検証

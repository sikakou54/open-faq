## 3. 詳細設計本文

### 3.1 構造化ログ JSON Schema(★TH-11)

すべての Worker は標準出力に JSON 1 行形式で構造化ログを出力する。Logpush / Logflare 等の集約基盤に送出され、Trace ID で横串集計可能。

```json
{
  "level": "debug | info | warn | error | critical",
  "timestamp": "2026-05-12T10:00:00Z",
  "worker": "AdminConsoleWorker",
  "request_id": "<cf-ray>",
  "trace_id": "01J...",
  "operator_id": "01J... | null",
  "contract_owner_user_id": "01J... | null",
  "action": "<action_code> | null",
  "endpoint": "/admin/api/v1/...",
  "method": "GET | POST | PUT | DELETE",
  "status_code": 200,
  "duration_ms": 123,
  "error_code": "E-OP-... | null",
  "error_message": "...",
  "stack_trace": "..."
}
```

必須フィールド: `level`、`timestamp`、`worker`、`request_id`。
任意フィールド: `trace_id`、`operator_id`、`contract_owner_user_id`、`action`、`endpoint`、`method`、`status_code`、`duration_ms`、`error_code`、`error_message`、`stack_trace`。

実装ファイル `admin/src/shared/logger.ts` の型定義と一致させること。

### 3.2 機密項目マスキング

ログ出力前に必ず次のフィールドをマスクする:

| フィールド | マスキング |
|---|---|
| `Authorization` ヘッダ | 完全削除 |
| `X-CSRF-Token` | 最初の 4 文字 + `***` |
| パスワード / MFA コード / リカバリーコード | 完全削除 |
| メール本文 PII / 電話番号 / クレカ | DD10 と同方式でマスク |
| Stripe `customer.id` | 末尾 4 文字 + `***`(法令対応で完全削除は避ける) |
| IP アドレス | `/24`(IPv4) / `/48`(IPv6)単位で切り捨て |

### 3.3 エラー ID 体系

エラー ID 体系の正本は [基本設計 / エラー設計](../02_基本設計/05_エラー設計.md)。`E-OP-*` 87 件、エラー分類 13 種、異常系共通方針、`critical` レベルエラーは当該ドキュメントを参照。

主な接頭辞:

| 接頭辞 | 範囲 | DD 参照 |
|---|---|---|
| `E-OP-AUTH-*` | 認証 / 認可 | DD01 |
| `E-OP-4EYES-*` | 4-eyes | DD02 |
| `E-OP-AUDIT-*` | 監査ログ | DD03 |
| `E-OP-WEBHOOK-*` | Webhook | DD04 |
| `E-OP-RESTORE-*` | 削除データ復元 | DD05 |
| `E-OP-ANN-*` | お知らせ | DD06 |
| `E-OP-PII-*` | PII 報告 | DD10 |
| `E-OP-STATE-*` | 状態遷移共通 | DD11 |

### 3.4 RFC 7807 problem+json 形式

エラーレスポンスは RFC 7807 に準拠:

```json
{
  "type": "https://admin.open-faq.example.com/errors/E-OP-AUTH-005",
  "title": "Re-authentication required",
  "status": 403,
  "code": "E-OP-AUTH-005",
  "detail": "Critical operation requires re-authentication within 5 minutes",
  "trace_id": "01J...",
  "instance": "/admin/api/v1/owners/01J.../suspend"
}
```

`type` フィールドは `code` フィールドと同一のエラー ID を URL 末尾に持ち、運営者がドキュメントへ到達可能にする。

### 3.11.1 SCR-096 異常検知バッジ → フィルタ自動投入(画面側ロジック)

SCR-096 運営者活動ダッシュボードの「ハッシュチェーン検証バッジ」「運営者操作頻度異常検知バッジ」は、基本設計 [01_画面設計.md §5.9](../02_基本設計/01_画面設計.md) で「異常時はクリックで一覧フィルタに条件を自動投入」と定義する。本節はその画面側ロジックを補足する。

| バッジ | 平常時表示 | 異常時表示 | クリック時のフィルタ自動投入 |
|---|---|---|---|
| ハッシュチェーン検証バッジ(MSG-SCR-096-BADGE-001 / 002)| ✅ 検証済(過去 24h)、クリック不可 | ⚠ 確認が必要(該当区間 `[from, to]` 含む)、クリック活性 | `occurred_at` フィルタに `[from, to]` を自動投入し、一覧を再検索。クイックフィルタチップ「異常検知あり」を選択状態にする |
| 運営者操作頻度異常検知バッジ(MSG-SCR-096-BADGE-003 / 004)| ✅ 操作頻度: 正常、クリック不可 | ⚠ 操作頻度異常 `actor_id={op_xxx}`、クリック活性 | `actor_id` フィルタに該当 `op_xxx` を自動投入し、`occurred_at` に検知ウィンドウ(直近 1 時間等)を自動投入。一覧を再検索 |

- 異常検知の出元は `AuditChainVerifierWorker`(運営者 [DD03_監査ハッシュチェーン.md](DD03_監査ハッシュチェーン.md))および `OperatorActivityAnomalyDetector`(検知ロジックはアラート閾値: 直近 1 時間で 200 操作以上 / actor)。検知結果は `GET /admin/v1/home` および `GET /admin/v1/audit-logs/anomalies` レスポンスに含めて画面に渡す。
- 画面側は、フィルタ自動投入後の URL にクエリパラメータを付与(`?actor_id=op_xxx&from=...&to=...`)して、戻る操作 / シェアにも対応する。
- 「✕ 全クリア(現在の絞り込み解除)」チップ(MSG-SCR-096-CHIP-CLEAR-001)はクイックフィルタチップ群に常時併置し、クリックで全フィルタを初期値に戻す。

### 3.11.2 SCR-HOME 緊急アラート 0 件時の「異常なし」バナー(画面側ロジック)

基本設計 [01_画面設計.md §5.3](../02_基本設計/01_画面設計.md) で定義する SCR-HOME 緊急アラート帯は、件数が 1 件以上ある場合は赤系アラートを表示するが、**全アラートが 0 件のときは緑系の「✅ 異常なし」バナー(MSG-SCR-HOME-OK-001)を表示する**。これは「アラート表示が壊れているのか平常なのか」をユーザーが判別できるようにするための明示的な肯定表示。

- 表示判定: `GET /admin/v1/home` のレスポンスで以下がすべて 0 のとき。
  - `approval.expiring_within_24h_count`
  - `webhook.dlq_stalled_24h_count`
  - `pii.sla_exceeded_count`
  - `audit.hash_chain_breach_count`
- バナー文言: 「✅ 緊急対応が必要な事案はありません(ハッシュチェーン: 検証済 / 操作頻度: 正常)」
- ハッシュチェーン / 操作頻度の状態は同じ API レスポンスに含めて返す(運営者 API 設計参照)。

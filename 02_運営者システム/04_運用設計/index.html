1. 運用方針

項目 内容
基本方針 運営者システムは契約管理・課金管理・監査・4-eyes 承認の正本系。メイン側障害時も独立稼働できる縮退運転を担保する
データリージョン 日本リージョン(apac)固定。データ越境ゼロ
SLO 目標 管理画面(運営者含む)月次 99.5%(NFR-202)、監査ログ検索 p95 ≤ 1000ms、監査ログエクスポート 10 万行/60s
4-eyes 承認 MVP ハードゲート 3 操作(契約物理削除 / AI 推論パラメータ変更 / マスター鍵ローテーション)+ 承認ログ 7 操作
オンコール SLA P1 = 15 分アサイン / 30 分着手、P2 = 1h、P3 = 翌営業日(NFR-820)
環境分離 dev / staging / prod の 3 環境。prod デプロイは 2 名承認、運営者 Worker はメイン Worker と独立(別 wrangler プロジェクト)
縮退運転 メイン側障害時も監査・課金 Webhook 受信・運営者操作などは継続可能

2. 運用対象

区分 対象 補足
アプリ @faq-saas/admin-api、AdminConsoleWorker、SREWorker、AnalyticsBatchWorker、AuditChainVerifierWorker、RetentionPurgeWorker、R2AuditArchiveWorker、DLQAutoBackoffWorker、MailSendWorker Cloudflare Workers / Pages
DB D1(admin-db-prod) D1 Time Travel(直近 30 日)
インフラ Cloudflare Workers + KV + R2 + Queues + Cron Triggers apac リージョン
外部サービス Stripe(Webhook 一次受信)、Resend(メール送信 + Webhook 受信集計) 連携 IF #10 / #12
バッチ RetentionPurgeWorker 日次 03:00 / AuditChainVerifierWorker 日次 / R2AuditArchiveWorker 年次 / DLQAutoBackoffWorker 5 分 cron / SLA エンジン日次 04:00 等 §06_運用手順 §1 を参照
ログ アプリログ(R2 / 90 日)、エラーログ(180 日)、監査ログ(1y / 5y / 7y、ハッシュチェーン管理) JSON Lines
監視対象 メイン NFR-804 (a)〜(h) + 本書独自 (k) 運営者操作頻度異常 + (l) 全契約横断 MAU・解決率 + Webhook 受信失敗率 + DLQ 滞留 + 監査ハッシュチェーン不一致 + AI 内部原価 Cloudflare Workers Analytics Engine

3. 運用体制

役割 担当範囲 連絡経路 エスカレーション
第 1 階層 on-call(運営者) 一次切り分け / Runbook 実行(RB-001〜RB-023) PagerDuty / Slack NFR-820 SLA
第 2 階層 on-call backup 第 1 階層の代替 PagerDuty —
PO + 開発リード 重大障害時の意思決定支援 メール + Slack 経営層へ
経営層(CTO / SRE Lead) データ損失リスク評価 / 顧客告知判断 個人携帯 —
法務担当 RB-023 EU 顧客対応 / 監査ログ改ざん検知時の判断 メール —
サポート 平日 9:00〜18:00(JST)、土日祝・年末年始対応外(セキュリティインシデント発動時のみ時間外) メール + チャット オンコールへ

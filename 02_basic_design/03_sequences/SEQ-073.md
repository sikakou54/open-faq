## 概要

招待された登録済みユーザー本人が招待内容を確認したうえで「参加を受諾する」を押下すると、サーバーが招待トークンの有効性(期限・使用済み)と招待先メールとログイン中ユーザーの一致(本人受諾)を再検証し、問題がなければ対象プロジェクトへの割当有効化・招待トークンの使用済み化・監査ログ記録を同一トランザクションで実行する。成功時は完了画面を表示し、失敗時は内容を確定しない。表示名・初回パスワード・規約同意の収集は行わない。

## シーケンス図

```mermaid
sequenceDiagram
  autonumber
  actor U as 利用者
  participant Screen as メンバーアカウント有効化
  participant Server as サーバー
  participant DB as DB
  U->>Screen: 参加を受諾するを押下
  Screen->>Server: 招待受諾要求
  Server->>Server: トークン有効性・本人一致を検証
  alt 検証 OK・トークン有効・本人一致
    Server->>DB: 業務データを更新(割当有効化・トークン使用済み化・監査ログ記録)
    DB-->>Server: 更新完了
    Server-->>Screen: 受諾完了
    Screen-->>U: 完了画面を表示
  else 本人不一致
    Server-->>Screen: 本人不一致
    Screen-->>U: 本人不一致エラー画面を表示
  else トークン無効 / 期限切れ・使用済み
    Server-->>Screen: トークン無効
    Screen-->>U: トークン無効 / 期限切れエラー画面を表示
  end
```

## 例外フロー

- 本人不一致: 招待先メールとログイン中ユーザーのメールが一致しない場合、本人不一致エラー画面を表示し、状態を確定しない。
- トークン無効 / 期限切れ / 使用済み: 招待トークンが期限切れ・使用済み・無効の場合、トークン無効 / 期限切れエラー画面を表示し、状態を確定しない。

## 詳細設計への移管候補

| 内容 | 移管先候補 | 理由 |
|---|---|---|
| 複数テーブル更新の同一トランザクション内の順序・ロールバック | 詳細設計 | 基本設計では 1 段に集約し、トランザクション内の実装順序は展開しない |

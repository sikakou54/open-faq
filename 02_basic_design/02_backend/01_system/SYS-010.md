# <span id="SYS-010"></span>SYS-010: 契約単位レート制限の適用

> **このページは、各機能へのリクエストに契約・機能種別ごとのレート制限を契約単位で適用し、DDoS・Bot・暴走による過負荷から防御するシステム処理 SYS-010 を定義します。** 処理概要 / 処理フロー図 / 入出力 / 処理項目定義 / 入出力一覧 / システムイベント一覧 の 6 セクションで記述します。

*種別 システム設計 ・ 優先度 P0 ・ ステータス ドラフト*

## 1. 処理概要

サービスの可用性と契約間の公平な利用を守るため、システムは各機能へのリクエストが発生するたびに、契約・機能種別ごとに定めたレート制限を契約単位で評価する。リクエストを受け付けると、システムはリクエスト元の契約と機能種別を識別し、当該契約・種別に適用される上限を評価する。上限の範囲内であればリクエストの処理を許可し、上限を超過していれば当該リクエストを抑制して後段の業務処理へ渡さない。評価の都度アクセス状況を記録し、超過の抑制は記録として残して濫用の検知につなげる。本処理はゲートウェイ層で全 API 横断に作用し、特定の機能 API に結線しない。レート制限は契約単位での適用とし、プロジェクト単位化(月次の上限件数・無料枠を除く)の対象外とする。

計数方式は固定ウィンドウカウンタ(設計値)とする。契約 × 機能種別ごとにカウンタを 1 つ持ち、固定長のウィンドウ(=1 分(設計値))内のリクエスト数を加算する。同一ウィンドウ内で上限に達した以降のリクエストは抑制し、ウィンドウが切り替わるとカウンタは初期化される。複数インスタンスで処理する分散環境を前提とするため、カウンタは各インスタンス内のメモリではなく共有の計数ストア(KV など。設計値)に保持し、契約をまたいだ競合なくウィンドウ単位で計数する。上限超過時はリクエストを後段へ渡さず、HTTP 429([ERR-011](../../05_errors/ERR-011.md#ERR-011) `RATE_LIMITED`)を返す境界とする。レート制限を実装する非機能要件は [NFR-017](../../../01_requirements/03_non_functional_requirement/07_nfr.md#NFR-017) に基づく。具体のしきい値(契約・種別ごとの上限回数)は要件(NFR-017 / 業務ルール RULE)へ委譲し、本設計では計数方式と境界のみを定める。

## 2. 処理フロー図

```mermaid
flowchart TB
  A([機能へのリクエストを受け付ける]) --> B[リクエスト元の契約と機能種別を識別する]
  B --> C[当該契約・種別に適用するレート制限の上限を取得する]
  C --> D[契約単位で適用状況を評価する]
  D --> E{上限の範囲内か}
  E -->|範囲内| F[リクエストの処理を許可し後段へ渡す]
  E -->|上限超過| G[当該リクエストを抑制し後段へ渡さない]
  F --> H[アクセス状況を記録する]
  G --> H
  H --> Z([評価完了])
```

## 詳細設計への移管候補

- 計数方式は固定ウィンドウカウンタ・ウィンドウ=1 分(いずれも設計値)を基本設計で定める。計数ストア(KV 等)の具体製品・キー設計・ウィンドウ境界近傍(ウィンドウ切替直後の集中)での扱い・カウンタ失効(TTL)の実装は詳細設計で定める。
- 上限の上書き設定(契約別)と基礎設定(種別別)の優先順位・合成規則は詳細設計で定める。
- 超過時は HTTP 429([ERR-011](../../05_errors/ERR-011.md#ERR-011))を返すことを基本設計で定める。抑制の通知形式(`Retry-After` 等の再試行可否の表現)と濫用検知への連携方式は詳細設計で定める。

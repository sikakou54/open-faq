## 処理概要

AI 回答を生成し質問ログを記録する。未回答時は未解決質問を同時生成する。

| 処理 ID | 処理内容                                             |
|---------|------------------------------------------------------|
| P-01    | セッショントークンとオリジンを検証する               |
| P-02    | オーナー単位のレート制限とプロジェクト月次上限を判定する |
| P-03    | 当該プロジェクトの公開 FAQ を全文検索([`TP_FAQ_FTS`](../04_database/TBL-030.md#TBL-030) trigram・[API-031](API-031.md#API-031) 相当)し、一致スコア降順で上位 N 件（N はシステム仕様書の設計値。[システム仕様書 §1](../../07_system-spec.md#1-aiしきい値)）を候補 FAQ とする。候補の最上位一致スコアを関連度(relevance)とする。この候補集合を AI 推論 IF へ渡して回答を生成し、同じ候補集合の上位から関連質問サジェスト(関連 FAQ の質問文・関連度順に最大 3 件・設計値)を併せて求める。所定時間（[システム仕様書 §3](../../07_system-spec.md#3-タイムアウトセッション認証)）内に応答しない、または AI プロバイダがエラーを返した場合は、処理エラーとして扱う |
| P-04    | 回答可否を判定する。信頼度(confidence) >= 信頼度しきい値 かつ 関連度(relevance) >= 関連度しきい値([システム仕様書 §1](../../07_system-spec.md#1-aiしきい値)・[RULE-012](../../../01_requirements/01_business_requirement/08_rule.md#RULE-012))の両方を満たせば回答可(`answered`)。いずれか未満は `low_confidence`、候補 FAQ が 0 件のときは `no_faq_match`、候補 FAQ 間に矛盾がある場合は `contradiction`(未回答)とする |
| P-05    | 回答可の場合、応答返却前に回答文の個人情報(メールアドレス・電話番号・クレジットカード番号)を検出し、種別ラベル付き伏字へ置換してマスキングする([RULE-024](../../../01_requirements/01_business_requirement/08_rule.md#RULE-024))。本工程は応答整形前に AI 推論 IF の呼び出し元が応答整形の前段で実施する。検出時は質問ログの `pii_masked=1` と検出種別 `pii_types` を記録する |
| P-06    | 質問ログを記録する(AI タイムアウト / プロバイダエラー時は `result_type=error`、`result_reason_code=ai_unavailable` として記録する)。DB書き込み自体が失敗する致命的エラー時は、追跡可能なエラー識別子を生成する。未回答時(低信頼度・該当 FAQ なし・候補 FAQ 間の矛盾の 3 理由)は未解決質問を同一トランザクションで生成する。AI タイムアウト / プロバイダエラー時は未解決質問を生成しない |
| P-07    | 回答または未回答理由・連絡先を返却する。回答時はマスキング済み回答文と関連質問サジェストを併せて返す。AI タイムアウト / プロバイダエラー時、およびDB書き込み失敗時などのシステムエラー時は処理エラー(エラー識別子付き)を返す |

## レスポンス

**レスポンス(200)**

answered:

```json
{
  "type": "answered",
  "answer": "返品は商品到着後 7 日以内に...",
  "confidence": 0.78,
  "referencedFaqs": [
    { "id": "faq_...", "question": "返品ポリシー", "answer": "..." }
  ],
  "suggestions": [
    { "faqId": "faq_...", "label": "支払い方法を変更したい" },
    { "faqId": "faq_...", "label": "領収書がほしい" }
  ],
  "questionLogId": "qlog_..."
}
```

| 項目             | 型              | 説明                            |
|------------------|-----------------|---------------------------------|
| `type`           | enum            | 回答種別。`answered`(回答あり)  |
| `answer`         | string          | AI が生成した回答本文           |
| `confidence`     | number          | 回答の信頼度(0〜1)              |
| `referencedFaqs` | array\<object\> | 回答の根拠とした参照 FAQ の配列 |
| `　└ id`         | string          | 参照 FAQ の ID                  |
| `　└ question`   | string          | 参照 FAQ の質問                 |
| `　└ answer`     | string          | 参照 FAQ の回答                 |
| `suggestions`    | array\<object\> | 関連質問サジェスト。候補 FAQ 照合結果から関連 FAQ の質問文を関連度順に最大 3 件(設計値)返す(ウィジェットはチップ表示) |
| `　└ faqId`      | string          | サジェスト元 FAQ の ID          |
| `　└ label`      | string          | チップ表示文(関連 FAQ の質問文) |
| `questionLogId`  | string          | 記録された質問ログの ID         |

**レスポンス(200)**

unanswered:

```json
{
  "type": "unanswered",
  "reason": "low_confidence" | "no_faq_match" | "contradiction" | "usage_limit_reached" | "payment_required",
  "questionLogId": "qlog_...",
  "inquiryId": "inq_...",
  "inquiryCode": "INQ-20260513-XYZ123",
  "contactEmail": "support@example.com"
}
```

| 項目 | 型 | 説明 |
|----|----|----|
| `type` | enum | 回答種別。`unanswered`(回答不可) |
| `reason` | enum | 未回答 / 受付停止理由。`low_confidence`(信頼度不足)/ `no_faq_match`(該当 FAQ なし)/ `contradiction`(候補 FAQ 間の矛盾)/ `usage_limit_reached`(質問数上限 100% 到達による受付停止)/ `payment_required`(支払方法ゲートによる受付停止) |
| `questionLogId` | string | 記録された質問ログの ID |
| `inquiryId` | string | 同時生成された未解決質問の ID |
| `inquiryCode` | string | 未解決質問の管理用識別子(ウィジェットには描画しない) |
| `contactEmail` | string | プロジェクトの問い合わせ先メールアドレス(未設定時は `null`) |

未回答理由 3 値(`no_faq_match` / `low_confidence` / `contradiction`)による `unanswered` では、質問ログと未解決質問を同一トランザクションで作成する。`inquiryCode` は管理用識別子としてレスポンス互換性のため返却するが、ウィジェットには描画しない。ウィジェットは回答できなかった旨と確認済みプロジェクト連絡先メールを表示し、FAQ 質問入力は継続可能とする。連絡先未設定時は `contactEmail=null` を返す。冪等性は `questionLogId` を基準に担保する。

AI 推論が所定時間（[システム仕様書 §3](../../07_system-spec.md#3-タイムアウトセッション認証)）内に応答しない、または AI プロバイダがエラーを返した場合は処理エラーとして扱う。質問ログを残す場合は `result_type=error`、`result_reason_code=ai_unavailable` を設定し、未解決質問は登録しない([FR-082](../../../01_requirements/02_functional_requirement/02_faq-ai-fr.md#FR-082))。

`suggestions`(関連質問サジェスト)は候補 FAQ 照合結果から関連 FAQ の質問文を関連度順に最大 3 件(設計値。0 件のときは空配列)返す。

**候補 FAQ 選定と回答可否判定** `answer` / `confidence` / `referencedFaqs` / `suggestions` は、当該プロジェクトの公開 FAQ を全文検索([`TP_FAQ_FTS`](../04_database/TBL-030.md#TBL-030) trigram・[API-031](API-031.md#API-031) 相当)して一致スコア降順で上位 N 件(N はシステム仕様書の設計値。[システム仕様書 §1](../../07_system-spec.md#1-aiしきい値))を候補 FAQ とし、この候補集合から求める。関連度(relevance)は候補最上位の検索一致スコアである。信頼度(confidence) >= 信頼度しきい値 かつ 関連度(relevance) >= 関連度しきい値の両方を満たす場合のみ `answered` を返し、いずれか未満は `low_confidence`、候補 FAQ 0 件は `no_faq_match`、候補 FAQ 間の矛盾は `contradiction`(いずれも `unanswered`)とする。しきい値の正本は [システム仕様書 §1](../../07_system-spec.md#1-aiしきい値) / [RULE-012](../../../01_requirements/01_business_requirement/08_rule.md#RULE-012)。`confidence` は質問ログの信頼度スコア、関連度は関連度スコアへ記録する([`H_QUESTION_LOGS`](../04_database/TBL-025.md#TBL-025))。

**回答文の個人情報マスキング** `answered` を返す前に、回答文に含まれる個人情報(メールアドレス・電話番号・クレジットカード番号)を検出し、種別ラベル付き伏字([メールアドレス] / [電話番号] / [カード番号])へ置換したうえで `answer` を返す([RULE-024](../../../01_requirements/01_business_requirement/08_rule.md#RULE-024))。氏名・住所は MVP 対象外(将来対応)。マスキングは応答整形前に AI 推論 IF の呼び出し元が応答整形の前段で実施する。検出時は質問ログの `pii_masked=1` と検出種別 `pii_types`(`email` / `phone` / `credit_card` のカンマ区切り)を記録する([`H_QUESTION_LOGS`](../04_database/TBL-025.md#TBL-025))。マスキングは `answered` のまま返し、未解決へ振り替えない。

## 列挙値

リクエスト / レスポンスで用いる enum 項目の取りうる値の全集合と意味を示す。レスポンスは回答可否で `type` が分岐し、未回答時のみ `reason` を返す。

| 項目 | 取りうる値 | 意味 |
|----|----|----|
| `type` | `answered` | 回答あり(AI が回答を生成) |
| `type` | `unanswered` | 回答不可(未解決質問を生成し連絡先を案内) |
| `reason` | `low_confidence` | 信頼度がしきい値未満(回答を生成できなかった) |
| `reason` | `no_faq_match` | 該当する FAQ が無い(候補 FAQ 0 件) |
| `reason` | `contradiction` | 候補 FAQ 間に矛盾があり断定回答しない([FR-060](../../../01_requirements/02_functional_requirement/02_faq-ai-fr.md#FR-060) / [RULE-023](../../../01_requirements/01_business_requirement/08_rule.md#RULE-023)) |
| `reason` | `usage_limit_reached` | 当該プロジェクトの質問数が月次上限の 100% に到達し新規質問受付を停止(受付停止) |
| `reason` | `payment_required` | 支払方法未登録のまま無料枠を超過し新規質問受付を停止(支払方法ゲート停止) |

> [!NOTE]
> **`reason` は `type=unanswered` のときのみ返す。** 質問ログ側のコード値([`H_QUESTION_LOGS.result_reason_code`](../04_database/TBL-025.md#TBL-025))は回答成功を示す `answered`、処理エラーを示す `ai_unavailable`、レート制限を示す `rate_limited` も保持するが、本 API のレスポンス `reason` には上記 5 値(未回答理由 3 値 `no_faq_match` / `low_confidence` / `contradiction` + 受付停止 2 値 `usage_limit_reached` / `payment_required`)が現れる。`ai_unavailable` は処理エラーとして扱い、レート制限は本 API では 429 [ERR-009](../../05_errors/ERR-009.md#ERR-009) で返す。回答文中の個人情報は未回答理由ではなく、`answered` のままマスキングして返す([RULE-024](../../../01_requirements/01_business_requirement/08_rule.md#RULE-024)。P-05)。

> [!IMPORTANT]
> **受付停止は 200(graceful)で返す。** `usage_limit_reached`(質問数上限 100% 到達)・`payment_required`(支払方法ゲート)による新規質問受付停止は、エラー(4xx/5xx)ではなく 200 の `type=unanswered`(対応する `reason`)として返す。ウィジェットは送信を無効化し、停止理由に応じた定型文を表示する。停止の契機・課金アカウント状態・解除条件の正本は [課金・請求設計 §2.1](../../05_billing-design.md#21-停止状態とウィジェット応答一覧) / [§4](../../05_billing-design.md#4-支払方法ゲート)。`contradiction` は候補 FAQ 間の矛盾により断定回答を避け未解決として扱うもので、未解決質問を生成し連絡先を案内する([FR-060](../../../01_requirements/02_functional_requirement/02_faq-ai-fr.md#FR-060) / [RULE-023](../../../01_requirements/01_business_requirement/08_rule.md#RULE-023))。なお課金アカウントが `suspended` の場合は受付停止ではなく別途エラー応答(`BILLING_ACCOUNT_SUSPENDED`)で扱う(詳細設計で確定)。

## バリデーション

入力値の検証規則は `## リクエスト` の各パラメータ説明を参照する。`question` は 1〜1000 文字とする(画面 SCR-030 の最大長 1000 文字と一致)。

## エラー

| HTTP ステータス | エラーコード         | 内容               |
|-----------------|----------------------|--------------------|
| 429 | `RATE_LIMITED` → [ERR-009](../../05_errors/ERR-009.md#ERR-009) | レート制限超過 |
| 403 | `DOMAIN_NOT_ALLOWED` → [ERR-027](../../05_errors/ERR-027.md#ERR-027) | 許可ドメイン外 |
| 500 | `INTERNAL_SERVER_ERROR` | DB書き込み失敗などの致命的エラー。一意なエラー識別子（FR-084等）を生成・レスポンスに含め、後続の有人調査を可能にする例外処理を実装する（詳細設計にて指示） |
| 503 | `AI_UNAVAILABLE` → [ERR-036](../../05_errors/ERR-036.md#ERR-036) | AI 推論タイムアウト / プロバイダエラー。質問ログには `result_type=error`、`result_reason_code=ai_unavailable` を記録し、未解決質問は登録しない |

> [!NOTE]
> **AI 推論のタイムアウト（[システム仕様書 §3](../../07_system-spec.md#3-タイムアウトセッション認証)）/ プロバイダエラーは未解決登録にしない。** これらは処理エラーとして扱い、質問ログに `result_type=error`、`result_reason_code=ai_unavailable` を記録する場合でも、未解決質問は登録しない([FR-082](../../../01_requirements/02_functional_requirement/02_faq-ai-fr.md#FR-082))。DB保存等の内部エラー時もシステムエラーとして扱うが、この場合はエラー識別子を生成しウィジェットに返すことでユーザーからの報告を可能とする。

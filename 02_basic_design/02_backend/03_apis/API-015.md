# <span id="API-015"></span>API-015: セキュリティ設定更新

> **このページは セキュリティ設定更新 API の仕様を定義します。** 認証済みのアカウント利用者が設定画面から自身のメールアドレス・パスワード(セキュリティ系)を更新する。いずれの変更も再認証トークンを要求する。メールアドレス変更時は現行ログインメールを即時には上書きせず、新アドレスを未確認として保持し、新アドレスへ確認メールを送信する。表示名(プロフィール名)の更新は [API-012](API-012.md#API-012) が担う(本 API では扱わない)。

| ID | 業務ユースケースID | イベント(画面ID EVT-NN) | テーブルID |
|----|----|----|----|
| API-015 | [UC-009](../../../01_requirements/04_business_usecases/UC-009.md#UC-009) | SCR-022 EVT-03 | [TBL-001](../04_database/TBL-001.md#TBL-001) ・ [TBL-002](../04_database/TBL-002.md#TBL-002) ・ [TBL-003](../04_database/TBL-003.md#TBL-003) ・ [TBL-014](../04_database/TBL-014.md#TBL-014) |

| API名 | エンドポイント | HTTPメソッド |
|----|----|----|
| セキュリティ設定更新 | `/me/settings` | PATCH |

## 処理概要

認証済みのアカウント利用者が設定画面から自身のメールアドレス・パスワード(セキュリティ系)を更新する。いずれの変更も再認証トークンを検証する。メールアドレス変更は「確認待ち → 確認完了で確定反映」の2段階とし、本 API では現行ログインメール(`M_USER.email`)を即時には書き換えず、新アドレスを未確認の変更中アドレス(`M_USER.pending_email`)として保持したうえで、新アドレスへ確認メールを送信する。新アドレスは確認メールの確認完了時に初めて現行ログインメールへ確定反映する(確定反映は確認完了処理が担う)。

| 処理 ID | 処理内容                                                       |
|---------|----------------------------------------------------------------|
| P-01    | セッションから認証済みのアカウント利用者を特定する                |
| P-02    | 入力値(メールアドレス形式・パスワード強度等)を検証する          |
| P-03    | メールアドレス・パスワード変更時は再認証トークン(リクエストボディ `reauthToken`)を検証する |
| P-04    | パスワード変更時はパスワードハッシュを `M_USER` に更新する          |
| P-05    | メールアドレス変更時は、現行ログインメール(`M_USER.email`)は変更せず新アドレスを未確認の変更中アドレス(`M_USER.pending_email`)として保持し、メール確認トークンを発行して新アドレスへ確認メールを送信する(確認完了時に現行ログインメールへ確定反映) |
| P-06    | 更新後の値を返却する                                           |

## リクエスト

#### 認証ヘッダ

本 API は利用者セッション(Cookie)と CSRF トークンを必要とする状態変更系である。

| ヘッダ | 要否 | 説明 |
|---|---|---|
| `Cookie: session=<token>` | 必須 | 利用者セッション |
| `X-CSRF-Token: <token>` | 必須 | CSRF 対策(状態変更系) |
| `Idempotency-Key: <ULID>` | 任意 | 冪等性(書き込み再送の重複防止) |

#### Query Parameters

```json
{}
```

#### Path Parameters

```json
{}
```

#### Request Body

```json
{
  "email": "new@example.com",
  "password": "P@ssw0rd-new",
  "passwordConfirm": "P@ssw0rd-new",
  "reauthToken": "reauth_..."
}
```

| 項目               | 型     | 説明                                                       |
|--------------------|--------|------------------------------------------------------------|
| `email`            | string | 新しいメールアドレス(任意・変更時は `reauthToken` 必須)。確認完了までは現行ログインメールを上書きせず、未確認の変更中アドレス(`M_USER.pending_email`)として保持する |
| `password`         | string | 新しいパスワード(任意・変更時は `passwordConfirm` と `reauthToken` 必須。12 文字以上、英大小文字 / 数字 / 記号のうち 3 種類以上) |
| `passwordConfirm`  | string | 新しいパスワード確認用(パスワード変更時必須・`password` と一致) |
| `reauthToken`      | string | 再認証トークン(メールアドレス・パスワード変更時必須・API-005 で取得) |

表示名(プロフィール名)は本 API では更新しない([API-012](API-012.md#API-012) が担う)。

## レスポンス

**レスポンス(200)**

```json
{
  "email": "owner@example.com",
  "pendingEmail": "new@example.com",
  "emailVerificationPending": true
}
```

| 項目                       | 型      | 説明                                                      |
|----------------------------|---------|-----------------------------------------------------------|
| `email`                    | string  | 現在のログインメールアドレス(`M_USER.email`。メール変更時も確認完了まで現行のまま) |
| `pendingEmail`             | string  | 未確認の変更中メールアドレス(`M_USER.pending_email`。メール未変更時は `null`) |
| `emailVerificationPending` | boolean | 確認メールを送信して検証待ちの場合 `true`(メールアドレス変更時) |

## バリデーション

入力値の検証規則は `## リクエスト` の各パラメータ説明を参照する。

## エラー

| HTTP ステータス | エラーコード                   | 内容                           |
|-----------------|--------------------------------|--------------------------------|
| 400 | `VALIDATION_ERROR` → [ERR-001](../../05_errors/ERR-001.md#ERR-001) | メールアドレス形式不正 / パスワード強度不足・不一致等の入力検証エラー |
| 401 | `REAUTH_REQUIRED` → [ERR-013](../../05_errors/ERR-013.md#ERR-013) | 再認証トークンが無効または未提示(メール・パスワード変更時) |
| 409 | `EMAIL_ALREADY_USED` → [ERR-014](../../05_errors/ERR-014.md#ERR-014) | 入力メールアドレスが既に使用中 |

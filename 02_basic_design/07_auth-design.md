# 認証・認可設計書

> **このページは、利用者側(オーナー / メンバー)の認証(本人確認)と認可(操作可否)の方針・判定フローを定義します。**認証主体は全ユーザー共通の `M_USER` で、セッション・トークンは `M_USER`(`user_id`)を保持します。種別はオーナー(`M_CONTRACT.user_id` 一致)/ メンバー(`M_PRJ_USERS` の有効割当)で導出します。メールは `M_USER` で一意(1 メール = 1 アカウント)です。メンバーは 1 オーナー配下に固定し、別オーナーへの重複所属は招待時に拒否します(FR-185)。

*版数 v2.3 ・ 更新 2026-06-20 ・ 承認済*

## <span id="1-認証方式"></span>1. 認証方式

本システムが採用する本人確認の手段を一覧で示します。次の表は、各方式の採否と用途、有効期限などの制約・正本リンクを並べたもので、利用者の登録から日常ログイン、パスワード再設定、メンバー有効化までを 1 つの方式表で見渡せます。

| 認証方式 | 採否 | 説明 | 制約・参照 |
|----|----|----|----|
| メールアドレス + パスワード | 採用 | 主たる認証方式。ログイン時にパスワードを検証する | パスワード要件は §2、ハッシュ方式は 要件のセキュリティ NFR が正本 |
| パスワード再設定リンク | 採用 | メールに送る短期トークンで本人到達を確認し再設定させる | 有効期限 **1 時間** / 1 回限り(FR-006)。トークン定義は [T_ACCESS_TOKENS](04_database/TBL-T-002.md) |
| 招待トークン(メンバー有効化) | 採用 | 招待リンクから本人が初回パスワード・氏名・規約同意を入力して有効化する | 有効期限 **7 日**(FR-018 / FR-032)。フローは §3 |
| メール確認トークン | 採用 | 新規登録後の本人確認(FR-003) | 画面は [画面設計.md](01_screens/index.md)(SCR-013) |
| 規約再同意の割込み | 採用 | 利用規約・プライバシーポリシー改定時にログイン後フローへ割込む(FR-010) | 認可割込みルールは §6 |
| Bot 対策(CAPTCHA 等) | 採用 | 必要に応じてログイン等で発動する(FR-149) | 判定方針は 要件のセキュリティ NFR が正本 |

### <span id="11-パスワード要件"></span>1.1 パスワード要件

利用者が設定できるパスワードの強度ルールをまとめます。次の表は最低長・文字種・再設定リンクの有効期限を定めたもので、ハッシュ化など保存方式は 要件のセキュリティ NFR が正本です(FR-144)。

| 項目 | 値・制約 |
|----|----|
| 最低長 | **12 文字** |
| 文字種 | 英大文字・英小文字・数字・記号のうち **3 種類以上** |
| パスワード再設定リンクの有効期限 | **1 時間**(超過時は再送導線を提示) |

> [!NOTE]
> **ハッシュ方式の正本** 保存時のハッシュアルゴリズムとパラメータ(ストレッチング等)は 要件のセキュリティ NFR を正本とし、本書では「ハッシュ化して保存する(FR-144)」方針のみを示します。

## <span id="2-再認証"></span>2. 重要操作の再認証

不可逆・高リスクな操作の直前に、ログイン済みでも改めて本人確認を求めます。最初の表で再認証の有効範囲(1 回かつ 15 分以内)を定め、続く表で対象となる重要操作を列挙します。対象は FR-005 で 5 種に固定されています。

| 項目 | 内容 |
|----|----|
| 有効範囲 | **「当該操作 1 回かつ 15 分以内」**。1 回の操作で消費し、15 分を超えた場合は無効となり再度求める |
| ダイアログ画面 | [画面設計.md](01_screens/index.md) が正本 |

| 対象の重要操作 | 説明 | 根拠 |
|----|----|----|
| パスワード変更 | 本人のパスワード変更 | FR-005 |
| 退会 | オーナーによる退会申請(SCR-014) | FR-005 |
| 課金情報変更 | 支払方法等の変更 | FR-005 |
| メンバーの登録・停止・削除 | メンバーの招待 / プロジェクト割当解除 / 削除などのメンバー管理操作 | FR-005 |
| 月次上限件数の変更 | プロジェクト単位の月次上限件数・無料枠の変更 | FR-005 |

> [!IMPORTANT]
> **対象範囲は要件で固定** 再認証の対象は上記 5 種に固定します(FR-005 / BR-002)。再認証が未保持または期限切れのまま対象操作の API に到達した場合は認可で拒否し、再認証を要求するエラー分類(`E-AUTH-REAUTH-REQUIRED`)に落とします。エラー定義は 各 API のエラー仕様 が正本です。

## <span id="3-認証フロー"></span>3. 認証フロー

ログインからログアウト、パスワード再設定、メンバー有効化、メール確認、強制ログアウトまで、認証にかかわる各フローの処理手順をまとめます。以降の各小節は、それぞれのフローを順序付きの手順表で示します。

### <span id="31-ログイン"></span>3.1 ログイン

通常ログイン時の処理を入口から初期画面遷移まで順に追います。資格情報は全ユーザー共通の `M_USER` で照合し、メールは `M_USER` で一意(1 メール = 1 アカウント)です。メンバーは 1 オーナー配下に固定し、別オーナーへの重複所属は招待時に拒否します(FR-185)。セッションは認証した `M_USER`(`user_id`)を保持し、オーナー / メンバーの種別は保持せず導出します。次のシーケンス図は、資格情報の検証・セッション確立に加え、規約再同意ゲート(§6)と契約状態ゲート(§5.7)をログイン直後に通す順序と、不一致・停止状態などの分岐を示します。

```mermaid
sequenceDiagram
  participant U as ブラウザ(利用者)
  participant W as worker-main-api
  participant D as D1 (M_USER)
  participant K as KV (session)

  U->>W: メール + パスワード送信
  W->>D: M_USER で資格情報を照会(メールで一意に識別)
  D-->>W: 照合結果(一致 / 不一致)
  alt 不一致
    W-->>U: 認証失敗(失敗カウント加算・ヒントなし)
  else 一致
    W->>K: セッション確立(認証した M_USER を user_id で保持)
    K-->>W: 
    W-->>U: セッション Cookie 発行
    W->>D: 規約再同意の要否を確認
    D-->>W: 未同意文書(あり / なし)
    opt 未同意あり
      W-->>U: 規約再同意へ割込み(SCR-015)
    end
    W->>D: 契約状態を確認
    D-->>W: 契約状態(active / suspended / deleted_pending / deleted)
    alt active
      W-->>U: 管理範囲に応じた初期画面へ遷移
    else suspended / deleted_pending / deleted
      W-->>U: 停止時アクセス制限(§5)を適用
    end
  end
``` > [!NOTE]
> **補足** 不一致時は失敗カウントを加算し、ヒントを与えない応答を返します(必要に応じ Bot 対策チャレンジを挟む、FR-149)。契約状態が `suspended` / `deleted_pending` / `deleted` の場合は §5 のアクセス制限に従います。

### <span id="32-ログアウト"></span>3.2 ログアウト

利用者が明示的にログアウトしたときの扱いを定めます。次の表のとおり、操作したセッションのみを失効させ、他デバイスのセッションは維持します(FR-182)。

| 項目 | 内容 | 参照 |
|----|----|----|
| 当該セッション | 失効させ、セッション Cookie を破棄してログイン画面へ戻す | — |
| 他セッション | 複数デバイスでログイン中の他セッションは維持する | FR-182 |

### <span id="33-パスワード再設定"></span>3.3 パスワード再設定

パスワードを忘れた利用者がメール経由のトークンで再設定する手順を示します。次のシーケンス図は、列挙攻撃を避ける一律応答、トークンの有効期限(1 時間・1 回限り)、トークンの有効 / 期限切れ分岐、再設定後の全セッション失効までを示します。

```mermaid
sequenceDiagram
  participant U as ブラウザ(本人)
  participant W as worker-main-api
  participant D as D1 (T_ACCESS_TOKENS)
  participant M as メール送信

  U->>W: メールアドレスを入力
  W->>D: リセットトークンを発行(有効期限 1 時間・1 回限り)
  D-->>W: 
  W->>M: リセットリンクを送付
  M-->>W: 
  W-->>U: 存在有無を判別できない応答(列挙攻撃対策)
  U->>W: トークン + 新パスワードを送信
  W->>D: トークンを検証
  D-->>W: 検証結果(有効 / 期限切れ・使用済み)
  alt 期限切れ・使用済み
    W-->>U: 無効を通知(再送導線を提示)
  else 有効
    W->>D: パスワード更新 + 全セッションを失効
    D-->>W: 
    W-->>U: ログイン画面へ誘導
  end
``` > [!NOTE]
> **補足** トークン定義(有効期限 1 時間・1 回限り)は [T_ACCESS_TOKENS](04_database/TBL-T-002.md) が正本。新パスワードは §1.1 の要件を満たすこと。

### <span id="34-招待受諾メンバー有効化"></span>3.4 招待受諾(メンバー有効化)

招待されたメンバーが、招待リンクから氏名・初回パスワード・規約同意を入力してアカウントを有効化するまでを示します。標準フローは次のシーケンス図、複数プロジェクト参加や既存アカウント一致といった例外の扱いは続く表で示します(FR-012 / FR-022)。

```mermaid
sequenceDiagram
  participant O as 招待元(オーナー / メンバー)
  participant W as worker-main-api
  participant M as メール送信
  participant I as 招待された本人
  participant D as D1

  O->>W: メールアドレスを指定して招待(氏名なし)
  W->>D: 招待トークン発行 + 割当を予約状態で保持(有効期限 7 日)
  D-->>W: 
  W->>M: 招待リンクを送付
  M-->>W: 
  W-->>O: 
  M-->>I: 招待リンク受領(有効化前はログイン不可)
  I->>W: 氏名・初回パスワード・規約同意を入力
  W->>D: 氏名・初回パスワード・規約同意・割当をまとめて確定
  D-->>W: 
  W-->>I: 有効化完了
```

| 状況 | 扱い | 参照 |
|----|----|----|
| 同一の人を複数プロジェクトへ参加させる | 各プロジェクトで個別に招待する | FR-012 |
| 同一オーナー配下で既存の有効・招待中アカウントとメールアドレスが一致 | 同一の利用者として扱い、新たに参加するプロジェクト分の割当のみを追加する(氏名入力を伴う有効化ページはバイパスされる) | FR-036 |

> [!WARNING]
> **有効化前 Principal の限定** 有効化ページは未認証の本人が招待トークン経由で到達するため、通常のセッションを持ちません。有効な招待トークン(期限内・未使用)の保持を唯一の認可条件とし、当該ページの有効化処理に限ってトークン由来の Principal を用います。本ページ以外でこの Principal は使用しません。

### <span id="35-メール確認"></span>3.5 メール確認

新規登録後の本人到達をメール確認トークンで検証します。次の表は本人確認の方法と、確認完了までアカウントを未確認状態として利用を制限する扱いを示します(FR-003)。

| 項目 | 内容 | 参照 |
|----|----|----|
| 本人確認 | 新規登録後にメール確認トークンを送り、リンク到達で本人確認を完了する | FR-003 |
| 確認前の扱い | 確認完了までアカウントは未確認状態とし、利用を制限する | §5.6 アカウント状態 |

### <span id="36-強制ログアウト"></span>3.6 強制ログアウト

契約停止イベントを契機に、配下利用者のセッションをどう失効させるかを定めます。次のシーケンス図は、手動停止・規約違反停止での全セッション無効化、長時間処理中の扱い、停止後の次回操作での再認証・停止時アクセス制限(§5)を示します(FR-011 / BR-007)。

```mermaid
sequenceDiagram
  participant OP as 契約管理
  participant W as worker-main-api
  participant K as KV (session)
  participant U as 配下利用者

  OP->>W: 停止 / 規約違反停止を契機として連携
  W->>K: 配下利用者の全セッションを速やかに無効化
  K-->>W: 
  W-->>OP: 
  Note over W: 進行中の長時間処理は完了まで継続(中断しない)
  U->>W: 次回操作
  W-->>U: 再認証を要求 / 停止時アクセス制限(§5)を適用
``` > [!NOTE]
> **強制ログアウトの契機** 強制ログアウトは契約停止イベントを契機とします。本書は「契機を受けてセッションをどう失効させるか」の方針に限定します。

## <span id="4-セッション設計"></span>4. セッション設計

ログイン状態をどれだけ保持し、いつ・どの契機で失効させるかをまとめます。次の表は無操作タイムアウト(30 分)・絶対タイムアウト(12 時間)・複数デバイス同時ログインなどの方針を並べたもので、ログインの寿命と失効ルールを一望できます(FR-008 / FR-182)。

| 項目 | 方針 | 根拠 |
|----|----|----|
| 無操作タイムアウト | **30 分**。最終操作から無操作が続くとセッションを失効させる | FR-008 / BR-004 |
| 絶対タイムアウト | **12 時間**。ログインからの経過で失効させ、1 営業日内に再認証を求める | FR-008 / BR-005 |
| 複数デバイス同時ログイン | 同一アカウントで複数デバイスから同時にログイン可 | FR-182 / BR-006 |
| 契約停止時の全失効 | 契約停止時、配下利用者の全セッションを速やかに無効化(§3.6) | FR-011 |
| 強制ログアウト | 当該アカウントの全セッションを強制失効(最優先) | FR-182 |
| 操作と同時失効時の挙動 | 操作中にセッションが失効した場合は明示メッセージを表示し再ログインへ誘導する | FR-008 |
| セッション内不正利用・改ざん対策 | 状態変更操作でセッションの不正利用・改ざんを検知・防止する(具体方式は 要件のセキュリティ NFR 正本) | BR-009 |

> [!NOTE]
> **アクティブセッションの自己管理** 利用者自身によるアクティブセッション一覧表示・自己セッション終了は MVP 対象外です(FR-182)。強制ログアウトは提供します。

### <span id="41-ログイン失敗ロックアウト"></span>4.1 ログイン失敗ロックアウト

連続したログイン失敗から総当たり攻撃を防ぐためのロックアウト条件を定めます。次の表は発動条件(5 回連続失敗)・効果(15 分ロック)・復旧手段・通知をまとめたもので、ロック中の到達は `E-AUTH-LOCKED` に落とします。

| 項目 | 内容(値・制約) | 参照 |
|----|----|----|
| 条件 | ログインが **5 回連続** 失敗 | — |
| 効果 | 当該アカウントを **15 分間** ロックする | — |
| 復旧 | **時間経過** または **オーナー / 当該プロジェクトのメンバーによる解除** | — |
| 通知・救済 | ロック発動時は該当本人および必要なオーナー / メンバーへ通知し、正規利用者の救済導線を提供する | 宛先・文面は [メール設計書](06_mail-design.md) 正本 |
| ロック中のログイン到達 | ロック分類のエラー(`E-AUTH-LOCKED`)に落とす | エラー定義は 各 API のエラー仕様 が正本 |

### <span id="42-失効の優先順位"></span>4.2 失効の優先順位

複数の失効要因が同時に当てはまるとき、どれを優先して判定するかを定めます。次の表は強制ログアウトを最優先とし、絶対タイムアウト・無操作タイムアウト・通常セッションの順で評価する優先順位です。重要操作の再認証(§2)はこの寿命判定とは別軸で働きます。

| \#  | 判定                   | 補足・参照                         |
|-----|------------------------|------------------------------------|
| 1   | **強制ログアウト**     | 最優先。全セッションを失効(FR-011) |
| 2   | **絶対タイムアウト**   | 12 時間(FR-008)                    |
| 3   | **無操作タイムアウト** | 30 分                              |
| 4   | **通常セッション**     | 有効期間内                         |

重要操作の再認証(§2、FR-005)は上記の寿命判定とは別軸で、通常セッションが有効でも対象操作の直前に追加の認証を要求する。

## <span id="5-認可モデルと判定フロー"></span>5. 認可モデルと判定フロー

誰が何を操作できるかを判定する認可モデルを定めます。ユーザー種別とオーナー判定、契約・プロジェクトの境界、判定段の順序、専有・保護制約、通知宛先解決、アカウント状態・契約状態によるアクセス制限を以降の小節で順に示します。

### <span id="51-ユーザー種別とオーナー判定"></span>5.1 ユーザー種別とオーナー判定

認可の起点となるユーザー種別と、その見分け方を定めます。認証主体は全ユーザー共通の `M_USER` で、セッション / トークンは `user_id` で `M_USER` を指します。種別は導出で判定し、次の表は認証主体が `M_CONTRACT.user_id` に一致すればオーナー、`M_PRJ_USERS` に有効割当を持てばメンバー(割当の有無で操作範囲が決まる)であることを示します(FR-013 / FR-014)。

| ユーザー種別 | 判定 | 権限の表し方 |
|----|----|----|
| オーナー | 認証主体(`M_USER`)が `M_CONTRACT.user_id` と一致(`isOwner`)。`M_CONTRACT.id` が契約境界キー本体で 1 契約 1 オーナーを担保する | 全プロジェクトへの全権 + オーナー専有機能。役割変更・降格・譲渡なし(FR-013) |
| メンバー | 認証主体(`M_USER`)が `M_PRJ_USERS` に有効割当(`valid=1`)を持つ。当該プロジェクトへの割当の有無で操作範囲が決まる(役割差なし) | 割り当てられたプロジェクト内で旧管理者相当の全操作。プロジェクト内の役割差は持たない(FR-014 / FR-186) |
| ウィジェット利用者 | 管理画面の主体行を持たない | 公開キーとウィジェットセッションで質問送信のみ。管理画面の Principal は構築しない |

> [!IMPORTANT]
> **オーナー判定 = isOwner bypass を先頭に** オーナーは `M_CONTRACT` マスタの独立した主体です。オーナーは自プロジェクトに `M_PRJ_USERS` のメンバー行を自動保持しますが、**認可権威は `M_CONTRACT.user_id` がセッションの `user_id` と一致すること(`isOwner=true`)による bypass を最優先**とし、`M_PRJ_USERS` の owner 行は一覧表示・通知宛先の網羅用で専有判定には用いません。オーナーは自契約配下の全プロジェクトに無条件でアクセスできます(FR-016 / FR-035)。メンバー権限は `M_PRJ_USERS` の有効割当(`valid=1`)でのみ付与されます。通知宛先解決と画面表示では、オーナーを `M_CONTRACT` から、当該プロジェクトの有効メンバーを `M_PRJ_USERS`(`valid=1`)から個別に解決します(§5.5)。なお監査ログ・通知宛先・作成者・申請者の記録に用いる `actor_type` + `actor_id`(`owner`→`M_CONTRACT` / `project_user`→`M_PRJ_USERS`)は、認証主体 `M_USER` から操作時に導出した種別を記録するための識別子であり、セッション自体は保持しません。規約同意は `M_USER` 直結(`user_id`)で記録するため、この `actor_type` の対象には含めません。

### <span id="52-オーナー境界判定"></span>5.2 オーナー境界判定

他契約のデータへ越境させないための境界チェックを定めます。次の表は操作者の契約境界キーと対象リソースの `contract_id` の一致を要求し、不一致は相手契約の存在を漏らさないため 404 偽装で拒否することを示します(FR-185)。

| 観点 | 内容 | 参照 |
|----|----|----|
| 判定 | 操作者の契約境界キーと、操作対象リソースの `contract_id` が一致すること | — |
| 不一致時 | 相手契約の存在を漏らさないため **404 偽装** で拒否する(`E-AUTHZ-OWNER-BOUNDARY`) | エラー方針は 各 API のエラー仕様 正本 |
| メンバーの 1 オーナー固定 | メールは `M_USER` で一意(1 メール = 1 アカウント)。メンバーは 1 オーナー配下に固定し、別オーナーへの重複所属は招待時に拒否する | FR-185 |

### <span id="53-認可判定の順序"></span>5.3 認可判定の順序

1 リクエストを許可するまでに通す認可判定の段を、上から評価する順序で定めます。次の表はセッション検証から始まり、契約状態ゲート・オーナー判定・プロジェクト境界・専有機能・再認証・利用上限までを 11 段に分け、各段の拒否時エラー分類を併記したものです。

| \# | 判定段 | 内容 | 拒否時のエラー・参照 |
|----|----|----|----|
| 1 | **セッション検証** | 無操作 30 分 / 絶対 12 時間を満たす有効セッションか | 失敗は `E-AUTH-SESSION-EXPIRED` |
| 2 | **アカウント有効性** | アカウントが利用可能状態か(無効化済みなら再ログインへ誘導、§5.6) | — |
| 3 | **規約再同意ゲート** | 改定済みで未同意の文書があれば SCR-015 割込みへ(§6) | `E-AUTHZ-TERMS` |
| 4 | **契約状態ゲート** | `suspended` / `deleted_pending` / `deleted` 時は §5.7 のアクセス制限を適用 | `E-BIZ-CONTRACT-SUSPENDED` 等 |
| 5 | **オーナー判定(isOwner bypass)** | 認証主体(`M_USER`)が `M_CONTRACT.user_id` と一致すれば自契約配下を無条件許可(§5.1) | — |
| 6 | **オーナー境界判定** | 非オーナーは契約境界キー一致を要求。不一致は **404 偽装**(§5.2) | `E-AUTHZ-OWNER-BOUNDARY` |
| 7 | **プロジェクト境界判定** | 対象プロジェクトへの割当があること(役割差なし)。割当なしは **404 偽装** | `E-AUTHZ-PROJECT-DENIED` / `E-AUTHZ-FORBIDDEN`(FR-189) |
| 8 | **オーナー専有機能判定** | 専有機能(§5.4)を非オーナーが要求した場合は 403 | `E-AUTHZ-OWNER-ONLY` |
| 9 | **オーナー保護・自己操作禁止** | §5.4 の不可制約に該当すれば拒否 | `E-AUTHZ-OWNER-PROTECTED` / `E-AUTHZ-SELF-MUTATION` |
| 10 | **再認証判定** | 重要操作(§2)で再認証が「当該操作 1 回 + 15 分以内」を満たすか | 未充足は `E-AUTH-REAUTH-REQUIRED` |
| 11 | **利用上限判定** | 認可通過後に上限を確認する(レート = 契約単位、月次上限件数・無料枠 = プロジェクト単位) | [課金・請求設計書](05_billing-design.md) が正本 |

> [!WARNING]
> **権限外は原則非表示・直アクセス拒否** 権限外の主ナビゲーションと操作は**原則として表示しない**。共同作業上、存在を知らせる価値がある項目のみ読み取り専用で表示し、必要権限と依頼先を説明する。権限のない画面への直アクセスは、同等の説明と戻り先を添えて拒否する(FR-191)。表示制御の画面仕様は [画面設計.md](01_screens/index.md) が正本。 > [!NOTE]
> **割当変更の反映タイミング(D-03)** プロジェクト割当の追加・剥奪では**既存セッションを即時失効させず、次回の認可チェック時点で反映**します(FR-188)。認可判定で参照する割当情報のキャッシュ反映上限(TTL)は **60 秒以内** とし、これを本書で正本化します(FR-188 が反映上限時間とキャッシュ方式を本書へ委任)。変更から最大 60 秒で次回認可チェックに反映されます。

### <span id="54-オーナー保護自己操作最後の管理者保護"></span>5.4 オーナー専有機能・保護制約

非オーナーに付与してはならない専有機能と、運用が止まらないための保護制約を定めます。次の表はオーナー専有機能・オーナー保護・自己操作の禁止をまとめたもので、各制約に該当する操作は §5.3 の対応する判定段で拒否されます(FR-183 / FR-184)。

| 制約 | 内容 | 根拠 |
|----|----|----|
| オーナー専有機能 | 課金情報・支払方法・請求履歴・退会・規約再同意・契約設定、プロジェクトの作成・編集・削除はオーナー専有。メンバーには付与しない。**プロジェクト単位の利用上限・無料枠は、オーナーまたは当該プロジェクトのメンバーが変更できる** | FR-015 / BR-017 |
| オーナー保護 | オーナーに対する退会・停止・削除・降格・譲渡を一切受け付けない。オーナーの自動メンバー行は解除・降格できない。1 契約 = 1 オーナーのため孤立状態は構造的に発生しない | FR-183 |
| 自己操作の禁止 | 自分自身の利用状態変更・削除はできない。自身のアカウント削除は常に不可で、他人を経由して自身を削除する経路も設けない | FR-184 |

### <span id="55-通知宛先解決"></span>5.5 critical 通知の宛先解決(共有概念正本)

critical 通知を「誰に送るか」を決める宛先解決ロジックを本書で正本化します。次の表は、オーナー(`M_CONTRACT`)と当該プロジェクトの有効メンバー(`M_PRJ_USERS valid=1`)を両マスタから個別に抽出して合算する手順、重複排除、プロジェクト限定通知の絞り込みをまとめたものです。配信契機・文面は [メール設計書](06_mail-design.md) / [メール設計書](06_mail-design.md) が正本です。

| 観点 | 内容 | 参照 |
|----|----|----|
| 網羅性 | オーナーは自プロジェクトにメンバー行を持つが、網羅と重複排除のため、宛先は `M_CONTRACT` 由来のオーナーと `M_PRJ_USERS valid=1` 由来の有効メンバーの **2 マスタを合算**して解決する | — |
| 重複排除 | 同一のプロジェクトユーザーが複数プロジェクトに割り当てられている場合やオーナーが両マスタに現れる場合の二重配信を避けるため、認証主体(`M_USER` の `user_id`)で重複排除する。メールは `M_USER` で一意(1 メール = 1 アカウント)のため、同一メールは常に同一 `M_USER`(同一宛先)として扱う(FR-185) | — |
| プロジェクト限定通知 | 質問数上限の選択アラート等は対象プロジェクトに限定し、当該プロジェクトの有効メンバー(オーナー + 当該 PJ の `valid=1` 割当)のみに配信する | FR-034 |

> [!NOTE]
> **配信契機・文面の正本** 通知契機・配信先一覧は [メール設計書](06_mail-design.md)、本文テンプレートは [メール設計書](06_mail-design.md) が正本です。本書は「誰が宛先か」を決める解決ロジックのみを定めます。

### <span id="56-アカウント状態"></span>5.6 アカウント状態と利用可否

アカウントの各状態でログインと利用をどこまで許すかを定めます。次の表は有効・招待中・メール未確認・ロック中・無効化の状態ごとに、ログイン可否と許される利用範囲を示したものです(FR-189)。

| 状態 | ログイン可否 | 利用範囲 |
|----|----|----|
| 有効 | 可 | 割当に応じた操作 |
| 招待中(有効化前) | 不可 | 招待リンクからの有効化のみ。割当は予約状態(FR-021) |
| メール未確認 | 制限 | メール確認の完了のみ(FR-003) |
| ロック中 | 不可(15 分) | 時間経過またはオーナー / メンバーによる解除で復旧(§4.1) |
| 無効化(最後の割当解除) | 不可 | 全セッション・未使用招待を失効。割当のないプロジェクトへの直アクセスは拒否(FR-189) |

> [!IMPORTANT]
> **最後の割当解除による自動無効化** メンバーの最後の有効なプロジェクト割当が解除されたとき、対象アカウントを**自動で無効化**し、全ログインセッションと未使用招待を失効させます(FR-031 / FR-189)。利用停止後に同じメールアドレスで再招待された場合は、過去の削除済みデータとは別の新しい利用者として扱います。一定期間(90 日)経過後にデータが消去される旨は割当解除の確認時に明示します。契約横断のメンバー管理画面は提供しません。

### <span id="57-契約状態によるアクセス制限"></span>5.7 契約状態によるアクセス制限

契約そのものの状態が利用に与える制限を定めます。次の表は停止中・退会申請中・退会済みの各 `status` について、管理画面で許す操作とセッションの扱いを示したものです。支払方法ゲートはこの契約停止とは別である点に注意します(表後の補足)。

| 契約状態 | 管理画面の許可操作 | セッション |
|----|----|----|
| 停止中(`suspended`) | 課金・退会のみ許可。その他は 403 | 手動停止 / 規約違反停止は全失効(§3.6)。支払猶予切れは既存セッション継続 |
| 退会申請中(`deleted_pending`) | 参照は可、新規書込は禁止。退会キャンセル動線を提供 | オーナーのみログイン可 |
| 退会済み(`deleted`) | 不可 | ログイン不可 |

> [!NOTE]
> **支払方法ゲートは契約停止ではない** 支払方法未登録 + 無料枠超過によるウィジェット質問受付の停止は契約サスペンションではありません(`status` は `active` のまま)。本表のアクセス制限・セッション無効化の対象外で、管理画面は通常どおり利用できます。詳細は [課金・請求設計書](05_billing-design.md) 正本。

## <span id="6-規約再同意の認可割込み"></span>6. 規約再同意の認可割込み

規約改定時に、ログイン後の認可フローへ再同意画面を割り込ませる発火条件と段階適用を定めます。次の表は判定対象(改定文書のみ)・割込み箇所(§5.3 \#3)・予告と同意期限・契約単位の段階適用をまとめたもので、制限内容の正本は 本書 です(FR-010)。

| 観点 | 内容 | 参照 |
|----|----|----|
| 判定対象 | 利用規約・プライバシーポリシーのうち**改定された文書のみ**。文書は `doc_type` 別に版管理し、最新版未同意のものだけ再同意を求める | — |
| 割込み | ログイン後の認可フロー(§5.3 \#3)で未同意を検知したら SCR-015 へ割込む。割込み中は他画面の操作を許さない | — |
| 予告と期限 | 発効 30 日前に予告し、同意期限は発効日 + 14 日 | FR-010 |
| 段階適用 | 全契約同日強制ではなく**契約単位で段階的に**発効させる | — |
| 非同意 | 退会の案内へ誘導する | — |
| 同意期限超過後の機能制限 | ログイン自体は可能とし、機能を制限する。本書は割込みの発火条件と段階適用を定める | 制限内容の正本は 本書 |

> [!IMPORTANT]
> **同意・割込みはオーナー専有** 規約再同意の同意 / 不同意操作はオーナー専有機能です(FR-015)。再同意未完了の契約に属するメンバーは、契約側の同意完了までゲートの影響を受けます。

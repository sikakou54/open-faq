# FAQ ドキュメント保守ルール

本ファイルはリポジトリレベルのドキュメント保守ルールの正本である。要件定義書・基本設計書・個別設計書・詳細設計書・画面遷移図(ワイヤーフレーム)から参照される。

---

## フェーズ境界

| ドキュメント | 範囲 | 含めない内容 |
|---|---|---|
| 要件定義書 | WHAT(何を提供するか)、優先度、制約、受入条件 | DDL、API スキーマ、cron 式、Worker 名、実装パラメータ |
| 基本設計書 | システム全体方針 / アーキテクチャ / 機能一覧 / 非機能方針 / 設計方針 / 個別設計書への参照 | 画面詳細、API 詳細、DB 詳細、個別エラー / メッセージ / 認可ルール / 課金詳細 |
| 個別設計書(11 種) | 各設計領域の正本(画面 / API / DB / 権限 / エラー / メッセージ / トレーサビリティ / 認証認可 / セキュリティ / 課金請求) | 他領域の詳細(参照のみ) |
| 画面遷移図(ワイヤーフレーム HTML) | エンドユーザー UI 表現のみ | FR / NFR / DB 列名 / 画面間設計根拠 |
| トレーサビリティマトリクス | 要件 → 設計 → テスト観点の対応 | 個別仕様の詳細本文 |
| 詳細設計書 | 実装関連の詳細(モジュール構成 / ロジック / バッチ / ログ / 監視 / テスト / 実装ガイドライン) | 個別設計書が正本とする DDL / Zod / API スキーマ / 画面項目 / 状態遷移 / 権限 / エラー / セキュリティ |
| 将来拡張ドキュメント | MVP 後の候補・設計 / 実装バックログ | MVP 本文ドキュメントからの参照 |

## 変更適用順序(5 段階)

仕様または設計の変更要求があった場合、ユーザーが特定のドキュメントだけを指し示していたとしても、必ず以下の **固定 5 段階順序** で全層を調査・適用すること。途中で省略すると下流に古い前提が残る。

1. **要件定義書** (`01_メインシステム/01_要件定義書.md`、`02_運営者システム/01_要件定義書.md`)
2. **基本設計書** (`01_メインシステム/02_基本設計書.md`、`02_運営者システム/02_基本設計書.md`) + **該当する個別設計書(11 種)** ── 下記「変更種別 → 影響ドキュメント・マトリクス」に従って機械的に判別
3. **画面遷移図(ワイヤーフレーム)** (`01_メインシステム/画面遷移図.html`、`02_運営者システム/画面遷移図.html`)
4. **トレーサビリティマトリクス** (`01_メインシステム/個別設計書群/08_トレーサビリティマトリクス.md`、`02_運営者システム/個別設計書群/08_トレーサビリティマトリクス.md`)
5. **詳細設計書** (`01_メインシステム/03_詳細設計書.md`、`02_運営者システム/03_詳細設計書.md`) ── 実装関連の詳細のみ更新(個別設計書が正本とする仕様は更新対象外)

各層について、変更により削除・改名された語句(FR ID、列挙値、列名、画面 / 項目ラベル、メッセージ ID、エラー ID)をファイル内検索して取りこぼしを修正する。最後に `99_スクリプト/仕様同期検査.sh` を実行する。ある層に関連内容が無いと判明した場合は、暗黙にスキップせず変更サマリへ明示的に記録する。

## モーダル画面

意味のある状態、バリデーション、業務ロジックを持つモーダルダイアログ(プロジェクト作成 / 編集、規約再同意、4-eyes 承認モーダル等)は、呼び出し画面に埋め込まず **独立した画面として SCR ID を付与して定義する**。

- **ID 規約**: 親画面の SCR ID に `-M<n>` サフィックスを付ける。親画面下の最初のモーダルは `SCR-<親>-M1`、同じ親に複数モーダルがあれば `-M2` / `-M3` …(例: SCR-010 の編集モーダル → `SCR-010-M1`)
- **配置**: モーダル行 / セクション / ワイヤーフレームブロックを、すべてのドキュメントで **親画面の直後** に置く ── 要件定義の SCR 一覧、画面設計書 §2 / §5、サイドメニュー除外一覧、画面遷移図、トレーサビリティマトリクス §3 / §4。SCR ID をアルファベット順にソートすると親画面の直下にモーダルが並ぶ
- **コンテンツ分割**: 親画面側は「モーダルを開く」操作のみを記載する。入力項目、バリデーション、保存 / キャンセルボタンはすべてモーダル SCR の項目表に記載する
- **モーダル画面に該当しないもの**: 単純な確認ダイアログ(「本当に削除しますか?」yes/no)や再認証チャレンジは独立 SCR にせず、呼び出しボタンの注記として `(確認ダイアログ)` のままインライン記載する

## 同期ルール

メイン / 運営者間の共有概念の正本ルールは [共有/共有概念.md](共有/共有概念.md) で正本化する。本書には主要ルールの要約のみ記載:

- 共有概念が変更される場合は、**共有概念.md 正本ルール表で示された正本ドキュメントを必ず更新** し、参照側ドキュメントはリンクのみ更新する(再掲禁止)
- 共有概念には以下を含む(全項目は共有概念.md 参照): `accounts.contract_status` / `case_status` / 通知重要度 / SCR ID / AC ID / IF #1〜#12 / retention class / 法令プライバシー制約 / **オーナー直接所有のユーザーモデル**(`accounts.is_owner` + `accounts.owner_account_id` + `account_permissions` + `account_project_grants` ── メインを正本)/ ハッシュチェーン監査(運営者を正本)/ IP allowlist(運営者を正本)/ PII 暗号化(メインを正本)/ 暗号鍵管理(メインを正本)/ 4-eyes 承認基盤(運営者を正本)/ Stripe Webhook 受信(運営者を正本)等
- `accounts.contract_status` は `active` / `suspended` / `deleted_pending` / `deleted` に固定し、オーナー行(`is_owner=1`)でのみ意味を持つ
- `case_status=closed` は自動 retention 処理で設定しない。管理者の確定(運営者によるクローズ要求の承認を含む)のみがクローズ可能とする
- 通知重要度は `low` / `normal` / `high` / `critical`。`critical` はメール送信が必須となるイベント(オーナー + 同一オーナースコープで `users:manage` フラグを保持する全メンバーへ配信)に予約
- 利用者側アカウントは **オーナー**(契約あたり 1 アカウント固定、全権、MVP では譲渡不可、`accounts.owner_account_id = accounts.id` で自己参照) と **メンバー**(0..N、`owner_account_id` でオーナーに紐付き、メンバー権限フラグ `faq:manage` / `chat:respond` / `users:manage` / `project:manage` / `logs:view` および `account_project_grants` のプロジェクト割当で制御) に分かれる。オーナー専有機能(課金、退会、規約再同意)はメンバーに付与できない。プロジェクト割当が 0 のメンバーはダッシュボードとプロジェクト非依存機能のみ利用可能
- `audit_logs.retention_class` は **クラス名表記**(`general` / `billing` / `operator_high_priv`)に統一(期間表記 `1y` / `5y` / `7y` は人間可読補助)

## 検証

要件定義書・設計書を編集した後は、同期スモークチェックを実行すること:

```sh
99_スクリプト/仕様同期検査.sh
```

仕様同期検査.sh は SC-001〜SC-010 を実装している:

| 検査 ID | 検査内容 |
|---|---|
| SC-001 | 各個別設計書ファイルの必須ヘッダ存在 |
| SC-002 | 共有概念正本ファイルでの必須キーワード出現 |
| SC-003 | 参照側ドキュメントでの再掲禁止(リンクのみで実装、深い相互参照は手動レビュー) |
| SC-004 | SCR ID クロスチェック(画面設計書 02 とトレーサビリティ 08 に全 SCR 出現) |
| SC-005 | メッセージ ID 体系 `MSG-SCR-*` の出現確認 |
| SC-006 | エラー ID 体系 `E-AUTH-*` / `E-AUTHZ-*` / `E-BIZ-*` / `E-OP-AUTH-*` / `E-OP-4EYES-*` の出現確認 |
| SC-007 | `accounts.contract_status` 4 値が課金・請求設計書とテーブル定義書に出現 |
| SC-008 | 通知重要度 4 値(`low` / `normal` / `high` / `critical`)+ 「強制送信」文言がメッセージ一覧に出現 |
| SC-009 | 連携 IF #1〜#12 がメイン側 API 設計書 + 運営者側 API 設計書(主管責任表)に出現 |
| SC-010 | 参照リンク到達性(主要相対パスのターゲット存在確認) |

このスモークチェックは完全なレビューの代替ではない。要件定義と基本設計と個別設計書のドリフトを引き起こしたことのある高リスクな退行を検出する。

## 11 ドキュメント体系の構成

各システムは以下の 11 個別設計書 + 索引 + 共有概念対応表を持つ。

```
01_メインシステム/個別設計書群/
├── 00_索引.md                         # 索引 + SCR × ドキュメント カバレッジ
├── 02_画面設計書.md                   # 画面設計書(SCR-001〜027)
├── 03_API設計書.md                    # API 設計書(管理 API + 連携 IF #1〜#12 送信側)
├── 04_テーブル定義書.md               # テーブル定義書(全テーブル + DDL)
├── 05_権限設計書.md                   # 権限設計書(5 種フラグ + オーナー専有)
├── 06_エラー設計書.md                 # エラー設計書(E-* 体系)
├── 07_メッセージ一覧.md               # メッセージ一覧(MSG-SCR-* + 通知契機)
├── 08_トレーサビリティマトリクス.md   # トレーサビリティマトリクス
├── 09_認証認可設計書.md               # 認証・認可設計書(オーナー境界 / Principal)
├── 10_セキュリティ設計書.md           # セキュリティ設計書(PII / 鍵管理 / 不正利用検知)
└── 11_課金請求設計書.md               # 課金・請求設計書(MVP 単一プラン + Stripe Smart Retries)

02_運営者システム/個別設計書群/
├── 00_索引.md                         # 索引 + 4-eyes 対象 10 操作カバレッジ
├── 02_画面設計書.md                   # 画面設計書(SCR-090〜099 / AUTH / HOME / APPROVALS)
├── 03_API設計書.md                    # API 設計書(管理 API + Stripe Webhook 一次受信 + 連携 IF 送信主管)
├── 04_テーブル定義書.md               # テーブル定義書(operator_* / audit_logs ハッシュチェーン / webhook_events 等)
├── 05_権限設計書.md                   # 権限設計書(service_operator 単一ロール + 4-eyes 10 操作)

├── 08_トレーサビリティマトリクス.md   # トレーサビリティマトリクス
├── 09_認証認可設計書.md               # 認証・認可設計書(6 段認可判定 + 4-eyes 承認フロー)
├── 10_セキュリティ設計書.md           # セキュリティ設計書(監査ハッシュチェーン正本 / 3 区分保持 / GDPR)
└── 11_課金請求設計書.md               # 課金・請求設計書(Stripe Webhook 一次受信 / DLQ 30 日リプレイ / 月次請求確定)

共有/
└── 共有概念.md                        # 共有概念対応表 + 検査キーワード
```

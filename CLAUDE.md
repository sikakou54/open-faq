# FAQ ドキュメント保守ルール

本書(CLAUDE.md)はリポジトリレベルのドキュメント保守・運用ルールの**正本**であり、Claude Code が自動読込する指示ファイルである。運用ルールは本書に集約し、ポータル(設計書 HTML 群)とは独立して維持する(ポータル内にルールの HTML ページは置かない)。

リポジトリは「メインシステム(利用者向け FAQ ウィジェット SaaS)」と「運営者システム(SaaS 提供事業者向け管理コンソール)」の **2 系統を並列で管理し、各系統に同一構成の 5 層ドキュメントを置く**。

## 正本と形式(最重要)

- **設計書の正本は HTML**(従来の Markdown 文書管理は廃止。リポジトリ内に設計書 `.md` は存在しない)。唯一の Markdown は本書 `CLAUDE.md`(運用ルール正本 / エージェント指示ファイル)のみ。
- リポジトリルートがそのまま静的ポータルサイト(入口 `index.html`、`file://` でも動作)。各ページは本文(`<article class="content">`)だけを持ち、サイドバー / 検索 / 目次 / パンくず / テーマ切替は `assets/js/portal.js` が読み込み時に注入する(共通シェルは各ページに焼き込まない)。
- **絵文字は使わない。** アイコンは Bootstrap Icons(CDN: `cdn.jsdelivr.net/npm/bootstrap-icons`)を `portal.js` が `<head>` に読み込み、`<i class="bi bi-...">` で利用する(`currentColor` 継承でダーク/ライト自動追従)。アイコン表示には通信が必要だが、テキストラベル・`aria-label` で機能は維持される。
- 設計書本文から**他ファイルへのリンク・参照はすべて `.html`** とする。本リポジトリ管理外の外部 / 運用 runbook の markdown 成果物(例 `docs/security/pentest/YYYY-MM-DD.md`)を指す記述のみ、実体に合わせて原表記を許容する。

## 記載スタイル標準(全ページ共通テンプレート)

全設計書ページは以下の固定骨格で記述する。目的は「冒頭で全体像が掴め、結論が先に読め、ページ間で構成が揃う」こと。`<article class="content">` 内のみを編集し、`<head>` / `<body data-page-id>` は触らない。

### ページ骨格(上から順に固定)

1. **`<h1>` タイトル** — ID 接頭辞 + 簡潔名(例 `FR05: AI 回答`)。
2. **ページ要約ブロック `aside.page-summary`** — H1 直後に必ず置く。従来の「0. 文書情報 / 1. 機能概要」表は廃止し、ここへ統合する。構成:
   - `<p class="ps-lead"><strong>このページは〜を定義します。</strong> 平易な 1〜2 文。</p>` ── 専門用語を避け、読者が最初に知るべき要点を述べる。
   - `<div class="ps-grid">` 内に 2 ブロック: 左=「このページでわかること」(読者の疑問に答える 3〜5 個の箇条書き)、右=「正本・関連」(リンク集。どこが正本か明示)。
   - **要約ブロック内のラベルは見出しタグ(`h2`〜`h6`)を使わず `<p class="ps-label">` を用いる**(目次を汚さないため)。
3. **文書メタ `<p class="doc-meta">`** — 要約ブロックの直後に 1 行で。`<span><b>版数</b> v1.5</span><span><b>更新</b> 2026-06-16</span><span class="dm-status">承認済</span>` 形式。版数・更新日・ステータスを保持。更新履歴の全文は本節末尾「更新履歴」見出し配下へ移す。
4. **本編(`<h2>` 以降)** — 各層の定型骨格(下記)に従う。
5. **更新履歴**(任意・末尾)— `<h2>更新履歴</h2>` + 表。冒頭メタを軽くするため詳細はここへ。

### 文章ルール(冷たく難しい文章の解消)

- **結論先出し(PREP)**: 各 `<h2>` / `<h3>` の直後に `<p class="section-lead">` で 1〜2 文の結論リードを置き、その節で何が決まるか・なぜそうするかを述べてから表へ入る。**表をいきなり置かない**。**ただし要件定義層は section-lead を置かない**(後述「要件定義テンプレート(v2)」が優先)。
- **一文一義**: 名詞の積み上げを避け、主語と述語を明確にする。長文は分割する。
- **能動・平易**: 受動の連鎖を避ける。社内固有語・略語は初出で 1 行説明するか正本へリンクする(`<span class="term-def" title="...">用語</span>` 可)。
- **箇条書きと表の使い分け**: 並列項目で属性が 1 種なら箇条書き、属性が 2 列以上なら表。1 行 1 列だけの「擬似表」は箇条書きか `doc-meta` に置き換える。

### 注記(callout)の使い分け

`<div class="callout 種別"><i class="bi bi-..."></i><div><span class="c-title">見出し</span>本文</div></div>` の形で使う(絵文字禁止、Bootstrap Icons)。

| 種別 | クラス | アイコン | 用途 |
|---|---|---|---|
| 補足 | `callout note` | `bi-info-circle` | 補足説明・前提 |
| 重要 | `callout important` | `bi-pin-angle` | 見落とすと困る正本ルール |
| 注意 | `callout warning` | `bi-exclamation-triangle` | 破壊的操作・誤りやすい点 |
| ヒント | `callout tip` | `bi-lightbulb` | 推奨・ベストプラクティス |

### 表の作法

- 列順は「識別子 → 名称 → 説明 → 値 / 制約 → 参照」を基本とする。
- コード値・ID・列名は `<code>` で表記。空欄は `—`(全角ダッシュ)で明示し空セルを残さない。
- 正本でない値の再掲は禁止。`<a href="...html">` で正本へ送る(同期ルール準拠)。

### 移行時の注意

- 既存の「0. 文書情報」「1. 機能概要」表の情報は欠落させず、要約ブロック + `doc-meta` + 各節リードへ振り分ける。FR ID / SCR ID / 列挙値などの識別子は削除・改名しない。
- 節番号を振り直した場合、同一ページ内および被参照ページのアンカー(`#...`)整合を確認する。
- 改修後は検証手順(検索インデックス再生成 → `check-spec-sync.sh`)を実行する。

## 要件定義テンプレート(v2 / 正本サンプル: `01_メインシステム/01_要件定義/FR01_アカウント管理.html`)

要件定義層(`<システム>/01_要件定義/FRxx_*.html`)は本テンプレートを**正本**とする(上記「記載スタイル標準」の page-summary 2 列グリッド・section-lead は要件定義では使わない)。

### ページ骨格(要件定義)

1. **`<h1>`** — `FRxx: 名称`。
2. **`aside.page-summary`** — `ps-lead` のみ。`<strong>この要件は、〜を目的とし、以下について記載します。</strong>` の **具体的な目的 1 文**(抽象語を避ける)+ その下に概要トピックの `<ul>`。**2 列グリッド(ps-grid)・「正本・関連」リンク集は置かない**。
3. **`<p class="doc-meta">`** — `版数 / 更新 / 機能グループ / 優先度 / ステータス` を 1 行。
4. **`<h2>1. 業務要件</h2>`** — 表(`<table class="br-table">`)。列 = `ID(BR) / 観点 / ステークホルダ / 業務要件 / 関連 NFR`。
5. **`<h2>2. 機能要件</h2>`** — 表(`<table class="req-table">`)。列 = `ID(FR) / 要件 / 関連業務要件 / 優先度`。
6. **未確定事項** — `callout tip`(無ければ「確定済」と明記)。
7. **`<h2>更新履歴</h2>`** — 表。

**section-lead(節の説明リード)は置かない。** 見出しは番号 + 平易名のみ(範囲表記 `(FR-001〜022)` などは付けない)。

### 業務要件(セクション 1)の作法

- **業務要件 = 実現すべきこと**を先に定義し、機能要件はその実現手段として後に置く(この順序)。
- **ID = `BR-###`**(業務要件 / Business Requirement)。**システムごとの連番**(メイン・運営者で別名前空間)。欠番(廃番)は再利用しない。付与した BR は**正本である `01_要件定義/index.html` §5 業務要件一覧にも登録**する。
- ID 列の定義セルは `<td id="BR-###">BR-###</td>`(**アンカーのみ・リンクにしない**)。
- **5W1H を意識**して「いつ・何を・どうする」を 1 文で書く。**主語(誰)は本文に書かず「ステークホルダ」列**に出す(例 `アカウント利用者` / `システム` / `運営者`)。
- ステークホルダによって内容が異なる場合は、業務要件セル内で `<ul class="role-req"><li><strong>役割：</strong>…</li></ul>` の**役割キー付き箇条書き**で分ける。
- **関連列は NFR のみ**(`関連 NFR`)。**機能要件(FR)への参照は業務要件側に置かない**(トレーサビリティは機能要件 → 業務要件の一方向)。NFR 参照は `index.html#NFR-###` へリンク。該当なしは `—`。

### 機能要件(セクション 2)の作法

- **ID = `FR-###`**。ID 列の定義セルは `<td id="FR-###">FR-###</td>`(アンカーのみ・非リンク)。**新規に付与する ID は枝番(`FR-015a` のような英字サフィックス)を使わず**、システムごとの連番で採る(欠番=廃番は再利用しない)。既存の枝番 ID は据え置く(メイン / 運営者は同一トークン形式の FR-ID 名前空間を共有し、全層で約 1,400 箇所が相互参照するため、機械的な全面リナンバリングは参照整合を壊すリスクが高く行わない)。
- 列 = `ID / 要件 / 関連業務要件 / 優先度`。**関連業務要件列に、その FR が実現する `BR-###` をリンク**(`#BR-###`)で記す。該当なしは `—`。
- 優先度は**バッジ**(`<span class="badge badge-p0">P0</span>` / `badge-p1` / `badge-p2`)。
- **長い要件文は箇条書きで整理する**: 条件・手順・例外が複数ある要件は、だらだらと一文に詰め込まず、先頭に主旨を 1 文で述べ、詳細を `<ol class="alpha">`((a)(b)(c)、入れ子は `<ol class="alpha-sub">`=(a-1))の箇条書きにする。1 つの観点だけの短い要件は 1 文のままでよい。

### 共通(箇条書き・用語・参照)

- **列挙や複数観点は箇条書き**にする。マーカーは `<ol class="alpha">` = `(a)(b)(c)`、入れ子は `<ol class="alpha-sub">` = `(a-1)(a-2)`。
- 要件は **MVP の内容のみ**記載する(「将来対応 / MVP 対象外」等は書かない。退避先は `05_future/`)。
- **他システム(運営者 / 顧客管理)の要件・整合参照は書かない**。ただし他システムの操作を「契機」として書くのは可(主語はシステム)。
- 用語: ログイン可能なアカウント保有者(オーナー / プロジェクト管理者 / メンバー)は **「アカウント利用者」**、ウィジェットのエンドユーザーは **「ウィジェット利用者」** と区別する。
- 参照 ID は定義へリンク(定義セル自身は非リンク)。同一ページ内は `#ID`、他ページは `相対パス#ID`、NFR は `index.html#NFR-###`。

## 基本設計・詳細設計テンプレート(シンプル版 / 正本サンプル: `01_メインシステム/02_基本設計/08_認証・認可設計.html`)

基本設計(`<システム>/02_基本設計/`)・詳細設計(`<システム>/03_詳細設計/`)は本テンプレートを正本とする。**概要情報は最小限**にする(主たる読者は設計者本人)。

### ページ骨格

各節(ページ全体および各 `<h2>` / `<h3>`)は **タイトル → 説明 → テーブル → 補足(あれば)** の順で構成する。読者が上から読んで理解できることを最優先する。

1. **タイトル `<h1>` / 見出し `<h2>` `<h3>`** — 簡潔名(例「認証・認可設計」)。
2. **概要 `aside.page-summary`** — ページ冒頭。`ps-lead` の **目的 1 文のみ**(「このページでわかること」「正本・関連」リンク集・2 列グリッドは置かない)。直後に `<p class="doc-meta">`(`版数 / 更新 / ステータス` の 1 行)。
3. **説明 `<p class="section-lead">`** — **各節の見出し直後に必ず 1〜2 文の説明を置く**。「この節で何を定義するか・要点・前提」を平易に述べてから表へ入る(**表をいきなり置かない**)。設計書はセクションが多く複雑なため、説明が無いと上から読んで理解できない。
4. **テーブル** — 説明の後に本体の表を置く。各節は**表を主体**にする(「内容は原則すべて表」)。
5. **補足(あれば)** — 表で表しにくい注意・正本委譲・横断的な但し書きは、**当該節の表の後に** callout(note / important / warning / tip)で置く。
6. (任意)**更新履歴**。未確定事項は callout か省略。

### 本文の作法

- 旧「0. 文書情報 / 1. 文書概要(目的・対象範囲・版数表・関連ドキュメント表)」は廃止し、ps-lead + doc-meta へ集約する。**関連ドキュメントの大きなリンク表は作らない**(本文中のインラインリンクとポータルのナビで足りる)。**詳細設計(DD)は末尾に簡潔な「4. 関連設計」節を置く**(`<p class="section-lead">` にインラインリンク数個。表にしない。テスト観点を §5 とし欠番を作らない)。基本設計は関連設計節を設けない。
- **内容は原則すべて表(`<table>`)にする**。項目・パラメータ・分類・対応関係・手順など構造化できるものはすべて表で表現し、散文(`<p>`)と箇条書き(`<ul>` / `<ol>`)は最小限にとどめる(表にしにくい短い補足・正本委譲のみ `<p>` か callout)。手順は先頭に順序列(`#`)を持つ表にする。
- **フロー(処理・手続きの流れ)を解説する節は、シーケンス図で示す**。並びは 説明 → シーケンス図 →(補足)。シーケンス図は Mermaid を使い `<pre class="mermaid"><code>sequenceDiagram …</code></pre>` で書く(矢印は HTML エスケープ: `-&gt;&gt;` / `--&gt;&gt;`)。portal.js が CDN 読込でテーマ追従描画する。登場主体(participant)は利用者 / Worker / D1 / KV / メール等、当該フローの実体に合わせる。**問い合わせ(リクエスト `-&gt;&gt;`)には必ず応答線(戻り `--&gt;&gt;`)を対で描く**(検証・取得・保存などの呼び出しは結果の戻りまで示し、片方向で終わらせない)。ただし**応答線のラベルは分岐・結果値など意味がある場合のみ付け**、完了・受付・ID 採番などの自明な戻りはラベルを省く(空ラベル `X--&gt;&gt;Y:` が有効。図が読みにくくならないようにする)。**応答が複数の状態(例: 一致 / 不一致、有効 / 期限切れ)を返し、その後の流れが分かれる場合は、`alt` / `else`(任意分岐は `opt`)で状態ごとのシーケンスに分岐させて表現する**。返却値のラベルは **`名詞(値 / 値 / …)`** の形に統一する(例: `照合結果(一致 / 不一致)` / `契約状態(active / suspended / …)`。「〜の有無」「等」のような不統一な表記にしない)。
- **シーケンス図をテキストに起こしただけの手順テーブルは置かない**(図と重複)。表は図に無い情報(パラメータ・上限値・例外・対応関係など)を補うときのみ追加し、その場合は図のやりとり・分岐と対応させる(列例 = `# / やりとり(誰→誰)/ 内容 / 分岐・補足`)。図の無い単純なフローは表のみでよい。
- 表は「識別子 → 名称 → 説明 → 値 / 制約 → 参照」列順。コード値・ID・テーブル名・列名は `<code>`、空欄は `—`。
- 参照(FR / BR / NFR / テーブル / エラー ID / SCR)は定義の正本へリンクする。
- 重要な注意・正本委譲は callout(note / important / warning / tip)。
- DDL・API スキーマ・擬似コード等は `<pre><code>` でそのまま保持する(設計の実体)。

## 画面設計テンプレート(6 セクション / 1 画面 = 1 ファイル / 正本サンプル: `01_メインシステム/02_基本設計/01_画面設計/SCR_005.html`・`SCR_005-001.html`)

**目標構成(移行中)**: 画面設計は **1 画面 = 1 ファイル**の独立ポータルページに分割し、`01_メインシステム/02_基本設計/01_画面設計/` に置く。**親画面は `SCR_<番号>.html`、従属画面は `SCR_<番号>-NNN.html`、モーダルは `SCR_<親>-M<n>.html`**(一覧 / 詳細などは別 ID = 別ファイルとし、1 ファイルに複数画面を束ねない)。各ファイルは上記「基本設計・詳細設計テンプレート」を前提に、**以下の 6 セクション固定構成**(`<h2>` 番号付き)で記述する。正本サンプルは `SCR_005.html`(要対応の質問一覧)+ `SCR_005-001.html`(要対応の質問詳細)。

- **画面遷移図は廃止予定**。レイアウト(`<div class="scr-mock">` 配下のワイヤーフレーム)は各 SCR ファイル内へ取り込む(§2 参照)。**全画面の移行が完了したら画面遷移図(`画面遷移図_*.html` + `.view.html`)を完全削除する**。移行が完了するまでは旧モノリス `01_画面設計.html` を暫定で残す(移行済み画面は新ファイルへリンクで委譲)。
- 1 画面ファイルは 3 階層深い(`01_メインシステム/02_基本設計/01_画面設計/`)。`PORTAL_ROOT` とアセット参照は `../../../`、同階層の基本設計ファイルは `../<名前>.html`、要件定義は `../../01_要件定義/...`。`data-page-id` はルートからの相対パス。`<head>` に portal.js 一式 + `screen-mock.css` + lucide CDN を読み込む。
- ページ追加時は `assets/js/nav-data.js` に登録し `python3 99_script/build-search-index.py` を実行する。**メインの画面設計はナビ上で 1 グループに集約**: 基本設計カテゴリの `pages` 内に `{ "url": ".../01_画面設計/index.html", "title": "画面設計(メイン)", "kind": "group", "children": [ …各 SCR… ] }` を置く(`kind:"group"` は portal.js が折り畳みサブグループとして描画。children は SCR 番号順)。新規画面は `children` に SCR 番号順で 1 行追記する(`build-search-index.py` は children も索引化する)。

### 各 SCR 節の固定骨格

各 SCR ファイルは `<h1>` = `SCR-<番号> 名称`、直後にページ要約 `aside.page-summary`(`ps-lead` のみ)+ `doc-meta` を置き、続けて以下の `<h2>` を**この順序で**並べる。各 `<h2>` 直後に `section-lead`(1〜2 文)を置いてから表へ入る(表をいきなり置かない)。**ただし §3 画面レイアウトは section-lead を置かず**、見出し直後にレイアウトを置く。

1. **`1. 画面概要`** — section-lead に「何を行う画面か」を簡潔 1 文。続けて **当該画面の表**(列 = `画面 ID / 画面名 / 機能概要`。1 ファイル = 1 画面なので通常 1 行)。画面 ID は **親画面が `SCR-<番号>`(子番号を付けない)、従属画面が `SCR-<番号>-NNN`(3 桁連番)** とする(「サブ画面」という別概念は設けない。例 `SCR-005` 一覧〔親〕/ `SCR-005-001` 詳細〔子〕。定義セルは `<td id="SCR-...">`)。続けて **`関連` の表**(列 = `関連 / 内容`。行に `FR / BR`〔定義の正本へリンク〕と `関連画面`〔親 / 子 / 主要遷移先を当該 SCR ファイルへリンク〕)。さらに **ステークホルダの表**(列 = `ステークホルダ / 対象`。`オーナー` / `プロジェクト管理者` / `メンバー` を行に、利用するロールを `◯`・対象外を `—`。見出しラベルは付けず表のみ)。画面間の遷移は §2 で図示し、§1 には図を置かない。
2. **`2. 画面遷移図`** — section-lead 1 文の後、Mermaid `flowchart` で**画面 ID + 画面名をノード、イベント(操作)をエッジのラベル**として画面間遷移を示す。**ノードは画面 ID と画面名のみ**(「絞り込み / 一覧 / CSV」等の機能列挙や補足は書かない)。エッジは遷移を起こすイベント(例 `問い合わせ ID クリック` / `FAQ 登録へ`)。他画面(例 `SCR-006`)への遷移も対象に含める。Mermaid の矢印は HTML エスケープ(`--&gt;`)。
3. **`3. 画面レイアウト`**(**section-lead は置かない**)— サンプルデータを入力した見た目イメージを、**当該 SCR ファイル内に直接インラインで取り込む**(旧 画面遷移図のリッチなワイヤーフレーム markup `.app-layout` 等)。画面遷移図は独自のデザイントークン / コンポーネント CSS を持ちポータル本体と一部衝突するため、**全セレクタを `.scr-mock` にスコープした `assets/css/screen-mock.css` を `<head>` で読み込み、レイアウト markup を `<div class="scr-mock">…</div>` で包む**。`screen-mock.css` は `python3 99_script/build-screen-mock-css.py <画面遷移図.html>` で生成する(画面遷移図 `<style>` を `.scr-mock` 配下へスコープ + 印刷用 `min-height/page-break` を打ち消す inline フィットを付与。直接編集しない)。状態・画面が複数(通常 / 空 / エラー / 権限不足 / 一覧 / 詳細等)あれば各々を `<div class="layout-embed"><div class="le-cap">状態名(または画面名)</div><div class="scr-mock">…</div></div>` で並べ、**当該画面が取りうる全表示状態を省略・割愛せず描画する**。**le-cap およびモック画面内(`screen-header` 等)には画面 ID(`SCR-###`)を表示しない**(画面 ID の管理は §1 画面概要の定義表で行う)。アイコンは画面遷移図が lucide(`data-lucide`)を使うため、SCR ファイル `<head>` に lucide CDN を読み込み末尾で `lucide.createIcons()` を呼ぶ。**画面遷移図の完全削除後は、`screen-mock.css` が当該レイアウト CSS の最終的な置き場(コミット済みの正本)となる**。レイアウトは概観用で、項目の正本は §4。
4. **`4. 画面項目定義`** — 入出力項目を表で定義。**列 = `項目 ID / 項目 / 入出力 / 種類 / 表示条件 / 備考`**。**`表示条件` 列には、その項目の表示に条件がある場合のみ条件を簡潔に記す**(例: ロール別の出し分け、ログイン要否、状態依存(`suspended` 時 / 上限 ON 時 / 既読時 等)、モード依存(編集 / 新規))。常時表示の項目は `—`。条件は本列に集約し `備考` には重複記載しない。**項目 ID は画面内連番 `IT-01`、`IT-02`…**(画面をまたいだ大域連番にはしない。定義セルは `<td id="IT-..">`)。**一覧 / 検索系画面は、一覧の表示項目(絞り込み・列・件数表示・空状態)も本表に統合する**(独立した「一覧表示」節は設けない)。一覧表に「操作」列は設けず詳細遷移は ID 列リンクに集約する旨を section-lead か備考に記す。**クリック・押下・タップ・ホバー等のユーザー操作に伴う画面挙動(遷移・モーダル表示・コピー・既読化等)は §6 画面イベント一覧に記載し、§4 備考には書かない**(§4 は項目の静的属性・表示条件までを記し、挙動は §6 のイベントとして `関連項目` 列で当該項目 ID に紐づける。ステッパー表示〔①→②→③〕や状態モデルの値遷移など、ユーザー操作イベントでない記述は §4 のままでよい)。複数画面に分かれる場合は `<p class="ps-label">` でグルーピングして複数表に分ける。
5. **`5. 入出力一覧`** — 当画面が読み書きするテーブル・ファイルと呼び出す API の一覧。列 = `入出力名 / 種別 / I/O / アクセス種別(CRUD)/ 備考`で、**`アクセス種別(CRUD)` は `C` `R` `U` `D` の 4 列マトリクス**(`<th colspan="4">アクセス種別(CRUD)</th>` のグループ見出し + 2 段目に `C` / `R` / `U` / `D`。他の列は `rowspan="2"`)。セルは該当を `◯`・非該当を `—`。**`入出力名` は和名(テーブルの論理名・API / ファイルの和名)で記し、物理名(`<code>inquiries</code>` / `<code>GET /inquiries</code>` 等)は `備考` に置く**。`種別` = テーブル / ファイル / API。`I/O` は画面視点の 入力 / 出力(読取=入力・書込=出力)。CRUD はテーブルのみ印を付け、API・ファイルは全て `—`。`備考` に用途・物理名を記し正本(テーブル設計 / API 設計)へリンクする。
6. **`6. 画面イベント一覧`** — 画面で発生するイベントを表で列挙(列 = `イベント ID / イベント / 発生タイミング / 概要 / 関連項目`)。**イベント ID は画面内連番 `EV-01`、`EV-02`…**(画面をまたいだ大域連番にはしない)。**`関連項目` 列に当該イベントが扱う §4 の項目 ID を `<a href="#IT-..">` の同一ページ内リンクで紐づける**(複数はカンマ区切り、該当なしは `—`)。状況遷移など正本が他ページにあるものは表の後に callout で正本委譲する(イベントの詳細仕様・シーケンス図の独立セクションは設けない)。

### 補足

- 親画面・従属画面・モーダルは**それぞれ別ファイル**にする(例 一覧 `SCR_005.html` / 詳細 `SCR_005-001.html` / モーダル `SCR_005-M1.html`)。各ファイルの §1 は当該画面 1 行、関連は §1 の `関連画面:` 行で相互リンク、画面間遷移は §2 に図示する。ID は **親 `SCR-<番号>`(子番号なし)+ 従属 `SCR-<番号>-NNN`**。**「サブ画面」という別概念・別ラベルは設けない**。
- モーダル画面(`SCR-..-M#`)も独立 SCR ファイルとして本 6 セクション構成で記述する(「モーダル画面」節の規約に従い親画面の直後に配置)。一覧を持たない画面は §4 画面項目定義に一覧表示項目を設けない。
- 旧構成(`画面概要` + `表示・操作項目` の自由形式)から移行する際、既存の FR ID / SCR ID / 項目名 / `inquiries.status` 等の識別子・列挙値は欠落・改名させず、上記 6 セクションへ振り分ける。
- 前提・制約・但し書きなどの補足事項は素の `<p>` ではなく **callout(`note` 既定 / 重要は `important` / 注意は `warning` / 推奨は `tip`)** で記す(例: 割当前提・権限不足表示の注記)。
- 各 SCR ファイルの `<h2>` の `id` は画面内で一意にする(例 `5-画面項目定義`)。被参照アンカーがある場合は節番号変更時に整合を確認する。

## ポータル運用(HTML 正本)

- **内容の編集**: 各設計書 HTML の `<article class="content">…</article>` の中身を直接編集する。`<head>` のスクリプト読み込みや `<body data-page-id="…">` は編集しない。
- **ページの追加**:
  1. 対象ディレクトリに本文のみの HTML を作成する(既存ページの `<head>` / `<body data-page-id>` 構造を踏襲。`data-page-id` はリポジトリルートからの相対パス、`PORTAL_ROOT` は階層分の `../`)。
  2. `assets/js/nav-data.js` の該当システム → カテゴリの `pages` に 1 行追記する(`{ "url": "...", "title": "...", "kind": "md" }`。画面遷移図ラッパーは `"kind": "diagram"`)。
  3. `python3 99_script/build-search-index.py` を実行して検索インデックス(`assets/js/search-index.js`)を再生成する。
- **画面遷移図 / ER 図**: 既存の自己完結型 HTML(運営者 `02_運営者システム/画面遷移図.html` / 各システム `02_基本設計/ER図.html`)を原本とし、対応する `*.view.html` ラッパー(`portal.js` のシェル内で iframe 表示)を nav に登録する。原本内のリンクは `.html` を指す。原本に絵文字を使う場合は CDN アイコンに統一し、原本 `<head>` に Bootstrap Icons の `<link>` を追加する(原本は `portal.js` を読み込まないため)。「別タブで開く」リンクは `<i class="bi bi-box-arrow-up-right"></i>` を用いる。**メインの画面遷移図(`画面遷移図_*.html` + `.view.html`)は削除済み**で、レイアウトは各 `01_画面設計/SCR_*.html` の §3 に取り込まれている。
- **共通アセット**: `assets/css/style.css`(スタイル)、`assets/js/portal.js`(シェル構築)、`assets/js/nav-data.js`(ナビ定義・手編集)、`assets/js/search-index.js`(検索索引・スクリプト再生成)。
- **運用ルールの更新**: 本書 `CLAUDE.md` を直接編集する(運用ルールの正本。ポータルには展開しない)。

## 変更適用順序(5 段階)

仕様または設計の変更要求があった場合、ユーザーが特定のドキュメントだけを指し示していたとしても、必ず以下の **固定 5 段階順序** で全層を調査・適用すること。途中で省略すると下流に古い前提が残る。

1. **要件定義**(`01_メインシステム/01_要件定義/`、`02_運営者システム/01_要件定義/`)── 該当 FR グループファイル + `index.html`
2. **基本設計**(`01_メインシステム/02_基本設計/`、`02_運営者システム/02_基本設計/`)── 「変更種別 → 影響ドキュメント・マトリクス」に従って該当する 11 個別設計ファイルを更新
3. **詳細設計**(`01_メインシステム/03_詳細設計/`、`02_運営者システム/03_詳細設計/`)── 該当する DD グループファイル + `index.html`(基本設計が正本とする仕様は更新対象外、参照のみ)
4. **画面遷移図 / 画面レイアウト**(メイン: 該当する `01_メインシステム/02_基本設計/01_画面設計/SCR_*.html` の §3 画面レイアウト〔`.scr-mock` 取込〕を更新。旧 `画面遷移図_*.html` 4 ファイルは削除済みで存在しない。運営者: `02_運営者システム/画面遷移図.html`)── 画面構造の変更がある場合のみ。**メインはレイアウトを各 SCR ファイルへインライン取込済みのため、画面ごとに該当 SCR ファイルを個別に更新すること**
5. **運用設計 + 将来対応**(`<システム>/04_運用設計/`、`<システム>/05_future/`)── 監視 / バックアップ / リリース等への影響、または将来検討項目への退避

各層について、変更により削除・改名された語句(FR ID、列挙値、列名、画面 / 項目ラベル、メッセージ ID、エラー ID)をファイル内検索して取りこぼしを修正する。ページの追加・改名・削除を伴う場合は `assets/js/nav-data.js` を更新し `python3 99_script/build-search-index.py` を実行する。最後に `bash 99_script/check-spec-sync.sh` を実行する。ある層に関連内容が無いと判明した場合は、暗黙にスキップせず変更サマリへ明示的に記録する。

## モーダル画面

意味のある状態、バリデーション、業務ロジックを持つモーダルダイアログ(プロジェクト作成 / 編集、規約再同意、4-eyes 承認モーダル等)は、呼び出し画面に埋め込まず **独立した画面として SCR ID を付与して定義する**。

- **ID 規約**: 親画面の SCR ID に `-M<n>` サフィックスを付ける。親画面下の最初のモーダルは `SCR-<親>-M1`、同じ親に複数モーダルがあれば `-M2` / `-M3` …(例: SCR-004 の編集モーダル → `SCR-004-M1`)
- **配置**: モーダル行 / セクション / ワイヤーフレームブロックを、すべてのドキュメントで **親画面の直後** に置く ── 要件定義の SCR 一覧、`02_基本設計/01_画面設計.html` §2 / §5、サイドメニュー除外一覧、画面遷移図、`02_基本設計/07_トレーサビリティマトリクス.html` §3 / §4。SCR ID をアルファベット順にソートすると親画面の直下にモーダルが並ぶ
- **コンテンツ分割**: 親画面側は「モーダルを開く」操作のみを記載する。入力項目、バリデーション、保存 / キャンセルボタンはすべてモーダル SCR の項目表に記載する
- **モーダル画面に該当しないもの**: 単純な確認ダイアログ(「本当に削除しますか?」yes/no)や再認証チャレンジは独立 SCR にせず、呼び出しボタンの注記として `(確認ダイアログ)` のままインライン記載する

## 同期ルール

メイン / 運営者間の共有概念の正本ルールは [共有/共有概念.html](共有/共有概念.html) で正本化する。本書には主要ルールの要約のみ記載:

- 共有概念が変更される場合は、**共有概念.html 正本ルール表で示された正本ドキュメントを必ず更新** し、参照側ドキュメントはリンクのみ更新する(再掲禁止)
- 共有概念には以下を含む(全項目は共有概念.html 参照): `contract_owners.contract_status` / `inquiries.status` / 通知重要度 / SCR ID / AC ID / IF #1〜#12 / retention class / 法令プライバシー制約 / **オーナー / プロジェクト管理者 / メンバーの 3 ロールユーザーモデル**(`users` マスタ + `contract_owners`(オーナー専有属性、行存在 = オーナー判定)+ `project_users`(プロジェクト割当、`role`='admin'/'member')、契約境界キー = `contract_owner_user_id` ── メインを正本)/ ハッシュチェーン監査(運営者を正本)/ 運営者ログイン向け IP allowlist = `operator_ip_allowlist`(運営者を正本)/ PII 暗号化(メインを正本)/ 暗号鍵管理(メインを正本)/ 4-eyes 承認基盤(運営者を正本)/ Stripe Webhook 受信(運営者を正本)等
- `contract_owners.contract_status` は `active` / `suspended` / `deleted_pending` / `deleted` に固定し、`contract_owners` 行を持つユーザー(= オーナー)でのみ意味を持つ。v1.10 で `contract_status='deleted'` 設定時は必ず `contract_owners.valid=0` も同時セットする(`contract_owners` テーブルの CHECK 制約で強制)
- `inquiries.status` は 2 値(`open` / `closed`)。未解決質問登録時 `open`、SCR-005 詳細画面の手動操作で `open` ↔ `closed` を切替する(再オープン回数制限なし)。FAQ 下書き保存・FAQ 公開・個別チャットの操作は本カラムを変更しない(連動ロジックなし、FR-077)
- 通知重要度は `low` / `normal` / `high` / `critical`。`critical` はメール送信が必須となるイベント(`project_users.role='admin' AND valid=1` を 1 件でも保持する全ユーザーへ配信。v1.10 でオーナーも自動 admin 行を保持するため、本クエリ単独で網羅可能)に予約
- 利用者側ユーザーは **オーナー**(契約あたり 1 ユーザー固定、全権、MVP では譲渡不可、`contract_owners.user_id PRIMARY KEY` で 1 契約 = 1 オーナー固定が DB レベル担保される、**v1.10 でプロジェクト作成時に `project_users(role='admin', valid=1)` 行が自動 INSERT** される明示的なプロジェクト管理者保持者。認可フロー先頭の `isOwner=true` bypass(= `contract_owners` 行存在判定)は維持し、admin 行は通知宛先解決と画面表示の事実情報として参照される)と **メンバー**(0..N、`project_users.contract_owner_user_id` でオーナーに紐付き、プロジェクトごとに `project_users.role` = `admin`(プロジェクト管理者)または `member`(メンバー)で制御)に分かれる。オーナー専有機能(プロジェクト作成・編集・削除、課金、退会、規約再同意、メンバーユーザー削除)はプロジェクト別ロールでは付与できない。プロジェクト割当が 0 のメンバーはダッシュボードのみ利用可能。**v1.10 でプロジェクト作成時はオーナーが自動で当該プロジェクトの管理者となり、SCR-004-M1 から「管理者を指定」UI は撤去**(FR-030a 改訂)。**v1.10 でメンバーユーザー削除は論理削除(`users.valid=0`)。プロジェクト編集・削除動線は SCR-004 / SCR-004-M1 モーダルに集約**(SCR-008 プロジェクトホーム・SCR-016 には削除動線を置かず、SCR-008 は概要表示に徹する。一覧の「操作」列は撤去し行内リンクで編集 / 詳細へ遷移する方針)。論理削除データは `updated_at` 基準で 90 日経過後に物理削除バッチ(DD13)で物理削除される。全削除対象 10 テーブルに `valid INTEGER NOT NULL DEFAULT 1 CHECK (valid IN (0,1))` カラムを追加(`users` / `contract_owners` / `project_users` / `projects` / `allowed_domains` / `faqs` / `question_logs` / `inquiries` / `billing_subscriptions` / `project_quota_limits`)。月次上限件数・無料枠はプロジェクト単位(`project_quota_limits`、SCR-021 上限管理。オーナー / 当該プロジェクト管理者が設定可、運営者上書きは SCR-093)、レート制限は契約単位(`owner_quota_overrides`)。`usage_metering` はプロジェクト単位計測 + 契約単位請求集計
- `audit_logs.retention_class` は **クラス名表記**(`general` / `billing` / `operator_high_priv`)に統一(期間表記 `1y` / `5y` / `7y` は人間可読補助)

## 検証

設計書を編集した後は、同期スモークチェックを実行すること:

```sh
bash 99_script/check-spec-sync.sh
```

ページの追加・改名・削除を行った場合は、先に検索インデックスを再生成する:

```sh
python3 99_script/build-search-index.py
```

check-spec-sync.sh は HTML 正本を対象に SC-001〜SC-012 を実装している:

| 検査 ID | 検査内容 |
|---|---|
| SC-001 | 各基本設計 11 ファイルの必須見出し(`<h1>` / `<h2>`)存在 |
| SC-002 | 共有概念正本ファイルでの必須キーワード出現 |
| SC-003 | 参照側ドキュメントでの再掲禁止(リンクのみで実装、深い相互参照は手動レビュー)|
| SC-004 | SCR ID クロスチェック(画面設計 01 とトレーサビリティ 07 に全 SCR 出現)|
| SC-005 | メッセージ ID 体系 `MSG-SCR-*` の出現確認 |
| SC-006 | エラー ID 体系 `E-AUTH-*` / `E-AUTHZ-*` / `E-BIZ-*` / `E-OP-AUTH-*` / `E-OP-4EYES-*` の出現確認 |
| SC-007 | `contract_owners.contract_status` 4 値が課金・請求設計とテーブル設計に出現 |
| SC-008 | 通知重要度 4 値(`low` / `normal` / `high` / `critical`)+ 「強制送信」文言がメッセージ一覧に出現 |
| SC-009 | 連携 IF #1〜#12 がメイン側 API 設計 + 運営者側 API 設計(主管責任表)に出現 |
| SC-010 | 参照リンク到達性(主要相対パスのターゲット存在確認)|
| SC-011 | 要件定義 FR / 詳細設計 DD / 将来対応 FUT グループファイルが両システムで揃っているか |
| SC-012 | FR / DD / FUT、SCR、テーブル、API 節、メッセージ節、メール節の連番と廃番識別子の再混入防止 |

このスモークチェックは完全なレビューの代替ではない。要件定義と基本設計と詳細設計のドリフトを引き起こしたことのある高リスクな退行を検出する。

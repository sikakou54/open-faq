# FAQ 設計ポータル 保守ルール

本書(CLAUDE.md)はリポジトリの**保守・運用ルール正本**。各層テンプレート・記載スタイルは [`templates/`](templates/README.md) 配下へ分離(本書はテンプレート本文を持たない)。「FAQ AI ウィジェット SaaS / メインシステム」の**要件定義書・基本設計書**を **Markdown** で管理。1 識別子 = 1 ファイルの個別ページ構成、[`README.md`](README.md) が入口、各識別子はフラット連番・ゼロ詰め 3 桁・欠番なし。

> [!IMPORTANT]
> **読み順(正本)** 要件定義 ＞ 業務UC ＞ 画面設計 ＞ システム設計 ＞ API ＞ DB ＞ シーケンス ＞ 権限/エラー/MSG。`要件↔UC↔(SCR/SYS)↔API↔TBL` のトレーサビリティは [トレーサビリティ一覧](02_basic_design/00_traceability/index.md) に**一元管理**(各設計書には業務的文脈を示すため `UC-NNN` を記載し、全層の厳密な紐付け `TR-NNN` はトレーサビリティ一覧のみに保持する。逆引きは行わない)。画面起点UC→SCR、システム起点(バッチ/Webhook/非同期/監視/通知)UC→SYS。

## 構成
ファイル名は英名(ASCII/ID ベース)、タイトル・見出しは日本語。3 グループ(`01_requirements`/`02_basic_design`/`03_future`)。各セクション = `index.md` + 配下個別ページ。

├── 03_non_functional_requirement/ #   NFR: 07_nfr.md(通し連番)

- ID 1件=1ファイル(原則)。**要件仕様(BR/FR/NFR/RULE)はカテゴリ別・種別別ファイルへ統合**(ID・アンカー・採番不変)。**BR は HTML テーブル行**(`<span id="ID"></span>` を ID セル先頭)、FR/NFR/RULE は節 `## <span id="ID"></span>ID: 名称` で保持。
- ページ間リンクはすべて `.md`。相互参照は下記「相互参照アンカー」参照。

> [!IMPORTANT]
> **横断正本(集約ドキュメント)** 次の3つは各層に散在しがちな情報の**単一の真実(正本)**。具体的な数値(タイムアウト、上限等)、状態(ステータス)の意味、データ保持期間は、それぞれシステム仕様書・状態モデルのみを正本とし、各設計書からは具体値を書かず正本へのリンクまたはID参照で示す(重複定義・本文外委譲・`等`での打ち切りをしない)。
> - **[システム仕様書](02_basic_design/07_system-spec.md)** — 設計値の正本(しきい値/課金単価/無料枠/タイムアウト/保持期間/AIしきい値の3階層継承)。値は業務的根拠 RULE/FR/NFR を併記。
> - **[状態モデル](02_basic_design/08_state-model.md)** — 状態の正本(アカウント/課金アカウント/メンバー割当/FAQ・未解決質問の一覧+遷移図)。状態名はここに統一。
> - **[用語集](01_requirements/00_glossary.md)** — 用語の正本(正式名/別名/意味)。表記揺れはここで吸収。
> - **enum(列挙値)** は API は §列挙値、TBL は §コード値 に**取りうる値の全集合と意味**を明記。全体共通の状態 enum は状態モデルを参照。

## ドキュメントの粒度分離ルール
**上位から下位へ粒度を分離し、層の役割を混在させない。** 業務要件 → 機能要件・非機能要件 → 業務UC → 基本設計

| 層 | 書くこと |
|----|----|
| BR | 業務上実現したいこと(誰が・何のために・何を可能にするか) |
| FR | システムが提供する機能 |
| NFR | 性能・セキュリティ・可用性・運用性などの品質要件 |
| UC | 業務目的を達成するための利用者・システムの流れ |
| 基本設計(SCR/SYS/API/TBL/SEQ ほか) | 要件をシステムとしてどう実現するか |

- **過剰な実装詳細を上位層へ書かない。** 業務要件・UCに 画面ID/API ID/テーブル名/物理名/カラム名/SQL/クラス名等を書かない。業務語へ言い換えるか下位層へ委ねる。

> [!IMPORTANT]
> **UC本文には基本設計レベルの情報を書かない。** SCR/EVT/API/TBL/SEQ/ERR/MSG、画面名・ボタン名・API名/エンドポイント/メソッド/ステータスコード、テーブル物理名・カラム名・SQL、クラス/関数名 — すべて業務語へ言い換えて除去。ただし、**ユーザー(U)・UI・サーバー(S)の3者に限定した要件定義レベルのシーケンス図**を ## 基本フロー の冒頭に配置することは、業務フローの具体化として推奨される。図内の文言は業務語に限定し、SCR-xxx/API-xxx/TBL名/カラム名等の実装詳細は含めない。

- **BR と FR で重複を作らない。** BR=業務意図(1〜2文)。FR=システム機能の細粒度。数値しきい値・手順番号・状態名・UI文言・FRとほぼ同文の機能記述は BR に書かず対応 FR/RULE へ委ねる。FR に無い業務理由・価値・ステークホルダ別役割・固有方針は BR に残す。

- **権限設計(PERM)の記述粒度**: 権限設計は「第2層: 境界判定（404偽装）」と「第3層: 操作権限（403拒否）」を分離して定義する。「境界判定」は PERM-005 が正本であり、割当なし・部外者を 404 偽装で除外する入場ゲートである。「操作権限」は境界を通過した関係者（オーナー / メンバー）に対して許可・拒否を定める。基本設計層の PERM 文書には物理実装（テーブル名・カラム名・SQL 条件式・ORM クエリ等）を記述せず、「対象プロジェクトの作成者であること」「当該プロジェクトへの有効な割当を持つこと」のような業務的な判定条件を記述する。物理実装の詳細は詳細設計に委ねる。

> [!WARNING]
> **層横断トレーサビリティはドキュメントに書かず [トレーサビリティ一覧](02_basic_design/00_traceability/index.md) に一元管理。** 各ドキュメントには業務的文脈を示すために業務ユースケースID（`UC-NNN`）を記載し、全層の厳密な紐付け（`TR-NNN`）はトレーサビリティ一覧にのみ保持する。個別設計書からの逆引き（TR記載）は行わない。禁止する逆引き記載(専用フィールド・節・表行): UCの`トレーサビリティID`、各設計の`トレーサビリティID`、SCRの`関連FR/BR`、APIの`トレーサビリティID`、SYSの関連表`FR/BR/RULE`行、TBLの`使用元`、SEQ§項目の`対応BR/FR`、PERMの`由来要件`。残してよいのは設計仕様そのもの(APIの`利用テーブル`・`エラー`、SCR/API/SYS/SEQ/PERM/ERRのヘッダーに置く`業務ユースケースID`、SCRの`入出力一覧`、TBLのFK、SEQ§項目の`関連SCR/API/TBL/EVT/ERR/MSG`、PERMの`対応SCR/EVT/API`等)。設計根拠の括弧引用(例:「12文字以上(FR-006)」)とSEQが主題とするUCの言及は許容(専用のTR対応表・結線節に列挙し直さない)。要件(BR/FR/NFR/RULE)は後工程へのリンク・委譲文を持たない。

## 設計変更時の影響調査・反映ルール
変更した層から**下流・上流・関連の双方向**に影響調査と反映を行う(省略不可)。影響調査と修正レビューは **NotebookLM を活用** し、指摘がなくなるまで繰り返す。

### 上流変更 → 下流影響調査
| 変更層 | 影響調査の対象 |
|----|----|
| BR | FR/NFR/UC/基本設計 |
| FR | UC/基本設計 |
| NFR | 基本設計/運用設計 |
| UC | 基本設計 |

### 下流(基本設計)追加・変更 → 上流・関連への反映
> [!IMPORTANT]
> **基本設計(SCR/SYS/API/TBL/SEQ/PERM/ERR/MSG)を追加・変更したら上流(要件・UC)と関連(同層・隣接層)へ必ず反映。** 新機能を基本設計に足す場合も同様。下流だけで完結させない。

| 追加・変更 | 反映・整合の対象 |
|----|----|
| SCR/SYS | 由来BR/FR/UC確認(無ければ起こす)。SEQ/PERM/ERR/MSG整備 |
| API | 対応UC・由来FR、利用TBL、SEQ、ERRへの反映 |
| TBL | 利用API・対応UC、ER図・index一覧への反映 |
| SEQ | 対応UC・関連SCR/API/TBL/ERR/MSGとの結線整合 |

- 新機能は対応 BR/FR/UC を上流に起こし双方向トレース(要件↔UC↔基本設計)を成立させる。既存機能の削除・改名・統合も同パスで反映。
- 影響なし時も「影響調査を実施した」ことを作業ログに残す。反映後は採番(欠番なし)・相互参照アンカー(0/0)・各index一覧を確認。

## ナビゲーション・索引
- **トップページ** = [`README.md`](README.md)(手書き最小)。各セクション `index.md` への入口。
- **各セクション一覧** = 各フォルダの `index.md`(手動保守)。右TOCはビューア自動生成のため埋め込まない。

### 相互参照アンカー(最重要)
**定義側**: `<span id="ID"></span>` を埋める(見出し先頭 `# <span id="FR-001"></span>FR-001: …` or 表のIDセル先頭)。新規定義行を追加したら必ず付ける。**参照側**: `[ID](相対パス.md#ID)` で送る。`../` の数は参照元ファイルの root からの深さで決まる。

- **要件仕様特例**: ファイル名は「カテゴリ+種別」(例 `02_faq-ai-fr.md`)。アンカーは `#<ID>`。
- **EVT特例**: 独立ページなし。SCR §6 に非リンクアンカー(画面ごとローカル・各画面でEVT-01から)。他層は `SCR-NNN EVT-MM` プレーンテキストで参照(リンク不可)。
- **SEV特例**: 広域ID。SYS §6 に非リンクアンカー。参照側はプレーンテキスト広域ID。
- **TR特例(最重要)**: 定義は[トレーサビリティ一覧](02_basic_design/00_traceability/index.md)のみ。個別ドキュメントから `[TR-NNN](...)` を記載しない。基本設計の各ヘッダーは業務ユースケースID（`UC-NNN`）と設計仕様上必要な関連ID（API/TBL/EVT等）だけを持つ。

相対パス早見表(参照元 → 参照先):

| 参照元 → 参照先 | 相対パス例 |
|----|----|
| 同一ページ | `[FR-001](#FR-001)` |
| 要件仕様 同種別フォルダ内 | `[FR-007](02_faq-ai-fr.md#FR-007)` / `[BR-001](01_account-br.md#BR-001)` |
| 要件仕様 種別跨ぎ・UC→要件 | `[FR-001](../02_functional_requirement/01_account-fr.md#FR-001)` / `[RULE-001](../01_business_requirement/08_rule.md#RULE-001)` |
| 同一フォルダ別ページ(SCR/SYS) | `[SCR-002](SCR-002.md#SCR-002)` / `[SYS-002](SYS-002.md#SYS-002)` |
| 基本設計 同グルーピング内(兄弟フォルダ) | API→TBL `[TBL-001](../04_database/TBL-001.md#TBL-001)` / SYS→API `[API-060](../03_apis/API-060.md#API-060)` |

| 各設計 → トレーサビリティ一覧 | 個別設計書には `TR-NNN` を記載しない。必要時は一覧ページ自体へのリンク `[トレーサビリティ一覧](../../00_traceability/index.md)` のみに留める |

ページ追加時: `<ID>.md` を作成し当該 `index.md`(必要なら [`README.md`](README.md))の一覧へ手で追記。[templates/](templates/README.md) の該当テンプレートを雛形にする。編集後は壊れリンク・アンカー 0/0 を確認。

## ページ形式
各ページは**自己完結 Markdown**。

- 図は ` ```mermaid ` コードフェンスで記述。**開き・閉じは必ず独立行**(同一行連結で描画破壊)。ノードラベルに丸括弧等を含む場合は `"..."` で囲む。**絵文字は使わない。**
- **テンプレート外側に記述しない。** 固定セクション・表の内側だけに記述。セクション外へ箇条書き・補足・`>[!NOTE]`等を勝手に足さない。列構成も増減しない。値の選択肢・状態パターンは所定の表(画面設計はデータパターン表)で表し表外の箇条書きで列挙しない。**別セクションで分かる内容を重複記載しない**(SCR: §2遷移図フローを§1/§7で散文化しない。§4/§5/§8は定義セクション外で繰り返さない)。置き場所が無い情報は GitHub Issue で課題化。
- **読めば分かることを説明文で繰り返さない。** 表・見出し・セクション構成・リンクを見れば把握できる事実(例「○○ / ○○ … の N セクションで記述します」のようなセクション一覧の言い換え、表のカラムや行の散文化、自明な定義の再掲)は説明文・前置きに書かない。説明文は「何を・何のために定義するページか」の要旨に留める。
- **基本設計に詳細設計粒度を書かない。** 変数名・JSONキー・物理カラム名・アルゴリズム・コードポイント等は書かない。画面イベント(SCR §7)等は業務的結果(遷移先・**呼び出すAPI**・表示/保存内容・成功/失敗結果)を記す。画面内部変数・状態保持・クエリ構築・処理ステップ列挙等の実装機構は詳細設計に委ねる。
- **画面設計(SCR)の遷移図・項目一覧では全画面共通のグローバルナビ(ダッシュボード/利用状況/プロジェクト/請求/設定)を省略する。**
- 各ページの骨格・文章ルール・表の作法は [`templates/00_common-style.md`](templates/00_common-style.md) を正本とする。

## 画面モックの画像化
GitHub は埋め込み HTML の `style=`/`class=` を除去するため §3 画面レイアウトは **PNG 画像**で表示。HTML は MD に埋め込まず `mocks/<画面>-<n>.html` に外出し(MD は `mocks/<画面>-<n>.png` の 1 行のみ)。**`../` 親参照不可**(Cursor/VS Code でプレビュー不可)→ `mocks/` は必ず SCR ファイルと同階層配下(`02_basic_design/01_frontend/01_screens/mocks/`)に置く。モック変更時は `.html` を直接編集し PNG を再生成:

```sh
node ~/.claude/skills/html-to-png/scripts/html_to_png.js 02_basic_design/01_frontend/01_screens/mocks --selector "#wrap"
```

## NotebookLM 活用方針
> [!IMPORTANT]
> **ドキュメントを読んで事実確認・仕様確認を行う作業はすべて NotebookLM に依頼する。** レビュー・影響調査・整合性確認いずれも同様。使用は必ずスキル(`full-layer-review`)経由。直接操作しない(`full-layer-review` が全層フルレビュー・`nlm` CLI 操作・判定スクリプトを内包)。
> **ドキュメントを更新したら NotebookLM のソースを必ず更新する。** 対象ノート: https://notebooklm.google.com/notebook/047ad8fb-70fd-4569-b5cb-4bddf0bd806a

レビュー観点: 構成・粒度の適切さ / 抜け漏れ / ドキュメント間整合性 / 粒度混在 / 上位変更の下位反映 / 逆参照有無

> [!WARNING]
> **指摘があれば修正し再レビュー。指摘が無くなるまで繰り返す。** 1回で打ち切らない。

## Agent 実行ルール
> [!IMPORTANT]
> **「統括Agent + 作業Agent」構成で実行する。** 1人のAgentだけで完結させない。

| 役割 | 担当 |
|----|----|
| 統括Agent | 方針決定・作業分解・進捗管理・成果物統合・レビュー依頼・最終確認 |
| 調査Agent | 割り当てられた範囲の調査(最大3Agent) |
| 作業Agent | 割り当てられた範囲の編集・修正(最大3Agent) |
| レビューAgent | 割り当てられた範囲のレビュー(最大3Agent) |
| Issue管理Agent | Issue登録の依頼受付・実施 |
| NoteBookLM管理Agent | 依頼事項をNoteBookLMへ受け渡し、得られた結果を整理して依頼元Agentへ返却する窓口役 |

## 課題管理(GitHub Issue)

作業中の課題は GitHub Issue に登録し解決したらクローズ(**記録なしで放置しない**)。

### 登録

**登録すべきケース:** 要件不足 / 仕様解釈が複数 / ドキュメント矛盾 / 影響範囲が広い / 関係者判断が必要 / その場で解決不可 / 後続作業に影響。`gh issue create` で起票、表題に区分(`traceability`/`design-gap`/`needs-review`等)を付ける。

> [!IMPORTANT]
> **Issue本文は中学生にも分かる表現で。** 専門用語には説明を添える。**必ず提案を添える。** 見出し構成: `# 概要`(何が問題) / `# 背景`(なぜ発生) / `# 影響`(放置すると何に困るか) / `# 提案`(対応案) / `# 確認したいこと`(関係者への判断依頼) / `# 関連ドキュメント`(ファイル名・パス)。

### クローズ

解決コミットに `Closes #<番号>` を入れるか `gh issue close <番号>` で Close。

> [!WARNING]
> **Close前に確認:** 課題解決済み / 関連ドキュメント更新済み / 影響調査完了 / NotebookLMレビューで問題なし / 対応内容がIssueにコメント済み。
